# State Machine

A Stacktape state machine uses AWS Step Functions to orchestrate multi-step workflows with branching, retries, parallel execution, and error handling — without writing orchestration code. Define your workflow in [Amazon States Language (ASL)](https://states-language.net/spec.html), and Stacktape handles the underlying AWS infrastructure. Pay ~$0.025 per 1,000 state transitions (Standard workflows).

## When to use

Stacktape state machines are the right choice when your workload involves coordinating multiple steps with dependencies between them. Use a state machine instead of chaining [Lambda function](/resources/compute/lambda-function) calls manually whenever you need visibility into execution state, built-in retries, or branching logic.

- **Multi-step processing pipelines** — order processing, ETL pipelines, or data ingestion flows where each step depends on the previous one's output.
- **Workflows with conditional branching** — approval flows, content moderation pipelines, or any process that takes different paths based on intermediate results.
- **Tasks requiring automatic retries** — payment processing, third-party API calls, or anything that may fail transiently and needs exponential backoff without custom retry logic.
- **Long-running orchestration** — processes that span minutes, hours, or days (AWS Standard workflows can run up to one year), such as batch processing or multi-stage deployments.
- **Parallel fan-out / fan-in** — splitting work across multiple Lambda functions in parallel and aggregating results.

## When NOT to use

Not every coordination problem needs a state machine. For straightforward async processing, single-step operations, or simple event fan-out, lighter Stacktape resources handle the job at lower cost and complexity. State machine transitions cost ~$0.025 per 1,000, which adds up at high volume.

| Scenario | Better alternative |
|---|---|
| Simple request-response API | [HTTP API Gateway](/resources/networking/http-api-gateway) with a single Lambda function |
| One function calling another | Direct invocation via AWS SDK — no orchestration needed |
| Decoupled async processing | [SQS queue](/resources/messaging/sqs-queue) trigger on a Lambda function |
| Fan-out to multiple consumers | [SNS topic](/resources/messaging/sns-topic) or [EventBridge event bus](/resources/messaging/event-bus) |
| High-throughput stream processing | [Kinesis stream](/resources/messaging/kinesis-stream) with a Lambda consumer |
| Simple scheduled task | A Lambda function with a [schedule trigger](/configuration/triggers/schedule-triggers) |

## Basic example

This example defines two [Lambda functions](/resources/compute/lambda-function) orchestrated by a state machine. The workflow validates an order, branches on the result, and either processes the order or fails. Task states use CloudFormation intrinsic functions to reference Lambda ARNs at deploy time.

<CodeBlock

  defineConfig,
  LambdaFunction,
  StateMachine,
  StacktapeLambdaBuildpackPackaging
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const validateOrder = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/validate-order.ts' })
  });

  const processOrder = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/process-order.ts' }),
    timeout: 60
  });

  const orderWorkflow = new StateMachine({
    definition: {
      Comment: 'Order processing workflow',
      StartAt: 'ValidateOrder',
      States: {
        ValidateOrder: {
          Type: 'Task',
          Resource: { 'Fn::GetAtt': ['ValidateOrderFunction', 'Arn'] },
          Next: 'CheckValidity',
          Catch: [{ ErrorEquals: ['States.ALL'], Next: 'OrderFailed' }]
        },
        CheckValidity: {
          Type: 'Choice',
          Choices: [{ Variable: '$.valid', BooleanEquals: true, Next: 'ProcessOrder' }],
          Default: 'OrderFailed'
        },
        ProcessOrder: {
          Type: 'Task',
          Resource: { 'Fn::GetAtt': ['ProcessOrderFunction', 'Arn'] },
          Retry: [
            {
              ErrorEquals: ['States.TaskFailed'],
              IntervalSeconds: 3,
              MaxAttempts: 2,
              BackoffRate: 2
            }
          ],
          End: true
        },
        OrderFailed: {
          Type: 'Fail',
          Error: 'OrderProcessingFailed',
          Cause: 'Validation or processing failed'
        }
      }
    }
  });

  return {
    resources: { validateOrder, processOrder, orderWorkflow }
  };
});
// [!code focus-end]`
    }
  ]}
/>

A state machine definition requires `StartAt` and `States`. `StartAt` names the first state to execute, and `States` maps state names to state objects — each with a `Type` field. Most non-terminal states transition with `Next` or finish with `End: true`; terminal `Succeed` and `Fail` states end execution without needing `Next` or `End`.

The logical names used in examples on this page are illustrative — always verify against the compiled template before hard-coding them.

## Workflow definition

The `StateMachine` resource accepts a single `definition` property using [Amazon States Language (ASL)](https://states-language.net/spec.html) — a JSON-based spec for defining Step Functions workflows. A definition requires `StartAt` (the first state to execute) and `States` (a map of state names to state objects). You can optionally set `Comment` for a human-readable description, `Version` for the ASL version, and `TimeoutSeconds` to cap the maximum execution time for the entire workflow.

### State types

The Stacktape `StateMachine` definition models eight ASL state types: Choice, Fail, Map, Parallel, Pass, Succeed, Task, and Wait. Each state type serves a distinct role in the workflow.

| State type | Purpose | Key properties |
|---|---|---|
| **Task** | Invokes a Lambda function or another supported ASL task target | `Resource`, `Retry`, `Catch`, `TimeoutSeconds`, `HeartbeatSeconds`, `Parameters` |
| **Choice** | Branches based on input conditions | `Choices` (condition rules), `Default` (fallback state) |
| **Parallel** | Runs multiple branches concurrently, waits for all to complete | `Branches`, `Retry`, `Catch` |
| **Map** | Iterates over an array, running a sub-workflow per item | `Iterator`, `ItemsPath`, `MaxConcurrency` |
| **Wait** | Pauses execution for a duration or until a timestamp | `Seconds`, `Timestamp`, `SecondsPath`, `TimestampPath` |
| **Pass** | Passes input to output, optionally injecting or transforming data | `Result`, `Parameters`, `ResultPath` |
| **Succeed** | Marks successful completion — terminates the workflow | — |
| **Fail** | Terminates execution with an error | `Error`, `Cause` |

**Task** is the most common state type — it's how the state machine invokes your Lambda functions and other AWS services. **Choice** is the branching mechanism (equivalent to if/else). **Parallel** and **Map** handle fan-out patterns. **Succeed** and **Fail** are terminal states that end execution without needing `Next` or `End`. See the [ASL specification](https://states-language.net/spec.html) for full syntax details.

### Referencing Lambda functions

Task states use the `Resource` property to identify a Lambda function or another supported ASL task target. The `Resource` value can be a string or an object. Use an object for CloudFormation intrinsic functions like `Fn::GetAtt` to reference Lambda ARNs that resolve at deploy time. Use a string when you have a concrete ARN or a Step Functions service integration identifier (e.g., `arn:aws:states:::dynamodb:putItem`).

```json
{
  "Type": "Task",
  "Resource": { "Fn::GetAtt": ["MyFunctionLogicalName", "Arn"] }
}
```

Run [`stacktape compile-template`](/cli/compile-template) to find the CloudFormation logical name for each resource in your generated template. Do not hard-code logical names without verifying them against the compiled output.

You can also set `TimeoutSeconds` on a Task state to limit how long the state machine waits for the task to complete, and `HeartbeatSeconds` to require periodic heartbeat signals from long-running tasks. The `Parameters` property lets you reshape the input before passing it to the task.

### Workflow timeout

Set `TimeoutSeconds` at the top level of your definition to limit the maximum execution time for the entire workflow. This is a safeguard against runaway executions — without it, AWS Standard Step Functions workflows can run for up to one year. For any workflow that should complete within a predictable window (minutes or hours), always set a timeout.

## Error handling

AWS Step Functions provides built-in error handling through `Retry` and `Catch` blocks on Task, Parallel, and Map states. This replaces the need for custom retry logic in your Lambda function code and gives you a declarative way to handle failures at the orchestration layer.

**Retry** accepts an array of rules, each specifying which errors to match (`ErrorEquals`), the initial delay (`IntervalSeconds`), maximum attempts (`MaxAttempts`), and backoff multiplier (`BackoffRate`). Multiple retry rules are evaluated in order — the first matching rule is used.

**Catch** routes to a fallback state when all retries are exhausted or for errors not covered by retry rules. Each catch rule matches error types (`ErrorEquals`) and redirects to a named state (`Next`). This lets you build graceful degradation paths — for example, notifying a support team when payment processing fails after retries.

<CodeBlock
  intellisense
  tabs={[

      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StateMachine,
  StacktapeLambdaBuildpackPackaging
} from 'stacktape';

export default defineConfig(() => {
  const chargePayment = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/charge.ts' }),
    timeout: 60
  });

  const notifySupport = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/notify-support.ts' })
  });

  // [!code focus-start]
  const paymentWorkflow = new StateMachine({
    definition: {
      StartAt: 'ChargePayment',
      States: {
        ChargePayment: {
          Type: 'Task',
          Resource: { 'Fn::GetAtt': ['ChargePaymentFunction', 'Arn'] },
          Retry: [
            {
              ErrorEquals: ['States.TaskFailed'],
              IntervalSeconds: 5,
              MaxAttempts: 3,
              BackoffRate: 2
            }
          ],
          Catch: [
            {
              ErrorEquals: ['States.ALL'],
              Next: 'NotifySupport'
            }
          ],
          Next: 'PaymentComplete'
        },
        NotifySupport: {
          Type: 'Task',
          Resource: { 'Fn::GetAtt': ['NotifySupportFunction', 'Arn'] },
          Next: 'PaymentFailed'
        },
        PaymentFailed: {
          Type: 'Fail',
          Error: 'PaymentError',
          Cause: 'Payment failed after 3 retries'
        },
        PaymentComplete: {
          Type: 'Succeed'
        }
      }
    }
  });
  // [!code focus-end]

  return {
    resources: { chargePayment, notifySupport, paymentWorkflow }
  };
});`
    }
  ]}
/>

In this example, `ChargePayment` retries up to 3 times on failure with exponential backoff (5s, 10s, 20s). If all retries fail, `Catch` redirects to `NotifySupport`, which alerts the support team before the workflow terminates with a `Fail` state. Without the retry block, any transient error would immediately fail the workflow.

ASL defines several reserved error names: `States.ALL` (matches any error), `States.TaskFailed` (Lambda errors and task failures), `States.Timeout` (task exceeded `TimeoutSeconds`), and `States.HeartbeatTimeout` (missed heartbeat signal). You can also match on custom error strings thrown by your Lambda functions.

## Starting executions

A Stacktape state machine does not start automatically — you start executions programmatically using the AWS SDK from another resource. Use [`connectTo`](/configuration/connecting-resources) to grant a Lambda function the IAM permissions it needs to interact with the state machine, and the [`$ResourceParam` directive](/configuration/directives) to inject the state machine ARN as an environment variable.

<CodeBlock

  defineConfig,
  LambdaFunction,

  HttpApiGateway,
  HttpApiIntegration
} from 'stacktape';

export default defineConfig(() => {
  const processOrder = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/process-order.ts' })
  });

  const orderWorkflow = new StateMachine({
    definition: {
      StartAt: 'ProcessOrder',
      States: {
        ProcessOrder: {
          Type: 'Task',
          Resource: { 'Fn::GetAtt': ['ProcessOrderFunction', 'Arn'] },
          End: true
        }
      }
    }
  });

  const apiGateway = new HttpApiGateway({});

  // [!code focus-start]
  const startOrder = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/start-order.ts' }),
    connectTo: ['orderWorkflow'],
    environment: [{ name: 'ORDER_WORKFLOW_ARN', value: "$ResourceParam('orderWorkflow', 'arn')" }],
    events: [
      new HttpApiIntegration({
        httpApiGatewayName: 'apiGateway',
        path: '/orders',
        method: 'POST'
      })
    ]
  });
  // [!code focus-end]

  return {
    resources: { processOrder, orderWorkflow, apiGateway, startOrder }
  };
});`
    }
  ]}
/>

Listing `'orderWorkflow'` in `connectTo` adds the IAM permissions required to interact with the state machine to the consuming workload's role. The `environment` entry uses `$ResourceParam('orderWorkflow', 'arn')` to inject the state machine ARN at deploy time. Run [`stacktape compile-template`](/cli/compile-template) to inspect the exact IAM actions granted. State machines expose `arn` and `name` as [referenceable parameters](/configuration/referenceable-parameters).

<Info>
State machines are not listed in the `connectTo` environment variable injection table — `connectTo` grants IAM permissions but does not automatically inject environment variables for state machines. Use the `$ResourceParam` directive to pass the state machine ARN explicitly.
</Info>

Use the ARN in your handler to start executions:

```typescript
import { Hono } from 'hono';
import { handle } from 'hono/aws-lambda';
import { SFNClient, StartExecutionCommand, DescribeExecutionCommand } from '@aws-sdk/client-sfn';

const sfn = new SFNClient({});
const stateMachineArn = process.env.ORDER_WORKFLOW_ARN!;

const app = new Hono();

app.post('/orders', async (c) => {
  const input = await c.req.text() || '{}';
  const result = await sfn.send(
    new StartExecutionCommand({ stateMachineArn, input })
  );
  return c.json({ executionArn: result.executionArn }, 202);
});

app.get('/orders/status/:executionArn{.+}', async (c) => {
  const executionArn = decodeURIComponent(c.req.param('executionArn'));
  const result = await sfn.send(new DescribeExecutionCommand({ executionArn }));
  return c.json({
    status: result.status,
    output: result.output ? JSON.parse(result.output) : null
  });
});

export default handle(app);
```

This pattern exposes an HTTP endpoint that starts a workflow execution and returns the execution ARN. A second endpoint lets clients poll for the execution status. The `@aws-sdk/client-sfn` package provides the `StartExecutionCommand`, `DescribeExecutionCommand`, `StopExecutionCommand`, and other Step Functions operations.

## Overrides

The `StateMachine` resource supports [overrides](/configuration/overrides-and-escape-hatches) for modifying the generated CloudFormation resources. The `StateMachineProps` type exposes only the ASL `definition` property — CloudFormation-level settings that are not modeled by Stacktape (such as workflow type, logging configuration, or tracing) can be set through `overrides` on the generated CloudFormation state machine resource.

Use [`stacktape compile-template`](/cli/compile-template) to find the logical names of the generated CloudFormation resources, then apply dot-notation overrides. See [overrides](/configuration/overrides-and-escape-hatches) for details on the syntax.

<Tip>
Stacktape automatically creates an execution role for state machines in your stack. Run [`stacktape compile-template`](/cli/compile-template) to inspect the exact IAM policy attached to the role. Use `overrides` if you need to restrict or extend the permissions.
</Tip>

## FAQ

### How do I reference Lambda functions in a state machine definition?

Use CloudFormation intrinsic functions in the Task state's `Resource` field — for example, `{ 'Fn::GetAtt': ['MyFunctionLogicalName', 'Arn'] }`. The logical name depends on the compiled CloudFormation template. Run [`stacktape compile-template`](/cli/compile-template) to find the exact logical name for each resource in your generated template before referencing them.

### How do I pass the state machine ARN to a Lambda function?

Use [`connectTo`](/configuration/connecting-resources) to grant IAM permissions, and add an `environment` entry with the [`$ResourceParam` directive](/configuration/directives) to inject the ARN. For example, `{ name: 'MY_WORKFLOW_ARN', value: "$ResourceParam('myWorkflow', 'arn')" }`. The state machine exposes `arn` and `name` as [referenceable parameters](/configuration/referenceable-parameters).

### Can I customize the underlying CloudFormation resources?

Yes. The `StateMachine` resource supports [overrides](/configuration/overrides-and-escape-hatches) for modifying the generated CloudFormation resources. The `StateMachineProps` type exposes only the ASL `definition`, so CloudFormation-level settings must be applied through `overrides`. Run [`stacktape compile-template`](/cli/compile-template) to find logical names.

### What permissions does the state machine execution role have?

Stacktape automatically creates an execution role for state machines in your stack. Run [`stacktape compile-template`](/cli/compile-template) to inspect the exact IAM policy attached to the role. Use [overrides](/configuration/overrides-and-escape-hatches) if you need to restrict or extend the permissions granted.

### How much do AWS Step Functions cost?

AWS Step Functions Standard workflows cost ~$0.025 per 1,000 state transitions. The AWS Free Tier includes 4,000 transitions per month. Each state in your workflow counts as one transition, so a 5-state workflow costs 5 transitions per execution. Express workflows offer higher throughput at lower per-transition cost but are limited to 5-minute maximum duration. See the [AWS Step Functions pricing page](https://aws.amazon.com/step-functions/pricing/) for current rates.

### What is Amazon States Language?

Amazon States Language (ASL) is the JSON-based specification used to define Step Functions workflows. It describes states, transitions, error handling, input/output processing, and parallel execution. Stacktape accepts the ASL definition as the `definition` property of the `StateMachine` resource. The Stacktape type system models eight state types: Choice, Fail, Map, Parallel, Pass, Succeed, Task, and Wait. The [full specification](https://states-language.net/spec.html) documents all state types and features.

### How long can a Step Functions execution run?

AWS Standard Step Functions workflows can run for up to one year. Set the `TimeoutSeconds` property in your state machine definition to impose a shorter limit. Individual Task states also support `TimeoutSeconds` to limit how long a single step can take. Use `HeartbeatSeconds` on Task states to require periodic signals from long-running tasks.

### Can Step Functions run tasks in parallel?

Yes. Use the **Parallel** state to run multiple branches concurrently — all branches must complete before the workflow continues. Use the **Map** state to iterate over an array and run a sub-workflow for each item, with `MaxConcurrency` controlling how many items process simultaneously. Both state types support `Retry` and `Catch` for error handling.

### When should I use a state machine vs an SQS queue?

Use a state machine when you need branching logic, ordered multi-step processing, built-in retries with backoff, or workflow visibility (execution history, status tracking). Use an [SQS queue](/resources/messaging/sqs-queue) for simple decoupled message processing where a single Lambda consumer handles each message independently. SQS is cheaper at high throughput and simpler to set up for one-step async processing.

### When should I use a state machine vs direct Lambda-to-Lambda invocation?

Direct invocation (calling one Lambda from another via the AWS SDK) works for simple two-function chains. Use a state machine when you need error handling with retries and fallbacks, conditional branching, parallel execution, or execution history. State machines also avoid the cascading timeout problem — if function A calls B calls C, A's timeout must account for the entire chain. A state machine manages each step's timeout independently.
