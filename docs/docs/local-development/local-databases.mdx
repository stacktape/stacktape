# Local Databases

Stacktape [dev mode](/local-development/dev-mode-overview) runs databases locally in Docker containers so you can develop without deploying or paying for AWS resources. Data persists between sessions, connection strings are injected as environment variables automatically, and the experience mirrors the deployed stack — with no manual Docker setup required.

## When to use local databases

Local databases are the default and recommended path for day-to-day development. They start in seconds, have zero network latency, cost nothing, and isolate each developer's data. Use local databases when you are iterating on application logic, testing schema migrations, or prototyping new features.

## When NOT to use local databases

Switch to [remote (deployed) databases](#using-remote-databases) when your work depends on behavior that local emulation cannot replicate — Aurora multi-AZ failover, DynamoDB Streams, OpenSearch security plugins, or engine-specific query plan differences. Also prefer remote databases when you need to test against a realistic data volume or share state across a team. Set `dev.remote: true` on the resource or pass the `--remoteResources` CLI flag to connect to the deployed AWS resource instead.

## Supported databases

Stacktape can locally emulate six database and data-store types. Each runs as a single Docker container with data persisted to your project directory.

| Resource type | Docker image | Default port | Local credentials | Readiness timeout |
|---|---|---|---|---|
| PostgreSQL | `postgres:<version>` | 5432 | `postgres` / `localdevpassword` | 30 s |
| MySQL | `mysql:<version>` | 3306 | `root` / `localdevpassword` | 60 s |
| MariaDB | `mariadb:<version>` | 3306 | `root` / `localdevpassword` | 60 s |
| Redis | `redis:<version>` | 6379 | `default` / configured password | 30 s |
| DynamoDB | `amazon/dynamodb-local` | 8000 | None (local mock auth) | 30 s |
| OpenSearch | `opensearchproject/opensearch:<version>` | 9200 | None (security disabled) | 60 s |

The local credentials shown above are the defaults when no credentials are configured. For [relational databases](/resources/databases/relational-database), the local username comes from `credentials.masterUserName` (defaults to `postgres` for PostgreSQL engines, `root` for MySQL/MariaDB), and the password comes from `credentials.masterUserPassword` (falls back to `localdevpassword` locally). Properties like `instanceSize` only apply to the deployed AWS resource and are ignored locally.

Aurora engine types — Aurora PostgreSQL, Aurora MySQL, Aurora Serverless v1, and Aurora Serverless v2 — are mapped to standard PostgreSQL or MySQL containers locally. Stacktape shows a warning at startup that Aurora-specific features (multi-AZ replication, serverless scaling, reader endpoints) are not available in the local emulation. The SQL interface and connection string format match the deployed behavior, which is sufficient for application-level development.

<Info>
Docker Desktop (or any Docker-compatible runtime) must be running before starting dev mode. If Docker is not available, Stacktape shows a clear error message with instructions to install or start Docker.
</Info>

## How it works

When you run [`stacktape dev`](/cli/dev), Stacktape identifies database resources in your config and starts Docker containers for each one.

1. **Container naming** — each container is named `stp-<stage>-<resourceName>`, so multiple stages can run in parallel without conflicts.
2. **Port allocation** — Stacktape applies a stage-based offset (derived from a hash of your stage name) to the default port, then searches sequentially for an available port. This prevents collisions when multiple developers or stages share a machine.
3. **Data persistence** — data is stored at `.stacktape/dev-data/<stage>/<resourceName>/data/` inside your project directory. Containers are reused across dev sessions, and data survives restarts.
4. **Readiness checks** — Stacktape polls the container until it responds to health checks. Timeouts range from 30 seconds (PostgreSQL, Redis, DynamoDB) to 60 seconds (MySQL, MariaDB, OpenSearch) depending on the database type.
5. **Environment variable injection** — referenceable parameters (host, port, connectionString, etc.) are injected into your workloads as `STP_<RESOURCE_NAME>_<PARAM>` environment variables, matching the deployed-stack behavior. See [Connecting resources](/configuration/connecting-resources) for the full pattern.
6. **Health monitoring** — a background monitor checks containers every 30 seconds. After 2 consecutive failures, the container is marked unhealthy and auto-restarted (up to 3 attempts). If all restart attempts fail, the resource enters a "failed" state requiring manual intervention.

## Relational databases

### PostgreSQL

Stacktape runs the official `postgres` Docker image. The version is taken from your engine configuration (or `latest` if not specified). The container is initialized with `POSTGRES_USER`, `POSTGRES_PASSWORD`, and `POSTGRES_DB` environment variables matching your config.

Readiness is checked using `pg_isready`, polling every 500 ms with a 30-second timeout.

In dev mode, this starts a `postgres:16.6` container. The `instanceSize` and `masterUserPassword` (when using `$Secret()`) are only relevant for the deployed AWS resource — locally, the password falls back to `localdevpassword`.

The injected parameters for PostgreSQL include `host`, `port`, `dbName`, `connectionString`, and `jdbcConnectionString`.

### MySQL and MariaDB

MySQL uses the `mysql` Docker image; MariaDB uses the `mariadb` image. Readiness is checked using `mysqladmin ping`, polling every 1 second with a 60-second timeout.

If the local data directory contains data from an incompatible MySQL version (for example, you changed `version` from `8.0` to `8.4`), Stacktape detects the incompatibility (via InnoDB or redo log errors) and automatically reinitializes the data directory with a warning.

### Aurora engines

All Aurora engine types — Aurora PostgreSQL, Aurora MySQL, Aurora Serverless v1, Aurora Serverless v2 — are mapped to standard PostgreSQL or MySQL containers locally. The mapping is:

| Aurora engine type | Local container |
|---|---|
| `aurora-postgresql`, `aurora-postgresql-serverless`, `aurora-postgresql-serverless-v2` | PostgreSQL |
| `aurora-mysql`, `aurora-mysql-serverless`, `aurora-mysql-serverless-v2` | MySQL |

This is a pragmatic tradeoff: the SQL interface, schema behavior, and connection string format match the deployed database, which is sufficient for application development. Aurora-specific behavior — ACU scaling, multi-AZ replication, reader endpoints, pause-after-inactivity — is only available on the deployed AWS resource. Test these features against a deployed stage using `dev.remote: true`.

### Oracle and SQL Server

RDS Oracle (`oracle-ee`, `oracle-se2`) and SQL Server (`sqlserver-ee`, `sqlserver-se`, `sqlserver-ex`, `sqlserver-web`) engines do not have local emulation support. These engines are not mapped to a local container type. If your config includes an Oracle or SQL Server database, use `dev.remote: true` to connect to the deployed AWS resource during development.

## Redis

Stacktape runs the official `redis` Docker image. The version defaults to `7.2` if not specified in your engine configuration. If `defaultUserPassword` is configured, the container starts with the `--requirepass` flag. Redis runs as a single node locally — cluster-mode features (hash slots, multi-node failover) are not available.

Readiness is checked using `redis-cli ping`, polling every 500 ms with a 30-second timeout.

## DynamoDB

Stacktape uses the `amazon/dynamodb-local` image with the `-sharedDb` flag, meaning all tables share a single underlying database file. Tables are automatically created from your config if a `primaryKey` is defined — you do not need to create them manually.

Readiness is checked by making an HTTP ListTables request, polling every 300 ms with a 30-second timeout.

Your application code should use the `endpoint` parameter to configure the AWS SDK DynamoDB client for local access.

<Warning>
DynamoDB Local supports basic CRUD operations, queries, and scans, but does not emulate DynamoDB Streams, Global Tables, TTL, or on-demand backups. Use `dev.remote: true` if your application depends on these features.
</Warning>

## OpenSearch

Stacktape runs the `opensearchproject/opensearch` image as a single-node cluster with the security plugin disabled for simpler local access. The JVM heap is limited to 512 MB (`-Xms512m -Xmx512m`) to reduce resource usage on your development machine.

The version defaults to `2.17.0` when your engine version is set to `latest`. Readiness is checked by making an HTTP GET request to the root endpoint, polling every 1 second with a 60-second timeout.

The injected parameters include `domainEndpoint` (e.g. `http://localhost:<port>`) and `arn` (a synthetic local ARN).

## Using remote databases

By default, all emulatable databases run locally. To connect to the real deployed AWS resource instead, set `dev.remote: true` on the resource in your config:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, RelationalDatabase, RdsEnginePostgresql } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const mainDb = new RelationalDatabase({
    credentials: {
      masterUserPassword: '$Secret(\'db.password\')'
    },
    engine: new RdsEnginePostgresql({
      version: '16.6',
      primaryInstance: {
        instanceSize: 'db.t4g.micro'
      }
    }),
    dev: {
      remote: true
    }
  });

  return {
    resources: { mainDb }
  };
});
// [!code focus-end]`
    }
  ]}
/>

You can also override at runtime using the `--remoteResources` CLI flag without changing your config. Pass one or more resource names:

```bash
stacktape dev --remoteResources mainDb
```

To specify multiple resources, separate them with commas:

```bash
stacktape dev --remoteResources mainDb,cache
```

### When to use remote databases

- **Testing against real data** — when you need to reproduce a production issue with actual records.
- **Aurora-specific behavior** — to test failover, reader endpoints, or serverless ACU scaling.
- **DynamoDB advanced features** — Streams, Global Tables, TTL, and backups are only available on the deployed resource.
- **Shared development database** — when multiple developers need to see the same state.

For most day-to-day development, local databases are faster (no network latency), free (no AWS charges), and isolated (no risk of corrupting shared data).

## Resetting local data

Data persists in `.stacktape/dev-data/<stage>/<resourceName>/data/` across dev sessions. To start fresh, use the `--freshDb` flag:

```bash
stacktape dev --freshDb
```

This deletes the data directory for all local databases before starting containers. The flag applies to every local resource in the session — individual databases cannot be reset selectively with it.

To reset a specific resource manually, delete its data directory:

```bash
rm -rf .stacktape/dev-data/<stage>/<resourceName>/data
```

<Tip>
Add `.stacktape/` to your `.gitignore` to avoid committing local database data to version control.
</Tip>

## Port allocation

Stacktape avoids port conflicts using a two-layer strategy:

1. **Stage-based offset** — a hash of your stage name produces a deterministic offset (0–4,900 in steps of 100). Two developers running different stages on the same machine get non-overlapping port ranges by default.
2. **Sequential search** — starting from the offset port, Stacktape checks both system port usage and an internal reservation set (to prevent race conditions when multiple resources start in parallel), then picks the first free port.

If the assigned port differs from the default, the actual port is displayed in the dev mode TUI and reflected in the injected connection string. Your application code does not need to handle port differences — the injected environment variables always contain the correct value.

## Databases without local emulation

Not all Stacktape database resources support local emulation:

| Resource | Local support | Alternative |
|---|---|---|
| RDS Oracle / SQL Server | No | Set `dev.remote: true` to use the deployed RDS instance |
| [MongoDB Atlas](/resources/databases/mongodb-atlas) | No | Use a local MongoDB container manually, or set `dev.remote: true` |
| [Upstash Redis](/resources/databases/upstash-redis) | No | Connect to the deployed Upstash instance (third-party managed) |

Oracle and SQL Server engines are not mapped to a local container type in Stacktape's engine mapping. MongoDB Atlas and Upstash Redis are third-party managed services without official local emulators in the dev mode resource system.

## FAQ

### Does local database data persist when I restart dev mode?

Yes. Data is stored on disk at `.stacktape/dev-data/<stage>/<resourceName>/data/` and survives both dev mode restarts and machine reboots. If a container with the same name is already running, Stacktape reuses it rather than creating a new one. Use the `--freshDb` flag to explicitly wipe all local data before starting.

### Can I run multiple stages locally at the same time?

Yes. Each stage gets its own container name (`stp-<stage>-<resourceName>`) and a stage-based port offset derived from a hash of the stage name, so two developers or two stages on the same machine do not collide. You can run `stacktape dev --stage dev-alice` and `stacktape dev --stage dev-bob` in parallel.

### How do I connect to the local database with a GUI tool?

Use the host `localhost` and the port shown in the dev mode TUI output. For PostgreSQL, the default credentials are `postgres` / `localdevpassword` (or your configured values). For MySQL/MariaDB, the defaults are `root` / `localdevpassword`. For Redis, use the connection string shown in the TUI. DynamoDB Local and OpenSearch use HTTP endpoints with no authentication required.

### Does local DynamoDB support Streams and Global Tables?

No. DynamoDB Local supports basic CRUD, queries, and scans, but does not emulate Streams, Global Tables, Time to Live (TTL), or on-demand backups. If your application depends on these features, set `dev.remote: true` on the [DynamoDB table](/resources/databases/dynamodb) resource to connect to the deployed AWS resource.

### How does Stacktape handle Aurora Serverless locally?

Aurora Serverless v1 and v2 engine types are mapped to standard PostgreSQL or MySQL containers locally. The serverless scaling (ACU-based), pause-after-inactivity, and automatic failover behaviors are AWS-only features. The SQL interface and connection string format are the same, so application code works unchanged. A warning is shown at startup explaining which Aurora features are unavailable.

### What happens if Docker is not running?

Stacktape detects Docker availability at startup. If Docker is not installed or the Docker daemon is not running, you get a clear error message directing you to install or start Docker Desktop. Dev mode cannot start local databases without a Docker-compatible runtime.

### Can I use the local database for running migrations?

Yes. The connection string is injected into your workloads the same way as in a deployed stack. Run migrations from your application code, a [deployment script](/resources/advanced/deployment-scripts), or manually via a database client connected to `localhost:<port>`. Schema changes persist across dev sessions because the data directory is preserved.

### What happens when I change the database version in my config?

For PostgreSQL, Redis, DynamoDB, and OpenSearch, the new version is pulled and the container is recreated. Existing data may or may not be compatible with the new version depending on the engine. For MySQL and MariaDB, Stacktape detects incompatible data (such as InnoDB redo log format mismatches) and automatically reinitializes the data directory with a warning — your previous local data is deleted in this case.

### Can I use Oracle or SQL Server databases locally?

No. RDS Oracle and SQL Server engine types do not have local emulation in Stacktape dev mode. Use `dev.remote: true` on the resource to connect to the deployed RDS instance during development. Alternatively, you can run your own Oracle or SQL Server container manually and configure your application to connect to it.

### How much memory does the local OpenSearch container use?

The OpenSearch container's JVM heap is limited to 512 MB (`-Xms512m -Xmx512m`) to reduce resource usage on development machines. The security plugin is disabled for simpler local access. If you need more capacity or security features, use `dev.remote: true` to connect to the deployed [OpenSearch domain](/resources/databases/opensearch).
