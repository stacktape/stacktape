# Agent Mode in Dev

<Warning title="Preview — in active development">
Agent mode is usable today but still evolving. Verify available endpoints by calling `GET /` on the running agent server. Some details on this page may change between Stacktape releases.
</Warning>

Stacktape [dev mode](/local-development/dev-mode-overview) can start an HTTP API server alongside your local workloads and databases. This **agent mode** gives AI coding assistants — and any HTTP client — programmatic access to check workload status, trigger rebuilds, read log file paths, query local databases, and call AWS SDK commands, enabling automated edit-test-fix loops without manual intervention.

## When to use

Agent mode is designed for workflows where an AI coding assistant drives the edit-test-fix loop. Enable it when:

- **AI-driven development** — You're using Claude Code, Cursor, Windsurf, or another AI coding assistant to build your application. The assistant starts `stacktape dev --agent` as a shell command, then uses the HTTP API to check status, rebuild workloads, read the log file path, query local databases, call AWS SDK commands, and stop dev mode.
- **Automated test harnesses** — You want a script to verify application behavior during development by hitting the status, env, or database endpoints.
- **Database inspection without client tools** — You need to run SQL queries, check Redis keys, scan DynamoDB items, or search OpenSearch indices against local dev databases without installing separate database clients.

## When NOT to use

- **Manual development** — If you're editing code yourself and checking results in a browser, standard [dev mode](/local-development/dev-mode-overview) without `--agent` is simpler. The TUI output is optimized for human reading.
- **Production observability** — Agent mode runs locally. For deployed stacks, use [`stacktape logs`](/cli/logs), [`stacktape metrics`](/cli/metrics), or the [Stacktape Console](/stacktape-console/console-overview).
- **CI/CD pipelines** — Use [`stacktape deploy`](/cli/deploy) directly in your pipeline. Agent mode is a development-time feature.

## Starting agent mode

Add the `--agent` flag to the [`stacktape dev`](/cli/dev) command to start the HTTP API server alongside your workloads.

```bash
stacktape dev --agent --agentPort 9900 --stage dev --region eu-west-1
```

| Flag            | Description                                                                              |
| --------------- | ---------------------------------------------------------------------------------------- |
| `--agent`       | Enables the agent HTTP API server alongside dev mode                                     |
| `--agentPort`   | Port for the agent HTTP server                                                           |
| `--stage`       | Stage used for dev mode and the `.stacktape/dev-data/{stage}/` directory                 |
| `--region`      | AWS region used by dev mode and as the default for `/aws/sdk` when `region` is omitted   |
| `--resources`   | Which workloads to run (`all` or comma-separated names)                                  |
| `--freshDb`     | Reset local database data on startup                                                     |

Once ready, Stacktape prints the agent URL and a summary of available endpoints. Local database data persists in `.stacktape/dev-data/{stage}/` between runs — use `--freshDb` to wipe it for a clean slate.

<Tip>

AI coding assistants can call `GET /` on the agent server to receive a full endpoint listing dynamically. Use `GET /?compact=true` for a shorter LLM-oriented response — it includes the current phase, workload names, available local resources with their primary query/schema endpoints, core action templates (rebuild, logs, env, aws, stop), and a pointer to `GET /` for full docs.

</Tip>

## Response format

Every agent endpoint wraps its result in a consistent JSON envelope so clients can handle success and error uniformly.

```json
{
  "v": "1.0",
  "ok": true,
  "code": "OK",
  "message": "OK",
  "data": {}
}
```

| Field     | Description                                                                          |
| --------- | ------------------------------------------------------------------------------------ |
| `v`       | Envelope version (currently `"1.0"`)                                                 |
| `ok`      | `true` on success, `false` on error                                                  |
| `code`    | Machine-readable status. Common values include `OK`, `NOT_FOUND`, `VALIDATION_ERROR`, `INTERNAL_ERROR`, `ALREADY_EXISTS`, `AUTH_ERROR`, and `READ_ONLY_VIOLATION` |
| `message` | Human-readable description (optional — most responses include it, but not all)       |
| `data`    | Endpoint-specific payload (optional — present when the endpoint returns structured data) |

Many not-found and validation errors include contextual hints in `data`, such as an `available` array of valid resource names or a `hint` string describing expected input. For example, referencing a non-existent workload returns the list of available workloads in `data.available` and a usage hint in `data.hint`. This makes it easy for AI agents to self-correct without extra calls.

Agent endpoints return a JSON envelope with `v`, `ok`, and `code` on every response; most successful responses also include `message` and endpoint-specific `data`. Response examples throughout this page may omit envelope fields like `v` for brevity.

## Core endpoints

### Status

The `/status` endpoint reports the current agent phase and a map of workload names to their local URLs (if running) or status strings. AI agents should poll this endpoint after starting dev mode and wait for `"ready"` before testing workload URLs.

```bash
curl http://localhost:9900/status
```

```json
{
  "v": "1.0",
  "ok": true,
  "code": "OK",
  "data": {
    "phase": "ready",
    "ready": true,
    "workloads": {
      "api": "http://localhost:3000",
      "worker": "running"
    },
    "pid": 12345
  }
}
```

The `phase` value is one of `starting`, `ready`, `rebuilding`, `stopping`, or `stopped`. In the current server code, `/status` moves phase from `starting` to `ready` once workloads are running or errored, and `/stop` moves phase to `stopping`. The `ready` field becomes `true` when there is at least one workload and every workload has reached `running` or `error` status.

For detailed information including uptime, per-workload type/port/error data, local resource status, and the log file path, use verbose mode.

```bash
curl "http://localhost:9900/status?verbose=true"
```

The verbose response replaces the workload string map with an array of objects containing `name`, `type`, `status`, `url`, `port`, and `error` fields, plus a `localResources` array, `uptime` in seconds, and the `logFile` path. Workload `status` is one of `pending`, `starting`, `running`, `error`, or `stopped`. Local resource status uses the same values.

### Logs

The `/logs` endpoint returns the path to the JSONL log file on disk. Logs are **not** streamed through the HTTP response — AI agents should read the file directly using standard tools like `tail`, `grep`, or `jq`.

```bash
curl http://localhost:9900/logs
```

```json
{
  "v": "1.0",
  "ok": true,
  "code": "OK",
  "data": {
    "logFile": ".stacktape/dev-data/dev/9900-logs.txt",
    "format": "jsonl",
    "hint": "Use standard tools: tail -n 50 <logFile> | jq ."
  }
}
```

Each line in the log file is a JSON object with these fields:

| Field    | Example                      | Description                           |
| -------- | ---------------------------- | ------------------------------------- |
| `ts`     | `"2025-03-15T14:30:15Z"`    | ISO timestamp                         |
| `source` | `"api"`                      | Workload name that produced the log   |
| `level`  | `"info"`, `"warn"`, `"error"` | Severity level                      |
| `msg`    | `"Server started on :3000"`  | Log message content                   |

View the last 50 lines:

```bash
tail -n 50 .stacktape/dev-data/dev/9900-logs.txt
```

Filter for errors only:

```bash
grep '"level":"error"' .stacktape/dev-data/dev/9900-logs.txt
```

Filter by a specific workload:

```bash
grep '"source":"api"' .stacktape/dev-data/dev/9900-logs.txt
```

This file-based approach is intentional — AI coding assistants already have file-reading tools, and `tail -f` provides real-time streaming without polling overhead.

### Environment variables

The `/env/{workload}` endpoint returns the environment variables available to a specific workload. For Lambda functions, variables are fetched from the deployed AWS function configuration. For other workload types whose env vars are tracked by dev mode (currently container, hosting-bucket, and nextjs-web workloads), variables are read from the local runner state.

```bash
curl http://localhost:9900/env/api
```

```json
{
  "v": "1.0",
  "ok": true,
  "code": "OK",
  "data": {
    "workload": "api",
    "type": "function",
    "source": "aws",
    "envVars": {
      "STP_NOTES_TABLE_TABLE_NAME": "my-project-dev-notesTable",
      "AWS_REGION": "eu-west-1"
    },
    "count": 2
  }
}
```

This endpoint is useful for debugging [`connectTo`](/configuration/connecting-resources) wiring. If an expected `STP_*` variable is missing, the workload's `connectTo` array likely doesn't reference the target resource. The response also includes the `source` field — `"aws"` for Lambda functions (fetched from the deployed function configuration) or `"local"` for other tracked workload types (currently container, hosting-bucket, and nextjs-web workloads, read from the local runner state).

### Rebuild

After editing application code, trigger a workload rebuild to pick up the changes. Rebuilding a specific workload is faster than rebuilding all of them — AI agents should target only the workload whose code changed.

Rebuild a specific workload:

```bash
curl -X POST http://localhost:9900/rebuild/api
```

Rebuild all workloads at once:

```bash
curl -X POST http://localhost:9900/rebuild/all
```

If a workload is currently in the `starting` state (mid-rebuild), the server rejects the request with an error message. The rebuild-all endpoint also rejects if any workload is mid-rebuild, listing the names of the workloads that are still starting.

### Liveness check

`GET /health` returns the current agent phase and can be used as a simple liveness check. Unlike `/status`, `/health` does not include workloads or local resources — it only returns `ok` and the current agent `phase` inside the standard envelope. Use `/status` for full workload details.

```bash
curl http://localhost:9900/health
```

### Resource health checks

Check the health of a specific local resource (database, cache). The agent monitors local resources automatically, but this endpoint triggers an immediate check and returns the result.

```bash
curl http://localhost:9900/health/myDb
```

The response includes `name`, `type`, `port`, `host`, and the `status` value returned by the local health monitor. If the resource name doesn't match any running local resource, the error response includes an `available` array listing valid resource names.

### Stop

Gracefully shut down dev mode, all workloads, and all local resources.

```bash
curl -X POST http://localhost:9900/stop
```

Calling stop when the agent is already stopping or stopped returns success without side effects.

## Database endpoints

Agent mode exposes HTTP endpoints for local resource instances started by dev mode. The current server has typed handlers for PostgreSQL, MySQL/MariaDB, Redis, DynamoDB, and OpenSearch resources, so AI agents can inspect schemas, run queries, and verify data without installing separate database client tools. `GET /` lists only the database endpoint groups for local resources currently running in this dev session. If a resource is not running or has a different local type, the endpoint returns an error with available resource names where applicable.

The `{name}` path parameter in every database endpoint matches the resource name in your Stacktape configuration. If you reference a name that doesn't exist, the error response includes an `available` array listing the valid resource names.

<Warning>

The PostgreSQL, MySQL, and Redis command/query endpoints accept an optional `confirm` field. The startup documentation warns that destructive operations require `{"confirm": true}` in the request body — without it, the request is rejected. The DynamoDB and OpenSearch delete handlers do not accept a `confirm` parameter; they require the key or document identifier fields shown below.

</Warning>

### PostgreSQL

Available when dev mode is running a local instance for a [relational database](/resources/databases/relational-database) with a PostgreSQL engine in your Stacktape configuration.

```bash
curl -X POST http://localhost:9900/postgres/myDb/query \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT * FROM users LIMIT 10"}'
```

| Endpoint                          | Method | Description                    | Parameters                            |
| --------------------------------- | ------ | ------------------------------ | ------------------------------------- |
| `/postgres/{name}/query`          | POST   | Execute SQL                    | `{sql, limit?, timeout?, confirm?}`   |
| `/postgres/{name}/schema`         | GET    | Column types and constraints   | `?tables=t1,t2` (optional filter)     |
| `/postgres/{name}/tables`         | GET    | List all tables                |                                       |
| `/postgres/{name}/explain`        | POST   | Query execution plan           | `{sql, analyze?}`                     |
| `/postgres/{name}/indexes`        | GET    | Index definitions              | `?table=name` (optional filter)       |
| `/postgres/{name}/stats`          | GET    | Table statistics               | `?table=name` (optional filter)       |
| `/postgres/{name}/sample`         | GET    | Sample rows from a table       | `?table=name&limit=5`                 |

```bash
curl -X POST http://localhost:9900/mysql/myDb/query \
```

| `/mysql/{name}/tables`          | GET    | List all tables                |                                                        |

### Redis

Available when dev mode is running a local instance for a [Redis cluster](/resources/databases/redis) in your Stacktape configuration. The `cmd` field accepts a Redis command as a single string (e.g. `"GET mykey"`, `"SET user:1 '{\"name\":\"Alice\"}'"`).

```bash
curl -X POST http://localhost:9900/redis/myCache/command \
  -H "Content-Type: application/json" \
  -d '{"cmd": "GET session:user123"}'
```

| Endpoint                       | Method | Description                  | Parameters                       |
| ------------------------------ | ------ | ---------------------------- | -------------------------------- |
| `/redis/{name}/command`        | POST   | Execute any Redis command    | `{cmd, confirm?}`               |
| `/redis/{name}/keys`           | GET    | List keys matching pattern   | `?pattern=*&limit=100`          |
| `/redis/{name}/get`            | GET    | Get value by key             | `?key=name`                     |
| `/redis/{name}/ttl`            | GET    | Get TTL for a key            | `?key=name`                     |
| `/redis/{name}/info`           | GET    | Server info                  | `?section=memory` (optional)    |
| `/redis/{name}/type`           | GET    | Get type of a key            | `?key=name`                     |

### DynamoDB

Available when dev mode is running a local instance for a [DynamoDB table](/resources/databases/dynamodb) in your Stacktape configuration. Key parameters (`pk`, `sk`) are objects mapping attribute names to values — for example, `{"userId": "123"}`.

```bash
curl -X POST http://localhost:9900/dynamodb/notesTable/scan \
  -H "Content-Type: application/json" \
  -d '{"limit": 10}'
```

| Endpoint                           | Method | Description                  | Parameters                                    |
| ---------------------------------- | ------ | ---------------------------- | --------------------------------------------- |
| `/dynamodb/{name}/query`           | POST   | Query by partition/sort key  | `{pk: {key: val}, sk?, index?, limit?}`       |
| `/dynamodb/{name}/scan`            | POST   | Full table scan              | `{filter?, limit?}`                           |
| `/dynamodb/{name}/get`             | POST   | Get single item              | `{pk: {key: val}, sk?}`                       |
| `/dynamodb/{name}/put`             | POST   | Put an item                  | `{item: {...}}`                               |
| `/dynamodb/{name}/delete`          | POST   | Delete an item               | `{pk: {key: val}, sk?}`                       |
| `/dynamodb/{name}/schema`          | GET    | Table key schema and indexes |                                                |
| `/dynamodb/{name}/sample`          | GET    | Sample items                 | `?limit=5`                                    |

### OpenSearch


## AWS SDK access

Agent mode provides direct access to AWS SDK commands through the `/aws/sdk` endpoint. The endpoint uses credentials returned by `getDevAgentCredentials()` — dev agent role credentials scoped to the stack, with a fallback to user credentials. If no usable access key and secret key are returned, the request fails with an error. The startup message describes this access as scoped to stack resources via IAM role.

```bash
curl -X POST http://localhost:9900/aws/sdk \
  -H "Content-Type: application/json" \
  -d '{"service": "lambda", "command": "ListFunctions", "input": {"MaxItems": 10}}'
```

The request body accepts `service` (required), `command` (required), `input` (optional, defaults to `{}`), and `region` (optional, defaults to the dev mode region). If `service` or `command` is missing, the agent returns a validation error with a hint showing the expected request shape.

To list all supported services and example commands:

```bash
curl http://localhost:9900/aws/sdk/services
```

`GET /aws/sdk/services` returns a successful envelope whose payload includes the supported services from `getSupportedServices()`. In the current implementation, the services payload is nested under `data.data` in the response. Use the running endpoint response as the authoritative shape and list of valid `service` values before constructing `/aws/sdk` calls. The service list in the startup message and on this page is a summary; the endpoint response is always up to date.

Common AWS SDK calls for AI agents:

| Use case                  | Service            | Command              | Example input                                                  |
| ------------------------- | ------------------ | -------------------- | -------------------------------------------------------------- |
| Read Lambda logs          | `logs`             | `FilterLogEvents`    | `{"logGroupName": "/aws/lambda/fn", "filterPattern": "ERROR"}` |
| Scan a DynamoDB table     | `dynamodb`         | `Scan`               | `{"TableName": "tbl", "Limit": 10}`                           |
| List S3 objects           | `s3`               | `ListObjectsV2`      | `{"Bucket": "bkt", "Prefix": "uploads/"}`                     |
| Invoke a Lambda           | `lambda`           | `Invoke`             | `{"FunctionName": "fn", "Payload": "{}"}`                     |
| Describe a stack          | `cloudformation`   | `DescribeStacks`     | `{"StackName": "my-stack"}`                                    |

<Info>

The startup message lists supported `service` identifiers: `lambda`, `dynamodb`, `s3`, `logs`, `cloudformation`, `cloudwatch`, `sqs`, `sns`, `sfn`, `eventbridge`, `secretsmanager`, `ssm`, `sts`, `iam`, `ec2`, `ecs`, `ecr`, `rds`, `ses`, `cognito`, `apigatewayv2`, `xray`, `kinesis`, `firehose`, and more. Call `GET /aws/sdk/services` on the running agent for the complete, up-to-date list.

### Example session

An AI coding assistant building a notes API would follow this sequence:

1. Write `stacktape.ts` with a Lambda function, DynamoDB table, and HTTP API Gateway trigger
2. Start agent mode: `stacktape dev --agent --agentPort 9900 --stage dev --region eu-west-1`
3. Poll `GET /status` until `phase` is `"ready"` and `ready` is `true`
4. Write the handler code in `src/handler.ts`
5. Call `POST /rebuild/api` to pick up the new code
6. Test the endpoint by calling the workload URL from the status response
7. If errors occur, get the log file path from `GET /logs`, then read it: `tail -n 50 <logFile>`
8. Check `GET /env/api` to verify that `STP_NOTES_TABLE_TABLE_NAME` is injected
9. Query the database directly via `POST /dynamodb/notesTable/scan` to verify stored data
10. Fix code, rebuild, and test again until everything works
11. Deploy to AWS with `stacktape deploy`

### Tips for AI agent integrations

- **Poll status before testing** — Wait for `"phase": "ready"` before making requests to workload URLs. Workloads may still be compiling during the `starting` phase.
- **Use compact docs** — Call `GET /?compact=true` at the start of a session to discover available endpoints with minimal token usage. The full `GET /` listing is more detailed but token-heavy.
- **Rebuild selectively** — `POST /rebuild/{name}` rebuilds only the named workload. Use it instead of `/rebuild/all` when you changed code in a single workload.
- **Check env vars early** — `GET /env/{workload}` reveals whether [`connectTo`](/configuration/connecting-resources) is wired correctly before you start debugging application code.
- **Read logs from the file** — The `/logs` endpoint returns a file path. Use `tail`, `grep`, or `jq` to filter the JSONL log file. Use `tail -f` for real-time streaming.
- **Inspect schemas before writing queries** — Call the `/schema` or `/tables` endpoint to discover table and column names before writing SQL or DynamoDB queries.
- **Use database endpoints for verification** — After writing data through your API, query the database directly to confirm it was stored correctly.

## MCP server integration

For MCP-based assistant setup, see the [Stacktape MCP server](/using-with-ai/mcp-server-setup) page. If your AI assistant supports the Model Context Protocol (Claude Code, Cursor, Windsurf), the MCP server provides a higher-level integration layer. Agent mode's HTTP API is the lower-level alternative that works with any tool capable of making HTTP requests.

## Security

Stacktape advertises the agent URL as `http://localhost:{port}`. The server sets `Access-Control-Allow-Origin: null` and returns `403` for OPTIONS preflight requests. The implementation comments describe this as blocking browser-based cross-origin requests while keeping curl, scripts, and local tools usable.

The `/aws/sdk` endpoint uses credentials returned by `getDevAgentCredentials()`. If no access key and secret key are available, the endpoint returns an error (`AWS credentials not available`). The startup message describes this access as scoped to stack resources via IAM role. Treat the endpoint as having whatever permissions those credentials grant — avoid exposing the agent port to untrusted networks.

## FAQ

### Should I use agent mode or the Stacktape MCP server?

Both work with assistants like Claude Code, Cursor, and Windsurf. Agent mode is a plain HTTP API that any tool can call with `curl` or HTTP libraries — no protocol integration needed, so it suits custom tooling or any client that can make HTTP requests. The [MCP server](/using-with-ai/mcp-server-setup) uses the Model Context Protocol standard that those assistants support natively, providing a more integrated, higher-level layer. If your AI tool supports MCP, prefer the MCP server; reach for agent mode when you need raw HTTP access.

### What databases can I query through the agent API?

Agent mode exposes typed HTTP endpoints for five local resource types started by dev mode — PostgreSQL, MySQL/MariaDB, Redis, DynamoDB, and OpenSearch — so agents can inspect schemas, run queries, and verify data without installing separate database client tools. Each type has its own endpoint group (SQL engines get `query`/`schema`/`tables`/`explain`/`indexes`/`stats`/`sample`, plus key-value and document operations for Redis, DynamoDB, and OpenSearch). `GET /` lists only the endpoint groups for resources currently running in the session.

### Is my data persisted between agent mode sessions?

Yes. The startup message confirms local database data is stored in `.stacktape/dev-data/{stage}/` and survives restarts. Use the `--freshDb` flag when starting dev mode to reset all local databases to a clean state.

### Are destructive database operations safe during AI sessions?

The PostgreSQL, MySQL, and Redis query/command endpoints accept an optional `confirm` field. The startup documentation warns that destructive operations require `{"confirm": true}` — without it, the request is rejected. DynamoDB and OpenSearch delete handlers do not accept a `confirm` parameter; they require the key or document identifier fields directly.

### What happens if a workload fails to start?

The agent server keeps running even if individual workloads fail. The `/status` endpoint reports per-workload status — in verbose mode, each workload includes `status`, `url`, `port`, and `error` fields. AI agents can read the error, fix the code, and call `POST /rebuild/{name}` to retry without restarting the entire dev session.

### Why does an `/aws/sdk` call fail or return access-denied?

The `/aws/sdk` endpoint runs against dev agent credentials, which the source describes as scoped to the stack's IAM role (falling back to user credentials). Those credentials determine what resources are reachable — a call can fail with `AWS credentials not available` if no usable key pair is returned, or with an access-denied error if the role lacks permission for that resource. The endpoint supports a broad set of services (`lambda`, `dynamodb`, `s3`, `logs`, `cloudformation`, `sqs`, `sns`, `sfn`, and more); call `GET /aws/sdk/services` for the authoritative, up-to-date list of valid `service` values before constructing a call.
