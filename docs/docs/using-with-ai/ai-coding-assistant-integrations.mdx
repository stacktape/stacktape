# AI Coding Assistant Integrations

<Warning>
**Preview — in active development.** The Stacktape MCP server is under active development. Setup and behavior may change between Stacktape CLI versions. Expect improvements and possible breaking changes.
</Warning>

Stacktape integrates with AI coding assistants through the [Model Context Protocol (MCP)](https://modelcontextprotocol.io). The [`stacktape mcp:add`](/cli/mcp-add) command configures Claude Code, Cursor, VS Code with GitHub Copilot, Windsurf, OpenAI Codex, and OpenCode to launch Stacktape's local MCP server. The generated config tells the assistant to run `stacktape mcp` as a local MCP server process.

## When to use

Use the MCP integration when your AI coding assistant supports MCP and you want it to launch Stacktape's local MCP server directly from the editor. See the [MCP server setup](/using-with-ai/mcp-server-setup) page for the current tool list and server behavior. The integration is most valuable when you want a tighter feedback loop between asking questions in your assistant and taking action on your Stacktape project.

## When NOT to use

Skip MCP setup if you prefer running Stacktape commands in a terminal, or if your AI client does not support MCP. For operations not covered by the MCP server, use the [CLI](/cli/deploy) directly. See [MCP server setup](/using-with-ai/mcp-server-setup) for the current tool list.

## Supported assistants

The [`stacktape mcp:add`](/cli/mcp-add) command checks known config-file paths and writes the correct MCP configuration for the clients you select. Six AI coding assistants are supported:

| Assistant | Config format | Root key | Project config path | Global config path |
|-----------|--------------|----------|--------------------|--------------------|
| Claude Code | JSON | `mcpServers` | `.mcp.json` | `~/.claude.json` |
| Cursor | JSON | `mcpServers` | `.cursor/mcp.json` | `~/.cursor/mcp.json` |
| VS Code / GitHub Copilot | JSON | `servers` | `.vscode/mcp.json` | Platform-specific `User/mcp.json` |
| Windsurf | JSON | `mcpServers` | `.codeium/windsurf/mcp_config.json` | `~/.codeium/windsurf/mcp_config.json` |
| OpenAI Codex | TOML | `mcp_servers` | `.codex/config.toml` | `~/.codex/config.toml` |
| OpenCode | JSON/JSONC | `mcp` | `opencode.jsonc` or `.opencode/config.json` | Platform-specific (see below) |

VS Code global paths vary by platform: on Windows `%APPDATA%\Code\User\mcp.json`, on macOS `~/Library/Application Support/Code/User/mcp.json`, on Linux `~/.config/Code/User/mcp.json`.

Windsurf global path resolves to the user's home directory on all platforms: `~/.codeium/windsurf/mcp_config.json` (on Windows, `%USERPROFILE%\.codeium\windsurf\mcp_config.json`).

OpenCode global paths vary by platform: on Windows `%APPDATA%\opencode\opencode.jsonc` or `%APPDATA%\opencode\config.json`; on macOS and Linux `~/.config/opencode/opencode.jsonc`, `~/.config/opencode/config.json`, or `~/.opencode/config.json`.

Generated configs launch `stacktape mcp` locally. Claude Code, Cursor, and VS Code configs explicitly set `type: 'stdio'`; Windsurf omits the type field; OpenCode uses `type: 'local'` with the command as an array; Codex uses TOML with no type field. Other MCP-compatible clients may work if they can launch a local command with arguments and environment variables, but `mcp:add` only writes configs for the six listed clients.

## One-command setup

The [`stacktape mcp:add`](/cli/mcp-add) command configures Stacktape MCP for one or more supported AI coding assistants in a single invocation. When run interactively, it presents a prompt where you select which clients to configure, then writes the correct MCP server entry to each client's config file. In agent mode or with `--autoConfirmOperation`, the prompt is skipped and the default selection is used. Run it in your project directory:

```bash
stacktape mcp:add
```

When prompting is available, the interactive prompt lists the supported clients. Clients whose config file already exists on disk are labeled `(detected)` and pre-selected. Select the clients you want, confirm, and the command creates or updates the `stacktape` MCP server entry in each client's config file.

Before modifying any existing file, `mcp:add` creates a timestamped `.bak` backup (e.g. `.mcp.json.bak.2025-03-15T10-30-00-000Z`). If something goes wrong, restore from the backup.

### Non-interactive mode

To skip the interactive prompt (useful in CI or scripted setups), pass `--autoConfirmOperation`. The prompt is also skipped automatically in agent mode.

```bash
stacktape mcp:add --autoConfirmOperation
```

The default selection is all detected clients. If no existing config files are detected, the default selection is all six clients. With `--autoConfirmOperation`, that default selection is used without prompting.

### How detection works

Detection checks whether a config file already exists at any of the candidate paths for a given client — it does not check whether the client application is installed. A freshly installed client that has never been opened will not have a config file yet. Select it manually from the interactive prompt, or configure it by hand using the examples in [Manual setup](#manual-setup).

### Local vs global Stacktape command

When your project lists `stacktape` as a dependency in `package.json` (in `dependencies`, `devDependencies`, `optionalDependencies`, or `peerDependencies`), `mcp:add` writes `node_modules/.bin/stacktape` as the command for project-level config files. Global config files always use the `stacktape` command. This ensures the MCP server uses your project's pinned Stacktape version for local configs and the globally installed version for global configs.

### Project-local vs global config

The `mcp:add` command checks project-level config paths first, then global paths. It writes to the first existing file it finds for each client. If no config file exists for a client, it creates a new project-level config file at the first candidate path.

## What the MCP server provides

The [`stacktape mcp:add`](/cli/mcp-add) command writes config that tells the AI assistant to run `stacktape mcp` as a local stdio server. That server exposes documentation search, project inspection, validated CLI planning and execution, and dev-mode control. For the authoritative tool list, safety gates, and authentication behavior, see [MCP server setup](/using-with-ai/mcp-server-setup).

Once configured, useful first prompts include:

- “Use Stacktape tools to inspect this repository and identify its config file, stage, and region.”
- “Search the Stacktape docs for the right resource type for this workload and cite the relevant pages.”
- “Plan a Stacktape deployment for this project. Do not execute it.”
- “Start Stacktape dev mode, wait until it is ready, and show me errors from the API workload.”

## Manual setup

Stacktape MCP can be configured manually by adding a `stacktape` server entry to your AI coding assistant's MCP config file. Use manual setup when you prefer not to run [`stacktape mcp:add`](/cli/mcp-add), or when configuring a client that `mcp:add` does not support. The examples below show the config structure that `mcp:add` generates for each of the six supported clients.

<Info>
The `mcp:add` command includes a `STACKTAPE_MCP_CLIENT_NAME` environment variable in generated configs. If you configure a client manually, include the same variable to match the generated setup.
</Info>

<Tabs>
<Tab label="Claude Code">

Add to `.mcp.json` in your project root or `~/.claude.json` globally:

```json
{
  "mcpServers": {
    "stacktape": {
      "type": "stdio",
      "command": "stacktape",
      "args": ["mcp"],
      "env": {
        "STACKTAPE_MCP_CLIENT_NAME": "claude-code"
      }
    }
  }
}
```

</Tab>
<Tab label="Cursor">

Add to `.cursor/mcp.json` in your project root or `~/.cursor/mcp.json` globally:

```json
{
  "mcpServers": {
    "stacktape": {
      "type": "stdio",
      "command": "stacktape",
      "args": ["mcp"],
      "env": {
        "STACKTAPE_MCP_CLIENT_NAME": "cursor"
      }
    }
  }
}
```

</Tab>
<Tab label="VS Code / Copilot">

Add to `.vscode/mcp.json` in your project root or your VS Code user-level `mcp.json`. VS Code uses `servers` as the root key, not `mcpServers`:

```json
{
  "servers": {
    "stacktape": {
      "type": "stdio",
      "command": "stacktape",
      "args": ["mcp"],
      "env": {
        "STACKTAPE_MCP_CLIENT_NAME": "vscode"
      }
    }
  }
}
```

</Tab>
<Tab label="Windsurf">

Add to `.codeium/windsurf/mcp_config.json` in your project root or `~/.codeium/windsurf/mcp_config.json` globally. Windsurf configs omit the `type` field:

```json
{
  "mcpServers": {
    "stacktape": {
      "command": "stacktape",
      "args": ["mcp"],
      "env": {
        "STACKTAPE_MCP_CLIENT_NAME": "windsurf"
      }
    }
  }
}
```

</Tab>
<Tab label="OpenAI Codex">

Add to `.codex/config.toml` in your project root or `~/.codex/config.toml` globally:

```toml
[mcp_servers.stacktape]
command = "stacktape"
args = ["mcp"]
enabled = true

[mcp_servers.stacktape.env]
STACKTAPE_MCP_CLIENT_NAME = "codex"
```

</Tab>
<Tab label="OpenCode">

Add to `opencode.jsonc` or `.opencode/config.json` in your project root, or to the global config path for your platform. OpenCode uses `mcp` as the root key, `"type": "local"`, and passes the command as an array:

```json
{
  "mcp": {
    "stacktape": {
      "type": "local",
      "command": ["stacktape", "mcp"],
      "enabled": true,
      "env": {
        "STACKTAPE_MCP_CLIENT_NAME": "opencode"
      }
    }
  }
}
```

</Tab>
</Tabs>

<Tip>
If your project has `stacktape` as an npm dependency, replace `"stacktape"` with `"node_modules/.bin/stacktape"` in project-level configs to pin the MCP server to your project's Stacktape version.
</Tip>

## Verifying the integration

After configuring Stacktape MCP with [`stacktape mcp:add`](/cli/mcp-add) or manual setup, verify the integration is working by opening your AI assistant and asking a Stacktape-related question. If the client reads the config file, it should run `stacktape mcp` as the configured local MCP server. If the assistant does not appear to use Stacktape context, check the following:

1. **The `stacktape` command is on your PATH.** Run `stacktape --version` in a terminal to confirm.
2. **The `mcp` subcommand exists.** Older Stacktape CLI versions may not include it. Run [`stacktape upgrade`](/cli/upgrade) to get the latest version.
3. **The config file has correct syntax.** For JSON configs, validate the file parses without errors. For JSONC files (OpenCode), ensure comments use `//` syntax. The `mcp:add` command creates `.bak` backups before modifying files, so you can restore if needed.
4. **Your AI client has MCP enabled.** Some clients require toggling MCP support in their settings before they read MCP config files.
5. **The config file is in the right location.** Some clients only read project-level configs when the project is open as a workspace. If in doubt, use a global config path.

## Keeping the integration up to date

Generated configs run whichever `stacktape` binary the config points to. Updating that binary — via [`stacktape upgrade`](/cli/upgrade) or updating the `stacktape` npm package — means the next MCP server launch uses the updated version automatically.

Re-run [`stacktape mcp:add`](/cli/mcp-add) only if you want to change which clients are configured or switch between project-level and global configs. The command is idempotent — running it again against an already-configured client leaves the config unchanged (status: `unchanged`) and does not create a backup.

## FAQ

### What is MCP and why does Stacktape use it?

[MCP (Model Context Protocol)](https://modelcontextprotocol.io) is an open standard for connecting AI assistants to external tools. Stacktape's coding-assistant integration uses local stdio transport: the generated config tells the assistant to launch `stacktape mcp` as a child process and exchange MCP messages over stdin and stdout.

### Which AI coding assistants work with Stacktape?

Six clients have automatic setup via [`stacktape mcp:add`](/cli/mcp-add): Claude Code, Cursor, VS Code with GitHub Copilot, Windsurf, OpenAI Codex, and OpenCode. Other MCP-compatible clients may work if they can launch a local command with arguments and environment variables; `mcp:add` only writes configs for the six listed clients.

### Should I use project-local or global config?

Use project-level config (`.mcp.json`, `.cursor/mcp.json`, etc.) when you want the MCP server pinned to your project's Stacktape version via `node_modules/.bin/stacktape`. Use global config (`~/.claude.json`, `~/.cursor/mcp.json`, etc.) when you want Stacktape MCP available across all projects using your globally installed CLI. The `mcp:add` command writes to the first existing file it finds for each client and creates project-level files when no config exists.

### Does `mcp:add` modify my existing config files?

Yes. The command reads your existing config, adds or updates the `stacktape` server entry under the appropriate root key (e.g. `mcpServers`, `servers`, or `mcp`), and writes the file back. It creates a timestamped `.bak` backup before any modification. If the config already has a `stacktape` entry with identical content, the file is left unchanged and no backup is created.

### My assistant isn't using Stacktape context — what's wrong?

Most failures come from a few causes: the `stacktape` command isn't on your PATH (run `stacktape --version` to check), your CLI is too old to include the `mcp` subcommand (run [`stacktape upgrade`](/cli/upgrade)), MCP support isn't enabled in your client's settings, or the config file is in a location the client doesn't read for the current workspace. See [Verifying the integration](#verifying-the-integration) for the full checklist.

### How do I remove the MCP configuration?

To remove the MCP configuration, open the config file for your client and delete the `stacktape` entry. For JSON configs, remove the `"stacktape": { ... }` object under the appropriate root key (`mcpServers`, `servers`, or `mcp`). For Codex TOML, remove the `[mcp_servers.stacktape]` section and its `[mcp_servers.stacktape.env]` subsection.

### How does MCP compare to using the Stacktape CLI directly?

The MCP integration lets your AI assistant run Stacktape's MCP server as a local process and exchange structured messages over the MCP protocol. This keeps the interaction inside your editor conversation rather than requiring you to copy-paste terminal output. The CLI gives you direct control over every flag and argument and is the primary interface for all Stacktape operations. See [MCP server setup](/using-with-ai/mcp-server-setup) for which operations the MCP server currently supports.
