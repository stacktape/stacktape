# deployment-script:run

The `deployment-script:run` command re-packages and invokes a single [deployment script](/resources/advanced/deployment-scripts) without running a full [deploy](/cli/deploy). Use it to iterate on script logic — fix a migration, adjust a seed routine, or re-run a post-deploy step — without waiting for a complete stack update. The command rebuilds the script's Lambda function code from your local source, hot-swaps it onto the existing Lambda, resolves the script's `parameters` from your current local configuration, invokes the function, and prints the response payload or error to your terminal.

<Warning>
This command updates the script's **source code** and resolves `parameters` from your local configuration. Changes to environment variables, memory, timeout, VPC settings, or other Lambda configuration require a full [`deploy`](/cli/deploy).
</Warning>

## Usage

```bash
stacktape deployment-script:run --stage production --region eu-west-1 --resourceName runMigrations
```

The three required flags identify which deployed script to target:

- **`--stage`** — the stage where the script is deployed (e.g. `production`, `staging`, `dev`).
- **`--region`** — the AWS region of the stack.
- **`--resourceName`** — the name of the deployment script resource in your Stacktape configuration.

The deployment script must already exist in your configuration **and** be deployed to the target stack. If the resource is not defined in your configuration file or not present in the deployed stack, the command fails. Add the resource and run [`deploy`](/cli/deploy) first.

## Examples

Re-run a database migration script against a staging stack:

```bash
stacktape deployment-script:run --stage staging --region us-east-1 --resourceName runMigrations
```

Re-run a seed script with debug-level logging for more detailed CLI logs:

```bash
stacktape deployment-script:run --stage dev --region eu-west-1 --resourceName seedData --logLevel debug
```

Point to a non-default config file location:

```bash
stacktape deployment-script:run --stage production --region us-west-2 --resourceName postDeployHook --configPath ./infra/stacktape.ts
```

## When to use this vs deploy

Use `deployment-script:run` when you are iterating on the script's handler code and want fast feedback. The command rebuilds and hot-swaps only the script's function code, then invokes the Lambda — skipping a full stack update. The script's `parameters` are resolved from your current local configuration and sent as the invocation payload, so parameter changes take effect immediately without redeploying. Switch to a full [`deploy`](/cli/deploy) when you change Lambda configuration such as environment variables, memory, timeout, packaging settings, or VPC settings.

## Related commands

- [`deploy`](/cli/deploy) — full stack deployment. Required to provision a new deployment script or update its configuration (environment, memory, timeout).
- [`script:run`](/cli/script-run) — runs a local script defined in your configuration. Unlike `deployment-script:run`, local scripts execute on your machine rather than as a Lambda function in AWS.
- [`delete`](/cli/delete) — tears down the entire stack, including any deployment scripts.
