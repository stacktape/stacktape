# deploy

Deploy your stack to AWS. If the stack does not exist yet, Stacktape creates it. If it already exists, Stacktape updates it in place.

> **Before you start:** You must be logged in to Stacktape and have an AWS account connected in the [Stacktape Console](https://console.stacktape.com). Run `stacktape login` to configure your API key. A valid Stacktape subscription is required to deploy.

## Usage

```bash
stacktape deploy --stage <stage> --region <region> [options]
```

`--stage` and `--region` are required. All other flags are optional.

## Important Flags

### Common flags

| Flag | Required | Description |
|------|----------|-------------|
| `--stage` | Yes | Deployment stage name (e.g. `prod`, `staging`). Max 12 characters. |
| `--region` | Yes | AWS region to deploy to (e.g. `eu-west-1`, `us-east-1`). |
| `--projectName` | No | Stacktape project name. Groups stages together in the Console — affects cost dashboards, team access, and the project URL. |
| `--configPath` | No | Path to your Stacktape config file. Defaults to `stacktape.yml` in the current directory. |
| `--hotSwap` | No | Attempt a fast deploy when Stacktape determines the template diff contains only hotswap-eligible changes (Lambda functions and ECS-based services). Falls back to a full CloudFormation deploy automatically if not eligible. Only attempted on existing stacks — first deploy is always a full create. |
| `--autoConfirmOperation` | No | Skip the pre-deploy confirmation prompt that shows the change diff before applying. Required for non-interactive CI/CD pipelines. |
| `--disableAutoRollback` | No | Prevent automatic rollback when a CloudFormation update fails. Artifact cleanup is also skipped — the stack and any uploaded artifacts stay in their failed state. Use this to inspect failures before deciding how to recover. |
| `--disableDriftDetection` | No | Skip CloudFormation drift detection (a check for resources modified outside of CloudFormation, e.g. manually in the AWS Console) before deploying. Disable in pipelines where out-of-band changes are expected. |
| `--profile` | No | AWS CLI profile to use for credentials. Useful when you have multiple AWS profiles configured on your machine. |
| `--noCache` | No | Force all workloads to be rebuilt from scratch, ignoring cached artifacts. |

### Advanced flags

| Flag | Required | Description |
|------|----------|-------------|
| `--awsAccount` | No | AWS account ID to deploy into. Useful for multi-account setups. |
| `--disableDockerRemoteCache` | No | Disable Docker layer caching from the remote registry. Disable if you suspect stale cache layers are causing incorrect builds — most teams never need this. |
| `--dockerArgs` | No | Extra arguments passed directly to the Docker build command. |
| `--showSensitiveValues` | No | Print secrets and connection strings in the deployment output. Off by default to protect credentials in logs. |
| `--disableLayerOptimization` | No | Disable Lambda layer deduplication. Lambda layers are a mechanism for sharing code across Lambda functions; by default, Stacktape deduplicates identical code into shared layers to reduce deployment size and speed up cold starts. Disable only if you encounter layer-related packaging issues. |
| `--preserveTempFiles` | No | Keep temporary build files after the command completes. Useful for debugging packaging issues. |
| `--logLevel` | No | Controls log verbosity. Accepted values: `debug`, `info`, `warn`, `error`. |

## How It Works

Stacktape runs these steps when you deploy:

1. **Validate** — Guardrails defined in your config are checked before any packaging begins. Guardrails are org-level policies (e.g. region restrictions, resource limits, security rules) configured in the Stacktape Console that can block a deploy before any AWS changes are made. Secrets and SSM parameters referenced in your config are also created interactively if they don't exist yet.
2. **Package and Register** — Stacktape builds and bundles your Lambda functions (serverless compute billed per invocation), container images, and other workload artifacts. In parallel, any required CloudFormation (AWS's infrastructure-as-code engine that Stacktape uses under the hood) private types are registered in your account if not already present.
3. **Prompt** — Unless `--autoConfirmOperation` is set, Stacktape shows a resource-level diff of the CloudFormation changes — what will be added, modified, or deleted in AWS — and asks you to confirm before proceeding. This is the last checkpoint before AWS resources are modified. Press `y` to continue or `n` to cancel.
4. **Upload** — Packaged artifacts are uploaded to an S3 artifacts bucket in your account.
5. **Deploy** — CloudFormation creates or updates your stack. With `--hotSwap`, eligible Lambda functions and ECS-based services (long-running containers managed by AWS, including web-service, private-service, and worker-service) are updated directly via AWS APIs, bypassing CloudFormation for a faster cycle.
6. **Post-deploy** — When running from the CLI, resource endpoints are printed. If your config includes hosting buckets, they are synced. If CDN distributions are configured, their caches are invalidated.

### Hotswap deploys

Pass `--hotSwap` to skip a full CloudFormation deploy when Stacktape determines the template diff contains only hotswap-eligible changes. Eligible resources are Lambda functions and ECS-based services (web-service, private-service, worker-service). Stacktape analyzes the diff using its internal template diffing logic — if any change requires CloudFormation (e.g. infrastructure additions, IAM changes, new resources), it falls back automatically to a full deploy with a warning.

Use hotswap during active development to cut deploy times from minutes to seconds. Always run a full deploy before releasing to production — hotswap intentionally bypasses the CloudFormation state machine, so infrastructure changes need a full deploy to land correctly.

### New stacks vs. updates

The command handles both cases transparently. For a new stack, Stacktape first creates artifact storage resources, then runs a CloudFormation create. For an existing stack, it runs a CloudFormation update.

After a successful first deploy from an interactive terminal (TTY session), Stacktape prompts you to set up CI/CD. This prompt fires automatically in TTY-only environments and is not controlled by `--autoConfirmOperation` — in a non-TTY pipeline it is silently skipped.

### Dev-stack protection

> **Warning:** You cannot use `stacktape deploy` to update a stage that was started with `stacktape dev`. Stacktape protects dev stacks from accidental production-style deploys. If you hit this error, either delete the dev stack first (`stacktape delete --stage <stage> --region <region>`) or deploy to a different stage name.

## What Happens When a Deploy Fails

**Failed update (existing stack):** By default, CloudFormation automatically rolls back to the last known good state, and Stacktape cleans up artifacts from the failed deploy. The stack ends up in `UPDATE_ROLLBACK_COMPLETE` and is usable again.

If you pass `--disableAutoRollback`, the stack stays in its failed state AND artifact cleanup is skipped — orphaned artifacts may remain in your S3 artifacts bucket. This is intentional: use it to inspect failed resources before deciding how to recover. Once you have diagnosed the issue, use `stacktape cf:rollback` to trigger CloudFormation's native recovery and return the stack to a usable state, then fix your config and redeploy. You can also inspect logs with `stacktape debug:logs`.

**Failed create (new stack):** A failed first-time CloudFormation create leaves the stack in `ROLLBACK_COMPLETE` state. This is non-recoverable without deletion — `cf:rollback` does not apply here. Delete the stack and redeploy once you have fixed the underlying issue.

## Examples

**Basic deploy to production:**

```bash
stacktape deploy --stage prod --region eu-west-1
```

**Deploy with a custom config path (CI/CD pipeline):**

```bash
stacktape deploy \
  --stage prod \
  --region eu-west-1 \
  --projectName my-api \
  --configPath infra/stacktape.yml \
  --autoConfirmOperation
```

**Fast iteration deploy using hotswap:**

```bash
stacktape deploy --stage dev --region eu-west-1 --hotSwap
```

**Rebuild everything from scratch without cache:**

```bash
stacktape deploy --stage staging --region eu-west-1 --noCache
```

**Keep the stack in its failed state for debugging:**

```bash
stacktape deploy --stage staging --region eu-west-1 --disableAutoRollback
```

**Deploy using a specific AWS profile:**

```bash
stacktape deploy --stage prod --region eu-west-1 --profile my-prod-profile
```
