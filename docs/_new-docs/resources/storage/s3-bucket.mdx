# S3 Bucket

An AWS S3 bucket is general-purpose object storage for files, images, user uploads, backups, and any binary data. Stacktape provisions the bucket, access policies, CORS, versioning, encryption, and an optional CloudFront CDN from a single config block — so you never touch raw IAM policies or bucket configurations manually.

## When should I use an S3 bucket?

S3 buckets are the right choice when you need to:

- **Store user-uploaded files** — profile photos, documents, videos, or any user-generated content
- **Hold application assets** — ML models, seed data files, exports, or large binary blobs read at runtime
- **Archive backups and logs** — lifecycle rules automatically expire or move objects to cheaper storage
- **Trigger event-driven workflows** — every file upload or deletion can fire a Lambda function directly or fan out through EventBridge
- **Host static assets with a CDN** — put CloudFront in front of the bucket for globally fast downloads at lower bandwidth cost

**When not to use a bucket:** S3 is optimized for object-level get/put. For structured querying, row-level access, or transactions, use a relational database or DynamoDB instead.

## How do I add an S3 bucket to a Stacktape config?

A Lambda function that reads from and writes to a private bucket:

```yaml
resources:
  uploadsBucket:
    type: bucket

  processUploads:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/process-uploads.ts
      connectTo:
        - uploadsBucket
```

`connectTo` grants the function IAM read/write/delete permissions on the bucket and injects `STP_UPLOADS_BUCKET_NAME` and `STP_UPLOADS_BUCKET_ARN` (ARN — Amazon Resource Name, the unique AWS identifier for this resource) as environment variables.

Stacktape generates a hash-derived bucket name scoped by region, account, and stack, which greatly reduces collision risk between projects and stages.

## How do I make a bucket public or private?

By default, buckets are **private** — only compute resources with explicit IAM permissions (via `connectTo` or `iamRoleStatements`) can access them.

```yaml
resources:
  assetsBucket:
    type: bucket
    properties:
      accessibility:
        accessibilityMode: public-read
```

Available modes:

| Mode | Who can read | Who can write |
|------|--------------|---------------|
| `private` (default) | IAM-authorized only | IAM-authorized only |
| `public-read` | Anyone | IAM-authorized only |
| `public-read-write` | Anyone | Anyone |

For fine-grained control beyond these presets, add `accessPolicyStatements` with raw IAM policy statements under `accessibility`. This is useful for granting access to specific AWS accounts, IP ranges, or with tag-based conditions.

## How do I configure CORS for browser file uploads?

Enable CORS when browser JavaScript needs to access the bucket directly — most commonly for uploading files via presigned URLs:

```yaml
resources:
  uploadsBucket:
    type: bucket
    properties:
      cors:
        enabled: true
        corsRules:
          - allowedOrigins:
              - https://myapp.com
            allowedMethods:
              - GET
              - PUT
              - POST
            allowedHeaders:
              - '*'
            maxAge: 3600
```

When `enabled: true` is set with no `corsRules`, Stacktape uses permissive defaults (`*` origins, all methods). Provide explicit rules for production workloads.

**Presigned URLs** are the standard AWS pattern for browser-to-S3 uploads. A presigned URL is a time-limited URL your server generates that lets a browser upload a file directly to S3 — without routing the file through your server and without exposing AWS credentials to the client. Generate one server-side using the AWS SDK:

```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const s3 = new S3Client({});

export const getUploadUrl = async (key: string) => {
  return getSignedUrl(
    s3,
    new PutObjectCommand({
      Bucket: process.env.STP_UPLOADS_BUCKET_NAME,
      Key: key,
    }),
    { expiresIn: 3600 }
  );
};
```

The browser then issues a `PUT` directly to the returned URL. The URL expires after the time you specify.

## Does `directoryUpload` delete existing files?

Use `directoryUpload` to sync a local folder into the bucket on every deploy. This is designed for static build output — not user-generated data.

<Warning>
`directoryUpload` **replaces all existing bucket contents** on every deploy. Files in the bucket that are not present in the local directory are deleted. Do not use it on buckets that also store user-uploaded or application-generated files.
</Warning>

```yaml
resources:
  assetsBucket:
    type: bucket
    properties:
      directoryUpload:
        directoryPath: dist/
        headersPreset: single-page-app
```

**Header presets** configure HTTP caching automatically for your framework:

| Preset | Behavior |
|--------|----------|
| `single-page-app` | HTML never cached; hashed assets cached forever. For React, Vue, Angular SPAs. |
| `static-website` | CDN-cached, never browser-cached. For generic static sites. |
| `astro-static-website` | Files under `_astro/**/*` cached immutably; HTML always fresh. |
| `sveltekit-static-website` | Files under `_app/**/*` cached immutably; HTML always fresh. |
| `nuxt-static-website` | Files under `_nuxt/**/*` cached immutably; HTML always fresh. |
| `gatsby-static-website` | Framework-specific caching for Gatsby's build output. |

Use `fileOptions` to override headers or add tags for specific file patterns — useful when a preset doesn't match your framework's output exactly:

```yaml
resources:
  assetsBucket:
    type: bucket
    properties:
      directoryUpload:
        directoryPath: dist/
        headersPreset: single-page-app
        fileOptions:
          - includePattern: '*.html'
            headers:
              - key: Cache-Control
                value: no-cache, no-store, must-revalidate
          - includePattern: 'assets/**'
            excludePattern: 'assets/fonts/**'
            headers:
              - key: Cache-Control
                value: public, max-age=31536000, immutable
            tags:
              - key: ContentType
                value: static-asset
```

## How do I enable versioning and encryption?

```yaml
resources:
  documentsBucket:
    type: bucket
    properties:
      versioning: true
      encryption: true
```

- **`versioning`** keeps previous versions of overwritten or deleted objects. Essential for recovering from accidental deletions. Combine with a `non-current-version-expiration` lifecycle rule to keep storage costs bounded.
- **`encryption`** encrypts all stored objects at rest using AES-256. Recommended for any bucket holding user data or sensitive files.

## How do lifecycle rules reduce S3 storage costs?

Lifecycle rules automatically expire or move objects to cheaper storage classes:

```yaml
resources:
  logsBucket:
    type: bucket
    properties:
      versioning: true
      lifecycleRules:
        - type: expiration
          properties:
            daysAfterUpload: 90
            prefix: logs/

        - type: non-current-version-expiration
          properties:
            daysAfterVersioned: 30

        - type: class-transition
          properties:
            daysAfterUpload: 30
            storageClass: STANDARD_IA
            prefix: archives/

        - type: abort-incomplete-multipart-upload
          properties:
            daysAfterInitiation: 7
```

**Lifecycle rule types:**

| Type | What it does |
|------|--------------|
| `expiration` | Deletes objects N days after upload |
| `non-current-version-expiration` | Deletes old versions N days after becoming non-current (requires `versioning: true`) |
| `class-transition` | Moves objects to a cheaper storage class N days after upload |
| `non-current-version-class-transition` | Moves old versions to a cheaper storage class |
| `abort-incomplete-multipart-upload` | Cleans up stuck multipart uploads after N days |

**Storage class transitions** — the core tradeoff is storage cost versus retrieval cost and speed. Supported transition targets in Stacktape:

| Class | Best for |
|-------|----------|
| `STANDARD_IA` | Infrequently accessed data that needs instant retrieval |
| `ONEZONE_IA` | Same as STANDARD_IA, stored in a single AZ (lower durability) |
| `INTELLIGENT_TIERING` | Unpredictable access patterns — AWS auto-moves between tiers |
| `GLACIER` | Archives you rarely access (retrieval: minutes to hours) |
| `DEEP_ARCHIVE` | Long-term cold storage — cheapest to store, slowest to retrieve (12+ hours) |

*See the [AWS S3 pricing page](https://aws.amazon.com/s3/pricing/) for current storage and retrieval costs by region.*

<Info>
`STANDARD_IA` and `ONEZONE_IA` charge per-GB retrieved — if you access data frequently, retrieval charges can exceed staying in `STANDARD`. `GLACIER` and `DEEP_ARCHIVE` have minimum billing durations (90 and 180 days respectively), so they only pay off for data you'll retain long-term.
</Info>

All rules support `prefix` and `tags` filters to target only specific objects.

## How do I add a CDN to an S3 bucket?

Put a CloudFront distribution in front of the bucket for faster global downloads and lower S3 bandwidth costs:

```yaml
resources:
  assetsBucket:
    type: bucket
    properties:
      cdn:
        enabled: true
        rewriteRoutesForSinglePageApp: true
```

`rewriteRoutesForSinglePageApp: true` routes any request without a file extension to `index.html`, enabling client-side routing for SPAs.

**URL normalization** is enabled by default, allowing HTML files to be served with clean URLs. Disable with `disableUrlNormalization: true`.

**Custom domains:** to serve your bucket under your own domain, configure `cdn.customDomains`. The domain must have a Route53 hosted zone in your AWS account, with your registrar's nameservers pointing to it. Stacktape automatically provisions a TLS certificate and creates the DNS record. See the CDN documentation for full `customDomains` options.

**Cache invalidation:** when you redeploy with `directoryUpload`, Stacktape invalidates CloudFront caches automatically by default (disable with `disableInvalidationAfterDeploy: true` on the CDN). Files with hashed names (e.g., `app.a1b2c3d4.js`) are also naturally cache-busted because the filename changes. Cache headers configured via `headersPreset` or `fileOptions` additionally control browser-level caching independent of CDN invalidation.

## How do I trigger a Lambda on S3 file uploads?

Two mechanisms — choose based on your use case:

**Mechanism 1: Direct S3→Lambda trigger (simpler, most common)**

Configure an `s3` integration directly on your Lambda function. This sets up a direct S3 bucket notification with no additional bucket configuration required:

```yaml
resources:
  uploadsBucket:
    type: bucket

  processUploads:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/process-uploads.ts
      connectTo:
        - uploadsBucket
      events:
        - type: s3
          properties:
            bucketArn: $ResourceParam('uploadsBucket', 'arn')
            s3EventType: 's3:ObjectCreated:*'
            filterRule:
              prefix: uploads/
              suffix: .jpg
```

Available `s3EventType` values include `s3:ObjectCreated:*`, `s3:ObjectCreated:Put`, `s3:ObjectRemoved:*`, `s3:ObjectRestore:*`, and more specific variants. Use `filterRule` to narrow by object key prefix or suffix so only relevant objects trigger the function.

**Mechanism 2: EventBridge notifications (fan-out, complex filtering)**

EventBridge is an AWS event routing service that lets multiple independent consumers subscribe to the same event stream. Set `enableEventBusNotifications: true` on the bucket to route all S3 events through it:

```yaml
resources:
  uploadsBucket:
    type: bucket
    properties:
      enableEventBusNotifications: true

  processUploads:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/process-uploads.ts
      events:
        - type: event-bus
          properties:
            useDefaultBus: true
            eventPattern:
              source:
                - aws.s3
              detail-type:
                - 'Object Created'
              resources:
                - $ResourceParam('uploadsBucket', 'arn')
```

The event pattern fields (`source`, `detail-type`, `resources`) follow standard AWS EventBridge pattern syntax.

<Info>
`enableEventBusNotifications` and the `s3` integration type are entirely separate mechanisms. EventBridge notifications require the `event-bus` integration on your Lambda (not the `s3` integration type). The `s3` integration type is a direct S3-to-Lambda connection that works independently of EventBridge.
</Info>

**Which to use?** Use the direct `s3` integration for simple single-consumer file-processing workflows. Use EventBridge when you need multiple independent consumers for the same events or richer content-based filtering.

## How do I connect a Lambda function to an S3 bucket?

Use `connectTo` in Lambda functions, container workloads, and batch jobs to grant access and inject connection details:

```yaml
resources:
  myFunction:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/handler.ts
      connectTo:
        - assetsBucket
```

This grants full S3 read/write/delete permissions on the bucket and injects:

- `STP_ASSETS_BUCKET_NAME` — the bucket name
- `STP_ASSETS_BUCKET_ARN` — the bucket ARN

The env var prefix is `STP_` followed by the resource name converted to `SCREAMING_SNAKE_CASE`. A resource named `assetsBucket` becomes `STP_ASSETS_BUCKET_*`.

```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';

const s3 = new S3Client({});

export const handler = async () => {
  await s3.send(new PutObjectCommand({
    Bucket: process.env.STP_ASSETS_BUCKET_NAME,
    Key: 'uploads/example.txt',
    Body: 'Hello from Lambda',
  }));
};
```

**Scripts and `connectTo`:** for scripts, `connectTo` injects environment variables with connection details but does not grant IAM permissions. To give a script actual AWS access to the bucket, use `assumeRoleOfResource` to assume the IAM role of a deployed workload:

```yaml
scripts:
  syncData:
    type: local-script
    properties:
      executeCommand: node scripts/sync.js
      connectTo:
        - assetsBucket
      assumeRoleOfResource: myFunction
```

<Info>
The resource named in `assumeRoleOfResource` (here, `myFunction`) must already be deployed before the script runs — the IAM role it assumes is created during deployment.
</Info>

## S3 bucket durability, limits, and pricing model

Stacktape creates an AWS S3 bucket and configures its policies, CORS rules, lifecycle configuration, versioning, encryption settings, and optionally a CloudFront distribution — all from a single resource block.

S3 is serverless: no instances to manage, no capacity to pre-provision. You pay per GB stored per month, per-request, and for data transfer out. S3 offers 99.999999999% (11 nines) durability by redundantly storing data across multiple facilities. Buckets are regional.
