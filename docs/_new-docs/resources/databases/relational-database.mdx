# Relational Database

A managed SQL database backed by AWS RDS or Aurora. Stacktape handles provisioning, backups, patching, failover, and connection string injection — you configure the engine and connect your workloads.

<Info>
RDS and Aurora databases take 5–15 minutes to provision on first deploy. CloudFormation waits for the database to be fully ready before the deployment completes — this is normal.
</Info>

## When to Use

Use a relational database when your application needs:

- SQL queries, joins, transactions, or a relational schema
- Compatibility with PostgreSQL, MySQL, MariaDB, Oracle, or SQL Server
- Managed backups, automated patching, and point-in-time recovery without operational overhead

**Consider alternatives when:**

- Your workload is primarily key-value or document-based with unpredictable scale — DynamoDB is a better fit
- You need sub-millisecond in-memory access — Redis is more appropriate
- You're managing an existing self-hosted database — importing it into RDS may not be worth the migration cost

## Basic Example

Before deploying, store your password as a Stacktape secret. Secrets are encrypted and referenced safely in your config:

```bash
stacktape secret:create --name db.password --value yourpassword
```

Then reference the secret in your config. The quickest path to a production-ready database is Aurora Serverless v2 — it auto-scales to demand and charges only for capacity consumed:

```yaml
resources:
  mainDatabase:
    type: relational-database
    properties:
      credentials:
        masterUserName: app_user
        masterUserPassword: $Secret('db.password')
      engine:
        type: aurora-postgresql-serverless-v2
        properties:
          version: '16.6'
```

```typescript
import type { StacktapeConfig } from 'stacktape';

const config: StacktapeConfig = {
  resources: {
    mainDatabase: {
      type: 'relational-database',
      properties: {
        credentials: {
          masterUserName: 'app_user',
          masterUserPassword: "$Secret('db.password')",
        },
        engine: {
          type: 'aurora-postgresql-serverless-v2',
          properties: {
            version: '16.6',
          },
        },
      },
    },
  },
};
```

Any workload with `mainDatabase` in its `connectTo` gets network access and the `STP_MAIN_DATABASE_CONNECTION_STRING` environment variable injected automatically. The variable name follows the pattern `STP_[RESOURCE_NAME]_CONNECTION_STRING` — a resource named `userDb` would inject `STP_USER_DB_CONNECTION_STRING`.

## Key Configuration Areas

### Engine

The engine choice is the most consequential decision. There are four engine families:

#### Aurora Serverless v2 — recommended for most projects

`aurora-postgresql-serverless-v2` and `aurora-mysql-serverless-v2` scale capacity from 0.5 to 128 Aurora Capacity Units (ACUs) in 0.5-ACU steps, where ~1 ACU ≈ 2 GB RAM.

**Cost anchor:** At the minimum 0.5 ACU, Aurora Serverless v2 costs roughly $40–50/month at idle (priced per ACU-hour; varies by AWS region). The database scales up automatically under load and you pay only for what you consume — making it cost-effective for both low-traffic and spiky workloads.

**Important:** Aurora Serverless v2 does not pause to zero when idle. The database always keeps at least `minCapacity` ACUs running. For truly occasional workloads where zero-cost idle matters, see Aurora Serverless v1 below.

Defaults: `minCapacity: 0.5`, `maxCapacity: 10`. The initial database is named `defdb` unless you set `dbName`.

```yaml
engine:
  type: aurora-postgresql-serverless-v2
  properties:
    version: '16.6'
    minCapacity: 0.5
    maxCapacity: 32
    dbName: myapp
```

Add reader instances to offload read traffic:

```yaml
engine:
  type: aurora-postgresql-serverless-v2
  properties:
    version: '16.6'
    serverlessReadersCount: 1  # writer + 1 reader; default 0
```

#### Standard RDS — predictable, fixed-size instances

`postgres`, `mysql`, `mariadb`, `oracle-ee`, `oracle-se2`, and SQL Server variants give you a single fixed instance. You choose the instance size and pay hourly. This is the simplest model when capacity is predictable and constant.

An **availability zone** is an isolated data center within an AWS region. Enabling `multiAz` creates a standby copy in a second data center so a hardware failure doesn't take your database offline.

```yaml
engine:
  type: postgres
  properties:
    version: '16.4'
    primaryInstance:
      instanceSize: db.t4g.medium
      multiAz: true        # standby replica in another AZ for automatic failover
    storage:
      initialSize: 50      # GB; auto-scales up; default 20
      maxSize: 500         # default 200
```

Instance sizing guidance:
- `db.t4g.micro` — dev/test, 1 GB RAM (~$12/mo)
- `db.t4g.medium` — small production, 4 GB RAM (~$50/mo)
- `db.r6g.large` — production, 16 GB RAM (~$180/mo)

`t` family instances are burstable; use `r` family for steady, CPU-intensive workloads.

Read replicas offload read traffic, each with its own endpoint:

```yaml
readReplicas:
  - instanceSize: db.t4g.medium
  - instanceSize: db.t4g.medium
```

**Minor version upgrades** (e.g., PostgreSQL 16.4 → 16.5) happen automatically during the maintenance window by default. Set `disableAutoMinorVersionUpgrade: true` on the engine properties to control when patches are applied.

**Major version upgrades** (e.g., PostgreSQL 15 → 16) may trigger a resource replacement depending on the engine type. Always test in a staging environment first and confirm you have recent backups before bumping a major version in production.

#### Aurora (provisioned) — high-performance clusters

`aurora-postgresql` and `aurora-mysql` give you a clustered setup with data replicated across three availability zones. Reads are load-balanced across all instances. Failover to a reader replica typically completes in about 30 seconds.

```yaml
engine:
  type: aurora-postgresql
  properties:
    version: '16.4'
    instances:
      - instanceSize: db.r6g.large   # first instance = writer
      - instanceSize: db.r6g.large   # subsequent instances = readers
```

The initial database is named `defdb` unless you set `dbName`.

#### Aurora Serverless v1 — idle-pause for infrequent workloads

`aurora-mysql-serverless` and `aurora-postgresql-serverless` can **pause completely when idle** — you pay only for storage while paused. This is the defining advantage over v2: if your database sits unused for long stretches, v1 can reach near-zero cost.

For new projects, prefer Aurora Serverless v2 unless zero-cost idle is a hard requirement. V2 has faster, more granular scaling and broader feature support. V1 has a cold-start delay when resuming from a paused state.

Configure pause behavior with `pauseAfterSeconds` (300–86400 seconds):

```yaml
engine:
  type: aurora-postgresql-serverless
  properties:
    minCapacity: 2        # default: 2
    maxCapacity: 4        # default: 4
    pauseAfterSeconds: 300   # pause after 5 minutes of inactivity
    dbName: myapp
```

Default capacities: `minCapacity: 2`, `maxCapacity: 4`. Valid PostgreSQL serverless v1 values are powers of 2 from 2 to 256.

### Credentials

`masterUserPassword` is required. Always store it as a Stacktape secret:

```bash
stacktape secret:create --name db.password --value yourpassword
```

```yaml
credentials:
  masterUserName: app_user       # default: db_master_user
  masterUserPassword: $Secret('db.password')
```

<Warning>
Changing `masterUserName` after the database is created replaces the database and permanently deletes all data. Avoid the characters `[]{}(),;?*=!@` in both the username and password.
</Warning>

### Accessibility

Controls who can connect at the network level. A **VPC (Virtual Private Cloud)** is a private network in AWS where your resources run — think of it as a walled garden: only components inside the VPC, or explicitly granted access, can connect.

<Warning>
For Aurora engines, `forceDisablePublicIp` can only be set at creation time and cannot be changed later. Decide whether to remove the public IP before your first deploy — changing this after the database is created requires recreating it.
</Warning>

The default mode is `internet` — anyone with valid credentials can connect. This is fine for development but not for production.

| Mode | Who can connect |
|---|---|
| `internet` | Anyone with credentials (default) |
| `vpc` | Resources in the same VPC; optionally add `whitelistedIps` |
| `scoping-workloads-in-vpc` | Only resources with this database in their `connectTo` |
| `whitelisted-ips-only` | Only the IP addresses listed in `whitelistedIps` |

For production, `scoping-workloads-in-vpc` is the most restrictive and safest option.

```yaml
accessibility:
  accessibilityMode: scoping-workloads-in-vpc
```

To allow connections from your office IP in addition to VPC access:

```yaml
accessibility:
  accessibilityMode: vpc
  whitelistedIps:
    - 203.0.113.50/32
```

<Info>
Aurora Serverless engines only support `vpc` or `scoping-workloads-in-vpc`.
</Info>

### Connecting to Other Resources

Add the database name to a workload's `connectTo` list. Stacktape automatically opens security group rules and injects environment variables with connection details.

```yaml
resources:
  mainDatabase:
    type: relational-database
    properties:
      credentials:
        masterUserPassword: $Secret('db.password')
      engine:
        type: aurora-postgresql-serverless-v2
        properties:
          version: '16.6'

  apiService:
    type: web-service
    properties:
      connectTo:
        - mainDatabase
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: src/server.ts
      resources:
        cpu: 0.5
        memory: 1024
```

**Environment variables injected for all relational databases:**

| Variable | Description |
|---|---|
| `STP_MAIN_DATABASE_CONNECTION_STRING` | Standard connection URL |
| `STP_MAIN_DATABASE_JDBC_CONNECTION_STRING` | JDBC connection string for Java ORM frameworks (Hibernate, JDBC drivers) |
| `STP_MAIN_DATABASE_HOST` | Database hostname |
| `STP_MAIN_DATABASE_PORT` | Database port |

**Additional variables injected for Aurora clusters:**

| Variable | Description |
|---|---|
| `STP_MAIN_DATABASE_READER_CONNECTION_STRING` | Reader endpoint connection URL |
| `STP_MAIN_DATABASE_READER_JDBC_CONNECTION_STRING` | Reader endpoint JDBC URL |
| `STP_MAIN_DATABASE_READER_HOST` | Reader endpoint hostname |

Additional values — such as reader port and read replica endpoints — are accessible via `$ResourceParam`. See [Referenceable Parameters](#referenceable-parameters) below.

### Cross-Stack VPC Access

If a workload in a different Stacktape stack needs to connect to this database, both stacks must share the same VPC.

<Warning>
Set `reuseVpc` when you first create the stack. Adding it to an already-deployed stack can cause resources to be replaced and data to be lost.
</Warning>

Configure the connecting stack to reuse this stack's VPC:

```yaml
# stacktape.yml in the other stack
stackConfig:
  vpc:
    reuseVpc:
      projectName: my-project
      stage: production
```

### SSL/TLS and ORM Configuration

The connection strings injected by Stacktape do not include SSL parameters by default. RDS and Aurora enforce SSL in recent engine versions, and most ORMs require explicit SSL configuration to connect successfully.

**Prisma** — append `?sslmode=require` to the connection string or set it in your schema datasource:

```
postgresql://user:pass@host:5432/dbname?sslmode=require
```

**Drizzle** — pass `ssl: 'require'` in the client options. For local migration scripts connecting through a bastion, also set `NODE_TLS_REJECT_UNAUTHORIZED=0` to allow self-signed RDS certificates:

```typescript
const db = drizzle(process.env.STP_MAIN_DATABASE_CONNECTION_STRING, {
  ssl: 'require',
});
```

**SQLAlchemy (Python):**

```python
engine = create_engine(os.environ["STP_MAIN_DATABASE_CONNECTION_STRING"] + "?sslmode=require")
```

**node-postgres (`pg`):**

```typescript
const client = new Client({
  connectionString: process.env.STP_MAIN_DATABASE_CONNECTION_STRING,
  ssl: { rejectUnauthorized: false },
});
```

### Lambda and Connection Pooling

<Warning>
Each Lambda invocation opens a new database connection and typically does not reuse it across invocations. Under load, many concurrent invocations can exhaust your database's connection limit — even Aurora Serverless v2 has a connection ceiling that scales with ACU count but can still be exceeded.

For Lambda-heavy architectures, consider AWS RDS Proxy (a fully managed connection pooler that multiplexes Lambda invocations over a persistent pool) or a client-side pooling library like `pg-pool` for PostgreSQL.
</Warning>

### Running Migrations

After deploying a database, your first task is usually running migrations. Use an `afterDeploy` hook that runs a script with `connectTo` on the database — Stacktape injects the connection string as an environment variable.

#### With internet or whitelisted-IP accessibility

If your machine can reach the database directly, use a `local-script`:

```yaml
hooks:
  afterDeploy:
    - scriptName: runMigrations

scripts:
  runMigrations:
    type: local-script
    properties:
      executeCommand: npx prisma migrate deploy
      connectTo:
        - mainDatabase
```

#### With scoping-workloads-in-vpc (production)

When the database is VPC-restricted, your local machine cannot connect directly. A **bastion** is a small EC2 instance Stacktape uses as a secure jump host to reach VPC-only resources from your local machine.

The `bastion` must be defined as a separate resource in your config:

```yaml
resources:
  bastion:
    type: bastion

hooks:
  afterDeploy:
    - scriptName: runMigrations

scripts:
  runMigrations:
    type: local-script-with-bastion-tunneling
    properties:
      bastionResource: bastion
      executeCommand: npx prisma migrate deploy
      connectTo:
        - mainDatabase
```

The script runs locally, but Stacktape opens an SSH tunnel through the bastion so the injected `STP_MAIN_DATABASE_CONNECTION_STRING` resolves to the tunneled endpoint. Your migration tool connects as if the database were local.

### Backups

Automated daily backups are retained for 1 day by default — this is too low for production. Set `automatedBackupRetentionDays` to at least 7, up to a maximum of 35:

```yaml
automatedBackupRetentionDays: 14
```

Set to `0` to disable automated backups entirely (RDS only; Aurora always retains at least 1 day).

### Maintenance Window

Control when AWS applies patches and minor version upgrades. The database may be briefly unavailable during this window; `multiAz` or Aurora clusters minimize downtime.

```yaml
preferredMaintenanceWindow: 'Sun:02:00-Sun:04:00'   # UTC
```

Format: `Ddd:HH:MM-Ddd:HH:MM` in UTC. Pick a low-traffic window for production. To prevent automatic minor upgrades, set `disableAutoMinorVersionUpgrade: true` in the engine properties.

### Logging

Logs are sent to CloudWatch and retained for 90 days by default.

```yaml
logging:
  retentionDays: 30
  logTypes:
    - postgresql
  engineSpecificOptions:
    log_min_duration_statement: 1000   # log queries slower than 1s
    log_statement: ddl                 # log CREATE/ALTER statements (default)
    log_connections: true
```

Available `logTypes` by engine:
- PostgreSQL: `postgresql`
- MySQL/MariaDB: `audit`, `error`, `general`, `slowquery`
- Oracle: `alert`, `audit`, `listener`, `trace`
- SQL Server: `agent`, `error`

PostgreSQL-specific options (defaults shown):
- `log_min_duration_statement: 10000` — log queries slower than 10s; `0` to log all, `-1` to disable
- `log_statement: "ddl"` — options: `none`, `ddl`, `mod`, `all`
- `log_connections: false`
- `log_disconnections: false`
- `log_lock_waits: false`

MySQL-specific options (defaults shown):
- `server_audit_events: ["QUERY_DDL"]`
- `long_query_time: 10` — slow query threshold in seconds; `-1` to disable

### Alarms

Set up alarms directly on the database or configure global alarms from the Stacktape Console. Use `disabledGlobalAlarms` to suppress specific global alarms on this database without removing them globally.

```yaml
alarms:
  - trigger:
      type: database-cpu-utilization
      properties:
        thresholdPercent: 80
    notificationTargets:
      - type: slack
        properties:
          webhookUrl: $Secret('slack.webhook')
  - trigger:
      type: database-free-storage
      properties:
        thresholdMB: 5000
  - trigger:
      type: database-connection-count
      properties:
        thresholdCount: 100

disabledGlobalAlarms:
  - global-db-low-memory
```

Available alarm trigger types: `database-cpu-utilization`, `database-free-memory`, `database-free-storage`, `database-read-latency`, `database-write-latency`, `database-connection-count`.

### Deletion Protection

Prevent accidental deletion by enabling `deletionProtection`. To delete the database, first redeploy with it disabled.

```yaml
deletionProtection: true
```

Default is `false`.

### Development Mode

During `stacktape dev`, the database runs locally in Docker by default — no AWS costs while developing. Set `remote: true` to connect to the deployed database instead, which is useful when you need to test against real data or validate migrations:

```yaml
dev:
  remote: true
```

## How It Works

Stacktape provisions the database as an RDS or Aurora resource in your AWS account via CloudFormation, placed inside your stack's VPC with subnets spread across multiple availability zones.

Connection details are injected into workloads at startup as environment variables. The password is stored securely and never exposed in plaintext.

For `scoping-workloads-in-vpc` accessibility, Stacktape creates security group rules at deploy time that allow only the specific workloads listed in `connectTo` to reach the database port.

Aurora clusters replicate data across three availability zones automatically. If the writer instance fails, a reader is promoted to writer (typically within 30 seconds). Standard RDS with `multiAz: true` achieves similar availability using a synchronous standby replica in a second AZ.

## Referenceable Parameters

Use `$ResourceParam('databaseName', 'paramName')` to reference these values elsewhere in your config — for example, in environment variables or scripts that don't use `connectTo`:

| Param | Description | Available for |
|---|---|---|
| `host` | Primary writer hostname | All engines |
| `hosts` | All cluster instance hostnames | Aurora |
| `connectionString` | Standard connection URL | All engines |
| `jdbcConnectionString` | JDBC connection URL | All engines |
| `port` | Database port | All engines |
| `dbName` | Initial database name | All engines |
| `readerHost` | Reader endpoint hostname | Aurora |
| `readerPort` | Reader endpoint port | Aurora |
| `readerConnectionString` | Reader endpoint connection URL | Aurora |
| `readerJdbcConnectionString` | Reader endpoint JDBC URL | Aurora |
| `readReplicaHosts` | All read replica hostnames | RDS with read replicas |
| `readReplicaConnectionStrings` | All read replica connection URLs | RDS with read replicas |
| `readReplicaJdbcConnectionStrings` | All read replica JDBC URLs | RDS with read replicas |
| `readReplica{N}Port` | Port for the Nth read replica (0-indexed) | RDS with read replicas |

Example:

```yaml
environment:
  - name: DB_READER_HOST
    value: $ResourceParam('mainDatabase', 'readerHost')
```
