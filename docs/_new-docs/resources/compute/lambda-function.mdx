# Lambda Function

Lambda functions are short-lived, event-driven compute units that run your code without managing servers. They start in milliseconds, scale automatically from zero to thousands of concurrent invocations, and you pay only for the compute time you use — measured in GB-seconds (memory × duration).

## When to Use

Lambda functions are the right default for most workloads in Stacktape. Choose them when:

- You're handling HTTP API requests, processing queue messages, reacting to file uploads, or running scheduled jobs
- Your workload is bursty or intermittent — Lambda scales to zero when idle, so you pay nothing during quiet periods
- Individual executions complete within 15 minutes

**When not to use Lambda:**
- Tasks that run longer than 15 minutes — use a `batch-job` or `worker-service` instead
- Workloads needing persistent local state, long-lived connections, or a fixed public IP — use a container-based service
- Continuously running CPU-intensive workloads (e.g., video encoding) — Lambda's per-invocation pricing works against you; containers are cheaper for sustained load

## Basic Example

```yaml
resources:
  myApi:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/handler.ts
      events:
        - type: http-api-gateway
          properties:
            method: GET
            path: /hello
      environment:
        - name: STAGE
          value: $Stage()
```

`$Stage()` is a built-in directive that resolves to the current deployment stage name (e.g., `production`, `staging`). Stacktape has other built-in directives like `$Secret()` for fetching secrets and `$ResourceParam()` for referencing resource outputs — see the [Directives reference](/core-concepts/directives) for the full list.

This defines a Lambda function triggered by HTTP GET requests via API Gateway. Stacktape bundles your TypeScript source, uploads it to AWS, and wires the API Gateway trigger automatically.

### Handler File

The `entryfilePath` points to a file that exports a `handler` function. The event shape your function receives depends on the trigger type.

**HTTP API Gateway handler:**

```typescript
// src/handler.ts
export const handler = async (event: any) => {
  const name = event.queryStringParameters?.name ?? 'World';
  return {
    statusCode: 200,
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ message: `Hello, ${name}!` }),
  };
};
```

**SQS handler** — events arrive as batches of records:

```typescript
export const handler = async (event: { Records: Array<{ body: string; messageId: string }> }) => {
  for (const record of event.Records) {
    const message = JSON.parse(record.body);
    console.info('Processing message:', message);
    // process each record — throw to NACK the whole batch
  }
};
```

**Schedule handler** — invoked on a cron or rate expression:

```typescript
export const handler = async (_event: any) => {
  console.info('Running scheduled task at', new Date().toISOString());
  // perform periodic work
};
```

For `http-api-gateway`, the event follows the AWS API Gateway v2 payload format. For `sqs`, it's a batch of records with retry/DLQ behavior controlled by the SQS trigger configuration. For `schedule`, it's a CloudWatch Events object. Stacktape does not alter the event format — it matches standard AWS Lambda event shapes.

The same config in TypeScript format:

```typescript
import type { StacktapeConfig } from 'stacktape';

const config: StacktapeConfig = {
  resources: {
    myApi: {
      type: 'function',
      properties: {
        packaging: {
          type: 'stacktape-lambda-buildpack',
          properties: { entryfilePath: 'src/handler.ts' },
        },
        events: [{ type: 'http-api-gateway', properties: { method: 'GET', path: '/hello' } }],
        environment: [{ name: 'STAGE', value: '$Stage()' }],
      },
    },
  },
};

export default config;
```

## Key Configuration Areas

### Packaging

How Stacktape builds and packages your code for Lambda.

**`stacktape-lambda-buildpack`** (recommended): Point to your source file. Stacktape bundles, tree-shakes, and uploads it automatically.

```yaml
packaging:
  type: stacktape-lambda-buildpack
  properties:
    entryfilePath: src/handler.ts
```

**`custom-artifact`**: Provide a pre-built `.zip` file. Use this when you manage your own build pipeline or need a custom runtime.

```yaml
packaging:
  type: custom-artifact
  properties:
    artifactPath: dist/my-function.zip
```

### Runtime

When using `stacktape-lambda-buildpack`, the runtime is auto-detected from your entry file extension (`.ts`/`.js` → Node.js, `.py` → Python, `.go` → Go). You only need to set `runtime` explicitly when you want to pin a specific version or use a runtime that cannot be inferred from the file extension.

```yaml
packaging:
  type: stacktape-lambda-buildpack
  properties:
    entryfilePath: src/handler.ts
runtime: nodejs20.x   # pin to Node.js 20 instead of the auto-detected default
```

Accepted values match AWS Lambda runtime identifiers. Common examples:

| Language | Runtime values |
|---|---|
| Node.js | `nodejs22.x`, `nodejs20.x` |
| Python | `python3.13`, `python3.12`, `python3.11` |
| Go | `go1.x`, `provided.al2023` |
| Java | `java21`, `java17` |
| .NET | `dotnet8` |
| Ruby | `ruby3.3` |

When using `custom-artifact`, you must always set `runtime` explicitly because Stacktape cannot infer it from a zip file.

### Events (Triggers)

Lambda functions can respond to a wide range of event sources. Stacktape configures IAM permissions automatically for each one.

| Trigger type | Use case |
|---|---|
| `http-api-gateway` | REST/HTTP APIs |
| `schedule` | Cron jobs, periodic tasks |
| `s3` | React to file uploads or deletions |
| `sqs` | Process queue messages |
| `sns` | Fan-out notifications |
| `kinesis` | Real-time stream processing |
| `dynamo-db` | React to table changes via DynamoDB Streams |
| `event-bus` | Event-driven architectures |
| `application-load-balancer` | HTTP via ALB |
| `cloudwatch-log` | Log-driven processing |
| `kafka-topic` | MSK/Kafka consumers |
| `alarm` | React to CloudWatch alarms |

Multiple triggers can be attached to a single function:

```yaml
events:
  - type: http-api-gateway
    properties:
      method: POST
      path: /process
  - type: schedule
    properties:
      scheduleRate: rate(1 hour)
```

Events and triggers are documented in depth in the [Events & Triggers](/events-and-triggers/overview) section.

### Connecting to Other Resources

Use `connectTo` to give a function access to databases, buckets, queues, and other resources. Stacktape automatically grants IAM permissions and injects environment variables.

```yaml
resources:
  myFunction:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/handler.ts
      connectTo:
        - myDatabase
        - myBucket
        - myQueue
```

The table below shows **selected common examples** of injected environment variables. Variable names follow the pattern `STP_[RESOURCE_NAME]_[PARAM]` (uppercased, dashes replaced by underscores).

| Resource type | Injected environment variables |
|---|---|
| `relational-database` | `STP_MY_DATABASE_CONNECTION_STRING`, `STP_MY_DATABASE_JDBC_CONNECTION_STRING`, `STP_MY_DATABASE_HOST`, `STP_MY_DATABASE_PORT`. Aurora clusters also get `STP_MY_DATABASE_READER_CONNECTION_STRING`, `STP_MY_DATABASE_READER_JDBC_CONNECTION_STRING`, `STP_MY_DATABASE_READER_HOST`. |
| `redis-cluster` | `STP_MY_REDIS_HOST`, `STP_MY_REDIS_READER_HOST`, `STP_MY_REDIS_PORT` |
| `bucket` | `STP_MY_BUCKET_NAME`, `STP_MY_BUCKET_ARN` |
| `dynamo-db-table` | `STP_MY_TABLE_NAME`, `STP_MY_TABLE_ARN`. `STP_MY_TABLE_STREAM_ARN` is also injected when the table has a stream configured (`streamType` is set). |
| `sqs-queue` | `STP_MY_QUEUE_ARN`, `STP_MY_QUEUE_NAME`, `STP_MY_QUEUE_URL` |
| `sns-topic` | `STP_MY_TOPIC_ARN`, `STP_MY_TOPIC_NAME` |
| `mongo-db-atlas-cluster` | `STP_MY_CLUSTER_CONNECTION_STRING` |
| `upstash-redis` | `STP_MY_REDIS_HOST`, `STP_MY_REDIS_PORT`, `STP_MY_REDIS_PASSWORD`, `STP_MY_REDIS_REST_TOKEN`, `STP_MY_REDIS_REST_URL`, `STP_MY_REDIS_REDIS_URL` |
| `function` | `STP_MY_FUNCTION_ARN` |
| `user-auth-pool` | `STP_MY_POOL_ID`, `STP_MY_POOL_CLIENT_ID`, `STP_MY_POOL_ARN` |
| `aws:ses` | No injected variables — grants full SES email-sending permissions via the function's IAM role. Use the AWS SDK directly with the assumed role. |

For permissions not covered by `connectTo` (e.g., AWS Bedrock, Rekognition, Textract), use `iamRoleStatements`:

```yaml
iamRoleStatements:
  - Effect: Allow
    Action:
      - bedrock:InvokeModel
    Resource:
      - "*"
```

### Memory and CPU

Lambda allocates CPU proportionally to memory. The formula: **1,769 MB = 1 vCPU**.

- Default memory is often low enough that CPU-intensive tasks are noticeably slow
- Increasing memory often makes a function faster *and* cheaper overall: higher per-GB rate but much shorter execution time
- Range: 128 MB to 10,240 MB

```yaml
memory: 1024   # ~0.58 vCPU — good for most I/O-heavy functions
```

```yaml
memory: 3538   # 2 vCPUs — for CPU-bound work (image processing, parsing)
```

### Architecture

Default is `x86_64`. Switch to `arm64` (AWS Graviton) for roughly 20% lower cost per GB-second with no code changes for most applications:

```yaml
architecture: arm64
```

When using `stacktape-lambda-buildpack`, Stacktape compiles for the selected architecture automatically. With `custom-artifact`, you must pre-build for the target architecture. Lambda Layers must also be built for the same architecture.

### Timeout

Default timeout is 10 seconds. Maximum is 900 seconds (15 minutes).

```yaml
timeout: 30
```

Functions are killed if execution exceeds the timeout. For workloads that regularly need more than 15 minutes, use `batch-job` or `worker-service`.

### Storage (`/tmp` Disk)

Each Lambda invocation gets a `/tmp` directory for temporary files. Default is 512 MB; you can increase it up to 10,240 MB.

```yaml
storage: 2048   # 2 GB of /tmp space
```

Increase this when your function downloads large files, unzips archives, loads ML model weights, or processes large media. **This storage is ephemeral** — it does not persist between invocations and is not shared across instances. For persistent shared storage, use an EFS volume mount (see below).

### Tags

Resource-level tags for this function, in addition to any stack-level tags. Max 50.

```yaml
tags:
  - name: feature
    value: user-notifications
  - name: team
    value: backend
```

Tags are useful for cost allocation in AWS Cost Explorer — you can filter Lambda costs per feature or team without waiting for stack-level breakdowns. Stacktape automatically adds `projectName`, `stage`, and `stackName` tags to every resource; your custom tags are merged on top.

### VPC Access

By default, Lambda functions run outside your VPC and have full internet access. Set `joinDefaultVpc: true` to place the function inside the VPC, enabling access to databases and Redis clusters that require network-level access.

```yaml
joinDefaultVpc: true
```

**Important tradeoff:** VPC-connected functions lose direct internet access. They can still reach S3 and DynamoDB (Stacktape creates VPC endpoints automatically), but calls to external APIs (Stripe, OpenAI, etc.) will fail. If you need both VPC access and internet from a Lambda function, use a `web-service` or `worker-service` instead.

Most resources that require VPC access (any `redis-cluster`, or a `relational-database` with `accessibilityMode: vpc`) will surface an error during deploy if `joinDefaultVpc` is missing. In practice, you may need to set it manually when connecting to VPC-only resources from a function.

### Lambda Layers

Layers are zip archives mounted into the function's execution environment before your handler runs. Common uses: shared native dependencies, custom runtimes, monitoring agents.

```yaml
layers:
  - arn:aws:lambda:eu-west-1:123456789012:layer:my-shared-layer:3
  - arn:aws:lambda:eu-west-1:901920949190:layer:NewRelicNodeJS:47
```

An ARN (Amazon Resource Name) is AWS's unique identifier for any resource, formatted as `arn:aws:service:region:account-id:resource-type:resource-id:version`. Layer ARNs come from the AWS console, third-party providers, or other stacks.

Maximum 5 layers per function. If your function uses `arm64` architecture, every attached layer must also be built for `arm64`.

### Function URL

Give a function its own HTTPS endpoint without an API Gateway:

```yaml
url:
  enabled: true
  authMode: NONE
  cors:
    enabled: true
    allowedOrigins:
      - https://myapp.com
```

URL format: `https://{id}.lambda-url.{region}.on.aws`

`authMode` options:
- `NONE` — public, anyone can call it (default)
- `AWS_IAM` — only AWS users/roles with explicit invoke permission

This is simpler and cheaper than API Gateway for single-function endpoints. Use API Gateway when you need routing across multiple functions, request transformation, usage plans, or custom authorizers.

**Response streaming** raises the maximum response size from 6 MB to 20 MB and improves time-to-first-byte. Enable it with:

```yaml
url:
  enabled: true
  responseStreamEnabled: true
```

Streaming requires the AWS streaming handler pattern in your function. Here is the required handler structure:

```typescript
import { streamifyResponse, ResponseStream } from 'aws-lambda';

export const handler = streamifyResponse(async (event, responseStream, context) => {
  responseStream.setContentType('text/plain');
  responseStream.write('First chunk\n');
  // stream additional chunks...
  responseStream.end();
});
```

### CDN

Put a CloudFront CDN in front of your function to cache responses at edge locations worldwide:

```yaml
cdn:
  enabled: true
```

This reduces function invocations and lowers latency for global users. Use CDN when your function serves public, cacheable content (e.g., static API responses, media metadata). Skip it for personalized responses, write operations, or workloads where caching would return stale data.

### Provisioned Concurrency (Eliminating Cold Starts)

When a Lambda function hasn't been called recently, the first request can take noticeably longer as AWS initializes the execution environment (a "cold start"). Cold start latency ranges from under a second for Node.js and Python to several seconds for JVM or .NET runtimes — actual duration depends heavily on runtime, memory size, and whether VPC networking is enabled.

```yaml
provisionedConcurrency: 2
```

This keeps 2 instances warm and ready at all times. Use it for user-facing APIs and latency-sensitive workloads. Skip it for background jobs, cron tasks, or pipelines.

**Cost note:** You pay for provisioned instances even when idle. It also increases deploy time by approximately 2–5 minutes.

### Reserved Concurrency

Cap how many concurrent instances this function can run:

```yaml
reservedConcurrency: 10
```

Common reasons to set this:
- Prevent overwhelming a database connection pool
- Throttle expensive downstream API calls (OpenAI, payment providers)
- Reserve guaranteed capacity for a critical function while protecting other functions from being starved

No additional cost.

> Note: Due to how Stacktape processes this value, setting `reservedConcurrency` to `0` has no effect — the limit is not applied. Use a low positive number (e.g., `1`) if you need aggressive throttling.

### Logging

By default, all output written to `stdout` and `stderr` is forwarded to CloudWatch Logs and retained for 180 days.

**Retention period:**

```yaml
logging:
  retentionDays: 30
```

Allowed values: `1`, `3`, `5`, `7`, `14`, `30`, `60`, `90`, `120`, `150`, `180`, `365`, `400`, `545`, `731`, `1827`, `3653`. Longer retention increases CloudWatch storage costs — set a shorter value for non-production functions.

**Disable logging** for high-volume functions where you forward logs elsewhere:

```yaml
logging:
  disabled: true
```

**Log forwarding** to external destinations (Datadog, Logtail, etc.) is also supported via `logging` configuration. See the [Monitoring & Observability](/monitoring-and-observability/logs) section for full details.

### Gradual Deployments

Shift traffic progressively to a new version instead of switching all at once. This uses AWS CodeDeploy under the hood.

```yaml
deployment:
  strategy: Canary10Percent5Minutes
  beforeAllowTrafficFunction: runSmokeTests
  afterTrafficShiftFunction: runPostDeployValidation
```

**`beforeAllowTrafficFunction`**: A Lambda function in your stack that runs before traffic begins shifting. Use it for smoke tests. If it signals failure to CodeDeploy, the deployment rolls back before any traffic moves.

**`afterTrafficShiftFunction`**: A Lambda function that runs after all traffic has shifted. Use it for post-deploy validation. If it signals failure, CodeDeploy rolls back.

Hook functions must explicitly signal success or failure to CodeDeploy using the AWS SDK. Without this call, CodeDeploy will time out and treat the hook as failed:

```typescript
import { CodeDeployClient, PutLifecycleEventHookExecutionStatusCommand } from '@aws-sdk/client-codedeploy';

const client = new CodeDeployClient({});

export const handler = async (event: any) => {
  const { DeploymentId, LifecycleEventHookExecutionId } = event;
  let status: 'Succeeded' | 'Failed' = 'Succeeded';

  try {
    // Run your validation logic here
    const res = await fetch(process.env.API_URL + '/health');
    if (!res.ok) status = 'Failed';
  } catch {
    status = 'Failed';
  }

  await client.send(new PutLifecycleEventHookExecutionStatusCommand({
    deploymentId: DeploymentId,
    lifecycleEventHookExecutionId: LifecycleEventHookExecutionId,
    status,
  }));
};
```

Available strategies:

- **Canary**: Send 10% of traffic first, then all traffic after a wait period
  - `Canary10Percent5Minutes`
  - `Canary10Percent10Minutes`
  - `Canary10Percent15Minutes`
  - `Canary10Percent30Minutes`
- **Linear**: Shift 10% of traffic at regular intervals
  - `Linear10PercentEvery1Minute`
  - `Linear10PercentEvery2Minutes`
  - `Linear10PercentEvery3Minutes`
  - `Linear10PercentEvery10Minutes`
- **AllAtOnce**: Instant traffic switch — CodeDeploy is still used but no gradual rollout occurs

### EFS Volume Mounts

Mount a persistent EFS filesystem at a path inside the function:

```yaml
joinDefaultVpc: true
volumeMounts:
  - type: efs
    properties:
      efsFilesystemName: myFilesystem
      mountPath: /mnt/data
      rootDirectory: /uploads
```

Unlike `/tmp`, EFS data persists indefinitely and can be shared across multiple functions or container services. Requires `joinDefaultVpc: true`.

### Async Destinations

Route the result of async invocations to another service. Destinations are specified as ARNs (AWS resource identifiers):

```yaml
destinations:
  onSuccess: arn:aws:sqs:eu-west-1:123456789012:results-queue
  onFailure: arn:aws:sqs:eu-west-1:123456789012:dead-letter-queue
```

Supports SQS, SNS, EventBus, and Lambda ARNs. Useful for fan-out pipelines and async error handling.

**Note on failure handling:** `onFailure` fires after the function exhausts all retry attempts (async invocations retry twice by default). This is distinct from configuring a DLQ on the SQS trigger itself — trigger-level DLQs capture messages that could not be delivered to the function, while `onFailure` destinations receive the function's invocation result after retries are exhausted. For most async error-handling patterns, `onFailure` is the right place to route failures.

### Alarms

Functions support resource-level alarms that merge with any global alarms configured in the Stacktape Console:

```yaml
alarms:
  - type: error-rate
    properties:
      threshold: 5
  - type: duration
    properties:
      threshold: 8000
```

Use `disabledGlobalAlarms` to suppress specific global alarm names for this function:

```yaml
disabledGlobalAlarms:
  - GlobalHighErrorRate
```

See the [Monitoring & Observability](/monitoring-and-observability/alarms) section for full alarm configuration options.

## How It Works

Lambda functions scale automatically. AWS maintains a pool of execution environments and assigns one to each incoming request. When traffic spikes, AWS adds more instances. When traffic drops to zero, all instances are reclaimed and you pay nothing.

**Cold starts** happen when there is no warm instance ready: AWS must initialize your runtime and run your module-level code before handling the request. Latency varies from under a second for Node.js and Python to several seconds for JVM or .NET runtimes, and is also affected by memory size and VPC networking. Provisioned concurrency eliminates cold starts by pre-warming instances.

**Execution model**: each invocation gets its own isolated environment. There is no shared in-process memory between concurrent invocations. Module-level variables persist across sequential invocations on the same warm instance, but you cannot rely on this for state that must be shared across all instances.

**Network**: without `joinDefaultVpc`, functions run in AWS-managed infrastructure with full internet access. With `joinDefaultVpc`, they run in your stack's VPC, losing direct internet access but gaining access to VPC-only resources (databases, Redis clusters). S3 and DynamoDB remain accessible via VPC endpoints that Stacktape creates automatically.

## Pricing

Lambda pricing has two components:

1. **Requests**: charged per invocation, with a free tier of 1 million requests per month
2. **Duration**: charged per GB-second (memory × execution time), rounded to 1ms — `arm64` is approximately 20% cheaper per GB-second than `x86_64`

The AWS free tier covers both requests and GB-seconds every month, not just the first year. For exact current pricing, see the [AWS Lambda pricing page](https://aws.amazon.com/lambda/pricing/).

**Practical implication**: Lambda is very cost-effective for intermittent and bursty workloads. For functions running under continuous heavy load, containers priced per-hour may become cheaper to operate.

## Observability

### Logs

All `stdout` and `stderr` output is automatically sent to CloudWatch Logs. You can view logs with `stacktape logs` or in the Stacktape Console.

### Issue Detection

Stacktape automatically detects runtime errors in Node.js/TypeScript Lambda functions — including uncaught exceptions, unhandled promise rejections, and `console.error` output — and reports them as **Issues** in the Stacktape Console. Issues are grouped by error signature so you can spot recurring failures without manually scanning logs.

This works out of the box without any code changes. To disable it for an entire stack:

```yaml
stackConfig:
  disableIssues: true
```

## Development Notes

Lambda functions support Stacktape dev mode (`stacktape dev`). In dev mode, your code runs locally with hot reload while staying connected to real AWS resources (databases, queues, etc.) in the cloud. Changes to your source file are reflected immediately without redeployment.

## Production Notes

For production Lambda functions, consider:

- **`arm64` architecture** for a straightforward ~20% cost reduction on most workloads
- **`retentionDays: 30`** (or shorter) on non-production functions to control CloudWatch costs
- **Provisioned concurrency** for latency-sensitive user-facing endpoints
- **Reserved concurrency** to protect databases from connection pool exhaustion
- **Gradual deployment** with lifecycle hooks for critical functions
- **Termination protection** on the stack to prevent accidental deletion:

```yaml
deploymentConfig:
  terminationProtection: true
```

To delete a stack with termination protection, first deploy with `terminationProtection: false`, then run the delete command.
