# Choosing Compute

Stacktape offers seven compute resource types. Picking the right one upfront avoids rework and controls cost.

The most common decision is **function vs. container service**. Here is the financial reality up front:

- A `function` costs **$0 at idle** — you pay only per invocation (roughly $0.20 per million requests at 128 MB / 100 ms each).
- A `web-service` on Fargate runs 24/7. At minimum size (0.25 vCPU / 512 MB), that's **~$9/month per stage** even with zero traffic. Run three stages (dev, staging, production) and that's ~$27/month before a single request arrives.

For most new projects, start with `function`. Move to a container service when you need always-on latency, WebSockets, GPU, or long-running processes.

## Quick Recommendation

- **Event-driven HTTP API or webhook** → [`function`](#lambda-function) with an HTTP trigger
- **Always-on API or WebSocket server** → [`web-service`](#web-service)
- **Finite data processing, ML inference, or batch jobs** → [`batch-job`](#batch-job)
- **Continuous queue or stream consumer** → [`worker-service`](#worker-service)
- **Internal microservice reachable only from other containers** → [`private-service`](#private-service)
- **Multiple primary containers each needing independent traffic sources** → [`multi-container-workload`](#multi-container-workload)
- **Request/response manipulation at the CDN edge** → [`edge-lambda-function`](#edge-lambda-function)

## Comparison Table

| Resource | Always on | Public URL | Max runtime | Billing | Cold starts | GPU |
|---|---|---|---|---|---|---|
| `function` | No | Optional | 15 min (hard cap) | Per invocation | Yes | No |
| `web-service` | Yes | Yes | Unlimited | Per hour | No | Yes (EC2) |
| `private-service` | Yes | No | Unlimited | Per hour | No | Yes (EC2) |
| `worker-service` | Yes | No | Unlimited | Per hour | No | Yes (EC2) |
| `multi-container-workload` | Yes | Optional | Unlimited | Per hour | No | Yes (EC2) |
| `batch-job` | No | No | Configurable | Per run | No | Yes (dedicated field) |
| `edge-lambda-function` | No | No (CDN only) | 5–30 sec | Per invocation | Yes | No |

For `web-service`, `private-service`, `worker-service`, and `multi-container-workload`, GPU requires selecting GPU-capable EC2 instance types (e.g., `g4dn.xlarge`) via `resources.instanceTypes`. `batch-job` has a dedicated `resources.gpu` field.

## Fargate vs EC2

Container-based resources (`web-service`, `private-service`, `worker-service`, `multi-container-workload`) can run on either:

- **Fargate** (default): AWS's serverless container engine. You specify `cpu` and `memory`; AWS provisions and manages the underlying servers. No EC2 instances to maintain or patch.
- **EC2 instance types**: You specify instance types (e.g., `t3.medium`, `g4dn.xlarge`) via `resources.instanceTypes`. Lower cost at sustained utilization, required for GPU workloads, and supports warm pools for faster scaling.

Most teams start with Fargate and switch to EC2 only when cost optimization or GPU workloads demand it.

## When to Choose Each Option

### Lambda Function

Lambda is the default for event-driven workloads. Each invocation runs in an isolated, ephemeral environment. Functions scale automatically from zero to thousands of concurrent instances with no provisioning.

**Use it when:**
- Code responds to events: HTTP requests, queue messages, file uploads, schedules, DynamoDB streams
- Tasks complete in under 15 minutes
- Traffic is bursty or unpredictable and you want to pay only when code runs
- You want the lowest operational overhead and the fastest path to production

**Avoid it when:**
- Tasks run longer than 15 minutes (use `batch-job` or `worker-service`)
- You need WebSocket connections that stay open (use `web-service` with ALB)
- You need GPU compute (use `batch-job`)
- Cold start latency is unacceptable and provisioned concurrency cost is not justified

**Minimal config:**

```yaml
resources:
  myApi:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/api.ts
      events:
        - type: http-api-gateway
```

**Function URL vs. API Gateway:** For a single-function HTTP endpoint, set `url.enabled: true` to give your function its own HTTPS URL — no API Gateway needed. A function URL is simpler and cheaper than an API Gateway for single-function endpoints. Use an HTTP API Gateway when you need multiple functions behind one endpoint, request routing, or JWT authorizers.

**Cold starts:** The first request after inactivity can take 1–5+ seconds while AWS initializes the environment. Node.js and Python typically cold-start faster than JVM-based runtimes. Set `provisionedConcurrency` to keep instances warm for latency-sensitive endpoints. This adds idle cost and approximately 2–5 minutes to deploys.

**Memory and CPU:** Lambda scales CPU proportionally to memory. 1,769 MB = 1 vCPU. If your function is slow, increasing memory gives it more CPU — which often makes it faster and cheaper overall by reducing execution time.

**VPC access:** `joinDefaultVpc: true` is required to reach databases and Redis. Once inside the VPC, the function loses direct internet access. Stacktape auto-creates VPC endpoints for S3 and DynamoDB, but calls to external APIs (Stripe, OpenAI, etc.) will fail. To reach both VPC resources and the internet, enable `usePrivateSubnetsWithNAT: true` (~$32/month per NAT Gateway) — or restructure to avoid the VPC constraint entirely.

**Dev mode:** Functions support `stacktape dev` with local emulation.

### Web Service

A container that runs 24/7 with a public HTTPS URL. Stacktape handles TLS, health checks, auto-scaling, and zero-downtime deployments.

**Use it when:**
- You need an always-on API or web backend with consistent latency
- WebSocket or long-lived HTTP connections are required
- You need GPU instances via EC2 instance types
- Traffic is high enough that per-request Lambda pricing becomes expensive

**Avoid it when:**
- Traffic is bursty or low — you pay for at least one running instance 24/7
- You only need event-driven invocations (`function` is cheaper and simpler)

**Minimal config:**

```yaml
resources:
  myApi:
    type: web-service
    properties:
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: src/server.ts
      resources:
        cpu: 0.5
        memory: 1024
```

**Load balancing options:**
- **`http-api-gateway`** (default): Pay-per-request, ~$1/million requests. Cheapest at low traffic.
- **`application-load-balancer`**: ~$18/month flat + usage. Required for gradual deployments, WAF, and WebSocket support. More cost-effective above ~500k requests/day.
- **`network-load-balancer`**: For non-HTTP traffic (TCP/TLS) like MQTT, game servers, or custom binary protocols. Does not support CDN, firewall, or gradual deployments.

**Scaling:** Configure `scaling.minInstances` and `scaling.maxInstances`. The minimum supported value for `minInstances` is 1 — container services cannot scale to zero.

**Sidecars and init containers:** Use `sideContainers` to run helper containers. `run-on-init` containers (e.g., database migrations) must exit with code 0 before the main container starts. `always-running` containers (e.g., log forwarders, monitoring agents) run alongside the main container for its entire lifetime.

<Warning>
Container services cannot scale to zero. You pay for at least one running instance at all times — including staging environments. If you run multiple stages (dev, staging, production), that base cost multiplies. For low-traffic or ephemeral environments, `function` or `batch-job` are significantly cheaper.
</Warning>

### Private Service

An always-on container with no public endpoint. Other containers in the same stack reach it by its resource name — for example, `http://myService:3000`. Built on the same Fargate/EC2 foundation as `web-service`.

**Use it when:**
- You're building an internal microservice, gRPC server, or backend-for-frontend
- The service should never be reachable from the public internet
- You need container-to-container communication with low overhead

**Avoid it when:**
- External clients need to reach it (use `web-service`)
- You only need occasional task execution (use `batch-job` or `function`)

**Minimal config:**

```yaml
resources:
  internalApi:
    type: private-service
    properties:
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: src/internal.ts
      resources:
        cpu: 0.25
        memory: 512
```

**Internal networking — a concrete decision:** The default `service-connect` load balancing (~$0.50/month for Cloud Map service discovery) is reachable only from other container-based resources in the same stack. If you need Lambda functions to call this service, switch `loadBalancing.type` to `application-load-balancer` — Lambda functions that have joined the VPC can then reach it by its ALB address.

**Sidecars and init containers:** Like `web-service`, `private-service` supports `sideContainers` with `run-on-init` and `always-running` container types.

### Worker Service

An always-on container with no public URL and no load balancer — it only initiates outbound connections. Runs inside your VPC and can be reached by other VPC resources, but has no public endpoint. Built on the same Fargate/EC2 foundation as other container services.

**Use it when:**
- You have a long-running process that continuously polls a queue or stream in an infinite loop — for example, an SQS consumer or Kinesis reader
- You need a stateful background worker that runs indefinitely
- You want container-based execution without any inbound network endpoint

**Avoid it when:**
- The task runs to completion and stops (use `batch-job`)
- You need to receive inbound requests (use `web-service` or `private-service`)
- Work is short-lived and event-triggered (use `function`)

**Minimal config:**

```yaml
resources:
  myWorker:
    type: worker-service
    properties:
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: src/worker.ts
      resources:
        cpu: 0.5
        memory: 1024
      connectTo:
        - myQueue
```

**Consuming SQS or Kinesis:** `worker-service` runs whatever code you write. Use `connectTo` to inject the queue URL or stream ARN as environment variables (`STP_MY_QUEUE_URL`, etc.), then poll from your application code using your preferred SDK. There is no built-in queue integration — your code implements the polling loop.

**Full container capabilities:** `worker-service` supports horizontal scaling via `scaling.minInstances`/`maxInstances`, EFS volume mounts, `sideContainers`, `arm64` architecture (~20% cheaper), and `usePrivateSubnetsWithNAT` for a static outbound IP address. The minimum `minInstances` is 1.

### Multi-container Workload

<Info>
For most sidecar and init-container patterns, you do not need `multi-container-workload`. All three simpler service types (`web-service`, `private-service`, `worker-service`) support `sideContainers` with `run-on-init` and `always-running` types. Use `multi-container-workload` only when the simpler types genuinely cannot cover your setup — for example, when you need multiple primary containers each receiving independent traffic.
</Info>

Runs multiple containers together as a single deployable unit on shared compute. All containers start, scale, and stop together.

**Use it when:**
- Multiple primary containers each need their own independent event integrations or traffic sources (e.g., two containers each handling different HTTP routes or queues)
- You need startup dependency ordering between multiple primary containers via `dependsOn` (conditions: `START`, `COMPLETE`, `SUCCESS`, `HEALTHY`). Note: sidecar containers on simpler service types also support `dependsOn`, so this only applies when primary-to-primary ordering is required.
- Your tightly-coupled processes genuinely cannot be modeled as one main container plus sidecars

**Avoid it when:**
- You only have one main container — use `web-service`, `private-service`, or `worker-service` instead
- You only need a sidecar or init container — use `sideContainers` on a simpler type
- Containers are loosely coupled and scale independently — deploy them as separate services

**Minimal config:**

```yaml
resources:
  myWorkload:
    type: multi-container-workload
    properties:
      resources:
        cpu: 1
        memory: 2048
      containers:
        - name: app
          packaging:
            type: stacktape-image-buildpack
            properties:
              entryfilePath: src/app.ts
          events:
            - type: http-api-gateway
        - name: sidecar
          packaging:
            type: stacktape-image-buildpack
            properties:
              entryfilePath: src/sidecar.ts
```

### Batch Job

Runs a container to completion on demand, then stops. AWS provisions compute capacity when the job is triggered. You pay only for the compute time consumed — no idle cost.

**Use it when:**
- Processing is finite: data imports, image/video encoding, ML inference, report generation
- Work is triggered by a schedule, HTTP request, queue message, or S3 upload
- Jobs need to run longer than 15 minutes (Lambda's hard limit)
- You need GPU instances for ML workloads

**Avoid it when:**
- You need a persistent process polling a queue continuously (use `worker-service`)
- Work must respond to HTTP requests interactively (use `function` or `web-service`)

**Minimal config:**

```yaml
resources:
  myJob:
    type: batch-job
    properties:
      resources:
        cpu: 2
        memory: 4096
      container:
        packaging:
          type: stacktape-image-buildpack
          properties:
            entryfilePath: src/job.ts
```

**Spot instances:** Set `useSpotInstances: true` to use spare AWS capacity and save up to 90%. Jobs can be interrupted — enable only if your job can safely restart. Combine with `retryConfig` to automatically retry on interruption.

**GPU:** Set `resources.gpu` to the number of GPUs required. AWS provisions a GPU-capable instance automatically.

**Retry and timeout:** Configure `retryConfig` with `attempts`, `retryIntervalSeconds`, and `retryIntervalMultiplier` for exponential backoff. `timeout` kills the job after the specified seconds; there is no stated upper cap unlike Lambda's hard 15-minute limit.

**Dev mode:** Batch jobs support `stacktape dev`.

---

### Edge Lambda Function

A Lambda function that runs at CloudFront edge locations, not in your stack's region. Invoked on CDN events: viewer request, viewer response, origin request, or origin response.

**Use it when:**
- You need to modify request or response headers at the edge (auth checks, security headers)
- You're implementing URL rewriting, A/B testing, or geo-based redirects before requests reach your origin

**Minimal config:**

```yaml
resources:
  myEdgeFn:
    type: edge-lambda-function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/edge.ts
```

**Important constraints:**
- Node.js and Python runtimes only
- Cannot use environment variables
- Cannot connect to VPC resources
- `connectTo` grants IAM permissions only (e.g., S3, DynamoDB) — no connection strings or network access
- Viewer events: max 128 MB memory, 5-second timeout
- Origin events: max 10,240 MB memory, 30-second timeout
- Must be referenced from a CDN `edgeFunctions` configuration — not usable standalone

**Skip this for most use cases.** If you're not specifically manipulating CDN-level request/response flow, a regular `function` with an HTTP trigger is simpler and more capable.
