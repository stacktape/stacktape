# Web Service

A web service is an always-on container with a public HTTPS URL. Use it for REST APIs, GraphQL servers, web apps, or any
workload that must be reachable from the internet 24/7.

Stacktape handles TLS certificates, load balancing, auto-scaling, health checks, and zero-downtime deployments
automatically. Under the hood, every web service runs on AWS ECS (Elastic Container Service).

## When to Use

**Choose a web service when you need:**

- A container that runs continuously and accepts HTTP/HTTPS traffic (APIs, backends, dashboards).
- WebSocket support (requires the Application Load Balancer option).
- Non-HTTP protocols like TCP or TLS (requires the Network Load Balancer option).
- Full control over the runtime: install system packages, run background threads, keep in-memory state.

**Consider alternatives when:**

- Your workload is event-driven or has long idle periods — a [Lambda function](/resources/compute/lambda-function) is
  cheaper because you pay only for invocations.
- You need a container that is not publicly reachable — use a [private service](/resources/compute/private-service).
- You need a container without any networking (queue consumers, background workers) — use a
  [worker service](/resources/compute/worker-service).
- You need to run a short-lived batch computation — use a [batch job](/resources/compute/batch-job).

**Cost baseline:** The minimum Fargate configuration (0.25 vCPU, 512 MB) costs roughly **$8–9/month** before load
balancer and transfer costs. A web service cannot scale to zero — at least one container instance is always running.

## Basic Example

A minimal web service that runs a Node.js app on Fargate and is reachable over HTTPS through the default API Gateway:

```yaml
resources:
  myApi:
    type: web-service
    properties:
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: ./src/index.ts
      resources:
        cpu: 0.25
        memory: 512
```

Your application must listen on the `PORT` environment variable (Stacktape injects `PORT=3000` automatically) and bind
to `0.0.0.0`. A simple Express example:

```ts
import express from 'express';

const app = express();
app.get('/', (req, res) => res.json({ status: 'ok' }));
app.listen(process.env.PORT, '0.0.0.0');
```

After deploying, Stacktape prints the public URL. You can also retrieve it with
`$ResourceParam('myApi', 'url')`.

## Key Configuration Areas

### Packaging

The `packaging` property determines how your container image is built. Five options are available:

| Type | When to use |
| --- | --- |
| `stacktape-image-buildpack` | Zero-config for JS/TS, Python, Java, Go, Ruby, PHP, .NET. Bundles code automatically. |
| `custom-dockerfile` | You need full control over the image (system deps, multi-stage builds, etc.). |
| `nixpacks` | Auto-detect language and build — similar to Heroku buildpacks. |
| `external-buildpack` | Use a specific Cloud Native Buildpack (buildpacks.io). |
| `prebuilt-image` | Use an existing image from Docker Hub, ECR, or any registry. |

**Stacktape buildpack** (most common):

```yaml
packaging:
  type: stacktape-image-buildpack
  properties:
    entryfilePath: ./src/index.ts
```

For Python frameworks like FastAPI, use the `entryfilePath` format `./app/main.py:app` and configure the runner:

```yaml
packaging:
  type: stacktape-image-buildpack
  properties:
    entryfilePath: ./app/main.py:app
    languageSpecificConfig:
      packageManagerFile: pyproject.toml
      runAppAs: ASGI
```

**Custom Dockerfile:**

```yaml
packaging:
  type: custom-dockerfile
  properties:
    buildContextPath: ./
    dockerfilePath: Dockerfile
```

**Prebuilt image** (e.g., running an open-source tool):

```yaml
packaging:
  type: prebuilt-image
  properties:
    image: louislam/uptime-kuma:1
```

For private registries, provide credentials via `repositoryCredentialsSecretArn` (a Secrets Manager ARN containing
`{"username":"...","password":"..."}`).

For a detailed comparison and language-specific configuration, see [Packaging](/packaging).

### Compute Resources

The `resources` property controls CPU, memory, and compute engine. Two engines are available:

**Fargate (default)** — serverless containers. Pick a CPU/memory combination and AWS manages the infrastructure:

```yaml
resources:
  cpu: 0.5
  memory: 1024
```

Valid Fargate CPU values: `0.25`, `0.5`, `1`, `2`, `4`, `8`, `16` vCPU. Memory must be compatible with the chosen CPU
tier:

| vCPU | Memory range (MB) |
| --- | --- |
| 0.25 | 512–2048 |
| 0.5 | 1024–4096 |
| 1 | 2048–8192 |
| 2 | 4096–16384 |
| 4 | 8192–30720 |
| 8 | 16384–61440 |
| 16 | 32768–122880 |

Use `architecture: arm64` for ~20% cost savings on Graviton processors (default is `x86_64`):

```yaml
resources:
  cpu: 1
  memory: 2048
  architecture: arm64
```

**EC2** — specify instance types for more control, GPU access, or potentially lower per-hour cost:

```yaml
resources:
  instanceTypes:
    - t3.medium
```

When using EC2, instances auto-scale and are refreshed weekly for patching. Use `enableWarmPool: true` to keep
pre-initialized instances ready for faster scale-out (works with a single instance type only).

### Scaling

By default, a web service runs a single container instance. Configure `scaling` to add auto-scaling:

```yaml
scaling:
  minInstances: 2
  maxInstances: 10
  scalingPolicy:
    keepAvgCpuUtilizationUnder: 70
```

Traffic is automatically distributed across all running instances. Scaling decisions are based on average CPU and/or
memory utilization across instances.

| Property | Default | Description |
| --- | --- | --- |
| `minInstances` | `1` | Minimum running containers. Cannot be 0. |
| `maxInstances` | `1` | Maximum running containers. |
| `scalingPolicy.keepAvgCpuUtilizationUnder` | `80` | Target CPU %. Scales out above, in below. |
| `scalingPolicy.keepAvgMemoryUtilizationUnder` | `80` | Target memory %. Scales out above, in below. |

### Load Balancing

The `loadBalancing` property controls how traffic reaches your containers. This is the most important architectural
choice for a web service — it affects pricing, features, and protocol support.

#### HTTP API Gateway (default)

Pay-per-request pricing (~$1 per million requests). Best for most apps, especially those with variable or low traffic.

```yaml
# This is the default — you can omit loadBalancing entirely
loadBalancing:
  type: http-api-gateway
```

Supports CORS (via the `cors` property) and CDN. Does **not** support WebSockets, gradual deployments, or firewalls.

#### Application Load Balancer

Flat ~$18/month base + usage charges. More cost-effective above roughly 500k requests/day. AWS Free Tier eligible (first
12 months).

```yaml
loadBalancing:
  type: application-load-balancer
  properties:
    healthcheckPath: /health
    healthcheckInterval: 10
```

**Required** when you need:

- Gradual deployments (`deployment` property)
- Web Application Firewall (`useFirewall`)
- WebSocket support

| Property | Default | Description |
| --- | --- | --- |
| `healthcheckPath` | `/` | Path the ALB pings to check container health. |
| `healthcheckInterval` | `5` | Seconds between health checks. |
| `healthcheckTimeout` | `4` | Seconds before a health check is considered failed. |

#### Network Load Balancer

For non-HTTP protocols (TCP, TLS) like MQTT brokers, game servers, or custom binary protocols. Requires explicit `ports`
configuration.

```yaml
loadBalancing:
  type: network-load-balancer
  properties:
    ports:
      - port: 8883
        protocol: TLS
        containerPort: 1883
```

When using the NLB, Stacktape does **not** inject the `PORT` environment variable — your app must know which port(s) to
listen on.

Does **not** support CDN, firewall, CORS, or gradual deployments.

| Property | Default | Description |
| --- | --- | --- |
| `ports[].port` | — | Public port on the load balancer (required). |
| `ports[].protocol` | `TLS` | `TLS` (encrypted) or `TCP` (raw). |
| `ports[].containerPort` | Same as `port` | Port on the container receiving traffic. |
| `healthCheckProtocol` | `TCP` | `TCP` (port check) or `HTTP` (path check). |
| `healthcheckPath` | `/` | Only used when `healthCheckProtocol` is `HTTP`. |
| `healthcheckInterval` | `5` | Seconds between health checks (5–300). |
| `healthcheckTimeout` | `4` | Seconds before a check fails (2–120). |
| `healthCheckPort` | Traffic port | Override the port used for health checks. |

### CORS

Configure Cross-Origin Resource Sharing when your API is called from browser JavaScript on a different domain. Only
works with `http-api-gateway` load balancing (the default).

```yaml
cors:
  enabled: true
  allowedOrigins:
    - https://myapp.com
  allowedMethods:
    - GET
    - POST
  allowCredentials: true
```

With `enabled: true` and no other options, permissive defaults apply (`*` origins, common headers).

CORS settings configured here override any CORS headers your application sends.

| Property | Default | Description |
| --- | --- | --- |
| `allowedOrigins` | `["*"]` | Origins allowed to make requests. |
| `allowedMethods` | Common methods | HTTP methods to allow. |
| `allowCredentials` | — | Whether to allow cookies and auth headers. |
| `exposedResponseHeaders` | — | Headers accessible to browser JavaScript. |
| `maxAge` | — | How long browsers cache preflight responses (seconds). |

### Custom Domains

Attach your own domain to the service. Stacktape automatically creates DNS records and provisions free TLS certificates
via AWS Certificate Manager.

```yaml
customDomains:
  - domainName: api.example.com
```

**Prerequisites:** Your domain must have a Route 53 hosted zone in your AWS account with your registrar's nameservers
pointing to it.

Set `disableDnsRecordCreation: true` if you manage DNS externally (e.g., Cloudflare). You will still get a TLS
certificate, but you must point your DNS to the service endpoint yourself.

Use `customCertificateArn` only if you have specific certificate requirements (e.g., EV/OV certificates). Stacktape
provisions and auto-renews free certificates by default.

### CDN

Put CloudFront in front of your service for caching and lower latency worldwide. Works with HTTP API Gateway and
Application Load Balancer, but **not** with Network Load Balancer.

```yaml
cdn:
  enabled: true
  cachingOptions:
    defaultTTL: 60
```

The CDN has no monthly base cost — you pay per request and per GB transferred. See [CDN](/resources/networking/cdn) for
the full configuration reference.

### Connecting to Other Resources

Use `connectTo` to grant your web service access to other resources in your stack. Stacktape automatically handles IAM
permissions, security group rules, and injects environment variables with connection details.

```yaml
resources:
  myApi:
    type: web-service
    properties:
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: ./src/index.ts
      resources:
        cpu: 0.25
        memory: 512
      connectTo:
        - myDatabase
        - myBucket
      environment:
        - name: CUSTOM_VAR
          value: hello

  myDatabase:
    type: relational-database
    properties:
      engine:
        type: postgres
        properties:
          version: '16'
      resources:
        cpu: 0.25
        memory: 0.5

  myBucket:
    type: bucket
```

With the config above, your web service container receives environment variables like
`STP_MY_DATABASE_CONNECTION_STRING`, `STP_MY_BUCKET_NAME`, and `STP_MY_BUCKET_ARN` automatically.

For fine-grained AWS permissions not covered by `connectTo`, use `iamRoleStatements`:

```yaml
iamRoleStatements:
  - Resource: ['*']
    Action:
      - 'rekognition:DetectLabels'
```

### Gradual Deployments

Shift traffic to new container versions gradually instead of all at once. **Requires** `application-load-balancer` load
balancing.

```yaml
loadBalancing:
  type: application-load-balancer
deployment:
  strategy: Canary10Percent5Minutes
```

Available strategies:

| Strategy | Behavior |
| --- | --- |
| `Canary10Percent5Minutes` | 10% of traffic to new version, then 100% after 5 minutes. |
| `Canary10Percent15Minutes` | 10% of traffic to new version, then 100% after 15 minutes. |
| `Linear10PercentEvery1Minutes` | Shift 10% more traffic every minute. |
| `Linear10PercentEvery3Minutes` | Shift 10% more traffic every 3 minutes. |
| `AllAtOnce` | Instant switch to the new version. |

You can run validation functions during the deployment:

- `beforeAllowTrafficFunction` — Lambda function name to invoke before traffic shifts to the new version. If it fails,
  the deployment rolls back.
- `afterTrafficShiftFunction` — Lambda function name to invoke after all traffic has shifted. Useful for post-deploy
  smoke tests.

### Firewall

Protect your service from common web exploits (SQL injection, XSS, etc.) using a Web Application Firewall. **Requires**
`application-load-balancer` load balancing.

```yaml
loadBalancing:
  type: application-load-balancer
useFirewall: myFirewall
```

The `myFirewall` resource must be defined as a `web-app-firewall` in your config. See
[Web Application Firewall](/resources/security/web-app-firewall).

### Alarms

Monitor your web service and get notified when something goes wrong. Alarms are defined per-resource and can also be
combined with global alarms configured in the Stacktape Console.

The alarm type depends on your load balancing choice:

- **HTTP API Gateway**: error rate and latency alarms.
- **Application Load Balancer**: error rate, unhealthy targets, and custom CloudWatch metric alarms.

```yaml
resources:
  myApi:
    type: web-service
    properties:
      packaging:
        type: stacktape-image-buildpack
        properties:
          entryfilePath: ./src/index.ts
      resources:
        cpu: 0.25
        memory: 512
      loadBalancing:
        type: application-load-balancer
      alarms:
        - trigger:
            type: ApplicationLoadBalancer.errorRate
            properties:
              thresholdPercent: 5
          evaluation:
            period: 60
            evaluationPeriods: 3
```

Use `disabledGlobalAlarms` to exclude specific global alarms (set in the Console) from this particular service.

### Side Containers

Run helper containers alongside your main container:

```yaml
sideContainers:
  - name: migrate
    containerType: run-on-init
    packaging:
      type: stacktape-image-buildpack
      properties:
        entryfilePath: ./migrate.ts
  - name: datadog-agent
    containerType: always-running
    packaging:
      type: prebuilt-image
      properties:
        image: datadog/agent:latest
```

- **`run-on-init`**: Runs to completion (must exit 0) before the main container starts. Use for database migrations or
  setup tasks.
- **`always-running`**: Runs for the entire container lifecycle. Can communicate with the main container over
  `localhost`. If it crashes, the whole task is replaced.

### Health Checks

An internal health check auto-replaces containers that become unhealthy:

```yaml
internalHealthCheck:
  healthCheckCommand: ['CMD-SHELL', 'curl -f http://localhost:3000/health || exit 1']
  intervalSeconds: 30
  retries: 3
  startPeriodSeconds: 60
```

This is separate from the load balancer health check. The internal check runs inside the container via ECS, while the
load balancer check hits your service over the network.

| Property | Default | Range |
| --- | --- | --- |
| `intervalSeconds` | `30` | 5–300 |
| `timeoutSeconds` | `5` | 2–60 |
| `retries` | `3` | 1–10 |
| `startPeriodSeconds` | — | 0–300 |

### Logging

Container output (`stdout`/`stderr`) is automatically sent to CloudWatch Logs and retained for 90 days by default. View
logs with `stacktape logs` or in the Stacktape Console.

```yaml
logging:
  retentionDays: 30
```

Set `disabled: true` to turn off CloudWatch logging entirely.

### Volume Mounts

Mount persistent EFS filesystems that survive container restarts and are shared across instances:

```yaml
volumeMounts:
  - type: efs
    properties:
      efsFilesystemName: myStorage
      mountPath: /data
```

The EFS filesystem must be defined as an `efs-filesystem` resource in your config. Data is encrypted in transit. Use
`rootDirectory` to restrict access to a subdirectory within the filesystem.

### Private Subnets with NAT

Deploy in private subnets so all outbound traffic routes through a NAT Gateway. This gives your containers a static IP
for whitelisting with external services (payment gateways, partner APIs).

```yaml
usePrivateSubnetsWithNAT: true
```

Adds NAT Gateway cost (~$32/month per AZ + data processing fees). Configure the number of NAT Gateways in
`stackConfig.vpc.nat.availabilityZones` (default: 2).

### Remote Sessions

Enable interactive shell access to running containers for debugging:

```yaml
enableRemoteSessions: true
```

Then use `stacktape container:session` to open a shell inside the container. Adds a small SSM agent sidecar.

## How It Works

When you deploy a web service, Stacktape creates the following AWS resources:

1. **ECS Service** running on either Fargate (serverless) or EC2 instances.
2. **Load balancer** — one of HTTP API Gateway, Application Load Balancer, or Network Load Balancer — that terminates
   TLS and routes traffic to your containers.
3. **ECR repository** to store container images (unless using a prebuilt image).
4. **CloudWatch Log Group** for container logs.
5. **IAM roles** with least-privilege permissions based on your `connectTo` and `iamRoleStatements`.
6. Optionally: **CloudFront distribution** (CDN), **Route 53 DNS records**, **ACM certificates**, **Auto Scaling
   policies**, and **CodeDeploy** (for gradual deployments).

### Request flow

For the default HTTP API Gateway setup:

```
Client → HTTPS → API Gateway → Container (port 3000)
```

For Application Load Balancer:

```
Client → HTTPS (port 443) → ALB → Container (port 3000)
```

For Network Load Balancer:

```
Client → TLS/TCP (custom port) → NLB → Container (custom port)
```

### Injected environment variables

Stacktape automatically injects these environment variables into your main container:

| Variable | Value | Condition |
| --- | --- | --- |
| `PORT` | `3000` | Not injected when using Network Load Balancer. |
| `HOST` | `0.0.0.0` | Always injected. |
| `DEPLOYMENT_TEST_PORT` | `8080` | Only when `deployment` is configured. |

Variables from `connectTo` are also injected automatically (e.g., `STP_MY_DATABASE_CONNECTION_STRING`).

### Zero-downtime deployments

By default, ECS performs a rolling update: new containers are started and pass health checks before old containers are
drained. With `deployment` configured (ALB only), traffic is shifted gradually using AWS CodeDeploy.

### Shutdown behavior

When a container is being replaced (during a deployment or scale-in), it receives `SIGTERM` first. Your application
should handle this signal to close connections gracefully. After `stopTimeout` seconds (default: 2, max: 120),
the container is force-killed with `SIGKILL`.

## Referenceable Parameters

Use `$ResourceParam('myApi', '<param>')` to reference these values in your config:

| Parameter | Description |
| --- | --- |
| `url` | The full HTTPS URL of the service. |
| `domain` | The domain name (without protocol). |
| `customDomains` | List of custom domain names. |
| `customDomainUrl` | URL using the first custom domain. |
| `customDomainUrls` | URLs for all custom domains. |
| `canonicalDomain` | Primary domain (custom if set, otherwise default). |
| `logGroupArn` | ARN of the CloudWatch Log Group. |
| `cdnDomain` | CloudFront domain (when CDN is enabled). |
| `cdnUrl` | CloudFront URL (when CDN is enabled). |
| `cdnCustomDomains` | Custom domains on the CDN. |
| `cdnCustomDomainUrls` | URLs for CDN custom domains. |
| `cdnCanonicalDomain` | Primary CDN domain. |
| `cdnCanonicalUrl` | Primary CDN URL. |
