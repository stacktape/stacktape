# Resource Limit Guardrails

Resource limit guardrails are preventive policies that cap the size and count of compute resources in your stack. They define maximum memory, CPU, timeout, and resource count limits that Stacktape enforces before any infrastructure changes occur — blocking non-compliant deployments with a clear error identifying the offending resource.

## How resource limit guardrails work

Stacktape evaluates guardrails during [`deploy`](/cli/deploy), [`codebuild:deploy`](/cli/codebuild-deploy), [`preview-changes`](/cli/preview-changes), and [`delete`](/cli/delete) commands. Guardrails are fetched from the Stacktape Console as part of your organization's global configuration. If any guardrail is violated, the command fails with a `GUARDRAIL` error before any infrastructure changes are made — no resources are created, updated, or torn down.

Guardrails are not defined in your `stacktape.ts` configuration file. They are managed at the organization level through the [Stacktape Console](/stacktape-console/console-overview), which means they apply across projects and stages without requiring individual project configuration. This separation ensures developers cannot bypass limits by editing their local config.

<Info>Guardrails are preventive — they block non-compliant configurations before deployment starts. For reactive monitoring of running resources (memory usage, CPU spikes, error rates), use [alarms](/observability/alarms) instead. The two complement each other: guardrails prevent misconfigurations, alarms detect runtime problems.</Info>

## When to use

Resource limit guardrails are most valuable when multiple developers or teams deploy to shared AWS accounts and you need to prevent cost surprises or capacity mistakes:

- **Cost control** — Cap Lambda memory and container CPU/memory to prevent developers from accidentally provisioning expensive resources. A single 10 GB Lambda or 16 vCPU container can dominate your AWS bill.
- **Staging environment discipline** — Restrict resource sizes across development and staging stages to keep non-production costs low. Combine with [deployment guardrails](/guardrails/deployment) for stage-level restrictions.
- **Stack sprawl prevention** — Limit the number of resources per stack to keep infrastructure manageable and avoid hitting AWS CloudFormation account-level limits.
- **Architectural enforcement** — Block resource types that are expensive, not yet approved, or unnecessary for your workloads (e.g., block `open-search-domain` or `redis-cluster` if your team only uses serverless databases).

## When NOT to use

- **Solo developers or small teams** — If everyone deploying understands the cost implications, guardrails add friction without much benefit. Start without them and add guardrails when you onboard new team members or get your first surprise bill.
- **Prototyping stages** — Strict limits can slow experimentation. Consider applying guardrails only once your team moves past early prototyping.
- **As a replacement for alarms** — Guardrails only check configuration at deploy time. They cannot detect runtime cost spikes, memory leaks, or throttling. Use [alarms](/observability/alarms) for runtime monitoring and [budgets](/managing-costs/budgets) for spend alerts.

## Function memory limit

The `function-memory-limit` guardrail sets the maximum memory (in MB) allowed for any [Lambda function](/resources/compute/lambda-function) or [edge function](/resources/compute/edge-function) in the stack. If a function's memory exceeds the limit, the deployment is blocked.

| Property | Type | Description |
|---|---|---|
| `maxMemoryMB` | `number` | Maximum memory in MB allowed for any function |

When a function does not explicitly set a `memory` value, Stacktape evaluates it against the guardrail using the Lambda function default of 1024 MB. This means a guardrail with `maxMemoryMB` set to 512 blocks functions that rely on the default — they must explicitly set `memory: 512` or lower to pass.

**When to set this:** Most Lambda workloads run well between 256 MB and 2048 MB. Setting a ceiling of 2048 MB or 3008 MB prevents developers from accidentally choosing 10 GB when they meant 1 GB, while still leaving room for memory-intensive tasks like image processing. AWS Lambda memory scales linearly with CPU — a function with 1,769 MB gets 1 full vCPU.

**Example violation message:**

```
Function "processImages" has memory 4096MB which exceeds the limit of 2048MB.
```

## Function timeout limit

The `function-timeout-limit` guardrail sets the maximum timeout (in seconds) for any [Lambda function](/resources/compute/lambda-function) or [edge function](/resources/compute/edge-function). Functions configured with a timeout above the limit are blocked at deploy time.

| Property | Type | Description |
|---|---|---|
| `maxTimeoutSeconds` | `number` | Maximum timeout in seconds allowed for any function |

When a function does not explicitly set a `timeout` value, Stacktape evaluates it against the guardrail using the Lambda function default of 20 seconds. A guardrail with `maxTimeoutSeconds` set to 10 blocks functions using the default timeout — they must explicitly configure a lower value.

**When to set this:** Long timeouts often indicate work that belongs in a [container workload](/resources/compute/web-service) or [batch job](/resources/compute/batch-job) rather than a Lambda function. A 30-second or 60-second limit covers most API and event-processing use cases. If you need functions that run for minutes, consider whether they should be Lambda functions at all — AWS Lambda supports up to 900 seconds, but per-invocation cost grows linearly with duration.

**Example violation message:**

```
Function "dataSync" has timeout 120s which exceeds the limit of 30s.
```

## Container resource limit

The `container-resource-limit` guardrail caps the CPU (in vCPU) and memory (in MB) for container workloads — [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload), and [batch jobs](/resources/compute/batch-job). You can set either or both limits independently.

| Property | Type | Description |
|---|---|---|
| `maxCpu` | `number` | Maximum vCPU allowed per container workload |
| `maxMemoryMB` | `number` | Maximum memory in MB allowed per container workload |

The guardrail checks the `resources.cpu` and `resources.memory` values configured on each container workload. If a workload does not explicitly configure CPU or memory, the guardrail does not block it — only explicitly set values are compared against the limit.

**When to set this:** Container costs scale directly with CPU and memory allocations. For AWS Fargate workloads, CPU and memory determine per-second pricing. Capping at 4 vCPU and 8192 MB covers most web APIs and background workers. Reserve higher limits for compute-heavy workloads like video encoding or ML inference — and require those teams to explicitly request a guardrail adjustment.

**Example violation messages:**

```
Container workload "api" has 8 vCPU which exceeds the limit of 4 vCPU.
```

```
Container workload "api" has 16384MB memory which exceeds the limit of 8192MB.
```

## Resource type restriction

The `resource-type-restriction` guardrail blocks specific Stacktape resource types entirely. Any stack containing a blocked resource type fails to deploy.

| Property | Type | Description |
|---|---|---|
| `blockedResourceTypes` | `string[]` | List of Stacktape resource types that cannot be deployed |

Resource types use their Stacktape identifier strings — for example, `open-search-domain` blocks [OpenSearch domains](/resources/databases/opensearch), `redis-cluster` blocks [Redis clusters](/resources/databases/redis), and `batch-job` blocks [batch jobs](/resources/compute/batch-job).

**When to set this:** Use this guardrail to enforce architectural decisions across your organization. Common scenarios:

- **Block expensive managed services** — Prevent teams from adding OpenSearch or Redis clusters when a simpler solution would suffice.
- **Enforce serverless-only architectures** — Block container-based resource types like `web-service`, `private-service`, `worker-service`, and `multi-container-workload` to ensure teams use Lambda functions.
- **Limit to approved resource types** — In regulated environments, block resource types that haven't been reviewed by your security or compliance team.

**Example violation message:**

```
Resource "search" uses type "open-search-domain" which is blocked. Blocked resource types: open-search-domain, redis-cluster.
```

## Resource count limit

The `resource-count-limit` guardrail caps the total number of Stacktape resources in a single stack. If the stack's resource count exceeds the limit, the deploy fails.

| Property | Type | Description |
|---|---|---|
| `maxResources` | `number` | Maximum number of resources allowed per stack |

The resource count is determined by `configManager.allConfigResources.length` — the total number of named resources defined in the stack configuration.

**When to set this:** Large stacks are harder to maintain, slower to deploy, and more likely to hit AWS CloudFormation limits. Each Stacktape resource generates multiple underlying AWS CloudFormation resources (IAM roles, security groups, log groups, etc.), so a stack with 30 Stacktape resources can produce 200+ CloudFormation resources. A limit of 20–30 resources per stack encourages teams to split large applications into separate projects.

**Example violation message:**

```
Stack has 35 resources which exceeds the limit of 30.
```

## Combining guardrails

Guardrails are independent — you can enable any combination. A common setup might include:

- **Function memory limit** of 2048 MB and **timeout limit** of 60 seconds to keep Lambda costs predictable
- **Container resource limit** of 4 vCPU and 8192 MB to prevent oversized Fargate tasks
- **Resource count limit** of 30 to encourage modular stacks
- **Resource type restriction** blocking expensive managed services

Combine resource limit guardrails with [deployment guardrails](/guardrails/deployment) (stage and region restrictions), [security guardrails](/guardrails/security-and-data-protection), and [database guardrails](/guardrails/databases) for comprehensive organizational governance. See the [guardrails overview](/guardrails/overview) for the full list of available guardrail types.

## FAQ

### How are resource limit guardrails configured?

Resource limit guardrails are managed through the [Stacktape Console](/stacktape-console/console-overview) as part of your organization's global configuration. They are not defined in your `stacktape.ts` file. Stacktape fetches the active guardrails from the Console at the beginning of each deploy, delete, preview-changes, or codebuild:deploy command. See the [guardrails overview](/guardrails/overview) for setup instructions.

### What's the difference between guardrails and alarms?

Guardrails are preventive — they block non-compliant deployments before any infrastructure changes occur. [Alarms](/observability/alarms) are reactive — they monitor running resources and alert you when metrics exceed thresholds. A function might pass guardrail validation with 1024 MB memory but still trigger an alarm if it consistently runs out of memory at runtime. Use both together for complete coverage.

### Do function guardrails check functions that use the default memory or timeout?

Yes. When a Lambda function does not explicitly set `memory`, the guardrail evaluates it using the default of 1024 MB. Similarly, functions without an explicit `timeout` are evaluated using the default of 20 seconds. If your guardrail sets a limit below these defaults, affected functions must explicitly configure a value within the allowed range.

### Are edge functions checked by function guardrails?

Yes. Both `function-memory-limit` and `function-timeout-limit` guardrails evaluate [Lambda functions](/resources/compute/lambda-function) and [edge functions](/resources/compute/edge-function) together.

### What happens if a container workload doesn't set CPU or memory?

The `container-resource-limit` guardrail only checks explicitly configured `resources.cpu` and `resources.memory` values. If a container workload omits these properties, the guardrail does not block it. To ensure all workloads are sized explicitly, combine guardrails with team conventions or code review processes that require explicit resource configuration.

### How much does a Lambda function cost per MB of memory?

AWS Lambda pricing is based on the number of requests and duration billed in GB-seconds. More memory also allocates proportionally more CPU — at 1,769 MB a function gets 1 full vCPU. Doubling memory doubles the per-millisecond cost but often halves execution time for CPU-bound workloads, making the total cost similar. Use the `function-memory-limit` guardrail to prevent extreme allocations (e.g., 10 GB) that are rarely cost-effective for typical workloads.

### How does the resource count limit relate to AWS CloudFormation limits?

AWS CloudFormation has a default limit of 500 resources per stack, but each Stacktape resource generates multiple CloudFormation resources (IAM roles, security groups, log groups, target groups, etc.). A stack with 30 Stacktape resources can produce 200+ CloudFormation resources. Setting a `resource-count-limit` guardrail keeps stacks well within CloudFormation limits and encourages modular architecture.

### When should I use resource type restriction vs size limits?

Use `resource-type-restriction` when an entire category of resource is off-limits — for example, blocking all container workloads to enforce a serverless architecture. Use `function-memory-limit`, `function-timeout-limit`, or `container-resource-limit` when the resource type is allowed but you want to cap its size. The two work well together: allow `web-service` as a type but cap it at 4 vCPU and 8192 MB memory.

### Do guardrails run during `stacktape delete`?

Yes. Guardrails are evaluated during [`delete`](/cli/delete) commands. For delete operations, resource-scoped guardrails (function limits, container limits, resource count, resource type restriction) only apply when a project configuration file is available. Stage, region, and command restrictions always apply regardless of whether a config file is present.

### Can I set different resource limits for different stages?

The guardrails source does not define per-stage exception mechanisms within a single guardrail. All configured guardrails apply uniformly. If you need different limits for production vs development, manage separate guardrail sets per environment through your organization's policy in the [Stacktape Console](/stacktape-console/console-overview). Combine with [deployment guardrails](/guardrails/deployment) to control which stages and regions are available.
