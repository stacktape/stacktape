# debug:aws-sdk

The `stacktape debug:aws-sdk` command executes read-only AWS SDK v3 commands against resources in your AWS account. Use it to inspect Lambda functions, scan DynamoDB tables, list S3 objects, query CloudWatch logs, or interrogate any supported AWS service without leaving your terminal.

## Usage

```bash
stacktape debug:aws-sdk --region eu-west-1 --service lambda --command ListFunctions
```

The command accepts three required flags: `--service` (the AWS service to call), `--command` (the SDK operation name), and `--region`. An optional `--input` flag passes JSON parameters to the operation.

## Read-only enforcement

Only operations whose names start with `List`, `Get`, `Describe`, `Head`, or `Batch` are allowed. Attempting a mutating operation (such as `PutItem` or `DeleteFunction`) returns an error. This makes the command safe for debugging production stacks without risk of accidental writes.

## Supported services

The command supports 35+ AWS services. Service names are case-insensitive. Some services have aliases (e.g. `stepfunctions` and `sfn` both target Step Functions).

| Service | Example commands |
|---------|----------------|
| `lambda` | ListFunctions, GetFunction, GetFunctionConfiguration |
| `dynamodb` | ListTables, DescribeTable, Scan, GetItem |
| `s3` | ListBuckets, ListObjectsV2, GetObject, HeadObject |
| `logs` | DescribeLogGroups, FilterLogEvents, GetLogEvents |
| `ecs` | ListClusters, ListServices, DescribeTasks |
| `cloudformation` | DescribeStacks, DescribeStackResources, ListStacks |
| `cloudwatch` | GetMetricData, ListMetrics, DescribeAlarms |
| `rds` | DescribeDBInstances, DescribeDBClusters |
| `sqs` | ListQueues, GetQueueAttributes |
| `sns` | ListTopics, ListSubscriptions, GetTopicAttributes |
| `secretsmanager` | ListSecrets, GetSecretValue, DescribeSecret |
| `cognito` | ListUserPools, ListUsers, DescribeUserPool |
| `ec2` | DescribeInstances, DescribeSecurityGroups, DescribeVpcs |
| `kinesis` | ListStreams, DescribeStream |
| `apigatewayv2` | GetApis, GetRoutes, GetStages |

Additional supported services: `sfn` / `stepfunctions`, `eventbridge` / `events`, `ssm`, `sts`, `iam`, `ecr`, `ses`, `sesv2`, `cloudfront`, `route53`, `acm`, `opensearch`, `wafv2`, `elb` / `elbv2`, `autoscaling`, `budgets`, `costexplorer`, `codebuild`, `codedeploy`, `servicediscovery`, `xray`, `firehose`.

## Passing input parameters

Use the `--input` flag with a JSON string to pass parameters to the SDK command. The JSON structure matches the AWS SDK v3 input type for that command.

List objects in a specific S3 bucket:

```bash
stacktape debug:aws-sdk --region eu-west-1 --service s3 --command ListObjectsV2 --input '{"Bucket": "my-bucket", "MaxKeys": 20}'
```

Scan a DynamoDB table with a limit:

```bash
stacktape debug:aws-sdk --region eu-west-1 --service dynamodb --command Scan --input '{"TableName": "my-table", "Limit": 10}'
```

Filter CloudWatch log events:

```bash
stacktape debug:aws-sdk --region eu-west-1 --service logs --command FilterLogEvents --input '{"logGroupName": "/aws/lambda/my-func", "startTime": 1700000000000}'
```

## Output

In normal mode, the command prints the service, command name, and the JSON response with `$metadata` stripped for readability. In agent mode (`--agent`), the output is a single JSON object with `ok`, `service`, `command`, and `data` fields — suitable for programmatic consumption by AI coding assistants.

## Does not require a config file

Unlike most Stacktape commands, `debug:aws-sdk` does not require a Stacktape configuration file. You only need valid AWS credentials (via an AWS profile or a connected AWS account) and the `--region` flag.

## Examples

### List all Lambda functions in a region

```bash
stacktape debug:aws-sdk --region us-east-1 --service lambda --command ListFunctions
```

### Describe a specific ECS service

```bash
stacktape debug:aws-sdk --region eu-west-1 --service ecs --command DescribeServices --input '{"cluster": "my-cluster", "services": ["my-service"]}'
```

### Check CloudFormation stack status

```bash
stacktape debug:aws-sdk --region eu-west-1 --service cloudformation --command DescribeStacks --input '{"StackName": "my-project-prod"}'
```

### Get a secret value

```bash
stacktape debug:aws-sdk --region eu-west-1 --service secretsmanager --command GetSecretValue --input '{"SecretId": "my-secret"}'
```

### List CloudWatch alarms in ALARM state

```bash
stacktape debug:aws-sdk --region eu-west-1 --service cloudwatch --command DescribeAlarms --input '{"StateValue": "ALARM"}'
```

## FAQ

### When should I use debug:aws-sdk instead of the other debug commands?

Use `debug:aws-sdk` when the specialized debug commands (`debug:logs`, `debug:dynamodb`, `debug:redis`, `debug:sql`) don't cover the AWS service you need to inspect. It gives you direct access to any supported AWS SDK v3 operation, making it the most flexible debugging tool — but also the most verbose to use. For common tasks, prefer the specialized commands.

### Can I use debug:aws-sdk to modify resources?

No. The command enforces read-only access by only allowing operations that start with `List`, `Get`, `Describe`, `Head`, or `Batch`. Attempting a mutating operation like `PutItem` or `DeleteFunction` returns an error. This safety restriction prevents accidental damage when debugging production stacks.

### Do I need a Stacktape config file to use this command?

No. Unlike `deploy` or `dev`, this command does not require a `stacktape.ts` or `stacktape.yml` file. You only need valid AWS credentials and the `--region` flag. This makes it useful for quick inspection of any AWS account you have access to.

### How do I know which commands are available for a service?

If you omit the `--command` flag, the CLI returns example commands for the specified service. You can also consult the [AWS SDK v3 documentation](https://docs.aws.amazon.com/AWSJavaScriptSDK/v3/latest/) for the full list of operations per service. Command names match the SDK class names without the `Command` suffix (e.g. `ListFunctions` maps to `ListFunctionsCommand`).

### What credentials does the command use?

The command uses the same credential resolution as other Stacktape CLI commands: your configured AWS profile (`--profile`), a connected AWS account (`--awsAccount`), or the environment's default credentials. When a stage is specified and a debug agent role is available, the command uses that role for scoped access.

### How is the --input flag structured?

The `--input` flag accepts a JSON string whose structure matches the AWS SDK v3 input type for the chosen command. For example, `ListObjectsV2` expects `{"Bucket": "name"}`, and `DescribeStacks` expects `{"StackName": "name"}`. Refer to the AWS SDK v3 documentation for the exact schema of each command's input.

### Can I use this command in CI/CD pipelines?

Yes. Use `--agent` or `--outputFormat jsonl` to get machine-readable JSON output. The response includes an `ok` field, the service/command names, and the result data — making it straightforward to parse in scripts or pass to other tools.

### What happens if I get an AccessDeniedException?

The error means your AWS credentials lack the IAM permissions required for that specific operation. Either adjust the IAM policy attached to your profile/role, or use `--awsAccount` to switch to an account with broader permissions. The error message includes a hint explaining the likely cause.
