# issues:list

The `issues:list` command displays detected runtime issues (errors) across your deployed Lambda functions and container workloads. Use it to monitor error patterns, track occurrence counts, and triage problems without leaving your terminal.

## Usage

```bash
stacktape issues:list
```

List all open issues for a specific project and stage:

```bash
stacktape issues:list --projectName my-app --stage production --issueStatus OPEN
```

## What it does

Stacktape automatically detects and groups runtime errors from your deployed functions and containers into issues. Each issue represents a unique error pattern with metadata including the error message, error type, originating function name, project, stage, occurrence count, and the timestamp of the last occurrence.

The `issues:list` command queries these issues and displays them in a table. When no filters are provided, it returns up to 25 issues across all projects and stages. Use `--projectName`, `--stage`, and `--issueStatus` to narrow results.

In agent mode (`--agent`), the command outputs raw JSON instead of a formatted table, making it suitable for scripting and AI coding assistants.

## Important flags

**`--issueStatus`** — Filter by issue status. Accepted values: `OPEN`, `RESOLVED`, `IGNORED`. Without this flag, issues of all statuses are returned.

**`--projectName`** — Scope results to a single project.

**`--stage`** — Scope results to a single stage (e.g., `production`, `staging`, `dev-john`).

**`--limit`** — Maximum number of issues to return. Defaults to 25 when not specified.

**`--agent`** — Enables agent mode. Outputs structured JSON (NDJSON) for programmatic consumption by scripts or AI coding assistants.

## Examples

List only ignored issues:

```bash
stacktape issues:list --issueStatus IGNORED
```

Return up to 100 issues across all projects:

```bash
stacktape issues:list --limit 100
```

Get JSON output for scripting:

```bash
stacktape issues:list --agent --projectName my-app --stage production
```

## Output columns

In interactive mode, the command prints a table with these columns:

| Column    | Description                                      |
| --------- | ------------------------------------------------ |
| ID        | Unique issue identifier (used with other issue commands) |
| Status    | `OPEN`, `RESOLVED`, or `IGNORED`                 |
| Error     | Truncated error message (up to 60 characters)    |
| Type      | Error type or class name                         |
| Function  | Name of the function or container that produced the error |
| Project   | Project name                                     |
| Stage     | Stage name                                       |
| Count     | Total number of occurrences                      |
| Last seen | Timestamp of the most recent occurrence          |

## Related commands

- [`issues:resolve`](/cli/issues-resolve) — Mark an issue as resolved. It reopens automatically if the same error recurs.
- [`issues:ignore`](/cli/issues-ignore) — Mark an issue as ignored. Ignored issues do not reopen and do not generate alerts.
- [`issues:reopen`](/cli/issues-reopen) — Reopen a resolved or ignored issue, setting its status back to OPEN.
- [`debug:logs`](/cli/debug-logs) — Fetch logs from a deployed resource to investigate root causes of issues.
