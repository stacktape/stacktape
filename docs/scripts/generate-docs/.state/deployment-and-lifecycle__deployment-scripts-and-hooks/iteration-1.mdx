# Deployment Scripts and Hooks

Stacktape provides two complementary mechanisms for automating work around deployments: **hooks** trigger named scripts before or after CLI commands (running on your machine or CI), and **deployment scripts** execute as Lambda functions within AWS during the CloudFormation lifecycle. Together they cover use cases from pre-deploy builds to post-deploy database migrations to pre-delete cleanup.

## When to use hooks vs deployment scripts

Hooks and deployment scripts serve different execution contexts. Choose based on where the work needs to happen and what failure semantics you need.

| | Hooks + Scripts | Deployment Scripts |
|---|---|---|
| **Runs on** | Your machine or CI runner | AWS Lambda |
| **Triggered by** | CLI commands (deploy, delete, dev, bucket-sync) | CloudFormation lifecycle |
| **Max duration** | No limit (runs locally) | 15 minutes (Lambda max) |
| **Failure on deploy** | Non-blocking (deploy completes with warning) | Blocking (CloudFormation rolls back) |
| **Access to local files** | Yes | No (packaged code only) |
| **VPC access** | Via bastion tunneling | Via `joinDefaultVpc` |
| **Best for** | Builds, linting, local migrations, seed scripts | Migrations requiring rollback safety, cleanup that must run in AWS |

**Use hooks** when you need access to your local filesystem (builds, linting, generating files), when the script has no time limit, or when failure should not roll back infrastructure changes.

**Use deployment scripts** when the script must run inside AWS (needs VPC access without a bastion), when failure should trigger a full rollback to protect data integrity, or when you need the script to execute even if the deploy was triggered from the Console or CI without a local machine.

## Hooks

Stacktape hooks run named scripts at specific points in the CLI command lifecycle. Each hook references a script defined in the top-level `scripts` section of your configuration. Hooks execute sequentially in the order they are listed.

### Available hook points

| Hook | Runs | Failure behavior |
|------|------|-----------------|
| `beforeDeploy` | Before packaging and deploying | Blocks — deploy does not start |
| `afterDeploy` | After deployment completes | Non-blocking — deploy is marked complete with a warning |
| `beforeDelete` | Before stack deletion begins | Blocks — delete does not start |
| `afterDelete` | After stack deletion completes | Non-blocking |
| `beforeDev` | Before dev mode starts | Blocks — dev mode does not start |
| `afterDev` | After dev mode exits | Non-blocking |
| `beforeBucketSync` | Before syncing bucket contents | Blocks |
| `afterBucketSync` | After syncing bucket contents | Non-blocking |

<Info>The `beforeDelete` and `afterDelete` hooks only run when `--configPath` and `--stage` are provided to the delete command.</Info>

### Basic hook example

This configuration builds the frontend before every deploy and runs database migrations after every deploy:

      afterDeploy: [{ scriptName: 'migrate' }]

        executeCommand: 'npm run build'

      migrate: new LocalScript({
        executeCommand: 'npx prisma migrate deploy',
        connectTo: [database]

    },

### Hook failure behavior

**Before-hooks** (like `beforeDeploy`) are blocking — if the script exits with a non-zero code, the command aborts immediately. This protects against deploying broken builds.

**After-hooks** (like `afterDeploy`) during the deploy command are non-blocking — the deployment is already complete, so a hook failure produces a warning but does not roll back infrastructure. The CLI reports the number of failed hooks and their error messages.

<Warning>After-deploy hooks do NOT re-execute during rollbacks. If you need rollback-safe post-deploy logic, use a [deployment script](#deployment-scripts-resource) instead.</Warning>

## Scripts

Scripts are named, reusable commands or code files defined in the top-level `scripts` section. They can be run manually with [`stacktape script:run`](/cli/script-run) or attached to hooks. Stacktape supports three script types.

### Local scripts

A `LocalScript` runs on the machine executing the Stacktape command — your laptop or a CI runner. It can execute shell commands or script files (JavaScript, TypeScript, Python).

You must specify exactly one of: `executeCommand`, `executeCommands`, `executeScript`, or `executeScripts`.

<CodeBlock

    }

/>

**Key properties:**

- **`connectTo`** — List resource names to auto-inject connection environment variables (`STP_[RESOURCE_NAME]_[PARAM]`). For example, connecting to a resource named `database` injects `STP_DATABASE_CONNECTION_STRING`.
- **`environment`** — Additional environment variables. Supports [`$ResourceParam()`](/configuration/directives) and [`$Secret()`](/configuration/directives) directives.
- **`assumeRoleOfResource`** — Assume the IAM role of a deployed workload, granting the script the same AWS permissions. Stacktape injects temporary AWS credentials automatically.
- **`cwd`** — Working directory for execution. Defaults to the directory where the Stacktape command was run.
- **`pipeStdio`** — Whether to pipe script output to your terminal. Defaults to `true`.

### Running scripts manually

Run any named script with the [`stacktape script:run`](/cli/script-run) command:

```bash
stacktape script:run --scriptName migrate --stage production
```

You can also pass additional environment variables at runtime:

```bash
stacktape script:run --scriptName seed --stage dev --env SEED_COUNT=500
```

## Deployment scripts (resource)

A Stacktape deployment script is a resource (type `deployment-script`) that executes as a Lambda function during the CloudFormation deployment lifecycle. Unlike hooks which run on your machine, deployment scripts run inside AWS and participate in CloudFormation's rollback semantics.

### Triggers

- **`after:deploy`** — Runs after all resources are created or updated. If the script fails, CloudFormation rolls back the entire deployment. Use this for database migrations or data transformations where a failure means the new code can't function correctly.
- **`before:delete`** — Runs before stack resources are deleted. If the script fails, deletion continues anyway. Use this for cleanup tasks like exporting data or deregistering from external services.

### Basic deployment script

<CodeBlock

    connectTo: [database]
  });

    routes: [{ path: '/{proxy+}', method: '*', integration: new HttpApiIntegration({ function: api }) }]
  });

  return {
    resources: { database, migrate, api, gateway }
  };
});

The handler receives the `parameters` object as the Lambda event:

```typescript
// scripts/migrate.ts
import { Client } from 'pg';

export const handler = async (event: Record<string, any>) => {
  const client = new Client({
    connectionString: process.env.STP_DATABASE_CONNECTION_STRING
  });
  await client.connect();

  await client.query(`
    CREATE TABLE IF NOT EXISTS users (
      id SERIAL PRIMARY KEY,
      email VARCHAR(255) UNIQUE NOT NULL,
      created_at TIMESTAMP DEFAULT NOW()
    )
  `);

  await client.end();
  return { success: true, message: 'Migration complete' };
};
```

```typescript
// scripts/cleanup.ts
import { S3Client, DeleteObjectsCommand, ListObjectsV2Command } from '@aws-sdk/client-s3';

export const handler = async (event: { bucketName: string; stage: string }) => {
  const s3 = new S3Client({});
  console.info(`Cleaning up bucket ${event.bucketName} for stage ${event.stage}`);

  const listed = await s3.send(new ListObjectsV2Command({ Bucket: event.bucketName }));
  if (listed.Contents?.length) {
    await s3.send(new DeleteObjectsCommand({
      Bucket: event.bucketName,
      Delete: { Objects: listed.Contents.map(obj => ({ Key: obj.Key })) }
    }));
  }

  return { deleted: listed.Contents?.length || 0 };
};
```

### VPC access

Set `joinDefaultVpc: true` when the deployment script needs to reach VPC-only resources like private databases or Redis clusters. Be aware that enabling VPC access means the Lambda function loses direct internet access — if your script needs to call external APIs, configure a NAT gateway in your VPC.

### Running deployment scripts manually

Use [`stacktape deployment-script:run`](/cli/deployment-script-run) to re-run a deployment script outside of the deploy lifecycle. This is useful during development or when a migration needs to be re-applied.

```bash
stacktape deployment-script:run --resourceName migrate --stage production --region eu-west-1
```

This command hot-swaps the script's code (updates the Lambda function code) before invoking it, so you always run the latest version. However, it does not update environment variables or other configuration — use a full `stacktape deploy` for that.

### Deployment script constraints

- **Timeout:** Maximum 900 seconds (15 minutes). Default is 10 seconds — increase this for long-running migrations.
- **Memory:** 128–10,240 MB. CPU scales proportionally (1,769 MB = 1 full vCPU).
- **Storage:** Ephemeral `/tmp` storage, 512–10,240 MB. Default is 512 MB.
- **Runtime:** Auto-detected from your file extension. Supports Node.js, Python, Java, Go, Ruby, PHP, and .NET.
- **Packaging:** Uses [Lambda packaging](/packaging/overview) — either `StacktapeLambdaBuildpackPackaging` (zero-config bundling from source) or `CustomArtifactLambdaPackaging` (pre-built zip).

## FAQ

### When should I use a hook vs a deployment script for database migrations?

Use a **hook** when you want to run your existing local migration tooling (Prisma, Drizzle, Knex) and are comfortable with migrations being non-blocking — a failed hook won't roll back infrastructure. Use a **deployment script** when you need rollback-safe migrations: if the script fails, CloudFormation undoes all infrastructure changes from that deploy, keeping your schema and code in sync.

### Can I access a private database from a hook?

Yes. Use a `LocalScriptWithBastionTunneling` script type, which runs locally but tunnels database connections through a [bastion host](/resources/security/bastion-host). The injected `connectTo` environment variables automatically point to the tunneled endpoint. Alternatively, use a deployment script with `joinDefaultVpc: true` to run the script inside AWS with direct VPC access.

### What happens if an afterDeploy hook fails?

The deployment itself is already complete — infrastructure changes are committed. The CLI reports a warning with the number of failed hooks and their error messages. The deploy is marked as successful with caveats. If you need failure to trigger a rollback, use a deployment script with `trigger: 'after:deploy'` instead.

### Can I run multiple hooks in sequence?

Yes. List multiple entries in the hook array — they execute sequentially in order. If a `beforeDeploy` hook fails, subsequent hooks in the same array do not run, and the deploy command aborts.

### How long can a deployment script run?

The maximum timeout is 900 seconds (15 minutes), which is the AWS Lambda execution limit. The default is 10 seconds, so you should explicitly set a higher `timeout` for migrations or data processing. If your workload needs more than 15 minutes, use a hook with a local script (no time limit) or consider a [batch job](/resources/compute/batch-job) for very long-running tasks.

### Do deployment scripts re-run on rollback?

No. An `after:deploy` deployment script runs once after resources are created or updated. If the script itself fails, CloudFormation rolls back the infrastructure, but the script does not run again in reverse. Similarly, scripts do NOT re-run during a [`stacktape rollback`](/cli/rollback) command — the CLI warns about this when `after:deploy` hooks or scripts are present in the config.

### Can I pass secrets to scripts?

Yes. For hooks and local scripts, use `$Secret()` in the `environment` array or rely on `connectTo` which auto-injects connection strings (including credentials). For deployment scripts, use `$Secret()` in either `environment` or `parameters`. Secrets are resolved at deploy time. Prefer `environment` over `parameters` for sensitive values — parameters are logged in CloudFormation events.

### What's the difference between script:run and deployment-script:run?

[`stacktape script:run`](/cli/script-run) executes a named script (local or bastion) from your `scripts` section — it runs on your machine or a bastion host. [`stacktape deployment-script:run`](/cli/deployment-script-run) hot-deploys and invokes a deployment script Lambda function in AWS. The latter is useful for re-running migrations or testing your Lambda-based script without a full deploy.

### How do deployment scripts interact with CloudFormation?

Deployment scripts are implemented as CloudFormation custom resources. For `after:deploy`, CloudFormation invokes the script Lambda after all dependent resources are ready. If the invocation fails (non-zero exit or thrown error), CloudFormation marks the custom resource as failed and initiates a stack rollback. For `before:delete`, the script runs when CloudFormation processes the custom resource deletion — failure is logged but does not block the delete.

### Can I use deployment scripts with a pre-built package?

Yes. Instead of `StacktapeLambdaBuildpackPackaging`, use `CustomArtifactLambdaPackaging` with a `packagePath` pointing to your pre-built zip or directory. Stacktape uploads it directly without bundling. This is useful if you have a custom build process or need to include binary dependencies that the buildpack doesn't handle.
