# codebuild:deploy

The `codebuild:deploy` command deploys your stack to AWS using AWS CodeBuild instead of your local machine. It offloads the entire build and deployment process to a dedicated VM in your AWS account, streaming logs back to your terminal in real time. Use it when local builds are too slow, resource-constrained, or when you need a clean, reproducible build environment.

## How it works

The command validates your configuration and resolves all resources locally before starting CodeBuild. This means configuration errors surface immediately rather than after waiting for the remote environment to provision. Once the build starts, your terminal acts as a log viewer — you can close it without cancelling the deployment.

## When to use

Use `codebuild:deploy` instead of [`deploy`](/cli/deploy) when:

- **Your machine is resource-constrained** — Docker builds for large container workloads can exhaust local RAM or CPU.
- **You need faster uploads** — CodeBuild runs inside AWS, so pushing container images to ECR is significantly faster than from a local machine on a slow connection.
- **You want a clean build environment** — eliminates "works on my machine" issues caused by local Docker cache, OS differences, or toolchain versions.
- **CI/CD pipelines** — when your CI runner lacks Docker or has insufficient resources. See [build runners](/ci-cd-and-gitops/build-runners) for the full comparison.

For most development workflows where builds are fast and your internet connection is decent, the standard [`deploy`](/cli/deploy) command is simpler and avoids the overhead of provisioning a CodeBuild environment.

## Usage

```bash
stacktape codebuild:deploy --stage production --region eu-west-1
```

With a custom config path:

```bash
stacktape codebuild:deploy --stage staging --region us-east-1 --configPath ./infra/stacktape.ts
```

Skip the confirmation prompt (useful in CI):

```bash
stacktape codebuild:deploy --stage production --region eu-west-1 --autoConfirmOperation
```

## Important flags

| Flag | Alias | Purpose |
|------|-------|---------|
| `--stage` | `-s` | **Required.** Target stage (e.g. `production`, `dev-jane`). Max 12 characters. |
| `--region` | `-r` | **Required.** AWS region to deploy into. |
| `--configPath` | `-cp` | Path to your Stacktape config file. Defaults to `stacktape.ts` or `stacktape.yml` in the working directory. |
| `--autoConfirmOperation` | `-aco` | Skip the interactive confirmation prompt. Required for non-interactive environments. |
| `--hotSwap` | `-hs` | Attempt a faster deployment for code-only changes by updating functions directly. Recommended only for development stages. |
| `--disableAutoRollback` | `-dar` | Keep the stack in `UPDATE_FAILED` state on failure instead of rolling back. Useful for diagnosing deploy issues. |
| `--noCache` | `-nc` | Force a fresh build, ignoring cached artifacts. |
| `--disableDockerRemoteCache` | `-drc` | Disable ECR-based Docker layer caching. |
| `--disableLayerOptimization` | `-dlo` | Bundle all code into each Lambda function instead of extracting shared layers. |

<Warning>
The `--showSensitiveValues` flag is automatically disabled inside the CodeBuild environment to prevent credentials from appearing in streamed logs. Sensitive values are still printed locally in the final stack info output.
</Warning>

## Examples

Deploy a production stage to EU West:

```bash
stacktape codebuild:deploy --stage production --region eu-west-1
```

Deploy with a fresh build (no cache) for troubleshooting:

```bash
stacktape codebuild:deploy --stage staging --region us-east-1 --noCache
```

Deploy without auto-rollback to inspect failures:

```bash
stacktape codebuild:deploy --stage dev --region eu-west-1 --disableAutoRollback
```

Use hotswap for faster iteration on a development stage:

```bash
stacktape codebuild:deploy --stage dev-jane --region eu-west-1 --hotSwap
```

## FAQ

### When should I use codebuild:deploy instead of deploy?

Use `codebuild:deploy` when your local machine is slow to build Docker images, has limited RAM, or has a slow upload connection. The CodeBuild environment runs inside AWS so image pushes to ECR are fast. For quick iterations on small projects, the standard [`deploy`](/cli/deploy) is simpler.

### Does codebuild:deploy cost extra?

Yes. AWS CodeBuild charges per build-minute based on the compute type. A typical deployment uses a `general1.small` or `general1.medium` instance. Builds that complete in under 1 minute are rounded up to 1 minute. For most deployments, this adds a few cents per deploy.

### Can I cancel a running codebuild:deploy?

Closing your terminal does not cancel the CodeBuild build — it continues running in AWS. To stop it, use the AWS Console or AWS CLI to stop the CodeBuild build directly. The stack will remain in whatever state it was in when the build was stopped.

### What happens if the CodeBuild deployment fails?

By default, the stack automatically rolls back to the last known good state. If you pass `--disableAutoRollback`, the stack stays in `UPDATE_FAILED` state so you can inspect what went wrong. Use [`cf:rollback`](/cli/cf-rollback) to recover manually, or fix the issue and redeploy.

### Does codebuild:deploy support the same flags as deploy?

Yes. Both commands accept identical flags — `--hotSwap`, `--noCache`, `--disableAutoRollback`, `--disableDockerRemoteCache`, and all others. The only behavioral difference is where the build and deployment process runs (remote VM vs. local machine).

### How does the project get uploaded to CodeBuild?

Stacktape creates a git archive (zip) of your project and uploads it to an S3 bucket in your AWS account. If S3 Transfer Acceleration is available in your deployment region, it uses accelerated uploads for faster transfer. The CodeBuild environment then downloads and extracts this archive before running the deployment.

### Can I use codebuild:deploy in CI/CD pipelines?

Yes. Pass `--autoConfirmOperation` to skip the interactive confirmation prompt. You can also use `--agent` for machine-readable JSONL output. See [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) for pipeline integration patterns.
