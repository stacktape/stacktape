# Debugging Containers

Stacktape provides two CLI commands for debugging deployed container workloads: [`container:session`](/cli/container-session) for interactive shell access and [`debug:container-exec`](/cli/debug-container-exec) for running one-off commands. Both connect to running ECS containers through Stacktape's ECS Exec and SSM session helpers.

## Supported resources

Container debugging commands target deployed resources that have an ECS task definition and a running ECS service. Both commands look up the resource by name against deployed workloads in the stack, then locate the corresponding `AWS::ECS::Service` (or `Stacktape::ECSBlueGreenV1::Service` for blue/green deployments). The supported resource types are:

- [Web service](/resources/compute/web-service)
- [Private service](/resources/compute/private-service)
- [Worker service](/resources/compute/worker-service)
- [Multi-container workload](/resources/compute/multi-container-workload)

These commands require a deployed ECS service to connect to. They are not the Lambda debugging commands — use [Debugging Lambda Functions](/local-development/debugging-lambda-functions) for Lambda-based resources like [Lambda functions](/resources/compute/lambda-function) and [edge functions](/resources/compute/edge-function). [Batch jobs](/resources/compute/batch-job) are not supported because they run tasks to completion rather than maintaining a persistent ECS service.

Use `container:session` when you need to poke around interactively — inspect files, check running processes, test network connectivity. Use `debug:container-exec` when you need to automate a check, pipe output into another tool, or run a quick command without holding a terminal open.

## Interactive shell sessions

The [`container:session`](/cli/container-session) command opens an interactive shell inside a running container. It calls Stacktape's ECS Exec SSM session helper, dropping you into a terminal session similar to SSH-ing into the container — but without SSH keys, open ports, or any additional setup in the container image.

```bash
stacktape container:session --stage production --region eu-west-1 --resourceName apiServer
```

The `--command` flag is optional for `container:session`. If you need a specific shell or program, pass it explicitly — for example, to use `bash` instead of the container's default shell:

```bash
stacktape container:session --stage production --region eu-west-1 --resourceName apiServer --command "/bin/bash"
```

### Multiple running tasks

When your service scales to multiple tasks (instances), Stacktape prompts you to choose which task to connect to. The prompt displays each task's ARN suffix and start time so you can identify the one you need.

### Common use cases

- **Inspect the filesystem** — check if config files, environment variables, or mounted volumes are correct.
- **Test connectivity** — run `curl`, `ping`, or `nc` from inside the container to verify network paths to databases or external APIs.
- **Debug running processes** — use `ps`, `top`, or language-specific tools to inspect application state.
- **Run ad-hoc migrations** — execute one-off scripts that need access to the container's environment.

<Warning>
The container must have a shell available (`/bin/sh` or `/bin/bash`). Minimal images built `FROM scratch` or distroless images without a shell will not work with this command.
</Warning>

## Running one-off commands

The [`debug:container-exec`](/cli/debug-container-exec) command executes a single command inside a running container and prints the result as structured JSON. It does not open an interactive session. The `--command` flag is required.

```bash
stacktape debug:container-exec --stage production --region eu-west-1 --resourceName apiServer --command "ls -la /app"
```

The JSON output includes the resource name, container name, task ARN, the command that was run, captured stdout/stderr output, and an `exitCode` field:

```json
{
  "resourceName": "apiServer",
  "containerName": "apiServer",
  "taskArn": "arn:aws:ecs:eu-west-1:123456789:task/cluster/abc123",
  "command": "ls -la /app",
  "output": "total 48\ndrwxr-xr-x 1 root root ...",
  "exitCode": 0
}
```

The underlying process has a 60-second timeout. If your command needs more time (for example, a large database export), use `container:session` interactively instead, or break the work into smaller steps.

### Targeting a specific task

When multiple tasks are running, `debug:container-exec` connects to the first available task by default. Use `--taskArn` to target a specific one:

```bash
stacktape debug:container-exec --stage production --region eu-west-1 --resourceName apiServer --taskArn abc123def --command "env"
```

You can provide the full ARN or just the task ID suffix. If the specified task is not found, Stacktape lists the available task IDs in the error message.

### Scripting and automation

Because `debug:container-exec` runs a single command and prints structured JSON, it works well in scripts and CI pipelines. The `exitCode` field in the JSON result reflects the exit code of the executed command — non-zero exit codes are captured in the output rather than causing the Stacktape CLI process to throw, so you can inspect both output and exit code programmatically.

Check a version file in staging:

```bash
stacktape debug:container-exec --stage staging --region eu-west-1 --resourceName worker --command "cat /app/version.txt"
```

Verify environment variables are set correctly:

```bash
stacktape debug:container-exec --stage staging --region eu-west-1 --resourceName apiServer --command "env | grep DATABASE"
```

## Multi-container workloads

When your resource runs multiple containers (side containers, init containers), you must specify which container to connect to using `--container`:

```bash
stacktape container:session --stage production --region eu-west-1 --resourceName apiServer --container nginx
```

```bash
stacktape debug:container-exec --stage production --region eu-west-1 --resourceName apiServer --container app --command "node -e 'console.log(process.env.PORT)'"
```

If your resource has multiple containers and you omit `--container`, Stacktape returns an error listing the available container names as defined in your configuration. For resources with a single container, the `--container` flag is optional — Stacktape connects to the only available container automatically.

## Viewing container logs

For log-based debugging, use [`stacktape debug:logs`](/cli/debug-logs) rather than exec-ing into a container to read log files. Container stdout/stderr is automatically captured by AWS CloudWatch:

```bash
stacktape debug:logs --stage production --region eu-west-1 --resourceName apiServer
```

Add `--follow` for live tailing. See [Logs](/observability/logs) for filtering, time ranges, and structured log queries.

## Deployed stacks only

Both `container:session` and `debug:container-exec` initialize Stacktape services for working with a deployed stack and require a deployed ECS service to connect to. They work against any deployed stage — production, staging, or any other stage created with [`stacktape deploy`](/cli/deploy).

These commands do not apply in [dev mode](/local-development/dev-mode-overview) (`stacktape dev`). Dev mode runs workloads locally, not on AWS ECS, so there is no deployed ECS service to connect to. For debugging workloads running locally in dev mode, use standard local debugging tools — your IDE's debugger, terminal output from the `stacktape dev` process, or any tooling appropriate for your local runtime.

## Prerequisites

Both commands require:

1. **A deployed stack** — The resource must be deployed and have at least one running ECS task. If no tasks are running (for example, the service scaled to zero or failed to start), the command returns an error.
2. **AWS Session Manager connectivity** — Both commands use Stacktape's SSM session helpers to connect to ECS Exec, so the local end of the connection is handled by Stacktape rather than requiring separate manual setup.

<Tip>
If you get "No running tasks found" errors, check that your service has healthy tasks. Use [`stacktape debug:logs`](/cli/debug-logs) to investigate why containers may be crashing before becoming reachable.
</Tip>

## Troubleshooting

### "Resource is not a valid container based resource"

This error means the `--resourceName` you specified doesn't match any deployed container workload with an ECS task definition in the stack. Verify:
- The resource name matches exactly what's in your Stacktape config (case-sensitive).
- The stack is deployed and the resource exists in it.
- You're targeting the correct `--stage` and `--region`.

### "Does not have a deployed ECS service"

The resource exists and has an ECS task definition, but no ECS service was found. This can happen if you're targeting a resource type that doesn't create a persistent ECS service, or if the stack deployment is incomplete.

### "No running tasks found"

The ECS service exists but has no healthy running tasks. Common causes:
- The container is crash-looping — check [`stacktape debug:logs`](/cli/debug-logs) for error output.
- A deployment is in progress and old tasks were stopped before new ones started.
- The service scaled to zero (if using scaling rules with a minimum of 0).

### Session disconnects immediately

If the shell session connects and immediately closes:
- The container may lack a shell (`/bin/sh`). Try `--command "/bin/bash"` or use a different base image that includes a shell.
- The container may be terminating due to health check failures or resource constraints.

### Specifying which container to connect to

If you see an error listing container names, your resource has multiple containers and Stacktape needs you to specify which one. Add `--container <name>` where the name matches a container defined in your Stacktape config.

## FAQ

### Which container resources support debugging?

All ECS-based container resources support both `container:session` and `debug:container-exec`: [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), and [multi-container workloads](/resources/compute/multi-container-workload). Both commands look for a deployed ECS service in the stack — resource types that don't create an ECS service are not supported.

### Can I debug containers in dev mode?

No. Dev mode runs workloads locally rather than on AWS ECS. The `container:session` and `debug:container-exec` commands require a deployed ECS service to connect to. Use your local debugging tools (IDE debugger, terminal output) for workloads running in dev mode.

### How does ECS Exec work under the hood?

AWS ECS Exec uses the AWS Systems Manager (SSM) agent running inside your container to establish a secure WebSocket connection. Stacktape's CLI uses its SSM session helpers to handle the local end of the connection. No SSH keys or open inbound ports are required — the connection is outbound-only through SSM.

### Which shell does container:session start?

The `--command` flag on `container:session` is optional. If you need a specific shell (for example `bash`), pass it explicitly with `--command "/bin/bash"`. If the container doesn't have your preferred shell available, the session will fail to start — try a different shell or use a base image that includes one.

### Can I use debug:container-exec in CI/CD pipelines?

Yes. `debug:container-exec` runs a single command, captures the output as JSON with an `exitCode` field, and exits. The underlying process has a 60-second timeout. This makes it well suited for health checks, smoke tests, or post-deploy verification in CI pipelines. Use `container:session` for interactive debugging that requires a terminal.

### What's the difference between container:session and SSH?

`container:session` provides SSH-like access without requiring SSH to be installed in the container or any inbound ports to be open. It uses AWS SSM for the connection, which means no SSH keys to manage, no open ports in your security groups, and all sessions are logged by AWS CloudTrail.

### How do I connect to a specific container in a multi-container workload?

Use the `--container` flag with the container name as defined in your Stacktape config. If you omit it and the workload has multiple containers, Stacktape returns an error listing the available container names. For single-container workloads, the flag is optional.

### Can I run destructive commands like rm or kill?

Yes — both commands give you full access to the container's filesystem and processes with the same permissions as the container's running process. Exercise caution in production. There are no guardrails preventing destructive operations once you're connected.

### Is there a timeout on debug:container-exec?

The `debug:container-exec` command runs the Session Manager plugin subprocess with a 60-second process timeout. If your command needs more time, consider using `container:session` interactively or breaking the work into smaller commands.

### How do I check if my container has running tasks?

Use [`stacktape debug:logs`](/cli/debug-logs) to check for recent container output, or look at your service in the [Stacktape Console](/stacktape-console/console-overview). If the container is crash-looping, the logs will show the error output. You can also use [`stacktape info:stack`](/cli/info-stack) to get an overview of your deployed resources and their status.
