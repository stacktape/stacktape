# Alert History

The Stacktape Console's alert history page gives you a single, chronological view of every alert your organization has received — across [alarms](/observability/alarms), [notifications](/observability/notifications), [budget alerts](/managing-costs/budgets), and [issues](/observability/issues). Instead of checking four separate history views, use alert history to see what happened, when, and whether the alert reached its destination.

## What alert history tracks

Every time an alarm fires, a notification rule matches, a budget threshold is crossed, or an issue is detected, Stacktape records an **alert event**. Each event captures:

- **Source** — which system generated the alert (Alarm, Notification, Budget, or Issue).
- **Event type** — the specific trigger, such as `Deploy failed`, `Alarm triggered`, `Budget threshold exceeded`, or `Issue detected`.
- **Severity** — one of four levels: INFO, WARNING, ERROR, or CRITICAL.
- **Project and stage** — which project and stage the event relates to, when applicable.
- **Title** — a human-readable summary of what happened.
- **Delivery status** — whether the alert was successfully sent to each configured [alert channel](/observability/alert-channels), with per-channel status tracking.

## Event types

Alert events span four source categories. The table below lists every event type the system records.

| Source | Event types |
|---|---|
| **Notification** | Deploy started, Deploy succeeded, Deploy failed, Delete started, Delete succeeded, Delete failed, Deploy rolled back, Rollback succeeded, Rollback failed, Preview deploy succeeded, Preview deploy failed, Git push deploy succeeded, Git push deploy failed, Script run succeeded, Script run failed, Secret created, Secret updated, Secret deleted |
| **Alarm** | Alarm triggered, Alarm resolved |
| **Budget** | Budget threshold exceeded, Budget forecasted overspend |
| **Issue** | Issue detected, Issue resolved |

Notification events cover the broadest range — deployments, rollbacks, previews, script runs, and secret changes. Alarm, budget, and issue events are narrower, each tracking the trigger-and-resolve lifecycle for their respective system.

## Severity levels

Each alert event is assigned a severity level that reflects the urgency of the underlying event. The four levels are:

| Severity | Typical use |
|---|---|
| **INFO** | Successful operations — a deploy completed, a secret was created |
| **WARNING** | Conditions worth noting — a budget forecast is trending toward overspend |
| **ERROR** | Failures that need attention — a deploy failed, an alarm triggered |
| **CRITICAL** | Urgent conditions requiring immediate action |

Severity is assigned by the source system, not by the user. For example, alarm rules set the severity when they fire, and budget alerts use WARNING for forecasted overspend and ERROR for threshold breaches.

## Filtering events

The alert history page includes a **source filter** at the top of the table. You can filter the view to show:

- **All sources** — the default, showing every event across all four categories.
- **Issues** — only issue detection and resolution events.
- **Alarms** — only alarm triggered and resolved events.
- **Notifications** — only deployment, rollback, script, and secret events.
- **Budgets** — only budget threshold and forecasted overspend events.

The table is sorted by time (newest first) and paginated. Use the source filter to narrow down a noisy timeline — for example, filtering to "Alarms" when investigating a spike in alarm activity.

## Delivery tracking

Each alert event includes delivery information for every [alert channel](/observability/alert-channels) that received it. This is the audit trail that answers "did the team actually get notified?"

Each delivery shows:

- **Status** — `Sent` (delivered), `Failed` (delivery attempt failed), or `Pending` (still being processed).
- **Channel** — which alert channel the delivery targeted (Slack, Microsoft Teams, Discord, email, or webhook) along with the channel name.

When an event has no deliveries, the table shows "No rules matched" — meaning no notification rule, alarm rule, or budget alert was configured to route that event to a channel. This is normal for events that don't have matching rules.

<Info>
A computed **delivery summary** rolls up per-channel statuses into a single label: Delivered (all channels succeeded), Partially failed (some succeeded, some failed), Failed (all failed), Pending (still processing), or No matching rules (no channels configured for this event type).
</Info>

### Diagnosing failed deliveries

If a delivery shows as `Failed`, the alert history records the error message from the failed attempt. Common causes include:

- **Slack** — invalid or revoked access token, wrong conversation ID.
- **Microsoft Teams / Discord** — expired or incorrect webhook URL.
- **Email** — invalid recipient address.
- **Webhook** — endpoint returned a non-2xx status code, or the request timed out (15-second timeout).

Fix the underlying channel configuration in [alert channels](/observability/alert-channels), and future alerts will deliver correctly. Failed deliveries are not automatically retried.

## Alert history vs. individual history pages

The Console also offers separate history views for [notifications](/observability/notifications), [alarms](/observability/alarms), and [budgets](/managing-costs/budgets) — accessible from the Monitoring sidebar under each category's "History" submenu. These per-category views show the same underlying events but are pre-filtered to a single source kind.

Use **alert history** when you need a cross-cutting view — for example, correlating a failed deploy notification with the alarm that fired shortly after. Use the individual history pages when you're focused on one system and want less noise.

## Accessing alert history

Alert history is available in the Stacktape Console at the organization level. You can reach it from:

- The **Overview** page's "Recent Alerts" widget, which links to the full alert history.
- The direct `/alert-history` route in the Console.

<Info>
Alert history is a Console-only feature. There is no CLI equivalent for viewing the unified alert history. For CLI-based alarm inspection, use [`stacktape debug:alarms`](/cli/debug-alarms) to check current CloudWatch alarm states.
</Info>

## FAQ

### What is the difference between alert history and the alarms history page?

Alert history is a unified view that combines events from all four sources — alarms, notifications, budgets, and issues — into a single timeline. The alarms history page (under Monitoring > Alarms > History) shows only alarm-triggered and alarm-resolved events. Use alert history when you need the full picture; use the alarms history page when you're focused on metric-based alerts.

### Can I filter alert history by project or stage?

The alert history table shows project and stage columns for each event, but the Console's source filter dropdown filters by event source (Alarm, Notification, Budget, Issue), not by project or stage. To find events for a specific project, scan the Project column in the table or use your browser's find-in-page feature.

### Are failed alert deliveries retried?

No. When a delivery to an alert channel fails (for example, a webhook returns a 500 error or a Slack token is invalid), the failure is recorded with the error message but the delivery is not retried. Fix the channel configuration to ensure future alerts deliver correctly.

### How long is alert history retained?

Alert events are stored in the Stacktape Console's database and are available as long as your organization exists. The history is paginated and loads on demand, so older events remain accessible.

### What severity levels does Stacktape use for alerts?

Stacktape uses four severity levels: INFO, WARNING, ERROR, and CRITICAL. Severity is assigned by the source system — alarm rules set severity when they fire, budget alerts use WARNING for forecasts and ERROR for exceeded thresholds, and notifications use severity appropriate to the event type (e.g., INFO for successful deploys, ERROR for failed deploys).

### Can I export alert history data?

Alert history is currently viewable in the Console only. There is no built-in export or API endpoint for downloading alert history as CSV or JSON.

### What does "No rules matched" mean in the Delivery column?

"No rules matched" means the event was recorded but no notification rule, alarm rule, or budget alert was configured to route it to an alert channel. The event is still logged in the history — it just was not delivered anywhere. This is expected for event types that you have not set up routing rules for.

### How do I set up alerts so events appear in alert history?

Alert events are created automatically when the underlying systems fire. Configure [alarm rules](/observability/alarms) for metric-based monitoring, [notification rules](/observability/notifications) for deployment and lifecycle events, [budget alerts](/managing-costs/budgets) for cost thresholds, and enable [issues](/observability/issues) for runtime error detection. Each of these routes events through [alert channels](/observability/alert-channels) and records them in alert history.
