# EFS Filesystem

A Stacktape EfsFilesystem creates an Amazon Elastic File System — shared, persistent file storage that multiple containers can read and write simultaneously. EFS volumes grow and shrink automatically with no capacity planning. Mount a volume to any container workload to share uploads, CMS media, ML model files, or data that must survive container restarts.

EFS Standard storage costs ~$0.30/GB/month with no minimum commitment — you pay only for storage used.

## When to use

Use an EfsFilesystem when multiple containers need to read and write the same files through a standard POSIX filesystem interface. EFS mounts like a local directory inside your containers, so existing code that uses `fs.readFile` or `open()` works without changes.

Common EFS use cases:

- **Shared uploads** — user-uploaded files accessed by multiple container replicas
- **CMS media directories** — WordPress, Strapi, or other CMS platforms that write to a local `/uploads` path
- **ML model files** — large model weights shared across inference containers
- **Configuration and state** — shared config files, SQLite databases, or application state across containers
- **Build caches** — persistent caches that survive container replacements in CI/CD workloads

## When NOT to use

EFS is not the right fit for every storage need. Consider these alternatives:

| Use case | Better option |
|---|---|
| Storing files accessed via API (uploads, backups, exports) | [Bucket (S3)](/resources/storage/s3-bucket) — cheaper per GB, higher durability, SDK-based access |
| Hosting a static website or SPA | [HostingBucket](/resources/frontend/static-hosting) — includes CDN, routing, and caching |
| Structured or queryable data | [RelationalDatabase](/resources/databases/relational-database) or [DynamoDbTable](/resources/databases/dynamodb) |
| Single-container high-performance block storage | EBS via [overrides](/configuration/overrides-and-escape-hatches) — lower latency, higher IOPS, but single-attach only |

**Cost comparison:** EFS at ~$0.30/GB/month is roughly 10x more expensive than S3 Standard (~$0.023/GB/month). Use EFS only when you need a mounted filesystem — for object storage patterns (upload, download, list via SDK), an [S3 Bucket](/resources/storage/s3-bucket) is more cost-effective.

## Throughput modes

The `throughputMode` property controls how EFS scales read/write performance. Stacktape supports three modes:

| Mode | How throughput scales | Cost model | Best for | Default |
|---|---|---|---|---|
| `elastic` | Auto-scales up and down with demand | Pay per GiB transferred | Spiky or unpredictable workloads | Yes |
| `provisioned` | Fixed throughput you configure | Pay for provisioned MiB/s + storage | Steady high-throughput workloads | No |
| `bursting` | Scales with storage size (50 KiB/s per GiB stored) | Included in storage price | Large filesystems with low throughput needs | No |

### Elastic (default)

Elastic throughput is the recommended mode for most workloads. EFS automatically scales throughput up when your containers are actively reading or writing, and scales down during idle periods. You pay per GiB of data transferred rather than for provisioned capacity, so costs track actual usage. Most teams should keep the default — elastic mode handles traffic spikes without running out of burst credits and requires no capacity planning.

### Provisioned

Provisioned throughput guarantees a fixed MiB/s rate regardless of how much data is stored. Set it with `provisionedThroughputInMibps`. This mode is appropriate when your workload needs consistent, high throughput — for example, a video processing pipeline that continuously reads and writes large files.

<CodeBlock

// [!code focus-start]
export default defineConfig(() => {
  const videoStorage = new EfsFilesystem({

The `provisionedThroughputInMibps` value sets guaranteed throughput — `100` means 100 MiB/s. Additional per-MiB/s fees apply on top of storage costs. You can adjust this value on subsequent deployments without downtime.

<Warning>
Provisioned throughput incurs additional fees based on the amount provisioned, regardless of actual usage. Only use this mode when elastic throughput cannot meet your performance requirements.
</Warning>

### Bursting

Bursting mode ties throughput to storage volume — 50 KiB/s per GiB of stored data, with the ability to burst higher for short periods using burst credits. This mode can work for large filesystems (hundreds of GBs or more) with infrequent access patterns. For small filesystems, burst credits deplete quickly, causing throughput to drop to the baseline. **For most workloads, elastic mode is a better choice** because it doesn't depend on storage size for throughput.

## Backups

Enable `backupEnabled` to turn on daily automatic backups with 35-day retention. Backups are incremental — only changes since the last backup are copied, keeping costs proportional to the rate of change rather than total filesystem size.

**When to enable:** Enable backups for any EFS filesystem storing data you cannot regenerate — user uploads, CMS media, application state. **When the default is fine:** Skip backups for ephemeral data like build caches, temporary processing files, or ML model weights that can be re-downloaded from their source. **Tradeoff:** AWS Backup charges for the storage consumed by backup copies, but incremental backups keep this cost low for filesystems that don't change rapidly.

## Mounting to containers

EFS filesystems are consumed by container workloads through the `volumeMounts` property. The volume appears as a standard directory inside the container — your application reads and writes files using normal filesystem calls. Data stored in EFS persists across container restarts and redeployments, and all data is encrypted in transit automatically.

Container workload types that support EFS volume mounts:

- [Web service](/resources/compute/web-service)
- [Private service](/resources/compute/private-service)
- [Worker service](/resources/compute/worker-service)
- [Multi-container workload](/resources/compute/multi-container-workload)

Each `volumeMounts` entry specifies:

| Property | Required | Description |
|---|---|---|
| `efsFilesystemName` | Yes | Name of the `EfsFilesystem` resource in your config |
| `mountPath` | Yes | Absolute path inside the container (e.g., `/data`, `/mnt/uploads`) |
| `rootDirectory` | No | Subdirectory within the EFS filesystem to mount (defaults to `/`, the entire filesystem) |

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  EfsFilesystem,
  WebService,
  StacktapeImageBuildpackPackaging
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const sharedStorage = new EfsFilesystem({
    backupEnabled: true
  });

  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({
      entryfilePath: './src/server.ts'
    }),
    resources: {
      cpu: 0.25,
      memory: 512
    },
    volumeMounts: [
      {
        type: 'efs',
        properties: {
          efsFilesystemName: 'sharedStorage',
          mountPath: '/mnt/uploads'
        }
      }
    ]
  });

  return {
    resources: { sharedStorage, api }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

In this example, the `api` web service mounts the `sharedStorage` EFS filesystem at `/mnt/uploads`. The application can then read and write files to `/mnt/uploads` using standard filesystem operations. When the service scales to multiple replicas, all replicas share the same files — a write from one replica is visible to all others.

Use `rootDirectory` to restrict a container's view to a subdirectory within the filesystem. This is useful when multiple workloads share one EFS filesystem but each should only access its own directory.

<Tip>
When multiple container replicas mount the same EFS filesystem, all replicas see the same files in real time. EFS handles concurrent reads and writes using standard POSIX file-locking semantics.
</Tip>

## FAQ

### How do I mount an EFS volume to a container?

Define an `EfsFilesystem` resource in your config, then add it to the `volumeMounts` array on any container workload ([web service](/resources/compute/web-service), [private service](/resources/compute/private-service), [worker service](/resources/compute/worker-service), or [multi-container workload](/resources/compute/multi-container-workload)). Each mount entry specifies the EFS resource name and a `mountPath` inside the container. Your application accesses it using standard file I/O — all data is encrypted in transit automatically.

### How much does Amazon EFS cost?

EFS Standard storage costs ~$0.30/GB/month with no upfront commitment. You pay only for the storage you use — the filesystem grows and shrinks automatically. Additional costs depend on throughput mode: elastic charges per GiB transferred, provisioned charges per MiB/s configured. Infrequent Access storage (configurable via [overrides](/configuration/overrides-and-escape-hatches)) can reduce costs to ~$0.016/GB/month for rarely accessed files. See [Managing Costs](/managing-costs/overview) for tips on monitoring AWS spend.

### Can multiple containers access the same EFS filesystem?

Yes. EFS is designed for shared access — multiple container replicas, and even containers from different workloads in the same stack, can mount the same filesystem simultaneously. EFS uses standard POSIX file-locking semantics to handle concurrent reads and writes. This makes it ideal for CMS platforms, shared upload directories, and any workload where replicas need consistent access to the same files.

### EFS vs S3 — when should I use each?

Use EFS when your application needs to read and write files using standard filesystem calls (`fs.readFile`, `open()`), especially when multiple containers need shared access to the same directory. Use an [S3 Bucket](/resources/storage/s3-bucket) when you interact with files through an API (upload, download, list) — S3 is cheaper per GB (~$0.023/GB/month vs ~$0.30/GB/month), offers eleven-nines durability, and scales without limits. Most web applications should default to S3 and only use EFS when they genuinely need a mounted filesystem.

### Does EFS encrypt data?

EFS encrypts all data in transit between containers and the filesystem automatically. For encryption at rest, EFS supports it but `EfsFilesystemProps` does not expose a dedicated toggle. To enable at-rest encryption, use [overrides](/configuration/overrides-and-escape-hatches) to set the `Encrypted` property on the underlying CloudFormation `AWS::EFS::FileSystem` resource. Once enabled on a filesystem, encryption at rest cannot be disabled without recreating it.

### Can I use EFS with Lambda functions in Stacktape?

EFS volume mounts in Stacktape are configured on container workloads via the `volumeMounts` property — [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), and [multi-container workloads](/resources/compute/multi-container-workload). Lambda functions do not expose `volumeMounts` in Stacktape's configuration. If you need shared persistent storage accessible from a [Lambda function](/resources/compute/lambda-function), use an [S3 Bucket](/resources/storage/s3-bucket) instead.

### What throughput mode should I choose?

Start with `elastic` (the default) — it auto-scales with demand and requires no capacity planning. Switch to `provisioned` only if you have a steady, high-throughput workload (video processing, large data pipelines) where you need guaranteed MiB/s and are willing to pay for it regardless of usage. Avoid `bursting` for small filesystems because throughput is tied to storage size and burst credits can deplete quickly. Most teams never need to change from `elastic`.

### Does EFS filesystem size have a limit?

EFS has no practical storage limit — filesystems can scale to petabytes automatically. Individual files can be up to 47.9 TiB. You don't need to provision capacity in advance; the filesystem grows as you write data and shrinks as you delete it. There are no minimum storage charges beyond what you actually use.

### How do I back up an EFS filesystem?

Set `backupEnabled: true` in your EfsFilesystem config to enable daily automatic backups through AWS Backup with 35-day retention. Backups are incremental, so only changes since the last backup are copied — storage costs stay proportional to the rate of change. For more control over backup schedules, retention policies, or cross-region replication, configure AWS Backup directly using [overrides](/configuration/overrides-and-escape-hatches).

### EFS vs EBS — what's the difference?

EFS is a shared network filesystem — multiple containers across availability zones can mount it simultaneously. EBS is block storage attached to a single compute instance with lower latency and higher IOPS, but accessible by only one instance at a time. In Stacktape, EFS is the built-in shared storage option via `EfsFilesystem` and `volumeMounts`. EBS is not directly exposed as a Stacktape resource type but can be configured through [overrides](/configuration/overrides-and-escape-hatches) for single-container, high-IOPS workloads.
