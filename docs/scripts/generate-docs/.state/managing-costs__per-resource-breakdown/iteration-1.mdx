# Per-Resource Cost Breakdown

Stacktape Console attributes your AWS spend to individual stacks and resources, so you can see exactly which Lambda function, database, or container is costing what. All costs are billed by AWS directly to your AWS account — the Stacktape subscription fee is separate and does not appear in these reports.

## How cost attribution works

Stacktape automatically tags every AWS resource it creates with metadata including the project name, stage, and stack identifier. When AWS generates daily Cost and Usage Reports, Stacktape's billing attribution system matches each cost line item back to the originating CloudFormation resource using these tags and resource identifiers. The result is a per-resource cost breakdown available in the Console within a day of the charges being incurred.

The attribution pipeline uses multiple matching strategies:

| Strategy | How it matches | Example |
|---|---|---|
| Physical Resource ID | Exact match against the AWS resource's physical ID | RDS instance, S3 bucket |
| CloudFormation logical ID | Matches via `aws:cloudformation:logical-id` tag | Most standard resources |
| ECS cluster tag | Matches via `aws:ecs:clusterName` tag | Container workloads |
| AutoScaling group tag | Matches via `stp:asg-name` tag | EC2-backed workloads |
| Stack-level fallback | Costs tagged with the stack hash but not matched to a specific resource | Edge functions, shared infrastructure |

Costs that cannot be attributed to any Stacktape-managed stack appear separately as non-attributed costs. These typically come from resources created outside of Stacktape or from AWS services that do not support cost allocation tags.

<Info>Cost reports are generated daily from AWS Cost and Usage Reports. After deploying a new stack, it can take up to a day before cost data appears in the Console.</Info>

## Viewing costs per stack

The **Costs per stack** page in the Stacktape Console shows month-to-date costs for every stack in a connected AWS account. Each row displays the project name, stage, region, and total cost for that stack.

<ConsoleScreenshot src="/static/screenshots/console-costs-per-stack.png" alt="Stacktape Console Costs per stack page showing a table of stacks sorted by month-to-date costs, with project name, stage, region, and cost columns" caption="Costs per stack view — filter by AWS account and month to see where spend is going." />

You can filter the view by:

- **AWS account** — if your organization has multiple [connected AWS accounts](/stacktape-console/connecting-your-aws-account), select which one to inspect.
- **Month** — choose the current month or any month going back roughly two years. Costs are aggregated per calendar month.

The table is sorted by cost (highest first) by default, making it straightforward to identify which stacks are driving your bill. Click any row to drill into the per-resource breakdown for that stack.

## Viewing costs per resource

When you click into a specific stack, the Console shows a per-resource cost table. Each row shows the Stacktape resource name, the AWS resource type, and the cost for that resource in the selected month. A total is displayed at the top of the table.

<ConsoleScreenshot src="/static/screenshots/console-costs-per-resource.png" alt="Stacktape Console per-resource cost breakdown showing individual resources like Lambda functions, RDS instances, and S3 buckets with their costs" caption="Per-resource breakdown for a single stack — sorted by cost to highlight the most expensive resources." />

This view helps you answer questions like:

- Which resource in this stage is the most expensive?
- Is the database or the compute driving costs?
- Did a recent change cause a cost spike for a specific resource?

Costs are displayed rounded to four decimal places, so even low-cost resources like infrequent Lambda invocations show meaningful data.

## Stack-level vs. resource-level costs

Some AWS charges cannot be matched to a specific Stacktape resource within a stack. These appear as "other costs" at the stack level. Common examples include:

- **Edge functions** — CloudFront-associated Lambda@Edge costs are sometimes reported without a physical resource ID that maps cleanly to a CloudFormation resource.
- **Data transfer** — cross-region or internet-bound data transfer charges are often aggregated by AWS rather than attributed to a single resource.
- **Shared infrastructure** — NAT gateways, VPC endpoints, and similar networking components may be shared across resources.

The stack total always includes both attributed resource costs and these unmatched stack-level costs, so the numbers you see at the stack level are comprehensive.

## Cost allocation tags

Stacktape applies the following cost allocation tags to every resource it manages:

| Tag | Purpose |
|---|---|
| `stp:project-name` | Identifies the Stacktape project |
| `stp:stage` | Identifies the stage (e.g. `production`, `staging`) |
| `stp:stack-name` | Full stack name used for attribution |
| `stp:globally-unique-stack-hash` | Unique identifier for the stack across regions |

These tags are activated as AWS Cost Allocation Tags automatically. If you use AWS Cost Explorer or third-party cost tools alongside Stacktape, you can filter by these tags to get the same project-and-stage grouping outside the Console.

<Tip>If you're running [multi-region deployments](/deployment-and-lifecycle/multi-region-deployments), the `stp:globally-unique-stack-hash` tag ensures that costs from different regions are attributed to the correct stack, even when stack names overlap.</Tip>

## Using cost data effectively

### Identify expensive resources

Sort the per-resource table by cost (the default) to immediately see which resources dominate your bill. In most stacks, one or two resources account for the majority of spend — often a relational database or a high-traffic container workload. Focus optimization efforts on these.

### Compare stages

Check the costs-per-stack view to compare your `production` and `staging` stages. If staging costs are close to production, you may be over-provisioning dev/test infrastructure. Consider using smaller [database instance classes](/resources/databases/relational-database) or lower [container CPU/memory](/resources/compute/web-service) for non-production stages.

### Track cost trends

Select different months to see how costs change over time. A sudden increase in a specific resource's cost often correlates with a recent deployment — check your [deployment history](/observability/overview) to identify what changed.

### Set up budget alerts

The per-resource breakdown helps you understand current spend, but [budgets](/managing-costs/budgets) alert you before costs exceed a threshold. Use the breakdown data to set realistic budget limits based on actual usage patterns.

## FAQ

### How often is cost data updated?

Cost reports are generated daily from AWS Cost and Usage Reports. After a new stack is deployed, it typically takes up to one day for cost data to appear in the Console. Within a given month, the figures shown are month-to-date totals that update daily.

### Why do some costs show as non-attributed?

AWS charges that cannot be matched to any Stacktape-managed CloudFormation stack appear as non-attributed. This happens when resources were created outside of Stacktape, when an AWS service does not support cost allocation tags, or when AWS aggregates charges (like data transfer) in a way that strips resource-level identifiers.

### Can I see costs in currencies other than USD?

The cost breakdown uses whichever currency your AWS account is configured to bill in. Most AWS accounts use USD, but if your account bills in EUR, GBP, or another currency, the Console reflects that. The currency is determined by your AWS billing settings, not by Stacktape.

### How does Stacktape attribute costs to resources?

Stacktape tags every AWS resource it creates with metadata tags (`stp:project-name`, `stp:stage`, `stp:stack-name`). A daily attribution pipeline reads AWS Cost and Usage Reports and matches each cost line item to a CloudFormation resource using physical resource IDs, CloudFormation logical IDs, and ECS/AutoScaling tags. Costs that match a stack but not a specific resource are grouped as stack-level "other costs."

### Does this include Stacktape's subscription fee?

No. The cost breakdown shows only AWS infrastructure charges billed directly by AWS to your AWS account. Stacktape's subscription fee is billed separately through the [Stacktape billing system](/stacktape-console/billing-and-subscription) and does not appear in these reports.

### Can I use AWS Cost Explorer alongside Stacktape?

Yes. The cost allocation tags Stacktape applies (`stp:project-name`, `stp:stage`, etc.) are standard AWS cost allocation tags. You can use them in AWS Cost Explorer, AWS Budgets, or third-party tools like CloudHealth or Vantage to filter and group costs the same way the Console does.

### How do I reduce costs for a specific resource?

Once you've identified the expensive resources, visit the [optimization tips](/managing-costs/optimization-tips) page for actionable strategies. Common wins include right-sizing database instances, switching Lambda functions to ARM architecture, and using reserved capacity for predictable workloads.

### Why is my database the most expensive resource?

Relational databases (RDS, Aurora) run continuously and charge by the hour for the instance class, plus storage and I/O. Unlike Lambda functions that scale to zero, databases maintain a baseline cost even with no traffic. For non-production stages, consider smaller instance classes. For production, Aurora Serverless v2 can reduce costs for variable workloads. See [relational databases](/resources/databases/relational-database) for configuration options.
