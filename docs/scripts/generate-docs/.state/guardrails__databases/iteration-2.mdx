# Database Guardrails

Stacktape database guardrails are preventive deployment policies that restrict which database engines, instance sizes, and accessibility modes your team can use. They block deployments containing [relational databases](/resources/databases/relational-database) that violate your organization's rules — catching policy violations before infrastructure changes are applied.

<Info>
Guardrails are **preventive** — they block non-compliant deployments before anything is provisioned. They are distinct from [alarms](/observability/alarms), which are **reactive** and notify you about runtime problems after deployment.
</Info>

Stacktape defines four database-specific guardrail types:

| Guardrail | Type identifier | Key property |
|---|---|---|
| [Engine restriction](#engine-restriction) | `database-engine-restriction` | `allowedEngines` — permitted engine types |
| [Instance restriction](#instance-restriction) | `database-instance-restriction` | `blockedInstanceSizes` — forbidden instance sizes |
| [VPC-only databases](#vpc-only-databases) | `require-vpc-databases` | `enabled` — require VPC-only accessibility |
| [Deletion protection](#deletion-protection) | `require-deletion-protection` | `enabled` — require deletion protection on all relational databases |

For other guardrail categories, see [deployment guardrails](/guardrails/deployment), [security guardrails](/guardrails/security-and-data-protection), and [resource limit guardrails](/guardrails/resource-limits).

## Engine restriction

The `database-engine-restriction` guardrail limits which database engine types your team can provision. It uses an `allowedEngines` list — any [relational database](/resources/databases/relational-database) whose engine type is not in the list is blocked. This is the most common database guardrail, typically used to standardize on a single engine family across an organization.

The `allowedEngines` property accepts an array of engine type strings. The guardrail type definition gives `postgres` and `aurora-postgresql` as examples. For the full list of available engine type identifiers, see the [relational database](/resources/databases/relational-database) documentation.

**When to enable:** Your organization wants to standardize on a database engine family (e.g., PostgreSQL only) or prevent teams from using engines that carry additional licensing costs. Setting `allowedEngines` to `['postgres', 'aurora-postgresql']` lets teams choose between single-instance RDS PostgreSQL and Aurora PostgreSQL while blocking all other engine families.

**When to skip:** Small teams or early-stage projects where engine flexibility matters more than standardization. If your team already follows a strong convention, a guardrail may add overhead without meaningful benefit.

**Allowlist vs blocklist:** Engine restriction uses an **allowlist** — you explicitly permit certain engines, and everything else is blocked. This is the safer default for standardization: when new engine types become available, they are blocked until you deliberately add them.

<Tip>
If you're standardizing on PostgreSQL, consider including multiple PostgreSQL-compatible engine types in the allowlist (e.g., `postgres`, `aurora-postgresql`) so teams can choose the deployment model that fits their workload — single-instance RDS or Aurora cluster.
</Tip>

## Instance restriction

The `database-instance-restriction` guardrail prevents teams from provisioning specific database instance sizes. Unlike engine restriction, this guardrail uses a **blocklist** via the `blockedInstanceSizes` property — you specify which instance sizes are forbidden rather than which are allowed.

The `blockedInstanceSizes` property accepts an array of instance size strings. The guardrail type definition gives `db.r5.4xlarge` and `db.r6g.8xlarge` as examples. Any relational database configuration that includes a blocked instance size is rejected.

**When to enable:** You want to prevent cost surprises from oversized instance classes. Large memory-optimized instances (such as r5 and r6g families in 4xlarge and above) can cost significantly more than smaller instances that handle the same workloads adequately. Blocking these proactively is simpler than reviewing every deployment manually.

**When to skip:** Teams with existing cost review processes, or organizations where every database configuration is reviewed before deployment. Also less relevant for teams that exclusively use serverless database engines, which scale capacity automatically rather than using fixed instance sizes.

**Blocklist vs allowlist:** Instance restriction uses a **blocklist** — you forbid specific sizes, and everything else is allowed. This is more practical than an allowlist for instance sizes because the number of valid sizes is large and changes as AWS introduces new instance classes. Block the outliers you want to prevent rather than maintaining a comprehensive allowlist.

<Warning>
Make sure to include all instance size variants you want to block. For example, if you block `db.r5.4xlarge`, the `db.r5.8xlarge` or `db.r6g.4xlarge` sizes remain available unless you also add them to the blocklist.
</Warning>

## VPC-only databases

The `require-vpc-databases` guardrail enforces that all databases use VPC-only network accessibility with no public internet access. When `enabled` is set to `true`, deployments containing databases exposed to the public internet are blocked.

The guardrail has a single `enabled` property. When active, all databases in the stack must use VPC-only accessibility. For details on the available accessibility modes and how to configure them, see the [relational database](/resources/databases/relational-database) documentation.

**When to enable:** Production environments, regulated industries, or any organization handling sensitive data — PII, financial records, or health information. Most compliance frameworks (SOC 2, HIPAA, PCI-DSS) require databases to be unreachable from the public internet. Enabling this guardrail ensures no team can accidentally expose a database.

**When to skip:** Development-only organizations where developers need to connect to databases directly from their local machines without a bastion or VPN. Consider enabling this guardrail selectively for production stages using [deployment guardrails](/guardrails/deployment) to restrict which stages the policy applies to.

**Tradeoff:** VPC-only databases require consuming workloads ([Lambda functions](/resources/compute/lambda-function), [web services](/resources/compute/web-service), and other compute resources) to be in the same VPC. This adds a configuration step for every resource that needs database access. For local access to VPC-only databases during development, use a [bastion host](/resources/security/bastion-host).

## Deletion protection

The `require-deletion-protection` guardrail enforces that all [relational databases](/resources/databases/relational-database) have `deletionProtection` set to `true`. When `enabled` is set to `true`, every relational database in the stack must explicitly include `deletionProtection: true` in its configuration.

The guardrail has a single `enabled` property. AWS RDS deletion protection prevents a database instance from being deleted — you must first disable the flag before the database can be removed. This two-step process is an AWS-level safeguard, and the guardrail ensures your team always has it active.

**When to enable:** Any production environment, or as a blanket policy once your organization wants to prevent accidental database deletion. The cost of losing production data far outweighs the minor friction of managing the protection flag explicitly.

**When to skip:** Development and testing stages where databases are ephemeral and frequently recreated. Enforcing deletion protection on short-lived dev databases adds friction without meaningful benefit.

**Tradeoff:** When you genuinely need to remove a database, you must first update the configuration to set `deletionProtection: false`, deploy that change, and then remove the database resource. This is intentional — the extra step prevents both accidental stack deletion and accidental resource removal from the configuration.

## Combining database guardrails

Database guardrails work together — you can enable multiple guardrails simultaneously to enforce a comprehensive database policy. A typical production setup enables all four: standardize on PostgreSQL engines, block oversized instances, require VPC isolation, and enforce deletion protection.

The following example shows a database configuration designed to comply with all four database guardrails at once.

    credentials: {
      masterUserName: 'app_admin',
      masterUserPassword: '$Secret(\'prod-db-password\')'

    engine: {
      type: 'aurora-postgresql-serverless-v2',
      properties: {
        version: '16.4',
        minCapacity: 0.5,
        maxCapacity: 8
      }
    },
    accessibility: {
      accessibilityMode: 'scoping-workloads-in-vpc'
    },
    deletionProtection: true
  });

  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    connectTo: [prodDb],
    joinDefaultVpc: true
  });

  return {
    resources: { prodDb, api }
  };
});
// [!code focus-end]`

This configuration addresses each guardrail:

- **Engine restriction** — uses `aurora-postgresql-serverless-v2`, a PostgreSQL-based engine that would pass an `allowedEngines` list including PostgreSQL variants
- **Instance restriction** — no instance sizes from a `blockedInstanceSizes` list are present in the configuration
- **VPC-only** — uses `scoping-workloads-in-vpc` accessibility, keeping the database off the public internet
- **Deletion protection** — explicitly sets `deletionProtection: true`

The [Lambda function](/resources/compute/lambda-function) sets `joinDefaultVpc: true` and uses `connectTo` to establish network and IAM access to the database.

## FAQ

### What's the difference between guardrails and alarms for databases?

[Guardrails](/guardrails/overview) are **preventive** — they block non-compliant deployments before infrastructure changes happen. [Alarms](/observability/alarms) are **reactive** — they monitor runtime metrics (CPU usage, connection count, storage) and notify you when thresholds are crossed. Use guardrails to enforce policy at deploy time and alarms to monitor database health at runtime. They complement each other but serve different purposes.

### Does the engine restriction use an allowlist or blocklist?

The `database-engine-restriction` guardrail uses an **allowlist** (`allowedEngines`). You specify which engine types are permitted, and any engine not in the list is blocked. This is the opposite of `database-instance-restriction`, which uses a **blocklist** (`blockedInstanceSizes`) of forbidden instance sizes. The allowlist approach is safer for standardization because new engine types are blocked by default until explicitly added.

### How is engine restriction different from resource type restriction?

The `resource-type-restriction` guardrail (covered in [resource limit guardrails](/guardrails/resource-limits)) blocks entire resource types — for example, blocking `relational-database` entirely so no one can provision SQL databases. The `database-engine-restriction` guardrail is more granular: it allows relational databases but limits which engines can be used within them. Use engine restriction when you want databases but need to standardize on specific engines.

### What engine values can I put in the allowedEngines list?

The guardrail type definition gives `postgres` and `aurora-postgresql` as examples. The full set of supported engine type identifiers matches the engine options available for [relational databases](/resources/databases/relational-database). Consult the relational database documentation for the complete list, which includes RDS single-instance, Aurora clustered, and Aurora Serverless variants across PostgreSQL and MySQL families.

### What instance sizes can I block?

The guardrail type definition gives `db.r5.4xlarge` and `db.r6g.8xlarge` as examples. You can add any valid RDS or Aurora instance size string to the `blockedInstanceSizes` array. The [relational database](/resources/databases/relational-database) documentation covers the available instance classes and their cost characteristics.

### When should I enable VPC-only databases?

Enable the `require-vpc-databases` guardrail for production environments and any stage handling sensitive data. VPC-only accessibility prevents databases from being reached over the public internet, which most compliance frameworks (SOC 2, HIPAA, PCI-DSS) require. For development stages where developers need direct database access from their local machines, consider using a [bastion host](/resources/security/bastion-host) alongside the guardrail, or leaving it disabled for non-production stages.

### Should I enable all four database guardrails at once?

Start with the guardrails that match your current priorities. VPC-only databases and deletion protection provide the highest-value safety net for production with minimal friction — they prevent the two most costly mistakes (data exposure and accidental deletion). Engine restriction is worth enabling once your team has standardized on an engine family. Instance restriction is most useful for cost control in larger teams where developers choose their own instance sizes.

### Can I apply different database guardrails to different stages?

Database guardrails apply as organization-level policies. For stage-level controls, see [deployment guardrails](/guardrails/deployment), which support stage restriction through the `stage-restriction` guardrail type. Combining database guardrails with stage restrictions lets you enforce strict database policies globally while controlling which stages can be deployed.

### How much does AWS RDS cost compared to Aurora Serverless?

AWS RDS single-instance databases are billed per hour of runtime based on the instance size, plus storage costs. Aurora Serverless v2 bills based on consumed capacity units (ACUs) and scales down when idle, making it cost-effective for variable workloads. The `database-instance-restriction` guardrail helps control RDS costs by blocking expensive instance sizes before they are provisioned. See the [managing costs overview](/managing-costs/overview) for Stacktape-specific cost monitoring tools.
