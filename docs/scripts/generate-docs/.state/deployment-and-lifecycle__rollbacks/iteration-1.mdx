# Rollbacks

Stacktape provides two rollback mechanisms for recovering from bad deployments: **version rollback** re-deploys a previous good version of your stack, and **CloudFormation rollback** recovers a stack stuck in a failed state. Both operate without requiring your local source code or config file — they work directly with artifacts already stored in AWS.

## When to use each rollback type

Stacktape's two rollback commands serve different failure scenarios. Choosing the wrong one wastes time or leaves the stack in a worse state.

| Scenario | Command | What it does |
|----------|---------|--------------|
| Last deploy succeeded but introduced a bug | `stacktape rollback` | Re-deploys a previous version's CloudFormation template as a new deployment |
| Deploy failed and stack is in `UPDATE_FAILED` or `UPDATE_ROLLBACK_FAILED` | `stacktape cf:rollback` | Triggers CloudFormation's native rollback to restore last known good state |
| You want to undo 2-3 deploys back | `stacktape rollback --rollbackSteps 3` | Targets an older version |
| A specific resource can't be rolled back by CloudFormation | `stacktape cf:rollback --resourcesToSkip LogicalId` | Skips problematic resources during recovery |

## Version rollback

The `stacktape rollback` command re-deploys a previously working CloudFormation template and reuses the original deployment artifacts (Lambda bundles, container images) from that version. It creates a **new** deployment version — it does not rewrite history.

This is the command you want when your stack is healthy (last deploy succeeded) but the application is broken due to a code or config change you want to undo.

### How it works

Every successful deployment stores a versioned snapshot of the CloudFormation template and all packaged artifacts (Lambda zips, container images) in your stack's deployment bucket. Versions follow the format `v000001`, `v000002`, etc.

The rolled-back template is deployed as a new version number (e.g., rolling back from `v000005` to `v000003` creates `v000006` containing the `v000003` template). This means you can always roll forward again.

### Basic usage

Roll back one version (the most common case):

```bash
stacktape rollback --stage production --region eu-west-1
```

Roll back to a specific version:

```bash
stacktape rollback --stage production --region eu-west-1 --targetVersion v000003
```

Roll back multiple steps:

```bash
stacktape rollback --stage production --region eu-west-1 --rollbackSteps 3
```

### Listing available versions

Before rolling back, check which versions are available:

```bash
stacktape rollback --stage production --region eu-west-1 --listVersions
```

This prints the current version and all versions that still have artifacts in the deployment bucket. Versions older than the retention limit are automatically cleaned up and cannot be rolled back to.

### Artifact retention

Stacktape keeps deployment artifacts for a configurable number of previous versions. Once a version's artifacts are cleaned up, you can no longer roll back to it.

Configure retention via `deploymentConfig.previousVersionsToKeep` in your Stacktape config. The default is 10 versions. Setting a higher value gives you more rollback targets but increases S3 and ECR storage costs.

### Rollback safety warnings

Not all deployments can be rolled back cleanly. When Stacktape detects potential issues, it prints warnings but still proceeds with the rollback. The warnings cover:

- **`$File` or other state-embedding directives** — The rolled-back template contains values that were baked in at the original deploy time. If the referenced file has changed since then, the rollback uses the old content.
- **TypeScript transforms** — Config-time transforms that generated dynamic CloudFormation are not re-executed; the old output is replayed as-is.
- **`after:deploy` hooks** — Hooks that ran after the original deploy (database migrations, cache warming, etc.) will NOT re-run during rollback.
- **Custom directives** — Results from custom directives were captured at original deploy time.

These warnings are informational. If your config is purely declarative (no `$File`, no hooks, no transforms), rollbacks replay cleanly every time.

### Automatic rollback on alarm

You can configure your stack to automatically roll back a deployment if a CloudWatch alarm fires during or shortly after the deploy. This is useful for production stacks where you want automated recovery when metrics degrade.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    memory: 1024
  });

  return {
    resources: { api },
    deploymentConfig: {
      triggerRollbackOnAlarms: ['api-error-rate-alarm'],
      monitoringTimeAfterDeploymentInMinutes: 5
    }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`triggerRollbackOnAlarms` accepts alarm names (from your stack's `alarms` section) or full ARNs. The alarm must already exist from a prior deployment — a newly created alarm only takes effect on the *next* deploy.

`monitoringTimeAfterDeploymentInMinutes` extends the monitoring window after CloudFormation reports the update as complete. If an alarm fires within this window, CloudFormation automatically rolls back the stack. Defaults to `0` (no post-deploy monitoring). Set this to 3-10 minutes for production stacks to catch latent regressions.

### Disabling automatic rollback

By default, a failed CloudFormation deployment automatically rolls back to the previous state. You can disable this to inspect the failed state before deciding how to proceed:

    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    })
  });

    resources: { api },
    deploymentConfig: {
      disableAutoRollback: true
    }
  };
});
// [!code focus-end]`
    }
  ]}
/>

When auto-rollback is disabled and a deployment fails, the stack enters `UPDATE_FAILED` state. You can then inspect CloudFormation events, fix the issue, and either redeploy or use `stacktape cf:rollback` to recover manually.

<Warning>
Only use `disableAutoRollback` in non-production stages for debugging. In production, automatic rollback is your safety net — disable it only if you have a specific reason to inspect failure states.
</Warning>

## CloudFormation rollback

The `stacktape cf:rollback` command triggers CloudFormation's native rollback mechanism. Use this when a deployment failed and left your stack in a broken state (`UPDATE_FAILED` or `UPDATE_ROLLBACK_FAILED`).

This is different from version rollback — it doesn't target a specific previous version. Instead, it tells CloudFormation to complete its interrupted rollback process and restore the stack to its last stable state.

### When to use cf:rollback

Use `stacktape cf:rollback` when:

- A deployment failed mid-way and the stack shows `UPDATE_FAILED` in the AWS Console
- A previous automatic rollback itself failed, leaving the stack in `UPDATE_ROLLBACK_FAILED`
- You used `disableAutoRollback: true` and now want to trigger the rollback manually

You do NOT need this command when:

- The stack is healthy and you want to go back to an earlier version — use `stacktape rollback` instead
- The deploy failed but auto-rollback succeeded — the stack is already recovered

### Basic usage

```bash
stacktape cf:rollback --stage production --region eu-west-1
```

### Skipping problematic resources

Sometimes a CloudFormation rollback fails because a specific resource can't be restored to its previous state (e.g., a manually deleted resource, or a resource whose previous configuration is no longer valid). In these cases, use `--resourcesToSkip` to skip those resources:

```bash
stacktape cf:rollback --stage production --region eu-west-1 --resourcesToSkip MyBrokenLambda MyOtherResource
```

The skipped resources remain in their current state while the rest of the stack rolls back. After recovery, redeploy to bring the skipped resources back into the desired state.

<Warning>
Skipping resources during rollback leaves them out of sync with the rest of your stack. Always follow up with a fresh `stacktape deploy` to reconcile the state.
</Warning>

After a successful `cf:rollback`, Stacktape automatically cleans up any deployment artifacts that were uploaded during the failed deployment.

## Deployment configuration reference

The `deploymentConfig` section in your Stacktape config controls rollback behavior, artifact retention, and deployment safety features.

| Property | Default | Description |
|----------|---------|-------------|
| `previousVersionsToKeep` | 10 | Number of old deployment versions to retain. Higher = more rollback targets, more storage cost |
| `triggerRollbackOnAlarms` | — | Alarm names or ARNs that trigger automatic rollback during deployment |
| `monitoringTimeAfterDeploymentInMinutes` | 0 | Minutes to continue monitoring alarms after deploy completes |
| `disableAutoRollback` | false | Keep stack in failed state instead of auto-rolling back |
| `terminationProtection` | false | Prevent accidental stack deletion |
| `cloudformationRoleArn` | — | IAM role for CloudFormation operations (persisted across deployments) |

## FAQ

### What happens to my database during a rollback?

A version rollback re-deploys the old CloudFormation template. If your database schema hasn't changed between versions, nothing happens to the database. If it has changed (e.g., you added a column via a deployment script), the rollback does NOT reverse database migrations — it only rolls back infrastructure. You'll need to handle data migrations separately.

### Can I roll back if I deleted a resource in a recent deploy?

Yes, but with caveats. The rolled-back template will attempt to recreate the deleted resource. If the resource was stateful (a database, a bucket with data), the data from before deletion is gone — CloudFormation creates a fresh resource. Stateless resources (Lambda functions, container services) recreate cleanly.

### How fast is a version rollback compared to a fresh deploy?

Version rollback skips all packaging and uploading steps — it reuses existing artifacts. The only time spent is the CloudFormation stack update itself, which typically takes 1-5 minutes depending on the number and type of resources being updated. This is significantly faster than a full deploy with build/package/upload.

### Does rollback work with gradual deployments?

Rollback replaces the entire CloudFormation template at once. If you were using a [gradual deployment](/deployment-and-lifecycle/gradual-deployments) strategy (canary or linear), the rollback bypasses it and immediately deploys the old version. This is by design — when you're rolling back, you want fast recovery, not gradual traffic shifting.

### What's the difference between `stacktape rollback` and `stacktape cf:rollback`?

`stacktape rollback` is a version-targeted operation — it picks a specific previous deployment version and re-deploys its template. Use it when your stack is healthy but the latest code/config is wrong. `stacktape cf:rollback` is a recovery operation — it unsticks a stack that CloudFormation left in a failed state. Use it when the stack itself is broken, not just the application.

### How do I know which version to roll back to?

Run `stacktape rollback --listVersions` to see all available versions. Versions are numbered sequentially (`v000001`, `v000002`, etc.), so you can correlate them with your deployment history. For most incidents, rolling back one step (the default) is correct — you're undoing the most recent change.

### Can I roll back a stack that was deployed from CI/CD?

Yes. The `stacktape rollback` command doesn't require the original source code or config file. It works entirely from stored artifacts in AWS. You can run it from any machine with Stacktape installed and access to the target AWS account — your laptop, a CI runner, or the Stacktape Console.

### What happens to static website content during rollback?

If your stack includes a [static hosting](/resources/frontend/static-hosting) resource with bucket sync, Stacktape attempts to restore the old file versions from a stored manifest. If the manifest isn't available (e.g., the version predates manifest tracking), you'll see a warning and may need to redeploy from the original commit to restore static content.

### How much does keeping old versions cost?

Lambda zip artifacts are typically a few MB each. Container images in ECR are deduplicated at the layer level, so incremental versions share most of their storage. For a typical stack with 2-3 Lambda functions, keeping 10 versions costs well under $1/month in S3 and ECR storage. Increase retention freely unless you have very large artifacts or many versions.

### Can I prevent accidental deletion of a production stack?

Yes — set `terminationProtection: true` in `deploymentConfig`. This blocks `stacktape delete` until you explicitly deploy with `terminationProtection: false` first. This is unrelated to rollback but is a common production safety measure configured alongside rollback alarms.
