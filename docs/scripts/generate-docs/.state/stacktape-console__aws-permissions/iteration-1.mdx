# AWS Permissions Needed

Stacktape connects to your AWS account by creating a cross-account IAM role through a CloudFormation stack. This page documents exactly what permissions the role receives, why they're needed, and how the security model works. For the setup flow itself, see [Connecting your AWS account](/stacktape-console/connecting-your-aws-account).

## What This Part of the Console Does

The AWS connection CloudFormation stack creates an IAM role in your AWS account that Stacktape assumes with temporary credentials. The role's permissions determine what Stacktape can do — deploy resources, read logs, fetch metrics, manage costs. The connection uses two permission modes that control which AWS managed policy is attached to the role.

## Connection Modes

Stacktape supports two connection modes, set when the CloudFormation connection stack is created:

| Mode | Managed policy attached | Use case |
|------|------------------------|----------|
| **PRIVILEGED** | `arn:aws:iam::aws:policy/AdministratorAccess` | Full deployments — create, update, and delete any AWS resource. This is what the Stacktape Console and CLI use by default. |
| **BASIC** | `arn:aws:iam::aws:policy/ReadOnlyAccess` | Read-only access — view resources, logs, and metrics without the ability to modify infrastructure. |

The Stacktape Console connects accounts in **PRIVILEGED** mode by default because deployments require creating and managing CloudFormation stacks, Lambda functions, ECS services, databases, networking, IAM roles, and other resources. The BASIC mode exists for scenarios where you only need observability access without deployment capabilities.

<Warning>If you connect in BASIC mode, Stacktape cannot deploy or delete stacks in that account. You'll only be able to view existing resources, logs, and metrics.</Warning>

## Privileged Mode Permissions

When connected in PRIVILEGED mode, the role receives the AWS-managed `AdministratorAccess` policy. This grants unrestricted access to all AWS services and resources in the account. Stacktape needs this breadth because a single deployment can create resources across many AWS services — CloudFormation, Lambda, ECS, RDS, DynamoDB, S3, CloudFront, API Gateway, IAM, VPC, ACM, Route 53, CloudWatch, Secrets Manager, Systems Manager, and others depending on your configuration.

### Why AdministratorAccess?

Stacktape provisions infrastructure-as-code through CloudFormation. A typical stack touches 10-30 AWS services depending on the resources defined in your configuration. Restricting to a narrow IAM policy would break unpredictably depending on which Stacktape resources you use. The AdministratorAccess approach ensures any valid Stacktape configuration can deploy without permission errors.

This is the same approach used by other infrastructure-as-code tools (AWS CDK, Terraform Cloud, Pulumi) when given cross-account deployment access.

### What Stacktape uses the permissions for

Stacktape assumes the cross-account role to:

- **Deploy and manage infrastructure** — create, update, and delete CloudFormation stacks containing your resources (Lambda functions, containers, databases, load balancers, etc.)
- **Read logs and metrics** — fetch CloudWatch logs and metrics for display in the Console and CLI
- **Manage secrets** — read and write values in AWS Secrets Manager and SSM Parameter Store
- **Cost reporting** — read Cost and Usage Reports for the cost dashboards
- **Development mode** — invoke Lambda functions and interact with resources during `stacktape dev`
- **Debugging** — execute commands in containers, open bastion tunnels, query databases

## Reporting Permissions (Both Modes)

In addition to the managed policy, both connection modes receive an inline policy called `StacktapeReporting` that grants permissions for cost tracking and stack discovery:

| Permission | Resource scope | Purpose |
|-----------|---------------|---------|
| `s3:GetObject`, `s3:GetObjectAcl`, `s3:ListBucket`, `s3:PutObject` | The connection's Cost and Usage Report S3 bucket | Read and write cost report data |
| `cloudformation:DescribeStacks`, `cloudformation:ListStackResources`, `cloudformation:ListStacks` | All (`*`) | Discover deployed stacks and their resources |
| `cur:PutReportDefinition`, `cur:ModifyReportDefinition`, `cur:DeleteReportDefinition` | The connection's specific CUR definition | Manage the Cost and Usage Report that feeds the cost dashboards |
| `ec2:DescribeRegions` | All (`*`) | List available AWS regions for the account |
| `ce:UpdateCostAllocationTagsStatus`, `ce:ListCostAllocationTags` | All (`*`) | Enable cost allocation tags for per-resource cost attribution |

These permissions are scoped as narrowly as possible — S3 and CUR actions are restricted to the specific bucket and report definition created by the connection stack.

## Security Model

The cross-account connection uses AWS security best practices:

### Temporary credentials only

Stacktape never stores or receives long-term AWS access keys. Each time Stacktape interacts with your account, it calls `sts:AssumeRole` to request temporary credentials. The maximum session duration is 12 hours, and most operations use shorter sessions.

### External ID protection

The role's trust policy requires a unique `ExternalId` (the connection ID) to prevent confused deputy attacks. Even if an attacker knows the role ARN, they cannot assume it without the correct external ID stored in Stacktape's backend.

### Single trusted principal

The trust policy allows only Stacktape's AWS account (`977946299200`) to assume the role. No other AWS account or service can use it.

### Isolated cost reporting bucket

The connection stack creates a dedicated, private S3 bucket for Cost and Usage Reports with:
- All public access blocked (`BlockPublicAcls`, `BlockPublicPolicy`, `IgnorePublicAcls`, `RestrictPublicBuckets`)
- Objects expire after 30 days via lifecycle rules
- Only the AWS billing service and the Stacktape role can access it

## Restricting Permissions

If your organization requires tighter controls than `AdministratorAccess`, you have two options:

### AWS Service Control Policies (SCPs)

Apply SCPs at the AWS Organizations level to restrict what any role — including Stacktape's — can do in the account. For example, you can block access to specific regions, prevent deletion of certain resource types, or deny actions on tagged resources. SCPs act as a ceiling on permissions regardless of what the role's policy allows.

<Warning>Narrowing permissions via SCPs may prevent certain Stacktape features from working. If a deployment fails with an `AccessDenied` error, check whether an SCP is blocking the required action.</Warning>

### Separate accounts for different stages

The most common approach is to connect separate AWS accounts — a production account with stricter SCPs and a development account with fewer restrictions. This provides blast-radius isolation without limiting developer velocity.

## Revoking Access

You can revoke Stacktape's access to your AWS account at any time through two paths:

- **Delete the connection in the Stacktape Console** — removes the connection from Stacktape's side. The CloudFormation stack and IAM role remain in your AWS account until you delete them manually.
- **Delete the CloudFormation connection stack in the AWS Console** — removes the IAM role and cost reporting bucket from your account. The connected account transitions to BLOCKED in Stacktape automatically.

Either method is immediate. After revocation, Stacktape cannot interact with that AWS account until you create a new connection.

## What the Connection Stack Creates

The CloudFormation stack (named `stacktape-{connectionId}`, deployed in eu-west-1) provisions four resources:

| Resource | Type | Purpose |
|----------|------|---------|
| `StacktapeRole` | `AWS::IAM::Role` | The cross-account role Stacktape assumes |
| `CostAndUsageReportBucket` | `AWS::S3::Bucket` | Private bucket for cost report storage |
| `CostAndUsageReportBucketPolicy` | `AWS::S3::BucketPolicy` | Grants billing service and Stacktape access to the bucket |
| `ReportingIAMPolicy` | `AWS::IAM::Policy` | Inline policy for cost reporting and stack discovery |

The stack also triggers a custom resource (`StacktapeHandleConnection`) that notifies Stacktape's backend when the connection is ready, enabling the automatic PENDING → ACTIVE transition in the Console.

## FAQ

### Does Stacktape store my AWS credentials?

No. Stacktape uses cross-account IAM role assumption with temporary credentials. Each time Stacktape needs to interact with your account, it calls AWS STS to assume the role and receives short-lived credentials. No long-term access keys are generated, stored, or exchanged.

### Why does Stacktape need AdministratorAccess?

Stacktape deploys infrastructure-as-code through CloudFormation. A typical deployment touches 10-30 different AWS services (Lambda, ECS, RDS, S3, CloudFront, API Gateway, IAM, VPC, and others). A narrow, service-specific policy would break unpredictably depending on which resources you configure. AdministratorAccess ensures any valid configuration deploys without permission errors — the same approach used by AWS CDK, Terraform Cloud, and other IaC platforms.

### Can I use a custom IAM policy instead of AdministratorAccess?

The connection stack uses the AWS-managed `AdministratorAccess` policy in PRIVILEGED mode. You cannot customize the policy attached to the role directly. However, you can layer AWS Service Control Policies (SCPs) on top to restrict specific actions at the organization level. Be aware that overly restrictive SCPs may cause deployment failures.

### What is the BASIC connection mode?

BASIC mode attaches `ReadOnlyAccess` instead of `AdministratorAccess`. It allows viewing resources, logs, and metrics but cannot deploy or modify infrastructure. Use BASIC when you want observability access to an account without granting deployment permissions — for example, a monitoring-only connection to a production account managed by a separate team.

### Is the connection free?

Yes. The CloudFormation stack creates an IAM role and an S3 bucket — both at no AWS cost for the resources themselves. The S3 bucket stores Cost and Usage Reports, which may incur minimal storage costs (objects expire after 30 days). You only pay for the AWS resources that Stacktape deploys into your account.

### What happens to existing resources if I revoke access?

Revoking access does not delete or modify any AWS resources in your account. Your deployed stacks, databases, Lambda functions, and other resources continue running. You simply lose the ability to manage them through Stacktape until you create a new connection.

### Can Stacktape access other AWS accounts I own?

No. The cross-account role is scoped to the specific AWS account where the connection stack was created. Stacktape can only assume the role in that one account. Each additional AWS account requires its own connection setup.

### What is the ExternalId and why does it matter?

The ExternalId is a unique identifier (the connection ID) required in the role's trust policy. It prevents confused deputy attacks — a class of security vulnerability where a malicious third party tricks a service into acting on an unintended target. Even knowing the role ARN is not enough to assume it without the correct ExternalId, which is stored securely in Stacktape's backend.

### How do I audit what Stacktape does in my account?

Enable AWS CloudTrail in your account. Every API call made by the assumed role appears in CloudTrail logs with the role session name `stacktape`. You can filter CloudTrail events by the role ARN (`arn:aws:iam::{your-account-id}:role/stacktape-{connectionId}`) to see exactly what actions Stacktape performed and when.

### Why is the connection stack in eu-west-1 if I deploy to other regions?

The connection stack only creates global resources (an IAM role and an S3 bucket for cost reports). IAM roles are global — once created in any region, they work across all regions. The eu-west-1 requirement is a deployment convention for the connection infrastructure. After connection, your projects deploy to whichever region you specify in your Stacktape configuration.
