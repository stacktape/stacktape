# Stacktape Buildpack for Containers

The Stacktape container buildpack (`StacktapeImageBuildpackPackaging`) builds an optimized OCI container image directly from your source code — no Dockerfile required. It applies to [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload), and [batch jobs](/resources/compute/batch-job). Stacktape handles bundling, dependency installation, and image creation for JavaScript, TypeScript, Python, Java, Go, Ruby, PHP, and .NET. The resulting image is pushed to a managed AWS ECR repository in your account.

## When to use

Use the Stacktape container buildpack when you want to deploy containerized workloads without writing or maintaining a Dockerfile. It is the best default for most teams starting with Stacktape container resources, especially for single-language applications in JavaScript/TypeScript, Python, or Java.

Concrete scenarios where the buildpack fits well:

- **Standard web APIs** — Express, Hono, Fastify, Flask, FastAPI, Django, or Spring Boot apps where the default Alpine-based image works.
- **Background workers** — worker services and batch jobs that process messages from queues or run scheduled tasks.
- **Rapid prototyping** — get a working container deployment with just an entry file path.
- **Teams without Docker expertise** — the buildpack handles base images, dependency installation, and image layering automatically.

## When NOT to use

The buildpack is not the right choice when you need full control over the container image or when your project has requirements outside the buildpack's scope:

- **Complex multi-stage builds** — if you need build-time caching across stages, intermediate build layers, or non-standard build toolchains, use a [custom Dockerfile](/packaging/containers/custom-dockerfile).
- **Existing container images** — if your CI pipeline already produces container images, use a [prebuilt image](/packaging/containers/prebuilt-image) to skip the build step entirely.
- **Unsupported languages** — Rust, Elixir, and other languages not in the supported list require a [custom Dockerfile](/packaging/containers/custom-dockerfile) or [Nixpacks](/packaging/containers/nixpacks).
- **Specialized base images** — if you need a specific OS distribution, GPU drivers, or compliance-certified base images, a custom Dockerfile gives you that control.

## Basic example

The smallest valid configuration points the buildpack at your entry file. Stacktape detects the language, bundles your code, installs dependencies, builds the container image, and pushes it to a managed ECR repository.

      api: {
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/server.ts'
        })

    },

  };
});

    }
  ]}
/>

The `entryfilePath` is the only required property. It points to your application's entry file, relative to your Stacktape configuration file. For JavaScript and TypeScript, the buildpack bundles your code into a single file using esbuild. For Python, Java, Go, and other languages, it packages the source with its dependencies.

## Supported languages

The buildpack supports eight languages. The optional `languageSpecificConfig` property lets you tune behavior per language. When omitted, Stacktape applies defaults based on the detected language.

| Language | Default version | Key options |
|---|---|---|
| JavaScript / TypeScript | Node.js 18 | Module format, bundle exclusions, source maps, decorator metadata |
| Python | 3.9 | WSGI/ASGI server mode, dependency file, minification |
| Java | 11 | Maven or Gradle, build file path |
| Go | — | — |
| Ruby | 3.3 | Version selection |
| PHP | 8.3 | Version selection |
| .NET | 8 | Version selection, project file path |

## JavaScript and TypeScript

For JavaScript and TypeScript projects, the buildpack uses esbuild to bundle your code into a single file. Dependencies with native binaries (like `bcrypt` or `sharp`) are detected and installed separately in the container rather than bundled. Source maps are generated by default.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const api = new WebService({
    containers: {
      api: {
        // [!code focus-start]
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/server.ts',
          languageSpecificConfig: {
            nodeVersion: 22,
            outputModuleFormat: 'esm',
            emitTsDecoratorMetadata: true,
            dependenciesToExcludeFromBundle: ['@prisma/client']
          }
        })
        // [!code focus-end]
      }
    },
    resources: {
      cpu: 0.25,
      memory: 512
    }
  });

  return {
    resources: { api }
  };
});
`
    }
  ]}
/>

### Module format

The `outputModuleFormat` property controls whether the bundle output is CommonJS (`'cjs'`, the default) or ES Modules (`'esm'`). ESM enables top-level `await`, but some npm packages don't support it and error stack traces may be less readable. Stick with `'cjs'` unless you specifically need ESM features.

### Node.js version

Set `nodeVersion` to choose the Node.js major version used in the container. Supported values: `16`, `17`, `18`, `19`, `20`, `21`, `22`, `23`, `24`. The default is `18`. For new projects, use `20` or `22` (the current LTS versions).

### TypeScript decorator metadata

Set `emitTsDecoratorMetadata: true` if your project uses TypeScript decorators with runtime metadata — required by NestJS, TypeORM, and similar frameworks that rely on `reflect-metadata`.

### Excluding dependencies from the bundle

Use `dependenciesToExcludeFromBundle` to keep specific packages out of the esbuild bundle. Excluded dependencies are treated as external and installed separately in the container's `node_modules`. This is necessary for packages that don't work when bundled — ORMs with dynamic imports, packages that use `__dirname` to find assets, or native add-ons that need to be loaded at runtime. Set the value to `['*']` to exclude all dependencies from bundling.

The separate `dependenciesToExcludeFromDeploymentPackage` property removes packages from the deployment package entirely — use it only for packages that are never needed at runtime. Set it to `['*']` to exclude all non-bundled dependencies.

### TypeScript path aliases

Set `tsConfigPath` to point to your `tsconfig.json` file. The buildpack reads it to resolve path aliases (like `@/src/...`) during the esbuild bundling step.

### Source maps

Source maps are generated by default. Set `disableSourceMaps: true` to skip them — this reduces image size but makes production errors harder to debug. Alternatively, use `outputSourceMapsTo` to save source maps to a local directory for uploading to external error tracking services (Sentry, Datadog), without including them in the container image.

## Python

For Python containers, the buildpack installs dependencies and optionally configures a production web server. Set `entryfilePath` to the `module/file.py:app` format when using `runAppAs` for web applications.

### WSGI and ASGI server mode

The `runAppAs` property is specific to the container buildpack. Set it to `'ASGI'` for async frameworks like FastAPI and Starlette, or `'WSGI'` for sync frameworks like Flask and Django. When set, the buildpack configures a production server that auto-binds to the `PORT` environment variable provided by Stacktape.

The `entryfilePath` must use the `module/file.py:app` format when `runAppAs` is set — for example, `./app/main.py:app` where `app` is the WSGI or ASGI application object in `main.py`.

### Python version

Supported Python versions: `2.7`, `3.6`, `3.7`, `3.8`, `3.9`, `3.11`, `3.12`, `3.13`, `3.14`. The default is `3.9`. For new projects, use `3.12` or later.

### Dependency management

Set `packageManagerFile` to point to your `requirements.txt`, `Pipfile`, or `pyproject.toml`. Stacktape uses `uv` for dependency resolution and installation.

### Minification

The `minify` property controls whether Python source files are minified to reduce image size. Minification makes production stack traces harder to read. Disable it with `minify: false` if you need readable tracebacks in production.

## Java

For Java projects, the buildpack uses Gradle by default. Set `useMaven: true` to switch to Maven. Use `packageManagerFile` to point to your `pom.xml` (Maven) or `build.gradle` (Gradle) file.

## Go, Ruby, PHP, and .NET

These languages are supported with minimal configuration:

- **Go** — no language-specific options. Point `entryfilePath` to your `main.go` file.
- **Ruby** — set `rubyVersion` to `3.2` or `3.3` (default: `3.3`).
- **PHP** — set `phpVersion` to `8.2` or `8.3` (default: `8.3`).
- **.NET** — set `dotnetVersion` to `6` or `8` (default: `8`) and `projectFile` to your `.csproj` path.

## Customizing the container image

### Installing system dependencies

Use `customDockerBuildCommands` to run shell commands during the Docker image build. Each string in the array becomes a `RUN` directive in the generated Dockerfile. This is the escape hatch for installing system packages, adding configuration files, or running build-time setup.

<CodeBlock
  intellisense

export default defineConfig(() => {
  const api = new WebService({
    containers: {
      api: {
        // [!code focus-start]
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/server.ts',

        // [!code focus-end]
      }
    },

      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { api }
  };
});
`
    }
  ]}
/>

<Info>
The default base image uses Alpine Linux, so use `apk` for package installation. If you enable `requiresGlibcBinaries`, the base switches to a Debian-based image — use `apt-get` instead.
</Info>

### Using glibc instead of Alpine musl

By default, the buildpack uses Alpine Linux with musl libc, producing smaller container images. Some native dependencies require glibc to work correctly. Set `requiresGlibcBinaries: true` to switch to a glibc-based (Debian) image.

Common packages that require glibc: `sharp`, `canvas`, `bcrypt`, `puppeteer`.

## Controlling packaged files

Use `includeFiles` and `excludeFiles` to control which files end up in the container image. Both accept glob patterns relative to your Stacktape configuration file. Use `excludeDependencies` to remove specific packages from the container entirely.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    containers: {
      worker: {
        // [!code focus-start]
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/worker.ts',
          includeFiles: ['./config/**', './templates/**'],
          excludeFiles: ['./tests/**', './**/*.test.ts'],
          excludeDependencies: ['jest', '@types/jest']
        })
        // [!code focus-end]
      }
    },
    resources: {
      cpu: 0.25,
      memory: 512
    }
  });

  return {
    resources: { worker }
  };
});
`
    }
  ]}
/>

- **`includeFiles`** — explicitly include files that aren't automatically picked up by the bundler, such as config files, templates, or static assets your app loads at runtime.
- **`excludeFiles`** — exclude files you don't need in production: tests, documentation, development configs. Reduces image size and attack surface.
- **`excludeDependencies`** — remove specific packages from the container. Useful for stripping dev-only dependencies that shouldn't ship to production.

## Comparing container packaging modes

The Stacktape container buildpack is one of five container packaging modes. Each applies to the same resource types ([web service](/resources/compute/web-service), [private service](/resources/compute/private-service), [worker service](/resources/compute/worker-service), [multi-container workload](/resources/compute/multi-container-workload), [batch job](/resources/compute/batch-job)) — they differ only in how the container image is produced.

| | Stacktape buildpack | [Custom Dockerfile](/packaging/containers/custom-dockerfile) | [Prebuilt image](/packaging/containers/prebuilt-image) | [Nixpacks](/packaging/containers/nixpacks) | [External buildpack](/packaging/containers/external-buildpack) |
|---|---|---|---|---|---|
| Dockerfile needed | No | Yes | No | No | No |
| Configuration effort | Minimal | Full control | None | Minimal | Minimal |
| Image customization | Moderate | Full | N/A | Moderate | Moderate |
| Best for | Most apps | Complex setups | Existing CI images | Broad auto-detection | Cloud Native Buildpacks |

**Use the Stacktape buildpack** for the fastest path to a working container with good defaults. **Use a [custom Dockerfile](/packaging/containers/custom-dockerfile)** when you need full control over every layer — multi-stage builds, precise base image selection, or non-standard toolchains. **Use a [prebuilt image](/packaging/containers/prebuilt-image)** when your CI already produces images and you want Stacktape to deploy them as-is. **Use [Nixpacks](/packaging/containers/nixpacks)** for broad language auto-detection beyond the eight languages the Stacktape buildpack supports. **Use an [external buildpack](/packaging/containers/external-buildpack)** if your team already uses Cloud Native Buildpacks (e.g. Paketo).

## FAQ

### Which languages does the Stacktape container buildpack support?

The Stacktape container buildpack supports JavaScript, TypeScript, Python, Java, Go, Ruby, PHP, and .NET. JavaScript and TypeScript get the deepest integration — esbuild-based bundling, source maps, module format selection, and decorator metadata support. Python includes built-in WSGI/ASGI server binding. Java supports both Gradle and Maven build systems. For per-language tuning options, see the [language-specific config page](/packaging/function/language-specific-config).

### Do I need to write a Dockerfile to use the Stacktape buildpack?

No. The buildpack generates a Dockerfile internally and builds the container image for you. You only provide an `entryfilePath` pointing to your application's entry point. If you need to run custom shell commands during the build (for example, installing system packages), use `customDockerBuildCommands` instead of writing a full Dockerfile.

### When should I use the Stacktape buildpack instead of a custom Dockerfile?

Use the buildpack when your app is a standard single-language project with typical dependencies — it covers the common 80% of containerized applications with minimal configuration. Switch to a [custom Dockerfile](/packaging/containers/custom-dockerfile) when you need multi-stage builds, a non-standard base image, precise layer caching, or system-level setup that goes beyond what `customDockerBuildCommands` can handle. The buildpack saves time; a custom Dockerfile gives you full control.

### How does the Stacktape buildpack handle Python web servers?

Set `runAppAs` to `'ASGI'` (for FastAPI, Starlette) or `'WSGI'` (for Flask, Django) in `languageSpecificConfig`. The buildpack configures a production server that auto-binds to the `PORT` environment variable provided by Stacktape. Your `entryfilePath` must use the `module/file.py:app` format (e.g. `./app/main.py:app`), where `app` is the WSGI or ASGI application object. This feature is specific to the container buildpack and is not available for Lambda packaging.

### What base image does the Stacktape container buildpack use?

By default, the buildpack uses an Alpine Linux base image, which keeps containers small thanks to the musl libc library. If your native dependencies require glibc (common for packages like `sharp`, `canvas`, `bcrypt`, or `puppeteer`), set `requiresGlibcBinaries: true` to switch to a Debian-based image. The tradeoff is a larger image footprint.

### How do I install system-level packages in the container?

Use the `customDockerBuildCommands` property to run shell commands during the Docker build. Each command becomes a `RUN` directive in the generated Dockerfile. On the default Alpine image, use `apk add --no-cache <package>`. If `requiresGlibcBinaries` is enabled (Debian-based), use `apt-get update && apt-get install -y <package>`. This covers cases like installing `ffmpeg`, native crypto libraries, or database client tools.

### What is the difference between the Stacktape buildpack and Nixpacks?

Both build container images from source without a Dockerfile. The Stacktape buildpack supports eight languages with deeper integration — esbuild bundling for JS/TS, WSGI/ASGI binding for Python, and fine-grained dependency exclusion controls. [Nixpacks](/packaging/containers/nixpacks) is a general-purpose tool that auto-detects a broader set of languages and frameworks. Choose the Stacktape buildpack for tighter control over bundling and language-specific tuning. Choose Nixpacks when your language or framework isn't in the buildpack's supported list.

### Where are built container images stored?

Stacktape automatically provisions and manages a private AWS Elastic Container Registry (ECR) repository in your AWS account. Built images are pushed to this repository after each build. You don't need to create or configure the registry — Stacktape handles it. AWS ECR pricing is based on storage used and data transfer out.

### Can I use the Stacktape buildpack with a multi-container workload?

Yes. Each container in a [multi-container workload](/resources/compute/multi-container-workload) has its own `packaging` property. You can use the Stacktape buildpack for some containers and other packaging modes for others in the same workload. A common pattern is using the buildpack for the main application container while using a [prebuilt image](/packaging/containers/prebuilt-image) for sidecars like metrics exporters or log agents.

### How do I keep my container images small?

Start with the default Alpine base — don't enable `requiresGlibcBinaries` unless you hit glibc-specific runtime errors. Use `excludeFiles` to strip tests, docs, and dev configs from the image. For JavaScript/TypeScript, let esbuild bundle as much as possible (bundled dependencies are tree-shaken, producing smaller output than separately installed `node_modules`). Use `excludeDependencies` to drop dev-only packages. Avoid adding unnecessary system packages via `customDockerBuildCommands`.
