# Kafka Topics

A Kafka topic trigger invokes a [Lambda function](/resources/compute/lambda-function) whenever new messages arrive on a self-managed Apache Kafka topic. Stacktape creates an AWS Lambda event-source mapping that polls your Kafka brokers, batches records, and delivers them to your function — with automatic IAM permissions for the authentication secrets you provide.

## When to use

Use a Kafka topic trigger when you already run self-managed Kafka (on EC2, EKS, Confluent Cloud, Aiven, or any broker reachable over the network) and want a serverless consumer that scales automatically without managing consumer group infrastructure.

Common scenarios:

- Processing order events, clickstreams, or CDC records from a Kafka topic
- Bridging Kafka into AWS-native workflows (persist to DynamoDB, forward to SQS, trigger Step Functions)
- Running lightweight transformations or enrichments on each batch of Kafka records

## When NOT to use

- **Amazon MSK (managed Kafka)** — if your cluster is MSK, AWS provides a dedicated MSK event-source mapping. The `kafka-topic` trigger in Stacktape is for self-managed Kafka clusters specifically.
- **Simple message queuing** — if you don't already run Kafka, an [SQS queue trigger](/configuration/triggers/sqs-events) is simpler and cheaper for standard async processing.
- **Fan-out to multiple consumers** — Kafka handles fan-out natively via consumer groups, but if your architecture is AWS-native, [SNS](/configuration/triggers/sns-events) or [EventBridge](/configuration/triggers/event-bus-events) is typically easier to operate.

This example creates a Lambda function that consumes messages from a Kafka topic using SASL/SCRAM-512 authentication.

The `bootstrapServers` array must contain at least one `host:port` endpoint for your Kafka cluster. Stacktape uses these to establish the initial connection and discover the full broker topology. The `topicName` identifies which topic to consume from.

## Authentication

Kafka topic triggers require authentication — there is no unauthenticated mode. Stacktape supports two authentication methods, and automatically grants your Lambda function's IAM role `secretsmanager:GetSecretValue` permission on the referenced secret ARNs.

### SASL authentication

SASL (Simple Authentication and Security Layer) authenticates with a username and password stored in AWS Secrets Manager. Three protocol variants are available:

| Type value | Protocol | When to use |
|---|---|---|
| `BASIC_AUTH` | SASL/PLAIN | Development or clusters that only support plaintext SASL |
| `SASL_SCRAM_256_AUTH` | SCRAM-SHA-256 | Production clusters with SCRAM-256 enabled |
| `SASL_SCRAM_512_AUTH` | SCRAM-SHA-512 | Production clusters with SCRAM-512 (strongest, recommended) |

The referenced secret must be a JSON object with `username` and `password` keys. You can create it using the [`stacktape secret:create`](/cli/secret-create) CLI command.

### Mutual TLS (MTLS) authentication

MTLS authenticates using a client-side TLS certificate. Use this when your Kafka cluster requires certificate-based identity verification. The `clientCertificate` secret must contain the certificate chain (X.509 PEM), private key (PKCS#8 PEM), and an optional private key password. Optionally provide a `serverRootCaCertificate` for clusters with self-signed or private CA certificates.

<Info>
The `serverRootCaCertificate` is optional. Omit it when your brokers use certificates issued by a publicly trusted CA. Include it when brokers use a private or self-signed CA.
</Info>

## Batching and throughput

Kafka topic triggers use AWS Lambda's event-source mapping to poll your brokers and deliver messages in batches. Two properties control how batches are formed:

| Property | Default | Maximum | Effect |
|---|---|---|---|
| `batchSize` | 100 | 10,000 | Maximum records delivered per invocation |
| `maxBatchWindowSeconds` | 0.5 | 300 | Maximum seconds to wait before invoking with whatever records are available |

Your function is invoked when either threshold is reached — whichever comes first. A larger `batchSize` reduces the number of invocations (lower cost) but increases per-invocation latency and memory usage. A longer `maxBatchWindowSeconds` accumulates more records into each batch, which is useful for low-throughput topics where you want to avoid many small invocations.

For high-throughput topics (thousands of messages per second), increase `batchSize` toward 1,000–5,000 and set `maxBatchWindowSeconds` to 5–10 seconds. For low-latency requirements, keep the defaults — 0.5 seconds ensures messages are processed quickly even on quiet topics.

<Warning>
If your function fails (throws an error or times out), the entire batch is retried. Make your handler idempotent — duplicate processing of the same record should produce the same result.
</Warning>

## Event payload

Your Lambda function receives a batch of Kafka records as an event object. Each record includes the topic, partition, offset, timestamp, key, and value. Values are base64-encoded.

```typescript
export const handler = async (event: any) => {
  for (const [topicPartition, records] of Object.entries(event.records)) {
    for (const record of records as any[]) {
      const value = Buffer.from(record.value, 'base64').toString('utf-8');
      const key = record.key ? Buffer.from(record.key, 'base64').toString('utf-8') : null;
      console.info(`Partition: ${record.partition}, Offset: ${record.offset}, Key: ${key}, Value: ${value}`);
    }
  }
};
```

The `event.records` object is keyed by `topic-partition` (e.g., `order-events-0`, `order-events-1`). Each record contains `topic`, `partition`, `offset`, `timestamp`, `timestampType`, `key`, `value`, and `headers`.

## Network connectivity

Your Kafka brokers must be reachable from AWS Lambda's execution environment. For public Kafka clusters (Confluent Cloud, Aiven, or brokers with public endpoints), no additional networking configuration is needed.

For private Kafka clusters (running inside a VPC, on-premises, or behind a VPN), configure your Lambda function with VPC access using `joinDefaultVpc: true` or by referencing a custom VPC. Ensure security groups and routing allow outbound connectivity to your broker endpoints on the configured ports.

## FAQ

### Can I consume from multiple Kafka topics with one Lambda function?

Yes. Add multiple `KafkaTopicIntegration` entries to your function's `events` array, each pointing to a different topic (or even different clusters). Stacktape creates a separate event-source mapping for each entry.

### Does the Kafka trigger support Amazon MSK?

The `kafka-topic` trigger type is designed for self-managed Kafka clusters. AWS provides a separate, dedicated MSK event-source mapping type that is aware of MSK-specific features (IAM auth, cluster ARN discovery). If you run MSK, check whether a dedicated MSK integration is available in your Stacktape version.

### How does scaling work for Kafka consumers?

AWS Lambda automatically scales the number of concurrent invocations based on the number of partitions in your topic. Each partition is processed by at most one concurrent invocation at a time. To increase parallelism, increase the number of partitions on your Kafka topic.

### What happens if my function fails?

If your function returns an error or times out, the same batch of records is retried. Unlike [SQS triggers](/configuration/triggers/sqs-events) or [Kinesis triggers](/configuration/triggers/kinesis-events), the Kafka topic trigger does not expose `maximumRetryAttempts` or `onFailure` destination properties. Failed records block the partition until they succeed or are skipped. Design your handler to be idempotent and consider implementing dead-letter logic within your function code.

### How do I create the Secrets Manager secret for authentication?

Use the Stacktape CLI to create a secret containing your Kafka credentials.

```bash
stacktape secret:create --name kafka-creds --value '{"username":"my-user","password":"my-password"}' --region eu-west-1
```

The command returns an ARN — use that ARN as the `authenticationSecretArn` value in your trigger configuration.

### What's the difference between SASL and MTLS authentication?

SASL authenticates with a username and password — simpler to set up, widely supported by managed Kafka providers (Confluent Cloud, Aiven). MTLS authenticates with a client certificate — stronger identity guarantees, typically used in enterprise or zero-trust environments. Choose based on what your Kafka cluster supports and your security requirements.

### Is there a cost for the Kafka event-source mapping?

The event-source mapping itself has no additional AWS charge beyond the Lambda invocation cost. You pay for the compute time of each invocation (duration x memory) and for Secrets Manager API calls to retrieve credentials. Secrets Manager charges per secret per month (~$0.40) plus per 10,000 API calls ($0.05).

### Can I filter messages before they reach my function?

The Kafka topic trigger does not support server-side message filtering (unlike [SQS](/configuration/triggers/sqs-events) or [SNS](/configuration/triggers/sns-events) triggers that support filter policies). All messages on the subscribed topic are delivered to your function. Implement filtering logic inside your handler if you need to skip certain records.

### How do I monitor my Kafka consumer?

Use [`stacktape debug:logs`](/cli/debug-logs) to tail your function's CloudWatch logs in real time. AWS also publishes metrics for the event-source mapping including `IteratorAge` (how far behind the consumer is) and `Invocations` — view these in the Stacktape Console or with [`stacktape debug:metrics`](/cli/debug-metrics).
