# Debug Commands Reference

Stacktape debug commands let you inspect logs, metrics, alarms, databases, and containers in a deployed stack directly from your terminal. Every command operates against a live AWS stack — you specify the target with `--projectName`, `--stage`, and `--region`, then point at a specific resource with `--resourceName`. All debug commands are read-only and do not modify your stack.

## Command overview

| Command | Purpose | Target resource types |
|---------|---------|---------------------|
| [`debug:logs`](/cli/debug-logs) | View CloudWatch logs | Lambda functions, container workloads, batch jobs |
| [`debug:metrics`](/cli/debug-metrics) | View CloudWatch metrics | Lambda functions, container workloads, databases, HTTP API gateways |
| [`debug:alarms`](/cli/debug-alarms) | List alarm states | All resources with configured alarms |
| [`debug:sql`](/cli/debug-sql) | Run read-only SQL queries | Relational databases (PostgreSQL, MySQL) |
| [`debug:dynamodb`](/cli/debug-dynamodb) | Query DynamoDB tables | DynamoDB tables |
| [`debug:redis`](/cli/debug-redis) | Inspect Redis keys and info | Redis clusters |
| [`debug:opensearch`](/cli/debug-opensearch) | Query OpenSearch indices | OpenSearch domains |
| [`debug:aws-sdk`](/cli/debug-aws-sdk) | Run read-only AWS SDK calls | Any AWS resource |
| [`debug:container-exec`](/cli/debug-container-exec) | Execute a command in a container | Web services, private services, worker services, multi-container workloads |
| [`container:session`](/cli/container-session) | Open an interactive shell | Web services, private services, worker services, multi-container workloads |
| [`bastion:tunnel`](/cli/bastion-tunnel) | Open a port-forwarding tunnel | Relational databases, Redis clusters, ALBs, private services |
| [`bastion:session`](/cli/bastion-session) | SSH into a bastion host | Bastion hosts |

<Info>All debug commands require a deployed stack. They work on any deployed stage — not just dev mode stacks.</Info>

## Shared flags

Every debug command accepts these standard flags for identifying the target stack:

- `--projectName` — the Stacktape project name
- `--stage` — the deployed stage name
- `--region` — the AWS region where the stack is deployed
- `--resourceName` — the name of the resource (as declared in your config) to target

## Viewing logs

The [`stacktape debug:logs`](/cli/debug-logs) command fetches CloudWatch log events for a specific resource. It supports Lambda functions, container workloads (web services, private services, worker services, multi-container workloads), and batch jobs.

### Basic usage

Fetch the last hour of logs for a resource:

```bash
stacktape debug:logs --projectName myProject --stage prod --region eu-west-1 --resourceName api
```

### Time range

Use `--startTime` with relative notation (`1h`, `30m`, `1d`, `5s`) or ISO timestamps. The default window is 1 hour from the current time.

Fetch the last 30 minutes:

```bash
stacktape debug:logs --resourceName api --startTime 30m
```

Fetch logs between two absolute timestamps:

```bash
stacktape debug:logs --resourceName api --startTime "2025-01-15T10:00:00Z" --endTime "2025-01-15T11:00:00Z"
```

### Filtering

Use `--filter` to pass a CloudWatch Logs filter pattern. This uses the standard CloudWatch filter syntax.

Filter for lines containing "ERROR":

```bash
stacktape debug:logs --resourceName api --filter "ERROR"
```

Filter JSON structured logs by a field:

```bash
stacktape debug:logs --resourceName api --filter '{ $.level = "error" }'
```

### CloudWatch Logs Insights queries

Use `--query` to run a full CloudWatch Logs Insights query against the resource's log group. This is more expressive than `--filter` and supports aggregations, sorting, and field extraction.

```bash
stacktape debug:logs --resourceName api --query "fields @timestamp, @message | filter @message like /ERROR/ | sort @timestamp desc | limit 20"
```

### Other flags

- `--limit` — maximum number of events to return (default: `100`)
- `--container` — for multi-container workloads, specify which container's logs to fetch
- `--raw` — output raw JSON instead of formatted, color-coded output

The formatted output color-codes log lines by severity: red for `ERROR`/`FATAL`/`CRITICAL`, yellow for `WARN`, gray for `DEBUG`.

## Viewing metrics

The [`stacktape debug:metrics`](/cli/debug-metrics) command fetches CloudWatch metric data and renders a sparkline chart in your terminal. It supports four resource types, each with a set of available metrics.

### Supported resource types and metrics

| Resource type | Available metrics |
|--------------|-------------------|
| `function` (Lambda) | `Invocations`, `Errors`, `Duration`, `Throttles`, `ConcurrentExecutions` |
| `multi-container-workload` | `CPUUtilization`, `MemoryUtilization` |
| `relational-database` | `CPUUtilization`, `DatabaseConnections`, `FreeStorageSpace`, `ReadLatency`, `WriteLatency` |
| `http-api-gateway` | `Count`, `4XXError`, `5XXError`, `Latency`, `IntegrationLatency` |

<Info>Web services, private services, and worker services are container workloads under the hood — use their `multi-container-workload` metrics.</Info>

### Basic usage

View Lambda invocations over the last hour:

```bash
stacktape debug:metrics --resourceName api --metric Invocations
```

### Customizing the query

- `--period` — aggregation interval in seconds (default: `300`, i.e. 5 minutes)
- `--stat` — aggregation statistic: `Average`, `Sum`, `Minimum`, `Maximum`, `SampleCount` (default: `Average`)
- `--startTime` — supports relative time (`1h`, `6h`, `1d`) or ISO timestamp (default: 1 hour ago)
- `--endTime` — end of the time window (default: now)

View database CPU over the last 24 hours with 1-hour granularity:

```bash
stacktape debug:metrics --resourceName mainDb --metric CPUUtilization --startTime 1d --period 3600
```

The terminal output shows a sparkline visualization along with min, max, average, and sum statistics.

## Viewing alarms

The [`stacktape debug:alarms`](/cli/debug-alarms) command lists all CloudWatch alarms associated with your stack and shows their current state.

### Basic usage

List all alarms in the stack:

```bash
stacktape debug:alarms --projectName myProject --stage prod --region eu-west-1
```

### Filtering

Filter by resource name to see only alarms for a specific resource:

```bash
stacktape debug:alarms --resourceName api
```

Filter by alarm state (`ALARM`, `OK`, or `INSUFFICIENT_DATA`):

```bash
stacktape debug:alarms --state ALARM
```

The output shows alarm name, associated resource, current state, metric, threshold, comparison operator, and last state change. For alarms in `ALARM` state, the state reason is included.

## Querying SQL databases

The [`stacktape debug:sql`](/cli/debug-sql) command executes read-only SQL queries against deployed [relational databases](/resources/databases/relational-database). It supports both PostgreSQL and MySQL engines and automatically detects the protocol from the connection string.

### Basic usage

```bash
stacktape debug:sql --resourceName mainDb --sql "SELECT * FROM users LIMIT 10"
```

### Read-only enforcement

Only these SQL statement prefixes are allowed: `SELECT`, `SHOW`, `DESCRIBE`, `EXPLAIN`. Any write operation (`INSERT`, `UPDATE`, `DELETE`, `DROP`, etc.) is rejected before the query reaches the database.

### VPC-only databases

If your database is in a private VPC without public access, use `--bastionResource` to route the connection through a [bastion host](/resources/security/bastion-host):

```bash
stacktape debug:sql --resourceName mainDb --bastionResource bastion --sql "SELECT count(*) FROM orders"
```

Stacktape automatically establishes an SSM port-forwarding tunnel, connects through it, and tears it down after the query completes.

### Other flags

- `--limit` — maximum rows returned (default: `1000`)
- `--timeout` — query timeout in milliseconds (default: `30000`)

The connection uses SSL with `rejectUnauthorized: false` to work with RDS self-signed certificates.

## Querying DynamoDB

The [`stacktape debug:dynamodb`](/cli/debug-dynamodb) command lets you inspect [DynamoDB table](/resources/databases/dynamodb) contents and schema without leaving the CLI.

### Operations

| Operation | Description | Required flags |
|-----------|-------------|---------------|
| `sample` (default) | Fetch sample items | `--resourceName` |
| `scan` | Scan the table | `--resourceName` |
| `query` | Query by partition key | `--resourceName`, `--pk` |
| `get` | Get a single item | `--resourceName`, `--pk` |
| `schema` | Show table key schema, GSIs, and LSIs | `--resourceName` |

### Examples

Get sample items from a table:

```bash
stacktape debug:dynamodb --resourceName ordersTable --operation sample --limit 5
```

Query by partition key:

```bash
stacktape debug:dynamodb --resourceName ordersTable --operation query --pk '{"userId": "user-123"}'
```

Query a global secondary index:

```bash
stacktape debug:dynamodb --resourceName ordersTable --operation query --pk '{"status": "pending"}' --index statusIndex
```

Get a specific item by primary key (partition + sort key):

```bash
stacktape debug:dynamodb --resourceName ordersTable --operation get --pk '{"userId": "user-123"}' --sk '{"orderId": "order-456"}'
```

### Flags

- `--operation` — one of `scan`, `query`, `get`, `schema`, `sample` (default: `sample`)
- `--pk` — partition key as JSON (required for `query` and `get`)
- `--sk` — sort key as JSON (optional, for composite keys)
- `--index` — name of a global secondary index to query
- `--limit` — maximum items returned (default: `100`)

## Querying Redis

The [`stacktape debug:redis`](/cli/debug-redis) command inspects keys and server info on a deployed [Redis cluster](/resources/databases/redis). Since Redis clusters live in a VPC, you typically need a `--bastionResource` to establish a tunnel.

### Operations

| Operation | Description | Required flags |
|-----------|-------------|---------------|
| `info` (default) | Redis server info | `--resourceName` |
| `keys` | List keys matching a pattern | `--resourceName` |
| `get` | Get a key's value | `--resourceName`, `--key` |
| `ttl` | Get a key's TTL | `--resourceName`, `--key` |
| `type` | Get a key's data type | `--resourceName`, `--key` |

### Examples

Get server info (memory, clients, replication):

```bash
stacktape debug:redis --resourceName cache --bastionResource bastion --operation info
```

List all keys matching a pattern:

```bash
stacktape debug:redis --resourceName cache --bastionResource bastion --operation keys --pattern "session:*"
```

Get a specific key's value:

```bash
stacktape debug:redis --resourceName cache --bastionResource bastion --operation get --key "session:abc123"
```

### Flags

- `--operation` — one of `keys`, `get`, `ttl`, `info`, `type` (default: `info`)
- `--key` — Redis key name (required for `get`, `ttl`, `type`)
- `--pattern` — glob pattern for key listing (default: `*`)
- `--section` — specific Redis INFO section (e.g. `memory`, `clients`, `replication`)
- `--limit` — maximum keys returned for `keys` operation (default: `100`)
- `--bastionResource` — bastion host to tunnel through (typically required)

## Querying OpenSearch

The [`stacktape debug:opensearch`](/cli/debug-opensearch) command queries deployed [OpenSearch domains](/resources/databases/opensearch) using the OpenSearch Query DSL.

### Operations

| Operation | Description | Required flags |
|-----------|-------------|---------------|
| `indices` (default) | List all indices | `--resourceName` |
| `count` | Count documents in an index | `--resourceName`, `--index` |
| `mapping` | Show index mapping | `--resourceName`, `--index` |
| `get` | Get a document by ID | `--resourceName`, `--index`, `--id` |
| `search` | Run a query DSL search | `--resourceName`, `--query` |

### Examples

List all indices:

```bash
stacktape debug:opensearch --resourceName search --operation indices
```

Search with query DSL:

```bash
stacktape debug:opensearch --resourceName search --operation search --query '{"match": {"title": "hello"}}' --index articles --limit 20
```

Get a specific document:

```bash
stacktape debug:opensearch --resourceName search --operation get --index articles --id "doc-123"
```

### Flags

- `--operation` — one of `search`, `get`, `indices`, `mapping`, `count` (default: `indices`)
- `--index` — index name (required for `mapping`, `count`, `get`)
- `--id` — document ID (required for `get`)
- `--query` — JSON query DSL string (required for `search`)
- `--limit` — max results for search (default: `10`)

## Running AWS SDK commands

The [`stacktape debug:aws-sdk`](/cli/debug-aws-sdk) command executes arbitrary read-only AWS SDK v3 commands against your account. This is a power-user escape hatch for inspecting AWS resources that don't have a dedicated debug command.

### Read-only enforcement

Only commands starting with `List`, `Get`, `Describe`, `Head`, or `Batch` are allowed. Any other prefix is rejected.

### Basic usage

List S3 objects in a bucket:

```bash
stacktape debug:aws-sdk --service s3 --command ListObjectsV2 --input '{"Bucket": "my-bucket-name"}'
```

Describe an EC2 instance:

```bash
stacktape debug:aws-sdk --service ec2 --command DescribeInstances --input '{"InstanceIds": ["i-123456"]}'
```

### Flags

- `--service` — AWS service name (e.g. `s3`, `dynamodb`, `ec2`, `lambda`, `ecs`, `rds`, `sqs`, `sns`)
- `--command` — SDK command name (e.g. `ListBuckets`, `GetObject`, `DescribeInstances`)
- `--input` — JSON input for the command (optional, depends on the command)
- `--region` — override the AWS region (defaults to the stack's region)

<Warning>The supported services include Lambda, DynamoDB, S3, CloudWatch, CloudFormation, SQS, SNS, Step Functions, EventBridge, Secrets Manager, SSM, STS, IAM, EC2, ECS, ECR, RDS, SES, Route53, ACM, Cognito, OpenSearch, WAFv2, ELB, AutoScaling, CodeBuild, CodeDeploy, X-Ray, Kinesis, and Firehose.</Warning>

## Executing commands in containers

The [`stacktape debug:container-exec`](/cli/debug-container-exec) command runs a one-shot command inside a running ECS container and returns the output. Use this for quick inspections — checking files, running scripts, or verifying environment variables.

### Basic usage

```bash
stacktape debug:container-exec --resourceName backend --command "ls -la /app"
```

### Flags

- `--command` — the shell command to execute inside the container (required)
- `--resourceName` — the container workload resource name (required)
- `--container` — specific container name when the task has multiple containers
- `--taskArn` — specific task ARN when multiple tasks are running (by default, the first running task is used)

The output includes the command result, exit code, container name, and task ARN.

## Interactive container shell

The [`stacktape container:session`](/cli/container-session) command opens an interactive shell session inside a running ECS container. Unlike `debug:container-exec` which runs a single command, this gives you a live terminal session.

```bash
stacktape container:session --resourceName backend
```

### Flags

- `--resourceName` — the container workload resource name (required)
- `--container` — specific container name when the task has multiple containers
- `--command` — initial command to run in the session (optional)

If multiple tasks are running for the service, Stacktape prompts you to select which one to connect to.

<Info>Both `debug:container-exec` and `container:session` require ECS Exec to be enabled on the service, which Stacktape configures automatically for deployed container workloads.</Info>

## Bastion tunneling

The [`stacktape bastion:tunnel`](/cli/bastion-tunnel) command opens a local port-forwarding tunnel through a [bastion host](/resources/security/bastion-host) to reach VPC-only resources. The tunnel stays open until you press Ctrl+C.

### Supported target resource types

- `relational-database`
- `redis-cluster`
- `application-load-balancer`
- `private-service` (only if it has a load balancer)

### Basic usage

Connect to a database through a bastion:

```bash
stacktape bastion:tunnel --bastionResource bastion --resourceName mainDb
```

This outputs the local port mapping, for example: `127.0.0.1:54321 -> mainDb-host:5432`. You can then connect with any local database tool using `localhost:54321`.

### Flags

- `--bastionResource` — the bastion host resource name (required)
- `--resourceName` — the target resource to tunnel to (required)
- `--localTunnelingPort` — override the local port (auto-assigned by default)

## Bastion shell session

The [`stacktape bastion:session`](/cli/bastion-session) command opens an interactive SSM shell session directly on the bastion EC2 instance.

```bash
stacktape bastion:session --bastionResource bastion
```

This is useful for manual network debugging, running `curl` against private endpoints, or verifying VPC connectivity.

## Agent mode output

All debug commands detect when they're invoked by an AI coding assistant (agent mode) and switch to structured JSON output. This makes debug commands work as data sources for [AI coding assistant integrations](/using-with-ai/ai-coding-assistant-integrations) and the [Stacktape MCP server](/using-with-ai/mcp-server-setup).

In agent mode, `debug:logs` returns:

```json
{
  "logGroup": "/aws/lambda/myStack-api",
  "events": [
    {
      "timestamp": "2025-01-15T10:30:00.000Z",
      "message": "Processing request",
      "logStream": "2025/01/15/[$LATEST]abc123"
    }
  ]
}
```

In interactive mode, the same command renders color-coded, human-readable output with severity highlighting and summary statistics.

## FAQ

### How do I view real-time logs as they arrive?

The `debug:logs` command fetches historical logs. For real-time log streaming, use [dev mode](/local-development/dev-mode-overview) which shows live logs as requests come in. For deployed stacks, re-run `debug:logs` with a short `--startTime` window (e.g. `--startTime 1m`) to get the most recent events.

### Can I use debug commands with dev mode stacks?

Yes. Debug commands work against any deployed stack, including stacks created via `stacktape dev`. Specify the same `--projectName`, `--stage`, and `--region` you used when starting dev mode. The stack must be fully deployed (not just running locally).

### Why does debug:sql only allow read-only queries?

The `debug:sql` command enforces read-only access (`SELECT`, `SHOW`, `DESCRIBE`, `EXPLAIN`) to prevent accidental data mutations from the CLI. For write operations, use [deployment scripts](/deployment-and-lifecycle/deployment-scripts-and-hooks) or connect directly via [`bastion:tunnel`](/cli/bastion-tunnel) with your preferred database client.

### How do I debug a VPC-only database without a bastion?

If your relational database or Redis cluster doesn't have public access and you haven't deployed a [bastion host](/resources/security/bastion-host), you won't be able to use `debug:sql` or `debug:redis` from the CLI. Add a bastion to your stack config, or use `debug:container-exec` to run queries from within a container that has VPC access to the database.

### What permissions do debug commands need?

Debug commands use a dedicated debug agent IAM role provisioned with your stack. This role has read-only access scoped to the resources in the stack. For `debug:aws-sdk`, `debug:dynamodb`, and `debug:opensearch`, the command explicitly uses this debug agent role. If the role assumption fails, it falls back to your configured AWS credentials.

### Can I pipe debug command output to other tools?

Use the `--raw` flag (for `debug:logs`) or agent mode to get JSON output that can be piped to `jq`, saved to files, or consumed by scripts. In interactive mode, the output includes ANSI color codes and formatting that isn't suitable for piping.

### What's the difference between debug:container-exec and container:session?

`debug:container-exec` runs a single command and returns the output — it's non-interactive and suitable for scripting or AI agent use. `container:session` opens a persistent interactive shell where you can run multiple commands. Use `container:session` for exploratory debugging; use `debug:container-exec` for automated checks.

### How do I know which metrics are available for my resource?

The `debug:metrics` command supports a fixed set of metrics per resource type (listed in the metrics table above). If you need a metric not in that list, use `debug:aws-sdk` with the CloudWatch `GetMetricData` or `ListMetrics` commands to access the full CloudWatch metrics catalog.

### Can bastion:tunnel forward to multiple resources simultaneously?

A single `bastion:tunnel` invocation targets one resource. To tunnel to multiple resources (e.g. both a database and a Redis cluster), run multiple `bastion:tunnel` commands in separate terminal windows pointing at the same bastion but different `--resourceName` targets.

### How much do debug commands cost?

Debug commands themselves are free — they're CLI operations. The underlying AWS API calls (CloudWatch GetLogEvents, GetMetricData, DescribeAlarms) incur standard AWS API charges, which are negligible for debugging use. CloudWatch Logs Insights queries (`--query` flag) are billed per GB of data scanned at standard CloudWatch Logs Insights pricing.
