# Custom Dockerfile

The `CustomDockerfilePackaging` mode builds a container image from your own Dockerfile during deployment. Stacktape runs `docker build`, pushes the resulting image to a managed ECR repository, and wires it into your resource automatically. Use this when you need full control over the container environment — system packages, multi-stage builds, or non-standard runtimes.

## When to use

Choose custom Dockerfile packaging when:

- You need system-level dependencies (native libraries, tools like `ffmpeg`, `wkhtmltopdf`, or GPU drivers).
- Your application requires a multi-stage build to keep the final image small.
- You already have a working Dockerfile and want to reuse it as-is.
- You need a specific base image (e.g., NVIDIA CUDA, distroless, or an internal corporate image).
- Your language or framework is not covered by the [Stacktape buildpack](/packaging/containers/stacktape-buildpack).

## When NOT to use

If your app is a standard Node.js, Python, Java, or Go service, the [Stacktape container buildpack](/packaging/containers/stacktape-buildpack) produces an optimized image with zero Dockerfile authoring. A custom Dockerfile adds maintenance overhead — you own base image updates, layer ordering, and security patching.

If you already have a built and pushed image in a registry, use [prebuilt image](/packaging/containers/prebuilt-image) instead. Stacktape skips the build step entirely.

| Mode | Best for |
|------|----------|
| [Stacktape buildpack](/packaging/containers/stacktape-buildpack) | Standard apps; zero config |
| **Custom Dockerfile** | Full control over the build |
| [Prebuilt image](/packaging/containers/prebuilt-image) | Image already exists in a registry |
| [Nixpacks](/packaging/containers/nixpacks) | Auto-detected builds without a Dockerfile |
| [External buildpack](/packaging/containers/external-buildpack) | Cloud Native Buildpacks (Paketo, Heroku) |

## Applicable resources

Custom Dockerfile packaging works with all container-based compute resources:

- [Web service](/resources/compute/web-service)
- [Private service](/resources/compute/private-service)
- [Worker service](/resources/compute/worker-service)
- [Multi-container workload](/resources/compute/multi-container-workload)
- [Batch job](/resources/compute/batch-job)

## Basic example

The minimal configuration requires only `buildContextPath`. If no `dockerfilePath` is specified, Stacktape looks for a file named `Dockerfile` in the build context directory.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, CustomDockerfilePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new WebService({
    packaging: new CustomDockerfilePackaging({
      buildContextPath: './'
    }),
    resources: {
      cpu: 0.25,
      memory: 512
    }
  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

A matching Dockerfile for a Node.js app might look like this:

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 80
CMD ["node", "src/server.js"]
```

<Info>
Your container must listen on port **80**. Stacktape routes traffic to this port for web services and private services.
</Info>

## Dockerfile path

By default, Stacktape looks for a file named `Dockerfile` at the root of `buildContextPath`. Use `dockerfilePath` to point to a different location — the path is relative to `buildContextPath`.

This is useful when you maintain multiple Dockerfiles (development, production, CI) in the same repository.

## Build context

The `buildContextPath` property defines the directory sent to `docker build` as context. All `COPY` and `ADD` instructions in your Dockerfile are relative to this path. The path itself is relative to your Stacktape config file.

A narrower build context speeds up builds because Docker sends fewer files to the build daemon. Combine with a `.dockerignore` file to exclude `node_modules`, `.git`, and other large directories.

```
# .dockerignore
node_modules
.git
dist
*.md
```

## Build arguments

Pass values into the Dockerfile at build time using `buildArgs`. Each argument maps to a Docker `ARG` instruction.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, CustomDockerfilePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new WebService({
    packaging: new CustomDockerfilePackaging({
      buildContextPath: './',
      buildArgs: [
        { argName: 'NODE_ENV', value: 'production' },
        { argName: 'APP_VERSION', value: '2.1.0' }
      ]
    }),
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

In the Dockerfile, declare the corresponding `ARG`:

```dockerfile
ARG NODE_ENV=development
ARG APP_VERSION=latest

FROM node:20-alpine
ENV NODE_ENV=${NODE_ENV}
LABEL version=${APP_VERSION}
WORKDIR /app
COPY . .
RUN npm ci
CMD ["node", "src/server.js"]
```

Build arguments are useful for toggling build behavior (install dev dependencies or not), injecting version labels, or selecting base image variants. They are baked into the image at build time and do not appear at runtime unless you explicitly copy them to `ENV`.

<Warning>
Do not pass secrets through `buildArgs`. They are visible in the image layer history. Use runtime environment variables or the Stacktape `$Secret()` directive for sensitive values.
</Warning>

## Overriding CMD and ENTRYPOINT

You can override the Dockerfile's `CMD` and `ENTRYPOINT` instructions directly in the Stacktape config without modifying the Dockerfile itself.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, CustomDockerfilePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new WebService({
    packaging: new CustomDockerfilePackaging({
      buildContextPath: './',
      entryPoint: ['/bin/sh', '-c'],
      command: ['node src/server.js']
    }),
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

- **`command`** overrides `CMD` — the default arguments passed to the entry point.
- **`entryPoint`** overrides `ENTRYPOINT` — the executable that runs when the container starts.

This is helpful when you use a generic base image and want to swap the start command per stage or environment without rebuilding.

<Info>
The `entryPoint` property is available for web-service, private-service, worker-service, and multi-container-workload. Batch jobs support `command` but not `entryPoint`.
</Info>

## Multi-stage builds

Multi-stage Dockerfiles are the recommended pattern for production containers. They separate the build environment (compilers, dev dependencies) from the runtime image, producing smaller and more secure images.

```dockerfile
# Build stage
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Production stage
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 80
CMD ["node", "dist/server.js"]
```

No special Stacktape configuration is required — multi-stage builds work with the same `buildContextPath` setup. Docker handles the intermediate stages internally.

## Caching behavior

Stacktape calculates a checksum of the build context directory before each deployment. If the checksum matches the previously deployed image, the build is skipped entirely. This means unchanged code does not trigger a rebuild or a push to ECR.

Docker's own layer cache also applies during builds. To maximize cache hits, order your Dockerfile instructions from least-frequently-changed (base image, system deps) to most-frequently-changed (application code).

## Best practices

**Keep images small.** Use Alpine-based or distroless base images. Multi-stage builds prevent build tools from bloating the final layer. Smaller images deploy faster and reduce ECR storage costs.

**Use `.dockerignore`.** Exclude `node_modules`, `.git`, build artifacts, and test files from the build context. This speeds up the context upload and prevents accidental inclusion of sensitive files.

**Pin base image versions.** Use `node:20.11-alpine` instead of `node:latest`. Pinned versions make builds reproducible and prevent unexpected breakage from upstream updates.

**Order layers by change frequency.** Copy dependency manifests (`package.json`, `requirements.txt`) and install before copying application code. This lets Docker cache the dependency layer across code changes.

**Do not run as root.** Add a `USER` instruction to switch to a non-root user in the final stage:

```dockerfile
RUN addgroup -S app && adduser -S app -G app
USER app
```

## FAQ

### What base images can I use?

Any publicly available or privately hosted Docker image works as a base. Common choices include official language images (`node:20-alpine`, `python:3.12-slim`), framework images, distroless images from Google, or your organization's hardened base images.

### How does Stacktape push the built image?

Stacktape manages an ECR (Elastic Container Registry) repository for each container resource in your stack. After `docker build` completes locally, Stacktape tags the image and pushes it to this repository. You do not need to configure ECR access or create repositories manually.

### Can I use private base images from a custom registry?

Yes. Docker pulls base images using your local Docker credentials. If your Dockerfile references a private registry in its `FROM` instruction, make sure you are logged in to that registry locally before running `stacktape deploy` or [`stacktape dev`](/cli/dev).

### Does Stacktape rebuild the image on every deploy?

No. Stacktape checksums the build context directory. If nothing changed since the last deploy, the build and push are skipped. Docker layer caching further speeds up builds when only application code changed but dependencies did not.

### How do I pass runtime secrets to the container?

Do not use `buildArgs` for secrets — they are baked into image layers. Instead, use the [`$Secret()` directive](/configuration/secrets) in your resource's `environment` property. Secrets are injected as environment variables at container start time and are never written to the image.

### What is the maximum image size?

AWS ECR does not impose a hard image size limit, but larger images increase deployment time and cold-start latency. Aim for under 500 MB for web services. Batch jobs can tolerate larger images since start-time sensitivity is lower.

### Can I build ARM/Graviton images?

Container architecture is controlled at the resource level (via compute settings), not in the packaging config. When your resource uses Graviton/ARM instances, Stacktape passes the `--platform linux/arm64` flag to `docker build` automatically. Make sure your Dockerfile produces a compatible image (use multi-arch base images or explicit `--platform` in your `FROM` instruction).

### How does this differ from the Stacktape container buildpack?

The [Stacktape container buildpack](/packaging/containers/stacktape-buildpack) auto-generates an optimized Dockerfile from your source code — you do not author or maintain one. It supports JS/TS, Python, Java, and Go. Custom Dockerfile gives you full control but requires you to own image maintenance, security patching, and layer optimization.

### Can I use Docker Compose with Stacktape?

No. Stacktape replaces Docker Compose for deployment purposes. Define each container as a separate resource (or use a [multi-container workload](/resources/compute/multi-container-workload) for tightly coupled containers). Your existing Dockerfiles transfer directly — only the orchestration layer changes.

### How do I debug a failing Docker build?

Run `docker build` locally with the same `buildContextPath` and `dockerfilePath` to reproduce the error outside Stacktape. Check that all files referenced by `COPY` instructions exist within the build context and are not excluded by `.dockerignore`. Stacktape surfaces the Docker build output in the terminal during deployment.
