# Custom Resources

A Stacktape custom resource runs a Lambda function during stack create, update, and delete operations to provision infrastructure or services not natively supported by Stacktape or CloudFormation. Use custom resources to integrate third-party APIs, SaaS platforms, or custom provisioning logic into your deployment lifecycle with automatic rollback support.

## When to use

Custom resources are the right choice when you need to manage an external resource's lifecycle in sync with your Stacktape stack. Common scenarios:

- **Third-party API setup** — creating webhooks, API keys, or accounts in external services (Stripe, PagerDuty, Cloudflare) that should be torn down when you delete the stack.
- **Cross-account AWS resources** — provisioning resources in a different AWS account that CloudFormation can't reach directly.
- **Custom validation or seed operations** — running checks or data seeding that must succeed before the stack is considered deployed.
- **External database provisioning** — creating databases or schemas in services outside AWS that need lifecycle management.

Custom resources participate in CloudFormation's lifecycle: if creation fails, CloudFormation rolls back and invokes the Delete handler. If an update fails, the previous version is restored. This makes custom resources safer than one-off scripts for managing external state that should stay in sync with your stack.

## When NOT to use

**Use a [deployment script](/resources/advanced/deployment-scripts) instead** if you need to run a command before or after deployment — database migrations, cache warming, or notification sends. Deployment scripts are simpler and don't require implementing the CloudFormation response protocol.

**Use [raw CloudFormation resources](/resources/advanced/raw-cloudformation-resources) instead** if the resource you need is a standard AWS resource type that Stacktape doesn't wrap. Raw CloudFormation is declarative and requires no Lambda code.

**Use an [AWS CDK construct](/resources/advanced/aws-cdk-constructs) instead** if you want to compose multiple AWS resources with higher-level abstractions. CDK constructs generate CloudFormation at build time and don't require runtime Lambda logic.

| | Custom resource | Deployment script | Raw CloudFormation | CDK construct |
|---|---|---|---|---|
| Runs code during deploy | Yes (Lambda) | Yes (local or bastion) | No | No (build-time only) |
| Manages external services | Yes | Yes | No | No |
| Automatic rollback | Yes (Delete event) | No | Yes (declarative) | Yes (declarative) |
| Stack-aware lifecycle | Create / Update / Delete | Before / After hooks | Declarative | Declarative |

## How it works

Stacktape custom resources use a two-part model:

**Custom resource definition** — configures a backing Lambda function with its packaging, timeout, memory, and resource access. The definition itself doesn't provision anything — it defines the code that runs when instances are created, updated, or deleted.

**Custom resource instance** — references a definition by name and passes `resourceProperties` to the backing Lambda. Each instance is an independent CloudFormation resource. When the stack is created, updated, or deleted, CloudFormation invokes the backing Lambda with the instance's properties and a `RequestType` of `Create`, `Update`, or `Delete`.

You can create multiple instances from a single definition, each with different properties. For example, one definition that manages DNS records could have separate instances for `api.example.com` and `app.example.com`.

## Basic example

This example defines a custom resource that registers a webhook with an external service during deployment and removes it when the stack is deleted.

**Stacktape configuration:**

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, CustomResource, StacktapeLambdaBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const webhookProvider = new CustomResource({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/webhook-provider.ts'
    }),
    timeout: 30,
    environment: [{ name: 'EXTERNAL_API_KEY', value: '$Secret(external-api-key)' }]
  });

  return {
    resources: { webhookProvider }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The `timeout` is set to 30 seconds (default is 10) to allow time for the external API call. The `environment` property injects a secret as an environment variable using the [`$Secret` directive](/configuration/directives).

**Lambda handler** (`src/webhook-provider.ts`):

```typescript
export const handler = async (event: any) => {
  const { RequestType, ResourceProperties, ResponseURL, StackId, RequestId, LogicalResourceId } = event;

  let status = 'SUCCESS';
  let reason = '';
  let data: Record<string, string> = {};
  let physicalResourceId = event.PhysicalResourceId || `webhook-${Date.now()}`;

  try {
    switch (RequestType) {
      case 'Create': {
        const response = await fetch('https://api.example.com/webhooks', {
          method: 'POST',
          headers: {
            Authorization: `Bearer ${process.env.EXTERNAL_API_KEY}`,
            'Content-Type': 'application/json'
          },
          body: JSON.stringify({ url: ResourceProperties.callbackUrl })
        });
        const webhook = await response.json();
        physicalResourceId = webhook.id;
        data = { webhookId: webhook.id };
        break;
      }
      case 'Delete': {
        await fetch(`https://api.example.com/webhooks/${event.PhysicalResourceId}`, {
          method: 'DELETE',
          headers: { Authorization: `Bearer ${process.env.EXTERNAL_API_KEY}` }
        });
        break;
      }
    }
  } catch (err: any) {
    status = 'FAILED';
    reason = err.message;
  }

  // CloudFormation requires a response to this pre-signed URL
  await fetch(ResponseURL, {
    method: 'PUT',
    body: JSON.stringify({
      Status: status,
      Reason: reason,
      PhysicalResourceId: physicalResourceId,
      StackId,
      RequestId,
      LogicalResourceId,
      Data: data
    })
  });
};
```

<Warning>
The Lambda **must** send a response to the `ResponseURL` from the event. If it doesn't respond, CloudFormation waits for up to one hour and then fails the stack operation. Always wrap your logic in try/catch and send a response in both success and failure paths.
</Warning>

## Writing the Lambda handler

The backing Lambda receives a CloudFormation custom resource event. Your handler must process the event and send a response to the pre-signed S3 URL provided in the event. The response tells CloudFormation whether the operation succeeded or failed.

### Event fields

| Field | Description |
|---|---|
| `RequestType` | `Create`, `Update`, or `Delete` — the stack operation that triggered this invocation |
| `ResourceProperties` | The `resourceProperties` key-value pairs from the custom resource instance config |
| `OldResourceProperties` | Previous properties (only present on `Update`) |
| `PhysicalResourceId` | Identifier returned by the previous Create or Update (not present on initial Create) |
| `ResponseURL` | Pre-signed S3 URL where you must POST the response |
| `StackId` | CloudFormation stack ARN |
| `RequestId` | Unique request identifier |
| `LogicalResourceId` | The resource's logical name in the CloudFormation template |

### Response format

Your handler must POST a JSON response to `ResponseURL`:

```json
{
  "Status": "SUCCESS",
  "PhysicalResourceId": "my-resource-id-123",
  "StackId": "from-event",
  "RequestId": "from-event",
  "LogicalResourceId": "from-event",
  "Data": { "outputKey": "outputValue" },
  "Reason": ""
}
```

### Handler rules

- **Always respond.** A missing response causes CloudFormation to wait until its internal timeout and then fail the stack operation.
- **Return a stable `PhysicalResourceId`.** If you return a different physical ID on an Update, CloudFormation interprets this as a resource replacement and sends a separate Delete event for the old physical ID.
- **Handle Delete gracefully.** The Delete handler runs during rollbacks too. If the resource was never fully created, your Delete handler should succeed silently rather than throw.
- **Keep the `Data` object small.** The maximum response payload is 4,096 bytes. The `Data` object is accessible via CloudFormation's `Fn::GetAtt` on the custom resource.

## Instances

A custom resource instance references a definition by name and forwards its `resourceProperties` to the backing Lambda during stack operations. Each instance is an independent CloudFormation resource with its own Create/Update/Delete lifecycle.

| Property | Required | Description |
|---|---|---|
| `definitionName` | Yes | Name of the custom resource definition in your config |
| `resourceProperties` | Yes | Key-value pairs forwarded to the Lambda as `event.ResourceProperties` |

Multiple instances can share one definition. This is useful when provisioning several similar external resources — DNS records for different subdomains, webhook registrations for different endpoints, or API configurations for different services. Each instance gets its own `PhysicalResourceId` and lifecycle.

<Info>
When a value in `resourceProperties` changes between deployments, CloudFormation sends an `Update` event to the backing Lambda with both the new `ResourceProperties` and `OldResourceProperties`. When an instance is removed from the config, CloudFormation sends a `Delete` event.
</Info>

## Packaging

The `packaging` property configures how the backing Lambda code is built and deployed. Stacktape supports two [Lambda packaging](/packaging/overview) modes:

### Stacktape Lambda buildpack

[Stacktape Lambda buildpack](/packaging/function/stacktape-buildpack) is zero-config packaging that bundles your source code automatically. It supports JavaScript, TypeScript, Python, Java, Go, Ruby, PHP, and .NET. This is the recommended choice for most custom resources — point it at your handler file and Stacktape handles bundling, dependency resolution, and upload.

### Custom artifact

[Custom artifact](/packaging/function/custom-artifact) lets you provide a pre-built zip or directory. Stacktape uploads it as-is (zipping directories automatically). Use this when you have a custom build pipeline or need precise control over the deployment package contents.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, CustomResource, CustomArtifactLambdaPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const provider = new CustomResource({
    packaging: new CustomArtifactLambdaPackaging({
      packagePath: './dist/custom-resource.zip',
      handler: 'index.handler'
    }),
    timeout: 60,
    memory: 256
  });

  return {
    resources: { provider }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The `memory` property sets Lambda memory in MB (128–10,240). CPU scales proportionally — 1,769 MB equals 1 vCPU. If your custom resource handler does CPU-intensive work like data transformation, increase memory to get more CPU. The `handler` property in the custom artifact config specifies the entry point in `filepath:functionName` format.

## Runtime

The `runtime` property sets the Lambda runtime (e.g., `nodejs22.x`, `python3.12`, `java17`). When using the Stacktape Lambda buildpack, the runtime is auto-detected from the entry file extension, so you rarely need to set this explicitly. Set it manually when using `custom-artifact` packaging or when you need a specific runtime version.

## Connecting to resources

Custom resource definitions support [`connectTo`](/configuration/connecting-resources) to give the backing Lambda access to other resources in your stack. Stacktape automatically grants IAM permissions and injects environment variables with connection details. This is useful when your provisioning logic needs to read from a database, write to a bucket, or interact with queues.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, CustomResource, StacktapeLambdaBuildpackPackaging, Bucket } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const configBucket = new Bucket({});

  const setupProvider = new CustomResource({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/setup-provider.ts'
    }),
    timeout: 60,
    connectTo: ['configBucket']
  });

  return {
    resources: { configBucket, setupProvider }
  };
});
// [!code focus-end]`
    }
  ]}
/>

In this example, the Lambda receives `STP_CONFIG_BUCKET_NAME` and `STP_CONFIG_BUCKET_ARN` as environment variables and has read/write IAM permissions to the bucket.

For AWS services not covered by `connectTo`, use `iamRoleStatements` to grant specific IAM permissions:

<CodeBlock
  intellisense
  tabs={[

// [!code focus-start]
export default defineConfig(() => {

/>

The `iamRoleStatements` property adds raw IAM policy statements to the Lambda's execution role. Use this for AWS services like SES, Rekognition, or Bedrock that aren't managed as Stacktape resources.

## FAQ

### What is a CloudFormation custom resource?

A CloudFormation custom resource is a stack resource backed by a Lambda function instead of a built-in AWS resource type. CloudFormation invokes the backing function on stack Create, Update, and Delete operations. The function manages the actual resource lifecycle and reports success or failure back to CloudFormation. Stacktape wraps this mechanism with its `CustomResource` class, adding automatic Lambda packaging, `connectTo` access, and environment variable injection.

### Can I use connectTo with custom resources?

Yes. The `connectTo` property on a custom resource definition works the same as on a [Lambda function](/resources/compute/lambda-function). Stacktape grants IAM permissions and injects environment variables like `STP_[RESOURCE_NAME]_CONNECTION_STRING` into the backing Lambda. Your custom resource handler can then interact with databases, buckets, queues, and other stack resources during provisioning.

### What happens if my custom resource Lambda fails during deployment?

If the Lambda sends a `FAILED` response or fails to respond before the timeout, CloudFormation marks the resource as failed and rolls back the entire stack to its previous state. During rollback, CloudFormation sends a Delete event to any custom resources that were successfully created in the failed deployment. Your Delete handler should tolerate being called for partially-created resources.

### How long can a custom resource Lambda run?

The maximum `timeout` is 900 seconds (15 minutes), matching the standard AWS Lambda limit. The default in Stacktape is 10 seconds. If your provisioning logic involves slow external APIs or multi-step setup, increase the timeout. CloudFormation itself waits up to one hour for a response from the Lambda — but the Lambda will be killed at its own timeout, so set the Lambda timeout to match your needs.

### When should I use a custom resource vs a deployment script?

Use a **custom resource** when the external resource's lifecycle should be tied to your stack — created on deploy, updated on redeploy, deleted on stack delete. Custom resources participate in CloudFormation rollbacks, so failed deployments clean up automatically. Use a **[deployment script](/resources/advanced/deployment-scripts)** for one-shot tasks that don't need lifecycle management: database migrations, cache warming, data seeding, or notifications. Deployment scripts don't require the CloudFormation response protocol and are simpler to write.

### Can I create multiple instances from one definition?

Yes. The definition defines the Lambda code once. You can create multiple custom resource instances that reference the same definition by name, each passing different `resourceProperties`. Each instance is an independent CloudFormation resource with its own Create/Update/Delete lifecycle. This avoids duplicating Lambda code when provisioning multiple similar external resources.

### What happens to custom resources during a stack rollback?

During rollback, CloudFormation sends a Delete event to custom resources that were successfully created during the failed operation, and an Update event to resources that were modified (reverting to previous properties). If the Delete handler also fails during rollback, CloudFormation logs the failure but continues — the resource may enter a `DELETE_FAILED` state requiring manual cleanup. Write your Delete handler defensively.

### Can custom resources return values for use elsewhere in the stack?

Custom resources can return data in the `Data` field of their CloudFormation response. This data is accessible via CloudFormation's `Fn::GetAtt` using the [overrides escape hatch](/configuration/overrides-and-escape-hatches). The maximum `Data` payload is 4,096 bytes. Custom resources do not expose Stacktape [referenceable parameters](/configuration/referenceable-parameters) — use overrides and CloudFormation intrinsic functions to reference custom resource outputs.

### Does AWS charge for custom resource Lambda invocations?

Custom resource Lambda invocations follow standard Lambda pricing: per-request fee plus duration-based charges. Since the Lambda only runs during stack operations (deploy, update, delete), costs are negligible — typically a few invocations per deployment. The Lambda is not invoked between deployments, so there is no ongoing compute cost. You pay only for the seconds of execution during stack operations.

### Can I use AWS CDK constructs instead of custom resources?

They solve different problems. [CDK constructs](/resources/advanced/aws-cdk-constructs) generate CloudFormation templates at build time — they compose AWS resources declaratively without runtime Lambda code. Custom resources run Lambda code during stack operations to manage resources that don't exist in CloudFormation's resource catalog. If the resource you need is a standard AWS service, prefer a CDK construct or [raw CloudFormation](/resources/advanced/raw-cloudformation-resources). Use a custom resource only when you need to call external APIs or run imperative provisioning logic during deployment.
