# Worker Service

A Stacktape worker-service runs an always-on container with no public URL for background workers, queue processors, and internal tasks. It runs 24/7 inside your VPC, is not reachable from the internet, and can connect to databases, queues, and other resources through the shared service-container configuration.

**Pricing context:** Worker services use AWS Fargate by default. vCPU billing ranges from approximately ~$7.29/month per 0.25 vCPU to ~$466/month per 16 vCPU, plus memory from approximately ~$1.60/month per 512 MB to ~$384/month per 120 GB. Idle tasks still bill; ARM (`arm64`) tasks are approximately ~20% cheaper than x86.

Worker-service properties extend the shared `SimpleServiceContainer` shape, so the resource supports packaging, compute resources, scaling, health checks, logging, resource access, and side containers.

## When to use

A worker-service is the right choice when a process must stay alive and does not need an inbound public endpoint. Common examples are queue consumers, background job processors, internal sync loops, long-running pollers, and containers that coordinate with databases, queues, buckets, or other stack resources through `connectTo`.

Use a worker-service when work is continuous or latency-sensitive enough that starting a new runtime for every event would be wasteful. A worker-service can keep clients warm and process work as soon as the worker sees it. The tradeoff is cost: at least one running task continues to bill unless your scaling configuration changes the number of running instances within supported limits.

## When NOT to use

- **Public HTTP containers** — use [web-service](/resources/compute/web-service) when the container needs a public HTTPS URL, load balancing, TLS, custom domains, CDN, or CORS.
- **VPC-only request/response services** — use [private-service](/resources/compute/private-service) when the container is meant to be addressed by other Stacktape resources; connected workloads receive a PrivateService `ADDRESS` parameter.
- **Short-lived work that runs to completion** — use a [batch job](/resources/compute/batch-job) for compute-heavy or finite jobs.
- **Event-driven code that can scale to zero** — use a [Lambda function](/resources/compute/lambda-function) when the work fits Lambda's execution model and you want request-based billing.
- **Multiple peer containers with separate lifecycle rules** — use [multi-container-workload](/resources/compute/multi-container-workload) when several containers must run as one ECS task with explicit per-container startup ordering.

## Basic example

This example runs a single TypeScript worker container from `./src/worker.ts`. The worker-service has no traffic configuration because a worker-service is not exposed through a public URL.

<CodeBlock

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({
      entryfilePath: './src/worker.ts'
    }),
    resources: {
      cpu: 0.5,
      memory: 1024
    },

    resources: { worker }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`cpu` and `memory` define the Fargate task size for the worker container. `environment` injects static runtime values into the container; values added through `connectTo` are injected automatically and use the `STP_[RESOURCE_NAME]_[PARAM]` naming pattern.

## Compute resources

A worker-service supports two compute engines through `resources`: **Fargate** by specifying `cpu` and `memory`, and **EC2** by specifying `instanceTypes`. Fargate is the default path for most worker services because Stacktape manages the serverless container capacity. EC2 is for teams that need instance-family control or different cost tradeoffs at sustained utilization.

**Fargate** accepts `cpu`, `memory`, and optional `architecture`. The source identifies Fargate as the default engine and allows ARM (`arm64`) for lower-cost Graviton tasks when your image and dependencies support that architecture. Start with Fargate for most queue consumers and background processors, then adjust task size after observing CPU, memory, and downstream pressure.

**EC2** mode uses `instanceTypes` instead of Fargate CPU and memory sizing. Use EC2 only when the instance shape itself matters or your team has a specific reason to manage that capacity choice. Most background workers should start with Fargate and revisit EC2 after measuring steady utilization and cost.

| Compute mode | When to use | Tradeoff |
|--------------|-------------|----------|
| Fargate | Most background workers, queue consumers, and internal loops | Pays for every running task; idle tasks still bill |
| Fargate ARM | Workers with compatible images and dependencies | Lower cost, but native dependencies must support ARM |
| EC2 | Sustained high utilization or strict instance selection | More capacity-shape decisions than Fargate |

## Scaling

A worker-service scales horizontally by changing the number of running container instances. Use `minInstances` to keep workers available and `maxInstances` to cap spend and downstream pressure. Scaling policies can target average CPU and memory utilization, which fits workers whose throughput is closely tied to compute or memory pressure.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    scaling: {
      minInstances: 2,
      maxInstances: 8,
      scalingPolicy: {
        keepAvgCpuUtilizationUnder: 70,
        keepAvgMemoryUtilizationUnder: 80
      }
    }
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

Start with a small fixed worker count when the workload is I/O-bound and predictable. Raise `maxInstances` only after confirming that the queue, database, or external API can handle parallel consumers. For queue processors, keep message handling idempotent because containers can be terminated and replaced when they fail health checks or during normal infrastructure operations.

<Warning>Setting a high `minInstances` keeps containers running all the time, so idle worker capacity still bills. Use conservative minimums and scale out only when the worker-service's downstream systems can absorb the extra concurrency.</Warning>

## Packaging

A worker-service uses container packaging. Stacktape supports five container packaging modes for worker services: Stacktape image buildpack, custom Dockerfile, prebuilt image, Nixpacks, and external buildpack. Most teams should start with the Stacktape image buildpack; switch modes when the build process or runtime image needs tighter control.

| Mode | When to use |
|------|-------------|
| [Stacktape image buildpack](/packaging/containers/stacktape-buildpack) | Source-based builds for common JavaScript, TypeScript, Python, Java, and Go workers |
| [Custom Dockerfile](/packaging/containers/custom-dockerfile) | Full control over system packages, base image, and startup behavior |
| [Prebuilt image](/packaging/containers/prebuilt-image) | An image is already built in Docker Hub or a private registry |
| [Nixpacks](/packaging/containers/nixpacks) | Auto-detected builds using Nixpacks conventions |
| [External buildpack](/packaging/containers/external-buildpack) | Cloud Native Buildpacks with a custom builder or buildpack set |

The packaging mode should match your build ownership. If the application team owns only source code, the Stacktape image buildpack keeps the config short. If the platform team already publishes hardened images, `prebuilt-image` avoids rebuilding them inside Stacktape.

## Connecting resources

Use `connectTo` to give a worker-service access to other stack resources. Stacktape grants IAM permissions, opens security-group access for databases and Redis, and injects environment variables with connection details. Environment variable names depend on the resource name, using the `STP_[RESOURCE_NAME]_[PARAM]` pattern.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    // [!code focus-start]
    connectTo: ['mainDatabase', 'jobsQueue'],
    // [!code focus-end]
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 }
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

If `mainDatabase` is a relational database, the worker receives variables such as `STP_MAIN_DATABASE_CONNECTION_STRING`, `STP_MAIN_DATABASE_HOST`, and `STP_MAIN_DATABASE_PORT`. If `jobsQueue` is an SQS queue, the worker receives queue details such as `STP_JOBS_QUEUE_ARN`, `STP_JOBS_QUEUE_NAME`, and `STP_JOBS_QUEUE_URL`. See [connecting resources](/configuration/connecting-resources) for the full injected variable list.

Use `iamRoleStatements` only for permissions not covered by `connectTo`, such as direct access to an AWS service that is not modeled as a Stacktape resource. Keep raw IAM statements narrow because Stacktape adds them as policy statements for the worker-service role.

## Health checks

A worker-service can use an internal health check to replace a container that is still running but no longer healthy. ECS runs the configured command inside the container; if the container fails the health check, Stacktape terminates and replaces the container. This is useful for workers with deadlocked event loops, broken dependencies, or stuck consumers.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    internalHealthCheck: {
      healthCheckCommand: ['CMD-SHELL', 'node ./healthcheck.js'],
      intervalSeconds: 30,
      timeoutSeconds: 5,
      retries: 3,
      startPeriodSeconds: 60
    }
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

Skip internal health checks for very simple workers where process exit already signals failure clearly. Add internal health checks when the process can hang without exiting, such as a polling loop that stops receiving work but keeps the event loop alive.

## Graceful shutdown

A worker-service can set `stopTimeout` to control how long Stacktape waits before force-killing the container during shutdown. The container receives `SIGTERM` first, then `SIGKILL` after the timeout. The value must be 2–120 seconds and defaults to 2 seconds.

Increase `stopTimeout` for queue consumers or background processors that need time to finish one message, flush telemetry, or release a lock after receiving `SIGTERM`. Keep the timeout low for fast-failing workers that should be replaced quickly. Long shutdown windows can slow replacement and deployment workflows, so size the timeout around real cleanup behavior.

## Side containers

A worker-service can run helper containers alongside the main worker. `run-on-init` helpers must exit successfully before the main container starts, which fits setup or migration steps. `always-running` helpers stay up for the worker-service lifecycle, which fits supporting processes that communicate with the main container over `localhost`.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  WorkerService,
  StacktapeImageBuildpackPackaging,
  CustomDockerfilePackaging
} from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    sideContainers: [
      {
        containerType: 'run-on-init',
        packaging: new CustomDockerfilePackaging({
          buildContextPath: './migrations',
          dockerfilePath: './Dockerfile'
        })
      }
    ]
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

Use side containers when the helper shares the worker-service lifecycle. If multiple containers need independent traffic routing, explicit startup dependencies, or peer-level configuration, use [multi-container-workload](/resources/compute/multi-container-workload) instead. Side containers are a convenience for common helper patterns, not a replacement for modeling several first-class containers.

## EFS storage

A worker-service can mount an [EFS filesystem](/resources/storage/efs-filesystem) through `volumeMounts` when the container needs persistent shared files. Data stored in EFS volumes persists even when containers are replaced, and multiple containers can mount the same volume. Use EFS when a worker needs files that persist across container replacement or must be shared by containers in the same workload.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  WorkerService,
  EfsFilesystem,
  StacktapeImageBuildpackPackaging
} from 'stacktape';

export default defineConfig(() => {
  const sharedData = new EfsFilesystem({});

  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    volumeMounts: [
      {
        type: 'efs',
        properties: {
          efsFilesystemName: 'sharedData',
          mountPath: '/mnt/shared',
          rootDirectory: '/workers'
        }
      }
    ]
    // [!code focus-end]
  });

  return { resources: { sharedData, worker } };
});`
    }
  ]}
/>

`mountPath` is the absolute path inside the container; `rootDirectory` selects the EFS directory mounted there. Skip EFS when the worker can store state in a database, object storage, or temporary container storage. A network filesystem adds latency and a separate storage cost model.

## Private subnets

A worker-service runs inside your VPC by definition. Set `usePrivateSubnetsWithNAT: true` when the worker should run without a public IP and send outbound internet traffic through NAT Gateway. NAT Gateways provide static Elastic IPs per configured AZ, so allowlisting may require more than one IP depending on `stackConfig.vpc.nat.availabilityZones`.

<CodeBlock

Private subnets with NAT add cost and networking complexity. The VPC NAT setting controls how many availability zones receive a NAT Gateway, and the source prices NAT Gateway at approximately ~$32/month per availability zone plus data processing fees. Skip this setting unless you need fixed outbound IPs or a stricter subnet posture.

## Logging

Container output is automatically sent to CloudWatch and retained for 90 days. View logs with [`stacktape debug:logs`](/cli/debug-logs) or in the [Stacktape Console](/stacktape-console/console-overview). Logs are usually the first place to inspect worker progress, failed message details, queue lag decisions, and shutdown behavior.

Set `logging.disabled: true` only for workers where CloudWatch logging is intentionally not needed. For most worker services, keep logging enabled and make log lines structured enough to identify the job, queue message, tenant, or resource the worker was processing.

## Remote sessions

Set `enableRemoteSessions: true` when you need interactive shell access to a running worker container through [`stacktape container:session`](/cli/container-session). Remote sessions are useful for inspecting environment variables, validating mounted files, or debugging a stuck worker in a deployed stack. The setting is disabled by default.

## FAQ

### What is a Stacktape worker-service?

A Stacktape worker-service is an always-on container with no public URL. It runs inside your VPC and is meant for background processing, queue consumers, internal loops, and tasks that need access to databases, queues, or other Stacktape resources. Use [worker-service](/resources/compute/worker-service) when the process should keep running without accepting inbound public HTTP traffic.

### Does a worker-service have a public endpoint?

No. A worker-service is documented as not reachable from the internet and has no public URL. Use [web-service](/resources/compute/web-service) for a public HTTPS container, or use [private-service](/resources/compute/private-service) when the container is meant to be addressed by other Stacktape resources.

### Does a worker-service scale to zero?

A worker-service is an always-on container resource, and the Fargate pricing source says idle tasks still bill because Fargate cannot scale to zero. If scale-to-zero billing is the main requirement, use a [Lambda function](/resources/compute/lambda-function) when the workload fits event-driven execution.

### How does a worker-service connect to a queue or database?

Add the resource name to `connectTo` on the worker-service. Stacktape grants IAM permissions, opens security-group access for databases and Redis, and injects environment variables with names such as `STP_MAIN_DATABASE_CONNECTION_STRING` when the connected resource is named `mainDatabase`. See [connecting resources](/configuration/connecting-resources) for the full list by resource type.

### How much does a Fargate worker-service cost?

Fargate bills by vCPU-hour and GB-hour for each running task, so a worker-service continues billing while idle. The provided pricing source gives ballpark us-east-1 ranges from approximately ~$7.29/month per 0.25 vCPU to ~$466/month per 16 vCPU, plus memory from approximately ~$1.60/month per 512 MB to ~$384/month per 120 GB. See [Managing Costs](/managing-costs/overview) for cost tracking guidance.

### Worker-service vs Lambda function: which should I use?

Use a worker-service for long-running processes, polling loops, warm clients, or background consumers that should stay alive. Use a [Lambda function](/resources/compute/lambda-function) for event-driven work that can start per invocation and benefit from scale-to-zero billing. The decision is mostly about lifecycle: always-on container versus per-event execution.

### Worker-service vs batch job: which should I use?

Use a worker-service for continuous background processing. Use a [batch job](/resources/compute/batch-job) for work that has a clear start and finish, especially compute-heavy or long-running jobs submitted on demand. A worker-service is operational capacity; a batch job is a run-to-completion unit.

### Can a worker-service use a custom Dockerfile?

Yes. A worker-service uses the shared container packaging model, which includes Stacktape image buildpack, custom Dockerfile, prebuilt image, Nixpacks, and external buildpack. Use [custom Dockerfile packaging](/packaging/containers/custom-dockerfile) when the worker needs specific system packages, startup behavior, or base-image control.

### How do I debug a running worker-service?

Start with logs from [`stacktape debug:logs`](/cli/debug-logs) or the Stacktape Console, because worker output is sent to CloudWatch Logs by default. For interactive inspection, enable `enableRemoteSessions` and use [`stacktape container:session`](/cli/container-session) to open a shell in the running container.

### Can a worker-service mount persistent storage?

Yes. A worker-service can use `volumeMounts` to mount an [EFS filesystem](/resources/storage/efs-filesystem). Use EFS when files must survive task replacement or be shared with helper containers; skip it when state belongs in a database, queue, or object store.
