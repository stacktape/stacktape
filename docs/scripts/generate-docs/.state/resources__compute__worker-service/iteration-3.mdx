# Worker Service

A Stacktape worker-service runs an always-on ECS container with no public URL. Use it for background workers, queue processors, and internal tasks that should keep running inside your VPC without accepting internet traffic. A worker-service can connect to databases, queues, buckets, and other stack resources through Stacktape-managed permissions and connection details.

**Pricing context:** Worker-services use AWS Fargate by default. The pricing source lists vCPU from approximately ~$7.29/month per 0.25 vCPU and memory from approximately ~$1.60/month per 512 MB in us-east-1. Idle Fargate tasks still bill, and ARM tasks are ~20% cheaper than x86.

## When to use

A worker-service is the right choice when a containerized process should keep running but should not receive public web traffic. Worker-services fit background processing patterns where the application code owns the loop: polling a queue, watching a broker, running a daemon, or coordinating work inside the stack.

- **Queue consumers** — workers that poll or consume jobs from [SQS queues](/resources/messaging/sqs-queue), Redis, or another broker
- **Background processors** — image processing, email delivery, report generation, or data enrichment that runs outside the request path
- **Internal daemons** — long-running containers that coordinate work inside your stack or VPC
- **Custom runtime workers** — workloads that need a full container environment instead of a [Lambda function](/resources/compute/lambda-function)

## When NOT to use

- **Public HTTP services** — use [web-service](/resources/compute/web-service) when the workload needs a public endpoint, load balancing, custom domains, or optional CDN configuration.
- **VPC-only request/response services** — use [private-service](/resources/compute/private-service) when other workloads need to call the container over the network.
- **Short-lived jobs** — use [batch job](/resources/compute/batch-job) when the workload should run to completion and stop.
- **Event-triggered functions** — use [Lambda function](/resources/compute/lambda-function) when the workload is better modeled as event-driven function invocations.
- **Multiple peer containers** — use [multi-container-workload](/resources/compute/multi-container-workload) when the workload is better modeled as multiple peer containers rather than one main worker with helper containers.

## Basic example

This example runs a background worker container and gives the worker access to an SQS queue. `connectTo` grants the worker permissions to use the queue and injects `STP_JOBS_QUEUE_ARN`, `STP_JOBS_QUEUE_NAME`, and `STP_JOBS_QUEUE_URL` into the container.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, SqsQueue, StacktapeImageBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const jobsQueue = new SqsQueue({});

  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({
      entryfilePath: './src/worker.ts'
    }),
    connectTo: ['jobsQueue'],
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { jobsQueue, worker }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The worker-service has no public endpoint configuration. For typical worker-services, write the container entrypoint as a long-running process, such as a queue polling loop, broker subscriber, or internal daemon. If the workload should run only after a specific event arrives, [Lambda function](/resources/compute/lambda-function) is usually the better starting point.

## Compute resources

Worker-services support two compute engines through the `resources` property: Fargate and EC2. The shared container source describes Fargate as the default, where you specify `cpu` and `memory`; EC2 mode is selected by specifying `instanceTypes`. Start with Fargate unless you have a concrete reason to choose instance families yourself.

<CodeBlock
  intellisense
  tabs={[

  });

  return { resources: { worker } };
});`
    }
  ]}
/>

`cpu` and `memory` size the running Fargate task. Fargate is the practical default for most worker-services because Stacktape can run the worker without you selecting or maintaining EC2 instance types. The tradeoff is cost: every running worker instance bills while waiting for work.

`instanceTypes` switches the worker-service to EC2-backed compute and lists preferred instance types. EC2 mode is useful when your team already has a reason to choose specific instance families or plan around reserved capacity; Fargate remains simpler for the common background-worker case.

| Compute mode | When to use | Tradeoff |
|--------------|-------------|----------|
| Fargate | Default choice for most background workers | Idle running tasks still bill |
| EC2 instance types | Specific instance-family needs or sustained capacity planning | More capacity behavior to reason about than Fargate |

<Info>Fargate pricing is based on the running task's vCPU and memory allocation. The pricing source lists ARM/Graviton tasks as ~20% cheaper than x86 where ARM is used.</Info>

## Scaling

Worker-service scaling is configured through the `scaling` property, which controls how many container instances Stacktape keeps running. Use scaling for steady background throughput, not for event-triggered execution. Queue-backed workers still need application-level polling behavior, and queue alarms belong on the queue resource rather than the worker-service itself.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    scaling: {
      minInstances: 1,
      maxInstances: 5
    }
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

Size worker-service scaling around the systems the worker talks to. More worker instances can drain work faster, but they can also increase database connections, external API calls, queue receive calls, and concurrent writes. For queue-backed workers, monitor the [SQS queue](/resources/messaging/sqs-queue) itself with [alarms](/observability/alarms), such as received-message count or queue-not-empty triggers.

<Warning>Worker-services are always-on containers, so at least one running instance can continue billing while idle. If your workload should only run in response to events, compare the worker-service design with [Lambda function](/resources/compute/lambda-function) before committing to an always-running container.</Warning>

## Packaging

A worker-service uses container packaging because the worker runs as an ECS container. Stacktape supports five container packaging modes: Stacktape image buildpack, custom Dockerfile, prebuilt image, Nixpacks, and external buildpack. Use the Stacktape image buildpack first unless you already have a Dockerfile or an image pipeline.

| Mode | When to use |
|------|-------------|
| [Stacktape image buildpack](/packaging/containers/stacktape-buildpack) | Build a container image from source with minimal configuration |
| [Custom Dockerfile](/packaging/containers/custom-dockerfile) | Control operating system packages, entrypoint, or image layout directly |
| [Prebuilt image](/packaging/containers/prebuilt-image) | Reuse an image that already exists in a registry |
| [Nixpacks](/packaging/containers/nixpacks) | Let Nixpacks detect and build the application image |
| [External buildpack](/packaging/containers/external-buildpack) | Use Cloud Native Buildpacks with a selected builder or buildpack set |

## Connecting to resources

Use `connectTo` to give a worker-service access to other resources in the same stack. Stacktape grants IAM permissions for supported resources, opens network access for relational databases and RedisCluster resources, and injects Stacktape-managed environment variables with connection details. This is the usual path for workers that read from queues, write to buckets, or connect to databases.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  WorkerService,
  SqsQueue,
  Bucket,
  StacktapeImageBuildpackPackaging
} from 'stacktape';

export default defineConfig(() => {
  const jobsQueue = new SqsQueue({});
  const exportsBucket = new Bucket({});

  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    connectTo: ['jobsQueue', 'exportsBucket']
    // [!code focus-end]
  });

  return { resources: { jobsQueue, exportsBucket, worker } };
});`
    }
  ]}
/>

For permissions not covered by `connectTo`, use `iamRoleStatements` to add raw IAM policy statements. Keep raw IAM narrow and prefer `connectTo` when the target is a Stacktape resource, because `connectTo` also handles the connection details that worker code usually needs. See [connecting resources](/configuration/connecting-resources) for the injected-variable model.

### Environment variables

Worker-service `environment` passes static values into the container at runtime. Values can be strings, numbers, or booleans. Use `environment` for application configuration and feature flags; use `connectTo` when the value should come from a Stacktape-managed resource such as a queue URL, database connection string, or bucket name.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    environment: [
      { name: 'WORKER_CONCURRENCY', value: 4 },
      { name: 'ENABLE_IMAGE_RESIZE', value: true }
    ]
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

The `environment` property passes the names you define; variables from `connectTo` are added separately using Stacktape-managed `STP_...` names. For example, an SQS queue named `jobsQueue` injects `STP_JOBS_QUEUE_URL`, and a bucket named `exportsBucket` injects `STP_EXPORTS_BUCKET_NAME`.

## Health checks

A worker-service can define `internalHealthCheck`, a container health check that Stacktape uses to replace unhealthy containers. Use an internal health check when a worker can become stuck while the process still appears alive. Skip it when process exit already reflects failure reliably and a cheap progress check would be hard to define.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    internalHealthCheck: {
      healthCheckCommand: ['CMD-SHELL', 'node ./healthcheck.js']
    }
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

A good worker health check verifies that the process can make progress without doing real job processing or mutating production data. For example, check local process state, a lightweight dependency connection, or a heartbeat file written by the worker loop. Detailed timing and retry fields are available in the API reference.

## Side containers

A worker-service can run helper containers beside the main worker through `sideContainers`. `run-on-init` helper containers run to completion before the main container starts, and `always-running` helper containers stay up for the lifetime of the task. Use side containers for setup work or tightly coupled helper processes; use separate resources when the helper should scale or fail independently.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  WorkerService,
  StacktapeImageBuildpackPackaging,
  CustomDockerfilePackaging
} from 'stacktape';

export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/worker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    sideContainers: [
      {
        containerType: 'run-on-init',
        packaging: new CustomDockerfilePackaging({
          buildContextPath: './migrations',
          dockerfilePath: './Dockerfile'
        })
      }
    ]
    // [!code focus-end]
  });

  return { resources: { worker } };
});`
    }
  ]}
/>

Use `run-on-init` for migrations, cache warming, or setup that must succeed before the worker starts. A `run-on-init` container must exit with code 0 before the main worker starts, so failed setup prevents the worker from processing jobs against an incomplete state.

Use `always-running` for helper processes that must share the worker lifecycle, such as local metrics agents or a companion process that talks to the main worker over `localhost`. If an `always-running` helper crashes, the whole task fails. Skip `always-running` when a separate [worker-service](/resources/compute/worker-service) or [private-service](/resources/compute/private-service) would give better isolation.

## EFS storage

A worker-service can mount an [EFS filesystem](/resources/storage/efs-filesystem) when the worker needs files that persist across container replacement or need to be mounted by multiple containers in the same task. Skip EFS when the worker can store durable data in an [S3 bucket](/resources/storage/s3-bucket), database, or queue instead.

The `volumeMounts` property attaches EFS storage to the worker-service. The source states that data in EFS volumes persists even when containers are replaced, multiple containers can mount the same volume, and all data is encrypted in transit. Use the API reference for the nested mount fields.

## Private subnets

Set `usePrivateSubnetsWithNAT: true` when a worker-service needs outbound traffic from a NAT Gateway instead of a public IP. This is mainly useful for external APIs that require IP allowlisting. Keep the default when the worker only calls AWS services, public APIs that do not allowlist source IPs, or resources connected through Stacktape-managed networking.

Private subnet mode routes outbound traffic through NAT Gateway. The source types call out NAT Gateway cost at approximately ~$32/month plus data processing fees, and the stack-level `stackConfig.vpc.nat` setting controls how many availability zones get NAT Gateways. Most worker-services should skip private subnets unless static outbound IPs are a concrete requirement.

## Logging

Worker-service container output is sent to CloudWatch Logs by default and retained for 90 days. The `logging` property configures container logging; see the API reference for available options. View logs with the [`debug:logs`](/cli/debug-logs) CLI command or in the [Stacktape Console](/stacktape-console/console-overview).

Worker logs are often more important than request logs because background failures can be invisible to users until queues back up or data is delayed. Keep structured logs with job IDs, queue message IDs, and retry counts. For external log pipelines, use [log forwarding](/observability/log-forwarding) rather than adding a custom helper container unless you need agent-level control.

## Remote sessions

Set `enableRemoteSessions: true` to allow interactive shell access to running worker containers through [`container:session`](/cli/container-session). The source describes this as SSH-like debugging through an added SSM agent. Enable remote sessions for troubleshooting stages where live inspection is useful, and keep remote sessions disabled by default to reduce the runtime surface area.

`stopTimeout` controls how long the worker gets after SIGTERM before the container is force-killed. The source requires a value from 2 to 120 seconds and defaults to 2 seconds. Increase `stopTimeout` for workers that need time to finish an in-flight job, release a lock, or flush telemetry before shutdown.

## FAQ

### Does a worker-service have a public URL?

No. A Stacktape worker-service is an always-on container with no public URL and is not reachable from the internet. Use [web-service](/resources/compute/web-service) when the workload needs a public HTTPS endpoint.

### Can a worker-service consume SQS messages?

Yes. A worker-service can connect to an [SQS queue](/resources/messaging/sqs-queue) with `connectTo`, which grants queue access and injects `STP_[RESOURCE_NAME]_ARN`, `STP_[RESOURCE_NAME]_NAME`, and `STP_[RESOURCE_NAME]_URL`. Worker-service code is responsible for polling or consuming messages because worker-services do not define event triggers.

### Does a worker-service scale to zero?

No. A worker-service runs continuously as an always-on container, and Fargate bills while tasks are running. If a workload is better modeled as event-driven function invocations, compare it with [Lambda function](/resources/compute/lambda-function).

### When should I use worker-service vs Lambda function?

Use a worker-service when the process needs a full container runtime, long-lived connections, custom polling behavior, or continuous background execution. Use a [Lambda function](/resources/compute/lambda-function) when the unit of work is better expressed as an event-triggered function invocation.

### When should I use worker-service vs batch job?

Use a worker-service for a process that should keep running and repeatedly handle work. Use a [batch job](/resources/compute/batch-job) for compute that runs to completion and stops, such as one-off exports or scheduled heavy processing.

### How much does an ECS Fargate worker cost?

AWS Fargate bills per running vCPU and memory allocation, so a worker-service costs money even when waiting for jobs. The provided pricing source lists vCPU billing from approximately ~$7.29/month per 0.25 vCPU and memory billing from approximately ~$1.60/month per 512 MB in us-east-1. ARM tasks are listed as ~20% cheaper than x86.

### How do I give a worker database or Redis access?

Use `connectTo` with the specific data resource. Relational databases inject `CONNECTION_STRING`, `JDBC_CONNECTION_STRING`, `HOST`, and `PORT`; Aurora databases also inject reader connection variables. RedisCluster injects `HOST`, `READER_HOST`, and `PORT`; UpstashRedis injects `REDIS_URL`, `HOST`, `PORT`, `PASSWORD`, `REST_TOKEN`, and `REST_URL`. See [connecting resources](/configuration/connecting-resources) for the full injected-variable model.

### Can a worker-service run sidecars?

Yes. A worker-service supports `sideContainers` with `run-on-init` and `always-running` modes. Use `run-on-init` for setup before the main worker starts, and `always-running` for helper processes that should live for the whole worker task.

### How should I monitor a worker-service?

Use structured CloudWatch logs, internal health checks, and alarms around resources that reveal worker health, such as queues or downstream databases. For queue-backed workers, monitor the SQS queue itself with alarms such as received-message count or queue-not-empty checks. See [alarms](/observability/alarms) for alerting setup.

### Can a worker-service use a static outbound IP?

Yes. Set `usePrivateSubnetsWithNAT: true` so outbound traffic routes through a NAT Gateway with a static Elastic IP. Use private subnets when a third-party API requires source IP allowlisting, and skip the setting for most workers because NAT Gateway adds cost and networking complexity.
