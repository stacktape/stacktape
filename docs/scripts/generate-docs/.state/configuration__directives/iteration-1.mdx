# Directives

Directives are built-in functions (prefixed with `$`) that inject dynamic values into your Stacktape configuration. They let you reference secrets, resource outputs, stage names, files, and cross-stack values — without hardcoding anything. Directives bridge the gap between static configuration and the dynamic reality of deployments where values like database URLs only exist after resources are created.

## Why directives exist

A Stacktape configuration file describes your entire infrastructure declaratively. But some values are inherently dynamic:

- A database connection string is only known after the database is provisioned.
- A secret should never be committed to source control.
- The stage name determines which domain prefix to use.
- A shared infrastructure stack exports values that dependent stacks consume.

Directives solve these problems by deferring value resolution to the right moment — either when Stacktape parses your config (local directives) or when CloudFormation deploys your resources (runtime directives).

## Local vs runtime directives

Stacktape divides directives into two categories based on **when** they resolve:

| Category | Resolved when | Use for |
|----------|---------------|---------|
| **Local** | Stacktape parses your config (before deployment) | Stage name, region, file contents, string formatting, CLI args, git info |
| **Runtime** | CloudFormation deploys resources | Secrets, resource parameters, cross-stack imports, runtime string composition |

The distinction matters because **runtime directives can only appear in the `resources` and `scripts` sections** of your config. Local directives work anywhere — including `variables`, top-level settings, and conditional logic.

<Warning>Using a runtime directive (like `$Secret` or `$ResourceParam`) outside the `resources` or `scripts` sections causes a deployment error. If you need a runtime value in a variable, use `$CfFormat` to compose it within a resource property.</Warning>

## Using directives in TypeScript

In TypeScript configuration, runtime directives are imported as functions from the `stacktape` package. Local directives that have native JavaScript equivalents (like `$Var` or `$Format`) aren't needed — use JS variables and template literals instead.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  RelationalDatabase,
  RdsEnginePostgresql,
  $Secret,
  $ResourceParam,
  $Stage,
  $Region
} from 'stacktape';

// [!code focus-start]
export default defineConfig(({ stage }) => {
  const database = new RelationalDatabase({
    engine: new RdsEnginePostgresql({
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    }),
    credentials: {
      masterUserPassword: $Secret('db-password')
    }
  });

  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      { name: 'DB_CONNECTION_STRING', value: $ResourceParam('database', 'connectionString') },
      { name: 'STAGE', value: $Stage() },
      { name: 'REGION', value: $Region() }
    ],
    connectTo: [database]
  });

  return { resources: { database, api } };
});
// [!code focus-end]
`
    }
  ]}
/>

The following directive functions are importable from the `stacktape` package: `$Secret`, `$SsmParam`, `$ResourceParam`, `$CfResourceParam`, `$CfFormat`, `$CfStackOutput`, `$GitInfo`, `$Stage`, `$Region`.

Directives that resolve locally (`$Var`, `$Format`, `$File`, `$FileRaw`, `$CliArgs`, `$StackOutput`, `$This`) don't have TypeScript equivalents because you can accomplish the same thing with native JavaScript — variables, template literals, `import`, and `fs.readFileSync`.

## Local directives reference

### $Stage()

Returns the current stage name passed to the CLI via `--stage`. Use it to differentiate resource naming, domain prefixes, or feature flags across stages.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway, $Stage } from 'stacktape';

// [!code focus-start]
export default defineConfig(({ stage }) => {
  const api = new HttpApiGateway({
    customDomains: [
      { domainName: \`\${stage}-api.example.com\` }
    ]
  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

In TypeScript, the `stage` value is also available directly from the `defineConfig` callback parameter — `defineConfig(({ stage }) => { ... })`. Both approaches are equivalent.

### $Region()

Returns the AWS region where the stack is being deployed (e.g. `us-east-1`, `eu-west-1`).

### $Format()

Performs string interpolation using `{}` placeholders. Each `{}` is replaced with the corresponding positional argument. Use it in YAML configs for dynamic string composition — in TypeScript, prefer template literals instead.

The number of `{}` placeholders must match the number of additional arguments.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway, $Stage } from 'stacktape';

// [!code focus-start]
export default defineConfig(({ stage, region }) => {
  const api = new HttpApiGateway({
    customDomains: [
      { domainName: \`\${stage}.\${region}.api.example.com\` }
    ]
  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

<Info>In TypeScript configs, native template literals replace `$Format()`. In YAML configs, use `$Format('{}-api.example.com', $Stage())` for the same result.</Info>

### $Var()

References a value from the `variables` section of your config. Access nested properties with dot notation after the closing parenthesis: `$Var().myProperty.nested`.

In TypeScript, this directive isn't needed — define a regular JavaScript variable and reference it directly. It exists primarily for YAML configs where you can't define variables natively.

### $File()

Loads and parses a file relative to your working directory. Supported formats:

| Format | Returns |
|--------|---------|
| `.env` | Array of `{ name, value }` objects |
| `.json` | Parsed JSON object |
| `.yml` / `.yaml` | Parsed YAML object |
| `.ini` | Parsed INI object |

Access nested properties with dot notation: `$File('config.json').database.host`.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';
import { readFileSync } from 'fs';

// [!code focus-start]
export default defineConfig(() => {
  const envVars = readFileSync('.env', 'utf-8')
    .split('\\n')
    .filter(line => line.includes('='))
    .map(line => {
      const [name, ...rest] = line.split('=');
      return { name: name.trim(), value: rest.join('=').trim() };
    });

  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: envVars
  });

  return { resources: { handler } };
});
// [!code focus-end]`
    }
  ]}
/>

<Tip>In TypeScript configs, use native `import` or `readFileSync` for file loading. The `$File()` directive is most useful in YAML configs where file loading isn't built in.</Tip>

### $FileRaw()

Reads a file as a raw UTF-8 string without parsing. Use it when you need the literal file content (e.g., embedding a JSON config as a string environment variable).

### $GitInfo()

Returns information about the current git repository. Access properties via the argument in YAML (`$GitInfo('branch')`) or via dot notation in TypeScript (`$GitInfo().branch`).

Available properties:

| Property | Description |
|----------|-------------|
| `commit` | Current commit SHA |
| `branch` | Current branch name |
| `username` | Git username |
| `gitUrl` | Repository URL |
| `hasUncommitedChanges` | Whether there are uncommitted changes |

<Info>In CI environments (GitHub Actions, GitLab CI, Bitbucket Pipelines), `$GitInfo` falls back to CI-specific environment variables when git commands aren't available.</Info>

### $CliArgs()

Accesses additional CLI arguments passed after `--` in the deploy command. Accepts a second optional argument as a default value if the argument isn't provided.

Pass custom arguments when deploying:

```bash
stacktape deploy --stage production --region eu-west-1 -- --featureFlag=true --apiVersion=v2
```

In your config, reference them with `$CliArgs('featureFlag')`. In TypeScript configs, the `defineConfig` callback doesn't expose CLI args directly, so use this directive or environment variables.

### $StackOutput()

References an exported output from another deployed Stacktape stack. The referenced stack must already be deployed. This is a **local** directive — it fetches the value at config-parse time by querying CloudFormation.

Arguments: `$StackOutput('stackName', 'outputKeyName')` with an optional third argument for region.

The `stackName` is the full CloudFormation stack name (formatted as `projectName-stage`). The `outputKeyName` is the CloudFormation output key — for Stacktape resources, output keys are in PascalCase format (e.g., a resource named `database` with parameter `connectionString` has output key `DatabaseConnectionString`).

<Warning>The referenced stack must be deployed before the stack that references it. If the referenced stack doesn't exist, deployment fails.</Warning>

### $Profile()

Returns the name of the AWS profile being used for the current deployment. Useful when you need to pass profile information to scripts or hooks.

### $AwsAccountId()

Returns the AWS account ID of the target account. Use it when you need the account ID for ARN construction or cross-account references.

## Runtime directives reference

Runtime directives resolve during CloudFormation deployment. They produce CloudFormation intrinsic functions under the hood, which means the actual values are only available when AWS provisions your resources.

### $Secret()

References a secret stored in AWS Secrets Manager. The secret must exist in the same region you're deploying to. Create secrets using [`stacktape secret:create`](/cli/secret-create) or through the [Stacktape Console](/configuration/secrets).

When prompted, provide a name (e.g., `api-keys`) and a JSON value (e.g., `{"stripe": "sk_...", "sendgrid": "SG..."}`).

### $SsmParam()

References a parameter from AWS Systems Manager Parameter Store. Supports both `String` and `SecureString` parameter types — the type is auto-detected. Use SSM parameters when you need values that change without redeployment (managed outside Stacktape) or when integrating with other AWS tools that write to Parameter Store.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $SsmParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      { name: 'FEATURE_FLAGS', value: $SsmParam('/myapp/feature-flags') },
      { name: 'DB_HOST', value: $SsmParam('/prod/database/host') }
    ]
  });

  return { resources: { handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

SSM parameters support hierarchical paths (e.g., `/myapp/prod/database/host`). Create and manage them through the Stacktape Console or AWS Console.

<Info>`$Secret` uses AWS Secrets Manager (designed for credentials with rotation support). `$SsmParam` uses SSM Parameter Store (designed for configuration values and feature flags). Choose based on whether the value is a credential or a configuration parameter.</Info>

### $ResourceParam()

References a parameter from another Stacktape resource in the same stack. The value resolves at deployment time because it depends on CloudFormation-provisioned outputs like URLs, ARNs, and connection strings. See [Referenceable parameters](/configuration/referenceable-parameters) for the full list of parameters available per resource type.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway, HostingBucket, $ResourceParam } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new HttpApiGateway({});

  const frontend = new HostingBucket({
    environment: [{ name: 'API_URL', value: $ResourceParam('api', 'url') }]
  });

  return { resources: { api, frontend } };
});
// [!code focus-end]
`
    }
  ]}
/>

The first argument is the resource name (as defined in the `resources` object you return from `defineConfig`). The second argument is the parameter name.

<Tip>For most resource-to-resource communication, prefer [`connectTo`](/configuration/connecting-resources) which automatically injects environment variables and grants IAM permissions. Use `$ResourceParam` when you need a value in a non-standard location or when `connectTo` doesn't cover your use case.</Tip>

### $CfResourceParam()

References a parameter from a raw CloudFormation resource defined in the `cloudformationResources` section. Use this when integrating with [raw CloudFormation resources](/resources/advanced/raw-cloudformation-resources) that aren't managed by Stacktape's resource abstraction.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [{ name: 'TOPIC_ARN', value: $CfResourceParam('mySnsTopic', 'TopicArn') }]
  });

  return {
    resources: { handler },
    cloudformationResources: {
      mySnsTopic: {
        Type: 'AWS::SNS::Topic',
        Properties: { TopicName: 'my-notifications' }
      }
    }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The first argument is the logical ID of the CloudFormation resource. The second argument is the attribute name — either a `GetAtt` attribute (like `TopicArn`, `Arn`, `DomainName`) or `Ref` (which returns the resource's primary identifier). Available attributes depend on the CloudFormation resource type — see the [AWS CloudFormation documentation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-template-resource-type-ref.html) for each type's return values.

### $CfFormat()

Performs string interpolation at CloudFormation deployment time, similar to `$Format()` but capable of including runtime directive values as arguments. Use `$CfFormat` when you need to combine multiple runtime values into a single string.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  RelationalDatabase,
  RdsEnginePostgresql,
  $Secret,
  $CfFormat,
  $ResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const database = new RelationalDatabase({
    engine: new RdsEnginePostgresql({
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    }),
    credentials: {
      masterUserPassword: $Secret('db-password')
    }
  });

  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      {
        name: 'JDBC_URL',
        value: $CfFormat(
          'jdbc:postgresql://{}:{}/{}',
          $ResourceParam('database', 'host'),
          $ResourceParam('database', 'port'),
          $ResourceParam('database', 'dbName')
        )
      }
    ]
  });

  return { resources: { database, handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

The key difference from `$Format()`: `$CfFormat()` accepts other runtime directives as arguments. `$Format()` only works with locally-resolved values. Under the hood, `$CfFormat` produces a CloudFormation `Fn::Sub` intrinsic function.

### $CfStackOutput()

References an exported output from another deployed stack, resolved at CloudFormation deployment time. Unlike `$StackOutput()` (which fetches the value locally), `$CfStackOutput()` creates a CloudFormation `ImportValue` dependency — this means **the referenced stack cannot be deleted** while this stack exists, protecting shared infrastructure from accidental removal.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfStackOutput
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      {
        name: 'SHARED_QUEUE_URL',
        value: $CfStackOutput('shared-infra-production', 'QueueQueueUrl')
      }
    ]
  });

  return { resources: { handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

Arguments: `$CfStackOutput('stackName', 'outputKeyName')`. The stack name is the full CloudFormation stack name (formatted as `projectName-stage`).

| Directive | Resolution time | Deletion protection | Use when |
|-----------|----------------|---------------------|----------|
| `$StackOutput()` | Config parse time (local) | None | You want a snapshot of the value; referenced stack can be deleted independently |
| `$CfStackOutput()` | CloudFormation deploy time | Referenced stack is locked | You need a live dependency that prevents accidental deletion of shared resources |

## Directive nesting

Directives can be nested, but only runtime directives can be nested inside other runtime directives. A local directive cannot depend on a runtime directive's result (because local directives resolve before deployment begins).

Valid nesting — a runtime directive inside another runtime directive:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfFormat,
  $ResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),

      {
        name: 'API_ENDPOINT',
        value: $CfFormat('{}/v1', $ResourceParam('api', 'url'))
      }

`
    }
  ]}
/>

In TypeScript configs, nesting constraints rarely apply because you use native JavaScript for local operations. The nesting rules matter primarily in YAML configs where you compose directives inside directive arguments.

## Complete directive summary

| Directive | Category | Arguments | Description |
|-----------|----------|-----------|-------------|
| `$Stage()` | Local | None | Current stage name |
| `$Region()` | Local | None | AWS region |
| `$Profile()` | Local | None | AWS profile name |
| `$AwsAccountId()` | Local | None | Target AWS account ID |
| `$Format()` | Local | `(template, ...values)` | String interpolation with `{}` placeholders |
| `$Var()` | Local | None (access via `.property`) | Reference the `variables` section |
| `$This()` | Local | None (access via `.property`) | Reference the raw config object |
| `$File()` | Local | `(filePath)` | Parse .env, JSON, YAML, or INI file |
| `$FileRaw()` | Local | `(filePath)` | Read file as raw string |
| `$GitInfo()` | Local | `(property)` | Git repository information |
| `$CliArgs()` | Local | `(argName, defaultValue?)` | CLI arguments passed after `--` |
| `$StackOutput()` | Local | `(stackName, outputKey, region?)` | Output from another deployed stack |
| `$Secret()` | Runtime | `(secretName)` | AWS Secrets Manager value |
| `$SsmParam()` | Runtime | `(paramName)` | SSM Parameter Store value |
| `$ResourceParam()` | Runtime | `(resourceName, param)` | Parameter from a Stacktape resource |
| `$CfResourceParam()` | Runtime | `(logicalId, attribute)` | Parameter from a raw CF resource |
| `$CfFormat()` | Runtime | `(template, ...values)` | Runtime string interpolation |
| `$CfStackOutput()` | Runtime | `(stackName, outputKey)` | Cross-stack import with deletion lock |

## When to use directives vs alternatives

Not every dynamic value needs a directive. Consider the alternatives:

| Need | Best approach |
|------|--------------|
| Pass a database URL to your code | Use [`connectTo`](/configuration/connecting-resources) — it injects env vars and grants permissions automatically |
| Compose strings from stage/region | In TypeScript: use template literals. In YAML: use `$Format()` |
| Share values across resources | In TypeScript: use a variable. In YAML: use `variables` + `$Var()` |
| Read config from a file | In TypeScript: use `import` or `readFileSync`. In YAML: use `$File()` |
| Reference a secret credential | `$Secret()` — no alternative; secrets must never be in source control |
| Build a string from multiple resource outputs | `$CfFormat()` — the only way to compose runtime values into a string |

## FAQ

### What's the difference between $Secret and $SsmParam?

`$Secret` reads from AWS Secrets Manager, which is designed for credentials, supports automatic rotation, and encrypts values at rest. `$SsmParam` reads from SSM Parameter Store, which supports both plain `String` and encrypted `SecureString` types and is typically used for configuration values, feature flags, and non-credential settings. Secrets Manager costs ~$0.40/secret/month; Parameter Store standard parameters are free.

### Can I use $ResourceParam to reference a resource in a different stack?

No. `$ResourceParam` only references resources within the same stack. For cross-stack references, use `$StackOutput` (local resolution, no deletion protection) or `$CfStackOutput` (runtime resolution with deletion protection). The referenced stack must export the output you're referencing.

### Why can't I use $Secret in the variables section?

Runtime directives like `$Secret` produce CloudFormation intrinsic functions that only resolve during deployment. The `variables` section is processed locally before the CloudFormation template is generated. If you need a secret value in multiple places, reference `$Secret('name')` directly in each resource property, or use `$CfFormat` to compose it with other values.

### How do I pass custom values at deploy time without changing the config?

Use `$CliArgs()` to read arguments passed after `--` in the deploy command. For example, `stacktape deploy --stage prod -- --dbSize=db.r6g.large` lets your config read `$CliArgs('dbSize')` (or use the `defineConfig` parameter in TypeScript). This is useful for values that change per-deploy but shouldn't be committed.

### Can I use directives in TypeScript configs?

Yes. Runtime directives (`$Secret`, `$ResourceParam`, `$CfFormat`, `$CfStackOutput`, `$SsmParam`, `$CfResourceParam`, `$GitInfo`, `$Stage`, `$Region`) are imported from the `stacktape` package. Local directives like `$Var`, `$Format`, `$File`, and `$CliArgs` don't need TypeScript equivalents because native JavaScript provides the same capability.

### What happens if a referenced secret doesn't exist?

Deployment fails with an error message identifying the missing secret and suggesting you create it with [`stacktape secret:create`](/cli/secret-create). Stacktape validates secret existence before submitting the CloudFormation template.

### How does $CfStackOutput prevent accidental deletion?

`$CfStackOutput` produces a CloudFormation `ImportValue` intrinsic function. AWS CloudFormation prevents any stack whose outputs are imported by another stack from being deleted. You must delete the consuming stack first (or remove the `$CfStackOutput` reference) before the shared stack can be removed.

### What's the CloudFormation output key format for Stacktape resources?

When using `$StackOutput` or `$CfStackOutput` to reference outputs from another Stacktape stack, the output key is in PascalCase format combining the resource name and parameter name. For example, a resource named `database` with parameter `connectionString` produces the output key `DatabaseConnectionString`.

### Can I nest $ResourceParam inside $Format?

No. `$Format` is a local directive and cannot accept runtime directives as arguments. Use `$CfFormat` instead — it resolves at deployment time and accepts runtime directives like `$ResourceParam` and `$Secret` as arguments. Example: `$CfFormat('{}/api', $ResourceParam('gateway', 'url'))`.

### How do I access JSON properties from $File or $Secret?

Both support dot-notation property access. For `$Secret`, use `$Secret('my-secret.apiKey')` where `my-secret` is the secret name and `apiKey` is a JSON property. For `$File` in YAML configs, use `$File('config.json').database.host`. The same pattern applies to `$Var()` for accessing nested variables.
