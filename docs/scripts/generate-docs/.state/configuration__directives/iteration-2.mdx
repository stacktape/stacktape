# Directives

Directives are built-in functions (prefixed with `$`) that inject dynamic values into your Stacktape configuration. They let you reference secrets, resource outputs, stage names, files, and cross-stack values without hardcoding anything. Directives bridge the gap between static config and the dynamic reality of deployments where values like database URLs exist only after resources are created.

## Why directives exist

A Stacktape configuration file describes your entire infrastructure declaratively. But some values are inherently dynamic:

- A database connection string is only known after the database is provisioned.
- A secret should never be committed to source control.
- The stage name determines which domain prefix to use.
- A shared infrastructure stack exports values that dependent stacks consume.

Directives solve these problems by deferring value resolution to the right moment — either when Stacktape parses your config (local directives) or when CloudFormation deploys your resources (runtime directives).

## Local vs runtime directives

Stacktape divides directives into two categories based on **when** they resolve:

| Category | Resolved when | Use for |
|----------|---------------|---------|
| **Local** | Stacktape parses your config (before deployment) | Stage name, region, file contents, string formatting, CLI args, git info |
| **Runtime** | CloudFormation deploys resources | Secrets, resource parameters, cross-stack imports, runtime string composition |

The distinction matters because **runtime directives can only appear in the `resources` and `scripts` sections** of your config. Local directives work anywhere — including `variables`, top-level settings, and conditional logic.

<Warning>Using a runtime directive (like `$Secret` or `$ResourceParam`) outside the `resources` or `scripts` sections causes a deployment error. If you need a runtime value in a variable, use `$CfFormat` to compose it within a resource property.</Warning>

## Using directives in TypeScript

In TypeScript configuration, runtime directives are imported as functions from the `stacktape` package. For local operations like variable access, string interpolation, and file loading, you can use native JavaScript instead — template literals, `import`, and `fs.readFileSync` cover most of what `$Var`, `$Format`, and `$File` do in YAML configs.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  RelationalDatabase,
  RdsEnginePostgresql,
  $Secret,
  $ResourceParam,
  $Stage,
  $Region
} from 'stacktape';

// [!code focus-start]
export default defineConfig(({ stage }) => {
  const database = new RelationalDatabase({
    engine: new RdsEnginePostgresql({
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    }),
    credentials: {
      masterUserPassword: $Secret('db-password')
    }
  });

  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      { name: 'DB_CONNECTION_STRING', value: $ResourceParam('database', 'connectionString') },
      { name: 'STAGE', value: $Stage() },
      { name: 'REGION', value: $Region() }
    ],
    connectTo: [database]
  });

  return { resources: { database, api } };
});
// [!code focus-end]
`
    }
  ]}
/>

The following directive functions are importable from the `stacktape` package: `$Secret`, `$SsmParam`, `$ResourceParam`, `$CfResourceParam`, `$CfFormat`, `$CfStackOutput`, `$GitInfo`, `$Stage`, `$Region`.

## Local directives reference

### $Stage()

Returns the current stage name passed to the CLI via `--stage`. Use it to differentiate resource naming, domain prefixes, or feature flags across stages.

<CodeBlock
  intellisense
  tabs={[

// [!code focus-start]
export default defineConfig(({ stage }) => {
  const api = new HttpApiGateway({

      { domainName: \`\${stage}-api.example.com\` }

  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

In TypeScript, the `stage` value is also available directly from the `defineConfig` callback parameter — `defineConfig(({ stage }) => { ... })`. Both approaches are equivalent.

### $Format()

Performs string interpolation using `{}` placeholders. Each `{}` is replaced with the corresponding positional argument. The number of `{}` placeholders must match the number of additional arguments. Use it in YAML configs for dynamic string composition — in TypeScript, prefer template literals instead.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(({ stage, region }) => {
  const api = new HttpApiGateway({
    customDomains: [
      { domainName: \`\${stage}.\${region}.api.example.com\` }
    ]
  });

  return { resources: { api } };
});
// [!code focus-end]`
    }
  ]}
/>

<Info>In TypeScript configs, native template literals replace `$Format()`. In YAML configs, use `$Format('{}-api.example.com', $Stage())` for the same result.</Info>

### $Var()

References a value from the `variables` section of your config. Access nested properties with dot notation: `$Var().myProperty.nested`.

In TypeScript, this directive isn't needed — define a regular JavaScript variable and reference it directly. `$Var()` exists primarily for YAML configs where native variables aren't available.

### $File()

Loads and parses a file relative to your working directory. Supported formats:

| Format | Returns |
|--------|---------|
| `.env` | Array of `{ name, value }` objects |
| `.json` | Parsed JSON object |
| `.yml` / `.yaml` | Parsed YAML object |
| `.ini` | Parsed INI object |

Access nested properties with dot notation: `$File('config.json').database.host`. If the file format is not one of the supported types, Stacktape throws a directive error suggesting you use `$FileRaw` instead.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';
import { readFileSync } from 'fs';

// [!code focus-start]
export default defineConfig(() => {
  const envVars = readFileSync('.env', 'utf-8')
    .split('\\n')
    .filter(line => line.includes('='))
    .map(line => {
      const [name, ...rest] = line.split('=');
      return { name: name.trim(), value: rest.join('=').trim() };
    });

  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: envVars
  });

  return { resources: { handler } };
});
// [!code focus-end]`
    }
  ]}
/>

<Tip>In TypeScript configs, use native `import` or `readFileSync` for file loading. The `$File()` directive is most useful in YAML configs where file loading isn't built in.</Tip>

### $FileRaw()

Reads a file as a raw UTF-8 string without parsing. Use it when you need the literal file content — for example, embedding a JSON config as a string environment variable or reading a template file.

### $GitInfo()

Returns information about the current git repository. Pass the property name as a string argument: `$GitInfo('branch')`.

Available properties:

| Property | Description |
|----------|-------------|
| `commit` | Current commit SHA |
| `branch` | Current branch name |
| `username` | Git username |
| `gitUrl` | Repository URL |

When git commands are unavailable (such as in some CI environments), Stacktape's git info manager can fall back to CI-specific environment variables for properties like commit SHA and branch name.

### $CliArgs()

Accesses additional CLI arguments passed after `--` in the deploy command. Accepts a second optional argument as a default value if the argument isn't provided.

Pass custom arguments when deploying:

```bash
stacktape deploy --stage production --region eu-west-1 -- --featureFlag=true --apiVersion=v2
```

In your config, reference them with `$CliArgs('featureFlag')`. The optional second argument provides a fallback: `$CliArgs('featureFlag', 'false')` returns `'false'` if `--featureFlag` wasn't passed.

### $StackOutput()

References an exported output from another deployed CloudFormation stack. The referenced stack must already be deployed. This is a **local** directive — it fetches the value at config-parse time by querying CloudFormation.

Arguments: `$StackOutput('stackName', 'outputKey')` with an optional third argument for `region` when importing from a stack in another AWS region.

The `stackName` is the full CloudFormation stack name (formatted as `projectName-stage`). The `outputKey` must match the key as it appears in the CloudFormation stack outputs. Use [`stacktape info:stack`](/cli/info-stack) to discover available output keys for a deployed stack.

<Warning>The referenced output must be exported from its stack. Unexported outputs cannot be referenced and will cause a deployment error. The referenced stack must also be deployed before the stack that uses the directive.</Warning>

### $Profile()

Returns the name of the AWS profile being used for the current deployment. Useful when you need to pass profile information to scripts or hooks.

### $AwsAccountId()

Returns the AWS account ID of the target account. Use it when you need the account ID for ARN construction or cross-account references.

### $This()

Returns the raw configuration object. Access nested properties with dot notation: `$This().resources.myApi`. This is primarily useful in YAML configs for self-referencing parts of the config.

## Runtime directives reference

Runtime directives resolve during CloudFormation deployment. They produce CloudFormation intrinsic functions under the hood, which means the actual values are only available when AWS provisions your resources.

### $Secret()

References a secret stored in AWS Secrets Manager. The secret must exist in the same region you're deploying to. Create secrets using [`stacktape secret:create`](/cli/secret-create) or through the [Stacktape Console](/configuration/secrets).

<CodeBlock
  intellisense

// [!code focus-start]

  const database = new RelationalDatabase({
    engine: new RdsEnginePostgresql({
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    }),

// [!code focus-end]`
    }
  ]}
/>

Deployment fails with an error if Stacktape cannot fetch the named secret, with a hint to create it using [`stacktape secret:create`](/cli/secret-create).

For JSON-formatted secrets, use dot notation to extract specific keys:

Create a JSON secret with the CLI:

```bash

```

When prompted, provide a name (e.g., `api-keys`) and a JSON value (e.g., `{"stripe": "sk_...", "sendgrid": "SG..."}`).

### $SsmParam()

References a parameter from AWS Systems Manager Parameter Store. `$SsmParam()` reads the parameter and uses the secure dynamic reference form (`ssm-secure`) when AWS reports the parameter type as `SecureString`; otherwise it uses the standard SSM dynamic reference form (`ssm`). Use SSM parameters when you need values that change without redeployment or when integrating with other AWS tools that write to Parameter Store.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $SsmParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      { name: 'FEATURE_FLAGS', value: $SsmParam('/myapp/feature-flags') },
      { name: 'DB_HOST', value: $SsmParam('/prod/database/host') }
    ]
  });

  return { resources: { handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

SSM parameters support hierarchical paths (e.g., `/myapp/prod/database/host`). The referenced parameter must exist before deployment — if missing, deployment fails with a directive error.

<Info>`$Secret` resolves values from AWS Secrets Manager. `$SsmParam` resolves values from SSM Parameter Store. Use `$Secret` for credentials and sensitive values. Use `$SsmParam` for configuration parameters, feature flags, and values managed outside Stacktape.</Info>

### $ResourceParam()

References a parameter from another Stacktape resource in the same stack. The value resolves at deployment time because it depends on CloudFormation-provisioned outputs like URLs, ARNs, and connection strings. See [Referenceable parameters](/configuration/referenceable-parameters) for the full list of parameters available per resource type.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  HttpApiGateway,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $ResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new HttpApiGateway({});

  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [{ name: 'API_URL', value: $ResourceParam('api', 'url') }]
  });

  return { resources: { api, handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

The first argument is the resource name (as defined in the `resources` object you return from `defineConfig`). The second argument is the parameter name. If the resource or parameter doesn't exist, Stacktape lists the available options in the error message.

<Tip>For most resource-to-resource communication, prefer [`connectTo`](/configuration/connecting-resources) which automatically injects environment variables and grants IAM permissions. Use `$ResourceParam` when you need a value in a non-standard location or when `connectTo` doesn't cover your use case.</Tip>

### $CfResourceParam()

References a parameter from a raw CloudFormation resource defined in the `cloudformationResources` section. Use this when integrating with [raw CloudFormation resources](/resources/advanced/raw-cloudformation-resources) that aren't managed by Stacktape's resource abstraction.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [{ name: 'TOPIC_ARN', value: $CfResourceParam('mySnsTopic', 'TopicArn') }]
  });

  return {
    resources: { handler },
    cloudformationResources: {
      mySnsTopic: {
        Type: 'AWS::SNS::Topic',
        Properties: { TopicName: 'my-notifications' }
      }
    }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The first argument is the logical ID of the CloudFormation resource. The second argument is the attribute name — either a `GetAtt` attribute (like `TopicArn`, `Arn`, `DomainName`) or a `Ref` attribute (which returns the resource's primary identifier). For `AWS::CloudFormation::CustomResource` types, any attribute name is accepted. Available attributes for standard types depend on the CloudFormation resource type — see the [AWS CloudFormation documentation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-template-resource-type-ref.html) for each type's return values.

### $CfFormat()

Performs string interpolation at CloudFormation deployment time, similar to `$Format()` but capable of including runtime directive values as arguments. Use `$CfFormat` when you need to combine multiple runtime values into a single string. Under the hood, `$CfFormat` produces a CloudFormation `Fn::Sub` intrinsic function.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  RelationalDatabase,
  RdsEnginePostgresql,
  $Secret,
  $CfFormat,
  $ResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const database = new RelationalDatabase({
    engine: new RdsEnginePostgresql({
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    }),
    credentials: {
      masterUserPassword: $Secret('db-password')
    }
  });

  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      {
        name: 'JDBC_URL',
        value: $CfFormat(
          'jdbc:postgresql://{}:{}/{}',
          $ResourceParam('database', 'host'),
          $ResourceParam('database', 'port'),
          $ResourceParam('database', 'dbName')
        )
      }
    ]
  });

  return { resources: { database, handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

The key difference from `$Format()`: `$CfFormat()` accepts other runtime directives as arguments. `$Format()` only works with locally-resolved values.

### $CfStackOutput()

References an exported output from another deployed stack, resolved at CloudFormation deployment time. Unlike `$StackOutput()` (which fetches the value locally), `$CfStackOutput()` creates a CloudFormation `ImportValue` dependency — this means **the referenced stack cannot be deleted** while this stack exists, protecting shared infrastructure from accidental removal.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfStackOutput
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      {
        name: 'SHARED_QUEUE_URL',
        value: $CfStackOutput('shared-infra-production', 'QueueQueueUrl')
      }
    ]
  });

  return { resources: { handler } };
});
// [!code focus-end]
`
    }
  ]}
/>

Arguments: `$CfStackOutput('stackName', 'outputKey', region?)`. The stack name is the full CloudFormation stack name (formatted as `projectName-stage`). The optional `region` argument allows importing outputs from a stack deployed in a different AWS region.

<Warning>The referenced output must be exported from its stack. Unexported outputs cannot be referenced and will cause a deployment error.</Warning>

| Directive | Resolution time | Deletion protection | Use when |
|-----------|----------------|---------------------|----------|
| `$StackOutput()` | Config parse time (local) | None | You want a snapshot of the value; referenced stack can be deleted independently |
| `$CfStackOutput()` | CloudFormation deploy time | Referenced stack is locked | You need a live dependency that prevents accidental deletion of shared resources |

## Directive nesting

Directives can be nested, but only runtime directives can be nested inside other runtime directives. A local directive cannot depend on a runtime directive's result because local directives resolve before deployment begins.

Valid nesting — a runtime directive inside another runtime directive:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfFormat,
  $ResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({ entryfilePath: './src/handler.ts' }),
    environment: [
      {
        name: 'API_ENDPOINT',
        value: $CfFormat('{}/v1', $ResourceParam('api', 'url'))
      }
    ]
  });

});

`
    }
  ]}
/>

In TypeScript configs, nesting constraints rarely apply because you use native JavaScript for local operations. The nesting rules matter primarily in YAML configs where you compose directives inside directive arguments.

## Complete directive summary

| Directive | Category | Arguments | Description |
|-----------|----------|-----------|-------------|
| `$Stage()` | Local | None | Current stage name |
| `$Region()` | Local | None | AWS region |
| `$Profile()` | Local | None | AWS profile name |
| `$AwsAccountId()` | Local | None | Target AWS account ID |
| `$Format()` | Local | `(template, ...values)` | String interpolation with `{}` placeholders |
| `$Var()` | Local | None (access via `.property`) | Reference the `variables` section |
| `$This()` | Local | None (access via `.property`) | Reference the raw config object |
| `$File()` | Local | `(filePath)` | Parse .env, JSON, YAML, or INI file |
| `$FileRaw()` | Local | `(filePath)` | Read file as raw string |
| `$GitInfo()` | Local | `(property)` | Git repository information |
| `$CliArgs()` | Local | `(argName, defaultValue?)` | CLI arguments passed after `--` |
| `$StackOutput()` | Local | `(stackName, outputKey, region?)` | Exported output from another deployed stack |
| `$Secret()` | Runtime | `(secretName)` | AWS Secrets Manager value |
| `$SsmParam()` | Runtime | `(paramName)` | SSM Parameter Store value |
| `$ResourceParam()` | Runtime | `(resourceName, param)` | Parameter from a Stacktape resource |
| `$CfResourceParam()` | Runtime | `(logicalId, attribute)` | Parameter from a raw CF resource |
| `$CfFormat()` | Runtime | `(template, ...values)` | Runtime string interpolation via `Fn::Sub` |
| `$CfStackOutput()` | Runtime | `(stackName, outputKey, region?)` | Cross-stack import with deletion lock |

## When to use directives vs alternatives

Not every dynamic value needs a directive. Consider the alternatives:

| Need | Best approach |
|------|--------------|
| Pass a database URL to your code | Use [`connectTo`](/configuration/connecting-resources) — it injects env vars and grants permissions automatically |
| Compose strings from stage/region | In TypeScript: use template literals. In YAML: use `$Format()` |
| Share values across resources | In TypeScript: use a variable. In YAML: use `variables` + `$Var()` |
| Read config from a file | In TypeScript: use `import` or `readFileSync`. In YAML: use `$File()` |
| Reference a secret credential | `$Secret()` — no alternative; secrets must never be in source control |
| Build a string from multiple resource outputs | `$CfFormat()` — the only way to compose runtime values into a string |
| Consume an output from another stack | `$StackOutput()` or `$CfStackOutput()` depending on whether you need deletion protection |

## FAQ

### What's the difference between $Secret and $SsmParam?

`$Secret` resolves values from AWS Secrets Manager, which is designed for credentials and sensitive data. `$SsmParam` resolves values from SSM Parameter Store, which supports both plain `String` and encrypted `SecureString` types and is commonly used for configuration values, feature flags, and non-credential settings. Use `$Secret` for database passwords, API keys, and tokens. Use `$SsmParam` for configuration that changes between environments or is managed by other AWS tools.

### Can I use $ResourceParam to reference a resource in a different stack?

No. `$ResourceParam` only references resources within the same stack. For cross-stack references, use `$StackOutput` (local resolution, no deletion protection) or `$CfStackOutput` (runtime resolution with deletion protection). Both require the referenced output to be exported from its source stack — unexported outputs cannot be referenced and cause a deployment error. See [Referenceable parameters](/configuration/referenceable-parameters) for per-resource parameter details.

### Why can't I use $Secret in the variables section?

Runtime directives like `$Secret` produce CloudFormation intrinsic functions that only resolve during deployment. The `variables` section is processed locally before the CloudFormation template is generated. If you need a secret value in multiple places, reference `$Secret('name')` directly in each resource property, or use `$CfFormat` to compose it with other values within a resource.

### How do I pass custom values at deploy time without changing the config?

Use `$CliArgs()` to read arguments passed after `--` in the deploy command. For example, `stacktape deploy --stage prod -- --dbSize=db.r6g.large` lets your config read `$CliArgs('dbSize')`. You can also pass an optional second argument as a default value: `$CliArgs('dbSize', 'db.t4g.micro')`. This is useful for values that change per-deploy but shouldn't be committed.

### Can I use directives in TypeScript configs?

Yes. Runtime directives (`$Secret`, `$ResourceParam`, `$CfFormat`, `$CfStackOutput`, `$SsmParam`, `$CfResourceParam`, `$GitInfo`, `$Stage`, `$Region`) are imported as functions from the `stacktape` package. For local operations like string formatting, file reading, and variable access, you can use native JavaScript instead of the corresponding YAML directives.

### What happens if a referenced secret doesn't exist?

Deployment fails with an error identifying the missing secret and suggesting you create it with [`stacktape secret:create`](/cli/secret-create). The same applies to `$SsmParam` — if the referenced SSM parameter doesn't exist, Stacktape reports the error before the deployment can proceed. Always create secrets and parameters before deploying stacks that reference them.

### How does $CfStackOutput prevent accidental deletion?

`$CfStackOutput` produces a CloudFormation `ImportValue` intrinsic function. AWS CloudFormation prevents any stack whose outputs are imported by another stack from being deleted. You must delete the consuming stack first (or remove the `$CfStackOutput` reference) before the shared stack can be removed. Use `$StackOutput` instead if you don't need this deletion protection.

### How do I find the output key for $StackOutput or $CfStackOutput?

Use [`stacktape info:stack`](/cli/info-stack) to list the available output keys for a deployed stack. The output key name passed to `$StackOutput` or `$CfStackOutput` must exactly match the key as it appears in the CloudFormation stack outputs. Both directives also accept an optional `region` argument for referencing stacks deployed in a different AWS region.

### Can I nest $ResourceParam inside $Format?

No. `$Format` is a local directive and cannot accept runtime directives as arguments. Use `$CfFormat` instead — it resolves at deployment time and accepts runtime directives like `$ResourceParam` and `$Secret` as arguments. Example: `$CfFormat('{}/api', $ResourceParam('gateway', 'url'))`.

### How do I access JSON properties from $Secret?

Use dot notation after the secret name: `$Secret('my-secret.apiKey')` where `my-secret` is the secret name and `apiKey` is a key in the JSON object. The secret value must be valid JSON — if it isn't, Stacktape reports a parsing error. The same dot-notation pattern works with `$File()` in YAML configs for accessing nested properties: `$File('config.json').database.host`.
