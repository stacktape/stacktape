# Custom Domains

Stacktape custom domains let you serve your resources at branded URLs like `api.example.com` instead of auto-generated AWS endpoints. Stacktape handles the hard parts automatically — provisioning free TLS certificates, creating DNS records in Route53, and renewing certificates before they expire. Custom domains work across networking resources, SSR frontends, and static hosting.

## When to use

Custom domains are essential for any production-facing service. Use them when you need:

- **Branded URLs** — customers and integrations see `api.yourcompany.com` instead of a random AWS-generated hostname
- **Stable endpoints** — your domain stays the same across redeployments, stage recreations, and infrastructure changes
- **Multi-stage routing** — use subdomains to separate stages: `api.example.com` for production, `api-staging.example.com` for staging
- **Shared domains across resources** — route `app.example.com` to your frontend and `api.example.com` to your API, both under one root domain

## When NOT to use

- **Development and testing stages** — auto-generated AWS URLs are fine for stages that only your team uses. Adding custom domains to throwaway stages adds setup overhead without benefit.
- **Internal-only services** — [private services](/resources/compute/private-service) that are only reachable within your VPC don't need public custom domains. Use service discovery instead.
- **Rapid prototyping** — if you're iterating quickly and haven't registered a domain yet, start without custom domains. You can add them later without rebuilding your stack.

<Tip>You don't need a custom domain to get HTTPS. Every Stacktape resource that exposes a public URL already has TLS enabled on its auto-generated endpoint.</Tip>

## Supported resources

Custom domains can be attached to a wide range of Stacktape resources. The `customDomains` property accepts an array of domain configurations on each.

| Resource | Page | Typical use case |
|----------|------|------------------|
| [HTTP API Gateway](/resources/networking/http-api-gateway) | Networking | `api.example.com` for Lambda-backed APIs |
| [Application Load Balancer](/resources/networking/application-load-balancer) | Networking | `app.example.com` for container workloads |
| [Network Load Balancer](/resources/networking/network-load-balancer) | Networking | TCP/TLS services at a custom domain |
| [CDN](/resources/networking/cdn) | Networking | `cdn.example.com` for cached content delivery |
| [Web service](/resources/compute/web-service) | Compute | `api.example.com` directly on a container service |
| [Next.js](/resources/frontend/nextjs) | Frontend | `www.example.com` for Next.js apps |
| [Astro](/resources/frontend/astro) | Frontend | SSR frontend at a custom domain |
| [Nuxt](/resources/frontend/nuxt) | Frontend | SSR frontend at a custom domain |
| [SvelteKit](/resources/frontend/sveltekit) | Frontend | SSR frontend at a custom domain |

The configuration shape is the same across all these resources — once you've set up a custom domain on one resource, the same pattern works everywhere.

## Basic example

Before attaching a custom domain, your domain must have a [Route53 hosted zone](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/hosted-zones-working-with.html) in your AWS account. Run [`stacktape domain:add`](/cli/domain-add) to create one — it walks you through creating the hosted zone, updating your registrar's nameservers to point to Route53, and provisioning TLS certificates.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myApi = new HttpApiGateway({
    customDomains: [{ domainName: 'api.example.com' }]
  });

  return {
    resources: { myApi }
  };
});
// [!code focus-end]`
    }
  ]}
/>

With just `domainName`, Stacktape automatically provisions a free TLS certificate via AWS Certificate Manager (ACM) and creates an alias DNS record in Route53 pointing to the resource's endpoint. The certificate is renewed automatically before expiration — no manual intervention needed.

The `domainName` value should not include a protocol prefix (`https://`). Use the bare domain: `api.example.com`, not `https://api.example.com`.

<Warning>The `domain:add` step is a one-time setup per root domain. If you've already added `example.com`, you can attach any subdomain (`api.example.com`, `app.example.com`, `staging-api.example.com`) without running `domain:add` again.</Warning>

## TLS certificates

Stacktape provisions and manages free TLS certificates by default. For most teams, this is all you need — certificates are issued via ACM, validated through DNS, and renewed automatically.

If you have specific certificate requirements — such as Extended Validation (EV) or Organization Validation (OV) certificates for compliance — provide your own ACM certificate ARN using `customCertificateArn`. The certificate must already exist in your AWS account.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myApi = new HttpApiGateway({
    customDomains: [
      {
        domainName: 'api.example.com',
        customCertificateArn: 'arn:aws:acm:eu-west-1:123456789012:certificate/abc-def-123'
      }
    ]
  });

  return {
    resources: { myApi }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

When you supply `customCertificateArn`, Stacktape skips automatic certificate provisioning and uses your certificate instead. You are responsible for renewing custom certificates before they expire.

<Info>For CDN-attached resources, AWS requires the certificate to be in the `us-east-1` region, regardless of your stack's region. Stacktape handles this automatically for auto-provisioned certificates, but if you provide a custom certificate for a CDN domain, ensure it's in `us-east-1`.</Info>

**When to use a custom certificate:** Only when compliance or organizational policy requires EV/OV certificates, or when you need to use a certificate from an external CA imported into ACM. For all other cases, the auto-provisioned free certificate is the right choice.

## External DNS

If you manage DNS through an external provider like Cloudflare, Namecheap DNS, or any non-Route53 service, set `disableDnsRecordCreation` to `true`. Stacktape will still provision and validate the TLS certificate, but it won't create or modify any DNS records in Route53. You create the DNS record yourself, pointing it to the resource's endpoint.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, ApplicationLoadBalancer } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myAlb = new ApplicationLoadBalancer({
    customDomains: [
      {
        domainName: 'app.example.com',
        disableDnsRecordCreation: true
      }
    ]
  });

  return {
    resources: { myAlb }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

After deploying, retrieve the resource's endpoint using `$ResourceParam` or [`stacktape param:get`](/cli/param-get), then create a CNAME or alias record in your DNS provider pointing `app.example.com` to that endpoint.

<Tip>Even with `disableDnsRecordCreation: true`, you still need a Route53 hosted zone for the domain. Stacktape uses it for DNS-based certificate validation — the hosted zone must exist so ACM can verify you own the domain. The `domain:add` command handles this setup.</Tip>

**When to use external DNS:** When your domain's authoritative DNS is managed outside Route53 (Cloudflare, Google Cloud DNS, etc.) and you don't want Stacktape touching your DNS records. This is common when Cloudflare is handling CDN/WAF at the DNS layer, or when your DNS is managed by a separate team.

## Multiple domains

You can attach multiple custom domains to a single resource by adding entries to the `customDomains` array. Each domain gets its own TLS certificate and DNS record (unless you disable DNS creation). This is useful for serving the same backend under different brands or domain names.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myApi = new HttpApiGateway({
    customDomains: [
      { domainName: 'api.example.com' },
      { domainName: 'api.mycompany.io' },
      { domainName: 'api-legacy.example.com', disableDnsRecordCreation: true }
    ]
  });

  return {
    resources: { myApi }
  };
});
// [!code focus-end]`
    }
  ]}
/>

Each domain in the array is configured independently — you can mix auto-managed DNS with external DNS, and auto-provisioned certificates with custom certificates, within the same resource. All domains route to the same underlying endpoint.

Each root domain (e.g., `example.com` and `mycompany.io`) must have its own Route53 hosted zone set up via [`stacktape domain:add`](/cli/domain-add).

## FAQ

### Which Stacktape resources support custom domains?

Custom domains work on [HTTP API Gateways](/resources/networking/http-api-gateway), [Application Load Balancers](/resources/networking/application-load-balancer), [Network Load Balancers](/resources/networking/network-load-balancer), [CDN](/resources/networking/cdn) distributions, [web services](/resources/compute/web-service), all SSR frontends ([Next.js](/resources/frontend/nextjs), [Astro](/resources/frontend/astro), [Nuxt](/resources/frontend/nuxt), [SvelteKit](/resources/frontend/sveltekit), [SolidStart](/resources/frontend/solidstart), [TanStack Start](/resources/frontend/tanstack-start), [Remix](/resources/frontend/remix)), and [static hosting](/resources/frontend/static-hosting). The `customDomains` configuration is the same across all of them.

### How do I set up a Route53 hosted zone for my domain?

Run [`stacktape domain:add`](/cli/domain-add). It creates a Route53 hosted zone, shows you the nameservers to configure at your domain registrar, verifies DNS ownership, and provisions TLS certificates. This is a one-time setup per root domain — after adding `example.com`, any subdomain works without repeating the process.

### Can I use a domain registered outside AWS?

Yes. You don't need to transfer your domain to AWS Route53. Register and keep the domain at any registrar (GoDaddy, Namecheap, Google Domains, etc.). Run [`stacktape domain:add`](/cli/domain-add), then update your registrar's nameservers to point to the Route53 hosted zone nameservers. Your registrar handles registration; Route53 handles DNS resolution.

### Does Stacktape handle TLS certificate renewal?

Yes. Auto-provisioned certificates are managed by AWS Certificate Manager and renewed automatically before expiration with no downtime. If you provide a custom certificate via `customCertificateArn`, you are responsible for renewing it yourself.

### How much do custom domains cost on AWS?

Route53 hosted zones cost $0.50/month per zone. DNS queries cost $0.40 per million queries. ACM certificates are free. The primary cost is the hosted zone — one zone covers unlimited subdomains, so `api.example.com`, `app.example.com`, and `staging.example.com` all share the same $0.50/month zone for `example.com`.

### How long does it take for a custom domain to start working?

TLS certificate provisioning via DNS validation typically completes within minutes after the Route53 hosted zone is properly configured. DNS propagation for the new records usually takes a few minutes but can take up to 48 hours in edge cases. Subsequent deployments that don't change the domain are faster since the certificate already exists.

### Can I use Cloudflare with Stacktape custom domains?

Yes. Set `disableDnsRecordCreation: true` so Stacktape doesn't create Route53 DNS records. You still need a Route53 hosted zone for certificate validation, but your live traffic DNS records stay in Cloudflare. Point the Cloudflare CNAME to the resource endpoint after deployment.

### Should I use custom domains on staging and development stages?

For most teams, no. Auto-generated AWS URLs are sufficient for non-production stages. If you do want branded staging URLs, use subdomain conventions like `api-staging.example.com` or `staging.api.example.com`. This adds no extra cost beyond DNS queries since all subdomains share the same hosted zone.

### Can I attach the same domain to multiple resources?

No. A domain name can only point to one resource at a time. To route different paths to different backends under the same domain, place an [HTTP API Gateway](/resources/networking/http-api-gateway) or [Application Load Balancer](/resources/networking/application-load-balancer) in front and use path-based routing to direct traffic to multiple Lambda functions or container workloads.

### What happens to my custom domain if I delete the stack?

Deleting a stack removes the DNS records and TLS certificate association, but does not delete the Route53 hosted zone or your domain registration. The hosted zone persists for reuse in future stacks. To fully clean up, delete the hosted zone manually in the AWS Console after confirming no other stacks reference the domain.
