# Custom Domains

Stacktape custom domains attach your own domain names (like `api.example.com`) to resources instead of using auto-generated AWS endpoints. When you add a custom domain, Stacktape provisions a free TLS certificate via AWS Certificate Manager, creates a DNS record in Route53, and renews the certificate automatically. The `DomainConfiguration` type is shared across networking resources, compute resources, and frontends.

## When to use

Custom domains are essential for any production-facing resource. Use them when you need:

- **Branded URLs** — serve your API at `api.yourcompany.com` instead of a random AWS-generated hostname
- **Stable endpoints** — your domain stays the same across redeployments, stage recreations, and infrastructure changes
- **Multi-stage routing** — use subdomains to separate stages: `api.example.com` for production, `api-staging.example.com` for staging
- **Shared root domain** — route `app.example.com` to your frontend and `api.example.com` to your API gateway, both under one root domain

## When NOT to use

- **Development and testing stages** — auto-generated AWS URLs work fine for stages that only your team uses. Adding custom domains to throwaway stages adds setup overhead without benefit.
- **Internal-only services** — [private services](/resources/compute/private-service) that are only reachable within your VPC don't need public custom domains. Use service discovery instead.
- **Rapid prototyping** — if you haven't registered a domain yet, start without custom domains. You can add them later without rebuilding your stack.

Every Stacktape resource that exposes a public URL already has TLS enabled on its auto-generated endpoint — you don't need a custom domain just to get HTTPS.

## Supported resources

The `customDomains` property accepts an array of `DomainConfiguration` objects. The domain resolver attaches domains to four core networking resource types:

| Resource | Page | Typical use case |
|----------|------|------------------|
| [HTTP API Gateway](/resources/networking/http-api-gateway) | Networking | `api.example.com` for Lambda-backed APIs |
| [Application Load Balancer](/resources/networking/application-load-balancer) | Networking | `app.example.com` for container workloads |
| [Network Load Balancer](/resources/networking/network-load-balancer) | Networking | TCP/TLS services at a custom domain |
| [CDN](/resources/networking/cdn) | Networking | `cdn.example.com` for cached content delivery |

Several other resource types also expose a `customDomains` property that accepts the same `DomainConfiguration` array — including [web services](/resources/compute/web-service), SSR frontends ([Next.js](/resources/frontend/nextjs), [Astro](/resources/frontend/astro), [Nuxt](/resources/frontend/nuxt), [SvelteKit](/resources/frontend/sveltekit), [SolidStart](/resources/frontend/solidstart), [TanStack Start](/resources/frontend/tanstack-start), [Remix](/resources/frontend/remix)), and [static hosting](/resources/frontend/static-hosting). On these resources, the domain is attached through the resource's underlying CDN or load balancer. See each resource's dedicated page for specifics on how its custom domain support works.

The core `DomainConfiguration` structure is the same across all resources that accept it. Once you understand the three properties (`domainName`, `customCertificateArn`, `disableDnsRecordCreation`), the pattern applies everywhere. Application Load Balancers and Network Load Balancers also accept a legacy string-array format for backward compatibility.

## Basic example

Before attaching a custom domain, your domain must have a [Route53 hosted zone](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/hosted-zones-working-with.html) in your AWS account, with your registrar's nameservers pointing to it. See [`stacktape domain:add`](/cli/domain-add) for the CLI workflow.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myApi = new HttpApiGateway({
    customDomains: [{ domainName: 'api.example.com' }]
  });

  return {
    resources: { myApi }
  };
});
// [!code focus-end]`
    }
  ]}
/>

With just `domainName`, Stacktape automatically creates a DNS record and provisions a free TLS certificate. The certificate is renewed automatically before expiration — no manual intervention needed.

The `domainName` value should not include a protocol prefix (`https://`). Use the bare domain: `api.example.com`, not `https://api.example.com`.

<Tip>The `domain:add` step is a one-time setup per root domain. After adding `example.com`, you can attach any subdomain (`api.example.com`, `app.example.com`, `staging-api.example.com`) without running `domain:add` again.</Tip>

## TLS certificates

Stacktape provisions and manages free TLS certificates by default. For most teams, this is all you need — certificates are issued via ACM, validated through DNS, and renewed automatically.

If you have specific certificate requirements — such as Extended Validation (EV) or Organization Validation (OV) certificates for compliance — provide your own ACM certificate ARN using `customCertificateArn`. The certificate must already exist in your AWS account.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myApi = new HttpApiGateway({
    customDomains: [
      {
        domainName: 'api.example.com',
        customCertificateArn: 'arn:aws:acm:eu-west-1:123456789012:certificate/abc-def-123'
      }
    ]
  });

  return {
    resources: { myApi }
  };
});
// [!code focus-end]`
    }
  ]}
/>

When you supply `customCertificateArn`, Stacktape skips automatic certificate provisioning and uses your certificate instead. You are responsible for renewing custom certificates before they expire.

<Tip>AWS CloudFront requires TLS certificates to be in the `us-east-1` region regardless of your stack's region. If you provide a custom certificate via `customCertificateArn` for a CDN-attached domain, ensure it's provisioned in `us-east-1`.</Tip>

**When to use a custom certificate:** Only when compliance or organizational policy requires EV/OV certificates, or when you need to use a certificate from an external CA imported into ACM. For all other cases, the auto-provisioned free certificate is the right choice — it costs nothing and renews itself.

## External DNS

If you manage DNS through an external provider like Cloudflare, Namecheap DNS, or any non-Route53 service, set `disableDnsRecordCreation` to `true`. Stacktape provisions the TLS certificate but does not create or modify the resource's DNS record. You create and manage the DNS record yourself, pointing it to the resource's endpoint after deployment.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, ApplicationLoadBalancer } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myAlb = new ApplicationLoadBalancer({
    customDomains: [
      {
        domainName: 'app.example.com',
        disableDnsRecordCreation: true
      }
    ]
  });

  return {
    resources: { myAlb }
  };
});
// [!code focus-end]`
    }
  ]}
/>

With `disableDnsRecordCreation: true`, you must create a CNAME or alias record in your DNS provider pointing your domain to the resource's endpoint. The `domainName` property still requires a Route53 hosted zone in your AWS account. With `disableDnsRecordCreation: true`, Stacktape provisions the TLS certificate but does not create or modify the resource DNS record.

<Tip>Stacktape still needs to validate domain ownership to provision the TLS certificate. Check [ACM documentation](https://docs.aws.amazon.com/acm/latest/userguide/dns-validation.html) for the validation options available with your DNS provider.</Tip>

**When to use external DNS:** When your domain's authoritative DNS is managed outside Route53 (Cloudflare, Google Cloud DNS, etc.) and you don't want Stacktape touching your DNS records. This is common when Cloudflare handles CDN/WAF at the DNS layer, or when your DNS is managed by a separate team.

## Multiple domains

You can attach multiple custom domains to a single resource by adding entries to the `customDomains` array. Stacktape configures certificate and DNS handling for each domain entry, unless DNS creation is disabled or a custom certificate is supplied. This is useful for serving the same backend under different brands or domain names.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, HttpApiGateway } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myApi = new HttpApiGateway({
    customDomains: [
      { domainName: 'api.example.com' },
      { domainName: 'api.mycompany.io' },
      { domainName: 'api-legacy.example.com', disableDnsRecordCreation: true }
    ]
  });

  return {
    resources: { myApi }
  };
});
// [!code focus-end]`
    }
  ]}
/>

Each domain in the array is configured independently — you can mix auto-managed DNS with external DNS, and auto-provisioned certificates with custom certificates, within the same resource. All domains route to the same underlying endpoint.

Each root domain (e.g., `example.com` and `mycompany.io`) needs its own Route53 hosted zone. See [`stacktape domain:add`](/cli/domain-add) for the CLI workflow.

## FAQ

### Which Stacktape resources support custom domains?

The four core domain-attachable networking resources are [HTTP API Gateways](/resources/networking/http-api-gateway), [Application Load Balancers](/resources/networking/application-load-balancer), [Network Load Balancers](/resources/networking/network-load-balancer), and [CDN](/resources/networking/cdn) distributions. [Web services](/resources/compute/web-service), SSR frontends ([Next.js](/resources/frontend/nextjs), [Astro](/resources/frontend/astro), [Nuxt](/resources/frontend/nuxt), [SvelteKit](/resources/frontend/sveltekit), [SolidStart](/resources/frontend/solidstart), [TanStack Start](/resources/frontend/tanstack-start), [Remix](/resources/frontend/remix)), and [static hosting](/resources/frontend/static-hosting) also accept `customDomains` — the domain is attached through the resource's underlying CDN or load balancer.

### How do I set up a Route53 hosted zone for my domain?

Run [`stacktape domain:add`](/cli/domain-add) to create a Route53 hosted zone for your domain. You'll need to update your domain registrar's nameservers to point to the Route53 hosted zone. This is a one-time setup per root domain — after adding `example.com`, any subdomain works without repeating the process.

### Can I use a domain registered outside AWS?

Yes. You don't need to transfer your domain to AWS Route53. Register and keep the domain at any registrar (GoDaddy, Namecheap, Google Domains, etc.), run [`stacktape domain:add`](/cli/domain-add), then update your registrar's nameservers to point to the Route53 hosted zone nameservers. Your registrar handles registration; Route53 handles DNS resolution.

### Does Stacktape handle TLS certificate renewal?

Yes. Auto-provisioned certificates are managed by AWS Certificate Manager and renewed automatically before expiration with no downtime. If you provide a custom certificate via `customCertificateArn`, you are responsible for renewing it yourself.

### How much do custom domains cost on AWS?

Route53 hosted zones cost $0.50/month per zone. DNS queries cost $0.40 per million queries. ACM certificates are free. The primary cost is the hosted zone — one zone covers unlimited subdomains, so `api.example.com`, `app.example.com`, and `staging.example.com` all share the same $0.50/month zone for `example.com`.

### How long does it take for a custom domain to start working?

TLS certificate provisioning via DNS validation typically completes within minutes after the Route53 hosted zone is properly configured. DNS propagation for the new records usually takes a few minutes but can take up to 48 hours in edge cases. Subsequent deployments that don't change the domain are faster since the certificate already exists.

### Can I use Cloudflare with Stacktape custom domains?

Yes. Set `disableDnsRecordCreation: true` so Stacktape doesn't create Route53 DNS records. Stacktape still needs to validate domain ownership to provision the TLS certificate — check ACM documentation for validation options with your DNS provider. Point the Cloudflare CNAME to the resource endpoint after deployment.

### Should I use custom domains on staging and development stages?

For most teams, no. Auto-generated AWS URLs are sufficient for non-production stages and carry no extra cost. If you do want branded staging URLs, use subdomain conventions like `api-staging.example.com` — this adds no extra hosted-zone cost since all subdomains share the same zone.

### Can I route one domain to multiple backends?

A single DNS name normally resolves to one target endpoint. To route different paths to different backends under the same domain, place an [HTTP API Gateway](/resources/networking/http-api-gateway) or [Application Load Balancer](/resources/networking/application-load-balancer) in front and use path-based routing to direct traffic to multiple Lambda functions or container workloads.

### What happens to my custom domain if I delete the stack?

The Route53 hosted zone created by [`domain:add`](/cli/domain-add) is managed independently of any stack and persists after stack deletion. Deleting a stack removes the DNS records and TLS certificate associations created for that stack, but the hosted zone and your domain registration remain intact for reuse in future stacks.
