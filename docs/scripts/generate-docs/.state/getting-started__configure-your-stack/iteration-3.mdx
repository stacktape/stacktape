# Configure Your Stack

A `stacktape.ts` file defines your AWS infrastructure — APIs, functions, databases, containers — using typed TypeScript classes. Stacktape reads this file for commands that need project configuration, and deployment commands use it to create and manage the resources in your AWS account. Create one from scratch, generate it from a starter, or use the Console's AI config generation.

<Info>Already have a `stacktape.ts`? Skip to [using dev mode](/getting-started/use-the-dev-mode) or [deploying your first stage](/getting-started/deploy-your-first-stage).</Info>

## Install the package

Add `stacktape` to your project. The package includes all resource classes, directive functions, and type definitions — giving you full IDE autocompletion and type checking as you write your config.

```bash
npm install stacktape --save-dev
```

## Your first stacktape.ts

Create a `stacktape.ts` file in your project root. This example deploys a [Lambda function](/resources/compute/lambda-function) behind an [HTTP API Gateway](/resources/networking/http-api-gateway) — a common starting point for REST APIs:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  HttpApiGateway,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  HttpApiIntegration
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new HttpApiGateway({});

  const myFunction = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    memory: 512,
    timeout: 30,
    events: [
      new HttpApiIntegration({
        httpApiGatewayName: api.resourceName,
        method: 'GET',
        path: '/hello'
      })
    ]
  });

  return {
    resources: { api, myFunction }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

Here's what each piece does:

- **`defineConfig`** wraps your config in a function that Stacktape calls at command time. It provides deployment context — stage, region, project name — as callback parameters you can use for [dynamic configuration](#dynamic-configuration).
- **Resource classes** like `LambdaFunction` and `HttpApiGateway` map to AWS resources. Each accepts a typed configuration object with full autocompletion in any TypeScript-capable editor (VS Code, WebStorm, etc.).
- **`StacktapeLambdaBuildpackPackaging`** points the Lambda function at an entry file such as `./src/handler.ts`. See [function packaging](/packaging/function/stacktape-buildpack) for packaging behavior and options.
- **`HttpApiIntegration`** connects HTTP requests to your function through the API Gateway. The `httpApiGatewayName` uses `api.resourceName` — each resource class exposes a `resourceName` getter that resolves to the key used in the `resources` return object, linking this integration to the `api` gateway.
- **The `resources` return** registers everything in your stack. The object keys (`api`, `myFunction`) become Stacktape resource names. These names are used by [`connectTo`](/configuration/connecting-resources) to wire up access between resources — for `connectTo`, environment variable names follow the pattern `STP_[RESOURCE_NAME]_[PARAM]` (e.g., a resource named `myFunction` produces `STP_MY_FUNCTION_ARN`). The same names work with [`$ResourceParam()`](/configuration/directives) to reference outputs from other resources.

The handler at `./src/handler.ts` is standard application code:

```typescript
export default async () => {
  return {
    statusCode: 200,
    body: JSON.stringify({ message: 'Hello from Stacktape!' })
  };
};
```

## Dynamic configuration

The `defineConfig` callback receives a context object you can use to vary infrastructure per [stage](/configuration/stages-and-environments). Since the config is TypeScript, conditionals, loops, shared modules, and any other language feature work naturally — you manage every stage from a single file with full type checking.

### Context properties

| Property      | Type       | Description                                            |
| ------------- | ---------- | ------------------------------------------------------ |
| `stage`       | `string`   | Current stage name (e.g., `dev`, `production`)         |
| `region`      | `string`   | AWS region (e.g., `us-east-1`, `eu-west-1`)            |
| `projectName` | `string`   | Project name                                           |
| `command`     | `string`   | CLI command being run (`deploy`, `delete`, `dev`, etc.) |
| `cliArgs`     | `Record<string, string \| number \| boolean>` | Additional arguments passed via the `--arg` CLI flag   |

See [configuration files](/configuration/configuration-files) for additional context properties.

### Stage-based patterns

Use TypeScript expressions to customize resources per stage. For example, pick a larger database instance in production while keeping development costs low, or enable a custom domain only for production:

```typescript
// Use stage from the callback context to vary per environment
const instanceSize = stage === 'production' ? 'db.t4g.medium' : 'db.t4g.micro';
const enableCustomDomain = stage === 'production';
const workerMemory = stage === 'production' ? 2048 : 512;
```

You can also use loops to generate resources dynamically, extract shared config into separate TypeScript modules, or use any npm package to compute values. [YAML configuration](/configuration/configuration-files) is also supported but cannot express this kind of logic inline.

## Quick-start with stacktape init

Use [`stacktape init`](/cli/init) to scaffold a project with a working `stacktape.ts` and application code:

```bash
npx stacktape@latest init
```

The command walks you through selecting a starter project and generates a ready-to-deploy setup. See the [`stacktape init` CLI reference](/cli/init) for the current options.

## Console alternatives

The [Stacktape Console](/stacktape-console/console-overview) offers two additional paths for creating a config:

**AI config generation** — In the Console, select a public or private Git repository and branch. Stacktape analyzes the codebase and creates a config draft for your review. This is the fastest path when you have an existing codebase and want Stacktape to determine the right infrastructure. See [AI config generation](/using-with-ai/config-generation-from-repository) for the full walkthrough.

**Config editor** — A browser-based editor where you can open a saved config template, edit it, and save changes. Some stage-created configs are read-only in the template editor and must be edited from the stage configuration page. Use the editor alongside AI config generation to review and refine generated configs before deploying. See the [visual config editor](/stacktape-console/visual-config-editor) for details.

Both paths work well for exploration and getting started. For production workflows, keep your `stacktape.ts` in version control alongside your application code — it's the single source of truth for your infrastructure.

## What you can configure

A Stacktape config has one required section (`resources`) and several optional sections you can add as your project grows:

- **[resources](/configuration/resources)** (required) — your infrastructure: APIs, databases, containers, functions, buckets, and more.
- **[scripts](/configuration/hooks-and-scripts)** — shell commands or code you can run with [`stacktape script:run`](/cli/script-run) or attach to lifecycle hooks. Common uses: database migrations, seed scripts, build steps.
- **[hooks](/configuration/hooks-and-scripts)** — run scripts automatically before or after `deploy`, `delete`, `dev`, or `bucket:sync` commands. Example: run Prisma migrations after every deploy.
- **[variables](/configuration/variables-and-reuse)** — reusable values you can reference anywhere with `$Var().variableName`.
- **[directives](/configuration/directives)** — register custom directive functions that dynamically compute config values at deploy time. Built-in directives like `$Secret()`, `$ResourceParam()`, and `$File()` are always available without registration.
- **[deploymentConfig](/deployment-and-lifecycle/deploying-stacks)** — advanced deployment settings: rollback behavior, termination protection, artifact retention.
- **stackConfig** — stack-wide settings: custom outputs, tags, and VPC configuration.
- **[cloudformationResources](/configuration/overrides-and-escape-hatches)** — an escape hatch to add raw CloudFormation resources alongside Stacktape-managed ones.
