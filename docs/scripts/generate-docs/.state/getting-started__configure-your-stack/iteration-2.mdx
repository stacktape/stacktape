# Configure Your Stack

A `stacktape.ts` file defines your entire AWS infrastructure — APIs, functions, databases, containers — using typed TypeScript classes. Stacktape reads this file when you run any command, provisions every resource in your AWS account, and gives you IDE autocompletion and type checking. This page walks you through creating one.

<Info>Already have a `stacktape.ts`? Skip to [using dev mode](/getting-started/use-the-dev-mode) or [deploying your first stage](/getting-started/deploy-your-first-stage).</Info>

## Install the package

Add `stacktape` to your project. The package includes all TypeScript resource classes, directive functions, and type definitions for your config.

```bash
npm install stacktape --save-dev
```

## Your first stacktape.ts

Create a `stacktape.ts` file in your project root. This example deploys a [Lambda function](/resources/compute/lambda-function) behind an [HTTP API Gateway](/resources/networking/http-api-gateway) — a common pattern for REST APIs:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  HttpApiGateway,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  HttpApiIntegration
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new HttpApiGateway({});

  const myFunction = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    memory: 512,
    timeout: 30,
    events: [
      new HttpApiIntegration({
        httpApiGatewayName: 'api',
        method: 'GET',
        path: '/hello'
      })
    ]
  });

  return {
    resources: { api, myFunction }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

Here's what each piece does:

- **`defineConfig`** wraps your config in a function that Stacktape calls when you run a command (`deploy`, `dev`, `delete`, etc.). It provides deployment context — stage, region, project name — as parameters.
- **Resource classes** like `LambdaFunction` and `HttpApiGateway` map directly to AWS resources. Each class accepts a typed configuration object with full autocompletion in your IDE.
- **`StacktapeLambdaBuildpackPackaging`** tells Stacktape how to build your code. Point it at a TypeScript or JavaScript entrypoint and Stacktape bundles it automatically — no build config needed. See [function packaging](/packaging/function/stacktape-buildpack) for details.
- **`HttpApiIntegration`** connects HTTP requests to your function through the API Gateway. The `httpApiGatewayName: 'api'` references the gateway by the name used in the `resources` return object.
- **The `resources` return** registers everything in your stack. The object keys (`api`, `myFunction`) become Stacktape resource names. When another workload or script uses [`connectTo`](/configuration/connecting-resources) with a resource name, Stacktape uses the name in injected environment variables such as `STP_MY_FUNCTION_ARN`.

The handler at `./src/handler.ts` is standard application code:

```typescript
export default async () => {
  return {
    statusCode: 200,
    body: JSON.stringify({ message: 'Hello from Stacktape!' })
  };
};
```

## Stage-based configuration

The `defineConfig` callback receives a context object you can use to vary infrastructure per [stage](/configuration/stages-and-environments). Since your config is TypeScript, you can use conditionals, loops, shared modules, and any other language feature. For example, write `stage === 'production' ? 'db.t4g.medium' : 'db.t4g.micro'` to pick a larger database instance in production while keeping development costs low.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, RelationalDatabase, RdsEnginePostgres, $Secret } from 'stacktape';

// [!code focus-start]
export default defineConfig(({ stage }) => {
  const database = new RelationalDatabase({
    engine: new RdsEnginePostgres({
      version: '16',
      primaryInstance: {
        instanceSize: stage === 'production' ? 'db.t4g.medium' : 'db.t4g.micro'
      }
    }),
    credentials: {
      masterUserPassword: $Secret('db-password')
    }
  });

  return {
    resources: { database }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The [`$Secret`](/configuration/directives) directive references a secret stored in the [Stacktape Console](/configuration/secrets) — it resolves at deploy time so credentials never appear in your config file. See [variables and reuse](/configuration/variables-and-reuse) for more patterns.

### Context properties

| Property      | Type       | Description                                            |
| ------------- | ---------- | ------------------------------------------------------ |
| `stage`       | `string`   | Current stage name (e.g., `dev`, `production`)         |
| `region`      | `string`   | AWS region (e.g., `us-east-1`, `eu-west-1`)            |
| `projectName` | `string`   | Project name                                           |
| `command`     | `string`   | CLI command being run (`deploy`, `delete`, `dev`, etc.) |
| `cliArgs`     | `string[]` | Additional CLI arguments after `--`                    |

Additional properties (`awsProfile`, `user`) are also available — see [configuration files](/configuration/configuration-files) for the full list.

## Quick-start with stacktape init

Use [`stacktape init`](/cli/init) to create a starter configuration:

```bash
npx stacktape@latest init
```

The command walks you through selecting a starter project and generates a working `stacktape.ts` along with application code and dependencies — ready to deploy. Starters cover common patterns including REST APIs with databases, Next.js apps, container services, and background workers. See [`stacktape init`](/cli/init) for the full option set.

## Console alternatives

The [Stacktape Console](/stacktape-console/console-overview) offers two additional paths for creating a config:

**AI config generation** — Select a Git repository (public or private) and a branch in the Console. Stacktape analyzes the codebase and generates a `stacktape.ts` for your project. This is the fastest path when you have an existing codebase and want Stacktape to figure out the infrastructure. See [AI config generation](/using-with-ai/config-generation-from-repository) for details.

**Config editor** — A browser-based editor where you can open a saved config template, edit it, and save changes. Use it alongside AI config generation to review and refine generated configs before deploying. See [visual config editor](/stacktape-console/visual-config-editor) for the walkthrough.

Both paths are useful for exploration and getting started quickly. For production workflows, keep your `stacktape.ts` in version control alongside your application code — it's the single source of truth for your infrastructure.

## What you can configure

Common top-level sections in a Stacktape config include:

- **[resources](/configuration/resources)** (required) — your app's infrastructure: APIs, databases, containers, functions, buckets, and more.
- **[scripts](/configuration/hooks-and-scripts)** — shell commands or code you can run with [`stacktape script:run`](/cli/script-run) or attach to lifecycle hooks. Common uses: database migrations, seed scripts, build steps.
- **[hooks](/configuration/hooks-and-scripts)** — run scripts automatically before or after `deploy`, `delete`, or `dev` commands. Example: run Prisma migrations after every deploy.
- **[variables](/configuration/variables-and-reuse)** — reusable values you can reference anywhere with `$Var().variableName`.
- **[directives](/configuration/directives)** — register custom directive functions from JS, TS, or Python files that dynamically compute config values. Built-in directives like `$Secret()`, `$ResourceParam()`, and `$File()` are always available without registration.
- **[deploymentConfig](/deployment-and-lifecycle/deploying-stacks)** — advanced deployment settings: rollback behavior, termination protection, artifact retention.
- **stackConfig** — stack-wide settings: custom outputs, tags, and VPC configuration.
- **[cloudformationResources](/configuration/overrides-and-escape-hatches)** — an escape hatch to add raw CloudFormation resources alongside Stacktape-managed ones.

You don't need any of these besides `resources` to get started. Add the other sections as your project grows.
