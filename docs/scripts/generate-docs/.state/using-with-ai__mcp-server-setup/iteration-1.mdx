# MCP Server Setup

<Warning>
The Stacktape MCP server is in preview and under active development. Some operations may not work end-to-end in all environments. Expect breaking changes as the server stabilizes.
</Warning>

The Stacktape MCP (Model Context Protocol) server lets AI coding assistants deploy infrastructure, query databases, read logs, and search Stacktape documentation — all through structured tool calls instead of raw CLI commands. It connects your assistant to four Stacktape tools that cover docs lookup, infrastructure operations, diagnostics, and dev mode control.

## When to use

Use the MCP server when you work with an AI coding assistant that supports the Model Context Protocol. Instead of pasting CLI output back and forth, the assistant calls Stacktape tools directly — getting structured JSON responses it can reason about. This is useful for:

- Deploying or deleting stacks without leaving your editor
- Querying logs, metrics, or database contents during debugging sessions
- Looking up Stacktape configuration syntax without switching to a browser
- Managing secrets from within an AI conversation

## Supported clients

The [`stacktape mcp:add`](/cli/mcp-add) command auto-detects installed clients and writes the correct MCP configuration for each one. Six clients are supported:

| Client | Project config path | Global config path |
|--------|--------------------|--------------------|
| Claude Code | `.mcp.json` | `~/.claude.json` |
| Cursor | `.cursor/mcp.json` | `~/.cursor/mcp.json` |
| VS Code / Copilot | `.vscode/mcp.json` | Platform-specific `User/mcp.json` |
| Windsurf | `.codeium/windsurf/mcp_config.json` | `~/.codeium/windsurf/mcp_config.json` |
| OpenAI Codex | `.codex/config.toml` | `~/.codex/config.toml` |
| OpenCode | `opencode.jsonc` or `.opencode/config.json` | Platform-specific |

When a project-level config file already exists, `mcp:add` writes there. Otherwise it falls back to the global config path. Existing files are backed up with a timestamp before modification.

## Automatic setup

Run [`stacktape mcp:add`](/cli/mcp-add) in your project directory. The command detects which clients are installed, prompts you to confirm, and writes the MCP server entry to each selected client's config file.

```bash
stacktape mcp:add
```

The interactive prompt lists detected clients with a `(detected)` label. Select the ones you want and confirm. If you pass `--autoConfirmOperation`, the command skips the prompt and installs into all detected clients.

```bash
stacktape mcp:add --autoConfirmOperation
```

When your project has `stacktape` as an npm dependency, the installer uses `node_modules/.bin/stacktape` for project-level configs and the global `stacktape` command for global configs. This ensures the MCP server uses the same Stacktape version as your project.

## Manual setup

If you prefer to configure the MCP server manually, point your client at the `stacktape mcp` command using stdio transport. The server communicates over stdin/stdout using the MCP protocol.

## Available tools

The MCP server exposes four tools. Each tool accepts structured parameters and returns JSON responses. Your AI assistant decides which tool to call based on your request.

### stacktape_docs

Searches Stacktape documentation for configuration syntax, resource types, deployment patterns, and troubleshooting guidance. The server maintains a full-text search index over Stacktape's documentation corpus, covering CLI references, resource configuration, concepts, recipes, and getting-started guides.

| Parameter | Type | Description |
|-----------|------|-------------|
| `query` | string (required) | Search query (e.g. "how to configure a lambda function") |
| `mode` | `answer` \| `reference` \| `snippet` | Response mode. `answer` returns full doc content, `reference` returns titles and paths, `snippet` returns code blocks. Default: `answer` |
| `resourceType` | string | Filter to a specific resource type (e.g. `function`, `web-service`, `relational-database`) |
| `docType` | `config-ref` \| `cli-ref` \| `concept` \| `recipe` \| `troubleshooting` \| `getting-started` | Filter to a doc category |
| `maxItems` | number | Max results to return. Default: 3 |

This tool is the default entry point for any Stacktape-related question. The assistant should search docs before answering configuration questions, since Stacktape syntax evolves between versions.

### stacktape_ops

Executes infrastructure operations: deploy, delete, preview changes, manage secrets, run deployment scripts, compile CloudFormation templates, and rollback deployments. Each operation runs as a subprocess and streams structured JSONL events.

| Parameter | Type | Description |
|-----------|------|-------------|
| `operation` | string (required) | One of: `deploy`, `delete`, `preview_changes`, `rollback`, `script_run`, `compile_template`, `secret_create`, `secret_get`, `secret_delete` |
| `args` | object | Operation arguments (e.g. `projectName`, `stage`, `region`, `configPath`, `secretName`, `secretValue`) |
| `confirm` | boolean | Required for `delete` operations. Must be `true` to proceed |

Most operations require `projectName`, `stage`, `region`, and `configPath`. Secret operations require `secretName` (and `secretValue` or `secretFile` for `secret_create`). The `deploy` and `delete` operations automatically set `--autoConfirmOperation` to avoid interactive prompts. Secret values are masked in responses.

### stacktape_diagnose

Inspects deployed infrastructure with read-only operations: view logs, metrics, and alarm states; query databases (SQL, DynamoDB, Redis, OpenSearch); execute commands in running containers; list stacks and deployment history; and call AWS SDK commands directly.

| Parameter | Type | Description |
|-----------|------|-------------|
| `operation` | string (required) | One of: `info_stack`, `info_stacks`, `info_operations`, `info_whoami`, `logs`, `metrics`, `alarms`, `container_exec`, `sql`, `dynamodb`, `redis`, `opensearch`, `aws_sdk` |
| `args` | object | Operation arguments. Most operations need `projectName`, `stage`, `region`, and `resourceName` |

The `logs` operation defaults to returning 50 entries when no `limit` is specified. Database query operations (`sql`, `dynamodb`, `redis`, `opensearch`) require both `resourceName` and the query/operation parameters. The `aws_sdk` operation requires `service` and `command` to call any AWS API directly.

### stacktape_dev

Controls [Stacktape dev mode](/local-development/dev-mode-overview): start a local development session, check its status, read structured logs, trigger workload rebuilds, and stop the session. The dev mode agent listens on a local HTTP port (default 7331) for control commands.

| Parameter | Type | Description |
|-----------|------|-------------|
| `operation` | string (required) | One of: `start`, `status`, `logs`, `rebuild`, `rebuild_all`, `stop` |
| `args` | object | Operation arguments. `start` accepts `projectName`, `stage`, `region`, `configPath`, `agentPort`. `rebuild` requires `workload`. `logs` accepts `cursor`, `limit`, `workload` |

The `start` operation launches dev mode as a subprocess with a 12-minute timeout for initial setup. Once started, subsequent operations (`status`, `logs`, `rebuild`, `stop`) communicate with the running dev session over its local HTTP API. Only one dev session can be active at a time within the MCP server process.

## How it works

The MCP server starts when your AI client launches the `stacktape mcp` command. It uses stdio transport — the client sends JSON-RPC requests over stdin and reads responses from stdout.

  { title: "Index built", description: "The server builds a full-text search index from Stacktape's documentation files" },
  { title: "Tools registered", description: "Four tools (docs, ops, diagnose, dev) are registered with the MCP protocol" },

For infrastructure operations (`stacktape_ops` and `stacktape_diagnose`), the MCP server spawns a Stacktape CLI subprocess for each command. It streams the CLI's JSONL output, parses structured events, and returns a consolidated response. Responses longer than 30,000 characters are truncated to keep context windows manageable.

For docs queries (`stacktape_docs`), the server uses a BM25-based search index with boosted scoring for titles and tags. It returns the most relevant documentation sections without needing network access — the index is built from local files bundled with the Stacktape CLI.

## Verifying the setup

After running `stacktape mcp:add` or configuring manually, verify the server starts correctly:

```bash
stacktape mcp
```

The server should start without errors and wait for input on stdin. Press `Ctrl+C` to stop it. If the command is not found, ensure Stacktape is installed and available on your PATH.

In your AI client, try a docs query like "how do I deploy a Lambda function" — the assistant should use the `stacktape_docs` tool and return configuration guidance.

## Troubleshooting

### Server fails to start

Ensure the `stacktape` command is available in your shell's PATH. If you installed Stacktape locally via npm, the global `stacktape` command may not exist. Either install globally (`npm install -g stacktape`) or use the local binary path in your MCP config (`node_modules/.bin/stacktape`).

### Operations require authentication

The `stacktape_ops` and `stacktape_diagnose` tools run real CLI commands against your AWS account. You must be logged in (`stacktape login`) and have an AWS profile configured ([`stacktape aws-profile:create`](/cli/aws-profile-create)) before these tools work.

### Config file conflicts

If `mcp:add` reports a JSON parse error, your existing config file may have syntax issues. The command parses JSON and JSONC (JSON with comments). Fix any syntax errors in the config file before retrying. Backups are saved with a `.bak.<timestamp>` suffix — you can restore from these if needed.

### Environment variable override

You can override the CLI command the MCP server uses internally by setting the `STACKTAPE_MCP_CLI_COMMAND` environment variable. This is useful if your Stacktape installation is in a non-standard location.

## FAQ

### What is the Model Context Protocol (MCP)?

MCP is an open standard that lets AI assistants call external tools through a structured JSON-RPC interface. Instead of parsing CLI output as text, the assistant receives typed JSON responses it can reason about programmatically. The Stacktape MCP server implements this protocol over stdio transport.

### Which AI assistants support MCP?

Claude Code, Cursor, VS Code with GitHub Copilot, Windsurf, OpenAI Codex, and OpenCode all support MCP servers. The `stacktape mcp:add` command can configure all six automatically. Any MCP-compatible client that supports stdio transport can use the Stacktape server.

### Does the MCP server store or transmit my credentials?

The MCP server runs locally as a subprocess of your AI client. It uses your existing Stacktape login session and AWS credentials — the same ones the Stacktape CLI uses. No credentials are sent to third parties beyond what normal CLI operations require (AWS API calls, Stacktape API for login). Secret values fetched via `secret_get` are masked in MCP responses.

### Can I use the MCP server without an AWS account?

The `stacktape_docs` tool works without an AWS account — it searches local documentation files bundled with the CLI. The `stacktape_ops` and `stacktape_diagnose` tools require a connected AWS account and active Stacktape login, since they execute real infrastructure operations.

### How does the docs search work?

The MCP server builds a BM25-based full-text search index from documentation files bundled with the Stacktape CLI. Titles and tags are weighted higher than body content. Results below 25% of the top score are filtered out. The index covers CLI references, resource configuration, concepts, recipes, and getting-started guides.

### Is the MCP server safe to use in production workflows?

The MCP server is currently in preview. While the `stacktape_docs` and `stacktape_diagnose` tools are read-only, the `stacktape_ops` tool can deploy and delete real infrastructure. Destructive operations like `delete` require explicit confirmation (`confirm: true`). Treat MCP-initiated deployments with the same care as manual CLI deployments.

### Can I run the MCP server alongside normal CLI usage?

Yes. The MCP server spawns separate CLI subprocesses for each operation, so it does not interfere with other Stacktape commands you run in a terminal. However, running concurrent deployments to the same stack from both the MCP server and a terminal can cause CloudFormation conflicts.

### How do I update the MCP server?

The MCP server ships as part of the Stacktape CLI. Updating Stacktape (`stacktape upgrade` or updating the npm package) updates the MCP server. After upgrading, you do not need to re-run `mcp:add` — the existing config points to the `stacktape mcp` command, which picks up the new version automatically.

### What happens if a tool call times out?

Infrastructure operations have a default timeout of 10 minutes. Dev mode startup has a 12-minute timeout. If a command exceeds its timeout, the MCP server terminates the subprocess and returns an error response. Long-running deployments may need to be run directly via the CLI instead.

### Can I scope the MCP server to specific operations?

Not currently. The MCP server exposes all four tools on every startup. Access control is handled at the AI client level — most clients let you approve or deny individual tool calls before they execute. This gives you per-call control over what the MCP server does.
