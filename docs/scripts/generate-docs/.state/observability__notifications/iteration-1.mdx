# Notifications

Stacktape notification rules alert your team when deployment lifecycle events occur — deploys, deletes, rollbacks, GitOps pushes, script runs, and secret changes. Each rule maps one or more event types to an [alert channel](/observability/alert-channels) and can be scoped to specific projects and stages, so the right people get notified about the right events.

<Info>
Notifications cover **operational events** (deployments, deletions, rollbacks). For **metric-based alerts** on running resources (CPU, error rates, latency), use [alarms](/observability/alarms). Both features deliver through the same [alert channels](/observability/alert-channels).
</Info>

## When to use notifications

Notification rules are most valuable when your team needs visibility into deployment activity without watching the CLI or Console. Common scenarios:

- **Production deploy alerts** — notify a Slack channel whenever a production deploy succeeds or fails, so the on-call engineer sees it immediately.
- **GitOps visibility** — alert the team when a git-push deploy or PR preview deploy fails, so broken builds don't go unnoticed.
- **Secret change auditing** — send an email or webhook when secrets are created, updated, or deleted, giving you an audit trail for sensitive configuration changes.
- **Multi-team coordination** — scope notifications per project so each team receives only their own deployment events.

If you only deploy occasionally and watch the CLI output yourself, you may not need notification rules. They become important once deployments happen automatically (via [GitOps](/ci-cd-and-gitops/gitops-with-console)) or across multiple projects and stages.

## Supported event types

Notification rules can watch for 17 event types, organized into six categories:

| Category | Event types |
|---|---|
| **Deployment** | Deploy started, Deploy succeeded, Deploy failed, Deploy rolled back |
| **Deletion** | Delete started, Delete succeeded, Delete failed |
| **Rollback** | Rollback succeeded, Rollback failed |
| **Git deployments** | Preview deploy succeeded, Preview deploy failed, Git push deploy succeeded, Git push deploy failed |
| **Scripts** | Script run succeeded, Script run failed |
| **Secrets** | Secret created, Secret updated, Secret deleted |

Each notification rule can watch for multiple event types at once. For example, a single rule can alert on both deploy failures and rollback failures, so you get one channel for all failure conditions.

### Deployment events

Deployment events fire for CLI-initiated and GitOps-triggered deploys. `Deploy started` fires at the beginning of a deployment, `Deploy succeeded` fires when the deployment completes, `Deploy failed` fires when it errors out, and `Deploy rolled back` fires when a deployment is automatically rolled back (for example, during a [gradual deployment](/deployment-and-lifecycle/gradual-deployments) that detects errors).

### Git deployment events

Git deployment events are specific to [GitOps](/ci-cd-and-gitops/gitops-with-console) workflows. `Git push deploy succeeded` and `Git push deploy failed` fire for branch-based auto-deploys. `Preview deploy succeeded` and `Preview deploy failed` fire for PR preview deployments. These are separate from standard deployment events, so you can set up distinct notification rules for automated vs. manual deploys.

### Secret events

Secret events fire when [secrets](/configuration/secrets) are created, updated, or deleted through the Console or CLI. These are useful for audit trails — knowing when and by whom a secret was changed can be critical for incident investigation.

## Creating a notification rule

Notification rules are created in the Stacktape Console under **Notifications > Rules**. Each rule has four fields:

1. **Alert channel** — the destination for notifications. You must [create a channel](/observability/alert-channels) first (Slack, Microsoft Teams, Discord, Email, or Webhook).
2. **Event types** — one or more event types from the table above. Select as many as needed; they are grouped by category in the dropdown.
3. **Active for projects** — which projects this rule applies to. Defaults to all projects.
4. **Active for stages** — which stages this rule applies to. Defaults to all stages.

<Tip>
Create separate rules for different severity levels. For example, route failure events to a high-urgency Slack channel and success events to a low-noise logging webhook.
</Tip>

## Scoping notifications

By default, a notification rule applies to all projects and all stages in your organization. You can narrow the scope in two ways:

- **Project scoping** — restrict the rule to specific projects. Only events from those projects trigger a notification. This is useful in multi-team organizations where each team owns different projects.
- **Stage scoping** — restrict the rule to specific stages (e.g., only `production`). This prevents noise from dev and staging deploys while keeping production alerts active.

Project and stage scoping can be combined. A rule scoped to project `api` and stage `production` only fires when that exact project-stage combination emits a matching event. If you leave both set to "All", the rule fires for every matching event across the entire organization.

## Notification delivery

When an event matches a notification rule, the alert is delivered to the configured [alert channel](/observability/alert-channels). Each delivery is tracked with a status:

| Status | Meaning |
|---|---|
| **Sent** | The notification was delivered to the channel |
| **Partial** | Some channels received the notification, but at least one delivery failed |
| **Failed** | Delivery to the channel failed (e.g., invalid webhook URL, expired token) |
| **Pending** | The notification is queued for delivery |
| **No match** | The event did not match any notification rule |

### Checking delivery history

The **Notifications > History** page in the Console shows a log of all notification events, including the event type, project, stage, title, and delivery status. Use this to verify that notifications are reaching their destinations and to troubleshoot delivery failures.

<ConsoleScreenshot src="/static/screenshots/console-notification-history.png" alt="Stacktape Console notification history page showing a table of notification events with time, event type, project, stage, title, and delivery status columns" caption="Notification history shows delivery status for every event" />

If a notification shows a `Failed` status, check the alert channel configuration — the most common causes are expired Slack tokens, changed webhook URLs, or email delivery issues. You can view per-channel delivery details by expanding the delivery status on the history page.

## Notifications vs. alarms vs. budgets

Stacktape has three alert systems that serve different purposes. All three deliver through the same [alert channels](/observability/alert-channels), and all three appear in the unified [alert history](/observability/alert-history).

| Feature | Watches for | Defined in |
|---|---|---|
| **Notifications** | Deployment lifecycle events (deploy, delete, rollback, scripts, secrets) | Console — Notifications > Rules |
| **[Alarms](/observability/alarms)** | CloudWatch metric thresholds (error rates, CPU, latency, queue depth) | Console — Alarms > Rules (applied on next deploy) |
| **[Budgets](/managing-costs/budgets)** | AWS spending thresholds | Console — Budgets > Rules |

Use notifications for operational awareness ("did the deploy finish?"), alarms for runtime health ("is the API throwing errors?"), and budgets for cost control ("are we overspending?"). Most production setups use all three.

## FAQ

### Can I create notification rules from the CLI?

Notification rules are managed exclusively in the Stacktape Console. There is no CLI command or config-file property for notification rules. The CLI fires the events that notifications watch for, but the rules themselves are created and managed in the Console.

### How many notification rules can I create?

There is no documented limit on the number of notification rules per organization. You can create multiple rules watching for different event types, scoped to different projects, stages, and channels.

### Can one event trigger multiple notification rules?

Yes. If an event matches multiple rules (for example, one rule watching all deploy events for all projects, and another watching deploy failures for `production` only), the event triggers all matching rules. Each rule delivers independently to its configured channel.

### What happens if my alert channel is misconfigured?

The notification is attempted and recorded with a `Failed` delivery status on the Notifications > History page. The event is not lost — you can see it in the history and fix the channel configuration. Stacktape does not retry failed deliveries automatically.

### Do notification rules require a new deployment to take effect?

No. Unlike [alarm rules](/observability/alarms), which are applied to AWS infrastructure on the next deployment, notification rules take effect immediately when created. They watch for events at the organization level, not at the AWS resource level.

### What is the difference between "Deploy failed" and "Deploy rolled back"?

`Deploy failed` fires when a deployment errors out — for example, a CloudFormation stack update fails. `Deploy rolled back` fires when a deployment is automatically reversed, which can happen during [gradual deployments](/deployment-and-lifecycle/gradual-deployments) that detect errors during the canary or linear rollout phase.

### Can I get notifications for PR preview deployments?

Yes. The `Preview deploy succeeded` and `Preview deploy failed` event types are specifically for [GitOps](/ci-cd-and-gitops/gitops-with-console) PR preview workflows. Create a notification rule with these event types to get alerts when preview environments are created or fail to deploy.

### Where can I see all alerts across notifications, alarms, and budgets?

The [alert history](/observability/alert-history) page in the Console shows a unified log of all alerts across notifications, alarms, and budgets. This gives you a single view of every alert your organization has received, regardless of which system generated it.
