# Prebuilt Image

Prebuilt image packaging references an existing container image from any registry — Docker Hub, Amazon ECR, GitHub Container Registry, or a private registry. Stacktape skips the build step entirely and deploys the image directly. Use this when your image is already built by an external CI/CD pipeline, or when you want to run a third-party image like Nginx, Redis, or Prometheus.

This packaging mode applies to [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload), and [batch jobs](/resources/compute/batch-job).

## When to use

Use prebuilt image packaging when:

- **Your CI/CD pipeline already builds images.** If GitHub Actions, GitLab CI, or another pipeline pushes tagged images to a registry, reference them directly instead of rebuilding inside Stacktape.
- **You're running third-party software.** Images like `nginx`, `redis`, `postgres`, or `grafana/grafana` are ready to use from Docker Hub.
- **You share images across multiple services.** A single image built once can be referenced by several Stacktape resources without redundant builds.
- **You need full control over the build environment.** Build externally with any tooling (Bazel, Nix, Kaniko) and hand the final image to Stacktape.

Use prebuilt image if your image already exists. Use [custom Dockerfile](/packaging/containers/custom-dockerfile) if you want Stacktape to build from a Dockerfile. Use [Stacktape buildpack](/packaging/containers/stacktape-buildpack) if you want zero-config builds from source code.

## When NOT to use

Skip prebuilt image packaging when:

- **You want Stacktape to handle the entire build.** The [Stacktape image buildpack](/packaging/containers/stacktape-buildpack) or [custom Dockerfile](/packaging/containers/custom-dockerfile) mode builds, tags, and pushes to ECR automatically.
- **You don't have an external image pipeline yet.** Setting up a separate registry and build system adds operational overhead. Let Stacktape build and push to its managed ECR repository instead.
- **You need Stacktape to detect code changes.** Prebuilt image mode deploys whatever tag you specify — Stacktape does not know whether the image content changed. If you want automatic rebuilds on source changes, use a build-based packaging mode.

## Basic example

The smallest valid configuration references a public image by name and tag:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, PrebuiltImagePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const webService = new WebService({
    packaging: new PrebuiltImagePackaging({
      image: 'httpd:latest'
    }),
    resources: {
      cpu: 0.25,
      memory: 512
    }
  });

  return {
    resources: { webService }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The `image` property accepts any valid container image reference: a Docker Hub short name (`httpd:latest`), a full registry URL (`123456789012.dkr.ecr.us-east-1.amazonaws.com/my-repo:v1.2.3`), or a GitHub Container Registry path (`ghcr.io/my-org/my-app:main`).

## Private registry images

To pull from a private registry, store your credentials in AWS Secrets Manager and reference the secret ARN via `repositoryCredentialsSecretArn`. The secret must be a JSON object with `username` and `password` keys.

Create the secret using the Stacktape CLI:

```bash
stacktape secret:create --name my-registry-creds --value '{"username":"my-user","password":"my-token"}'
```

Then reference the secret ARN in your config:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, PrebuiltImagePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new WebService({
    packaging: new PrebuiltImagePackaging({
      image: 'registry.example.com/my-org/my-app:v2.1.0',
      repositoryCredentialsSecretArn:
        'arn:aws:secretsmanager:us-east-1:123456789012:secret:my-registry-creds-AbCdEf'
    }),
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { api }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

Stacktape automatically grants the ECS task execution role permission to read the referenced secret from Secrets Manager. At container start, AWS ECS fetches the credentials and authenticates with the registry before pulling the image.

<Info>
Amazon ECR images in the same AWS account do not need `repositoryCredentialsSecretArn` — ECS authenticates to ECR natively using the task execution role.
</Info>

## Overriding startup behavior

You can override the image's default startup instructions without rebuilding it.

### Command override

The `command` property overrides the image's `CMD` instruction. Use this to pass different arguments or run an alternative script:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WorkerService, PrebuiltImagePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const worker = new WorkerService({
    packaging: new PrebuiltImagePackaging({
      image: 'my-org/data-pipeline:latest',
      command: ['node', 'dist/worker.js', '--mode', 'batch']
    }),
    resources: {
      cpu: 1,
      memory: 2048
    }
  });

  return {
    resources: { worker }
  };
});
// [!code focus-end]`
    }
  ]}
/>

### Entrypoint override

The `entryPoint` property overrides the image's `ENTRYPOINT` instruction. This is useful when you need to wrap the default process with an init script or switch the executable entirely:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, PrivateService, PrebuiltImagePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const internal = new PrivateService({
    packaging: new PrebuiltImagePackaging({
      image: 'my-org/api-server:v3.0.0',
      entryPoint: ['/app/entrypoint.sh'],
      command: ['--config', '/etc/app/config.yaml']
    }),
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { internal }
  };
});
// [!code focus-end]`
    }
  ]}
/>

<Info>
The `entryPoint` property is available on web services, private services, worker services, and multi-container workloads. Batch jobs support only `image` and `command`.
</Info>

## Multi-container workloads

When using prebuilt images in a [multi-container workload](/resources/compute/multi-container-workload), each container specifies its own packaging independently. This lets you combine prebuilt third-party images (like a sidecar proxy) with your application containers:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  MultiContainerWorkload,
  PrebuiltImagePackaging,
  StacktapeImageBuildpackPackaging
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const app = new MultiContainerWorkload({
    containers: [
      {
        name: 'api',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/server.ts'
        })
      },
      {
        name: 'envoy-proxy',
        packaging: new PrebuiltImagePackaging({
          image: 'envoyproxy/envoy:v1.28-latest',
          command: ['envoy', '-c', '/etc/envoy/envoy.yaml']
        })
      }
    ],
    resources: {
      cpu: 1,
      memory: 2048
    }
  });

  return {
    resources: { app }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

## Image tagging strategy

Since Stacktape does not rebuild prebuilt images, the tag you specify determines which version gets deployed. Choose a tagging approach that fits your workflow:

| Strategy | Example | Tradeoff |
|----------|---------|----------|
| Immutable tags (recommended) | `my-app:v1.2.3` or `my-app:abc1234` | Every deploy is deterministic. You always know exactly which code is running. |
| `latest` tag | `my-app:latest` | Convenient for dev, but non-deterministic — the same tag can point to different content across deploys. |
| Branch-based tags | `my-app:main` | Good for staging environments that track a branch. Still mutable. |

<Tip>
Use immutable tags (semantic versions or commit SHAs) for production. Mutable tags like `latest` can cause surprising behavior when ECS scales out and pulls a newer image than what's running on existing tasks.
</Tip>

## FAQ

### Which resources support prebuilt image packaging?

Prebuilt image packaging is supported by all container-based compute resources in Stacktape: [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload), and [batch jobs](/resources/compute/batch-job). It is not available for [Lambda functions](/resources/compute/lambda-function), which use function packaging modes instead.

### How does ECS pull images from a public registry?

AWS ECS Fargate pulls public images (Docker Hub, GitHub Container Registry, Quay) using anonymous access. No credentials are needed for public repositories. For Docker Hub specifically, anonymous pulls are subject to rate limits (100 pulls per 6 hours per IP). If you hit rate limits, consider mirroring images to Amazon ECR or providing Docker Hub credentials.

### Can I use Amazon ECR images without specifying credentials?

Yes. When you reference an ECR image in the same AWS account and region, ECS authenticates automatically using the task execution role. No `repositoryCredentialsSecretArn` is needed. For cross-account ECR access, configure the ECR repository policy to allow the task execution role.

### When should I use prebuilt image vs custom Dockerfile?

Use prebuilt image when the image already exists — built by your CI pipeline, pulled from a public registry, or shared across services. Use [custom Dockerfile](/packaging/containers/custom-dockerfile) when you want Stacktape to build the image from source during deployment. Prebuilt image is faster to deploy (no build step) but requires external infrastructure to produce and push images.

### How do I update the deployed image version?

Change the image tag in your Stacktape config and redeploy. Stacktape passes the new image reference to ECS, which performs a rolling update — starting new tasks with the updated image before draining old tasks. If you use a mutable tag like `latest`, you need to redeploy even if the tag hasn't changed in your config, because ECS caches the image digest from the previous deployment.

### Is there an image size limit?

AWS ECS Fargate supports container images up to 20 GB (uncompressed). Larger images increase task startup time because Fargate must pull and decompress the image before starting your container. For fast startups, keep images lean — prefer multi-stage Docker builds and alpine-based images when possible.

### Should I build images in CI or let Stacktape build them?

Let Stacktape build images (via [buildpack](/packaging/containers/stacktape-buildpack) or [custom Dockerfile](/packaging/containers/custom-dockerfile)) when you want the simplest setup — no registry management, no CI build steps for containers. Build in CI and use prebuilt image when you need custom tooling, want to share images across environments, run security scanning before deploy, or already have a container pipeline. Teams with mature CI systems often prefer prebuilt images for production and Stacktape buildpack for dev stages.

### Can I use ARM-based images?

Yes. If your container resource is configured to run on Graviton (ARM) instances, provide an ARM-compatible image. Multi-architecture images (built with `docker buildx`) work on both ARM and x86 without changes to the image reference.

### How do I debug startup failures with a prebuilt image?

Check container logs using [`stacktape debug:logs`](/cli/debug-logs) with the resource name. Common causes of startup failures: the image doesn't expose the expected port, the `command` override has a typo, or the image expects environment variables that aren't set. If the container exits immediately, verify your `entryPoint` and `command` produce a long-running process.
