# SQS Queue

Stacktape's `sqs-queue` resource provisions a fully managed Amazon SQS queue that lets you decouple services by passing messages asynchronously. A producer sends a message; a consumer retrieves and processes it independently. Queues are serverless — there are no servers to provision, and you pay only per message sent or received.

> **Pricing:** Standard queues cost approximately $0.40 per million requests. FIFO queues cost approximately $0.50 per million. The first 1 million requests per month are free. Costs scale with message volume, not queue count.

## When to use an SQS queue

SQS is the right tool when you want to:

- **Offload background work** — image resizing, email sending, report generation — away from your request path
- **Buffer traffic spikes** — protect a downstream service or database from bursts by queuing work ahead of it
- **Build retry resilience** — messages stay in the queue until a consumer successfully processes and deletes them; if a consumer crashes, the message becomes visible again automatically
- **Decouple services** — allow producers and consumers to evolve and scale independently without direct dependencies

Messages are stored redundantly across multiple availability zones, so your data is durable even if individual infrastructure fails.

### When not to use SQS

If multiple independent services need to react to the same event, use an SNS topic or EventBridge event bus instead — a single Stacktape Lambda SQS trigger should have only one consumer function per queue (for fan-out, use SNS or EventBridge). SQS is also not suited for real-time bidirectional communication or high-throughput ordered streaming (use Kinesis for that).

## Basic example

This creates a standard SQS queue and a Lambda function that processes messages from it in batches of up to 10.

```yaml
resources:
  jobQueue:
    type: sqs-queue

  jobProcessor:
    type: function
    properties:
      packaging:
        type: stacktape-lambda-buildpack
        properties:
          entryfilePath: src/process-job.ts
      events:
        - type: sqs
          properties:
            sqsQueueName: jobQueue
            batchSize: 10
```

```typescript
import type { StacktapeConfig } from 'stacktape';

const config: StacktapeConfig = {
  resources: {
    jobQueue: {
      type: 'sqs-queue',
    },
    jobProcessor: {
      type: 'function',
      properties: {
        packaging: {
          type: 'stacktape-lambda-buildpack',
          properties: { entryfilePath: 'src/process-job.ts' },
        },
        events: [
          {
            type: 'sqs',
            properties: {
              sqsQueueName: 'jobQueue',
              batchSize: 10,
            },
          },
        ],
      },
    },
  },
};
```

Stacktape creates an event source mapping that polls the queue and invokes the function with batches of messages. AWS automatically scales Lambda invocations as queue depth increases.

Your handler receives an `event` with a `Records` array — each record's `body` field holds the raw message string:

```typescript
import type { SQSHandler } from 'aws-lambda';

export const handler: SQSHandler = async (event) => {
  for (const record of event.Records) {
    const job = JSON.parse(record.body);
    // process job...
    console.info('Processing job:', job.jobId);
  }
};
```

## Configuring an SQS queue in Stacktape

### Visibility timeout

When a consumer retrieves a message, SQS hides it from other consumers for `visibilityTimeoutSeconds` (default: **30 seconds**). The consumer must fully process and delete the message before this window closes — otherwise the message becomes visible again and another consumer will pick it up.

**Set this higher than your expected processing time.** If your function takes up to 2 minutes, use at least 150 seconds.

```yaml
resources:
  slowQueue:
    type: sqs-queue
    properties:
      visibilityTimeoutSeconds: 300  # 5 minutes
```

### FIFO queues

Standard queues optimize for throughput. Enable `fifoEnabled` when processing order is critical — financial transactions, state transitions, sequential workflows.

FIFO queues guarantee:
- **Exactly-once delivery** — messages are not duplicated
- **Strict ordering** within each `MessageGroupId`

A `MessageGroupId` is a string you attach to each message when calling `SendMessage` (via the AWS SDK). Messages with the same `MessageGroupId` are processed in strict order. If you enable `contentBasedDeduplication`, identical messages are deduplicated within a 5-minute window based on a SHA-256 hash of the message body.

FIFO queues have lower throughput than standard queues — approximately 300 messages/second without batching, approximately 3,000 with batching.

```yaml
resources:
  orderedQueue:
    type: sqs-queue
    properties:
      fifoEnabled: true
      contentBasedDeduplication: true
```

For higher throughput on FIFO queues, enable `fifoHighThroughput`. This partitions messages by `MessageGroupId` and supports up to approximately 70,000 messages/second, but ordering is only guaranteed within a group — not across groups.

### Dead-letter queues

A dead-letter queue (DLQ) receives messages that failed processing too many times. After `maxReceiveCount` failed attempts, SQS automatically moves the message to the DLQ so you can inspect and reprocess it without blocking the main queue.

```yaml
resources:
  jobDlq:
    type: sqs-queue

  jobQueue:
    type: sqs-queue
    properties:
      visibilityTimeoutSeconds: 120
      redrivePolicy:
        targetSqsQueueName: jobDlq  # queue in the same stack
        maxReceiveCount: 3
```

To use a DLQ in another stack or AWS account, use `targetSqsQueueArn` instead of `targetSqsQueueName`:

```yaml
redrivePolicy:
  targetSqsQueueArn: arn:aws:sqs:eu-west-1:123456789012:shared-dlq
  maxReceiveCount: 3
```

<Info>Set `maxReceiveCount` to 3–5 for most workloads. Use a lower value for tasks that fail fast and should not be retried aggressively.</Info>

### Long polling

By default, SQS uses short polling (`longPollingSeconds: 0`) — consumers get an immediate response even if the queue is empty, and must poll again immediately. This generates unnecessary API calls and cost.

Set `longPollingSeconds` to `1`–`20` to enable long polling. The queue waits up to that many seconds for a message before returning. **20 seconds is recommended** — it's the most cost-effective and dramatically reduces empty responses.

```yaml
resources:
  jobQueue:
    type: sqs-queue
    properties:
      longPollingSeconds: 20
```

<Info>`longPollingSeconds` matters when anything other than a Lambda trigger is consuming the queue — container worker services, scripts, or application code polling directly via the AWS SDK. For Lambda-triggered queues, AWS manages polling as part of the event source mapping.</Info>

### Message delay

`delayMessagesSecond` (0–900) makes newly added messages invisible to consumers for a fixed period. Useful when a downstream service needs preparation time before processing (for example, waiting for related records to appear in a database).

```yaml
resources:
  delayedQueue:
    type: sqs-queue
    properties:
      delayMessagesSecond: 60  # messages held for 1 minute before delivery
```

### Message retention

Messages that are not processed are retained for `messageRetentionPeriodSeconds` (default: **4 days**, 345,600 seconds). Range: 60 seconds to 14 days (1,209,600 seconds). Increase this if consumers might fall behind or be temporarily offline.

### Large payloads

The maximum message size is 256 KB (`maxMessageSizeBytes: 262144`, the default). For larger payloads, store the data in S3 and send a reference (URL or key) as the message body.

### EventBridge integration

You can route events from an EventBridge event bus directly into an SQS queue. EventBridge is a serverless event router — it delivers events matching a pattern to targets including SQS queues, letting you build event-driven pipelines without writing fan-out code.

```yaml
resources:
  orderEvents:
    type: event-bus

  orderQueue:
    type: sqs-queue
    properties:
      events:
        - type: event-bus
          properties:
            eventBusName: orderEvents
            eventPattern:
              detail-type:
                - OrderPlaced
```

When the target queue has `fifoEnabled: true`, `messageGroupId` is required:

```yaml
events:
  - type: event-bus
    properties:
      eventBusName: orderEvents
      messageGroupId: orders  # required for FIFO queues
      eventPattern:
        detail-type:
          - OrderPlaced
```

### Alarms

Monitor queue health with built-in alarm types:

- **`sqs-queue-not-empty`** — fires when any of these queue metrics are non-zero: visible messages, in-flight messages, messages received, or messages sent
- **`sqs-queue-received-messages-count`** — fires when the number of received messages crosses a threshold

```yaml
resources:
  jobQueue:
    type: sqs-queue
    properties:
      alarms:
        - trigger:
            type: sqs-queue-not-empty
          notificationTargets:
            - type: slack
              properties:
                webhookUrl: $Secret('slack-webhook')
```

### Cross-account and SNS access

<Info>Most users don't need `policyStatements`. Use it only when granting access to external AWS accounts or AWS services such as SNS — `connectTo` handles permissions for resources within your own stack.</Info>

Use `policyStatements` to grant access to external AWS accounts or services like SNS that need to publish to this queue.

```yaml
resources:
  inboundQueue:
    type: sqs-queue
    properties:
      policyStatements:
        - Effect: Allow
          Action:
            - sqs:SendMessage
          Principal:
            Service: sns.amazonaws.com
          Condition:
            ArnEquals:
              aws:SourceArn: arn:aws:sns:eu-west-1:123456789012:my-topic
```

## Lambda trigger configuration

When you add an `sqs` event to a Lambda function, Stacktape creates an AWS event source mapping. Two properties directly affect cost and latency:

| Property | Default | Description |
|---|---|---|
| `batchSize` | 10 | Max messages per invocation (up to 10,000) |
| `maxBatchWindowSeconds` | not set | How long Lambda waits to fill a full batch before invoking |

**`maxBatchWindowSeconds`** (0–300 seconds) buffers incoming messages to assemble a larger batch before invoking. When not set, Lambda is invoked as soon as any messages are available — lower latency but higher invocation count and cost.

Use `maxBatchWindowSeconds: 5`–`30` on low-volume queues to reduce invocations. Use `0` (or leave unset) for latency-sensitive processing.

```yaml
events:
  - type: sqs
    properties:
      sqsQueueName: jobQueue
      batchSize: 10
      maxBatchWindowSeconds: 15  # wait up to 15s to fill a batch
```

<Warning>The combined payload of a batch cannot exceed 6 MB. If your messages are large, reduce `batchSize` accordingly to avoid hitting this limit silently.</Warning>

## How Stacktape manages SQS queue lifecycle

Each `sqs-queue` resource creates an AWS SQS queue. Stacktape manages the queue lifecycle through CloudFormation — creation, updates, and deletion are handled automatically during `deploy` and `delete` commands.

When you use `connectTo` on a workload to reference an SQS queue, Stacktape grants the workload's IAM role send, receive, and delete permissions on the queue, and injects environment variables with the queue's connection details.

When you add an `sqs` event trigger to a Lambda function, Stacktape creates an event source mapping that polls the queue and invokes the function with batches of messages. If the function fails, messages become visible again after the visibility timeout and will be retried — up to `maxReceiveCount` times if a DLQ is configured.

## Connecting to other resources

Add the queue to `connectTo` on any workload that needs to send or receive messages. Stacktape grants permissions and injects connection details automatically.

```yaml
resources:
  myApi:
    type: web-service
    properties:
      connectTo:
        - jobQueue
      # Injects:
      #   STP_JOB_QUEUE_ARN
      #   STP_JOB_QUEUE_NAME
      #   STP_JOB_QUEUE_URL
```

### Injected environment variables

| Variable | Example (resource named `jobQueue`) | Description |
|---|---|---|
| `STP_[NAME]_ARN` | `STP_JOB_QUEUE_ARN` | Queue ARN |
| `STP_[NAME]_NAME` | `STP_JOB_QUEUE_NAME` | Queue name |
| `STP_[NAME]_URL` | `STP_JOB_QUEUE_URL` | Queue URL — pass this to the AWS SDK to send/receive |

### Sending a message from application code

Use `STP_JOB_QUEUE_URL` with the AWS SDK to enqueue a job:

```typescript
import { SQSClient, SendMessageCommand } from '@aws-sdk/client-sqs';

const client = new SQSClient({ region: 'eu-west-1' });

await client.send(
  new SendMessageCommand({
    QueueUrl: process.env.STP_JOB_QUEUE_URL,
    MessageBody: JSON.stringify({ jobId: '123', type: 'resize-image' }),
  })
);
```
