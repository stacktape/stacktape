# Overrides and Escape Hatches

Stacktape generates AWS CloudFormation templates from your configuration. When you need to customize the underlying AWS resources beyond what Stacktape exposes — set a parameter Stacktape doesn't surface, tune a database engine setting, or add an AWS resource type Stacktape doesn't natively support — overrides and escape hatches give you direct access to the CloudFormation layer without leaving your Stacktape project.

## Why this exists

Stacktape covers the high-signal 80% of AWS resource configuration through its typed properties. But AWS services expose hundreds of settings, and Stacktape intentionally doesn't surface every one. Overrides and escape hatches close the gap: you keep Stacktape's deployment, observability, and lifecycle management while reaching through to any CloudFormation property or resource type.

## When to use escape hatches

Use overrides and escape hatches when:

- You need a CloudFormation property Stacktape doesn't expose (e.g., reserved concurrency on a Lambda function, enhanced monitoring on RDS)
- You need to add an AWS resource Stacktape doesn't have a built-in type for (e.g., a custom CloudWatch dashboard, an SES configuration set)
- You need to compute or dynamically modify CloudFormation properties based on existing values (transforms)
- You need full CDK-level control for complex infrastructure (CDK constructs)

## When NOT to use escape hatches

Check Stacktape's built-in properties first. Overrides bypass Stacktape's validation and auto-configuration, so they're harder to maintain and can conflict with future Stacktape updates. If a property is available directly on the resource (like `memory` on a Lambda function or `scaling` on a web service), use the native property instead.

## Overrides

Stacktape resource overrides let you set specific CloudFormation properties on the child resources that Stacktape generates. Every Stacktape resource creates one or more CloudFormation resources internally — for example, a [Lambda function](/resources/compute/lambda-function) creates an `AWS::Lambda::Function`, an `AWS::IAM::Role`, and an `AWS::Logs::LogGroup`. Overrides target these child resources by their logical name, using dot-notation paths to set specific properties.

The `overrides` property is available on all Stacktape resource types.

### How overrides work

Overrides use dot-notation paths to set values on child CloudFormation resource properties. The type definition is:

```typescript
type ResourceOverrides = {
  [cfLogicalName: string]: { [resourcePath: string]: any };
};
```

Each key is a CloudFormation logical name identifying which child resource to modify, and each value is an object mapping dot-notation property paths to their desired values. For top-level object properties, Stacktape performs a deep merge with existing values. For dot-notation paths, it sets the value at the specified path directly.

### Finding child resource logical names

To override a property, you need the logical name of the child CloudFormation resource. There are three ways to find it:

Run [`stacktape compile-template`](/cli/compile-template) to generate the full CloudFormation template and inspect the logical names directly.

```bash
stacktape compile-template --stage dev --region eu-west-1
```

This writes a `compiled-template.yaml` file you can search for the resource you want to modify.

Alternatively, use the Stacktape Console's resource inspector, which displays child resources for each Stacktape resource in your stack.

In TypeScript configs using `defineConfig`, you can also use user-friendly property names (like `lambda`, `lambdaRole`, `lambdaLogGroup`) instead of full logical names. The SDK automatically resolves these to the correct CloudFormation logical names based on the resource type.

### Basic override example

This example sets reserved concurrent executions on a Lambda function's underlying `AWS::Lambda::Function` resource:

### Common child resource names by resource type

When using user-friendly names in TypeScript configs, these are the most commonly overridden child resources:

| Stacktape resource type | User-friendly name | AWS resource type |
|---|---|---|
| Lambda function | `lambda` | `AWS::Lambda::Function` |
| Lambda function | `lambdaRole` | `AWS::IAM::Role` |
| Lambda function | `lambdaLogGroup` | `AWS::Logs::LogGroup` |
| Web service | `ecsTaskDefinition` | `AWS::ECS::TaskDefinition` |
| Web service | `ecsService` | `AWS::ECS::Service` |
| Relational database | `dbInstance` | `AWS::RDS::DBInstance` |
| Relational database | `dbInstanceParameterGroup` | `AWS::RDS::DBParameterGroup` |
| Bucket | `bucket` | `AWS::S3::Bucket` |
| HTTP API Gateway | `httpApi` | `AWS::ApiGatewayV2::Api` |
| Redis cluster | `redisReplicationGroup` | `AWS::ElastiCache::ReplicationGroup` |
| Redis cluster | `redisParameterGroup` | `AWS::ElastiCache::ParameterGroup` |

<Tip>
You can always fall back to using the full CloudFormation logical name directly (e.g., the PascalCase name from `compile-template` output) if the user-friendly name isn't available or if you're targeting a conditional child resource.
</Tip>

## Transforms

Transforms are JavaScript functions that receive the current CloudFormation properties of a child resource and return modified properties. They're more flexible than overrides because you can compute values, conditionally modify properties, or append to arrays.

Transforms are only available in TypeScript configurations using the `defineConfig` pattern. They are not supported in YAML configs.

### How transforms work

Each transform function receives the full `Properties` object of a CloudFormation resource and must return the modified properties object:

```typescript
type CfResourceTransform = (props: Record<string, any>) => Partial<Record<string, any>>;
```

Transforms run after Stacktape has fully resolved and built the CloudFormation template, so you see the final property values.

### Basic transform example

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    transforms: {
      lambda: (props) => ({
        ...props,
        Layers: [
          ...(props.Layers || []),
          'arn:aws:lambda:eu-west-1:123456789012:layer:my-shared-utils:3'
        ]
      })
    }
  });

  return { resources: { api } };
});
// [!code focus-end]
`
    }
  ]}
/>

This appends a Lambda layer ARN to whatever layers Stacktape already configured. With a plain override, you'd have to know the full final list — with a transform, you append to the existing value.

## Overrides vs transforms

| Aspect | Overrides | Transforms |
|---|---|---|
| Syntax | Static key-value pairs | JavaScript function |
| Config format | TypeScript and YAML | TypeScript only (`defineConfig`) |
| Use case | Set known property values | Compute, append, or conditionally modify |
| Merge behavior | Dot-path set or deep merge for objects | Full control — you return the final object |
| Validation | Validated against child resource existence | No validation on returned properties |

Use **overrides** when you know the exact property and value — they work in both TypeScript and YAML configs and are simpler to read. Use **transforms** when you need to append to arrays, conditionally modify properties, or compute values based on existing configuration.

## Raw CloudFormation resources

The `cloudformationResources` top-level config property lets you add any AWS CloudFormation resource directly to your stack. This is the escape hatch for AWS services Stacktape doesn't have a built-in resource type for.

Raw CloudFormation resources are merged into the final template as-is. They don't count toward your Stacktape resource limit, but they also don't benefit from Stacktape's automatic IAM permissions, `connectTo` integration, or simplified configuration.

### Adding a raw CloudFormation resource

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  $CfResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const processor = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/processor.ts'
    }),
    environment: [{ name: 'TOPIC_ARN', value: $CfResourceParam('AlertTopic', 'TopicArn') }]
  });

  return {
    resources: { processor },
    cloudformationResources: {
      AlertTopic: {
        Type: 'AWS::SNS::Topic',
        Properties: {
          TopicName: 'alert-notifications',
          DisplayName: 'Alert Notifications'
        }
      }
    }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

### Referencing CloudFormation resources from Stacktape

Use the [`$CfResourceParam`](/configuration/directives) directive to reference attributes of raw CloudFormation resources. It accepts the logical resource name and the CloudFormation attribute to retrieve:

```typescript
// Get the ARN of an SNS topic
$CfResourceParam('AlertTopic', 'TopicArn')

// Get a resource's physical ID (Ref)
$CfResourceParam('AlertTopic', 'Ref')
```

Under the hood, `$CfResourceParam` resolves to CloudFormation intrinsic functions (`Fn::GetAtt` or `Ref`) and automatically creates a stack output for the referenced value.

### Referencing Stacktape resources from CloudFormation

Use the [`$ResourceParam`](/configuration/referenceable-parameters) directive inside `cloudformationResources` to reference attributes of Stacktape-managed resources:

```typescript
cloudformationResources: {
  InvokePermission: {
    Type: 'AWS::Lambda::Permission',
    Properties: {
      FunctionName: '$ResourceParam(\'processor\', \'arn\')',
      Action: 'lambda:InvokeFunction',
      Principal: 'sns.amazonaws.com',
      SourceArn: '$CfResourceParam(\'AlertTopic\', \'TopicArn\')'
    }
  }
}
```

### DependsOn for ordering

Use the standard CloudFormation `DependsOn` property to control creation order between raw resources:

```typescript
cloudformationResources: {
  TopicSubscription: {
    Type: 'AWS::SNS::Subscription',
    DependsOn: ['AlertTopic'],
    Properties: {
      TopicArn: '$CfResourceParam(\'AlertTopic\', \'TopicArn\')',
      Protocol: 'email',
      Endpoint: 'alerts@example.com'
    }
  }
}
```

### Limitations of raw CloudFormation resources

- No automatic IAM permissions — you must manage roles and policies yourself
- No `connectTo` support — environment variables and security group rules aren't injected automatically
- No Stacktape-level validation — CloudFormation errors surface only at deploy time
- Property updates may require resource replacement — some CloudFormation properties can't be updated in-place

## AWS CDK constructs

For complex escape-hatch scenarios where raw CloudFormation is too verbose, Stacktape supports embedding [AWS CDK](https://aws.amazon.com/cdk/) constructs directly in your stack. Write a CDK construct class in TypeScript or JavaScript, and Stacktape synthesizes and deploys it alongside your other resources.

CDK constructs are the right choice when you need multiple interrelated AWS resources with complex logic (e.g., a custom monitoring dashboard, a VPC peering setup, or a multi-step Step Functions workflow that Stacktape's built-in [state machine](/resources/orchestration/state-machine) doesn't cover).

### CDK construct configuration

The `AwsCdkConstruct` resource type takes three properties:

| Property | Required | Description |
|---|---|---|
| `entryfilePath` | Yes | Path to a `.ts` or `.js` file exporting a CDK construct class |
| `exportName` | No | Name of the exported class (defaults to `"default"`) |
| `constructProperties` | No | Props object passed to the construct's constructor |

### Example CDK construct

First, write a construct class that extends `Construct` from `aws-cdk-lib`:

```typescript
// cdk/monitoring.ts
import { Construct } from 'constructs';
import * as cloudwatch from 'aws-cdk-lib/aws-cloudwatch';

export class MonitoringDashboard extends Construct {
  constructor(scope: Construct, id: string, props: { functionName: string }) {
    super(scope, id);

    new cloudwatch.Dashboard(this, 'Dashboard', {
      dashboardName: `${props.functionName}-metrics`,
      widgets: [
        [
          new cloudwatch.GraphWidget({
            title: 'Invocations',
            left: [
              new cloudwatch.Metric({
                namespace: 'AWS/Lambda',
                metricName: 'Invocations',
                dimensionsMap: { FunctionName: props.functionName }
              })
            ]
          })
        ]
      ]
    });
  }
}
```

Then reference it in your Stacktape config:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  AwsCdkConstruct,
  $ResourceParam
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/api.ts'
    })
  });

  const monitoring = new AwsCdkConstruct({
    entryfilePath: './cdk/monitoring.ts',
    exportName: 'MonitoringDashboard',
    constructProperties: {
      functionName: $ResourceParam('api', 'name')
    }
  });

  return { resources: { api, monitoring } };
});
// [!code focus-end]
`
    }
  ]}
/>

You can pass dynamic values from other Stacktape resources into construct properties using `$ResourceParam()` and `$Secret()` [directives](/configuration/directives).

For more detail, see [AWS CDK constructs](/resources/advanced/aws-cdk-constructs).

## Inspecting the generated template

To verify that your overrides, transforms, and raw resources produce the expected CloudFormation output, use [`stacktape compile-template`](/cli/compile-template):

```bash
stacktape compile-template --stage dev --region eu-west-1
```

This writes the final CloudFormation template to `compiled-template.yaml` (or a custom path via `--outFile`). The template includes all resolved overrides, applied transforms, synthesized CDK constructs, and raw CloudFormation resources. Inspect it before deploying to catch errors early — CloudFormation deployment failures are slower to debug than template inspection.

## Best practices

- **Try native properties first.** Overrides are a last resort. Check the resource's API reference for the property you need before reaching for an override.
- **Use transforms for array modifications.** If you need to append to an existing array (layers, environment variables, mount points), a transform is safer than an override because it preserves whatever Stacktape already configured.
- **Keep overrides minimal.** Each override is a maintenance obligation — Stacktape may change its internal logical names or resource structure in future versions.
- **Validate with `compile-template`.** Always inspect the compiled output after adding overrides or transforms. Overrides bypass Stacktape's validation, so errors only surface at deploy time if you skip this step.
- **Prefer CDK constructs for multi-resource setups.** If you need more than 2-3 related raw CloudFormation resources with cross-references, a CDK construct is cleaner and benefits from CDK's type-safe APIs.
- **Document why, not what.** When using overrides, add a code comment explaining why the override is needed — not what it does (the code shows that).

## FAQ

### Can I use overrides in YAML configs?

Yes. Overrides work in both TypeScript and YAML configurations. In YAML, specify overrides using the full CloudFormation logical name (from `compile-template` output) and dot-notation paths for nested properties. The user-friendly property names (like `lambda`, `ecsTaskDefinition`) are only available in TypeScript configs using `defineConfig`.

### Are transforms available in YAML configs?

No. Transforms require JavaScript functions and only work in TypeScript configurations using the `defineConfig` pattern. For YAML configs, use static overrides instead.

### What happens if I override a property that Stacktape also sets?

Your override wins. Stacktape applies overrides after it finishes generating the template, so your values take precedence. For object properties at the top level, Stacktape deep-merges your override with existing values. For dot-notation paths, it sets the value directly.

### Will overrides break when Stacktape updates?

Possibly. If Stacktape changes the internal logical name of a child resource, your override will fail with an error indicating the logical name isn't a valid child of that resource. User-friendly property names (in TypeScript) are more stable than raw logical names because Stacktape maps them internally. If you use full logical names from `compile-template`, expect occasional breakage on major updates.

### How do I find what CloudFormation attributes I can reference with $CfResourceParam?

Check the [AWS CloudFormation Resource Reference](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/aws-template-resource-type-ref.html) for the resource type you're targeting. Each resource type lists its return values — these are the attributes you can pass as the second argument to `$CfResourceParam`. Common ones include `Arn`, `Ref` (physical ID), and resource-specific attributes like `TopicArn` for SNS or `BucketName` for S3.

### Do raw CloudFormation resources get automatic IAM permissions?

No. Unlike Stacktape-managed resources (which get permissions automatically via `connectTo`), raw CloudFormation resources require you to manage IAM roles and policies manually. You'll need to add your own `AWS::IAM::Role` or `AWS::IAM::Policy` resources, or use overrides on existing Stacktape-managed roles to add additional policy statements.

### When should I use a CDK construct instead of raw CloudFormation?

Use CDK constructs when you need multiple interrelated resources with complex configuration logic — for example, a CloudWatch dashboard with dynamic widgets, a custom VPC peering setup, or a Step Functions state machine with many states. For a single additional resource (like an SNS topic or a Secrets Manager secret), raw CloudFormation in `cloudformationResources` is simpler.

### Can I reference CDK construct outputs in my Stacktape resources?

Not directly through Stacktape's type system. CDK constructs are synthesized into CloudFormation resources, so you can reference their outputs using `$CfResourceParam` with the synthesized logical names. Run `compile-template` to discover the logical names that CDK generates.

### Does the `cloudformationResources` section count toward the Stacktape resource limit?

No. Resources added via `cloudformationResources` are merged directly into the CloudFormation template and do not count toward Stacktape's managed resource limit. However, AWS CloudFormation has its own per-stack resource limit (500 resources per stack by default).
