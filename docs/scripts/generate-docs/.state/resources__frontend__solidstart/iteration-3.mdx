# SolidStart

A Stacktape SolidStart resource deploys a SolidStart app with SSR on AWS Lambda, static assets on S3, and a CloudFront CDN. Point `appDirectory` at the directory containing `app.config.ts`, and Stacktape handles the SSR function, asset bucket, CDN routing, custom domains, and environment variable injection.

## When to use

A Stacktape SolidStart resource is the right choice when your frontend is a SolidStart app that needs server rendering. The resource is designed around the SolidStart build output: set `appDirectory` to the directory containing `app.config.ts`, and Stacktape wires the SSR function, static asset bucket, and CDN.

Common use cases:

- **Server-rendered apps** — dashboards, portals, and content sites that need dynamic pages instead of only static files
- **SolidStart monorepos** — set `appDirectory` to the workspace containing `app.config.ts`, rather than the repository root
- **Solid's reactive SSR** — SolidStart's fine-grained reactivity model produces fast SSR; the Lambda + CloudFront architecture keeps infrastructure simple

## When NOT to use

- **Pure static websites** — use [static hosting](/resources/frontend/static-hosting) when your Solid app is exported as static files and does not need SSR.
- **Containerized custom servers** — use a [web-service](/resources/compute/web-service) when you want to run a custom Node server or another HTTP process in a container.
- **General backend APIs** — use a [Lambda function](/resources/compute/lambda-function) or [web-service](/resources/compute/web-service) for standalone APIs that are not part of a SolidStart app.
- **Other meta-frameworks** — Stacktape has dedicated resources for [Next.js](/resources/frontend/nextjs), [Astro](/resources/frontend/astro), [Nuxt](/resources/frontend/nuxt), [SvelteKit](/resources/frontend/sveltekit), [TanStack Start](/resources/frontend/tanstack-start), and [Remix](/resources/frontend/remix). Choose the dedicated resource for the framework your team uses.

## Basic example

The smallest useful SolidStart config points Stacktape at the app directory containing `app.config.ts`. In a single-app repository that is often the repository root (the default `"."`); in a monorepo it is the package or workspace directory for the SolidStart app.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const web = new SolidStartWeb({
    appDirectory: './apps/web'
  });

  return {
    resources: { web }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`appDirectory` defaults to `"."` and must point to the directory containing `app.config.ts`. Stacktape uses that directory for the Vinxi build, then deploys the SSR Lambda, uploads static assets to S3, and configures CDN routing.

## Build and dev

Stacktape runs `vinxi build` from `appDirectory` by default. `buildCommand` lets teams replace this when the repository has a custom build script — for example, a monorepo using `pnpm --filter web build` or an app that needs a prebuild step. Leave `buildCommand` unset when a normal Vinxi build works from `appDirectory`; fewer custom commands make CI and local behavior easier to reason about.

`dev` configures the local dev server used by [`stacktape dev`](/local-development/dev-mode-overview). The SolidStart resource defaults to `vinxi dev`. Override `dev.command` when your monorepo expects a workspace-aware dev script, and use `dev.workingDirectory` to set the working directory relative to the project root.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb } from 'stacktape';

export default defineConfig(() => {
  const web = new SolidStartWeb({
    appDirectory: './apps/web',
    // [!code focus-start]
    buildCommand: 'pnpm --filter web build',
    dev: {
      command: 'pnpm --filter web dev'
    }
    // [!code focus-end]
  });

  return { resources: { web } };
});`
    }
  ]}
/>

`dev.command` only affects [`stacktape dev`](/local-development/dev-mode-overview); it does not change the deployed build. Use it when the local dev server needs a package-manager script, monorepo filter, or another wrapper around the default `vinxi dev`.

## Server Lambda

A Stacktape SolidStart resource serves SSR through a Lambda function. `serverLambda` customizes that SSR function, including memory, timeout, VPC access, and logging. The default memory is 1024 MB and the default timeout is 30 seconds (also the maximum). Memory can range from 128 to 10,240 MB; AWS Lambda allocates CPU proportionally — 1,769 MB corresponds to 1 vCPU.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb } from 'stacktape';

export default defineConfig(() => {
  const web = new SolidStartWeb({
    appDirectory: './apps/web',
    // [!code focus-start]
    serverLambda: {
      memory: 2048,
      timeout: 25
    }
    // [!code focus-end]
  });

  return { resources: { web } };
});`
    }
  ]}
/>

Increase `memory` when SSR is CPU-bound or the app initializes large dependencies — doubling from 1024 to 2048 MB also doubles available CPU. Keep the default when pages render quickly and memory pressure is low. Set `joinDefaultVpc: true` when the SSR Lambda must reach VPC-protected resources such as databases or Redis; note that the function loses direct internet access when joined to the VPC.

`serverLambda.logging` configures CloudWatch log behavior, including retention period and the option to disable logging entirely. Logs default to 180-day retention. Use [`stacktape debug:logs`](/cli/debug-logs) or the [Stacktape Console](/stacktape-console/console-overview) to view runtime logs. See the API reference at the bottom of this page for the full `serverLambda` property surface.

## Domains and firewall

A Stacktape SolidStart resource can attach custom domains to the CloudFront CDN and associate a Web Application Firewall with it. `customDomains` configures DNS records and TLS certificates, while `useFirewall` references a `web-app-firewall` resource whose `scope` must be `cdn`.

Use `customDomains` for production and shared staging apps where users need a stable branded URL. A Route 53 hosted zone for the domain must exist in your AWS account. Stacktape creates DNS records and provisions free TLS certificates automatically; set `customCertificateArn` only when you have specific certificate requirements, and `disableDnsRecordCreation` when you manage DNS elsewhere (for example through Cloudflare).

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb, WebAppFirewall } from 'stacktape';

export default defineConfig(() => {
  const firewall = new WebAppFirewall({
    scope: 'cdn'
  });

  const web = new SolidStartWeb({
    appDirectory: './apps/web',
    // [!code focus-start]
    customDomains: [{ domainName: 'www.example.com' }],
    useFirewall: 'firewall'
    // [!code focus-end]
  });

  return { resources: { firewall, web } };
});`
    }
  ]}
/>

Enable `useFirewall` when your public SolidStart app handles user input and needs protection against common web exploits like SQL injection and XSS, or when a public-facing API needs request rate limiting at the CDN edge. AWS WAF adds per-rule and per-million-request costs on top of the base CloudFront charges. Skip it for internal tools, prototypes, or early development stages where WAF rules add configuration work before there is a clear threat model. See [web application firewall](/resources/security/web-application-firewall) for configuring rules.

## Static files and CDN

A Stacktape SolidStart resource uploads static assets to S3 and serves the app through CloudFront. Two properties control file and CDN behavior: `fileOptions` sets custom headers such as `Cache-Control` for uploaded files matching a glob pattern, and `cdn` configures CDN cache behavior for SSR routes and specific path patterns.

Use `fileOptions` when static files need explicit cache or metadata headers that differ from the generated defaults. Leave it unset first; SolidStart and Stacktape already provide the standard split between static assets and SSR responses.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb } from 'stacktape';

export default defineConfig(() => {
  const web = new SolidStartWeb({
    appDirectory: './apps/web',
    // [!code focus-start]
    fileOptions: [
      {
        includePattern: 'assets/**',
        headers: [{ key: 'Cache-Control', value: 'public, max-age=31536000, immutable' }]
      }
    ]
    // [!code focus-end]
  });

  return { resources: { web } };
});`
    }
  ]}
/>

The `includePattern` is a glob that selects which uploaded files the rule applies to. Use a long `max-age` with `immutable` only for fingerprinted assets whose filenames change on each build; use shorter caching for files whose URL stays stable across deploys.

`cdn` configures caching for SSR responses. By default, the CDN cache is flushed on every deploy (`disableInvalidationAfterDeploy` defaults to `false`) so users always see the latest version. Use `defaultCachingOptions` to override the default SSR caching TTLs for all server routes, and `pathCachingOverrides` to set per-path caching rules for specific URL patterns. Most teams should keep the default invalidation behavior and only tune caching after measuring real traffic patterns. See the API reference below for the full `cdn` property surface.

## Connecting resources

`connectTo` gives the SolidStart SSR function access to other Stacktape resources and injects connection details as environment variables following the `STP_[RESOURCE_NAME]_[PARAM]` naming pattern. Use `connectTo` for resources such as relational databases, buckets, DynamoDB tables, SQS queues, SNS topics, and user auth pools. Use `iamRoleStatements` only for AWS permissions not covered by the resource connection model.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb, RelationalDatabase } from 'stacktape';

export default defineConfig(() => {
  const mainDatabase = new RelationalDatabase({
    engine: 'postgres-16',
    instanceClass: 'db.t4g.micro'
  });

  const web = new SolidStartWeb({
    appDirectory: './apps/web',
    // [!code focus-start]
    connectTo: ['mainDatabase'],
    serverLambda: {
      joinDefaultVpc: true
    }
    // [!code focus-end]
  });

  return { resources: { mainDatabase, web } };
});`
    }
  ]}
/>

The `engine` property selects the database engine and version — `'postgres-16'` uses PostgreSQL 16. Other supported engines include earlier PostgreSQL versions and MySQL variants. The `instanceClass` property determines the compute size — `'db.t4g.micro'` is a small ARM-based burstable instance suitable for development and light workloads; production apps typically use larger classes like `db.t4g.small` or `db.t4g.medium`. See the [relational database](/resources/databases/relational-database) page for the full set of engines and instance classes.

With the resource named `mainDatabase`, the SolidStart SSR function receives `STP_MAIN_DATABASE_CONNECTION_STRING`, `STP_MAIN_DATABASE_HOST`, and `STP_MAIN_DATABASE_PORT` as environment variables. Set `joinDefaultVpc: true` on `serverLambda` when connecting to VPC-protected resources like databases or Redis clusters. For the full connection model and injected variables by resource type, see [connecting resources](/configuration/connecting-resources).

## Environment variables

A Stacktape SolidStart resource can set explicit `environment` variables for the SSR function. Use these for application configuration, feature flags, public service URLs needed by server code, and values produced by directives such as `$ResourceParam()` or `$Secret()`.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SolidStartWeb } from 'stacktape';

export default defineConfig(() => {
  const web = new SolidStartWeb({
    appDirectory: './apps/web',
    // [!code focus-start]
    environment: [
      { name: 'APP_ENV', value: 'production' },
      { name: 'STRIPE_KEY', value: '$Secret(stripe-key)' }
    ]
    // [!code focus-end]
  });

  return { resources: { web } };
});`
    }
  ]}
/>

Prefer `connectTo` for Stacktape-managed resources because it also handles permissions and, where needed, network access. Use explicit `environment` entries for values that are not resource connections — application secrets, feature flags, or third-party API keys. For sensitive values, reference secrets through the [secrets](/configuration/secrets) and [directives](/configuration/directives) flow instead of hard-coding credentials in config.

## FAQ

### What does Stacktape create for a SolidStart app?

A Stacktape SolidStart resource contains nested `bucket` and `serverFunction` resources. The `serverFunction` is a Lambda that handles SSR, the `bucket` holds static assets on S3, and a CloudFront CDN sits in front of both to route requests. See the [resources overview](/configuration/resources) for how Stacktape resources map to AWS.

### Where should `appDirectory` point?

`appDirectory` should point to the directory containing `app.config.ts`. It defaults to `"."`, the repository root. In a monorepo, set it to the SolidStart workspace rather than the root. The basic example on this page uses `./apps/web` to make that monorepo shape explicit.

### Can I use a custom domain with SolidStart?

Yes. Configure `customDomains` on the SolidStart resource, and make sure a Route 53 hosted zone for the domain exists in your AWS account. Stacktape creates the DNS record and manages the TLS certificate automatically unless you provide `customCertificateArn` or set `disableDnsRecordCreation`. See [custom domains](/resources/networking/custom-domains) for the broader domain model.

### Can I protect a SolidStart app with AWS WAF?

Yes. Set `useFirewall` to the name of a [web application firewall](/resources/security/web-application-firewall) resource whose `scope` is `cdn`. This protects the CloudFront CDN path. AWS WAF adds per-rule and per-million-request costs; it is separate from load-balancer firewall attachment paths used by compute resources.

### How do I connect to a database from SolidStart server functions?

Use `connectTo` to reference the database resource, and set `serverLambda.joinDefaultVpc: true` for VPC-protected databases like RDS or Redis. Stacktape injects connection details as environment variables following the `STP_[RESOURCE_NAME]_[PARAM]` pattern. Access these in SolidStart server functions via `process.env`. See [connecting resources](/configuration/connecting-resources) for details.

### What is the default build command?

The default build command is `vinxi build`, which is the standard SolidStart build tool. Override it with `buildCommand` when your monorepo needs a workspace-aware command like `pnpm --filter web build` or when the app has a custom build script. For local development, the default dev command is `vinxi dev`, configurable through `dev.command`.

### How much does hosting SolidStart on Lambda and CloudFront cost?

AWS bills the underlying services separately: Lambda invocations and duration for SSR, S3 storage and requests for static assets, and CloudFront data transfer and requests for CDN traffic. Lambda's free tier includes 1 million requests and 400,000 GB-seconds of compute per month. CloudFront offers 1 TB of data transfer free in the first year. Use [cost dashboards](/managing-costs/dashboards) after deployment to inspect real spend by stack.

### When should I use static hosting instead of the SolidStart resource?

Use [static hosting](/resources/frontend/static-hosting) when the site can be built into static files and does not need SSR. Static hosting has fewer moving parts, no Lambda cold starts, and lower per-request cost since every response is served from S3 through the CDN. Use the SolidStart resource when the app needs runtime server rendering for dynamic content, authentication, or data fetching during SSR.

### When should I use a web service instead of the SolidStart resource?

Use a [web-service](/resources/compute/web-service) when you want to run a custom HTTP server in a container, control the server process directly, need long-running WebSocket connections, or deploy a framework that is not modeled by the frontend resources. The SolidStart resource is purpose-built for the standard SolidStart build output — it manages SSR, static assets, and CDN routing without requiring a container.

### How does SolidStart SSR compare to Next.js SSR on Stacktape?

Both deploy with the same underlying architecture: Lambda for SSR, S3 for static assets, and CloudFront for CDN. The [Next.js resource](/resources/frontend/nextjs) has its own framework-specific configuration surface with additional options. The SolidStart resource is a focused deployment model for standard SolidStart SSR. Choose based on which framework your team uses and the framework-level features you need.
