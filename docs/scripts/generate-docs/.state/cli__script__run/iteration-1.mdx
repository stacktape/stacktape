# script:run

The `script:run` command executes a script defined in the `scripts` section of your Stacktape configuration. Use it to run database migrations, seed data, invoke build steps, or perform any custom task that needs access to your stack's environment variables, secrets, and AWS credentials.

## Usage

```bash
stacktape script:run --scriptName myMigration --stage production
```

The command resolves all directives (`$Secret`, `$ResourceParam`) in the script's environment, injects `connectTo` variables, and then executes the script locally or on a bastion host depending on the script type.

## When to use

Use `script:run` when you need to execute operational tasks that depend on your deployed stack's context — connection strings, secrets, or IAM credentials. Common use cases:

- Running database migrations after a deploy
- Seeding data into a database or DynamoDB table
- Executing one-off maintenance commands against VPC-private resources (via bastion tunneling)
- Running integration test suites that need real AWS credentials

For scripts that must run automatically during deployment, define them as [deployment scripts](/resources/advanced/deployment-scripts) instead. For ad-hoc shell access to a container, use [`container:session`](/cli/container-session).

## Important flags

### --scriptName (required)

The name of the script as defined in your configuration's `scripts` section. Must match exactly.

### --stage (required)

The stage whose deployed resources are used to resolve directives and environment. For example, `--stage production` resolves `$Secret(...)` and `$ResourceParam(...)` against the production stack.

### --env

Pass additional environment variables to the script execution context. Repeat the flag for multiple variables.

```bash
stacktape script:run --scriptName seed --stage dev --env DB_SEED_COUNT=100 --env VERBOSE=true
```

### --assumeRoleOfResource

Assume the IAM role of a deployed resource during script execution. This gives your script the same AWS permissions as that resource — useful when the script needs to access AWS services (S3, DynamoDB, SQS) with the same grants your workload has.

```bash
stacktape script:run --scriptName cleanup --stage production --assumeRoleOfResource myWorker
```

If `assumeRoleOfResource` is also defined in the script's configuration properties, the CLI flag takes precedence.

## Examples

Run a database migration script against the staging stack:

```bash
stacktape script:run --scriptName migrate --stage staging --region eu-west-1
```

Run a script with a custom config path and extra environment variables:

```bash
stacktape script:run --scriptName seed --stage dev --configPath ./infra/stacktape.ts --env SEED_FILE=./data/seed.json
```

Run a script assuming the IAM role of a deployed Lambda function (so the script has the same S3/DynamoDB permissions):

```bash
stacktape script:run --scriptName exportData --stage production --assumeRoleOfResource dataProcessor
```

## Errors

If the `--scriptName` value does not match any key in the `scripts` section of your configuration, the command fails with:

> Script **\{scriptName\}** is not defined in the 'scripts' section of the configuration.

Double-check the script name in your `stacktape.ts` and ensure it matches exactly (case-sensitive).
