# Managing Costs Overview

AWS infrastructure costs are billed directly to your AWS account, separate from Stacktape's subscription fee. The Stacktape Console provides cost dashboards, per-stack cost attribution, and budget alerts to help you understand and control your AWS spend. This section also covers optimization strategies to help you reduce costs.

Managing costs involves three activities, each covered in a dedicated page:

| Activity | Question it answers | Details |
|----------|---------------------|---------|
| **Viewing costs** | What am I spending and where? | [Dashboards](/managing-costs/dashboards), [Per-resource breakdown](/managing-costs/per-resource-breakdown) |
| **Controlling costs** | How do I prevent overspend? | [Budgets](/managing-costs/budgets) |

<Tip>
If you are new to cost management, start by checking the overview dashboard for your current month-to-date spend, then set up an organization-wide budget alert as a safety net. Those two steps cover most teams.
</Tip>

## Stacktape charges vs. AWS charges

Stacktape costs and AWS costs are independent charges on separate invoices. AWS infrastructure costs are charges in your connected AWS account. Stacktape subscription and billing are managed separately.

| Charge | Billed by | What it covers |
|--------|-----------|----------------|
| AWS infrastructure costs | AWS (to your AWS account) | Compute, storage, networking, databases — everything your resources use |
| Stacktape subscription | Stacktape (to your organization) | Console access, GitOps, build runners, observability features, support |

Stacktape does not handle or intermediate your AWS charges — AWS bills you directly. Stacktape reads cost data from your connected AWS account for display in the Console. Subscription plans and billing are managed on the [Billing page](/stacktape-console/billing-and-subscription).

## Viewing costs

The Stacktape Console shows AWS cost data in two places: the organization overview dashboard and the dedicated costs per stack page. Cost reports are generated daily from AWS Cost and Usage Reports, so figures reflect spend with up to a one-day delay.

The **overview dashboard** displays a month-to-date AWS costs KPI with a trend comparison against the previous month (percentage change), a costs-by-project chart (top 10 projects plus an "Other" bucket), a costs-by-resource-type chart, and a monthly cost trend sparkline. These give you a quick spending snapshot without navigating away from the main dashboard. Budget alert status also appears on the overview dashboard, showing each configured budget with its name, scope, alert percentages, and threshold.

The **costs per stack page** lets you select a connected AWS account and a specific month, then shows a table of every attributed stack with its project name, stage, region, and month-to-date spend — sortable by cost. Clicking a row navigates to the per-stage cost detail view. This is where you go to compare spend across stages or find your most expensive stack for a given month.

<Info>
New stacks may take up to a day to appear in cost data after their first deployment.
</Info>

For the full walkthrough of these views, see [Dashboards](/managing-costs/dashboards). For guidance on reading the costs-by-resource-type chart, see [Per-resource breakdown](/managing-costs/per-resource-breakdown).

### Cost attribution

The costs-per-stack view uses attributed stack data to show costs by project and stage. Per-project and per-stage cost attribution only works for resources deployed through Stacktape, because attribution relies on deployment metadata that Stacktape attaches during deployment. Stacks not deployed through Stacktape do not appear in the per-stack cost table.

Stack-scoped cost and budget views depend on this deployment metadata. When you deploy a stack through Stacktape, the metadata is attached automatically — no additional configuration is needed.

## Controlling costs with budgets

Budget alerts let you set spending thresholds and receive notifications before costs exceed them. Stacktape supports two scopes:

- **Organization-wide budgets** — applied to an entire AWS account. Created in the selected AWS account immediately when you save the budget configuration.
- **Per-stack budgets** — scoped to specific projects and stages. These depend on Stacktape deployment metadata, so you need to deploy matching stacks after creating or changing stack-scoped alerts.

Each budget has a threshold, a currency, and one or more alert percentages. When month-to-date spend crosses a percentage threshold (e.g., 50%, 80%, 100% of the budget), an alert is sent to your configured [alert channels](/observability/alert-channels).

Organization-wide budgets are the simpler option — they cover all spend in an AWS account and activate immediately. Per-stack budgets give you granular control but require a redeploy of each matching stack so the deployment metadata is in place. Most teams start with an organization-wide budget and add per-stack budgets for production stages as they grow.

<Tip>
Start with a single organization-wide budget at a comfortable monthly cap. Add per-stack budgets later for production stages that you want to monitor individually.
</Tip>

For setup details, see [Budgets](/managing-costs/budgets).

## Reducing costs

The largest cost-reduction levers in a typical Stacktape deployment are:

1. **Right-sizing compute** — matching memory, CPU, and instance types to actual workload needs rather than over-provisioning. A [Lambda function](/resources/compute/lambda-function) with 512 MB of memory costs half as much per millisecond as one with 1,024 MB — if your function doesn't need the extra memory, you're paying for nothing.
2. **Scaling to zero** — using Lambda functions or scale-to-zero container configurations for workloads with intermittent traffic. If a workload handles fewer than a few requests per second on average, Lambda's pay-per-invocation model is usually cheaper than an always-on container.
3. **Choosing the right database tier** — using serverless database engines for variable workloads and provisioned instances only when sustained throughput justifies the cost. A [relational database](/resources/databases/relational-database) running a `db.t4g.micro` instance costs a fraction of an Aurora Serverless cluster at idle, but Aurora Serverless scales automatically under load.
4. **CDN caching** — offloading repetitive responses to [CDN](/resources/networking/cdn) edge locations to reduce origin compute. If your API returns the same response for the same request, caching it at the edge avoids hitting your backend entirely.
5. **Cleaning up unused stages** — [deleting](/deployment-and-lifecycle/destroying-stacks) preview and dev stages that are no longer needed. Every running stage incurs costs for its databases, containers, and other always-on resources, even if no traffic is hitting them.

For concrete, actionable guidance on each of these, see [Optimization tips](/managing-costs/optimization-tips).

## FAQ

### How long does it take for cost data to appear?

Cost data comes from AWS Cost and Usage Reports, which AWS generates daily. After deploying a new stack, expect up to 24 hours before its costs appear in the Console. Existing stacks update once per day. This delay is an AWS-side constraint — Stacktape fetches reports as soon as they are available.

### Are AWS costs billed through Stacktape?

No. AWS bills your AWS account directly. Stacktape does not handle or intermediate your AWS charges — it reads cost data from your connected AWS account for display in the Console. Stacktape's own subscription is a separate charge billed by Stacktape to your organization. See [Billing and subscription](/stacktape-console/billing-and-subscription) for subscription details.

### How do budget alerts work?

You define a spending threshold, a currency, and one or more percentage triggers (e.g., alert at 50%, 80%, 100%). When your month-to-date spend crosses a trigger, Stacktape sends an alert to your configured [alert channels](/observability/alert-channels). Organization-wide budgets activate immediately; per-stack budgets require a deploy of matching stacks so the deployment metadata is attached. See [Budgets](/managing-costs/budgets) for setup steps.

### Can I set budgets per stage?

Yes. Stack-scoped budgets can target specific projects and stages, which is useful for monitoring production spend separately from development stages. After creating a stack-scoped budget, deploy the matching stacks so the deployment metadata is attached. Organization-wide budgets are simpler if you just want a single safety net across all stages.

### What AWS services typically cost the most in Stacktape deployments?

For most deployments, the top cost drivers are compute (ECS Fargate tasks or Lambda invocations), databases (RDS or Aurora instances), and data transfer. The exact breakdown depends on your architecture. The overview dashboard's costs-by-resource-type chart shows your actual distribution, so you can identify your biggest cost driver at a glance.

### How can I reduce Lambda function costs?

Reduce Lambda costs by right-sizing memory (smaller memory means lower per-millisecond cost), reducing execution duration through code optimization, and using ARM/Graviton architecture which is approximately 20% cheaper per GB-second than x86. For high-throughput functions that run continuously, evaluate whether a container-based [web service](/resources/compute/web-service) would be cheaper at sustained load. See [Optimization tips](/managing-costs/optimization-tips) for detailed guidance.

### How often are cost reports updated?

AWS generates Cost and Usage Reports once per day. The Stacktape Console fetches these reports daily, so the data you see is at most 24 hours old. Real-time cost monitoring is not available through this mechanism — use AWS CloudWatch billing alarms if you need sub-daily alerts on spend thresholds.

### What is the difference between the overview dashboard and the costs page?

The overview dashboard shows a spending summary — month-to-date total, trend vs. last month, costs by project (top 10), and costs by resource type. The dedicated costs per stack page provides a full table of every attributed stack with sortable columns, AWS account and month selectors, and navigation to per-stage detail views. See [Dashboards](/managing-costs/dashboards) for a walkthrough of both.

### How does Stacktape cost tracking compare to AWS Cost Explorer?

The Stacktape Console surfaces cost data organized by your Stacktape projects and stages, giving you stack-level attribution without manually configuring AWS tags or Cost Explorer filters. AWS Cost Explorer offers deeper analysis capabilities — custom reports, forecasting, savings plan recommendations, and full-account visibility across all services. Use the Stacktape Console for day-to-day stack-level cost awareness, and AWS Cost Explorer when you need advanced analysis or custom report exports.

### Can I see costs for resources not created by Stacktape?

Per-project and per-stage cost attribution only works for resources deployed through Stacktape, because attribution relies on deployment metadata that Stacktape attaches during deployment. Resources created outside of Stacktape (manually in the AWS console or by other tools) are not shown in the per-stack cost table. To view all AWS spending including non-Stacktape resources, use AWS Cost Explorer directly in your AWS account.
