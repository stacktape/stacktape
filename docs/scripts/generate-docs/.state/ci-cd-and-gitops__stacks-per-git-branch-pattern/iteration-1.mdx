# Stacks per Git Branch Pattern

Every Stacktape stage is a complete, isolated stack — its own databases, functions, containers, and endpoints. The stacks-per-branch pattern maps git branches and pull requests to stages so each line of development gets independent infrastructure. This eliminates environment contention, lets reviewers test against real AWS resources, and gives teams a clean promotion path from development to production.

## How the pattern works

A Stacktape stage is identified by its name (e.g. `prod`, `staging`, `pr-42`). The stage name is the only input that separates one deployed stack from another within the same project and region. By deriving that stage name from a git branch or PR number, you get a 1:1 mapping between code branches and live infrastructure.

The pattern has three common shapes:

| Shape | Git event | Stage name | Lifecycle |
|-------|-----------|-----------|-----------|
| **Long-lived branches** | Push to `main`, `develop` | Fixed name like `prod`, `staging` | Permanent — updated on every push |
| **PR previews** | PR opened against `main` | `pr-{number}` (e.g. `pr-42`) | Ephemeral — destroyed when PR closes |
| **Feature branches** | Push to `feat/auth` | Short name like `auth` | Medium-lived — destroyed when feature merges |

All three shapes use the same underlying mechanism: deploy with a derived `--stage` value. The only differences are how you derive the name, how long the stage lives, and how cleanup happens.

## Long-lived branch stages

Long-lived branch stages are the backbone of most deployment setups. They map permanent branches to permanent stages that are updated on every push.

A typical mapping:

| Branch | Stage | Purpose |
|--------|-------|---------|
| `main` | `prod` | Production — receives code after merge |
| `develop` | `staging` | Pre-production integration and QA |
| `release/*` | `release` | Release candidate testing |

The stage name does not have to match the branch name. Most teams use short, meaningful names (`prod`, `staging`) rather than the literal branch name. The mapping is configured either in the [Stacktape Console's GitOps settings](/ci-cd-and-gitops/gitops-with-console) (where you create a "Push to Branch" trigger) or in your [custom CI/CD pipeline](/ci-cd-and-gitops/custom-ci-cd).

### Setting up with GitOps

In the Console, create a GitOps configuration with "Push to Branch" as the trigger, select the branch to watch, and set your desired stage name. Every push to that branch triggers a deployment to the named stage. See [GitOps with Console](/ci-cd-and-gitops/gitops-with-console) for setup details.

### Setting up with custom CI/CD

In your pipeline file, derive the stage name from the branch name or use a static mapping. For most pipelines, the branch name is available as an environment variable.

GitHub Actions example deploying `main` to `prod`:

```bash
npx stacktape deploy --stage prod --region eu-west-1
```

Or dynamically from the branch:

```bash
npx stacktape deploy --stage "${GITHUB_REF_NAME}" --region eu-west-1
```

<Warning>
When using the branch name directly as the stage name, ensure the branch name meets stage naming constraints — lowercase alphanumeric and dashes only, minimum 2 characters. Branch names like `feature/add-auth` contain a slash and won't work. Sanitize them first (e.g. extract only the suffix).
</Warning>

## PR preview environments

PR previews give every pull request a dedicated stage. Reviewers test changes against a complete, isolated stack before merging — no shared staging environment, no risk of one PR overwriting another's state.

### How PR previews work with GitOps

When you configure a "PR Opened" trigger in the Console, the GitHub App handler performs these steps:

1. **PR opened** — The handler starts a deployment to stage `pr-{number}` (e.g. `pr-42`). It posts a comment on the PR with deployment status and a link to monitor progress in the Console.
2. **Subsequent pushes** — A temporary internal configuration is created so that pushes to the PR branch also trigger redeployments to the same `pr-{number}` stage. The preview stays current with the latest code.
3. **PR closed or merged** — If auto-delete is enabled, the handler starts a stack deletion for the preview stage and removes the temporary configuration.

The PR comment is updated throughout the lifecycle — showing pending status, success with quick links to deployed resources, or failure details with a link to deployment logs.

### How PR previews work with custom CI/CD

In custom pipelines, you control the naming and lifecycle explicitly. Most CI systems expose the PR number as an environment variable.

Deploy on PR open or push:

```bash
npx stacktape deploy --stage "pr-${PR_NUMBER}" --region eu-west-1 --autoConfirmOperation
```

Delete on PR close (add this as a separate pipeline step triggered on PR close/merge):

```bash
npx stacktape delete --stage "pr-${PR_NUMBER}" --region eu-west-1 --autoConfirmOperation
```

<Tip>
Use `--autoConfirmOperation` in CI pipelines to skip the interactive confirmation prompt that would otherwise block the job.
</Tip>

### Auto-delete on PR close

When creating a PR preview GitOps configuration in the Console, you can enable the "Delete stack after pull request is merged or closed" option. This is the recommended default — it prevents preview stages from accumulating and incurring ongoing AWS costs for databases, load balancers, or container workloads that no one is using.

Disable auto-delete only when you have a specific reason to keep preview stages alive after merge, such as:

- Post-merge QA validation that runs against the deployed preview
- Debugging a deployment issue after the PR was merged
- Compliance workflows that require a frozen snapshot

Even with auto-delete disabled, you can always remove a leftover preview stage manually with [`stacktape delete`](/cli/delete).

## Feature branch stages

Feature branch stages sit between long-lived stages and PR previews. They're deployed from a named branch (e.g. `feat/payments`) and live until the feature is merged or abandoned.

Unlike PR previews, feature stages don't auto-create from a trigger — you deploy them explicitly when you want a live environment for a long-running feature. Unlike long-lived stages, they're torn down once the feature ships.

Deploy a feature branch stage:

```bash
npx stacktape deploy --stage payments --region eu-west-1
```

Clean up when the feature ships:

```bash
npx stacktape delete --stage payments --region eu-west-1
```

Feature branches are most useful when:

- A feature takes weeks and needs a stable test environment beyond local dev mode
- Multiple team members are collaborating on the feature and need a shared environment
- The feature requires integration testing against third-party services that need real endpoints

For short-lived features (a day or two), PR previews are simpler because cleanup is automated.

## Stage naming conventions

Stage names must follow these constraints:

| Constraint | Value |
|-----------|-------|
| **Allowed characters** | Lowercase letters (a-z), digits (0-9), and hyphens (-) |
| **Minimum length** | 2 characters |
| **Maximum length** | 12 characters (enforced in Console GitOps configurations) |

<Warning>
Keep stage names short. The stage name is incorporated into AWS resource names (CloudFormation stacks, S3 buckets, Lambda functions), which have their own length limits. A long stage name combined with a long resource name can exceed these limits and cause deployment failures.
</Warning>

### Recommended naming patterns

| Pattern | Convention | Examples |
|---------|-----------|----------|
| Production | `prod` | — |
| Staging | `staging` | — |
| PR previews | `pr-{number}` | `pr-42`, `pr-137` |
| Feature branches | Short feature name | `auth`, `payments` |
| Developer personal | Developer initials or short name | `mc`, `dev-mc` |

Avoid using the full branch name as the stage name. Branch names like `feature/add-new-auth-flow` are too long and contain invalid characters (slashes). Instead, extract a short identifier.

## Cost management

Each stage is a complete independent stack. Every resource defined in your configuration is provisioned separately for each stage — separate databases, separate Lambda functions, separate containers. This means PR previews and feature branches incur real AWS costs.

### Strategies to control costs

**Right-size preview resources.** Use [stage-specific configuration](/configuration/stages-and-environments) or [directives](/configuration/directives) to give preview environments smaller database instances, lower container counts, or fewer replicas than production.

**Auto-delete PR previews.** Enable the "delete stack after PR close" option in GitOps, or add a deletion step in your custom CI pipeline. Stale previews running unused databases or containers are the most common source of unexpected bills.

**Use serverless resources in previews.** Lambda functions and DynamoDB tables have pay-per-use pricing with generous free tiers. A preview stage with only serverless resources costs nearly nothing when idle. Container workloads and relational databases incur costs whether traffic is flowing or not.

**Set budget alerts.** Configure [budgets](/managing-costs/budgets) to notify your team when spend exceeds a threshold. This catches runaway costs from stale stages.

**Audit regularly.** Use [`stacktape info:stacks`](/cli/info-stacks) to list all deployed stages and identify stale ones that should be deleted.

## Cleanup strategies

Stale stages are the primary risk of the stacks-per-branch pattern. A forgotten feature branch stage running a database instance costs money every hour.

| Strategy | How | Best for |
|----------|-----|----------|
| **GitOps auto-delete** | Enable "delete on PR close" in GitOps configuration | PR previews |
| **CI pipeline step** | Add a `stacktape delete` step triggered on PR close or branch deletion | Custom CI/CD setups |
| **Manual cleanup** | Run `stacktape delete --stage {name}` | Feature branches, one-off stages |
| **Periodic audit** | Schedule a script that lists stacks and deletes those older than N days | Teams with many developers |

<Info>
Deleting a stage removes all AWS resources provisioned for that stage — databases, containers, buckets, and their data. This is irreversible. Make sure the stage is no longer needed before deleting.
</Info>

## Combining patterns

Most teams combine two or three patterns. A common setup:

| Pattern | Trigger | Stage | Lifecycle |
|---------|---------|-------|-----------|
| Production | Push to `main` | `prod` | Permanent |
| Staging | Push to `develop` | `staging` | Permanent |
| PR previews | PR opened against `main` | `pr-{number}` | Auto-deleted on close |

This gives you:
- Continuous deployment to staging on every push to `develop`
- Production deployment on merge to `main`
- Isolated preview environments for every PR targeting `main`, automatically cleaned up

You can set up all three as GitOps configurations in the Console without writing any pipeline files. See [GitOps with Console](/ci-cd-and-gitops/gitops-with-console) for the full setup walkthrough.

## When to use this pattern

The stacks-per-branch pattern is the right default for most teams using Stacktape. It works particularly well when:

- Multiple developers work in parallel and need isolation from each other
- Code reviewers want to see changes running in a real environment
- You want a clear promotion path (feature → staging → production)
- Your stack includes stateful resources (databases) where shared environments cause conflicts

## When NOT to use this pattern

Skip stacks-per-branch (or use it minimally) when:

- **Your stack is expensive to provision.** A stage with large RDS instances, multiple container services, or GPU-backed batch jobs costs significant money per hour. In this case, limit previews to serverless-only subsets or share a single staging environment.
- **Provisioning time is too long.** Complex stacks with many resources can take 10-15 minutes to deploy. If reviewers won't wait that long, a shared staging environment with feature flags may work better.
- **Data isolation isn't needed.** If your service is stateless and doesn't depend on databases, testing against a shared staging environment is simpler and cheaper.
- **You have very few developers.** A solo developer or pair rarely needs per-branch isolation — `prod` and `staging` (or just `prod` and dev mode) may be sufficient.

## FAQ

### How does a PR preview environment differ from staging?

A PR preview is an isolated, ephemeral stage created for a single pull request. It gets its own databases, functions, and endpoints, completely separate from staging. Staging is a long-lived shared environment updated on every push to a specific branch. PR previews eliminate the "who broke staging" problem and let multiple PRs be tested simultaneously without interference.

### How much does a PR preview environment cost?

Each preview stage provisions the same AWS resources as any other stage. For serverless-only stacks (Lambda functions, DynamoDB, API Gateway), idle previews cost nearly nothing due to pay-per-use pricing. For stacks with always-on resources (RDS databases, ECS containers), each preview incurs the same hourly cost as production. Enable auto-delete on PR close and right-size preview resources to control costs.

### Can I use different configuration for preview stages vs production?

Yes. Stacktape supports stage-specific configuration through [directives](/configuration/directives) and conditional logic in your TypeScript config. Use smaller database instances, fewer container replicas, or skip expensive resources entirely in preview stages. The `defineConfig` function receives the stage name, so you can branch on it directly.

### What happens to data when a preview stage is deleted?

All resources are destroyed, including databases and their data. This is by design — preview stages are ephemeral and should not contain data you need to keep. If your workflow requires data persistence across PR iterations, use seed scripts in [deployment hooks](/configuration/hooks-and-scripts) to populate the database on each deployment.

### Can I deploy the same PR to multiple regions?

Yes. Create multiple PR preview GitOps configurations targeting the same branch but different AWS regions or accounts. Each PR will get a stage in each configured region. The stage name (`pr-{number}`) is the same across regions, but the stacks are completely independent.

### How do I debug a failing PR preview deployment?

Check the PR comment posted by the Stacktape GitHub App — it includes a link to the deployment logs in the Console. You can also use [`stacktape debug:logs`](/cli/debug-logs) with `--stage pr-{number}` to tail logs from the deployed resources, or check [`stacktape info:operations`](/cli/info-operations) to see the deployment history.

### What's the maximum number of stages I can deploy?

There is no Stacktape-imposed limit on the number of stages. However, AWS accounts have service quotas (e.g., CloudFormation stacks per region, IAM roles per account) that can become limiting with many simultaneous stages. For teams with many concurrent PRs, monitor your AWS service quotas and request increases proactively. Cleaning up stale stages is the best mitigation.

### Can I share a database between stages?

Not directly within the stacks-per-branch pattern — each stage provisions its own resources by design. If you need a shared database (e.g., a read-only reference dataset), define it in a separate stack and reference it via environment variables or [`$ResourceParam()`](/configuration/directives) from the consuming stages. This is an advanced pattern and adds operational complexity.

### How do I run database migrations in a PR preview?

Use [deployment scripts](/resources/advanced/deployment-scripts) or [hooks](/configuration/hooks-and-scripts) to run migrations automatically during deployment. Since each PR preview has its own database, migrations run in isolation without affecting other stages. The migration runs against the preview's database connection string, which is available via `connectTo` environment variables.

### GitOps vs custom CI/CD for the stacks-per-branch pattern — which should I choose?

GitOps with Console is simpler to set up and maintain — no pipeline files, automatic PR lifecycle management, and built-in status comments on PRs. Choose [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) when you need pre-deployment tests, approval gates, custom build steps, or integration with existing pipelines. Many teams start with GitOps and move to custom CI/CD only when they outgrow the managed flow.
