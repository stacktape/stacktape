# Organizations, Projects, and Stages

The Stacktape Console organizes all work into a three-level hierarchy: organizations contain projects, and projects contain stages. This structure maps cleanly to how teams ship software — one organization per company or team, one project per application (typically one Git repository), and one stage per deployed environment (production, staging, dev). Every resource, cost metric, alarm, and deployment lives within this hierarchy.

## How the hierarchy works

### Organizations

An organization is the top-level boundary for team collaboration, billing, and access control. When you sign up, Stacktape creates a default personal organization tied to your email. You can create additional organizations to separate work between teams, clients, or business units.

Organizations own:

- Connected AWS accounts
- Team members and their roles
- Projects and all their stages
- Budgets, guardrails, alert channels, and notification rules
- Build runner settings (CodeBuild compute type, base image)
- Git provider and third-party integrations

You switch between organizations using the dropdown in the Console header. The selected organization scopes everything you see — projects, costs, alarms, and team members all belong to the currently active organization.

You can also manage organizations from the CLI with [`stacktape org:create`](/cli/org-create) and [`stacktape org:list`](/cli/org-list).

### Projects

A project represents a single application. It usually maps 1:1 to a Git repository. Each project has a name (lowercase letters, numbers, and dashes) and optionally a connected Git repository for automated deployments.

In the Console, navigate to **Projects** to see all projects in the current organization. Each project card shows:

- The project name and connected Git URL
- Number of deployed stages
- Last deployment time
- Current month's cost with a trend indicator (rising, falling, or flat compared to last month)

Projects are sorted by most recently updated stage. You can filter the list using the search input at the top of the page.

### Stages

A stage is a running deployment of a project — a complete, isolated set of AWS resources. Common stage names include `production`, `staging`, `dev`, or `test`. Stages within the same project share no resources with each other: each gets its own databases, functions, containers, and networking.

Stage names must be 1–10 characters long, lowercase alphanumeric with dashes allowed.

You can share the same Stacktape configuration across all stages of a project (most common), or maintain separate configurations per stage when environments need different resource shapes.

<Info>
Stages are sometimes called "environments" in other tools. Stacktape uses "stage" as the canonical term, but they mean the same thing — an isolated deployment of your project.
</Info>

## Walkthrough: creating a project and deploying a stage

### Step 1 — Create a project

In the Console, navigate to **Projects** and click **Create new project**. You can create a project from a starter template, from an existing Git repository, or as an empty project.

If you connect a Git repository, Stacktape can deploy automatically on push (see [GitOps with Console](/ci-cd-and-gitops/gitops-with-console)).

From the CLI, use [`stacktape project:create`](/cli/project-create) with the `--projectName` flag.

### Step 2 — Create a stage

Inside the project view, click **New stage** in the left sidebar. Enter a stage name (e.g., `production` or `staging`). The Console creates an undeployed stage record and routes you to the deployment setup.

If the project has a Git repository connected, the Console guides you through configuring the deployment branch and triggers. If no Git repository is connected, the Console shows CLI instructions for deploying manually.

### Step 3 — Deploy

Run [`stacktape deploy`](/cli/deploy) with `--projectName` and `--stage` flags, or trigger a deployment through GitOps. The Console tracks deployment progress in real time and shows the stage as active once complete.

```bash
stacktape deploy --projectName my-app --stage production --region eu-west-1
```

Once deployed, the stage appears in the left sidebar of the project view with its status, and you can navigate to its overview, logs, metrics, costs, and configuration.

## Common tasks

### Switching organizations

Click the organization name in the Console header. A dropdown lists all organizations you belong to. Click any organization to switch context — the page reloads scoped to that organization's projects and settings.

You can also create a new organization directly from this dropdown.

### Filtering and finding projects

On the Projects page, use the filter input to search by project name or Git URL. The fuzzy search narrows results as you type. Projects are sorted by the most recently deployed stage, so active projects appear first.

### Viewing stage details

Click a project card to open the project view. The left sidebar lists all stages (deployed and undeployed). Click a stage to see its:

- **Overview** — resource list, deployment status, last operation
- **Logs** — real-time log streaming
- **Metrics** — CloudWatch metrics for all resources
- **Costs** — per-resource cost breakdown
- **Activity** — deployment history
- **Configuration** — the Stacktape config used for this stage

Dev-mode stacks (created via [`stacktape dev`](/cli/dev)) appear at the bottom of the stage list, visually separated from regular stages.

### Redeploying a stage

From the stage overview, you can trigger a redeploy of a specific commit. This uses the same deployment runner (CodeBuild or EC2) configured for the project.

### Deleting a stage or project

To delete a deployed stage, use [`stacktape delete`](/cli/delete) from the CLI or trigger deletion from the Console's stage danger zone. Deletion removes all AWS resources associated with that stage.

To delete a project, open the project settings dropdown (gear icon next to the project name in the left sidebar) and select **Delete project**. If the project still has deployed stages, the Console warns you and recommends deleting them first.

<Warning>
Deleting a project does not automatically delete its deployed stages. Delete all stages first to avoid orphaned AWS resources that continue incurring costs.
</Warning>

### Configuring build runners

Organization-level build runner settings control the default CodeBuild compute type and base image for all projects. In the Console, navigate to **Projects**, click the settings gear icon, and select **Configure Codebuild settings**.

Available compute types:

| Compute type | CPU | Memory |
|---|---|---|
| BUILD_GENERAL1_SMALL | 2 vCPU | 4 GB |
| BUILD_GENERAL1_MEDIUM | 4 vCPU | 8 GB |
| BUILD_GENERAL1_LARGE | 8 vCPU | 16 GB |
| BUILD_GENERAL1_XLARGE | 16 vCPU | 32 GB |
| BUILD_GENERAL1_2XLARGE | 72 vCPU | 144 GB |

You can also set additional install commands that run before every build. For per-project runner configuration, open the project settings and select **Configure Runner**.

See [Build runners](/ci-cd-and-gitops/build-runners) for details.

### Configuring project-level deployment settings

From the project view, click the gear icon next to the project name. The settings dropdown offers:

- **Configure Git deployments** — set branch-to-stage mappings, auto-deploy triggers (visible only when a Git repository is connected)
- **Configure Runner** — override the organization-level runner settings for this project
- **Configure GitHub Actions runner** — set up a [self-hosted GitHub Actions runner](/ci-cd-and-gitops/self-hosted-github-actions-runners) for this project (visible only when a Git repository is connected)
- **Delete project** — remove the project from the organization

## Access control

Stacktape uses four roles to control what team members can do within an organization:

| Role | Scope | Key capabilities |
|---|---|---|
| **Owner** | Organization-wide | Full control: delete org, manage billing, manage AWS accounts, all other permissions |
| **Admin** | Organization-wide | Everything except deleting the organization |
| **Developer** | Organization-wide or project-scoped | Deploy, delete non-production stages, manage secrets, manage domains, run scripts |
| **Viewer** | Organization-wide or project-scoped | Read-only access to projects, issues, and observability |

Developer and Viewer roles can be scoped to specific projects — a team member with the Developer role assigned to only two projects cannot see or deploy to others.

Key permission boundaries:

- Only Owners can delete the organization or manage billing
- Only Owners and Admins can invite members, manage AWS accounts, and configure guardrails
- Developers cannot delete production stages (requires `deployments:delete-production`, granted only to Owner and Admin)
- Viewers see project state and logs but cannot modify anything

See [Team and access control](/stacktape-console/team-and-access-control) for inviting members and assigning roles.

## Troubleshooting

### Stage shows as "errored" but deployment succeeded

If a deployment fails mid-way and leaves the CloudFormation stack in a broken state, the Console marks the stage as errored. You can dismiss the failed stage from the stage list (right-click or use the context menu) to clear the error indicator. On the next successful deployment, the stage returns to normal status.

### Undeployed stages clutter the sidebar

Stages created in the Console but never deployed appear in the sidebar as undeployed. You can delete these by clicking the delete icon next to the stage name in the sidebar. This removes only the Console record — no AWS resources are affected because none were created.

### Project card shows $0/mo despite active resources

Cost data comes from AWS Cost and Usage Reports, which can take up to 24 hours to appear. Newly deployed stages may show zero cost until the next billing cycle report is processed.

### Cannot create a project — name validation fails

Project names must be lowercase letters, numbers, and dashes only. No spaces, underscores, or uppercase characters. This constraint exists because the name becomes part of AWS resource identifiers (CloudFormation stack names, S3 bucket prefixes) which have strict character requirements.

## FAQ

### What is the difference between an organization and a project in Stacktape?

An organization is the top-level container for team collaboration, billing, and access control. A project lives inside an organization and represents a single application — usually one Git repository. You can have many projects in one organization, and each project can have multiple stages (deployed environments). Think of organizations as "teams" and projects as "apps".

### Can I move a project between organizations?

Moving projects between organizations is not currently supported in the Console. To relocate a project, create a new project in the target organization with the same Git repository, deploy stages there, and delete the old project and its stages.

### How many stages can a project have?

There is no hard limit on the number of stages per project. Stage names must be 1–10 characters. In practice, most teams run 2–5 stages (production, staging, and a few dev/test stages). Each stage deploys a full CloudFormation stack, so AWS account limits (like IAM roles per account — default 1,000) are the practical ceiling.

### Do stages share anything with each other?

No. Stages are fully isolated. Each stage gets its own databases, functions, containers, networking, and secrets. No data or state leaks between stages. This makes it safe to test destructive changes in a staging environment without any risk to production.

### How does stage naming affect AWS resources?

The stage name becomes part of every AWS resource's identifier. A project named `my-app` with stage `prod` produces a CloudFormation stack named `my-app-prod`. This is why stage names are limited to 10 characters — longer names would exceed AWS naming limits on downstream resources.

### What happens when I delete an organization?

Deleting an organization removes all Console metadata: projects, stage records, team memberships, alert configurations, and budget rules. It does NOT delete deployed AWS resources. You must delete all stages (and their AWS resources) before deleting the organization to avoid orphaned infrastructure that continues incurring costs. Only the Owner role can delete an organization.

### Can two team members deploy to the same stage simultaneously?

Stacktape prevents concurrent deployments to the same stage. If a deployment is already in progress for a stage, a second deployment attempt will be blocked until the first completes. The Console shows an "in progress" indicator on stages with active deployments.

### How do I set up a branch-per-stage workflow?

Connect a Git repository to your project, then configure Git deployments from the project settings. You can map branches to stages (e.g., `main` → `production`, `develop` → `staging`) and enable auto-deploy on push. See [Stacks per Git branch](/ci-cd-and-gitops/stacks-per-git-branch-pattern) for the full pattern.

### What is the default organization created on signup?

When you create a Stacktape account, a personal organization is automatically created using your email address. This default organization works exactly like any other organization — you can connect AWS accounts, create projects, and deploy stages. Create additional organizations when you need to separate work between teams or clients.

### How do roles interact with project-scoped access?

Owner and Admin roles always have organization-wide access and cannot be scoped to specific projects. Developer and Viewer roles can be assigned to the entire organization or restricted to specific projects. A project-scoped Developer can only see and deploy to their assigned projects — other projects are hidden from their Console view entirely.
