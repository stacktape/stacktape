# Using Console UI

The Stacktape Console gives you a complete operations dashboard the moment you deploy a stack. Logs, metrics, alarms, issues, cost breakdown, and secrets management are all available without extra configuration. Every stack you deployed in the previous step is already reporting data here.

<Info>
Already familiar with the Console? Skip ahead to [CI/CD](/getting-started/ci-cd) to set up automated deployments.
</Info>

## The Overview dashboard

The Overview page is the Console's default landing route. After your first deployment, it shows KPI cards for total deployments, average duration, error rate, automation rate, and AWS costs — all with sparkline trends. Below the KPIs, a configurable grid displays deployment frequency charts, recent deployments, open issues, recent alerts, cost breakdowns by project and resource type, budget status, and automation ratio.

Use the time range selector (7d, 14d, 30d, 90d, 6m, 1y) at the top right to adjust the window. Each gadget links through to the relevant detail page, so the Overview acts as your daily check-in surface.

## Logs

Stacktape streams logs from every Lambda function and container workload to CloudWatch. You can live-tail or search historical logs directly from the Console, filtered by project, stage, and resource. The CLI equivalent is [`stacktape debug:logs`](/cli/debug-logs), which supports filter patterns and time ranges from your terminal.

For the full guide on log querying, filtering, and forwarding to external services, see [Observability — Logs](/observability/logs).

## Metrics

CloudWatch metrics for your resources — invocation counts, error rates, duration percentiles, CPU/memory utilization, request counts — are surfaced in the Console with no setup. Stacktape automatically collects metrics from every resource type that emits them (Lambda functions, databases, load balancers, container workloads).

See [Observability — Metrics](/observability/metrics) for details on available metric types and how to query them from the CLI with [`stacktape debug:metrics`](/cli/debug-metrics).

## Alarms

Alarm rules monitor AWS CloudWatch metrics and alert you when thresholds are breached. The Console supports triggers for Lambda error rate, Lambda duration, database CPU/memory/storage/latency/connections, HTTP API Gateway error rate and latency, and Application Load Balancer error rate and unhealthy targets. Each rule is scoped to specific projects and stages, with configurable evaluation periods.

Creating or changing a rule saves it in the Console immediately. The actual CloudWatch alarms and EventBridge notification routing are created or updated on the next deployment of each matching stack.

<ConsoleScreenshot src="/static/screenshots/console-alarm-rules.png" alt="Stacktape Console Alarm Rules page showing a list of alarm rules with name, trigger type, and scoped projects/stages" caption="Alarm rules — define thresholds once, and every matching stack gets a CloudWatch alarm on the next deploy." />

Alarms require [alert channels](/observability/alert-channels) (Slack, Teams, email, Discord, or webhook) to deliver notifications. Set up a channel first, then reference it in your alarm rule. See [Observability — Alarms](/observability/alarms) for the full setup guide.

## Issues (automatic error tracking)

Issues is a built-in error inbox. Stacktape detects runtime errors in your Lambda functions and buildpack containers by scanning log patterns — unhandled exceptions, panics, tracebacks, exit errors, and handler-not-found failures across TypeScript/JavaScript, Python, Go, Java, .NET, Ruby, and PHP. Repeated errors are grouped into a single issue with the error message, stack trace, resource name, occurrence count, and last-seen timestamp.

Enable issue detection for all projects or selected projects from the Issues page. Matching stages need to be deployed with Stacktape CLI 3.8.0 or newer for the CloudWatch Logs subscription filters to be added. Issues can be filtered by status (Open, Resolved, Ignored), project, stage, and search text.

<ConsoleScreenshot src="/static/screenshots/console-issues-page.png" alt="Stacktape Console Issues page showing summary cards (Open/Resolved/Ignored counts), filters, and an issues table with error messages, types, resources, and occurrence counts" caption="The Issues inbox groups repeated runtime errors so you don't search logs by hand." />

Detection is log-pattern based, not a full APM trace. It catches most recurring runtime failures but can miss errors that don't match standard patterns. For most teams, this replaces the need for a separate error-tracking service. See [Observability — Issues](/observability/issues) for configuration details and the CLI commands [`stacktape issues:list`](/cli/issues-list) and [`stacktape issues:resolve`](/cli/issues-resolve).

## Cost dashboards

The Costs page shows your AWS spend broken down by project, stage, and region. Reports are generated daily from AWS Cost and Usage Reports. Select an AWS account and a month to see per-stack costs, then drill into an individual stage for a resource-level breakdown.

The Overview dashboard also surfaces costs by AWS service and by Stacktape resource type, so you can quickly identify which resources drive spend. See [Managing Costs — Dashboards](/managing-costs/dashboards) and [Managing Costs — Budgets](/managing-costs/budgets) to set up spending alerts.

## Secrets management

The Secrets page lets you create, view, update, and delete secrets stored in AWS Secrets Manager. Secrets are scoped to an AWS account and region. Reference them in your Stacktape configuration using the `$Secret()` [directive](/configuration/directives) — they're injected at runtime without appearing in your config file or version control.

```bash
stacktape secret:create --secretName MY_API_KEY --secretValue sk-abc123 --region eu-west-1
```

You can also manage secrets from the CLI with [`stacktape secret:create`](/cli/secret-create), [`stacktape secret:get`](/cli/secret-get), and [`stacktape secret:delete`](/cli/secret-delete). See [Configuration — Secrets](/configuration/secrets) for the full workflow.

## What you got for free

Every feature on this page — the overview dashboard, log streaming, metrics collection, alarm infrastructure, automatic error tracking, cost attribution, and secrets management — was provisioned by the single `stacktape deploy` command you ran in the previous step. There is no separate monitoring stack to configure, no agents to install, and no third-party SaaS to wire up.

| Feature | Where in Console | CLI equivalent |
|---------|-----------------|----------------|
| Logs | Per-stage detail | [`stacktape debug:logs`](/cli/debug-logs) |
| Metrics | Per-stage detail | [`stacktape debug:metrics`](/cli/debug-metrics) |
| Alarms | Monitoring → Alarms | [`stacktape debug:alarms`](/cli/debug-alarms) |
| Issues | Monitoring → Issues | [`stacktape issues:list`](/cli/issues-list) |
| Costs | Organization → Costs | — |
| Secrets | Configuration → Secrets | [`stacktape secret:create`](/cli/secret-create) |
