# Deploying Stacks

The [`stacktape deploy`](/cli/deploy) command packages your application code, generates a CloudFormation template from your configuration, and creates or updates all AWS resources as a single atomic operation. If any resource fails to provision, the entire stack rolls back to its previous working state automatically. Every stage of a project maps to one CloudFormation stack in your AWS account.

## How deployment works

Stacktape deployment runs through five sequential phases. Understanding these phases helps you diagnose slow deployments and interpret the deploy output.

### Validate and prepare

Stacktape parses and validates your configuration, then checks any [guardrails](/guardrails/overview) configured for your organization. If your config references [secrets](/configuration/secrets) via `$Secret()` that don't exist yet, the CLI prompts you to create them — auto-generating database passwords or asking you to enter values for application secrets. Missing SSM parameters are handled the same way.

### Package

Stacktape packages your workloads in parallel. [Lambda functions](/resources/compute/lambda-function) are bundled and packaged, while [container workloads](/resources/compute/web-service) build OCI images from your source or Dockerfile. After packaging, Stacktape resolves all resource dependencies and generates a CloudFormation template that defines every AWS resource your stack needs.

### Upload

Stacktape uploads packaged artifacts to an S3 deployment bucket and pushes container images to ECR repositories.

### Deploy

For new stacks, Stacktape creates the CloudFormation stack. For existing stacks, it updates the stack by applying only the changed resources. CloudFormation handles dependency ordering, parallelizes independent resource creation, and rolls back automatically on failure. If you used the `--hotSwap` flag and only code or container definitions changed, Stacktape bypasses CloudFormation entirely and updates Lambda functions and container workloads directly.

### Post-deploy

After the stack is stable, Stacktape runs any user-defined `afterDeploy` [hooks](/deployment-and-lifecycle/deployment-scripts-and-hooks). The most common use case is running database migrations after infrastructure is ready — for example, executing `npx prisma migrate deploy` as an `afterDeploy` script that connects to the newly provisioned database. See [Deployment scripts and hooks](/deployment-and-lifecycle/deployment-scripts-and-hooks) for configuration details.

Stacktape then syncs [hosting buckets](/resources/frontend/static-hosting) with your local build output, invalidates [CDN](/resources/networking/cdn) caches if needed, saves stack info to disk if configured, and sends deployment notifications to any configured [alert channels](/observability/alert-channels). On the first deploy of a new stack, the CLI also prompts you to set up [CI/CD](/ci-cd-and-gitops/overview).

## Running a deployment

The minimum deploy command requires a stage name and an AWS region:

```bash
stacktape deploy --stage dev --region eu-west-1
```

A **stage** is an isolated copy of your infrastructure — typically one per environment (`dev`, `staging`, `production`) or per developer (`dev-john`). Stage names can be at most 12 characters long. A **region** determines the AWS data center where your resources are created. Choose a region close to your users for lower latency.

Stacktape looks for `stacktape.ts` (or `stacktape.yml`) in the current directory by default. Use `--configPath` to point to a different file.

<Tip>
Use [`stacktape defaults:configure`](/cli/defaults-configure) to set a default stage, region, and AWS profile. After that, you can run `stacktape deploy` without flags during development.
</Tip>

### Key options

| Flag | Description |
| --- | --- |
| `--stage` (`-s`) | **(required)** Stage name — `dev`, `staging`, `production`, etc. Max 12 characters. |
| `--region` (`-r`) | **(required)** AWS region — `eu-west-1`, `us-east-1`, etc. |
| `--configPath` (`-cp`) | Path to config file. Defaults to `stacktape.ts` or `stacktape.yml`. |
| `--hotSwap` (`-hs`) | Fast deploy for code-only changes. Falls back to full deploy if infra changed. |
| `--autoConfirmOperation` (`-aco`) | Skip the confirmation prompt. Required for CI/CD pipelines. |
| `--disableAutoRollback` (`-dar`) | Keep failed CloudFormation state for debugging instead of rolling back. |
| `--noCache` (`-nc`) | Force fresh builds, ignoring cached artifacts. |
| `--preserveTempFiles` (`-ptf`) | Keep generated CloudFormation templates in `.stacktape/`. |
| `--logLevel` (`-ll`) | Set to `debug` for verbose output. |
| `--awsProfile` | AWS profile to use instead of the default. |
| `--projectName` (`-prj`) | Stacktape project name. |

For the full flag reference, see the [`deploy` CLI reference](/cli/deploy).

### Full deployment

Full deployment is the default mode. Stacktape generates a CloudFormation template and lets CloudFormation create, update, or delete resources as needed. CloudFormation handles dependency ordering, parallelizes independent resource operations, and rolls back the entire stack if any resource fails.

```bash
stacktape deploy --stage production --region eu-west-1
```

Use full deployment for production, staging, and any environment where infrastructure consistency matters. Full deploys are the only mode that can add or remove resources, change IAM permissions, update API routes, or modify resource configuration like memory and timeouts.

### Hot-swap deployment

Hot-swap deployment bypasses CloudFormation and updates [Lambda functions](/resources/compute/lambda-function) and [container workloads](/resources/compute/web-service) directly through the AWS APIs. This makes code iterations significantly faster during development.

```bash
stacktape deploy --stage dev --region eu-west-1 --hotSwap
```

When you pass `--hotSwap`, Stacktape analyzes the CloudFormation template diff to determine whether a hot-swap is possible. If Stacktape detects infrastructure changes — new resources, deleted resources, permission changes, configuration changes — it falls back to a full CloudFormation deployment automatically and warns you:

> Hot-swap not possible; running full CloudFormation deploy.

Hot-swap is only available for **updates to existing stacks**. The first deployment of a new stage always runs a full deploy regardless of the flag. Not all Lambda functions in your stack are eligible for hot-swap — edge functions and some framework-internal functions (such as Next.js API route handlers) are excluded and require a full deploy for code changes.

<Warning>
Hot-swap updates resources outside of CloudFormation, which creates state drift between the CloudFormation template and the actual infrastructure. Use hot-swap only for development stages. Always use full deployment for production.
</Warning>

Hot-swap capabilities differ between Lambda functions and container workloads:

**Lambda function hot-swap updates:**
- Function code

**Container workload hot-swap updates:**
- Container images
- Container environment variables
- Port mappings, health checks, secrets, and mount points

**What hot-swap cannot update (requires full deploy):**
- New or deleted resources
- IAM permissions and roles
- API Gateway routes
- Task-level configuration (memory, CPU, timeout)
- Lambda environment variables

### CodeBuild deployment

For large container builds, monorepos, or machines with limited resources, offload the entire build and deploy process to [AWS CodeBuild](/ci-cd-and-gitops/build-runners):

```bash
stacktape codebuild:deploy --stage production --region eu-west-1
```

CodeBuild deployment zips your project files (respecting `.gitignore`), uploads the archive to S3, and starts a CodeBuild build that runs the full deployment inside your AWS account. Logs stream to your terminal. This is useful when local Docker builds are slow, your CI runner is underpowered, or you want to keep large image layers close to ECR for faster pushes.

See the [`codebuild:deploy` CLI reference](/cli/codebuild-deploy) for available flags and configuration details.

## Automatic rollback

By default, Stacktape enables CloudFormation auto-rollback. If any resource fails during a full deployment, CloudFormation reverts the entire stack to its previous working state. For new stacks that fail on first creation, CloudFormation rolls back and deletes the provisioned resources, leaving the stack in a `ROLLBACK_COMPLETE` state. You must delete this empty stack before deploying again — see the [troubleshooting section](#stack-is-in-rollback_complete-state) below.

When auto-rollback is enabled (the default), Stacktape also cleans up deployment artifacts from the rolled-back deployment, so you don't accumulate orphaned build artifacts.

### Debugging failed deployments

To inspect a failed deployment before it rolls back, disable auto-rollback:

```bash
stacktape deploy --stage dev --region eu-west-1 --disableAutoRollback
```

Disabling auto-rollback leaves the stack in an `UPDATE_FAILED` state, which you can inspect in the AWS CloudFormation console or via [`stacktape debug:logs`](/cli/debug-logs). After diagnosing the issue, fix the problem and deploy again — Stacktape includes cleanup logic for deploying over a stack that previously failed an update. You can also roll back manually with [`stacktape rollback`](/cli/rollback).

<Warning>
Don't leave production stacks in a failed state. Fix the issue and redeploy promptly, or roll back manually with [`stacktape rollback`](/cli/rollback).
</Warning>

## Deployment confirmation

Before modifying an existing stack, Stacktape computes the CloudFormation template diff and may prompt you to confirm before proceeding. This helps catch potentially destructive changes — such as resource replacements or deletions — before they happen.

Skip the prompt in non-interactive environments with `--autoConfirmOperation`:

```bash
stacktape deploy --stage production --region eu-west-1 --autoConfirmOperation
```

<Tip>
Always pass `--autoConfirmOperation` in CI/CD pipelines. Without it, the deploy command will hang waiting for input and eventually time out.
</Tip>

## Previewing changes

Before deploying, you can see exactly what would change with [`stacktape preview-changes`](/cli/preview-changes):

```bash
stacktape preview-changes --stage dev --region eu-west-1
```

This generates the CloudFormation template and computes the diff against the currently deployed stack without modifying anything. Use it to verify that your config changes produce the expected infrastructure changes before committing to a deploy. See [Previewing changes](/deployment-and-lifecycle/previewing-changes) for details.

## CloudFormation stack states

Every Stacktape stage maps to a CloudFormation stack. The stack's state determines what operations are available. Understanding these states helps when troubleshooting stuck or failed deployments.

| State | Meaning | What you can do |
| --- | --- | --- |
| `CREATE_COMPLETE` | Stack created and healthy | Deploy updates, delete |
| `UPDATE_COMPLETE` | Last update succeeded | Deploy updates, delete |
| `UPDATE_ROLLBACK_COMPLETE` | Last update failed and rolled back | Deploy again (fix the issue first), delete |
| `UPDATE_FAILED` | Update failed, auto-rollback disabled | Fix and redeploy, or [rollback manually](/cli/rollback) |
| `ROLLBACK_COMPLETE` | Initial creation failed and rolled back | Delete the stack and try again |
| `*_IN_PROGRESS` | Operation running | Wait for it to finish |
| `DELETE_COMPLETE` | Stack deleted | Deploy to create a new stack |

## Deploying from the Stacktape Console

Instead of running `stacktape deploy` from your terminal, you can connect your Git repository to the [Stacktape Console](/stacktape-console/console-overview) and have deployments triggered automatically on push. This GitOps approach means every push to a configured branch triggers a deployment to the mapped stage — for example, pushing to `main` deploys to `production`. Pull requests can create temporary preview stages that are cleaned up when the PR is closed. GitOps-based deployment is a good fit for teams that want automated deployments without maintaining their own CI/CD pipeline. See [GitOps with Console](/ci-cd-and-gitops/gitops-with-console) for setup and configuration.

For teams that already have a CI/CD system (GitHub Actions, GitLab CI, etc.), you can run `stacktape deploy` directly inside your pipeline with an API key. See [Custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) for integration examples.

## Dev mode stacks and deployment

Stacks created with [`stacktape dev`](/cli/dev) (dev mode) cannot be updated with `stacktape deploy`. Dev mode stacks use a lightweight infrastructure configuration optimized for local development, and deploying a full production config on top of them would create conflicts. If you need to move from dev mode to a deployed stage, use a different stage name:

```bash
# Dev mode — runs workloads locally
stacktape dev --stage dev-local --region eu-west-1

# Full deployment — separate stage
stacktape deploy --stage dev --region eu-west-1
```

## Troubleshooting

The most common deploy failures stem from CloudFormation stack state issues, secret configuration gaps, and resource naming limits — each with a specific resolution path.

### Stack is in ROLLBACK_COMPLETE state

This means the initial stack creation failed and CloudFormation rolled back all provisioned resources. The stack itself is an empty shell that blocks the name. Delete it and deploy again:

```bash
stacktape delete --stage dev --region eu-west-1
stacktape deploy --stage dev --region eu-west-1
```

### Stack name too long — resource names obfuscated

The CloudFormation stack name is derived from your project name and stage. If the combined name is too long, Stacktape automatically shortens (obfuscates) individual resource names to stay within AWS naming limits. You'll see a warning during deploy. This doesn't affect functionality, but resource names in the AWS console become less readable. Keep project names and stage names short to avoid this.

### No changes to deploy

CloudFormation detected that your template matches the currently deployed stack. Verify that you saved your config file, that you're deploying the right stage, and that your changes affect Stacktape-managed resources (not raw [CloudFormation overrides](/configuration/overrides-and-escape-hatches), which may require a separate update path).

### Deployment is slow

Common causes of slow deployments:

- **Large container images** — Use multi-stage Docker builds or [Stacktape buildpack packaging](/packaging/containers/stacktape-buildpack) to reduce image size.
- **Many resources** — CloudFormation serializes some dependent resources. This is normal for complex stacks.
- **Slow local builds** — Consider [`codebuild:deploy`](/cli/codebuild-deploy) to offload builds to AWS.
- **First deploy to a new region** — Initial infrastructure setup adds one-time overhead.

Run with `--logLevel debug` to see detailed timing for each phase.

### Secrets prompt blocking CI/CD

If your config references `$Secret()` values that don't exist, the CLI prompts you to create them interactively. In CI/CD, this causes the deploy to hang. Create secrets before deploying:

```bash
stacktape secret:create --name mySecret --stage production --region eu-west-1
```

Or pass `--autoConfirmOperation` to auto-generate database passwords. Application secrets must be created beforehand via [`secret:create`](/cli/secret-create) or the [Stacktape Console](/configuration/secrets).

## FAQ

### How long does a Stacktape deployment take?

Deployment duration depends on the number and type of resources in your stack. A simple Lambda function deploys in under 2 minutes. Stacks with databases, container workloads, and networking typically take 3–10 minutes. Hot-swap deployments for code-only changes complete in seconds. The packaging and upload phases scale with your codebase size and number of container images.

### What happens if a deployment fails midway through?

CloudFormation automatically rolls back the entire stack to its previous working state when auto-rollback is enabled (the default). Resources that were partially created are deleted, and resources that were modified are reverted. You can disable auto-rollback with `--disableAutoRollback` to keep the failed state available for inspection. See [Rollbacks](/deployment-and-lifecycle/rollbacks) for manual rollback options.

### Can I deploy multiple stages from the same config?

Yes. The same `stacktape.ts` config produces independent stacks per stage. Deploy `--stage dev` and `--stage production` from the same config, and they create separate AWS resources with no shared state. Use [directives](/configuration/directives) like `$Stage()` inside your config to vary behavior per stage — for example, smaller instances in dev and larger in production.

### What is the difference between Stacktape deploy and raw CloudFormation?

Stacktape generates a CloudFormation template from your config, but adds a full deployment pipeline around it: code packaging, artifact management, container builds, secret management, automatic CDN invalidation, and hosting bucket syncing. With raw CloudFormation, you manage all of these yourself. Stacktape also provides hot-swap for fast iterations and [CodeBuild offloading](/cli/codebuild-deploy) for remote builds.

### Should I use hot-swap or full deployment?

Use hot-swap (`--hotSwap`) when iterating on Lambda or container code during development. It bypasses CloudFormation and pushes changes directly — Lambda code updates and container definition changes (images, environment variables) take effect in seconds. Use full deployment for any infrastructure changes (new resources, permission updates, config changes) and always for production. Hot-swap creates drift between your CloudFormation state and actual resources, which is acceptable in dev but not in production.

### When should I use CodeBuild deployment?

Use [`codebuild:deploy`](/cli/codebuild-deploy) when your local machine is slow for Docker builds, when building large container images, or when your CI runner has limited CPU/memory. CodeBuild runs the build and deploy in an AWS-hosted environment. It's also useful for monorepos where building multiple container images locally is time-consuming. For simple Lambda deployments, local deploy is usually faster because it avoids the overhead of uploading and starting a CodeBuild environment.

### Does Stacktape deploy require an AWS account?

Yes. Stacktape deploys to your own AWS account. You also need a Stacktape account (free or paid) for authentication and access to the deployment pipeline. Connect your AWS account through the [Stacktape Console](/stacktape-console/connecting-your-aws-account) or use an AWS profile directly with `--awsProfile`.

### How do I deploy from a CI/CD pipeline?

Set the `STACKTAPE_API_KEY` environment variable in your CI runner and pass `--autoConfirmOperation` to skip interactive prompts. Install Stacktape in your pipeline, then run `stacktape deploy`. See [Custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) for GitHub Actions, GitLab CI, and other provider examples. For push-to-deploy without maintaining your own pipeline, use [GitOps with Console](/ci-cd-and-gitops/gitops-with-console).

### Can I deploy to multiple AWS regions?

Yes. Run separate deploy commands with different `--region` flags. Each region gets its own independent stack. For serving users globally, you can deploy the same config to multiple regions and route traffic with DNS. See [Multi-region deployments](/deployment-and-lifecycle/multi-region-deployments) for patterns and tradeoffs.

### What is the difference between first deploy and subsequent deploys?

The first deployment creates the stack and all resources defined in your configuration, along with any internal resources Stacktape needs for artifact management. Subsequent deployments only update what changed — CloudFormation computes the diff and applies the minimum set of changes. Unchanged resources are left untouched. This is why first deploys are typically slower than updates.
