# Deploying Stacks

The [`stacktape deploy`](/cli/deploy) command packages your application code, generates a CloudFormation template from your configuration, and creates or updates all AWS resources as a single atomic operation. If any resource fails to provision, the entire stack rolls back to its previous working state automatically. Every stage of a project maps to one CloudFormation stack in your AWS account.

## How deployment works

Stacktape deployment runs through five sequential phases. Understanding these phases helps you diagnose slow deployments and interpret the deploy output.

### Validate and prepare

Stacktape parses and validates your configuration, then checks any [guardrails](/guardrails/overview) configured for your organization. If your config references [secrets](/configuration/secrets) via `$Secret()` that don't exist yet, the CLI prompts you to create them ã auto-generating database passwords or asking you to enter values for application secrets. Missing SSM parameters are handled the same way.

### Package

All workloads are packaged in parallel. [Lambda functions](/resources/compute/lambda-function) are bundled (tree-shaken, minified, and zipped), while [container workloads](/resources/compute/web-service) build OCI images from your source or Dockerfile. After packaging, Stacktape resolves all resource dependencies and generates a CloudFormation template that defines every AWS resource your stack needs.

### Upload

Packaged artifacts are uploaded to an S3 deployment bucket, and container images are pushed to ECR repositories. Stacktape tracks content digests to skip re-uploads when artifacts haven't changed.

### Deploy

For new stacks, Stacktape creates the CloudFormation stack. For existing stacks, it updates the stack by applying only the changed resources. CloudFormation handles dependency ordering, parallelizes independent resource creation, and rolls back automatically on failure. If you used the `--hotSwap` flag and only code changed, Stacktape bypasses CloudFormation entirely and updates Lambda functions and ECS services directly.

### Post-deploy

After the stack is stable, Stacktape syncs [hosting buckets](/resources/frontend/static-hosting) with your local build output, invalidates [CDN](/resources/networking/cdn) caches if needed, saves stack info to disk if configured, and sends deployment notifications to any configured [alert channels](/observability/alert-channels). On the first deploy of a new stack, the CLI also prompts you to set up [CI/CD](/ci-cd-and-gitops/overview).

## Running a deployment

The minimum deploy command requires a stage name and an AWS region:

```bash
stacktape deploy --stage dev --region eu-west-1
```

A **stage** is an isolated copy of your infrastructure ã typically one per environment (`dev`, `staging`, `production`) or per developer (`dev-john`). Stage names must be lowercase alphanumeric with dashes, at least 2 characters, and at most 12 characters.

A **region** determines the AWS data center where your resources are created. Choose a region close to your users for lower latency.

Stacktape looks for `stacktape.ts` (or `stacktape.yml`) in the current directory by default. Use `--configPath` to point to a different file.

<Tip>
Use [`stacktape defaults:configure`](/cli/defaults-configure) to set a default stage, region, and AWS profile. After that, you can run `stacktape deploy` without flags during development.
</Tip>

### Key options

| Flag | Alias | Description |
| --- | --- | --- |
| `--stage` | `-s` | **(required)** Stage name ã `dev`, `staging`, `production`, etc. |
| `--region` | `-r` | **(required)** AWS region ã `eu-west-1`, `us-east-1`, etc. |
| `--configPath` | `-cp` | Path to config file. Defaults to `stacktape.ts` or `stacktape.yml`. |
| `--hotSwap` | `-hs` | Fast deploy for code-only changes. Falls back to full deploy if infra changed. |
| `--autoConfirmOperation` | `-aco` | Skip the confirmation prompt. Required for CI/CD pipelines. |
| `--disableAutoRollback` | `-dar` | Keep stack in failed state for debugging instead of rolling back. |
| `--noCache` | `-nc` | Force fresh builds, ignoring cached artifacts. |
| `--preserveTempFiles` | `-ptf` | Keep generated CloudFormation templates in `.stacktape/`. |
| `--logLevel` | `-ll` | Set to `debug` for verbose output. |
| `--profile` | `-p` | AWS profile to use instead of the default. |
| `--disableDriftDetection` | `-ddd` | Skip detection of manual changes made outside CloudFormation. |
| `--projectName` | `-prj` | Stacktape project name. |

For the full flag reference, see the [`deploy` CLI reference](/cli/deploy).

### Full deployment

Full deployment is the default mode. Stacktape generates a CloudFormation template and lets CloudFormation create, update, or delete resources as needed. CloudFormation handles dependency ordering, parallelizes independent resource operations, and rolls back the entire stack if any resource fails.

```bash
stacktape deploy --stage production --region eu-west-1
```

Use full deployment for production, staging, and any environment where infrastructure consistency matters. Full deploys are the only mode that can add or remove resources, change IAM permissions, update API routes, or modify resource configuration like memory and timeouts.

### Hot-swap deployment

Hot-swap deployment bypasses CloudFormation and updates [Lambda function](/resources/compute/lambda-function) code and [container workload](/resources/compute/web-service) images directly through the AWS APIs. This makes code iterations significantly faster during development.

```bash
stacktape deploy --stage dev --region eu-west-1 --hotSwap
```

When you pass `--hotSwap`, Stacktape analyzes the CloudFormation template diff to determine whether a hot-swap is possible. Hot-swap works when the only changes are to Lambda function code or ECS task definitions (container images, environment variables). If Stacktape detects infrastructure changes ã new resources, deleted resources, permission changes, configuration changes ã it falls back to a full CloudFormation deployment automatically and warns you:

> Hot-swap not possible; running full CloudFormation deploy.

Hot-swap is only available for **updates to existing stacks**. The first deployment of a new stage always runs a full deploy regardless of the flag.

<Warning>
Hot-swap updates resources outside of CloudFormation, which creates state drift between the CloudFormation template and the actual infrastructure. Use hot-swap only for development stages. Always use full deployment for production.
</Warning>

**What hot-swap can update:**
- Lambda function code (detected by content digest comparison)
- ECS task definitions (container images, environment variables)
- ECS service task references

**What hot-swap cannot update:**
- New or deleted resources
- IAM permissions
- API Gateway routes
- Resource configuration (memory, timeout, instance size)
- CloudFormation conditions, mappings, or parameters

### CodeBuild deployment

For large container builds, monorepos, or machines with limited resources, offload the entire build and deploy process to [AWS CodeBuild](/ci-cd-and-gitops/build-runners):

```bash
stacktape codebuild:deploy --stage production --region eu-west-1
```

CodeBuild deployment works by zipping your git-tracked project files, uploading the archive to S3, and starting a CodeBuild build that runs the full deployment inside AWS. Logs stream back to your terminal in real time. This approach is useful when local Docker builds are slow, your CI runner is underpowered, or you want to keep large image layers close to ECR for faster pushes.

The CodeBuild environment validates your config and resolves resources locally first ã if there's a configuration error, you'll see it immediately rather than waiting for the remote build to fail.

<Info>
CodeBuild deployment accepts the same flags as `deploy`. The confirmation prompt runs locally before the remote build starts, so you won't be blocked waiting for input in CodeBuild. See the [`codebuild:deploy` CLI reference](/cli/codebuild-deploy) for details.
</Info>

## Automatic rollback

By default, Stacktape enables CloudFormation auto-rollback. If any resource fails during a full deployment, CloudFormation reverts the entire stack to its previous working state. For new stacks that fail on first creation, the stack and all its resources are deleted automatically.

Stacktape also cleans up deployment artifacts (S3 objects, ECR images) from rolled-back deployments, so you don't accumulate orphaned build artifacts.

### Debugging failed deployments

To inspect a failed deployment before it rolls back, disable auto-rollback:

```bash
stacktape deploy --stage dev --region eu-west-1 --disableAutoRollback
```

The stack remains in the `UPDATE_FAILED` state, letting you examine which resource failed and why in the AWS CloudFormation Console or via [`stacktape debug:logs`](/cli/debug-logs). After diagnosing the issue, you can either deploy again (Stacktape will continue from the failed state) or manually roll back with [`stacktape rollback`](/cli/rollback).

<Warning>
A stack in `UPDATE_FAILED` state blocks further updates until you either fix and redeploy or roll back. Don't leave production stacks in this state.
</Warning>

## Deployment confirmation

Before modifying an existing stack, Stacktape analyzes the CloudFormation template diff and checks whether any resources would be deleted or replaced. If the diff contains potentially destructive changes, the CLI prompts you to confirm before proceeding. This prompt shows the stack name, creation time, resource count, and a list of resources that may be affected.

Skip the prompt in non-interactive environments with `--autoConfirmOperation`:

```bash
stacktape deploy --stage production --region eu-west-1 --autoConfirmOperation
```

<Tip>
Always pass `--autoConfirmOperation` in CI/CD pipelines. Without it, the deploy command will hang waiting for input and eventually time out.
</Tip>

### Drift detection

By default, Stacktape checks whether resources were manually modified outside of CloudFormation (drift detection). If drift is detected, you'll be warned before the deployment proceeds. Disable drift detection with `--disableDriftDetection` if it slows down your deployment cycle during development, but leave it enabled for production to catch untracked manual changes.

## Previewing changes

Before deploying, you can see exactly what would change with [`stacktape preview-changes`](/cli/preview-changes):

```bash
stacktape preview-changes --stage dev --region eu-west-1
```

This generates the CloudFormation template and computes the diff against the currently deployed stack without modifying anything. Use it to verify that your config changes produce the expected infrastructure changes before committing to a deploy. See [Previewing changes](/deployment-and-lifecycle/previewing-changes) for details.

## CloudFormation stack states

Every Stacktape stage maps to a CloudFormation stack. The stack's state determines what operations are available. Understanding these states helps when troubleshooting stuck or failed deployments.

| State | Meaning | What you can do |
| --- | --- | --- |
| `CREATE_COMPLETE` | Stack created and healthy | Deploy updates, delete |
| `UPDATE_COMPLETE` | Last update succeeded | Deploy updates, delete |
| `UPDATE_ROLLBACK_COMPLETE` | Last update failed and rolled back | Deploy again (fix the issue first), delete |
| `UPDATE_FAILED` | Update failed, auto-rollback disabled | Fix and redeploy, or [rollback manually](/deployment-and-lifecycle/rollbacks) |
| `ROLLBACK_COMPLETE` | Initial creation failed and rolled back | Delete the stack and try again |
| `*_IN_PROGRESS` | Operation running | Wait for it to finish (Stacktape polls automatically) |
| `DELETE_COMPLETE` | Stack deleted | Deploy to create a new stack |

<Info>
Stacktape waits automatically if a stack operation is already in progress. You don't need to manually check the stack state before running a deploy.
</Info>

## Deploying from the Stacktape Console

You can trigger deployments from the [Stacktape Console](/stacktape-console/console-overview) instead of the CLI by connecting your repository for [GitOps](/ci-cd-and-gitops/gitops-with-console). Push-to-deploy, branch-to-stage mapping, and pull request preview environments are all configured through the Console. The underlying deployment runs the same pipeline ã config validation, packaging, CloudFormation ã but executes on Stacktape-managed [build runners](/ci-cd-and-gitops/build-runners) instead of your local machine.

For custom CI/CD setups (GitHub Actions, GitLab CI, etc.), see [Custom CI/CD](/ci-cd-and-gitops/custom-ci-cd).

## Dev mode stacks and deployment

Stacks created with [`stacktape dev`](/cli/dev) (dev mode) cannot be updated with `stacktape deploy`. Dev mode stacks use a lightweight infrastructure configuration optimized for local development, and deploying a full production config on top of them would create conflicts. If you need to move from dev mode to a deployed stage, use a different stage name:

```bash
# Dev mode ã runs workloads locally
stacktape dev --stage dev-local --region eu-west-1

# Full deployment ã separate stage
stacktape deploy --stage dev --region eu-west-1
```

## Troubleshooting

### Stack is in ROLLBACK_COMPLETE state

This means the initial stack creation failed and CloudFormation cleaned up all resources. The stack itself is an empty shell that blocks the name. Delete it and deploy again:

```bash
stacktape delete --stage dev --region eu-west-1
stacktape deploy --stage dev --region eu-west-1
```

### Stack name too long ã resource names obfuscated

The CloudFormation stack name is derived from your project name and stage. If the combined name is too long, Stacktape automatically shortens (obfuscates) individual resource names to stay within AWS naming limits. You'll see a warning during deploy. This doesn't affect functionality, but resource names in the AWS Console become less readable. Keep project names and stage names short to avoid this.

### No changes to deploy

CloudFormation detected that your template matches the currently deployed stack. Verify that you saved your config file, that you're deploying the right stage, and that your changes affect Stacktape-managed resources (not raw [CloudFormation overrides](/configuration/overrides-and-escape-hatches), which may require a separate update path).

### Deployment is slow

Common causes of slow deployments:

- **Large container images** ã Use multi-stage Docker builds or [Stacktape buildpack packaging](/packaging/containers/stacktape-buildpack) to reduce image size.
- **Many resources** ã CloudFormation serializes some dependent resources. This is normal for complex stacks.
- **Slow local builds** ã Consider [`codebuild:deploy`](/cli/codebuild-deploy) to offload builds to AWS.
- **First deploy to a new region** ã Initial infrastructure setup (deployment bucket, ECR repos) adds one-time overhead.

Run with `--logLevel debug` to see detailed timing for each phase.

### Secrets prompt blocking CI/CD

If your config references `$Secret()` values that don't exist, the CLI prompts you to create them interactively. In CI/CD, this causes the deploy to hang. Create secrets before deploying:

```bash
stacktape secret:create --name mySecret --stage production --region eu-west-1
```

Or pass `--autoConfirmOperation` to auto-generate database passwords. Application secrets must be created beforehand via [`secret:create`](/cli/secret-create) or the [Stacktape Console](/configuration/secrets).

## FAQ

### How long does a Stacktape deployment take?

Deployment duration depends on the number and type of resources in your stack. A simple Lambda function deploys in under 2 minutes. Stacks with databases, container workloads, and networking typically take 3ã10 minutes. Hot-swap deployments for code-only changes complete in seconds. The packaging and upload phases scale with your codebase size and number of container images.

### What happens if a deployment fails midway through?

CloudFormation automatically rolls back the entire stack to its previous working state. Resources that were partially created are deleted, and resources that were modified are reverted. Stacktape also cleans up deployment artifacts from the failed attempt. You can disable auto-rollback with `--disableAutoRollback` to inspect the failure before it's cleaned up. See [Rollbacks](/deployment-and-lifecycle/rollbacks) for manual rollback options.

### Can I deploy multiple stages from the same config?

Yes. The same `stacktape.ts` config produces independent stacks per stage. Deploy `--stage dev` and `--stage production` from the same config, and they create separate AWS resources with no shared state. Use [directives](/configuration/directives) like `$Stage()` inside your config to vary behavior per stage (e.g., smaller instances in dev, larger in production).

### What is the difference between Stacktape deploy and raw CloudFormation?

Stacktape generates a CloudFormation template from your config, but adds a full deployment pipeline around it: code packaging, artifact management, container builds, secret management, deployment versioning, automatic CDN invalidation, and hosting bucket syncing. With raw CloudFormation, you manage all of these yourself. Stacktape also handles hot-swap for fast iterations and CodeBuild offloading for remote builds ã neither is available in vanilla CloudFormation workflows.

### Should I use hot-swap or full deployment?

Use hot-swap (`--hotSwap`) when iterating on Lambda or container code during development. It's faster because it bypasses CloudFormation and pushes code changes directly. Use full deployment for any infrastructure changes (new resources, permission updates, config changes) and always for production. Hot-swap creates drift between your CloudFormation state and actual resources, which is acceptable in dev but dangerous in production.

### When should I use CodeBuild deployment?

Use [`codebuild:deploy`](/cli/codebuild-deploy) when your local machine is slow for Docker builds, when building large container images, or when your CI runner has limited CPU/memory. CodeBuild runs the entire build and deploy in an AWS-hosted environment with configurable compute resources. It's also useful for monorepos where building multiple container images locally is time-consuming. For simple Lambda deployments, local deploy is usually faster because it avoids the overhead of zipping, uploading, and starting a CodeBuild environment.

### Does Stacktape deploy require an AWS account?

Yes. Stacktape deploys to your own AWS account. You also need a Stacktape account (free or paid) for authentication and access to the deployment pipeline. Connect your AWS account through the [Stacktape Console](/stacktape-console/connecting-your-aws-account) or use an AWS profile directly with `--profile`.

### How do I deploy from a CI/CD pipeline?

Set the `STACKTAPE_API_KEY` environment variable in your CI runner and pass `--autoConfirmOperation` to skip interactive prompts. Install Stacktape in your pipeline, then run `stacktape deploy`. See [Custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) for GitHub Actions, GitLab CI, and other provider examples. For push-to-deploy without maintaining your own pipeline, use [GitOps with Console](/ci-cd-and-gitops/gitops-with-console).

### Can I deploy to multiple AWS regions?

Yes. Run separate deploy commands with different `--region` flags. Each region gets its own independent stack. For serving users globally, you can deploy the same config to multiple regions and route traffic with DNS. See [Multi-region deployments](/deployment-and-lifecycle/multi-region-deployments) for patterns and tradeoffs.

### What gets deployed on the first run vs subsequent runs?

The first deployment creates all resources from scratch, including the S3 deployment bucket, ECR repositories, IAM roles, and every resource in your config. Subsequent deployments only update what changed ã CloudFormation computes the diff and applies the minimum set of changes. Unchanged resources are left untouched. This is why first deploys are typically slower than updates.
