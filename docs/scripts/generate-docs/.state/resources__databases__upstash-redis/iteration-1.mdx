# Upstash Redis

Upstash Redis is a serverless, pay-per-request Redis database managed by Upstash. Stacktape provisions the database as part of your stack, handling credentials and region selection automatically. Upstash Redis is accessible over both HTTPS (REST API) and the standard Redis protocol with TLS, making it a natural fit for [Lambda functions](/resources/compute/lambda-function) and other serverless workloads where a persistent connection isn't practical.

## When to use

Upstash Redis is the right choice when your workload is serverless and bursty. Use it for caching, session storage, rate limiting, or lightweight pub/sub in Lambda-based applications. Because pricing is per-request with no idle costs, it avoids the minimum monthly spend of a traditional [Redis cluster](/resources/databases/redis).

Choose Upstash Redis when:

- Your compute is primarily [Lambda functions](/resources/compute/lambda-function) — the HTTPS REST API avoids connection-pooling headaches.
- Traffic is unpredictable or low-volume — you pay only for the commands you run, not for idle hours.
- You want zero operational overhead — no instance sizing, no patching, no replication configuration.
- You need a Redis-compatible store accessible from outside a VPC without bastion tunneling.

## When NOT to use

Upstash Redis is not a drop-in replacement for every Redis workload. Prefer a managed [Redis cluster](/resources/databases/redis) (ElastiCache) when:

- You need sub-millisecond latency at high throughput — ElastiCache runs inside your VPC with no HTTPS overhead.
- Your workload is container-based ([web services](/resources/compute/web-service), [worker services](/resources/compute/worker-service)) with persistent connections — the per-request pricing of Upstash becomes expensive at sustained high request rates.
- You rely on Redis features that Upstash restricts on the serverless tier (e.g. Lua scripting, certain cluster commands).
- You need data residency guarantees in a region Upstash does not support.

| Criteria | Upstash Redis | Redis cluster (ElastiCache) |
|---|---|---|
| Pricing model | Per-request, no idle cost | Per-hour per-node, fixed monthly minimum |
| Connection model | HTTPS REST or `rediss://` | Standard Redis protocol (TCP) |
| VPC required | No | Yes |
| Latency | Low (over HTTPS/TLS) | Sub-millisecond (in-VPC) |
| Operational overhead | None | Instance sizing, patching, failover config |
| Best for | Serverless / bursty workloads | High-throughput container workloads |

## Provider configuration

Upstash Redis requires an Upstash account. You must provide your account email and API key in the `providerConfig` section of your Stacktape config. Create an API key in the [Upstash console](https://console.upstash.com) under Account > API Keys.

Store the API key as a [secret](/configuration/secrets) to keep it out of version control.

## Eviction

By default, Upstash Redis does not evict keys — writes fail once the database reaches its memory limit. For caching workloads where losing old data is acceptable, enable eviction. When enabled, Upstash prioritizes evicting keys that have a TTL set before removing other keys (volatile-LRU policy).

Enable eviction when the database is used as a cache or for transient data (sessions, rate-limit counters). Leave it disabled when every key matters and you want writes to fail loudly rather than silently discard data.

## Connecting from workloads

Use [`connectTo`](/configuration/connecting-resources) to give a compute resource access to your Upstash Redis database. Stacktape injects environment variables with connection details into the consuming workload.

For a resource named `cache`, the following environment variables are injected:

| Environment variable | Description |
|---|---|
| `STP_CACHE_HOST` | Redis host endpoint |
| `STP_CACHE_PORT` | Redis port |
| `STP_CACHE_PASSWORD` | Database password |
| `STP_CACHE_REST_TOKEN` | Token for the Upstash REST API |
| `STP_CACHE_REST_URL` | Full HTTPS URL for the REST API |
| `STP_CACHE_REDIS_URL` | Full `rediss://` connection URL (includes password, host, and port) |

The variable names follow the pattern `STP_[RESOURCE_NAME]_[PARAM]`, where the resource name is converted to SCREAMING_SNAKE_CASE.

### Using the REST API

The Upstash REST API lets you run Redis commands over HTTPS — no persistent connection or Redis client library required. This is the recommended approach for Lambda functions.

```typescript
// Lambda handler using Upstash REST API
export const handler = async () => {
  const restUrl = process.env.STP_CACHE_REST_URL;
  const restToken = process.env.STP_CACHE_REST_TOKEN;

  // SET a key
  await fetch(`${restUrl}/set/greeting/hello`, {
    headers: { Authorization: `Bearer ${restToken}` }
  });

  // GET a key
  const response = await fetch(`${restUrl}/get/greeting`, {
    headers: { Authorization: `Bearer ${restToken}` }
  });

  const data = await response.json();
  return { statusCode: 200, body: JSON.stringify(data) };
};
```

You can also use the `@upstash/redis` npm package, which wraps the REST API with a familiar Redis-like interface:

```typescript
import { Redis } from '@upstash/redis';

const redis = new Redis({
  url: process.env.STP_CACHE_REST_URL!,
  token: process.env.STP_CACHE_REST_TOKEN!
});

export const handler = async () => {
  await redis.set('greeting', 'hello');
  const value = await redis.get('greeting');
  return { statusCode: 200, body: JSON.stringify({ value }) };
};
```

### Using the Redis protocol

Upstash Redis also supports the standard Redis protocol over TLS (`rediss://`). Use the `STP_CACHE_REDIS_URL` environment variable with any Redis client library:

```typescript
import { Redis } from 'ioredis';

const redis = new Redis(process.env.STP_CACHE_REDIS_URL!);

export const handler = async () => {
  await redis.set('key', 'value');
  const result = await redis.get('key');
  return { statusCode: 200, body: JSON.stringify({ result }) };
};
```

<Tip>

For Lambda functions, prefer the REST API or `@upstash/redis` over a traditional Redis client. Traditional clients open persistent TCP connections that don't map well to Lambda's ephemeral execution model and can exhaust connection limits under high concurrency.

</Tip>

## Region selection

Stacktape automatically selects the closest available Upstash region to your stack's deployment region. Upstash Redis is available in the following regions:

| AWS region | Location |
|---|---|
| `us-east-1` | US East (N. Virginia) |
| `us-west-1` | US West (N. California) |
| `us-west-2` | US West (Oregon) |
| `eu-west-1` | Europe (Ireland) |
| `eu-central-1` | Europe (Frankfurt) |
| `ap-southeast-1` | Asia Pacific (Singapore) |
| `ap-southeast-2` | Asia Pacific (Sydney) |
| `sa-east-1` | South America (São Paulo) |

If your stack is deployed to a region not in this list, Stacktape picks the geographically closest available region. You do not need to configure the region manually.

## FAQ

### When should I use Upstash Redis vs a Redis cluster (ElastiCache)?

Use Upstash Redis for serverless and bursty workloads — primarily Lambda functions — where you want pay-per-request pricing and zero operational overhead. Use a [Redis cluster](/resources/databases/redis) (ElastiCache) when your workload is container-based with sustained high throughput, needs sub-millisecond in-VPC latency, or requires advanced Redis features like Lua scripting. The break-even point depends on volume: at thousands of commands per second sustained, ElastiCache's per-hour pricing becomes cheaper.

### What does Upstash Redis cost?

Upstash uses a pay-per-request pricing model. You pay per command executed, with no minimum monthly charge and no cost when idle. Upstash offers a free tier with limited daily commands and storage. Paid plans scale based on throughput and storage usage. Check the [Upstash pricing page](https://upstash.com/pricing) for current rates.

### How do I connect to Upstash Redis from a Lambda function?

Add the Upstash Redis resource name to your Lambda function's [`connectTo`](/configuration/connecting-resources) list. Stacktape injects environment variables (`STP_[NAME]_REST_URL`, `STP_[NAME]_REST_TOKEN`, etc.) into your function. Use the `@upstash/redis` npm package or raw `fetch()` calls against the REST URL — both avoid the connection-pooling issues that traditional Redis clients have in Lambda.

### Does Upstash Redis support the standard Redis protocol?

Yes. Upstash Redis is accessible over both HTTPS (REST API) and the standard Redis protocol with TLS (`rediss://`). The `redisUrl` referenceable parameter provides a full connection URL you can use with any Redis client library like `ioredis` or `redis`. For Lambda functions, the REST API is generally a better fit because it avoids persistent TCP connections.

### Can I use Upstash Redis for rate limiting?

Yes. Upstash Redis supports atomic operations like `INCR` and `EXPIRE`, which are the building blocks of most rate-limiting algorithms (fixed window, sliding window, token bucket). The `@upstash/ratelimit` npm package provides a ready-made implementation designed for serverless environments.

### What environment variables does connectTo inject for Upstash Redis?

When you add an Upstash Redis resource to a workload's `connectTo`, Stacktape injects six environment variables: `HOST`, `PORT`, `PASSWORD`, `REST_TOKEN`, `REST_URL`, and `REDIS_URL`, each prefixed with `STP_[RESOURCE_NAME]_`. For example, a resource named `cache` produces `STP_CACHE_HOST`, `STP_CACHE_REST_URL`, and so on.

### How do I store the Upstash API key securely?

Use the [`$Secret()` directive](/configuration/directives) to reference a secret stored in AWS Secrets Manager. Create the secret with `stacktape secret:create`, then reference it in your provider config as `apiKey: '$Secret(upstash-api-key)'`. The API key never appears in plaintext in your config file or CloudFormation template.

### Does Upstash Redis run inside my VPC?

No. Upstash Redis is a fully managed third-party service that runs outside your AWS VPC. Communication happens over HTTPS or TLS-encrypted Redis protocol over the public internet. This means Lambda functions do not need VPC access to reach Upstash Redis, which avoids cold-start penalties associated with VPC-attached Lambdas.

### What happens if I deploy to a region Upstash doesn't support?

Stacktape automatically selects the geographically closest available Upstash region. For example, deploying your stack to `ap-northeast-1` (Tokyo) would route to `ap-southeast-1` (Singapore). This adds some cross-region latency but requires no manual configuration.
