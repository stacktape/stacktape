# Triggers Overview

Triggers define what causes a Stacktape resource to execute. A Lambda function sitting idle costs nothing — a trigger
connects it to an event source (an HTTP request, a file upload, a queue message, a schedule) so it runs when work
arrives. Triggers are configured in the `events` property of Lambda functions, batch jobs, and container workloads.

## Why triggers exist

Every compute resource in Stacktape needs at least one reason to run. Without a trigger, a [Lambda function](/resources/compute/lambda-function) has no way to receive traffic, and a [batch job](/resources/compute/batch-job) has no way to start. Triggers solve this by declaring the binding between an event source and your compute — Stacktape handles all underlying IAM permissions, event-source mappings, and routing rules automatically.

You configure triggers once in your `events` array, and Stacktape provisions the AWS plumbing: API Gateway integrations, SQS event-source mappings, S3 notification configurations, EventBridge rules, or load balancer listener rules depending on the trigger type.

## Trigger categories

Stacktape triggers fall into two broad categories based on how they deliver work to your compute resources.

### Synchronous triggers

Synchronous triggers deliver a request and wait for a response. The caller blocks until your function or container returns a result. Use these for APIs, webhooks, and any workflow where the caller needs an immediate answer.

| Trigger | Type value | Works with | Dedicated page |
|---------|-----------|------------|----------------|
| HTTP API Gateway | `http-api-gateway` | Lambda, batch jobs, containers | [HTTP triggers](/configuration/triggers/http-triggers) |
| Application Load Balancer | `application-load-balancer` | Lambda, batch jobs, containers | [HTTP triggers](/configuration/triggers/http-triggers) |
| Network Load Balancer | `network-load-balancer` | Containers only | [HTTP triggers](/configuration/triggers/http-triggers) |

### Asynchronous triggers

Asynchronous triggers deliver events without waiting for a response. Your function processes the event and the caller moves on. Use these for background processing, reactions to state changes, and scheduled jobs.

| Trigger | Type value | Works with | Dedicated page |
|---------|-----------|------------|----------------|
| Schedule (cron/rate) | `schedule` | Lambda, batch jobs | [Schedule triggers](/configuration/triggers/schedule-triggers) |
| S3 bucket events | `s3` | Lambda, batch jobs | [S3 events](/configuration/triggers/s3-events) |
| SQS queue | `sqs` | Lambda, batch jobs | [SQS events](/configuration/triggers/sqs-events) |
| SNS topic | `sns` | Lambda, batch jobs | [SNS events](/configuration/triggers/sns-events) |
| DynamoDB stream | `dynamo-db-stream` | Lambda, batch jobs | [DynamoDB streams](/configuration/triggers/dynamodb-streams) |
| Kinesis stream | `kinesis-stream` | Lambda, batch jobs | [Kinesis events](/configuration/triggers/kinesis-events) |
| EventBridge event bus | `event-bus` | Lambda, batch jobs | [Event bus events](/configuration/triggers/event-bus-events) |
| CloudWatch Logs | `cloudwatch-log` | Lambda | [CloudWatch logs](/configuration/triggers/cloudwatch-logs) |
| CloudWatch Alarm | `cloudwatch-alarm` | Lambda | [Alarms as triggers](/configuration/triggers/alarms-as-triggers) |
| Kafka topic | `kafka-topic` | Lambda | [Kafka topics](/configuration/triggers/kafka-topics) |

### Container-internal triggers

Container workloads ([web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [multi-container workloads](/resources/compute/multi-container-workload)) support two additional trigger types for intra-workload and intra-stack communication:

| Trigger | Type value | Purpose |
|---------|-----------|---------|
| Workload Internal | `workload-internal` | Opens a container port for communication between containers in the same workload |
| Service Connect | `service-connect` | Opens a container port for service-to-service discovery across your stack |

## How triggers work

When you define a trigger in the `events` array, Stacktape translates it into the appropriate AWS construct during deployment:

- **HTTP triggers** create API Gateway routes or load balancer listener rules that forward matching requests to your function or container.
- **Poll-based triggers** (SQS, Kinesis, DynamoDB streams, Kafka) create AWS Lambda event-source mappings that poll the source and invoke your function with batches of records.
- **Push-based triggers** (S3, SNS, CloudWatch Logs, schedules, EventBridge, alarms) configure the source service to invoke your function directly when an event occurs.

For poll-based triggers, you control batching behavior with `batchSize` and `maxBatchWindowSeconds`. The function waits until either the batch size is reached or the time window expires, then fires with a batch of records. This lets you balance latency against efficiency.

## Batching and throughput

Poll-based triggers (SQS, Kinesis, DynamoDB streams, Kafka) deliver records in batches. Two properties control batch behavior:

- **`batchSize`** — maximum records per invocation. Higher values mean fewer invocations but larger payloads. Maximums vary by source: 10,000 for SQS and Kinesis, 1,000 for DynamoDB, 10,000 for Kafka.
- **`maxBatchWindowSeconds`** — maximum seconds to wait before invoking. Caps at 300 seconds for all sources. Reduces invocation count when traffic is bursty. If omitted on SQS, the function fires as soon as messages are available.

Your function receives the full batch as a single event. If processing fails, the entire batch is retried (for stream-based sources). Design your handlers to be idempotent.

## Error handling and retries

Stream-based triggers (Kinesis, DynamoDB streams) retry failed batches automatically. You control retry behavior with:

- **`maximumRetryAttempts`** — how many times to retry a failed batch before giving up.
- **`onFailure`** — an SQS queue or SNS topic that receives batches that exhaust all retries (a dead-letter destination).
- **`bisectBatchOnFunctionError`** — splits a failed batch in half before retrying, useful when a single bad record poisons a large batch.
- **`parallelizationFactor`** — process multiple batches from the same shard concurrently.

For push-based triggers (SNS, EventBridge), you can configure `onDeliveryFailure` to route undeliverable events to an SQS queue.

<Warning>
For stream-based triggers (Kinesis, DynamoDB), a failed batch blocks the shard until retries are exhausted or the batch succeeds. Always set `maximumRetryAttempts` and `onFailure` to avoid stalled processing.
</Warning>

## Multiple triggers on one resource

A single Lambda function or batch job can have multiple triggers of any supported type. Common patterns:

- An API handler that also runs on a schedule for warm-keeping or maintenance tasks.
- A function that processes uploads from multiple S3 buckets with different filter rules.
- A function that consumes from both an SQS queue and an EventBridge rule.

Each trigger operates independently — Stacktape configures separate event-source mappings and permissions for each entry in the `events` array.

## FAQ

### Can a single function have multiple triggers?

Yes. The `events` property accepts an array, and you can mix trigger types freely. A Lambda function can simultaneously serve HTTP requests via API Gateway, process SQS messages, and run on a schedule. Each trigger gets its own IAM permissions and event-source mapping configured automatically by Stacktape.

### What is the difference between SQS, SNS, and EventBridge triggers?

SQS is point-to-point: one message, one consumer. Use it for work queues and decoupling producers from consumers. SNS is fan-out: one message delivered to multiple subscribers simultaneously. EventBridge adds content-based routing with pattern matching on event fields — use it when you need to route different events to different targets based on payload content. See [SQS events](/configuration/triggers/sqs-events), [SNS events](/configuration/triggers/sns-events), and [Event bus events](/configuration/triggers/event-bus-events).

### Why can't container workloads use queue or stream triggers?

Container workloads (web services, private services, worker services) are always-on processes that manage their own event loop. They receive traffic through load balancers or API Gateway. If you need a container to process queue messages, have the container poll the queue directly using the AWS SDK and [connect to the queue](/configuration/connecting-resources) for permissions. For event-driven processing without managing a polling loop, use a Lambda function instead.

### How does Stacktape handle permissions for triggers?

Stacktape automatically configures all IAM permissions each trigger requires. For poll-based sources (SQS, Kinesis, DynamoDB), it grants your function's execution role permission to read from the source. For push-based sources (S3, SNS), it grants the source service permission to invoke your function. You do not need to manage any of this manually.

### What happens if my function fails while processing a batch?

Behavior depends on the trigger type. For stream-based sources (Kinesis, DynamoDB), the entire batch — including successfully processed records — is retried. Your function must be idempotent. Set `maximumRetryAttempts` to limit retries and `onFailure` to route exhausted batches to a dead-letter queue. For SQS, individual failed messages return to the queue after the visibility timeout expires; successfully processed messages are deleted. Use `bisectBatchOnFunctionError` on stream sources to isolate bad records.

### When should I use HTTP API Gateway versus an Application Load Balancer?

Use an [HTTP API Gateway](/resources/networking/http-api-gateway) for most serverless APIs — it is cheaper (per-request pricing with no idle cost), simpler to configure, and handles path/method routing automatically. Use an [Application Load Balancer](/resources/networking/application-load-balancer) when you need advanced routing (header-based, query-parameter-based, source IP filtering), when you already have an ALB for container workloads, or when you need persistent connections like WebSockets.

### Can I trigger a batch job on a schedule?

Yes. Batch jobs support the `schedule` trigger type with the same `rate()` and `cron()` expressions available to Lambda functions. Stacktape creates a helper Lambda function behind the scenes that starts the batch job when the schedule fires. See [Schedule triggers](/configuration/triggers/schedule-triggers).

### What is the maximum batch size for event-source triggers?

Maximum batch sizes vary by source: SQS supports up to 10,000 records per batch, Kinesis supports up to 10,000, DynamoDB streams support up to 1,000, and Kafka supports up to 10,000. The default batch size is 10 for SQS, Kinesis, and Kafka; 100 for DynamoDB streams and Kafka (when using `maxBatchWindowSeconds`). Choose based on your function's processing time and memory — larger batches are more efficient but require more memory and risk longer retries on failure.

### How do I filter which events trigger my function?

Most trigger types support filtering. S3 triggers accept `prefix` and `suffix` filters on object keys. SNS supports `filterPolicy` for attribute-based filtering. EventBridge uses `eventPattern` for content-based routing on any event field. CloudWatch Logs accepts a `filter` pattern. For SQS and Kinesis, filtering happens in your handler code — all messages are delivered and you discard irrelevant ones.
