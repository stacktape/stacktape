# Batch Job

A batch job runs a container to completion and exits — for data processing, ML training, video encoding, ETL pipelines, and any task with a defined start and end. Stacktape provisions compute matching your CPU, memory, and GPU requirements. You pay only for the time the job runs.

**Pricing**: Costs depend on CPU, memory, and GPU requested and the duration of each run — no charges between runs. Spot instances (`useSpotInstances`) can reduce costs by up to 90% for interruptible workloads.

## When to use

**Use a batch job when:**

- Your task has a defined start and end — it processes data and then stops
- You need GPU compute (ML training, inference, video encoding)
- Your task runs longer than a [Lambda function](/resources/compute/lambda-function)'s timeout limit
- You want automatic retries with configurable backoff on failure
- You need container-level control over the runtime environment

**Use something else when:**

| Situation | Better option |
|-----------|---------------|
| Event-driven request/response handlers | [Lambda function](/resources/compute/lambda-function) — pay per invocation, scales to zero |
| Always-on API or web app | [Web service](/resources/compute/web-service) — persistent container with a public URL |
| Continuous background processing | [Worker service](/resources/compute/worker-service) — always-on container without a public URL |

## Basic example

This configures a batch job that processes data using a Node.js script:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const processor = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/process.ts',
      }),
    },
    resources: {
      cpu: 2,
      memory: 4096,
    },
  });

  return {
    resources: { processor },
  };
});
// [!code focus-end]`
    }
  ]}
/>

```typescript
// src/process.ts
async function main() {
  const data = await fetchData();
  const results = await processData(data);
  await saveResults(results);
  console.info('Job complete');
}

main().catch((err) => {
  console.error('Job failed:', err);
  process.exit(1);
});
```

Your container code runs to completion and exits — a zero exit code means success, anything else means failure. When a batch job is triggered, the trigger event payload is available in the `STP_TRIGGER_EVENT_DATA` environment variable.

The Stacktape image buildpack builds a container image from your source code automatically. For other packaging options (custom Dockerfile, prebuilt image, Nixpacks, external buildpack), see [container packaging](/packaging/containers/).

## Resources

The `resources` property sets CPU, memory, and optional GPU for the job. By default, Stacktape lets AWS select compute based on those requirements.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const processor = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/process.ts',
      }),
    },
    // [!code focus-start]
    resources: {
      cpu: 4,
      memory: 7680,
    },
    // [!code focus-end]
  });

  return {
    resources: { processor },
  };
});`
    }
  ]}
/>

<Tip>AWS reserves some memory on each instance for system processes. Request slightly less than a power of 2 — for example, use `7680` instead of `8192` to fit on a standard 8 GB instance without over-provisioning.</Tip>

Set `gpu` to run on a GPU-capable instance such as NVIDIA A100 or A10G. Omit `gpu` for CPU-only workloads.

## Environment variables

Use `container.environment` to inject custom environment variables into your job. Values can be static strings, `$ResourceParam()` references, or `$Secret()` directives.

<CodeBlock
  intellisense
  tabs={[
    {

      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const processor = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/process.ts',
      }),
      // [!code focus-start]
      environment: [
        { name: 'BATCH_SIZE', value: '1000' },
        { name: 'API_KEY', value: '$Secret(my-api-key)' },
      ],
      // [!code focus-end]
    },
    resources: {
      cpu: 2,
      memory: 4096,
    },
  });

  return {
    resources: { processor },
  };
});`
    }
  ]}
/>

Variables from `connectTo` (e.g., `STP_MY_DATABASE_CONNECTION_STRING`) are added automatically alongside any custom variables you define here.

## Spot instances

Enable `useSpotInstances` to use spare AWS capacity at up to 90% discount. If AWS reclaims the capacity, your container receives a `SIGTERM` signal and has 120 seconds to shut down gracefully before being terminated.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const importer = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/import.ts',
      }),
    },
    resources: {
      cpu: 2,
      memory: 4096,
    },
    // [!code focus-start]
    useSpotInstances: true,
    retryConfig: {
      attempts: 3,
      retryIntervalSeconds: 30,
    },
    // [!code focus-end]
  });

  return {
    resources: { importer },
  };
});`
    }
  ]}
/>

**Use Spot when** your job can safely restart from the beginning — data imports, image processing, ML training with checkpoints.

**Avoid Spot when** your job has non-repeatable side effects (sending emails, charging payments) or must finish within a strict deadline.

<Tip>Always pair `useSpotInstances` with `retryConfig` so interrupted jobs are automatically resubmitted.</Tip>

## Retries

Configure `retryConfig` to automatically retry jobs on failure, timeout, or Spot interruption.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const etl = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/etl.ts',
      }),
    },
    resources: {
      cpu: 2,
      memory: 4096,
    },
    // [!code focus-start]
    retryConfig: {
      attempts: 5,
      retryIntervalSeconds: 10,
      retryIntervalMultiplier: 2,
    },
    // [!code focus-end]
  });

  return {
    resources: { etl },
  };
});`
    }
  ]}
/>

Wait times increase with each attempt. Each wait = `retryIntervalSeconds` × attempt number × `retryIntervalMultiplier`. With the config above, waits between retries are 20s, 40s, 60s, 80s. Defaults: `attempts: 1`, `retryIntervalSeconds: 0`, `retryIntervalMultiplier: 1`.

Your container receives `STP_CURRENT_ATTEMPT` and `STP_MAXIMUM_ATTEMPTS` environment variables so you can implement checkpoint/resume logic. Make your jobs idempotent so retries produce the same result.

## Timeout

Set `timeout` to limit how long a job can run, in seconds. If the job exceeds this limit, it's terminated. If `retryConfig` is configured, the job is retried automatically after a timeout.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const etl = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/etl.ts',
      }),
    },
    resources: {
      cpu: 2,
      memory: 4096,
    },
    // [!code focus-start]
    timeout: 3600,
    // [!code focus-end]
    retryConfig: {
      attempts: 3,
    },
  });

  return {
    resources: { etl },
  };
});`
    }
  ]}
/>

When `timeout` is omitted, the job runs until it completes or fails on its own. Use `timeout` for safety on jobs where runaway execution could waste resources.

## Triggers

Batch jobs are started by events. Use the `events` property to configure what triggers the job — schedules, HTTP requests, S3 uploads, queue messages, and more.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging, HttpApiGateway } from 'stacktape';

export default defineConfig(() => {
  const gateway = new HttpApiGateway({});

  const report = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/generate-report.ts',
      }),
    },
    resources: {
      cpu: 2,
      memory: 4096,
    },
    // [!code focus-start]
    events: [
      {
        type: 'schedule',
        properties: {
          scheduleRate: 'rate(1 day)',
        },
      },
      {
        type: 'http-api-gateway',
        properties: {
          httpApiGatewayName: 'gateway',
          method: 'POST',
          path: '/reports/generate',
        },
      },
    ],
    // [!code focus-end]
  });

  return {
    resources: { gateway, report },
  };
});`
    }
  ]}
/>

Supported trigger types:

| Trigger | Use case |
|---------|----------|
| `schedule` | Periodic jobs — daily reports, hourly data syncs |
| `http-api-gateway` | On-demand jobs triggered by API calls |
| `s3` | Process files on upload (images, CSVs, videos) |
| `sqs` | Process queue messages |
| `sns` | React to published notifications |
| `kinesis-stream` | Process streaming data |
| `dynamo-db-stream` | React to database changes |
| `cloudwatch-log` | Process log events |
| `application-load-balancer` | HTTP-triggered jobs via an Application Load Balancer |
| `event-bus` | React to EventBridge events |

The trigger event payload is available in your container as the `STP_TRIGGER_EVENT_DATA` environment variable.

## Resource connections

Use `connectTo` to grant your batch job access to databases, buckets, queues, and other resources in your stack. Stacktape grants the relevant IAM permissions, opens network access where applicable, and injects resource parameters as environment variables.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, StacktapeImageBuildpackPackaging, RelationalDatabase } from 'stacktape';

export default defineConfig(() => {
  const db = new RelationalDatabase({
    engine: {
      type: 'postgres',
      properties: { version: '16.4' },
    },
    sizing: {
      type: 'serverless-v2',
      properties: { minCapacity: 0.5, maxCapacity: 2 },
    },
  });

  const migrator = new BatchJob({
    container: {
      packaging: new StacktapeImageBuildpackPackaging({
        entryfilePath: './src/migrate.ts',
      }),
    },
    resources: {
      cpu: 1,
      memory: 2048,
    },
    // [!code focus-start]
    connectTo: ['db'],
    // [!code focus-end]
  });

  return {
    resources: { db, migrator },
  };
});`
    }
  ]}
/>

Environment variables are injected in the format `STP_[RESOURCE_NAME]_[PARAM]`. For example, connecting to a relational database named `db` injects `STP_DB_CONNECTION_STRING`, `STP_DB_HOST`, `STP_DB_PORT`, and more.

For the full list of injected variables per resource type, see [connecting resources](/configuration/connecting-resources).

For permissions not covered by `connectTo` (e.g., AWS Rekognition, Textract, Bedrock), use `iamRoleStatements` to add raw IAM policy statements.

## Logging

Container output (`stdout`/`stderr`) is automatically sent to CloudWatch Logs and retained for 90 days by default. View logs with `stacktape logs`. Adjust retention with the `logging.retentionDays` property, or disable logging entirely with `logging.disabled: true`.

Batch job logging also supports forwarding logs to external destinations such as Datadog, Highlight.io, or any HTTP endpoint via the `logging.logForwarding` property.

## Dev mode

Batch jobs support [dev mode](/local-development/dev-mode) (`stacktape dev`), letting you iterate on your job code locally without redeploying to AWS.

## FAQ

### How do Spot instances work with batch jobs?

Enable `useSpotInstances` to run on spare AWS capacity at up to 90% discount. If AWS reclaims the capacity, your container receives `SIGTERM` and has 120 seconds to shut down. Pair with `retryConfig` so interrupted jobs are automatically resubmitted.

### What triggers can start a batch job?

Batch jobs support schedules (cron/rate), HTTP API Gateway requests, S3 events, SQS messages, SNS notifications, Kinesis streams, DynamoDB streams, CloudWatch log events, Application Load Balancer requests, and EventBridge events.

### How do I connect a batch job to a database?

Use `connectTo` to reference a database resource by name. Stacktape grants IAM permissions, opens network access, and injects connection details as environment variables (e.g., `STP_MY_DB_CONNECTION_STRING`).

### How do I retry a failed batch job?

Configure `retryConfig` with `attempts`, `retryIntervalSeconds`, and `retryIntervalMultiplier`. Jobs retry on failure, timeout, or Spot interruption. Make your job idempotent so retries produce the same result.

### What GPU options are available?

Set the `gpu` property to run on GPU-capable instances such as NVIDIA A100 or A10G. AWS selects the appropriate instance based on your CPU, memory, and GPU requirements.
