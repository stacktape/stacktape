# Batch Job

A Stacktape batch job runs one containerized task to completion for data processing, ML training, video encoding, imports, and similar work. Use a batch job when the workload has a clear end, may run for a long time, needs CPU, memory, or GPU sizing, and should start from a schedule, event, queue message, or HTTP request.

**Pricing context:** Batch jobs use Fargate-style vCPU and memory pricing while they run: approximately ~$7.29/month per 0.25 vCPU and ~$1.60/month per 512 MB when running continuously in us-east-1. Actual cost depends on job duration, resource size, and run frequency.

## When to use

A Stacktape batch job is the right choice when work should start, run, and finish instead of staying online as a service. Common use cases:

- **Scheduled processing** — nightly imports, report generation, cleanup tasks, and periodic data syncs triggered with a schedule
- **File-driven processing** — video transcoding, image processing, or document parsing triggered by S3 object events
- **Event-driven jobs** — heavier work started by SQS, SNS, Kinesis, DynamoDB streams, CloudWatch logs, EventBridge, HTTP API Gateway, or an Application Load Balancer
- **Compute-heavy jobs** — CPU- or GPU-sized work where `resources.cpu`, `resources.memory`, and optionally `resources.gpu` describe the job requirements
- **Retryable tasks** — workloads that can safely run again after failure, timeout, or Spot interruption

## When NOT to use

- **Always-on HTTP services** — use [web-service](/resources/compute/web-service) when a container should continuously serve public HTTPS traffic.
- **Internal always-on services** — use [private-service](/resources/compute/private-service) when other resources need a stable VPC-only endpoint.
- **Long-running background consumers** — use [worker-service](/resources/compute/worker-service) when a process should stay alive and continuously poll or process work.
- **Brief event handlers** — use a [Lambda function](/resources/compute/lambda-function) for brief event handlers that fit the Lambda runtime model. Use a batch job for containerized work, larger CPU or memory requirements, GPU workloads, or jobs modeled as run-to-completion containers.
- **Non-idempotent side effects without safeguards** — avoid automatic retries and Spot capacity for jobs that charge cards, send emails, or mutate external systems in ways that cannot be safely repeated.

## Basic example

This example runs a containerized export job every night. The `resources` block asks for 2 vCPUs and 3840 MB of memory; the memory value is intentionally below 4096 MB because the source recommends using slightly less than powers of 2 to avoid over-provisioning when AWS reserves memory for system processes.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, CustomDockerfilePackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const nightlyExport = new BatchJob({
    container: {
      packaging: new CustomDockerfilePackaging({
        buildContextPath: './jobs/nightly-export',
        dockerfilePath: './Dockerfile',
        command: ['node', 'dist/export.js']
      }),
      environment: [{ name: 'EXPORT_FORMAT', value: 'parquet' }]
    },
    resources: {
      cpu: 2,
      memory: 3840
    },
    timeout: 3600,
    retryConfig: {
      attempts: 3,
      retryIntervalSeconds: 30,
      retryIntervalMultiplier: 2
    },
    events: [
      {
        type: 'schedule',
        properties: {
          scheduleRate: 'cron(0 2 * * ? *)'
        }
      }
    ]
  });

  return {
    resources: { nightlyExport }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`timeout` is the maximum runtime in seconds; when the job exceeds it, Stacktape kills the job and retries it if `retryConfig` is set. `retryIntervalMultiplier: 2` turns the retry delay into exponential backoff, so a 30-second interval becomes 30 seconds, 60 seconds, 120 seconds, and so on. AWS cron expressions in `scheduleRate` use UTC, so convert local business times before scheduling production jobs.

## Container

The `container` block defines the image and runtime environment for a Stacktape batch job. A batch job has one container, unlike a [multi-container-workload](/resources/compute/multi-container-workload). Use `environment` for static job settings, and prefer [`$Secret()`](/configuration/secrets), [`$ResourceParam()`](/configuration/referenceable-parameters), or `connectTo` for credentials and resource connection details.

Batch jobs support five container packaging modes: [Stacktape image buildpack](/packaging/containers/stacktape-buildpack), [custom Dockerfile](/packaging/containers/custom-dockerfile), [prebuilt image](/packaging/containers/prebuilt-image), [Nixpacks](/packaging/containers/nixpacks), and [external buildpack](/packaging/containers/external-buildpack). Most teams should start with the Stacktape image buildpack; use a custom Dockerfile or prebuilt image when the runtime environment is already standardized outside Stacktape.

| Mode | When to use |
|------|-------------|
| [Stacktape image buildpack](/packaging/containers/stacktape-buildpack) | Build an image from source with minimal container configuration |
| [Custom Dockerfile](/packaging/containers/custom-dockerfile) | Control the operating system packages, build steps, runtime image, and startup command |
| [Prebuilt image](/packaging/containers/prebuilt-image) | Run an image that already exists in a registry and optionally override its command |
| [Nixpacks](/packaging/containers/nixpacks) | Let Nixpacks detect the app and build an image from a source directory |
| [External buildpack](/packaging/containers/external-buildpack) | Use Cloud Native Buildpacks with optional builder or buildpack selection |

## Compute resources

The `resources` block tells Stacktape how much compute the batch job needs. `cpu` and `memory` are required, while `gpu` is optional. Stacktape lets AWS select matching capacity from those requirements, so most batch-job configs should describe CPU, memory, and optional GPU needs instead of choosing infrastructure directly.

Use more CPU for parallel processing, compression, rendering, or model inference that can use multiple cores. Use more memory for large in-memory transforms, batch aggregation, model loading, and jobs that process large files. Use `gpu` only for jobs that need GPU-backed compute, such as ML training or GPU-accelerated inference.

## Triggers

BatchJobProps accepts integration shapes used for schedules, HTTP API Gateway, S3, SNS, SQS, Kinesis, DynamoDB streams, CloudWatch logs, Application Load Balancer, and EventBridge events. Choose the trigger based on how work enters your system, then keep the batch-job handler idempotent because event retries can happen at the trigger layer or job layer.

Use a schedule for periodic work, S3 for file-driven processing, SQS for job queues, EventBridge for domain events, and HTTP API Gateway only when callers should directly start a job. For the detailed shape of each event source, see the [triggers overview](/configuration/triggers/overview) and the dedicated trigger pages.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, Bucket, CustomDockerfilePackaging } from 'stacktape';

export default defineConfig(() => {
  const uploads = new Bucket({});

  const processor = new BatchJob({
    container: {
      packaging: new CustomDockerfilePackaging({
        buildContextPath: './jobs/process-upload',
        dockerfilePath: './Dockerfile'
      })
    },
    resources: { cpu: 2, memory: 3840 },
    // [!code focus-start]
    events: [
      {
        type: 's3',
        properties: {
          bucketArn: '$ResourceParam(uploads, arn)',
          s3EventType: 's3:ObjectCreated:*',
          filterRule: {
            prefix: 'incoming/',
            suffix: '.csv'
          }
        }
      }
    ]
    // [!code focus-end]
  });

  return { resources: { uploads, processor } };
});`
    }
  ]}
/>

`s3EventType: 's3:ObjectCreated:*'` starts the job for object-created events. `filterRule.prefix` and `filterRule.suffix` narrow the trigger to relevant objects, which is usually better than starting expensive batch work for every file in a bucket.

## Retries and timeouts

Batch-job retries are useful when failures are transient, interruptions are expected, or the job can safely be repeated from the beginning. `retryConfig.attempts` defaults to 1, `retryIntervalSeconds` defaults to 0, and `retryIntervalMultiplier` defaults to 1 when the properties are omitted.

Set `timeout` when a stuck job should be terminated instead of running indefinitely. If the timeout is exceeded, Stacktape kills the job and retries it when retry configuration is present. Keep retries conservative for jobs that mutate external systems, and make job handlers idempotent before increasing retry attempts.

<CodeBlock

    },
    resources: { cpu: 4, memory: 7680 },

    timeout: 7200,
    retryConfig: {

      retryIntervalMultiplier: 2

/>

`PrebuiltImagePackaging` uses an existing container image, and the batch-job packaging type supports a `command` override when the image's Dockerfile `CMD` should not be used. Use prebuilt images when your CI pipeline already builds and publishes immutable images; use Stacktape-managed image builds when you want deployment to build the image from source.

## Spot capacity

`useSpotInstances` tells Stacktape to use discounted spare AWS capacity for a batch job. The source documents savings up to 90%, but Spot jobs can be interrupted; when interruption happens, the container receives `SIGTERM` and has 120 seconds to shut down gracefully.

Enable Spot for restartable jobs such as imports, image processing, and ML training with checkpoints. Keep regular capacity for strict deadlines, irreversible side effects, or jobs that cannot tolerate being restarted from the beginning. Combining Spot with `retryConfig` is the practical default when the job is idempotent.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, NixpacksPackaging } from 'stacktape';

export default defineConfig(() => {
  const imageResizer = new BatchJob({
    container: {
      packaging: new NixpacksPackaging({
        sourceDirectoryPath: './jobs/image-resizer',
        startCmd: 'node dist/resize.js'
      })
    },
    resources: { cpu: 2, memory: 3840 },
    retryConfig: { attempts: 5, retryIntervalSeconds: 30 },
    // [!code focus-start]
    useSpotInstances: true
    // [!code focus-end]
  });

  return { resources: { imageResizer } };
});`
    }
  ]}
/>

`useSpotInstances` defaults to `false`, so omitting it keeps the job on regular capacity. In this example, `NixpacksPackaging` builds from `sourceDirectoryPath`, and `startCmd` overrides the command Nixpacks would otherwise infer.

<Warning>Spot interruption is not a graceful deploy signal. Treat interrupted jobs as failed attempts, persist checkpoints outside the container, and design the handler so running the same input twice does not corrupt data.</Warning>

## Connecting to other resources

Use `connectTo` on a batch job when the job needs access to other Stacktape resources. Stacktape uses `connectTo` to grant modeled access and inject environment variables such as bucket names, bucket ARNs, queue URLs, topic ARNs, database connection details, and BatchJob reference parameters documented in [connecting resources](/configuration/connecting-resources).

Prefer `connectTo` for Stacktape-managed resources because the relationship is readable and Stacktape can add the corresponding permissions or connection details. Use `iamRoleStatements` for AWS permissions that are not modeled as Stacktape resource connections, such as direct access to services like Textract, Rekognition, or Bedrock.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, BatchJob, Bucket, CustomDockerfilePackaging } from 'stacktape';

export default defineConfig(() => {
  const exportsBucket = new Bucket({});

  const reportJob = new BatchJob({
    // [!code focus-start]
    connectTo: ['exportsBucket'],
    iamRoleStatements: [
      {
        Action: ['textract:StartDocumentAnalysis'],
        Resource: ['*']
      }
    ],
    // [!code focus-end]
    container: {
      packaging: new CustomDockerfilePackaging({
        buildContextPath: './jobs/build-report',
        dockerfilePath: './Dockerfile'
      })
    },
    resources: { cpu: 2, memory: 3840 }
  });

  return { resources: { exportsBucket, reportJob } };
});`
    }
  ]}
/>

Because the example resource is named `exportsBucket`, injected variables follow the `STP_EXPORTS_BUCKET_*` pattern, including bucket `NAME` and `ARN`. The `iamRoleStatements` example adds a separate IAM policy statement alongside the generated permissions; keep these statements narrow in real stacks instead of using broad resources unless the AWS API requires it.

## Logging

Batch-job logging controls container `stdout` and `stderr` sent to CloudWatch. `logging.disabled` defaults to `false`, and `logging.retentionDays` defaults to 90. Supported retention values include short windows such as 1, 3, and 7 days as well as long windows up to 3653 days.

Keep logging enabled for production jobs because failures often happen after the triggering request has already returned. Shorten retention for noisy, frequent jobs to control log storage cost. Disable logging only for low-value jobs where the loss of debugging history is an acceptable tradeoff. The source documents `stacktape logs` as the command for viewing container logs.

`retentionDays: 30` keeps one month of logs instead of the default 90 days. Use a shorter value for high-volume jobs whose logs are only useful during immediate incident response, and keep the default when failed job investigation often happens days or weeks later.

## FAQ

### What is a Stacktape batch job?

A Stacktape batch job is a containerized task that runs to completion instead of staying online as a service. The BatchJob resource is designed for data processing, ML training, video encoding, imports, and similar work that has a beginning and an end. Use [web-service](/resources/compute/web-service) or [worker-service](/resources/compute/worker-service) for always-on containers.

### How do I trigger a Stacktape batch job?

A Stacktape batch job accepts `schedule`, `http-api-gateway`, `s3`, `sns`, `sqs`, `kinesis-stream`, `dynamo-db-stream`, `cloudwatch-log`, `application-load-balancer`, and `event-bus` event integrations. Pick the trigger that matches how work enters your system: schedules for periodic work, S3 for file events, SQS for queued jobs, and EventBridge for domain events. See [triggers overview](/configuration/triggers/overview).

### Can a Stacktape batch job use a GPU?

Yes. Set `resources.gpu` when the batch job needs GPU-backed compute, and omit `gpu` for CPU-only work. The active BatchJob resource shape exposes CPU, memory, and optional GPU requirements, then Stacktape lets AWS match capacity.

### How do retries work for Stacktape batch jobs?

`retryConfig` controls automatic retries after failure, timeout, or Spot interruption. `attempts` defaults to 1, `retryIntervalSeconds` defaults to 0, and `retryIntervalMultiplier` defaults to 1. Increase retries only for idempotent jobs that can safely run the same input again.

### Should I use Spot capacity for a Stacktape batch job?

Use `useSpotInstances: true` when the batch job can be interrupted and restarted from the beginning, such as image processing or ML training with checkpoints. The source documents savings up to 90%, but interrupted containers receive `SIGTERM` and only 120 seconds to shut down gracefully. Keep regular capacity for strict deadlines or non-repeatable side effects.

### How much does a Stacktape batch job cost?

The provided pricing source maps `batch-job` to Fargate-style vCPU and memory billing while the job runs. It lists approximately ~$7.29/month per 0.25 vCPU and ~$1.60/month per 512 MB when running continuously in us-east-1, with ARM around 20% cheaper where supported. For cost visibility, use [per-resource breakdown](/managing-costs/per-resource-breakdown).

### Batch job vs Lambda function: which should I use?

Use a [Lambda function](/resources/compute/lambda-function) for brief event handlers that fit the Lambda runtime model. Use a batch job for containerized work, larger CPU or memory requirements, GPU workloads, or jobs that are better modeled as a run-to-completion container. Batch jobs are usually the clearer fit for heavy data or media processing.

### Batch job vs worker service: which should I use?

Use a [worker-service](/resources/compute/worker-service) when a container should remain running and continuously process background work. Use a batch job when each unit of work should start a fresh container run, complete, and stop. Batch jobs are a cleaner fit for scheduled jobs, event-triggered processing, and workloads where idle containers would waste cost.

### Where do Stacktape batch job logs go?

Batch-job `stdout` and `stderr` are controlled by the `logging` block and sent to CloudWatch unless `logging.disabled` is set. `retentionDays` defaults to 90 and can be set to one of the supported CloudWatch retention values. The BatchJob source documents `stacktape logs` as the command for viewing container logs.

### Can another resource reference a batch job?

Resources and scripts that expose `connectTo` can connect to a BatchJob target. The source documents BatchJob reference values `JOB_DEFINITION_ARN` and `STATE_MACHINE_ARN`; the BatchJob resource also exposes `jobDefinitionArn`, `stateMachineArn`, and `logGroupArn` as referenceable parameters. See [connecting resources](/configuration/connecting-resources) and [referenceable parameters](/configuration/referenceable-parameters).
