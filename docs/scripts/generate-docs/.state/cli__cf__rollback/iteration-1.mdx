# cf:rollback

The `cf:rollback` command recovers a CloudFormation stack stuck in a failed state (`UPDATE_FAILED`, `CREATE_FAILED`, or `UPDATE_ROLLBACK_FAILED`) by triggering CloudFormation's native rollback mechanism. Use it when a deployment fails mid-way and leaves your stack unable to accept new updates.

## Usage

```bash
stacktape cf:rollback --region eu-west-1 --stage production
```

## What it does

When a Stacktape deployment fails, CloudFormation may leave the stack in one of several broken states:

- **`UPDATE_FAILED`** or **`CREATE_FAILED`** — the update or creation was interrupted. `cf:rollback` initiates a rollback to the last stable configuration.
- **`UPDATE_ROLLBACK_FAILED`** — a previous rollback itself failed. `cf:rollback` calls CloudFormation's `ContinueUpdateRollback` API to retry the rollback, optionally skipping resources that cannot be restored.

After rolling back the stack, the command also cleans up deployment artifacts that were uploaded for the failed deployment.

<Info>
This command does NOT roll back to a previous deployment version. It restores the stack to its last successfully deployed state as recorded by CloudFormation. To roll back to an earlier version of your configuration, use [`rollback`](/cli/rollback) instead.
</Info>

## When to use cf:rollback vs rollback

| Scenario | Command |
|----------|---------|
| Stack stuck in `UPDATE_FAILED` or `UPDATE_ROLLBACK_FAILED` | `cf:rollback` |
| Want to revert to a specific previous deployment version | [`rollback`](/cli/rollback) |
| Stack is healthy but you want to undo the last deploy | [`rollback`](/cli/rollback) |

Use `cf:rollback` as a recovery tool when your stack is broken. Use [`rollback`](/cli/rollback) as a deliberate version revert when your stack is healthy but you want to go back to an earlier configuration.

## Important flags

### --resourcesToSkip

When a rollback itself fails (`UPDATE_ROLLBACK_FAILED` state), specific resources may be preventing the rollback from completing — for example, a resource that was manually deleted or depends on an external service that changed. Pass their logical resource IDs to skip them during the retry.

```bash
stacktape cf:rollback --region eu-west-1 --stage production --resourcesToSkip MyCustomResource AnotherBrokenResource
```

<Warning>
Skipped resources remain in an inconsistent state. After a successful rollback with skipped resources, redeploy your stack to bring everything back in sync.
</Warning>

## Examples

Recover a stuck stack in eu-west-1:

```bash
stacktape cf:rollback --region eu-west-1 --stage production --projectName my-api
```

Retry a failed rollback, skipping a resource that cannot be restored:

```bash
stacktape cf:rollback --region us-east-1 --stage staging --resourcesToSkip BrokenLambdaPermission
```
