# Referenceable Parameters

The `$ResourceParam` directive lets you reference dynamic outputs from one resource inside another resource's configuration — URLs, ARNs, hostnames, ports, and connection strings that are only known after deployment. This page lists every available parameter, grouped by resource type.

## Syntax

`$ResourceParam` takes two arguments: the resource name (as defined in your config) and the parameter name.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  HttpApiGateway
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    environment: [
      {
        name: 'GATEWAY_URL',
        value: "$ResourceParam('gateway', 'url')"
      }
    ]
  });

  const gateway = new HttpApiGateway({});

  return {
    resources: { api, gateway }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The directive resolves at deploy time. The resulting value is injected as a string — suitable for environment variables, script inputs, or anywhere a string value is accepted in your config.

<Info>

Some parameters are only available when specific features are enabled. For example, CDN parameters require a `cdn` configuration, and reader parameters on databases require Aurora engines or read replicas. Conditions are noted in each parameter's description.

</Info>

## Using parameters in practice

### Environment variables

The most common use of `$ResourceParam` is injecting connection details into workloads via environment variables. Use it when `connectTo` doesn't cover the specific parameter you need, or when you need to pass a value to a resource that `connectTo` doesn't support.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  RelationalDatabase,
  RdsEnginePostgresql
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const mainDb = new RelationalDatabase({
    engine: new RdsEnginePostgresql({ version: '16' }),
    instanceSize: 'db.t4g.micro'
  });

  const migrator = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/migrate.ts'
    }),
    environment: [
      {
        name: 'DB_HOST',
        value: "$ResourceParam('mainDb', 'host')"
      },
      {
        name: 'DB_PORT',
        value: "$ResourceParam('mainDb', 'port')"
      },
      {
        name: 'DB_NAME',
        value: "$ResourceParam('mainDb', 'dbName')"
      }
    ],
    connectTo: ['mainDb']
  });

  return {
    resources: { mainDb, migrator }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

### Difference from connectTo

[`connectTo`](/configuration/connecting-resources) is the preferred way to wire resources together — it grants IAM permissions, opens network access, and injects environment variables automatically. Use `$ResourceParam` when you need:

- A parameter that `connectTo` doesn't auto-inject (e.g., `dbName`, `logGroupArn`, `archiveArn`)
- To pass a value into a non-environment context (stack outputs, other resource properties)
- To reference a parameter from a resource type that `connectTo` doesn't cover

## FAQ

### What is $ResourceParam used for in Stacktape?

`$ResourceParam` is a [directive](/configuration/directives) that resolves a named output from any resource in your stack at deploy time. It returns values like URLs, ARNs, hostnames, and connection strings that are only known after AWS provisions the resource. Use it in environment variables, stack outputs, or any config property that accepts a string.

### How is $ResourceParam different from connectTo?

[`connectTo`](/configuration/connecting-resources) is a higher-level abstraction that grants IAM permissions, opens security group rules, and injects a predefined set of environment variables automatically. `$ResourceParam` gives you access to individual parameters without side effects — no permissions are granted and no security groups are modified. Use `connectTo` for standard access patterns; use `$ResourceParam` for specific values or non-standard wiring.

### Can I use $ResourceParam to reference resources in other stacks?

No. `$ResourceParam` resolves parameters within the same stack only. To reference values from another stack, use the `$CfStackOutput` directive, which reads exported CloudFormation outputs from a different stack. The source stack must explicitly export the value with `export: true` in its stack outputs.

### When are $ResourceParam values resolved?

Values are resolved at deploy time when Stacktape compiles your configuration into a CloudFormation template. The directive produces a CloudFormation intrinsic function (like `Fn::GetAtt` or `Fn::Sub`) that AWS resolves during stack creation or update. During [dev mode](/local-development/dev-mode-overview), values are resolved from the deployed stack's outputs.

### What happens if I reference a parameter that doesn't exist?

Stacktape validates `$ResourceParam` references before deployment. If you reference a resource name that doesn't exist in your config, or a parameter name that isn't supported for that resource type, the deployment fails immediately with an error listing the valid parameters for that resource.

### Are CDN parameters always available?

CDN parameters (`cdnDomain`, `cdnUrl`, `cdnCustomDomains`, `cdnCustomDomainUrls`) are only available on resources that have a `cdn` configuration block. If you haven't configured a [CDN](/resources/networking/cdn) on the resource, referencing these parameters causes a deployment error.

### Which database parameters require Aurora engines?

The `readerHost`, `readerConnectionString`, and `readerJdbcConnectionString` parameters are only available on [relational databases](/resources/databases/relational-database) using Aurora PostgreSQL or Aurora MySQL engines. Standard RDS engines (single-instance Postgres, MySQL, MariaDB) expose `host`, `port`, `connectionString`, and `jdbcConnectionString`. Read replica parameters (`readReplicaHosts`, `readReplicaConnectionStrings`) require explicitly configured read replicas.

### Can I use $ResourceParam in hooks and scripts?

Yes. Scripts defined in the `scripts` section and [lifecycle hooks](/configuration/hooks-and-scripts) resolve `$ResourceParam` from the deployed stack's stored outputs. The referenced resource must already be deployed — if you reference a resource in an `afterDeploy` hook, it's available because deployment completed. For `beforeDeploy` hooks, only resources from a previous deployment are accessible.
