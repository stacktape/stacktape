# Security and Data Protection Guardrails

Stacktape security guardrails are preventive policies that block deployments violating your team's data-protection and network-security standards. They enforce requirements like VPC-only database access, deletion protection, dead-letter queues, WAF on load balancers, custom domains, and approved database engines — catching misconfigurations before they reach production.

## How security guardrails work

Security guardrails are preventive policy definitions — they validate resource configurations against your security requirements and block non-compliant deployments. This is fundamentally different from [alarms](/observability/alarms), which reactively notify you when a running system behaves unexpectedly at runtime.

Each guardrail has a `type` and a `properties` object. Security guardrails use either an `enabled: true` toggle (for `require-*` guardrails) or an allowlist (for restriction guardrails like `database-engine-restriction`). When a deployment includes a resource that violates an enabled guardrail, the deployment is rejected with an error identifying the offending resource.

| Aspect | Guardrails | Alarms |
|--------|-----------|--------|
| **Purpose** | Prevent non-compliant configurations | Detect runtime anomalies |
| **Timing** | Deployment time (preventive) | Runtime (reactive) |
| **Action** | Blocks deployment | Sends notification |
| **Scope** | Policy across resources | Per-resource monitoring |

## When to use

Enable security guardrails when your team needs to enforce compliance requirements without relying on code reviews alone to catch misconfigurations. They are most valuable for:

- **Multi-developer teams** where anyone can deploy independently — guardrails prevent a developer from accidentally deploying an internet-exposed database or a queue without a dead-letter queue.
- **Compliance-regulated organizations** (SOC 2, HIPAA, PCI-DSS) that mandate specific network isolation, data-protection, or encryption controls as baseline requirements.
- **Platform teams** providing Stacktape to internal customers who shouldn't need to memorize every security baseline to deploy safely.

## When NOT to use

Skip guardrails in early prototyping or single-developer projects where speed matters more than governance. Guardrails add a validation step to every deployment — for a solo developer who understands the tradeoffs, they slow iteration without adding safety. You can always enable guardrails later when the team grows or the project reaches production.

## Require VPC databases

The `require-vpc-databases` guardrail ensures all databases use VPC-only accessibility with no public internet access. When enabled, any database configured with internet-facing accessibility is blocked from deploying.

**When to enable:** Enable in production environments or any organization handling sensitive data. VPC-only databases cannot be reached from the public internet, eliminating an entire class of network-based attack vectors. The tradeoff is that you need a [bastion host](/resources/security/bastion-host) or VPC-connected workloads to access the database for administration and debugging. For most production workloads, VPC-only is the right default.

**When the default is fine:** During early development or prototyping, internet-accessible databases are more convenient for local development and ad-hoc queries. The risk is low when the database holds no real data.

A compliant [relational database](/resources/databases/relational-database) configuration:

The `accessibilityMode` property controls how the database is reachable. Setting it to `'vpc'` restricts access to resources within the same VPC. With this guardrail enabled, any database not using VPC-only accessibility causes the deployment to be rejected.

## Require deletion protection

The `require-deletion-protection` guardrail requires all [relational databases](/resources/databases/relational-database) to have `deletionProtection` set to `true`. Deletion protection is an AWS safeguard that prevents a database instance from being deleted through normal API operations — an explicit disable step is required before removal.

**When to enable:** Enable for any stage where accidental data loss is unacceptable — production, staging with real data, or shared development databases. The tradeoff is minimal: you must explicitly disable deletion protection before intentionally removing a database, which adds one deliberate step to prevent accidents.

**When the default is fine:** Skip for throwaway development stages where the database is recreated frequently and holds no important data.

<CodeBlock
  intellisense

  const mainDatabase = new RelationalDatabase({
    engine: new RdsEnginePostgres16({

  };
});
// [!code focus-end]`
    }

With `deletionProtection: true`, AWS rejects any API call that would delete the database instance. This protects against accidental deletion during stack teardowns or configuration changes. The `instanceSize` property controls the compute and memory capacity of the database — `db.t4g.micro` is a cost-effective choice for development and low-traffic production workloads. See the [relational database page](/resources/databases/relational-database) for a full list of available instance sizes.

## Require dead-letter queue

The `require-dead-letter-queue` guardrail ensures every [SQS queue](/resources/messaging/sqs-queue) in the stack has a `redrivePolicy` (dead-letter queue) configured. A dead-letter queue captures messages that fail processing repeatedly, preventing silent data loss in message-driven architectures.

**When to enable:** Enable when your system processes messages representing real business events — orders, payments, notifications — where losing a message means losing data. Dead-letter queues give you a second chance to inspect and reprocess failed messages instead of having them silently disappear after exhausting retries.

**When the default is fine:** Skip for ephemeral or best-effort queues where occasional message loss is acceptable, such as analytics event buffers or non-critical notification streams.

A compliant SQS queue configuration:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, SqsQueue } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const deadLetterQueue = new SqsQueue({});

  const orderQueue = new SqsQueue({
    redrivePolicy: {
      deadLetterQueue: 'deadLetterQueue',
      maxReceiveCount: 3
    }
  });

  return {
    resources: { deadLetterQueue, orderQueue }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The `redrivePolicy` specifies which queue receives failed messages (`deadLetterQueue`) and how many receive attempts are allowed before a message is moved there (`maxReceiveCount`). A `maxReceiveCount` of 3–5 is typical — low enough to catch persistent failures quickly, high enough to tolerate transient errors. Monitor your dead-letter queue with [alarms](/observability/alarms) to get notified when messages land there.

## Require WAF

The `require-waf` guardrail requires all [application load balancers](/resources/networking/application-load-balancer) to have a [web application firewall](/resources/security/web-application-firewall) attached. AWS WAF inspects incoming HTTP/HTTPS requests and blocks those matching configured rules, protecting against common web exploits like SQL injection, cross-site scripting (XSS), and bot traffic.

**When to enable:** Enable for public-facing applications that handle user input, authentication, or sensitive data. WAF provides protection against OWASP top-10 threats without requiring application-level code changes. Most production APIs benefit from WAF.

**When the default is fine:** Skip for internal-only load balancers, development stages, or stages behind a separate security layer (like a third-party API gateway with its own WAF). Also consider skipping for very low-traffic applications where the per-request WAF cost isn't justified.

**Cost tradeoff:** AWS WAF charges per rule group and per million requests evaluated. The exact cost depends on the rules you configure and your traffic volume — check AWS WAF pricing for current rates. For high-traffic production APIs, WAF is a standard best practice. For low-traffic internal tools, the cost may not be justified.

## Require custom domain

The `require-custom-domain` guardrail ensures public-facing [web services](/resources/compute/web-service) and [hosting buckets](/resources/frontend/static-hosting) have a [custom domain](/resources/networking/custom-domains) configured. This prevents production resources from being served on auto-generated AWS URLs that expose implementation details.

**When to enable:** Enable for production stages where URL stability, branding, and control over the domain matter. Custom domains give your users a predictable, professional endpoint that you fully control.

**When the default is fine:** Skip for development and staging stages where custom domains add DNS propagation delay and certificate provisioning time without adding value. Auto-generated URLs are perfectly adequate for internal testing.

## Database engine restriction

The `database-engine-restriction` guardrail limits which database engine types can be used. You define an `allowedEngines` list — for example, `["postgres", "aurora-postgresql"]` — and any database using an engine not on the list is blocked from deploying.

**When to enable:** Enable when your organization needs to standardize on specific database engines for security, compliance, or operational reasons:

- **Block deprecated or insecure engines** — prevent teams from deploying older engine versions that have known vulnerabilities or are approaching end-of-life.
- **Enforce approved engines** — restrict deployments to engines your DBA team actively supports and has runbooks for. This reduces operational risk when incidents happen at 2 AM.
- **Meet compliance requirements** — some compliance frameworks mandate specific database technologies or encryption capabilities only available on certain engines.

**When the default is fine:** Skip when your team is small enough that engine choice is a conversation rather than a policy, or when you're exploring different database technologies and don't want to constrain options prematurely. This guardrail is also covered on the [database guardrails](/guardrails/databases) page, which includes the related `database-instance-restriction` guardrail for controlling instance sizes.

## Resource type restriction (security use)

The `resource-type-restriction` guardrail blocks specified resource types from being deployed. While covered in detail on the [deployment guardrails](/guardrails/deployment) page, it has a clear security application: blocking resource types that don't meet your organization's compliance or operational requirements.

For example, you might block resource types that are difficult to secure in your environment, that your security team hasn't reviewed, or that expose unnecessary attack surface. The `blockedResourceTypes` property accepts a list of Stacktape resource type identifiers — for example, `["open-search-domain", "redis-cluster"]` as shown in the source type definition.

## Combining security guardrails

Most production organizations enable multiple security guardrails together. A typical production-security baseline:

| Guardrail | Purpose | Recommended for |
|-----------|---------|-----------------|
| `require-vpc-databases` | Network isolation for databases | All production stages |
| `require-deletion-protection` | Prevent accidental data loss | All stages with persistent data |
| `require-dead-letter-queue` | Message durability | Event-driven architectures |
| `require-waf` | Application-layer protection | Public-facing APIs |
| `require-custom-domain` | URL stability and branding | Production web services |
| `database-engine-restriction` | Engine standardization | Compliance-regulated teams |
| `resource-type-restriction` | Block non-compliant resource types | Security-sensitive environments |

These guardrails complement each other. VPC databases prevent network-level exposure. WAF prevents application-level attacks. Deletion protection and dead-letter queues prevent data loss. Custom domains ensure stable, branded endpoints. Engine restrictions enforce your approved technology list.

<Tip>
Start with `require-vpc-databases` and `require-deletion-protection` — they provide the highest security value with the least friction. Add `require-waf`, `require-dead-letter-queue`, and `require-custom-domain` as your production security posture matures.
</Tip>

## Related guardrails

This page covers security and data-protection guardrails. Stacktape includes additional guardrail types for deployment governance and resource limits:

- **Deployment guardrails** — `stage-restriction`, `region-restriction`, `command-restriction`, and `resource-type-restriction` control where and how stacks can be deployed. See [deployment guardrails](/guardrails/deployment).
- **Resource limit guardrails** — `function-memory-limit`, `function-timeout-limit`, `container-resource-limit`, and `resource-count-limit` cap the size and count of compute resources. See [resource limit guardrails](/guardrails/resource-limits).
- **Database guardrails** — `database-engine-restriction` and `database-instance-restriction` control which database engines and instance sizes are allowed. See [database guardrails](/guardrails/databases).

For a high-level overview of all guardrail types, see the [guardrails overview](/guardrails/overview).

## FAQ

### How do guardrails differ from alarms?

Guardrails are preventive — they validate resource configurations at deployment time and block non-compliant deployments before resources are modified. [Alarms](/observability/alarms) are reactive — they monitor running infrastructure and send notifications when metrics cross thresholds. Use guardrails to prevent insecure configurations from being deployed. Use alarms to detect unexpected behavior in already-deployed infrastructure.

### What happens when a guardrail is violated?

The deployment is rejected with an error identifying which resource violated which policy. The resource configuration must be updated to satisfy the guardrail before the deployment can proceed. This fail-fast behavior ensures that non-compliant configurations never reach your AWS infrastructure.

### What is VPC-only database accessibility?

VPC-only accessibility means the database has no public IP address and can only be reached from within the AWS VPC. Workloads running in the same VPC (such as Lambda functions or container workloads configured with [`connectTo`](/configuration/connecting-resources)) connect normally. For administration and debugging, use a [bastion host](/resources/security/bastion-host) or the [`bastion:tunnel`](/cli/bastion-tunnel) CLI command to create a secure tunnel from your local machine.

### What does AWS WAF protect against?

AWS WAF inspects incoming HTTP/HTTPS requests and blocks those matching configured rules. It defends against OWASP top-10 threats including SQL injection, cross-site scripting (XSS), and bot traffic. WAF rules can also implement rate limiting and geographic restrictions. See [web application firewall](/resources/security/web-application-firewall) for configuration details within Stacktape.

### Do security guardrails add cost to my AWS bill?

The guardrails themselves are policy checks — they don't create AWS resources directly. However, the configurations they require may have cost implications. VPC-only databases may need NAT gateways if workloads also require internet access. AWS WAF charges per rule and per request evaluated. Custom domains require DNS hosted zones. Check current AWS pricing for each service when planning which guardrails to enable.

### Can I restrict which database engines my team uses?

Yes. The `database-engine-restriction` guardrail lets you define an `allowedEngines` list. Any database using an engine not on the list is blocked from deploying. This is useful for standardizing on engines your DBA team supports, blocking deprecated versions with known vulnerabilities, or meeting compliance requirements that mandate specific database technologies. See [database guardrails](/guardrails/databases) for the full database-related guardrail set.

### When should I use guardrails instead of code reviews?

Use both — they serve different purposes. Code reviews catch logic errors, design issues, and context-dependent decisions that require human judgment. Guardrails enforce binary policy requirements (VPC-only databases, deletion protection, dead-letter queues) that reviewers might miss under time pressure. Guardrails are especially valuable as teams grow, because policy enforcement doesn't depend on reviewer vigilance or familiarity with every security baseline.

### How do I access a VPC-only database for debugging?

Use the [`bastion:tunnel`](/cli/bastion-tunnel) CLI command to create a secure SSH tunnel from your local machine to the database through a [bastion host](/resources/security/bastion-host). This lets you connect your local database client (psql, MySQL Workbench, etc.) to the VPC-only database without exposing it to the internet. The bastion host itself runs inside the VPC and is accessed through AWS Systems Manager, requiring no open inbound ports.

### What is the difference between deletion protection and backups?

Deletion protection prevents the database instance itself from being deleted — it's a safeguard against accidental removal of the resource via API calls. Backups (automated snapshots, point-in-time recovery) protect against data corruption or accidental data changes within a running database. Both serve different purposes: deletion protection keeps the database from being removed, while backups let you recover data if something goes wrong inside it. The `require-deletion-protection` guardrail enforces the former.

### Can I block specific CLI commands for certain stages?

Yes, but through the `command-restriction` guardrail covered on the [deployment guardrails](/guardrails/deployment) page, not through security guardrails. The `command-restriction` guardrail lets you define `blockedCommands` — for example, blocking `delete` and `rollback` on production stages. Combined with `require-deletion-protection`, this provides both resource-level and command-level protection against accidental data loss.
