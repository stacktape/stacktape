# Multi-Container Workload

A multi-container workload runs multiple containers together as a single unit with shared compute resources on AWS ECS Fargate or EC2. Use it for peer containers — independent containers that each have their own packaging, events, and lifecycle — init containers, or co-located processes that scale together. Stacktape manages container orchestration, auto-scaling, and deployments.

**Pricing**: Fargate bills per vCPU and memory — a minimal task (0.25 vCPU, 512 MB) costs approximately $9/month in us-east-1 (see [compute resources](#compute-resources) for common sizes). ARM (Graviton) tasks cost ~20% less. Fargate cannot scale to zero.

## When to use

**Use a multi-container workload when:**

- You need multiple peer containers in a shared task — each with its own packaging, events, traffic routing, and lifecycle
- Your workload requires init containers that must complete before the main containers start
- You need per-container traffic routing from different sources (e.g., one container serves HTTP, another handles TCP)
- You want explicit control over container startup ordering and dependencies

**Use something else when:**

| Situation | Better option |
|-----------|---------------|
| Single HTTP-facing service | [Web service](/resources/compute/web-service) — built-in load balancing, CDN, CORS, custom domains |
| Single internal service | [Private service](/resources/compute/private-service) — simpler config for internal traffic |
| Background worker, no inbound traffic | [Worker service](/resources/compute/worker-service) |
| Main container + helper sidecars | Web service, private service, or worker service with `sideContainers` |
| Serverless, pay-per-request | [Lambda function](/resources/compute/lambda-function) — scales to zero |
| One-off or scheduled batch processing | [Batch job](/resources/compute/batch-job) |

<Tip>Web services, private services, and worker services all support `sideContainers` for init and always-running helper containers. Prefer `sideContainers` when you have one primary container plus helpers. Use a multi-container workload when containers should be modeled as peers — each with its own events and traffic routing — or when your setup doesn't fit the single-main-container shape.</Tip>

## Basic example

This deploys two containers as a single task: an API server that receives HTTP traffic through an [HTTP API Gateway](/resources/networking/http-api-gateway), and a background worker that runs alongside it.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, MultiContainerWorkload, HttpApiGateway, StacktapeImageBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const gateway = new HttpApiGateway({});

  const workload = new MultiContainerWorkload({
    containers: [
      {
        name: "api",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/api.ts",
        }),
        events: [
          {
            type: "http-api-gateway",
            properties: {
              httpApiGatewayName: "gateway",
              containerPort: 3000,
              method: "*",
              path: "/{proxy+}",
            },
          },
        ],
      },
      {
        name: "worker",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/worker.ts",
        }),
      },
    ],
    resources: {
      cpu: 0.5,
      memory: 1024,
    },
  });

  return {
    resources: { gateway, workload },
  };
});
// [!code focus-end]`
    }
  ]}
/>

```typescript
// src/api.ts
import { serve } from '@hono/node-server';
import { Hono } from 'hono';

const app = new Hono();
app.get('/', (c) => c.json({ status: 'ok' }));
app.get('/health', (c) => c.text('healthy'));

serve({ fetch: app.fetch, port: 3000 });
```

The `httpApiGatewayName: "gateway"` references the resource named `gateway` in the `resources` object. The path `/{proxy+}` is an API Gateway greedy path parameter that catches all incoming request paths and forwards them to the container.

Both containers share the task's CPU and memory. The `worker` container runs without external traffic — it can reach the API container via `localhost:3000` because they share a network namespace.

The Stacktape image buildpack builds container images from your source code automatically. For other packaging options (custom Dockerfile, prebuilt image, Nixpacks, external buildpack), see [container packaging](/packaging/containers/).

Multi-container workloads support [dev mode](/local-development/dev-mode) (`stacktape dev`) for local iteration during development.

## Container configuration

The `containers` array defines every container in the workload. Each container has its own `packaging`, `environment`, `events`, and health checks, but all containers share the task's CPU and memory pool.

By default, every container is `essential: true` — if it exits or crashes, the entire task restarts. Set `essential: false` for containers meant to run once and exit (like database migrations or cache warming scripts). Without this, the task would restart every time an init container exits successfully.

CPU and memory are configured at the workload level in the `resources` property. Size the task for the combined needs of all containers.

Containers within the same task share a network namespace and can communicate via `localhost`. Add a `workload-internal` event to formally declare which ports a container exposes to other containers in the workload. This creates a port mapping in the task definition.

Use `dependsOn` to control startup order — for example, wait for a migration container to finish before starting the app. See [container dependencies](#container-dependencies) for details.

## Traffic routing

Multi-container workloads route traffic independently per container using `events`. Each container can integrate with a different traffic source.

Five event types are available:

| Event type | Use case | External resource required |
|---|---|---|
| `http-api-gateway` | HTTP APIs, request routing | [HTTP API Gateway](/resources/networking/http-api-gateway) |
| `application-load-balancer` | Advanced routing (path, headers, query params) | [Application Load Balancer](/resources/networking/application-load-balancer) |
| `network-load-balancer` | TCP/TLS traffic (MQTT, game servers) | [Network Load Balancer](/resources/networking/network-load-balancer) |
| `workload-internal` | Container-to-container within the same task | None |
| `service-connect` | Service-to-service discovery across resources | None |

Custom domains are configured on the gateway or load balancer resource, not on the workload itself. See [HTTP API Gateway](/resources/networking/http-api-gateway) or [Application Load Balancer](/resources/networking/application-load-balancer) docs for custom domain setup.

### Service-connect

Use `service-connect` to make a container reachable from other compute resources in the same stack via service discovery. Other resources connect using `protocol://alias:port`.

<CodeBlock
  intellisense

export default defineConfig(() => {

      {
        name: "api",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/api.ts",
        }),
        // [!code focus-start]
        events: [
          {
            type: "service-connect",
            properties: {
              containerPort: 3000,
              alias: "my-api",
              protocol: "http",
            },
          },
        ],
        // [!code focus-end]
      },
    ],
    resources: {
      cpu: 0.5,
      memory: 1024,
    },
  });

  return {
    resources: { workload },
  };
});`
    }
  ]}
/>

Other compute resources in the stack can reach this container at `http://my-api:3000`. If `alias` is omitted, it defaults to `{resourceName}-{containerName}` (e.g., `workload-api` for a resource named `workload` with a container named `api`).

### Application Load Balancer

Use ALB events when you need routing by path, method, host, headers, query parameters, or source IP. Define an [Application Load Balancer](/resources/networking/application-load-balancer) resource and reference it by name.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, MultiContainerWorkload, ApplicationLoadBalancer, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const alb = new ApplicationLoadBalancer({});

  const workload = new MultiContainerWorkload({
    containers: [
      {
        name: "api",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/api.ts",
        }),
        // [!code focus-start]
        events: [
          {
            type: "application-load-balancer",
            properties: {
              loadBalancerName: "alb",
              containerPort: 3000,
              priority: 1,
              paths: ["/*"],
            },
          },
        ],
        loadBalancerHealthCheck: {
          healthcheckPath: "/health",
          healthcheckInterval: 10,
        },
        // [!code focus-end]
      },
    ],
    resources: {
      cpu: 0.5,
      memory: 1024,
    },
  });

  return {
    resources: { alb, workload },
  };
});`
    }
  ]}
/>

Load balancer health check defaults: path `/`, interval `5` seconds, timeout `4` seconds. ALB defaults to `HTTP` health check protocol.

### Network Load Balancer

Use NLB events for non-HTTP traffic (TCP/TLS) — MQTT brokers, game servers, custom binary protocols. Define a [Network Load Balancer](/resources/networking/network-load-balancer) resource and reference it by name.

NLB events use `loadBalancerName`, required `listenerPort`, and `containerPort`. They do not support the ALB routing properties (paths, methods, hosts, headers, query parameters).

## Compute resources

The `resources` property sets CPU, memory, and the compute engine shared by all containers in the workload.

Two compute modes are available:

- **Fargate** — set `cpu` and `memory`. Serverless, no infrastructure to manage.
- **EC2** — set `instanceTypes` for specific instance types, more control, or GPU access. Instances auto-scale and are refreshed weekly for patching.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, MultiContainerWorkload, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const workload = new MultiContainerWorkload({
    containers: [
      {
        name: "app",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/app.ts",
        }),
      },
    ],
    // [!code focus-start]
    resources: {
      cpu: 1,
      memory: 2048,
      architecture: "arm64",
    },
    // [!code focus-end]
  });

  return {
    resources: { workload },
  };
});`
    }
  ]}
/>

Fargate CPU and memory must use valid combinations. Approximate monthly costs (us-east-1, single task, computed from per-vCPU and per-MB rates):

| vCPU | Memory | ~Cost/month |
|------|--------|-------------|
| 0.25 | 512 MB | ~$9 |
| 0.5 | 1024 MB | ~$18 |
| 1 | 2048 MB | ~$36 |
| 2 | 4096 MB | ~$71 |
| 4 | 8192 MB | ~$142 |

<Tip>Set `architecture: 'arm64'` to use ARM (Graviton) tasks for ~20% lower compute cost. The default is `x86_64`. Architecture is ignored when using EC2 `instanceTypes`.</Tip>

For EC2, specify `instanceTypes` instead of `cpu`/`memory`. The first type in the list is preferred. Set `enableWarmPool: true` (single instance type only) to keep pre-initialized instances ready for faster scaling.

## Scaling

Auto-scaling (default: 1 instance, 80% CPU/memory target) adds or removes task instances based on CPU and memory utilization. All containers in a task scale together — you cannot scale individual containers independently.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, MultiContainerWorkload, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const workload = new MultiContainerWorkload({
    containers: [
      {
        name: "app",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/app.ts",
        }),
      },
    ],
    resources: {
      cpu: 0.5,
      memory: 1024,
    },
    // [!code focus-start]
    scaling: {
      minInstances: 2,
      maxInstances: 10,
      scalingPolicy: {
        keepAvgCpuUtilizationUnder: 70,
      },
    },
    // [!code focus-end]
  });

  return {
    resources: { workload },
  };
});`
    }
  ]}
/>

Without scaling configured, a single task runs (`minInstances` and `maxInstances` both default to `1`). Setting `minInstances` to 0 is not supported — at least one task is always running and billing. For pay-per-request compute that scales to zero, use a [Lambda function](/resources/compute/lambda-function).

Auto-scaling adds instances when utilization exceeds the target and removes instances when it drops below. For user-facing production workloads, consider `minInstances: 2` so more than one instance can receive traffic.

## Container dependencies

Use `dependsOn` to control container startup order within a task. This is essential for init-container patterns where setup must complete before the main application starts.

Available conditions:

| Condition | Meaning |
|-----------|---------|
| `START` | The dependency container has started |
| `HEALTHY` | The dependency has passed its first health check |
| `SUCCESS` | The dependency has exited with code 0 |
| `COMPLETE` | The dependency has finished executing (regardless of exit code) |

### Init container pattern

Run a database migration before starting the application:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, MultiContainerWorkload, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const workload = new MultiContainerWorkload({
    containers: [
      // [!code focus-start]
      {
        name: "migrate",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/migrate.ts",
        }),
        essential: false,
      },
      {
        name: "app",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/app.ts",
        }),
        dependsOn: [{ containerName: "migrate", condition: "SUCCESS" }],
      },
      // [!code focus-end]
    ],
    resources: {
      cpu: 0.5,
      memory: 1024,
    },
    connectTo: ["myDatabase"],
  });

  return {
    resources: { workload },
  };
});`
    }
  ]}
/>

The `app` container waits until `migrate` exits with code 0 (`SUCCESS` condition) before starting. Setting `essential: false` on `migrate` prevents the task from restarting when it exits — by default (`essential: true`), the entire workload restarts when any container stops.

Use the `HEALTHY` condition when the dependency is a long-running container that exposes a health check — the dependent container waits until the first health check passes.

## EFS volume mounts

Mount [EFS filesystems](/resources/storage/efs-filesystem) for persistent storage that survives container restarts and is shared across containers in the task. Useful when multiple containers need access to the same files.

<CodeBlock

      lang: 'typescript',

export default defineConfig(() => {
  const storage = new EfsFilesystem({});

  const workload = new MultiContainerWorkload({
    containers: [
      {
        name: "app",
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: "./src/app.ts",
        }),
        // [!code focus-start]
        volumeMounts: [
          {
            type: "efs",
            properties: {
              efsFilesystemName: "storage",
              mountPath: "/data",
            },
          },
        ],
        // [!code focus-end]
      },
    ],
    resources: {
      cpu: 0.5,
      memory: 1024,
    },
  });

  return {
    resources: { storage, workload },
  };
});`
    }
  ]}
/>

Each volume mount specifies the EFS filesystem resource name and the `mountPath` inside the container. Optionally set `rootDirectory` to restrict the mount to a subdirectory within the filesystem (defaults to `/`).

## Connecting to other resources

Use `connectTo` to grant the workload access to databases, buckets, queues, and other resources in your stack. Stacktape automatically grants IAM permissions, opens network access for databases and Redis, and injects environment variables with connection details into all containers.

Environment variables are injected in the format `STP_[RESOURCE_NAME]_[PARAM]`. For example, connecting to a relational database named `myDatabase` injects `STP_MY_DATABASE_CONNECTION_STRING`, `STP_MY_DATABASE_HOST`, `STP_MY_DATABASE_PORT`, and more.

For the full list of injected variables per resource type, see [connecting resources](/configuration/connecting-resources).

For permissions not covered by `connectTo` (e.g., AWS Rekognition, Bedrock, Textract), use `iamRoleStatements` to add raw IAM policy statements.

## Gradual deployments

Gradual deployments shift traffic incrementally from the old version to the new one during deployment. You can optionally run validation Lambda functions before (`beforeAllowTrafficFunction`) and after (`afterTrafficShiftFunction`) the traffic shift to perform smoke tests.

Available strategies:

- **Canary** — send 10% of traffic to the new version first, then shift all traffic after a validation window (`Canary10Percent5Minutes`, `Canary10Percent15Minutes`)
- **Linear** — increase traffic to the new version by 10% at regular intervals (`Linear10PercentEvery1Minutes`, `Linear10PercentEvery3Minutes`)
- **AllAtOnce** — instant switch

<Warning>Gradual deployments require at least one container with an `application-load-balancer` event. They are not available with HTTP API Gateway or Network Load Balancer events.</Warning>

## Static outbound IP

To get a static outbound IP — for IP whitelisting with third-party APIs, payment gateways, or partner services — enable private subnets with NAT:

export default defineConfig(() => {

        }),
      },

    resources: {
      cpu: 0.5,
      memory: 1024,
    },
    // [!code focus-start]
    usePrivateSubnetsWithNAT: true,
    // [!code focus-end]
  });

  return {
    resources: { workload },
    stackConfig: {
      vpc: {
        nat: { availabilityZones: 2 },
      },
    },
  };
});`

With `usePrivateSubnetsWithNAT`, the workload runs in private subnets and uses NAT Gateways for outbound internet. Each NAT Gateway gets a static Elastic IP that persists across deployments. Configure the number of NAT Gateways (1–3) in `stackConfig.vpc.nat`. More gateways mean higher availability but higher cost.

<Warning>Each NAT Gateway costs ~$32/month plus data processing fees. Only enable this when you need a static outbound IP.</Warning>

## Health checks

Two types of health checks protect your workload:

- **Load balancer health check** — configured per-container via `loadBalancerHealthCheck` (only for containers with ALB or NLB events). Checks a path or port on the container at regular intervals. Defaults: path `/`, interval `5s`, timeout `4s`.
- **Internal health check** — configured per-container via `internalHealthCheck`. Runs a command inside the container (e.g., `["CMD-SHELL", "curl -f http://localhost/ || exit 1"]`). If a container marked as `essential` fails, the task is replaced. Defaults: interval `30s`, timeout `5s`, `3` retries before marking unhealthy.

For most workloads with load balancer events, the load balancer health check is sufficient. Use `internalHealthCheck` when you need to verify internal state beyond HTTP reachability.

## Logging

Container output (`stdout`/`stderr`) is automatically sent to CloudWatch Logs and retained for 90 days by default. Each container has its own logging configuration. View logs with `stacktape logs`. Adjust retention per-container with the `logging.retentionDays` property.

## Remote debugging

Enable `enableRemoteSessions: true` to allow interactive shell access to running containers via `stacktape container:session`. Useful for debugging issues that only reproduce in the deployed environment.
