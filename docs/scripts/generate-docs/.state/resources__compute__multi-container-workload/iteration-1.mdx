# Multi-Container Workload

A multi-container workload runs multiple containers as a single AWS ECS task with shared CPU and memory. Use it for sidecars, init containers, or workloads that need multiple co-located processes with independent traffic routing and lifecycle management. Built on Fargate (serverless) or EC2 instances, with horizontal auto-scaling.

**Pricing**: A minimal Fargate task (0.25 vCPU, 512 MB) costs ~$9/month for compute. ARM (Graviton) tasks cost ~20% less. Cost scales linearly with CPU, memory, and instance count. Fargate cannot scale to zero.

## When to use

**Use a multi-container workload when:**

- You need multiple peer containers in a shared task — each with its own packaging, traffic routing, and lifecycle
- Your workload requires init containers that must complete before the main containers start
- You need per-container traffic routing from different sources (e.g., one container serves HTTP, another handles TCP)
- You want explicit control over container startup ordering and dependencies

**Use something else when:**

| Situation | Better option |
|-----------|---------------|
| Single HTTP-facing service | [Web service](/resources/compute/web-service) — built-in load balancing, CDN, CORS, custom domains |
| Single internal service | [Private service](/resources/compute/private-service) — simpler config for VPC-internal APIs |
| Background worker, no inbound traffic | [Worker service](/resources/compute/worker-service) |
| Main container + helper sidecars | Web service, private service, or worker service with `sideContainers` |
| Serverless, pay-per-request | [Lambda function](/resources/compute/lambda-function) — scales to zero |
| One-off or scheduled batch processing | [Batch job](/resources/compute/batch-job) |

<Tip>Web services, private services, and worker services all support `sideContainers` for init and always-running helper containers. Only use a multi-container workload when you need multiple peer containers with independent traffic routing — not just a main container with helpers.</Tip>

## Basic example

This deploys two containers as a single task: an API server that receives HTTP traffic through an API Gateway, and a background worker. Both containers share compute resources and can communicate via `localhost`.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, ContainerWorkload, HttpApiGateway, StacktapeImageBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const gateway = new HttpApiGateway({});

  const workload = new ContainerWorkload({
    containers: [
      {
        name: 'api',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/api.ts'
        }),
        events: [
          {
            type: 'http-api-gateway',
            properties: {
              httpApiGatewayName: 'gateway',
              containerPort: 3000,
              method: '*',
              path: '/{proxy+}'
            }
          }
        ]
      },
      {
        name: 'worker',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/worker.ts'
        })
      }
    ],
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { gateway, workload }
  };
});
// [!code focus-end]`
    }
  ]}
/>

```typescript
// src/api.ts
import { serve } from '@hono/node-server';
import { Hono } from 'hono';

const app = new Hono();
app.get('/', (c) => c.json({ status: 'ok' }));
app.get('/health', (c) => c.text('healthy'));

serve({ fetch: app.fetch, port: 3000 });
```

The `api` container receives HTTP traffic routed through the API Gateway. The `worker` container runs in the same task without external events — it can communicate with the API container via `localhost:3000`.

The Stacktape image buildpack builds container images from your source code automatically. For other packaging options (custom Dockerfile, prebuilt image, Nixpacks, external buildpack), see [container packaging](/packaging/containers/).

## Containers

The `containers` array defines every container in the workload. Each container has its own packaging, environment variables, events, and health checks, but all containers share the task's CPU and memory pool.

Key per-container properties:

- **`name`** — unique identifier within the workload
- **`packaging`** — how to build or reference the container image (`stacktape-image-buildpack`, `custom-dockerfile`, `prebuilt-image`, `nixpacks`, or `external-buildpack`)
- **`essential`** — if `true` (default), the entire task restarts when this container stops or fails. Set to `false` for init containers that exit after completing their work.
- **`events`** — how this container receives traffic (API Gateway, load balancer, internal, or service-connect)
- **`dependsOn`** — control startup order by waiting for other containers to reach a specific state
- **`environment`** — per-container environment variables (use `$ResourceParam()` or `$Secret()` for dynamic values)
- **`volumeMounts`** — mount [EFS filesystems](/resources/storage/efs-filesystem) for persistent, shared storage across containers

Containers within the same task communicate via `localhost` on the ports opened by `workload-internal` events.

## Traffic routing

Unlike [web services](/resources/compute/web-service) (which have built-in load balancing), multi-container workloads route traffic per-container through `events`. Each container can integrate with different traffic sources independently.

Five event types are available:

| Event type | Use case | External resource required |
|---|---|---|
| `http-api-gateway` | HTTP/HTTPS traffic, REST APIs | [HTTP API Gateway](/resources/networking/http-api-gateway) |
| `application-load-balancer` | WebSocket, advanced routing, WAF | [Application Load Balancer](/resources/networking/application-load-balancer) |
| `network-load-balancer` | TCP/TLS traffic (MQTT, game servers) | [Network Load Balancer](/resources/networking/network-load-balancer) |
| `workload-internal` | Container-to-container within the same task | None |
| `service-connect` | Service-to-service discovery across resources | None |

<Warning>Only one container in a workload can use load balancer events (`application-load-balancer` or `network-load-balancer`). All ALB listener rules must target the same container and port.</Warning>

### Service-connect

Use `service-connect` to make a container reachable from other compute resources in the same stack. Other resources connect using `protocol://alias:port`.

<CodeBlock

          entryfilePath: './src/api.ts'

        // [!code focus-start]
        events: [
          {
            type: 'service-connect',
            properties: {
              containerPort: 3000,
              alias: 'my-api',
              protocol: 'http'
            }
          }
        ]
        // [!code focus-end]
      }
    ],

      cpu: 0.5,
      memory: 1024

Other resources in the stack can reach this container at `http://my-api:3000`. If `alias` is omitted, it defaults to a name derived from the resource and container names (e.g., `workload-api`).

<Info>Service-connect is not compatible with gradual deployment strategies. Use `http-api-gateway` or `application-load-balancer` events if you need canary/linear deployments.</Info>

### Application Load Balancer

Use ALB events for WebSocket support, advanced routing (path, headers, query parameters), or web application firewalls. Define an [Application Load Balancer](/resources/networking/application-load-balancer) resource and reference it by name.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, ContainerWorkload, ApplicationLoadBalancer, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const alb = new ApplicationLoadBalancer({});

  const workload = new ContainerWorkload({
    containers: [
      {
        name: 'api',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/api.ts'
        }),
        // [!code focus-start]
        events: [
          {
            type: 'application-load-balancer',
            properties: {
              loadBalancerName: 'alb',
              containerPort: 3000,
              priority: 1,
              paths: ['/*']
            }
          }
        ],
        loadBalancerHealthCheck: {
          healthcheckPath: '/health',
          healthcheckInterval: 10
        }
        // [!code focus-end]
      }
    ],
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { alb, workload }
  };
});`
    }
  ]}
/>

Load balancer health check defaults: path `/`, interval `5` seconds, timeout `4` seconds.

## Resources and compute

The `resources` property sets CPU, memory, and the compute engine shared by all containers in the workload.

Two compute engines are available:

- **Fargate** (default) — set `cpu` and `memory`. Serverless, no infrastructure to manage.
- **EC2** — set `instanceTypes` for more control, GPU access, or lower cost on steady workloads. Instances auto-scale and are refreshed weekly for patching.

<CodeBlock

      memory: 2048,
      architecture: 'arm64'
    }

  });

  return {
    resources: { workload }

});`

Fargate CPU and memory must use valid combinations. Common sizes and approximate monthly costs (us-east-1, single task):

| vCPU | Memory | ~Cost/month |
|------|--------|-------------|
| 0.25 | 512 MB | ~$9 |
| 0.5 | 1024 MB | ~$18 |
| 1 | 2048 MB | ~$36 |
| 2 | 4096 MB | ~$71 |
| 4 | 8192 MB | ~$142 |

<Tip>Set `architecture: 'arm64'` to use ARM (Graviton) tasks for ~20% lower compute cost. The default is `x86_64`. Architecture is ignored when using EC2 `instanceTypes`.</Tip>

For EC2, specify `instanceTypes` instead of `cpu`/`memory`. The first type in the list is preferred. Set `enableWarmPool: true` (single instance type only) to keep pre-initialized instances ready for faster scaling.

## Scaling

Auto-scaling adds or removes task instances based on CPU and memory utilization. All containers in a task scale together — you cannot scale individual containers independently.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, ContainerWorkload, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const workload = new ContainerWorkload({
    containers: [
      {
        name: 'app',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/app.ts'
        })
      }
    ],
    resources: {
      cpu: 0.5,
      memory: 1024
    },
    // [!code focus-start]
    scaling: {
      minInstances: 2,
      maxInstances: 10,
      scalingPolicy: {
        keepAvgCpuUtilizationUnder: 70
      }
    }
    // [!code focus-end]
  });

  return {
    resources: { workload }
  };
});`
    }
  ]}
/>

Without scaling configured, a single task runs (`minInstances` and `maxInstances` both default to `1`). Minimum instances cannot be set to 0. For production workloads, set at least `minInstances: 2` so the workload stays available if one task fails.

Scaling policy defaults: scale out when average CPU or memory exceeds 80%, scale in when utilization drops below the target.

## Container dependencies

Use `dependsOn` to control container startup order within a task. This is essential for init-container patterns where setup must complete before the main application starts.

Available conditions:

| Condition | Meaning |
|-----------|---------|
| `START` | The dependency container has started |
| `HEALTHY` | The dependency has passed its first health check |
| `SUCCESS` | The dependency has exited with code 0 |
| `COMPLETE` | The dependency has finished executing (regardless of exit code) |

### Init container pattern

Run a database migration before starting the application:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, ContainerWorkload, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const workload = new ContainerWorkload({
    containers: [
      // [!code focus-start]
      {
        name: 'migrate',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/migrate.ts'
        }),
        essential: false
      },
      {
        name: 'app',
        packaging: new StacktapeImageBuildpackPackaging({
          entryfilePath: './src/app.ts'
        }),
        dependsOn: [
          { containerName: 'migrate', condition: 'SUCCESS' }
        ]
      }
      // [!code focus-end]
    ],
    resources: {
      cpu: 0.5,
      memory: 1024
    },
    connectTo: ['myDatabase']
  });

  return {
    resources: { workload }
  };
});`
    }
  ]}
/>

The `migrate` container runs first and exits. Setting `essential: false` prevents the task from restarting when it exits successfully. The `app` container starts only after `migrate` exits with code 0 (`SUCCESS` condition).

Use the `HEALTHY` condition when the dependency is a long-running container that exposes a health check — the dependent container waits until the first health check passes.

## Connecting to other resources

Use `connectTo` to grant the workload access to databases, buckets, queues, and other resources in your stack. Stacktape automatically grants IAM permissions, opens network access (security group rules for databases and Redis), and injects environment variables with connection details into all containers.

Environment variables are injected in the format `STP_[RESOURCE_NAME]_[PARAM]`. For example, connecting to a relational database named `myDatabase` injects `STP_MY_DATABASE_CONNECTION_STRING`, `STP_MY_DATABASE_HOST`, `STP_MY_DATABASE_PORT`, and more.

For the full list of injected variables per resource type, see [connecting resources](/configuration/connecting-resources).

For permissions not covered by `connectTo` (e.g., AWS Rekognition, Bedrock, SES), use `iamRoleStatements` to add raw IAM policy statements.

## Gradual deployments

Shift traffic gradually from the old version to the new one during deployments. If health checks fail during the shift, traffic rolls back automatically.

Available strategies:

- **Canary** — send 10% of traffic to the new version first, then shift all traffic after a validation window (`Canary10Percent5Minutes`, `Canary10Percent15Minutes`)
- **Linear** — increase traffic to the new version by 10% at regular intervals (`Linear10PercentEvery1Minutes`, `Linear10PercentEvery3Minutes`)
- **AllAtOnce** — instant switch with automatic rollback on failure

You can optionally run validation Lambda functions before (`beforeAllowTrafficFunction`) and after (`afterTrafficShiftFunction`) the traffic shift to perform smoke tests.

<Warning>Gradual deployments require at least one container with an `application-load-balancer` event. They are not available with HTTP API Gateway, Network Load Balancer, or service-connect events.</Warning>

## Static outbound IP

By default, your task's outbound IP address changes with each deployment. To get a static outbound IP — for IP whitelisting with third-party APIs, payment gateways, or partner services — enable private subnets with NAT:

          entryfilePath: './src/app.ts'

      cpu: 0.5,
      memory: 1024

    usePrivateSubnetsWithNAT: true
    // [!code focus-end]
  });

  return {
    resources: { workload },
    stackConfig: {
      vpc: {
        nat: { availabilityZones: 2 }
      }
    }
  };
});`
    }

/>

All outbound traffic routes through NAT Gateways, each with a static Elastic IP that persists across deployments. Configure the number of NAT Gateways (1–3) in `stackConfig.vpc.nat`. More gateways mean higher availability but higher cost.

<Warning>Each NAT Gateway costs ~$32/month plus data processing fees. Only enable this when you need a static outbound IP.</Warning>

## Health checks

Two types of health checks protect your workload:

- **Load balancer health check** — configured per-container via `loadBalancerHealthCheck` (only for containers with ALB or NLB events). The load balancer pings a path on the container and stops routing traffic to unhealthy instances. Defaults: path `/`, interval `5s`, timeout `4s`.
- **Internal health check** — configured per-container via `internalHealthCheck`. Runs a command inside the container (e.g., `["CMD-SHELL", "curl -f http://localhost/ || exit 1"]`). If a container marked as `essential` fails, the task is replaced. Defaults: interval `30s`, timeout `5s`, `3` retries before marking unhealthy.

## Logging

Container output (`stdout`/`stderr`) is automatically sent to CloudWatch Logs and retained for 90 days by default. Each container has its own logging configuration. View logs with `stacktape logs` or in the Stacktape Console. Adjust retention per-container with the `logging.retentionDays` property.

## Remote debugging

Enable `enableRemoteSessions: true` to allow SSH-like access to running containers via `stacktape container:session`. This installs a small SSM agent using minimal CPU/memory. Useful for debugging issues that only reproduce in the deployed environment.
