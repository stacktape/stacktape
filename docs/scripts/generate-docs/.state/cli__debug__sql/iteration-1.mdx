# debug:sql

The `debug:sql` command executes read-only SQL queries against a deployed [relational database](/resources/databases/relational-database). It supports PostgreSQL and MySQL engines and restricts queries to `SELECT`, `SHOW`, `DESCRIBE`, and `EXPLAIN` statements, preventing accidental writes to production data.

## Usage

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName myDatabase --sql "SELECT * FROM users LIMIT 10"
```

The command connects to the database using credentials stored in AWS SSM Parameter Store during deployment. It resolves the connection string automatically — you only need to provide the resource name from your Stacktape configuration.

## When to use

Use `debug:sql` when you need to inspect data in a deployed database without setting up a local database client or managing connection credentials manually. Common scenarios include verifying that a migration ran correctly, checking row counts after a deployment, or investigating a production issue by examining recent records.

For write operations (INSERT, UPDATE, DELETE), use [`bastion:tunnel`](/cli/bastion-tunnel) to open a secure tunnel and connect with a full-featured database client like `psql`, `mysql`, or a GUI tool.

## Important flags

**`--sql`** — The SQL query to execute. Only read-only statements are accepted: `SELECT`, `SHOW`, `DESCRIBE`, and `EXPLAIN`. The command validates the query prefix before connecting to the database and rejects anything that could modify data.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName myDatabase --sql "DESCRIBE users"
```

**`--bastionResource`** — Required when the database is inside a VPC with no public access. The command tunnels through the specified [bastion host](/resources/security/bastion-host) using an SSM port-forwarding session. The tunnel is established automatically and cleaned up when the query completes.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName myDatabase --bastionResource myBastion --sql "SELECT COUNT(*) FROM orders"
```

**`--limit`** — Maximum number of rows to return. Defaults to 1000. Use this to prevent large result sets from overwhelming your terminal. The output indicates when results are truncated.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName myDatabase --sql "SELECT * FROM events" --limit 50
```

**`--timeout`** — Maximum time in milliseconds to wait for the query to complete. Defaults to 30000 (30 seconds). Increase this for complex queries that scan large tables.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName myDatabase --sql "SELECT * FROM large_table" --timeout 60000
```

## Read-only enforcement

The command only allows queries that begin with one of these prefixes (case-insensitive):

- `SELECT` — retrieve rows
- `SHOW` — display database metadata (tables, databases, variables)
- `DESCRIBE` — inspect table structure
- `EXPLAIN` — analyze query execution plans
- `\d` — PostgreSQL shorthand for describe

Any other statement (`INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, etc.) is rejected before a connection to the database is opened.

## VPC-only databases

Databases deployed without public accessibility require a [bastion host](/resources/security/bastion-host) to be reachable from your local machine. When you pass `--bastionResource`, the command establishes an SSM port-forwarding tunnel through the bastion, routes the query through `localhost`, and tears down the tunnel after the query completes. No SSH keys or inbound security group rules are needed — the tunnel uses AWS Systems Manager.

<Info>
When the database has a reader endpoint (Aurora clusters with read replicas), `debug:sql` automatically routes through the reader endpoint to avoid placing load on the primary writer instance.
</Info>

## Agent mode output

When running with `--agent` (or in a non-interactive environment), the command outputs structured JSON instead of a formatted table. The JSON payload includes `rows`, `fields`, `rowCount`, `truncated`, and the engine type. This makes it straightforward to pipe results into scripts or consume them from AI coding agents.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName myDatabase --sql "SELECT id, email FROM users LIMIT 5" --agent
```

## Examples

Query a PostgreSQL database for recent orders.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName mainDatabase --sql "SELECT id, status, created_at FROM orders ORDER BY created_at DESC LIMIT 20"
```

Show all tables in a MySQL database.

```bash
stacktape debug:sql --stage staging --region us-east-1 --resourceName appDb --sql "SHOW TABLES"
```

Check the execution plan for a slow query.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName mainDatabase --sql "EXPLAIN SELECT * FROM users WHERE email = 'test@example.com'"
```

Inspect table schema through a bastion host.

```bash
stacktape debug:sql --stage prod --region eu-west-1 --resourceName mainDatabase --bastionResource bastion --sql "DESCRIBE orders"
```

## FAQ

### Which database engines does debug:sql support?

The command supports PostgreSQL and MySQL databases deployed as a Stacktape [relational database](/resources/databases/relational-database). It detects the engine from the connection string protocol (`postgresql://` or `mysql://`) and uses the appropriate driver. Other database types like DynamoDB or Redis have their own dedicated debug commands.

### Can I run write queries with debug:sql?

No. The command enforces read-only access by checking that the SQL statement begins with `SELECT`, `SHOW`, `DESCRIBE`, `EXPLAIN`, or `\d`. Any other statement is rejected before a database connection is opened. For write operations, use [`bastion:tunnel`](/cli/bastion-tunnel) to connect with a full database client.

### How does debug:sql connect to a VPC-only database?

Pass `--bastionResource` to tunnel through a [bastion host](/resources/security/bastion-host) in your stack. The command creates an SSM port-forwarding session to the database endpoint, runs your query through the tunnel, and cleans up automatically. No SSH keys or open inbound ports are needed.

### What happens if my query returns too many rows?

Results are capped by the `--limit` flag, which defaults to 1000 rows. If the query returns more rows than the limit, the output is truncated and marked as such. Set a lower limit for large tables to keep terminal output manageable, or use `LIMIT` directly in your SQL.

### Does debug:sql use the reader or writer endpoint?

When the database has a reader endpoint (Aurora clusters with read replicas), the command automatically uses it. This avoids placing query load on the primary writer instance. For single-instance databases, the primary endpoint is used.

### How do I increase the query timeout?

Pass `--timeout` with a value in milliseconds. The default is 30000 (30 seconds). For complex analytical queries or large table scans, increase it as needed — for example, `--timeout 120000` for two minutes.

### Do I need to provide database credentials?

No. The command retrieves connection credentials automatically from AWS SSM Parameter Store, where Stacktape stores them during deployment. You only need permissions to read SSM parameters in the stack's region.

### Can I use debug:sql in CI or scripts?

Yes. Use the `--agent` flag to get structured JSON output instead of a formatted table. The JSON includes `rows`, `fields`, `rowCount`, and a `truncated` flag, making it easy to parse with `jq` or consume from scripts and AI agents.
