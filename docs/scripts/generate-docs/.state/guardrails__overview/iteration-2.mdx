# Guardrails Overview

Stacktape guardrails are organization-wide preventive rules that define allowed or blocked values for stages, regions, commands, resource types, and resource settings. Unlike [alarms](/observability/alarms) which notify you about runtime problems after deployment, guardrails validate your configuration at deploy time and reject violations with a clear error message.

## Guardrails vs alarms

Guardrails and [alarms](/observability/alarms) serve fundamentally different purposes. Guardrails are preventive — they block non-compliant configurations before anything changes. Alarms are reactive — they detect problems at runtime and send notifications. A well-managed production stack uses both.

| | Guardrails | Alarms |
|---|---|---|
| **When** | At deploy time | After deployment, at runtime |
| **Action** | Rejects the deployment | Sends a notification |
| **Scope** | Organization-wide, all projects | Per-resource, per-stack |
| **Purpose** | Prevent policy violations | Detect runtime issues |
| **Managed in** | [Stacktape Console](/stacktape-console/console-overview) (organization settings) | Stacktape config or Console |

## How guardrails work

Stacktape guardrails constrain what your team can deploy and how. Each guardrail targets a specific aspect of the deployment — the stage name, the AWS region, the CLI command being run, the resource types in the configuration, or the settings on individual resources like memory, timeout, or VPC accessibility. When a deployment violates an active guardrail, the operation is rejected.

Guardrails are configured at the organization level in the [Stacktape Console](/stacktape-console/console-overview). Once activated, a guardrail applies to all projects and stages within that organization.

Stacktape supports 15 guardrail types, organized into four categories:

| Category | What it controls | Example |
|---|---|---|
| [Deployment restrictions](/guardrails/deployment) | Stages, regions, commands, resource types | Allow only `eu-west-1` and `us-east-1` |
| [Security and data protection](/guardrails/security-and-data-protection) | VPC access, deletion protection, DLQ, WAF, custom domains | Require VPC for all databases |
| [Resource limits](/guardrails/resource-limits) | Function memory/timeout, container CPU/memory, stack size | Cap Lambda memory at 1024 MB |
| [Database restrictions](/guardrails/databases) | Allowed engines, blocked instance sizes | Allow only `postgres` and `aurora-postgresql` |

## Deployment restrictions

Deployment restriction guardrails control where and how your team can deploy. They operate on deployment parameters — stage name, region, and command — rather than on resource configuration.

- **Stage restriction** — Only allow deployments to specific stages. Configure an `allowedStages` list (e.g., `["production", "staging"]`); deployments to any other stage name are blocked. Useful for enforcing naming conventions or limiting which stages can exist.
- **Region restriction** — Only allow deployments to specific AWS regions. Configure an `allowedRegions` list (e.g., `["eu-west-1", "us-east-1"]`); deployments to other regions are blocked. Essential for data residency compliance and cost control.
- **Command restriction** — Block specific CLI commands. The `blockedCommands` array accepts command names such as `delete` and `rollback`. Use this to prevent accidental production deletions or restrict destructive operations.

For full details, see [Deployment guardrails](/guardrails/deployment).

## Security and data protection

Security guardrails enforce best practices that protect data and reduce attack surface. Each is a simple toggle — enabled or disabled — with no additional parameters to configure. Enabling one of these guardrails means every deployment in the organization must comply.

- **Require VPC for databases** — When enabled, all databases must use VPC-only accessibility with no public internet access. This prevents accidentally exposing a [relational database](/resources/databases/relational-database) to the public internet.
- **Require deletion protection** — When enabled, all [relational databases](/resources/databases/relational-database) must have `deletionProtection` set to `true`. Prevents accidental data loss from stack deletions or CloudFormation drift.
- **Require dead-letter queue** — When enabled, all [SQS queues](/resources/messaging/sqs-queue) must have a `redrivePolicy` (dead-letter queue) configured. Ensures failed messages are captured rather than silently lost.
- **Require WAF on load balancers** — When enabled, all [application load balancers](/resources/networking/application-load-balancer) must have a [web application firewall](/resources/security/web-application-firewall) attached. Adds a layer of protection against common web exploits.
- **Require custom domain** — When enabled, public-facing [web services](/resources/compute/web-service) and [hosting buckets](/resources/frontend/static-hosting) must have a [custom domain](/resources/networking/custom-domains) configured. Prevents auto-generated AWS URLs from reaching production.

For full details, see [Security and data protection guardrails](/guardrails/security-and-data-protection).

## Resource limits

Resource limit guardrails cap compute usage and stack size to control costs and prevent over-provisioned resources. Each guardrail accepts a numeric threshold.

- **Function memory limit** — Set a maximum memory allocation (in MB) for all [Lambda functions](/resources/compute/lambda-function). For example, setting `maxMemoryMB` to `1024` blocks any Lambda configured with more than 1024 MB. See [Lambda functions](/resources/compute/lambda-function) for supported memory values.
- **Function timeout limit** — Set a maximum timeout (in seconds) for all [Lambda functions](/resources/compute/lambda-function). For example, setting `maxTimeoutSeconds` to `30` prevents long-running Lambda configurations. See [Lambda functions](/resources/compute/lambda-function) for timeout constraints.
- **Container resource limit** — Set maximum vCPU (`maxCpu`) and memory in MB (`maxMemoryMB`) for container workloads including [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), and [multi-container workloads](/resources/compute/multi-container-workload).
- **Resource count limit** — Set the maximum number of resources allowed per stack (`maxResources`). Prevents stack sprawl that makes infrastructure hard to manage and expensive to maintain.

For full details, see [Resource limit guardrails](/guardrails/resource-limits).

## Database restrictions

Database guardrails standardize which database engines and instance sizes your team can use. This category also includes the resource type restriction guardrail, which can block any Stacktape resource type.

- **Allowed database engines** — Configure which database engine types are permitted via the `allowedEngines` list. Deployments using engines not in the list are blocked. The available values match the engine identifiers used by Stacktape [relational databases](/resources/databases/relational-database) — see that page for supported engines.
- **Blocked instance sizes** — Block specific RDS instance sizes via the `blockedInstanceSizes` list (e.g., `["db.r5.4xlarge", "db.r6g.8xlarge"]`). Prevents teams from provisioning expensive database instances without approval.
- **Blocked resource types** — Block specific Stacktape resource types via the `blockedResourceTypes` list. Despite its placement in the database category, this guardrail can block **any** resource type — not just databases. For example, you can block `function`, `web-service`, `bucket`, `bastion`, `open-search-domain`, `redis-cluster`, or any other Stacktape resource type.

<Info>
The blocked resource types guardrail appears under database restrictions in the Stacktape Console, but it accepts any Stacktape resource type. Use it to prevent teams from provisioning resource types your organization hasn't approved.
</Info>

For full details, see [Database guardrails](/guardrails/databases).

## Managing guardrails

Guardrails are managed in the [Stacktape Console](/stacktape-console/console-overview) at the organization level. Navigate to the Guardrails page in your organization settings to configure them.

Toggle guardrails (like "Require VPC for databases") are a single on/off switch. Value-based guardrails (like "Stage restriction" or "Function memory limit") open a configuration form where you specify the allowed values, blocked values, or numeric thresholds.

Changes take effect on the next deployment. Existing running stacks are not retroactively checked — guardrails apply when a deployment operation targets a stack.

For details on team roles and permissions, see [Team and access control](/stacktape-console/team-and-access-control).

## When to use

Guardrails are most valuable when:

- **Multiple developers deploy independently** — guardrails prevent one team from accidentally exposing a database or using an unapproved region without relying on code review to catch it.
- **You need compliance boundaries** — data residency (region restriction), network isolation (VPC databases), or audit requirements (deletion protection) can be enforced consistently.
- **Cost control is important** — resource limits prevent over-provisioned Lambda functions or expensive database instances from slipping through.
- **You want consistent production standards** — require custom domains and WAF for all public-facing services as a baseline policy.

## When NOT to use

- **Solo developers or small teams** — if you trust everyone deploying and rely on code review, guardrails add overhead without much benefit. Skip them until the team grows or compliance requirements arrive.
- **Early experimentation** — during the "move fast" phase of a project, restrictive guardrails slow development. Consider enabling guardrails only once the project matures toward production readiness.
- **Highly heterogeneous resource needs** — guardrails apply uniformly within an organization. If different projects need fundamentally different resource policies, you may need to structure your organizations accordingly.

## FAQ

### What happens when a guardrail is violated?

The deployment operation is rejected and Stacktape reports which guardrail was violated and what resource or value triggered it. This makes it clear what needs to change in the configuration or deployment parameters before retrying.

### Can I set different guardrails for production vs development stages?

Guardrails are configured at the organization level and apply to all projects and stages within that organization. You can use the stage restriction guardrail to limit which stage names are allowed, but resource-level guardrails (memory limits, VPC requirements) apply uniformly. Set thresholds that accommodate your development needs while still preventing egregious over-provisioning.

### How do Stacktape guardrails differ from AWS Service Control Policies?

AWS Service Control Policies (SCPs) operate at the AWS Organizations level and restrict which AWS API calls IAM principals can make. Stacktape guardrails operate at the configuration level and validate resource definitions before deployment. Guardrails are faster to configure (no IAM policy language), specific to Stacktape concepts (stages, resource types, packaging), and provide clearer error messages. SCPs are broader and can restrict AWS actions Stacktape doesn't control. For defense-in-depth, use both.

### Can I block specific resource types entirely?

Yes. The resource type restriction guardrail accepts a `blockedResourceTypes` list that can block any Stacktape resource type — `function`, `web-service`, `redis-cluster`, `bastion`, `open-search-domain`, and many others. Block resource types your organization hasn't approved for use. See [Database guardrails](/guardrails/databases) for details.

### Do guardrails apply to already-deployed stacks?

No. Guardrails are evaluated during deployment operations, not against running infrastructure. Stacks deployed before a guardrail was activated continue running. The guardrail applies when the next deployment targets that stack — forcing the team to fix the violation before any further changes can be made.

### Can guardrails prevent accidental stack deletion?

Yes. Use the command restriction guardrail to add `delete` to the `blockedCommands` list. This prevents anyone from running [`stacktape delete`](/cli/delete) across the organization. Combine with the stage restriction guardrail to control which stages can be targeted.

### What is the relationship between guardrails and IAM policies?

Guardrails operate within Stacktape's deployment pipeline — they check your configuration against organization rules. IAM policies operate at the AWS API level — they control what AWS actions can be performed. Guardrails catch configuration issues earlier and with clearer messages, but they don't replace IAM. A team member with direct AWS Console access bypasses Stacktape guardrails entirely. Use IAM policies for AWS-level access control and guardrails for Stacktape-level policy.

### How are guardrails managed?

Guardrails are managed in the [Stacktape Console](/stacktape-console/console-overview) under organization settings. Toggle-based guardrails (VPC requirement, deletion protection, etc.) are a single switch. Value-based guardrails (allowed stages, memory limits, blocked resource types) have a configuration form. See [Team and access control](/stacktape-console/team-and-access-control) for information about who can manage organization settings.

### Can I restrict which AWS regions my team deploys to?

Yes. The region restriction guardrail accepts an `allowedRegions` list of AWS region identifiers (e.g., `["eu-west-1", "us-east-1"]`). Deployments to any region not in the list are blocked. This is the recommended approach for data residency compliance — it's simpler than managing per-region IAM policies and catches violations before any AWS resources are provisioned. See [Deployment guardrails](/guardrails/deployment) for details.

### What is the difference between blocking a resource type and blocking a database engine?

The resource type restriction blocks entire categories of Stacktape resources — for example, blocking `redis-cluster` prevents anyone from adding a Redis resource to their stack. The database engine restriction is narrower: it controls which engine types (e.g., `postgres`, `aurora-mysql`) are allowed within [relational database](/resources/databases/relational-database) resources that are still permitted. Use resource type restriction to ban entire service categories; use engine restriction to standardize on specific database engines within allowed database resources.
