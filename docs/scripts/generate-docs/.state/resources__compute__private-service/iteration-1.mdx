# Private Service

A Stacktape private service runs always-on containers with no public endpoint, reachable only from other resources in your stack or VPC. Use it for internal microservices, gRPC backends, or any service that must never be exposed to the internet. Stacktape handles service discovery, health checks, auto-scaling, and zero-downtime deployments automatically.

**Pricing context:** The default `service-connect` networking adds approximately **~$0.50/month**. Switching to an Application Load Balancer adds approximately **~$18/month** in base fees. A minimal Fargate task (0.25 vCPU, 512 MB) costs approximately **~$9/month** for compute alone.

## When to use

A Stacktape private service is the right choice when your stack has internal components that other workloads call, but that should never be reachable from the public internet:

- **Internal microservices** â€” payment processors, auth validators, or email senders called only by your own services
- **gRPC backends** â€” low-latency RPC services consumed by other containers in the same stack
- **Data pipelines** â€” aggregation or transformation services invoked by other workloads
- **Background APIs** â€” services triggered by other stack resources, not by end users

### When NOT to use

- **Public-facing APIs** â€” use a [web service](/resources/compute/web-service) for containers that need a public HTTPS endpoint
- **Background workers without inbound calls** â€” use a [worker service](/resources/compute/worker-service) (no load balancing overhead)
- **Sporadic or event-driven workloads** â€” use a [Lambda function](/resources/compute/lambda-function) for pay-per-request pricing and scale-to-zero behavior
- **One-off or batch workloads** â€” use a [batch job](/resources/compute/batch-job) for tasks that run to completion and stop

## Compute resources

Every private service task runs with a fixed CPU and memory allocation. Stacktape supports two compute engines.

**Fargate (default)** â€” serverless containers. Specify `cpu` and `memory`; AWS manages the infrastructure. Valid CPU values are `0.25`, `0.5`, `1`, `2`, `4`, `8`, and `16` vCPU. Memory must be compatible with the CPU tier (for example, 0.25 vCPU supports 512â€“2048 MB). Set `architecture` to `arm64` to use Graviton processors at ~20% lower cost (default is `x86_64`).

**EC2** â€” specify `instanceTypes` (e.g., `t3.medium`, `c6g.large`) for more control and potentially lower cost at scale. Use `enableWarmPool` with a single instance type to keep pre-initialized instances ready for faster scaling.

## Scaling

Stacktape private services auto-scale horizontally â€” adding or removing container instances based on CPU and memory utilization. Requests are automatically distributed across all running instances.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, PrivateService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const internalApi = new PrivateService({
    packaging: new StacktapeImageBuildpackPackaging({
      entryfilePath: './src/internal-api.ts'
    }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    scaling: {
      minInstances: 2,
      maxInstances: 10,
      scalingPolicy: {
        keepAvgCpuUtilizationUnder: 70,
        keepAvgMemoryUtilizationUnder: 80
      }
    }
    // [!code focus-end]
  });

  return { resources: { internalApi } };
});
`
    }
  ]}
/>

| Property | Default | Description |
|----------|---------|-------------|
| `minInstances` | 1 | Minimum running instances. Cannot be 0 â€” Fargate does not scale to zero. |
| `maxInstances` | 1 | Maximum running instances. |
| `scalingPolicy.keepAvgCpuUtilizationUnder` | 80 | Target average CPU utilization (%). Scales out when exceeded. |
| `scalingPolicy.keepAvgMemoryUtilizationUnder` | 80 | Target average memory utilization (%). Scales out when exceeded. |

<Warning>Setting `minInstances` and `maxInstances` to the same value disables auto-scaling. You pay for all running instances even during low traffic.</Warning>

## Load balancing

The `loadBalancing` property controls how other resources in your stack reach the private service. Two modes are available, with different cost and reachability tradeoffs.

**Service Connect (default)** is the right choice for most teams. At ~$0.50/month it provides direct container-to-container connectivity with built-in load balancing and service discovery. Only container-based workloads in the same stack can reach the service â€” web services, private services, worker services, and multi-container workloads.

**Application Load Balancer** is needed when non-container VPC resources must reach the private service â€” for example, a Lambda function running inside the VPC, a [bastion host](/resources/security/bastion-host), or EC2 instances. It adds a ~$18/month base fee plus usage charges. For most internal service-to-service calls between containers, paying that base fee is unnecessary.

## Connecting to other resources

Use `connectTo` to give your private service access to other resources in your stack. Stacktape automatically grants IAM permissions, opens network access (security groups for databases and Redis), and injects environment variables with connection details. For the full list of injected variables per resource type, see [connecting resources](/configuration/connecting-resources).

When you connect to a relational database, the private service receives variables like `STP_MY_DATABASE_CONNECTION_STRING`, `STP_MY_DATABASE_HOST`, and `STP_MY_DATABASE_PORT`. For a bucket, it receives `STP_MY_BUCKET_NAME` and `STP_MY_BUCKET_ARN`.

Other resources that call this private service can reference it the same way. A web service with `connectTo: ['internalApi']` automatically receives `STP_INTERNAL_API_ADDRESS`. For permissions not covered by `connectTo` (e.g., accessing AWS Bedrock), add raw IAM statements via `iamRoleStatements`.

## Health checks

Stacktape private services support an internal container health check configured via `internalHealthCheck`. ECS runs a command inside the container and replaces it if it fails repeatedly â€” catching cases where the process is alive but not functioning correctly.

    }

Internal health check defaults: `intervalSeconds` `30` (range 5â€“300), `timeoutSeconds` `5` (range 2â€“60), `retries` `3` (range 1â€“10). The `startPeriodSeconds` grace period (0â€“300) gives the container time to initialize before health check failures count against the retries.

## Side containers

Side containers run alongside your main container within the same task. Two modes are available:

- **`run-on-init`** â€” runs to completion before the main container starts. Use for database migrations, cache warming, or setup scripts. Must exit with code 0.
- **`always-running`** â€” runs for the entire lifecycle of the task. Use for log forwarders, monitoring agents, or sidecar proxies. Can communicate with the main container via `localhost`.

## Logging

Container output (`stdout`/`stderr`) is automatically sent to CloudWatch Logs and retained for 90 days by default. View logs with [`stacktape debug:logs`](/cli/debug-logs) or in the [Stacktape Console](/stacktape-console/console-overview). Retention is configurable via `logging.retentionDays`. Set `logging.disabled: true` to stop sending logs to CloudWatch. You can also set up [log forwarding](/observability/log-forwarding) to external services like Datadog.

Set the top-level `protocol` property (`http`, `http2`, or `grpc`) to enable protocol-specific CloudWatch metrics â€” for example, HTTP 5xx error tracking for HTTP services or request count metrics for gRPC servers.

## Remote sessions

Set `enableRemoteSessions: true` to allow interactive shell access to running containers via [`stacktape container:session`](/cli/container-session). This adds a small SSM agent sidecar that uses minimal CPU and memory. Useful for debugging production issues without needing a public endpoint, but disabled by default to minimize attack surface.

## Private subnets

By default, private service containers run in public subnets with direct internet access for outbound calls. Set `usePrivateSubnetsWithNAT: true` to deploy in private subnets where all outbound traffic routes through a NAT Gateway. This gives you a static outbound IP that external services can whitelist.

<Warning>NAT Gateways add approximately ~$32/month per availability zone plus data processing fees. Configure the number of NAT Gateways (1â€“3 AZs) in `stackConfig.vpc.nat`.</Warning>

## FAQ

### How do I connect to a private service from a web service?

Add the private service to the web service's `connectTo` array: `connectTo: ['myPrivateService']`. Stacktape injects `STP_MY_PRIVATE_SERVICE_ADDRESS` into the calling resource's environment automatically. Use that variable as the base URL for HTTP or gRPC calls. No manual DNS configuration is needed â€” service discovery is handled entirely by Stacktape. See [connecting resources](/configuration/connecting-resources).

### What hostname do other containers use to reach the service?

With the default `service-connect` load balancing, the hostname is the lowercased resource name. A service named `myAuthService` is reachable at `myauthservice` on its configured `port` (default `3000`). The full internal address is `http://myauthservice:3000`. Using `connectTo` is the recommended approach â€” it injects the full address automatically so you don't need to hard-code it.

### When should I use `application-load-balancer` vs `service-connect`?

Use `service-connect` (default, ~$0.50/month) when only container-based workloads in the same stack need to reach the service. Switch to `application-load-balancer` (~$18/month) when non-container VPC resources also need access â€” such as a Lambda function running inside the VPC, a [bastion host](/resources/security/bastion-host), or EC2 instances. The cost difference makes `service-connect` the right default for most teams.

### Does a private service scale to zero?

No. Like all Fargate-based Stacktape workloads, a private service requires at least one running task. `minInstances` defaults to 1 and cannot be set to 0. You pay for compute at all times â€” typically ~$9/month for the smallest configuration. If the service is called infrequently, a [Lambda function](/resources/compute/lambda-function) scales to zero and charges only on invocation.

### How much does a private service cost to run?

The main costs are Fargate compute plus networking. A minimal 0.25 vCPU / 512 MB task costs ~$9/month in us-east-1. The default `service-connect` networking adds ~$0.50/month. Switching to an Application Load Balancer adds ~$18/month. Setting `architecture: 'arm64'` cuts Fargate compute by ~20% with no application code changes.

### Can I run a gRPC server as a private service?

Yes. Set `port` to your gRPC server's port (e.g., `50051`) and set `protocol: 'grpc'` to enable gRPC-specific CloudWatch metrics. Other containers in the stack connect using the lowercased resource name as the hostname: `grpc://[name]:50051`. The Stacktape image buildpack supports Node.js, Python, Java, and Go gRPC servers.

### How fast does a Fargate private service cold-start?

Cold-start typically takes 30â€“90 seconds from scaling event to task healthy. Image size and ECR pull time dominate. To eliminate cold-start latency entirely, keep `minInstances` at 1 or higher so there is always a warm instance available. New instances launched during scale-out take the same time, so set auto-scaling thresholds conservatively for latency-sensitive internal services.

### Private service vs worker service â€” which should I use?

Use a **private service** when other parts of your stack need to actively call it over HTTP, gRPC, or TCP â€” it accepts and routes inbound connections. Use a [worker service](/resources/compute/worker-service) when the workload only consumes messages from a queue, stream, or event source and never needs to respond to inbound requests. Worker services have no load balancing cost at all.

### Private service vs Lambda function for internal APIs â€” which is cheaper?

It depends on call volume. A Lambda function charges only when invoked and scales to zero â€” typically cheaper for infrequent calls (fewer than a few thousand per day). A private service charges for Fargate compute 24/7 regardless of traffic. For high-throughput, sustained internal traffic with consistent request rates, the flat Fargate cost beats per-invocation Lambda pricing.

### Can I expose a private service to the public internet later?

Not directly â€” there is no property on a private service that adds a public endpoint. To give a service a public URL, use a [web service](/resources/compute/web-service) instead, which is the same container model with public load balancing added. If you are unsure whether a service will eventually need a public endpoint, start with a web service and restrict access at the application layer.
