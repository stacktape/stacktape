# Custom Artifact

Custom artifact packaging lets you provide a pre-built Lambda deployment package — a zip file or directory — and Stacktape uploads it as-is. This applies only to [Lambda function](/resources/compute/lambda-function) resources. Use it when your build pipeline already produces the artifact or when the [Stacktape buildpack](/packaging/function/stacktape-buildpack) doesn't fit your workflow.

## When to use

Choose custom artifact packaging when:

- Your CI/CD pipeline already produces a deployment zip (Webpack, esbuild, custom scripts, Bazel, etc.) and you want Stacktape to deploy it without re-bundling.
- You need full control over what goes into the package — native binaries, pre-compiled assets, vendored dependencies.
- You're migrating an existing Lambda project to Stacktape and want to keep your existing build step unchanged.
- You use a language or toolchain that the Stacktape buildpack doesn't cover (e.g., Rust compiled with Cargo Lambda, C++ custom runtimes).

## When NOT to use

If you're writing TypeScript, JavaScript, Python, Java, Go, Ruby, PHP, or .NET and don't have a custom build step, the [Stacktape buildpack](/packaging/function/stacktape-buildpack) is simpler. It handles bundling, tree-shaking, source maps, and dependency installation automatically — zero configuration beyond an entry file path.

Use the buildpack if you want Stacktape to manage the build. Use custom artifact if you want to manage the build yourself.

## Basic example

The minimal configuration points `packagePath` at your pre-built artifact. Stacktape uploads it directly to Lambda.

## Configuration

### packagePath

The `packagePath` property is the only required field. It specifies the path to your pre-built deployment package, relative to the Stacktape config file.

Stacktape accepts three input shapes:

| Input | Behavior |
|-------|----------|
| A `.zip` file | Uploaded directly to Lambda |
| A directory | Automatically zipped, then uploaded |
| A non-zip file | Automatically zipped, then uploaded |

This means you don't need to zip your output manually if your build produces a directory. Point `packagePath` at the directory and Stacktape handles the rest.

### handler

The `handler` property tells Lambda which function to invoke. The syntax is `filepath:functionName` — for example, `src/index.handler` means "call the `handler` export from `src/index.js` inside the zip".

If you omit `handler`, Lambda uses its default handler resolution. Set it explicitly when your zip contains multiple files or when the entry point isn't at the root.

<CodeBlock

  const processor = new LambdaFunction({
    packaging: new CustomArtifactLambdaPackaging({

// [!code focus-end]`
    }

In this example, `packagePath` points to a directory. Stacktape zips it automatically, and Lambda invokes the `main` export from `handlers/process.js` inside the resulting package.

## Using with a custom build step

A common pattern is to run your own build command before deploying. You can wire this into Stacktape using a [hook](/configuration/hooks-and-scripts) that runs before packaging:

```bash
npm run build
```

Then point `packagePath` at the build output directory. This keeps your build tool decoupled from Stacktape — use Webpack, esbuild, Rollup, Cargo, `go build`, or any other tool.

## Directory packaging example

When your build produces a directory (common with Python virtualenvs or vendored Node.js projects), point `packagePath` at the directory root:

The directory is zipped at deploy time. The resulting zip preserves the directory's internal structure, so `handler: 'app.handler'` resolves to `app.js` (or `app.py`, depending on runtime) at the zip root.

## Comparison with Stacktape buildpack

| | Custom Artifact | Stacktape Buildpack |
|---|---|---|
| **Build responsibility** | You build the artifact | Stacktape builds from source |
| **Configuration** | `packagePath` + optional `handler` | `entryfilePath` + language config |
| **Supported languages** | Any (you control the runtime) | JS, TS, Python, Java, Go, Ruby, PHP, .NET |
| **Tree-shaking / bundling** | Your responsibility | Automatic |
| **Source maps** | Your responsibility | Automatic (JS/TS) |
| **Best for** | Custom toolchains, Rust, C++, pre-built CI artifacts | Standard app code with minimal config |

## FAQ

### When should I use custom artifact instead of the Stacktape buildpack?

Use custom artifact when your project has a build step that Stacktape's buildpack can't replicate — Rust with Cargo Lambda, C++ custom runtimes, complex Webpack configurations with plugins, or when your CI pipeline already produces a tested zip. If you're writing standard TypeScript or Python, the [buildpack](/packaging/function/stacktape-buildpack) is simpler and handles bundling automatically.

### Does Stacktape rebuild the artifact on every deploy?

No. Stacktape uploads whatever is at `packagePath` at deploy time. If the file hasn't changed since the last deploy, the upload still happens but Lambda may skip updating the function code if the hash matches. Your build step runs outside Stacktape — use a [hook](/configuration/hooks-and-scripts) or your CI pipeline to produce fresh artifacts before `stacktape deploy`.

### What's the maximum size for a Lambda deployment package?

AWS Lambda accepts deployment packages up to 50 MB (zipped) for direct upload, or up to 250 MB (unzipped) when deployed via S3. Stacktape handles the S3 upload path automatically for larger packages. If your artifact exceeds 250 MB unzipped, consider using a [container-based workload](/resources/compute/web-service) instead.

### Can I use a directory instead of a zip file?

Yes. If `packagePath` points to a directory, Stacktape automatically zips it before uploading. The internal directory structure is preserved in the resulting zip. This is convenient when your build tool outputs to a folder rather than producing a zip directly.

### How do I set the handler for Python or Go?

The `handler` property uses the format `filepath:functionName`. For Python, use `app.handler` to call the `handler` function in `app.py` at the zip root. For Go, compiled binaries typically use `bootstrap` as the handler (custom runtime). For Java, use the full class path like `com.example.Handler::handleRequest`.

### Can I use custom artifact with Lambda layers?

Yes. Custom artifact packaging controls only the function code package. Lambda layers, environment variables, memory, timeout, VPC configuration, and all other [Lambda function](/resources/compute/lambda-function) properties work the same regardless of packaging mode.

### How do I debug deployment issues with custom artifacts?

If your function fails after deploy, check that the `handler` path matches the file structure inside your zip. Unzip the artifact locally and verify the handler file exists at the expected path. Use [`stacktape debug:logs`](/cli/debug-logs) to view Lambda invocation errors, which often show "cannot find module" messages when the handler path is wrong.

### Does Stacktape cache custom artifacts between deploys?

Stacktape does not cache your custom artifact — it reads from `packagePath` on every deploy. If you want to skip unnecessary uploads, ensure your build step is idempotent and produces identical zips for unchanged source. The actual Lambda update is skipped by AWS when the code hash matches the previous deployment.
