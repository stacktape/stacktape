# Web Service

A Stacktape web service runs one container continuously with a public HTTPS URL. Use it for APIs, web apps, and internet-facing services that need TLS, health checks, auto-scaling, and deployment updates handled through the Stacktape resource instead of manually wiring container infrastructure, gateways, certificates, and load balancing.

**Pricing context:** A web service uses AWS Fargate by default. Source pricing lists about ~$7.29/month per 0.25 vCPU plus about ~$1.60/month per 512 MB in us-east-1. Idle tasks still bill because Fargate cannot scale to zero, and ARM tasks are about ~20% cheaper.

## When to use

A Stacktape web service is the right default for a single public HTTP container. Choose a web service when the container should always be reachable from the internet, needs a stable HTTPS endpoint, and benefits from Stacktape-managed TLS, load balancing, health checks, auto-scaling, custom domains, and optional CDN or firewall configuration.

Common use cases:

- **HTTP APIs** — REST, GraphQL, Hono, Express, FastAPI, Django, or similar services listening on a container port.
- **Server-rendered web apps** — web applications that run as a long-lived container instead of a framework-specific frontend resource.
- **WebSocket services** — use `application-load-balancer` load balancing when WebSocket support is required.
- **High-traffic public services** — use `application-load-balancer` when flat load balancer pricing is a better fit than per-request HTTP API pricing.
- **TCP or TLS services** — use `network-load-balancer` for non-HTTP protocols such as MQTT, game servers, or custom protocols.

## When NOT to use

A web service is not the best fit for every container. A web service keeps at least one container instance running, so workloads that only need to run on demand, should stay private, or should run without public traffic usually have better Stacktape resource choices.

- **Event-driven compute** — use a [Lambda function](/resources/compute/lambda-function) when code can run per request or per event and does not need an always-running container.
- **Internal-only services** — use a [private-service](/resources/compute/private-service) when the container should not have a public endpoint.
- **Background workers** — use a [worker-service](/resources/compute/worker-service) for always-on workers without public traffic.
- **Multiple peer containers** — use a [multi-container-workload](/resources/compute/multi-container-workload) when several containers need independent traffic routing or explicit peer-container lifecycle control.
- **Run-to-completion jobs** — use a [batch job](/resources/compute/batch-job) for workloads that start, process work, and exit.

## Basic example

A basic web service points Stacktape at the container source and declares the compute allocation. The default traffic path is `http-api-gateway`, which gives the Stacktape web service a public HTTPS URL and uses pay-per-request pricing.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({
      entryfilePath: './src/server.ts'
    }),
    resources: {
      cpu: 0.5,
      memory: 1024
    }
  });

  return {
    resources: { api }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`entryfilePath` points to the application entry point used by the Stacktape image buildpack. The `resources` block configures Fargate CPU and memory; use the smallest allocation that handles normal traffic, then raise it when CPU or memory pressure shows up in metrics.

The application must listen on the port expected by the container image or load-balancing configuration. For a typical Node.js service, that can be a fixed application port such as `3000` unless your container startup command or framework expects a different value.

```typescript
import express from 'express';

const app = express();
const port = 3000;

app.get('/health', (_req, res) => {
  res.status(200).send('ok');
});

app.get('/orders/:id', (req, res) => {
  res.json({ id: req.params.id, status: 'processing' });
});

app.listen(port, () => {
  console.log(`API listening on ${port}`);
});
```

## Traffic

A Stacktape web service supports three traffic front doors: `http-api-gateway`, `application-load-balancer`, and `network-load-balancer`. The choice affects pricing, protocol support, firewall eligibility, CDN support, and deployment options, so choose the narrowest traffic path that matches how clients connect to the container.

| Type | Use when | Tradeoff |
|------|----------|----------|
| `http-api-gateway` | Most public HTTP APIs and web apps | Default and about ~$1/million requests, but request cost grows with volume |
| `application-load-balancer` | WebSockets, gradual deployments, top-level `useFirewall`, or higher steady traffic | About ~$18/month plus usage, but more cost-effective above about 500k requests/day |
| `network-load-balancer` | TCP/TLS protocols such as MQTT, game servers, or custom protocols | Requires explicit `ports`; does not support CDN, top-level firewall, or gradual deployments |

### HTTP API Gateway

`http-api-gateway` is the default load balancing mode for a web service and is the best starting point for most HTTP APIs. HTTP API Gateway is pay-per-request, supports CORS configuration on the web service, and avoids the fixed monthly load balancer baseline that makes less sense for small or spiky services.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/server.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    loadBalancing: {
      type: 'http-api-gateway'
    }
    // [!code focus-end]
  });

  return { resources: { api } };
});`
    }
  ]}
/>

You can omit `loadBalancing` when using HTTP API Gateway, because `http-api-gateway` is the default. Keep the explicit form in shared examples or templates when you want readers to see the chosen traffic mode.

### Application Load Balancer

`application-load-balancer` is the better web service option when you need WebSockets, top-level `useFirewall`, gradual deployments, or sustained request volume where flat load balancer pricing is preferable. Stacktape documents Application Load Balancer as about ~$18/month plus usage and more cost-effective above about 500k requests per day.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/server.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    loadBalancing: {
      type: 'application-load-balancer',
      properties: {
        healthcheckPath: '/health',
        healthcheckInterval: 10,
        healthcheckTimeout: 5
      }
    }
    // [!code focus-end]
  });

  return { resources: { api } };
});`
    }
  ]}
/>

`healthcheckPath` is the path the load balancer pings; the default is `/`. `healthcheckInterval` defaults to `5` seconds, and `healthcheckTimeout` defaults to `4` seconds. Use a cheap endpoint such as `/health` that checks whether the application can accept traffic without doing expensive downstream work.

### Network Load Balancer

`network-load-balancer` is for TCP and TLS protocols rather than normal HTTP routing. Use Network Load Balancer for MQTT brokers, game servers, custom binary protocols, or cases where the public port and the container port need explicit transport-layer mapping.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const broker = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/broker.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    loadBalancing: {
      type: 'network-load-balancer',
      properties: {
        healthCheckProtocol: 'TCP',
        ports: [
          {
            port: 8883,
            protocol: 'TLS',
            containerPort: 1883
          }
        ]
      }
    }
    // [!code focus-end]
  });

  return { resources: { broker } };
});`
    }
  ]}
/>

Each NLB `ports` entry exposes a public load balancer port and forwards to `containerPort`, which defaults to the same value as `port` when omitted. `protocol` can be `TLS` or `TCP`, with `TLS` as the default. `healthCheckProtocol` can be `TCP` or `HTTP`; the default is `TCP`, and `healthcheckPath` only matters for HTTP health checks.

<Warning>`network-load-balancer` does not support CDN, top-level `useFirewall`, or gradual deployments on a web service. Use `application-load-balancer` when you need those web-specific controls.</Warning>

## Domains and CORS

A Stacktape web service can attach custom domains and configure CORS at the web service layer. Custom domains give clients a stable hostname such as `api.example.com`, while CORS is only supported with the default `http-api-gateway` load balancing mode and overrides CORS headers returned by the application.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/server.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    customDomains: [
      {
        domainName: 'api.example.com'
      }
    ],
    cors: {
      allowedOrigins: ['https://app.example.com'],
      allowedMethods: ['GET', 'POST', 'OPTIONS'],
      allowedHeaders: ['content-type', 'authorization']
    }
    // [!code focus-end]
  });

  return { resources: { api } };
});`
    }
  ]}
/>

`domainName` should not include `https://`. The domain must already exist as a Route53 hosted zone in your AWS account, and Stacktape can create DNS records and provision a free TLS certificate. Set `disableDnsRecordCreation: true` only when you manage DNS elsewhere; Stacktape still handles the certificate path described by the domain configuration.

CORS belongs in Stacktape when browser clients should receive consistent cross-origin headers regardless of application responses. Keep CORS in the application when behavior differs per route or tenant. For the broader domain model, see [custom domains](/resources/networking/custom-domains).

## CDN and firewall

A Stacktape web service can put a CDN in front of HTTP traffic and can attach a web application firewall through two distinct paths. Top-level `useFirewall` attaches a regional firewall to an Application Load Balancer, while `cdn.useFirewall` protects the CDN path with a CDN-scoped firewall.

Use a CDN when responses are cacheable, clients are globally distributed, or you want CloudFront in front of the web service. Skip a CDN for low-traffic internal APIs, strongly personalized responses, or services where every request must reach the container and edge caching adds configuration without benefit. For CDN routing and caching options, use the [CDN resource](/resources/networking/cdn) page.

`cdn.useFirewall` references a [web application firewall](/resources/security/web-application-firewall) whose `scope` is `cdn`. Do not treat this as the same constraint as top-level `useFirewall`: top-level `useFirewall` requires `application-load-balancer`, while the CDN firewall path protects the CloudFront distribution.

For an Application Load Balancer firewall, use top-level `useFirewall` with `loadBalancing.type: 'application-load-balancer'`. This is useful for public APIs that need regional WAF rules without adding a CDN in front.

<CodeBlock

  defineConfig,
  WebService,
  WebAppFirewall,

## Compute resources

A Stacktape web service supports two compute engines through `resources`: Fargate by CPU and memory, or EC2 by `instanceTypes`. Fargate is the default and the better choice for most teams because Stacktape only needs the container size; EC2 mode is for workloads that need specific instance families or sustained capacity economics.

**Fargate** — specify `cpu` and `memory`. Source pricing covers Fargate sizes from 0.25 to 16 vCPU and memory billing from 512 MB to 122880 MB, but the API reference should be treated as the source for exact accepted configuration shape. Use `architecture: 'arm64'` when your image and dependencies support ARM; source pricing notes ARM tasks are about ~20% cheaper than x86.

});`
    }

**EC2** — specify `instanceTypes` when you want service capacity on EC2 instances rather than Fargate. The source describes EC2 mode as more control and potentially lower cost, while Fargate remains the default path for most web services.

`instanceTypes` lists the EC2 instance types the web service can use. Use EC2 mode only when the extra capacity choice matters; for ordinary APIs, Fargate keeps the resource easier to operate and aligns with the default Stacktape abstraction.

<Info>Fargate cannot scale to zero. A web service with one minimum instance continues billing while idle because the container stays running.</Info>

## Scaling

A Stacktape web service scales horizontally by adding or removing running container instances. Traffic is distributed across the healthy instances, and the scaling policy can react to average CPU and memory utilization across the web service.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/server.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    scaling: {
      minInstances: 2,
      maxInstances: 10,
      scalingPolicy: {
        keepAvgCpuUtilizationUnder: 70,
        keepAvgMemoryUtilizationUnder: 80
      }
    }
    // [!code focus-end]
  });

  return { resources: { api } };
});`
    }
  ]}
/>

| Property | Default | Description |
|----------|---------|-------------|
| `minInstances` | 1 | Minimum running instances. Cannot be 0 for Fargate workloads. |
| `maxInstances` | 1 | Maximum running instances. |
| `scalingPolicy.keepAvgCpuUtilizationUnder` | 80 | Scale out when average CPU utilization exceeds this percentage. |
| `scalingPolicy.keepAvgMemoryUtilizationUnder` | 80 | Scale out when average memory utilization exceeds this percentage. |

Set `minInstances` above `1` for public services that need redundancy during deployments or traffic spikes. Keep `maxInstances` conservative until you understand cost and downstream capacity; scaling a web service horizontally can also scale database connections, queue consumers, and third-party API calls.

<Warning>Setting `minInstances` and `maxInstances` to the same value fixes capacity at that instance count. You still pay for every running instance even when traffic is low.</Warning>

## Packaging

A Stacktape web service uses container packaging, and the `packaging` property supports the same five container modes used by other container workloads. Most teams should start with the Stacktape image buildpack, then move to Dockerfile or prebuilt-image packaging when the build process needs explicit control.

| Mode | When to use |
|------|-------------|
| [Stacktape image buildpack](/packaging/containers/stacktape-buildpack) | Build an image from app source with minimal configuration |
| [Custom Dockerfile](/packaging/containers/custom-dockerfile) | Control the Dockerfile, build context, build args, command, or entry point |
| [Prebuilt image](/packaging/containers/prebuilt-image) | Use an existing image from a public or private registry |
| [Nixpacks](/packaging/containers/nixpacks) | Let Nixpacks detect the app and build the container image |
| [External buildpack](/packaging/containers/external-buildpack) | Use Cloud Native Buildpacks with a builder and optional buildpack list |

The basic example uses `StacktapeImageBuildpackPackaging` with `entryfilePath`; the source describes this mode as a zero-config buildpack for JavaScript, TypeScript, Python, Java, and Go container apps. For detailed packaging options, use the [packaging overview](/packaging/overview) and the dedicated container packaging pages rather than expanding every packaging property on this resource page.

## Environment variables

A Stacktape web service can receive explicit runtime variables through `environment`, and variables from `connectTo` are added automatically alongside them. Use explicit variables for application settings such as `NODE_ENV`, feature flags, or non-secret constants; use [secrets](/configuration/secrets) for sensitive values.

<CodeBlock
  intellisense
  tabs={[

export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/server.ts' }),

    ]
    // [!code focus-end]
  });

  return { resources: { api } };
});`
    }
  ]}
/>

Environment variable values can be strings, numbers, or booleans. Stacktape converts numbers and booleans to strings before injecting them into the container, so application code should read them as normal process environment variables.

## Connecting to other resources

A Stacktape web service can use `connectTo` to access other resources in the same stack. Stacktape grants IAM permissions where applicable, opens network access for supported private resources such as relational databases and Redis, and injects environment variables with connection details into the container.

For a connected resource named `database`, the container receives variables such as `STP_DATABASE_CONNECTION_STRING`, `STP_DATABASE_HOST`, and `STP_DATABASE_PORT`. In general, `connectTo` variables use `STP_[RESOURCE_NAME]_[PARAM]`; for example, a resource named `myDatabase` would use the `STP_MY_DATABASE_` prefix. For the complete table, see [connecting resources](/configuration/connecting-resources).

Use `connectTo` for Stacktape-managed access paths and `iamRoleStatements` only for raw IAM permissions not covered by `connectTo`, such as direct calls to AWS services outside the connected resource list. Keep explicit runtime variables in `environment`; Stacktape adds connected-resource variables automatically without replacing your own variables.

## Health checks

A Stacktape web service can use load balancer health checks and internal container health checks. Load balancer health checks control whether traffic is routed to a container instance, while internal health checks let ECS replace a container that is still running but failing its own command-based check.

For Application Load Balancer mode, configure health checks under `loadBalancing.properties`. Defaults are `healthcheckPath: '/'`, `healthcheckInterval: 5`, and `healthcheckTimeout: 4`. Network Load Balancer mode supports `TCP` or `HTTP` health checks; with `TCP`, the health check verifies the port rather than an HTTP path.

Internal health check defaults are `intervalSeconds` `30`, `timeoutSeconds` `5`, and `retries` `3`. `startPeriodSeconds` can give the application startup time before failures count. Use internal checks when the process can stay alive while the application is not actually serving requests correctly.

## Side containers

A Stacktape web service can run helper containers beside the main container through `sideContainers`. Use `run-on-init` for setup that must finish before the web service starts, and `always-running` for helper processes such as log forwarders or monitoring agents that run for the main container's lifecycle and can reach it via `localhost`.

<CodeBlock
  intellisense

`run-on-init` must exit with code 0 before the main container starts, which makes it a good fit for database migrations or setup tasks. `always-running` runs alongside the main container for its entire lifetime. Use a [multi-container-workload](/resources/compute/multi-container-workload) when helper containers need independent traffic routing or explicit peer lifecycle configuration.

## EFS storage

A Stacktape web service can mount an [EFS filesystem](/resources/storage/efs-filesystem) when data must survive container replacement. Use EFS for shared files or persistent runtime state that cannot be kept in memory, but prefer object storage or a database when the data model fits those services better.

The `volumeMounts` property accepts EFS mounts for the container. Data stored in EFS volumes persists even when containers are replaced, and the source states that data on mounted EFS volumes is encrypted in transit. EFS adds network filesystem behavior to the request path, so skip it for ordinary request-local temporary files.

## Gradual deployments

A Stacktape web service supports gradual traffic shifting through `deployment`, but the source requires `loadBalancing.type: 'application-load-balancer'`. Use gradual deployments for user-facing production services where a bad image should reach only a fraction of traffic before the deployment fully shifts.

For the deployment model and available strategies, see [gradual deployments](/deployment-and-lifecycle/gradual-deployments).

## Private subnets

A Stacktape web service can run in private subnets with NAT by setting `usePrivateSubnetsWithNAT: true`. The container no longer has a public IP, and outbound traffic routes through NAT Gateway, which gives external systems a stable IP path they can whitelist.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, WebService, StacktapeImageBuildpackPackaging } from 'stacktape';

export default defineConfig(() => {
  const api = new WebService({
    packaging: new StacktapeImageBuildpackPackaging({ entryfilePath: './src/server.ts' }),
    resources: { cpu: 0.5, memory: 1024 },
    // [!code focus-start]
    usePrivateSubnetsWithNAT: true
    // [!code focus-end]

    resources: { api },
    stackConfig: {
      vpc: {
        nat: {
          availabilityZones: 2
        }

`availabilityZones` controls how many NAT Gateways are created and can be `1`, `2`, or `3`; the default is `2`. Each NAT Gateway is described in the source as about ~$32/month plus data processing fees, so leave private subnets off unless static outbound IPs or VPC routing requirements justify the extra cost.

## Logging and sessions

Stacktape automatically sends container `stdout` and `stderr` to CloudWatch Logs and retains them for 90 days. Use the `logging` property when you need to disable logging or adjust container log behavior, and use the [logs](/observability/logs) page for day-to-day log viewing guidance.

`enableRemoteSessions: true` allows interactive shell access to running containers through [`container:session`](/cli/container-session). The source marks remote sessions as disabled by default and notes that enabling them adds a small SSM agent. Enable remote sessions for targeted debugging, then keep them off for services where reducing operational access paths matters more.

## FAQ

### What is a Stacktape web service?

A Stacktape web service is an always-running container resource with a public HTTPS URL. It is designed for APIs, web apps, and other internet-facing services that need TLS, health checks, auto-scaling, and deployment updates handled by Stacktape. For event-driven code that does not need a running container, use a [Lambda function](/resources/compute/lambda-function).

### Does a Stacktape web service scale to zero?

No. A web service uses always-on container infrastructure, and the source pricing context for Fargate explicitly says idle tasks still bill because Fargate cannot scale to zero. Use a [Lambda function](/resources/compute/lambda-function) when scale-to-zero billing is a primary requirement.

### Can I use a custom domain with a web service?

Yes. Configure `customDomains` with a domain such as `api.example.com`; the domain must exist as a Route53 hosted zone in your AWS account. Stacktape can create DNS records and provision a TLS certificate, while `disableDnsRecordCreation` is available when you manage DNS records yourself. See [custom domains](/resources/networking/custom-domains).

### Which load balancing mode should I choose?

Use `http-api-gateway` for most HTTP APIs because it is the default and costs about ~$1/million requests. Use `application-load-balancer` for WebSockets, gradual deployments, top-level `useFirewall`, or higher steady request volume. Use `network-load-balancer` only for TCP/TLS protocols that are not normal HTTP traffic.

### Can I put CloudFront in front of a web service?

Yes. Configure the web service `cdn` property to put a CDN in front of the service. Use `cdn.useFirewall` with a CDN-scoped [web application firewall](/resources/security/web-application-firewall) when the firewall should protect the CloudFront path. Do not confuse this with top-level `useFirewall`, which attaches to an Application Load Balancer.

### How much does a Fargate web service cost?

Fargate bills by vCPU and memory while the task is running, including idle time. The provided pricing source lists about ~$7.29/month per 0.25 vCPU and about ~$1.60/month per 512 MB in us-east-1, with ARM tasks about ~20% cheaper than x86. Load balancing can add request-based HTTP API charges or flat Application Load Balancer charges.

### Web service vs Lambda function: which should I use?

Use a web service when the application is naturally a long-running HTTP server, needs container-specific dependencies, or requires protocols and load balancer features that fit ECS. Use a [Lambda function](/resources/compute/lambda-function) when work is event-driven, intermittent, or benefits from scale-to-zero billing. Many teams use web services for main APIs and Lambda functions for asynchronous tasks.

### Web service vs private service: what's the difference?

A web service is internet-facing and receives a public HTTPS URL. A [private-service](/resources/compute/private-service) is for containers reachable only inside your stack or VPC. Use a private service for internal APIs, admin backends, or service-to-service traffic that should not be exposed publicly.

### Can a web service use WebSockets?

Yes, but choose `application-load-balancer` load balancing. The source describes WebSocket support as an Application Load Balancer reason, not as a default HTTP API Gateway claim. If you also need gradual deployments or top-level firewall protection, the same Application Load Balancer mode satisfies those requirements.

### How do I run database migrations for a web service?

Use a `run-on-init` side container when the migration must finish before the main container starts. The side container exits with code 0 before the main web container starts, making it suitable for schema setup or migrations packaged separately. For more complex multi-container lifecycle or independent traffic routing, use a [multi-container-workload](/resources/compute/multi-container-workload).
