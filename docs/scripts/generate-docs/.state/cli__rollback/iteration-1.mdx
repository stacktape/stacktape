# rollback

The `stacktape rollback` command reverts a deployed stack to a previous deployment version. It downloads the CloudFormation template from an earlier version and deploys it, reusing existing artifacts (Lambda zips, container images) so no rebuild is needed. This is the fastest way to undo a bad deployment.

## When to use

Use `rollback` when a recent deployment introduced a bug, performance regression, or broken configuration and you want to restore the previous working state. Unlike redeploying from an older Git commit, rollback skips packaging and uses the original artifacts directly — making it significantly faster.

<Warning>
`rollback` reverts your *infrastructure and code artifacts* to a previous deployment version. It does not revert data changes (database migrations, S3 objects written by your application, etc.). If your deployment included a database migration, rolling back the code without rolling back the schema may cause errors.
</Warning>

If your stack is stuck in an `UPDATE_FAILED` or `UPDATE_ROLLBACK_FAILED` state and cannot accept new updates, use [`cf:rollback`](/cli/cf-rollback) instead — it uses CloudFormation's native recovery mechanism.

## Usage

Roll back one version (the default):

```bash
stacktape rollback --stage production --region eu-west-1
```

Roll back to a specific version:

```bash
stacktape rollback --stage production --region eu-west-1 --targetVersion v000003
```

Roll back multiple steps:

```bash
stacktape rollback --stage production --region eu-west-1 --rollbackSteps 3
```

List available versions before deciding:

```bash
stacktape rollback --stage production --region eu-west-1 --listVersions
```

## Important flags

### `--targetVersion`

Specifies the exact deployment version to roll back to, for example `v000003`. Versions use the format `v` followed by a six-digit zero-padded number. Use `--listVersions` to see which versions are available in your deployment bucket.

### `--rollbackSteps`

Number of versions to step back from the current version. Defaults to `1` (the immediately previous deployment). For example, if the current version is `v000008`, `--rollbackSteps 3` targets `v000005`.

### `--listVersions`

When set, the command prints all available previous deployment versions and the current version, then exits without performing any rollback. Use this to inspect what's available before committing to a rollback.

### `--configPath`

The rollback command does not require a Stacktape configuration file — it works entirely from the stored CloudFormation template and artifacts. However, you can optionally provide `--configPath` if your project setup requires it for stack identification.

## How rollback works

Stacktape stores CloudFormation templates and build artifacts (Lambda zips, container images) for each deployment version in an S3 deployment bucket. When you run `rollback`:

1. **Version resolution** — the target version is determined from `--targetVersion`, or calculated by subtracting `--rollbackSteps` from the current version.
2. **Artifact verification** — Stacktape checks that the CloudFormation template, Lambda artifacts, and container images for the target version still exist in the deployment bucket.
3. **Safety check** — the command inspects rollback safety metadata stored during the original deployment. It warns about situations where the rollback may not fully restore the previous state (see [Safety warnings](#safety-warnings) below).
4. **Template preparation** — the old CloudFormation template is downloaded, its version output is updated to a new version number, and re-uploaded. This preserves the deployment version history.
5. **Deployment** — CloudFormation applies the old template. Since artifacts already exist, no packaging or uploading occurs.
6. **Bucket content restoration** — if the stack includes static hosting with bucket-synced content, Stacktape attempts to restore the bucket contents from a version manifest.

<Info>
Each rollback creates a new deployment version. Rolling back from `v000008` to `v000005` produces `v000009` — so you maintain a full audit trail and can roll forward again if needed.
</Info>

## Safety warnings

The rollback command checks metadata from the original deployment and warns about situations where a template-only rollback may not fully restore the previous behavior:

- **`$File` or other state-embedding directives** — directives like `$File` embed local file contents at deploy time. The rolled-back template contains the values from the *original* deployment, which may differ from your current local files. This is expected and usually correct for a rollback.
- **TypeScript transforms** — if your config uses TypeScript transforms that generate dynamic values, those values are not captured in the CloudFormation template and will not be re-evaluated during rollback.
- **`after:deploy` hooks** — lifecycle hooks that run after deployment are not re-executed during rollback. If the original deployment relied on post-deploy scripts (e.g., database seeding), you may need to run them manually.
- **Custom directives** — custom directive results were baked into the template at deploy time and will not be re-evaluated.

If the stack was deployed before rollback support was added, the safety metadata may be missing. The rollback still proceeds, but with a warning about potential unpredictability.

## Version retention

Old deployment versions are automatically cleaned up after successful deployments based on the `previousVersionsToKeep` setting in your deployment configuration. If a version has been cleaned up, rollback to that version is not possible — `--listVersions` shows only versions that still have their artifacts available.

<Tip>
If you anticipate needing to roll back further than the default retention window, increase `previousVersionsToKeep` in your configuration before deploying.
</Tip>

## Examples

### Quick rollback after a bad deploy

The most common use case — revert to the version that was running before the last deployment:

```bash
stacktape rollback --stage production --region eu-west-1
```

This is equivalent to `--rollbackSteps 1`. No config file is needed.

### Inspect available versions first

Before rolling back, check which versions are available:

```bash
stacktape rollback --stage production --region eu-west-1 --listVersions
```

The output shows each stored version and marks which one is currently deployed. Pick a version and roll back to it:

```bash
stacktape rollback --stage production --region eu-west-1 --targetVersion v000005
```

### Rollback in CI/CD

Use `--agent` for machine-readable output in automated pipelines:

```bash
stacktape rollback --stage production --region eu-west-1 --rollbackSteps 1 --agent
```

The `--agent` flag switches output to JSONL format, disables interactive prompts, and auto-confirms the operation.

## FAQ

### What is the difference between `rollback` and `cf:rollback`?

The `rollback` command deploys a previously stored CloudFormation template as a new version — it is for reverting a *successful* deployment that introduced a problem. The [`cf:rollback`](/cli/cf-rollback) command uses CloudFormation's native rollback mechanism to recover a stack stuck in `UPDATE_FAILED` or `UPDATE_ROLLBACK_FAILED` state. Use `rollback` for "the deploy worked but the app is broken" and `cf:rollback` for "the deploy itself failed."

### Does rollback rebuild my code?

No. Rollback reuses the Lambda zips and container images that were uploaded during the original deployment. This makes rollback faster than redeploying from source, since it skips packaging entirely.

### Does rollback affect my database?

Rollback reverts infrastructure and code artifacts only. It does not undo database migrations, data changes, or any other side effects of the original deployment. If your deployment included a schema migration, you may need to manually reconcile the database state after rolling back.

### How many versions can I roll back?

This depends on your `previousVersionsToKeep` setting. Old deployment artifacts are cleaned up after successful deployments. Use `--listVersions` to see which versions still have their artifacts available in the deployment bucket.

### Can I roll forward after a rollback?

Yes. Each rollback creates a new deployment version (e.g., rolling back from `v000008` to `v000005` produces `v000009`). You can deploy again normally, or roll back the rollback by targeting the version you came from.

### Will my `after:deploy` hooks run during rollback?

No. Lifecycle hooks configured with `after:deploy` are not re-executed during rollback. If your deployment relies on post-deploy scripts (database seeding, cache warming, etc.), you need to run them manually after the rollback. The command warns you about this when applicable.

### Does rollback restore static website content?

Stacktape attempts to restore bucket-synced content (static websites) from a version manifest stored during the original deployment. If the manifest is not available, the command warns you and suggests redeploying from the original commit to restore static content.

### Can I use rollback without a config file?

Yes. The `rollback` command works entirely from stored CloudFormation templates and deployment artifacts — no local configuration file is required. You only need `--stage` and `--region` to identify the stack.
