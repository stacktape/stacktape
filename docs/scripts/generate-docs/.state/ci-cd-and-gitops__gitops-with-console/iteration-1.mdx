# GitOps with Console

Stacktape Console can automatically deploy your stack whenever you push to a branch or open a pull request. You connect a Git provider, map branches to stages, and every matching push triggers a full `stacktape deploy` on your AWS account — no CI pipeline to maintain. PR preview environments get their own isolated stage and are optionally torn down when the PR closes.

## When to use

GitOps with Console is the fastest path from "code pushed" to "stack deployed" for most teams. Use it when:

- You want push-to-deploy without writing CI/CD pipeline files. The Console handles webhook registration, runner provisioning, and deployment orchestration.
- You need per-PR preview environments so reviewers can test changes against a real stack before merging.
- You want a single pane of glass for deployment history, logs, and cost — all tied back to the Git commit that triggered them.

## When NOT to use

GitOps with Console runs `stacktape deploy` for you. If your deployment workflow requires custom build steps, test suites, approval gates, or multi-step pipelines, use [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) instead. You can still use Stacktape CLI commands inside GitHub Actions, GitLab CI, or any other pipeline runner. See the [CI/CD overview](/ci-cd-and-gitops/overview) for a comparison of all options.

## Prerequisites

Before creating a GitOps configuration, you need two things connected in the Stacktape Console:

1. **A Git provider installation.** Stacktape supports GitHub, GitLab, and Bitbucket. Install the Stacktape GitHub App (or connect GitLab/Bitbucket via OAuth) at the organization level. Once installed, all repositories in that installation are available for GitOps.
2. **A connected AWS account.** Each GitOps configuration deploys to a specific [AWS account and region](/stacktape-console/connecting-your-aws-account). You must have at least one AWS account connected before creating deployment rules.

Your Stacktape project must also be linked to a repository through its Git provider installation. The Console uses this link to list available branches and config files.

## Push-to-deploy

Push-to-deploy is the most common GitOps pattern. Every push to a specified branch triggers a deployment to a fixed stage.

### How push-to-deploy works

1. You create a GitOps configuration that watches a branch (e.g. `main`) and maps it to a stage name (e.g. `production`).
2. When you push commits to that branch, the Console receives a webhook event from your Git provider.
3. The Console checks out your code at the pushed commit, resolves your Stacktape configuration, and runs `stacktape deploy` on the configured AWS account and region.
4. If the stage already exists, the deployment updates it. If the stage does not exist yet, a new stack is created.

You can create multiple push-to-deploy configurations for the same project — for example, `main` → `production` and `develop` → `staging`, each targeting different AWS accounts or regions.

### Configuration source

When creating a GitOps configuration, you choose where the Stacktape config comes from:

| Source | When to use |
|--------|-------------|
| **Git repository** | Your `stacktape.ts` or `stacktape.yml` lives in the repo alongside your application code. You specify the path to the config file (e.g. `stacktape.ts`). This is the standard approach — config changes are versioned with your code. |
| **Stacktape Console** | You use a template saved in the Console's [visual config editor](/stacktape-console/visual-config-editor). Useful when non-developers manage infrastructure or when you want to decouple config changes from code changes. |

Most teams should use the Git repository source. It keeps your infrastructure definition versioned, reviewable, and co-located with application code.

## PR preview environments

PR preview environments give every pull request its own isolated stage. Reviewers can test the exact changes in a real AWS environment before merging.

### How PR previews work

1. You create a GitOps configuration with the "Pull request opened" trigger and select the target branch (e.g. `main` — the branch PRs are opened against).
2. When a PR is opened against that branch, the Console automatically deploys a new stage named `pr-{number}` (e.g. `pr-42`).
3. Every subsequent push to the PR branch triggers a redeployment of that `pr-{number}` stage, so the preview stays current with the latest code.
4. When the PR is closed or merged, the Console can optionally delete the preview stage and its AWS resources.

<Info>
The `pr-{number}` stage naming convention is automatic. You do not need to configure stage names for PR preview environments — the Console derives the name from the pull request number.
</Info>

### Auto-delete on PR close

When creating a PR preview configuration, you can enable **"Delete stack after pull request is merged or closed"**. When enabled, the Console automatically runs `stacktape delete` on the `pr-{number}` stage as soon as the PR is closed or merged. This prevents preview stages from accumulating and incurring ongoing AWS costs.

When disabled, the preview stage persists after the PR closes. You would need to delete it manually through the Console or CLI. Enable auto-delete unless you have a specific reason to keep preview stages around after merge (e.g. post-merge QA on the deployed preview).

<Tip>
For a deeper look at the per-PR stage pattern — including naming conventions, cost management, and cleanup strategies — see [Stacks per Git branch pattern](/ci-cd-and-gitops/stacks-per-git-branch-pattern).
</Tip>

## Build runners

Every GitOps deployment needs a compute environment to run `stacktape deploy`. The Console supports two runner types:

| Runner | How it works | Best for |
|--------|-------------|----------|
| **CodeBuild** (default) | The Console downloads your repository, uploads it to S3, and starts an AWS CodeBuild project that runs the deployment. CodeBuild provisions a fresh container for each build. | Most teams. No persistent infrastructure to manage. Pay only for build minutes. |
| **EC2 runner** | The Console provisions a persistent EC2 instance and executes the deployment via AWS Systems Manager (SSM). The instance stays running between deployments. | Teams with large monorepos, heavy Docker builds, or dependencies that benefit from a warm cache. Faster iteration because the instance retains build artifacts between runs. |

The runner type is configured at the project level, not per GitOps configuration. All GitOps deployments for a project use the same runner type. For a detailed comparison of runners, see [Build runners](/ci-cd-and-gitops/build-runners).

## Creating a GitOps configuration

Each GitOps configuration in the Console specifies:

- **Deployment trigger** — "Push to branch" or "Pull request opened".
- **Branch** — The branch to watch. For push-to-deploy, pushes to this branch trigger deployment. For PR previews, pull requests targeting this branch trigger deployment.
- **Stage name** — The stage to deploy to (push-to-deploy only). PR previews auto-name stages as `pr-{number}`.
- **AWS account** — The [connected AWS account](/stacktape-console/connecting-your-aws-account) where the stack is deployed.
- **AWS region** — The region for the deployment (e.g. `eu-west-1`, `us-east-1`).
- **Configuration source** — Whether to use a config file from the Git repository or a template stored in the Console.
- **Delete on PR close** — Whether to auto-delete the preview stage when the PR is closed (PR previews only).

You can create multiple configurations per project. A typical setup might include:

- `main` → `production` (push-to-deploy)
- `develop` → `staging` (push-to-deploy)
- PRs targeting `main` → `pr-{number}` (PR preview with auto-delete)

<Warning>
Stage names must be at most 10 characters. Choose concise names like `prod`, `staging`, or `dev`. The `pr-{number}` pattern used by PR previews stays within this limit for PR numbers under 1,000,000.
</Warning>

For PR preview environments, the Console also creates a temporary internal configuration that watches for subsequent pushes to the PR branch. This temporary configuration is automatically cleaned up when the PR closes — you do not need to manage it.

### Deployment identity

When a push triggers a deployment, the Console attempts to match the pusher's email to a Stacktape user in your organization. If found, the deployment runs under that user's identity. If no matching user is found, the deployment runs under the organization's service account. This affects audit trails and deployment history attribution.

## Managing GitOps configurations

The GitOps page in the Console lists all active configurations for a project, showing each configuration's trigger type, branch, stage, region, and AWS account. From this page you can:

- **Create** a new GitOps configuration.
- **Delete** a configuration you no longer need. Deleting a configuration stops future webhook-triggered deployments but does not delete any currently deployed stages.

<Info>
Deleting a GitOps configuration does not delete the deployed stage. If you have a `main` → `production` push-to-deploy config and you delete the config, your `production` stage keeps running. To remove the stage, use [`stacktape delete`](/cli/delete) or delete it through the Console.
</Info>

## FAQ

### How do I set up push-to-deploy with Stacktape?

In the Stacktape Console, navigate to your project's GitOps page and create a new configuration. Select "Push to branch" as the trigger, choose the branch to watch, set a stage name, pick your AWS account and region, and select a configuration source. Every push to that branch will now trigger a deployment automatically.

### Can I use GitLab or Bitbucket instead of GitHub?

Yes. Stacktape supports GitHub, GitLab, and Bitbucket as Git providers. Connect your provider at the organization level in the Console, then link it to your project. The GitOps workflow — push-to-deploy and PR previews — works the same way across all three providers.

### How do PR preview environments get cleaned up?

When you create a PR preview configuration, you can enable "Delete stack after pull request is merged or closed." When enabled, the Console automatically runs `stacktape delete` on the preview stage as soon as the PR closes. When disabled, the preview stage persists until you delete it manually. Enable auto-delete to avoid accumulating unused AWS resources. See [Stacks per Git branch pattern](/ci-cd-and-gitops/stacks-per-git-branch-pattern) for more cleanup strategies.

### What happens if a GitOps deployment fails?

The deployment status is tracked in the Console's deployment history for the project. For PR previews, the Console posts a status comment on the pull request. You can view deployment logs in the Console to diagnose failures. Common causes include invalid Stacktape configuration, missing secrets, or AWS permission issues on the connected account.

### CodeBuild vs EC2 runner — which should I choose?

CodeBuild is the default and the right choice for most teams. It provisions a fresh container per build, requires no persistent infrastructure, and you pay only for build minutes. Choose EC2 runners if you have large monorepos, heavy Docker image builds, or dependencies that benefit from a warm cache across deployments. EC2 runners are faster for repeated builds but cost more because the instance runs continuously. See [Build runners](/ci-cd-and-gitops/build-runners) for a detailed comparison.

### How much does CodeBuild cost for GitOps deployments?

AWS CodeBuild pricing is based on build minutes and compute type. The AWS free tier includes 100 build minutes per month on the smallest instance. Beyond that, pricing varies by compute type and region. A typical Stacktape deployment takes 2–8 minutes depending on stack complexity and packaging. Check the [AWS CodeBuild pricing page](https://aws.amazon.com/codebuild/pricing/) for current rates.

### Can I deploy to multiple AWS accounts from the same repository?

Yes. Each GitOps configuration targets a specific connected AWS account and region. You can create multiple configurations for the same branch deploying to different accounts — for example, `main` → `production` on your production AWS account and `main` → `staging` on a separate staging account.

### How is GitOps with Console different from custom CI/CD?

GitOps with Console is a managed, zero-configuration deployment pipeline. You configure triggers in the Console UI and Stacktape handles webhook registration, runner provisioning, and deployment execution. [Custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) means you write your own pipeline files (GitHub Actions, GitLab CI, etc.) and call Stacktape CLI commands as steps. Custom CI/CD gives you full control over build steps, test suites, and approval gates. GitOps with Console is faster to set up but less flexible.

### Can I run tests before deploying with GitOps?

GitOps with Console runs `stacktape deploy` directly — there is no built-in test step. If you need to run tests, linting, or other checks before deploying, use [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) where you control every pipeline step. Alternatively, use your Git provider's native CI (GitHub Actions, GitLab CI) for testing and rely on branch protection rules to prevent merging broken code into the branch that triggers GitOps deployment.

### Do I need to install anything in my repository to use GitOps?

No changes to your repository are needed. The Stacktape GitHub App (or GitLab/Bitbucket OAuth integration) is installed at the organization level and receives webhook events automatically. Your repository just needs a valid Stacktape configuration file — or you can use a template stored in the Console instead.
