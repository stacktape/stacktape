# GitOps with Console

Stacktape Console can automatically deploy your stack whenever you push to a branch or open a pull request. You connect GitHub, map branches to stages, and every matching push triggers a deployment on your AWS account — no CI pipeline to maintain. PR preview environments get their own isolated stage and are torn down when the PR closes.

## When to use

GitOps with Console is the fastest path from "code pushed" to "stack deployed" for most teams. Use it when:

- You want push-to-deploy without writing CI/CD pipeline files. The Console handles webhook registration, runner provisioning, and deployment orchestration.
- You need per-PR preview environments so reviewers can test changes against a real stack before merging.
- You want deployment history tied back to the Git commit that triggered each deployment, visible in the Console.

## When NOT to use

GitOps with Console triggers a deployment for every matching Git event. If your workflow requires custom build steps, test suites, approval gates, or multi-step pipelines before deploying, use [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) instead. You can call Stacktape CLI commands as steps inside GitHub Actions, GitLab CI, or any other pipeline runner. See the [CI/CD overview](/ci-cd-and-gitops/overview) for a comparison of all options.

If you only need to deploy occasionally or manually, running [`stacktape deploy`](/cli/deploy) from the CLI or a one-off CI job is simpler than setting up GitOps.

## Prerequisites

Before creating a GitOps configuration, you need:

1. **A connected Git provider.** The provided GitOps webhook integration covers the GitHub App. Install the Stacktape GitHub App at the organization level in the Console. Once installed, repositories in that installation are available for GitOps.
2. **A connected AWS account.** Each GitOps configuration deploys to a specific [AWS account and region](/stacktape-console/connecting-your-aws-account). You must have at least one AWS account connected before creating deployment rules.
3. **A linked repository.** Your Stacktape project must be linked to a repository through the Git provider installation. The Console uses this link to list available branches.

<Info>
The Console also supports GitLab and Bitbucket as Git providers. The specific connection and webhook mechanics may differ from the GitHub App flow described on this page.
</Info>

## Push-to-deploy

Push-to-deploy is the most common GitOps pattern. Every push to a specified branch triggers a deployment to a fixed stage name.

### Setting up push-to-deploy

You create a GitOps configuration in the Console that maps a branch to a stage:

1. Navigate to your project's GitOps page and click **New GitOps configuration**.
2. Select **Push to Branch** as the deployment trigger.
3. Choose the branch to watch (e.g. `main`).
4. Set a stage name (e.g. `production`).
5. Select the target AWS account and region.

When the GitHub App receives a push event for that branch, the Console determines which GitOps configurations match and starts a deployment for each one.

You can create multiple push-to-deploy configurations for the same project — for example, `main` → `production` and `develop` → `staging`, each targeting different AWS accounts or regions.

<Warning>
Stage names must be at most 10 characters. Choose concise names like `prod`, `staging`, or `dev`.
</Warning>

## PR preview environments

PR preview environments give every pull request its own isolated stage. Reviewers can test the exact changes in a real AWS environment before merging.

### Setting up PR previews

1. Create a GitOps configuration with the **PR Opened** trigger and select the target branch (e.g. `main` — the branch PRs are opened against).
2. When a PR is opened against that branch, the Console automatically deploys a new stage. The stage name defaults to `pr-{#number}` (e.g. `pr-42`) when no explicit stage name is set.
3. The GitHub App handler creates a temporary internal configuration that watches for subsequent pushes to the PR branch, so the preview stays current with the latest code.
4. When the PR is closed or merged, the temporary configuration is automatically cleaned up.

### Auto-delete on PR close

When configuring PR previews, you can enable deletion of the preview stack after the PR is merged or closed. When this option is set, the Console runs a stack deletion on the `pr-{#number}` stage as soon as the PR closes. This prevents preview stages from accumulating and incurring ongoing AWS costs.

When auto-delete is not enabled, the preview stage persists after the PR closes and must be deleted manually through the Console or with [`stacktape delete`](/cli/delete). Enable auto-delete unless you have a specific reason to keep preview stages around after merge (e.g. post-merge QA).

<Tip>
For a deeper look at the per-PR stage pattern — including naming conventions, cost management, and cleanup strategies — see [Stacks per Git branch pattern](/ci-cd-and-gitops/stacks-per-git-branch-pattern).
</Tip>

## Build runners

Every GitOps deployment needs a compute environment to run the deployment. The Console supports two runner types — **CodeBuild** and **EC2 runners**. CodeBuild is the default and the right choice for most teams: it requires no persistent infrastructure and you pay only for build minutes. EC2 runners keep a persistent instance running between deployments, which benefits teams with large monorepos or heavy Docker builds that gain from a warm cache.

For a detailed comparison of runner types, configuration options, and cost tradeoffs, see [Build runners](/ci-cd-and-gitops/build-runners).

## How the webhook flow works

Understanding the event flow helps with debugging when a push doesn't trigger a deployment.

1. **Git push or PR event** — Your Git provider sends a webhook to the Stacktape Console's GitHub App handler.
2. **Event matching** — The handler determines which GitOps configurations match the event based on branch and trigger type. A single event can match multiple configurations.
3. **Deployment or deletion started** — For each matching configuration, the handler starts a deployment. For PR close events where auto-delete is enabled, it starts a stack deletion instead.
4. **PR-specific bookkeeping** — When a PR is opened, the handler creates a temporary GitOps configuration that watches for subsequent pushes to the PR branch. When a PR is closed, this temporary configuration is cleaned up automatically.
5. **Status tracking** — Deployment progress and results are visible in the Console's deployment history for the project.

<Info>
The temporary configuration created for PR pushes is filtered out of the GitOps configuration list in the Console — you only see the configurations you explicitly created.
</Info>

## Managing GitOps configurations

The GitOps page in the Console lists all active configurations for a project. Each row shows:

| Column | Description |
|--------|-------------|
| **Deployment trigger** | Either **Push to Branch** or **PR Opened**. |
| **From branch** | The branch being watched. |
| **Stage name** | The target stage name, or `pr-{#number}` for PR preview configurations. |
| **Region** | The AWS region for deployments. |
| **AWS Account** | The connected AWS account (name and account ID). |

From this page you can:

- **Create** a new configuration by clicking **New GitOps configuration**.
- **Delete** a configuration you no longer need.

<Warning>
Deleting a GitOps configuration stops future webhook-triggered deployments but does not delete any currently deployed stages. If you have a `main` → `production` push-to-deploy config and you delete the config, your `production` stage keeps running. To remove the stage, use [`stacktape delete`](/cli/delete) or delete it through the Console.
</Warning>

## Typical setup

A common multi-environment GitOps setup for a single project:

| Configuration | Trigger | Branch | Stage | Purpose |
|---------------|---------|--------|-------|---------|
| Production | Push to Branch | `main` | `prod` | Deploys on every merge to main |
| Staging | Push to Branch | `develop` | `staging` | Deploys on every push to develop |
| PR Previews | PR Opened | `main` | `pr-{#number}` | Creates a preview for every PR targeting main, auto-deletes on close |

This gives you continuous deployment to staging on every push, production deployment on merge to main, and isolated preview environments for every PR — all without a single pipeline file.

## FAQ

### How do I set up push-to-deploy with Stacktape?

In the Stacktape Console, navigate to your project's GitOps page and create a new configuration. Select "Push to Branch" as the trigger, choose the branch to watch, set a stage name, and pick your AWS account and region. Every push to that branch will trigger a deployment automatically. No pipeline files or CI configuration is needed.

### How do PR preview environments get cleaned up?

When you create a PR preview configuration, you can enable auto-deletion on PR close. When enabled, the Console starts a stack deletion as soon as the PR is closed or merged. The temporary internal configuration that watches for pushes to the PR branch is also cleaned up automatically. Enable auto-delete to avoid accumulating unused AWS resources. See [Stacks per Git branch pattern](/ci-cd-and-gitops/stacks-per-git-branch-pattern) for more cleanup strategies.

### What happens if a GitOps deployment fails?

Deployment status is tracked in the Console's deployment history for the project. You can view deployment details and logs in the Console to diagnose failures. Common causes include invalid Stacktape configuration, missing [secrets](/configuration/secrets), or AWS permission issues on the connected account. The failed deployment does not affect your currently running stage — the existing stack remains unchanged.

### CodeBuild vs EC2 runner — which should I choose?

CodeBuild is the default and the right choice for most teams. It requires no persistent infrastructure and you pay only for build minutes used. Choose EC2 runners if you have large monorepos, heavy Docker image builds, or dependencies that benefit from a warm cache across deployments. EC2 runners are faster for repeated builds but cost more because the instance runs between deployments. See [Build runners](/ci-cd-and-gitops/build-runners) for a detailed comparison.

### How much does AWS CodeBuild cost for GitOps deployments?

AWS CodeBuild pricing is based on build minutes and compute type. The AWS free tier includes 100 build minutes per month on the smallest instance type. Beyond that, pricing varies by compute type and region. A typical Stacktape deployment takes a few minutes depending on stack complexity and packaging mode. Check the [AWS CodeBuild pricing page](https://aws.amazon.com/codebuild/pricing/) for current rates.

### Can I deploy to multiple AWS accounts from the same repository?

Yes. Each GitOps configuration targets a specific connected AWS account and region. You can create multiple configurations for the same branch deploying to different accounts — for example, `main` → `prod` on your production AWS account and `main` → `staging` on a separate staging account. Each configuration is independent and can target any region.

### How is GitOps with Console different from custom CI/CD?

GitOps with Console is a managed deployment pipeline. You configure triggers in the Console and Stacktape handles webhook registration, runner provisioning, and deployment execution. [Custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) means you write your own pipeline files (GitHub Actions, GitLab CI, etc.) and call Stacktape CLI commands as steps. Custom CI/CD gives you full control over build steps, test suites, and approval gates. GitOps with Console is faster to set up but does not support pre-deployment testing or approval workflows.

### Can I run tests before deploying with GitOps?

GitOps with Console does not include a built-in test step. If you need to run tests, linting, or other checks before deploying, use [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) where you control every pipeline step. Alternatively, use your Git provider's native CI for testing and rely on branch protection rules to prevent merging broken code into the branch that triggers GitOps deployment.

### Can I use Stacktape-managed GitHub Actions runners with GitOps?

These are separate features. GitOps with Console is a fully managed deployment pipeline triggered by webhooks. [Self-hosted GitHub Actions runners](/ci-cd-and-gitops/self-hosted-github-actions-runners) let you run your own GitHub Actions workflows on Stacktape-managed infrastructure. If you want the full flexibility of GitHub Actions but with managed runners, use self-hosted runners with a [custom CI/CD](/ci-cd-and-gitops/custom-ci-cd) workflow instead of Console GitOps.

### Do I need to install anything in my repository to use GitOps?

No changes to your repository code are needed. The Stacktape GitHub App is installed at the organization level and receives webhook events automatically. Your repository needs a valid Stacktape configuration file (a `stacktape.ts` or `stacktape.yml` in the repo). The GitOps configuration in the Console points to the branch and your project handles the rest.
