# Connecting Resources

The `connectTo` property is how resources in a Stacktape stack communicate with each other. When you add a resource name to a workload's `connectTo` list, Stacktape automatically grants IAM permissions, opens network access via security groups, and injects environment variables with connection details — replacing dozens of lines of manual AWS configuration with a single reference.

## When to use

Use `connectTo` whenever one resource needs to interact with another. A [Lambda function](/resources/compute/lambda-function) that queries a database, a [web service](/resources/compute/web-service) that publishes to a queue, a [deployment script](/resources/advanced/deployment-scripts) that runs migrations — all of these are `connectTo` use cases. It is the primary mechanism for wiring resources together and should be your default choice over manual IAM and environment variable configuration.

## When NOT to use

Skip `connectTo` when you need narrow, specific permissions that differ from the defaults. For example, if a function should only read from a bucket but never write, `connectTo` grants full read/write access — use `iamRoleStatements` instead. You can also combine both: use `connectTo` for env vars and network access, then override permissions with custom IAM statements.

## Basic example

<CodeBlock
  intellisense
  tabs={[

// [!code focus-start]
export default defineConfig(() => {
  const myDatabase = new RelationalDatabase({
    credentials: {
      masterUserPassword: '$Secret(\\'database.password\\')'
    },
    engine: new RdsEnginePostgres({
      version: '16',
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    })
  });

  const uploads = new Bucket({});

  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),

  };
});
// [!code focus-end]`
    }
  ]}
/>

The `connectTo: ['myDatabase', 'uploads']` line replaces manual IAM policies, security group rules, VPC configuration, and environment variable setup. The Lambda function gets full access to the database and the bucket, with connection details injected as `STP_*` environment variables.

## What connectTo does

When you add a resource name to `connectTo`, Stacktape performs three actions automatically:

1. **Grants IAM permissions** — the consuming resource's IAM role gets a policy allowing it to interact with the target resource (e.g., `s3:GetObject`, `dynamodb:PutItem`, `sqs:SendMessage`).
2. **Opens network access** — for VPC-bound resources like databases and Redis, Stacktape creates security group rules allowing the consuming resource to reach the target on the correct port.
3. **Injects environment variables** — all referenceable parameters of the target resource are injected as `STP_[RESOURCE_NAME]_[PARAM]` environment variables into the consuming resource.

## Environment variable naming

Stacktape converts the resource name and parameter name to `UPPER_SNAKE_CASE` and joins them with the `STP_` prefix:

```
STP_[RESOURCE_NAME]_[PARAMETER_NAME]
```

For example, a resource named `myDatabase` with parameter `connectionString` becomes `STP_MY_DATABASE_CONNECTION_STRING`. A resource named `uploads` with parameter `name` becomes `STP_UPLOADS_NAME`.

CamelCase resource names are split at word boundaries: `jobQueue` → `JOB_QUEUE`, `usersTable` → `USERS_TABLE`.

## Using injected variables in code

Access the `STP_*` variables through `process.env` in your application code:

```typescript
import { Client } from 'pg';

export const handler = async () => {
  const client = new Client({
    connectionString: process.env.STP_MY_DATABASE_CONNECTION_STRING
  });

  await client.connect();
  const result = await client.query('SELECT NOW()');
  await client.end();

  return { statusCode: 200, body: JSON.stringify(result.rows) };
};
```

For S3 operations, use the injected bucket name:

```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';

const s3 = new S3Client({});

export const handler = async () => {
  await s3.send(new PutObjectCommand({
    Bucket: process.env.STP_UPLOADS_NAME,
    Key: 'hello.txt',
    Body: 'Hello, world!'
  }));

  return { statusCode: 200, body: 'Uploaded' };
};
```

<Tip>
IAM permissions from `connectTo` are applied to the resource's execution role automatically. AWS SDK calls (like the S3 example above) work without explicit credentials — the SDK picks up the role's temporary credentials from the environment.
</Tip>

## IAM permissions granted

Each resource type gets a different set of IAM actions. These are broad enough for typical usage:

| Target resource | Permissions granted |
|---|---|
| [Bucket](/resources/storage/s3-bucket) | `s3:ListBucket`, `s3:GetBucket*`, `s3:*Object*`, `s3:Abort*`, `s3:*MultipartUpload*` |
| [DynamoDB table](/resources/databases/dynamodb) | `dynamodb:BatchGet*`, `dynamodb:Get*`, `dynamodb:Query`, `dynamodb:Scan`, `dynamodb:BatchWrite*`, `dynamodb:PutItem`, `dynamodb:UpdateItem`, `dynamodb:DeleteItem`, `dynamodb:ConditionCheckItem`, `dynamodb:DescribeTable`, `dynamodb:DescribeStream` |
| [SQS queue](/resources/messaging/sqs-queue) | `sqs:SendMessage`, `sqs:ReceiveMessage`, `sqs:DeleteMessage`, `sqs:ChangeMessageVisibility`, `sqs:GetQueueAttributes`, `sqs:GetQueueUrl`, `sqs:PurgeQueue` |
| [SNS topic](/resources/messaging/sns-topic) | `sns:Publish`, `sns:Subscribe`, `sns:Unsubscribe`, `sns:ConfirmSubscription`, `sns:GetTopicAttributes`, `sns:ListSubscriptionsByTopic` |
| [Event bus](/resources/messaging/event-bus) | `events:PutEvents` |
| [Lambda function](/resources/compute/lambda-function) | `lambda:InvokeFunction` |
| [Batch job](/resources/compute/batch-job) | `batch:SubmitJob`, `batch:DescribeJobs`, `batch:TerminateJob`, `batch:ListJobs`, `states:StartExecution`, `states:StopExecution`, `states:ListExecutions`, `states:DescribeExecution` |
| [State machine](/resources/orchestration/state-machine) | `states:StartExecution`, `states:StopExecution`, `states:ListExecutions`, `states:DescribeExecution` |

| [OpenSearch domain](/resources/databases/opensearch) | `es:ESHttp*` |
| [Kinesis stream](/resources/messaging/kinesis-stream) | `kinesis:DescribeStream`, `kinesis:GetRecords`, `kinesis:GetShardIterator`, `kinesis:ListShards`, `kinesis:PutRecord`, `kinesis:PutRecords` |

| [Redis cluster](/resources/databases/redis) | Network access (security group rules) |

Databases and Redis are network-bound resources — access is controlled through VPC security groups rather than IAM policies. Stacktape opens the correct port between the consuming resource and the database automatically.

## AWS service macros

In addition to stack resources, `connectTo` supports the `aws:ses` macro for granting AWS SES (Simple Email Service) permissions without a dedicated resource:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const emailSender = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/send-email.ts'
    }),
    connectTo: ['aws:ses']
  });

  return {
    resources: { emailSender }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The `aws:ses` macro grants full SES permissions (`ses:*`) to the consuming resource. No environment variables are injected — you use the AWS SDK directly with the SES service.

## Custom permissions with iamRoleStatements

When `connectTo` grants more permissions than you need, or you need access to an AWS service not covered by `connectTo`, use `iamRoleStatements` for fine-grained control:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging, Bucket } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const uploads = new Bucket({});

  const reader = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/reader.ts'
    }),
    connectTo: ['uploads'],
    iamRoleStatements: [
      {
        Resource: ["$ResourceParam('uploads', 'arn')/*"],
        Action: ['s3:GetObject', 's3:ListBucket']
      }
    ]
  });

  return {
    resources: { uploads, reader }
  };
});
// [!code focus-end]`
    }
  ]}
/>

You can use `connectTo` and `iamRoleStatements` together. The `connectTo` handles env var injection and network access, while `iamRoleStatements` adds or narrows specific permissions. Custom IAM statements are added as a separate policy alongside the auto-generated one.

## Scripts with connectTo

[Scripts](/configuration/hooks-and-scripts) support `connectTo` for running local commands that need access to deployed resources — database migrations, seed scripts, or health checks:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, RelationalDatabase, RdsEnginePostgres } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myDatabase = new RelationalDatabase({
    credentials: {
      masterUserPassword: '$Secret(\\'database.password\\')'
    },
    engine: new RdsEnginePostgres({
      version: '16',
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    })
  });

  return {
    resources: { myDatabase },
    scripts: {
      migrate: {
        type: 'local-script-with-bastion-tunneling',
        properties: {
          executeCommand: 'npx prisma migrate deploy',
          connectTo: ['myDatabase']
        }
      }
    }
  };
});
// [!code focus-end]`
    }
  ]}
/>

Run the script with [`stacktape script:run`](/cli/script-run):

```bash
stacktape script:run --scriptName migrate --stage production --region eu-west-1
```

The `local-script-with-bastion-tunneling` type tunnels connections through a [bastion host](/resources/security/bastion-host), giving your local machine secure access to VPC-only databases. The injected environment variables are adjusted to route through the tunnel automatically.

For databases with `accessibility` set to `internet`, a plain `local-script` type works without tunneling.

The `connectTo` property is available on all compute resources and scripts:

## VPC and networking

When you connect to VPC-bound resources (databases, Redis clusters with `accessibility: 'scoping-workloads-in-vpc'`), Stacktape handles VPC placement automatically:

- **Lambda functions** are placed inside the VPC so they can reach the database
- **Container workloads** are placed in the VPC
- **Security group rules** are created to allow traffic on the correct port

<Warning>
Lambda functions placed in a VPC lose direct internet access. If your function needs to call external APIs while also connecting to a VPC database, configure NAT Gateways in `stackConfig.vpc.nat` or use a database with `accessibility: 'internet'` for development stages.
</Warning>

## FAQ

### How does connectTo differ from manually setting environment variables?

The `connectTo` property does three things that manual `environment` entries cannot: it grants IAM permissions, opens security group rules for network-bound resources, and resolves dynamic values (like hostnames only known after deployment). Manual environment variables only set static or directive-resolved values without affecting permissions or networking.

### Can I connect to resources in a different stack?

Not directly through `connectTo` — it only references resources within the same stack. To access resources from another stack, use [`$CfStackOutput()`](/configuration/directives) to import exported outputs, or share a VPC between stacks using `stackConfig.vpc.reuseVpc` and communicate via known endpoints.

### Does connectTo work with container side containers?

Yes. The `connectTo` property on a container workload applies to all containers in the task, including side containers. All containers in the same task share the same IAM role and environment variables, so side containers have access to the same `STP_*` variables and permissions.

### What happens if I remove a resource from connectTo?

On the next deployment, Stacktape removes the IAM permissions, security group rules, and environment variables for that resource. Your application code will no longer have the `STP_*` variables available, and AWS SDK calls to that resource will fail with permission errors.

### Can I use connectTo with a DynamoDB table that has streams enabled?

Yes. When a DynamoDB table has `streamType` configured, `connectTo` injects the `streamArn` parameter in addition to `name` and `arn`. The IAM permissions include `dynamodb:DescribeStream` and other stream-related actions.

### How do I connect to an external database not managed by Stacktape?

Use manual `environment` variables with the connection string. `connectTo` only works with resources defined in the same Stacktape config. For external services, set environment variables directly or use [`$Secret()`](/configuration/secrets) to reference credentials stored in the Stacktape Console.

### Does connectTo add latency to my function?

No. `connectTo` is a deploy-time configuration mechanism — it sets IAM policies, security group rules, and environment variables during deployment. At runtime, there is no additional middleware, proxy, or indirection between your code and the target resource.

### Can I limit which actions connectTo grants?

Not directly — `connectTo` grants a fixed set of permissions per resource type. If you need narrower permissions (e.g., read-only access to a bucket), use `iamRoleStatements` with specific actions instead of or alongside `connectTo`. You can still use `connectTo` for the env var injection and network access while restricting IAM separately.

### What is the aws:ses macro and when should I use it?

The `aws:ses` string in a `connectTo` array grants full AWS Simple Email Service permissions (`ses:*`) to the consuming resource. Use it when your function or service sends emails via the AWS SDK. Unlike resource names, it does not inject environment variables — it only adds IAM permissions.

### How do I see which environment variables are injected for my resources?

Run [`stacktape info:stack`](/cli/info-stack) after deploying to see all referenceable parameters for each resource. The injected env var names follow the `STP_[RESOURCE_NAME]_[PARAM]` pattern. You can also use [`stacktape debug:logs`](/cli/debug-logs) to inspect the runtime environment of a running workload.
