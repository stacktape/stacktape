# Connecting Resources

The `connectTo` property wires resources together inside a Stacktape stack. When you add a resource name to a workload's `connectTo` list, Stacktape grants IAM permissions, injects environment variables with connection details, and — for relational databases and Redis clusters — opens security group access. A single line replaces manual IAM policies, networking rules, and environment variable configuration.

## When to use

Use `connectTo` whenever one resource needs to interact with another at runtime. A [Lambda function](/resources/compute/lambda-function) querying a database, a [web service](/resources/compute/web-service) publishing to a queue, a [deployment script](/resources/advanced/deployment-scripts) running migrations — all are `connectTo` use cases. It is the default mechanism for connecting resources and should be your first choice over manual IAM and environment configuration.

## When NOT to use

Skip `connectTo` when you need permissions narrower than the defaults it grants. For example, `connectTo` gives a workload full read/write access to a bucket — if you only need read access, use `iamRoleStatements` with specific actions and set environment variables manually via [`$ResourceParam()`](/configuration/directives). You cannot selectively reduce the permissions `connectTo` grants for a given target; it's all-or-nothing per resource.

## Basic example

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging, RelationalDatabase, RdsEnginePostgres, Bucket } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myDatabase = new RelationalDatabase({
    credentials: {
      masterUserPassword: '$Secret(\\'database.password\\')'
    },
    engine: new RdsEnginePostgres({
      version: '16',
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    })
  });

  const uploads = new Bucket({});

  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    connectTo: ['myDatabase', 'uploads']
  });

  return {
    resources: { myDatabase, uploads, api }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The `connectTo: ['myDatabase', 'uploads']` line gives the Lambda function full access to both the database and the bucket. Stacktape injects connection details as `STP_MY_DATABASE_*` and `STP_UPLOADS_*` environment variables, grants IAM permissions for the bucket, and opens security group access to the database.

## What connectTo does

When you add a resource name to `connectTo`, Stacktape performs up to three actions depending on the target resource type:

1. **Grants IAM permissions** — the consuming resource's execution role gets a policy allowing it to interact with the target (e.g., `s3:*Object*` for buckets, `sqs:SendMessage` for queues). Network-bound resources like relational databases and Redis clusters don't use IAM for data-plane access.

2. **Opens security group access** — for relational databases and Redis clusters, Stacktape creates security group rules allowing the consuming resource to reach the target on the correct port. Lambda functions are automatically placed in the VPC when they connect to these resources.

3. **Injects environment variables** — Stacktape injects connection parameters as `STP_[RESOURCE_NAME]_[PARAM]` environment variables into the consuming resource.

## Environment variable naming

Stacktape converts the resource name and parameter name to `UPPER_SNAKE_CASE` with the `STP_` prefix:

```
STP_[RESOURCE_NAME]_[PARAMETER_NAME]
```

A resource named `myDatabase` with parameter `connectionString` becomes `STP_MY_DATABASE_CONNECTION_STRING`. A resource named `uploads` with parameter `name` becomes `STP_UPLOADS_NAME`. CamelCase names are split at word boundaries: `jobQueue` → `JOB_QUEUE`, `usersTable` → `USERS_TABLE`.

## Environment variables by resource type

The following tables document the environment variables injected by `connectTo` for each supported resource type. Additional parameters may be accessible via [`$ResourceParam()`](/configuration/directives) — use that directive when you need a value not listed here.

### Relational database

| Parameter | Environment variable example | Description |
|---|---|---|
| `connectionString` | `STP_MY_DB_CONNECTION_STRING` | Full connection string with embedded credentials |
| `host` | `STP_MY_DB_HOST` | Database hostname |
| `port` | `STP_MY_DB_PORT` | Database port |
| `readerConnectionString` | `STP_MY_DB_READER_CONNECTION_STRING` | Reader endpoint connection string (Aurora only) |
| `readerHost` | `STP_MY_DB_READER_HOST` | Reader endpoint hostname (Aurora only) |

<Info>
The connection string includes the master username and password embedded in the URL. Use `$Secret()` for the password in your config to keep credentials out of source control. Additional database parameters like `dbName` are available via `$ResourceParam()`.
</Info>

## Resources that grant IAM permissions only

The following resource types can be used as `connectTo` targets to grant IAM permissions, but do not have documented `connectTo` environment variable injection. Use [`$ResourceParam()`](/configuration/directives) to access their parameters as environment variables.

| Target resource | Permissions granted |
|---|---|
| [State machine](/resources/orchestration/state-machine) | `states:StartExecution`, `states:StopExecution`, `states:ListExecutions`, `states:DescribeExecution` |
| [OpenSearch domain](/resources/databases/opensearch) | `es:ESHttp*` |
| [Kinesis stream](/resources/messaging/kinesis-stream) | `kinesis:DescribeStream`, `kinesis:DescribeStreamSummary`, `kinesis:GetRecords`, `kinesis:GetShardIterator`, `kinesis:ListShards`, `kinesis:ListStreams`, `kinesis:PutRecord`, `kinesis:PutRecords` |

## Using injected variables in code

Access the `STP_*` variables through `process.env` in your application code:

```typescript
import { Client } from 'pg';

export const handler = async () => {
  const client = new Client({
    connectionString: process.env.STP_MY_DATABASE_CONNECTION_STRING
  });

  await client.connect();
  const result = await client.query('SELECT NOW()');
  await client.end();

  return { statusCode: 200, body: JSON.stringify(result.rows) };
};
```

For S3 operations, use the injected bucket name:

```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';

const s3 = new S3Client({});

export const handler = async () => {
  await s3.send(new PutObjectCommand({
    Bucket: process.env.STP_UPLOADS_NAME,
    Key: 'hello.txt',
    Body: 'Hello, world!'
  }));

  return { statusCode: 200, body: 'Uploaded' };
};
```

<Tip>
IAM permissions from `connectTo` are applied to the resource's execution role automatically. AWS SDK calls work without explicit credentials — the SDK uses the role's temporary credentials from the environment.
</Tip>

## AWS service macros

In addition to stack resources, `connectTo` supports the `aws:ses` macro for granting AWS SES (Simple Email Service) permissions without a dedicated resource:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const emailSender = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/send-email.ts'
    }),
    connectTo: ['aws:ses']
  });

  return {
    resources: { emailSender }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The `aws:ses` macro grants full SES permissions (`ses:*`) to the consuming resource. No environment variables are injected — you call the SES API directly via the AWS SDK.

## Custom permissions with iamRoleStatements

When you need access to an AWS service not covered by `connectTo`, or need different permissions than `connectTo` provides, use `iamRoleStatements`:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, LambdaFunction, StacktapeLambdaBuildpackPackaging, Bucket } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const uploads = new Bucket({});

  const reader = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/reader.ts'
    }),
    environment: [{ name: 'BUCKET_NAME', value: "$ResourceParam('uploads', 'name')" }],
    iamRoleStatements: [
      {
        Resource: ["$ResourceParam('uploads', 'arn')/*"],
        Action: ['s3:GetObject']
      }
    ]
  });

  return {
    resources: { uploads, reader }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

In this example, the reader function gets read-only access to the bucket via `iamRoleStatements` instead of the full read/write that `connectTo` would grant. The bucket name is injected manually via `$ResourceParam()`.

The `iamRoleStatements` property adds a separate IAM policy alongside any auto-generated policies. If you use both `connectTo` and `iamRoleStatements` targeting the same resource, both policies apply — `iamRoleStatements` does not override or narrow `connectTo` permissions.

## Scripts with connectTo

[Scripts](/configuration/hooks-and-scripts) support `connectTo` for running local commands that need access to deployed resources — database migrations, seed scripts, or health checks:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, RelationalDatabase, RdsEnginePostgres } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myDatabase = new RelationalDatabase({
    credentials: {
      masterUserPassword: '$Secret(\\'database.password\\')'
    },
    engine: new RdsEnginePostgres({
      version: '16',
      primaryInstance: { instanceSize: 'db.t4g.micro' }
    })
  });

  return {
    resources: { myDatabase },
    scripts: {
      migrate: {
        type: 'local-script-with-bastion-tunneling',
        properties: {
          executeCommand: 'npx prisma migrate deploy',
          connectTo: ['myDatabase']
        }
      }
    }
  };
});
// [!code focus-end]`
    }
  ]}
/>

Run the script with [`stacktape script:run`](/cli/script-run):

```bash
stacktape script:run --scriptName migrate --stage production --region eu-west-1
```

The `local-script-with-bastion-tunneling` type tunnels connections through a [bastion host](/resources/security/bastion-host), giving your local machine secure access to VPC-only databases. The injected environment variables are automatically adjusted to route through the tunnel. For databases with `accessibilityMode` set to `'internet'`, a plain `local-script` type works without tunneling.

The `connectTo` property is available on all compute resources and scripts:

- [Worker service](/resources/compute/worker-service)

- Scripts (`local-script`, `local-script-with-bastion-tunneling`, `bastion-script`)

## VPC and networking

When you connect to VPC-bound resources (relational databases or Redis clusters with `accessibilityMode` set to `'scoping-workloads-in-vpc'`), Stacktape handles VPC placement automatically:

- **Lambda functions** are placed inside the VPC so they can reach the database or Redis cluster
- **Container workloads** are placed in the VPC
- **Security group rules** are created to allow traffic on the correct port

<Warning>
Lambda functions placed in a VPC lose direct internet access. If your function needs to call external APIs while also connecting to a VPC database, configure NAT Gateways in `stackConfig.vpc.nat` or use a database with `accessibilityMode` set to `'internet'` for development stages.
</Warning>

## FAQ

### How does connectTo differ from manually setting environment variables?

The `connectTo` property does three things that manual `environment` entries cannot: it grants IAM permissions, opens security group rules for network-bound resources, and resolves dynamic values (like hostnames only known after deployment). Manual environment variables only set static or directive-resolved values without affecting permissions or networking.

### Can I connect to resources in a different stack?

Not directly through `connectTo` — it only references resources within the same stack. To access resources from another stack, use [`$CfStackOutput()`](/configuration/directives) to import exported outputs, or share a VPC between stacks using `stackConfig.vpc.reuseVpc` and communicate via known endpoints.

### Does connectTo work with container side containers?

Yes. The `connectTo` property on a container workload applies to all containers in the task, including side containers. All containers in the same ECS task share the same IAM role and environment variables, so side containers have access to the same `STP_*` variables and permissions.

### What happens if I remove a resource from connectTo?

On the next deployment, Stacktape removes the IAM permissions, security group rules, and environment variables for that resource. Your application code will no longer have the `STP_*` variables available, and AWS SDK calls to that resource will fail with permission errors.

### Can I use connectTo with a DynamoDB table that has streams enabled?

Yes. When a DynamoDB table has `streamType` configured, `connectTo` injects `STP_[TABLE_NAME]_STREAM_ARN` in addition to `NAME` and `ARN`. The IAM permissions include `dynamodb:DescribeStream` for reading from the stream.

### How do I connect to an external database not managed by Stacktape?

Use manual `environment` variables with the connection string. `connectTo` only works with resources defined in the same Stacktape config. For external services, set environment variables directly or use [`$Secret()`](/configuration/secrets) to reference credentials stored in the Stacktape Console.

### Does connectTo add latency to my function?

No. `connectTo` is a deploy-time configuration mechanism — it sets IAM policies, security group rules, and environment variables during CloudFormation deployment. At runtime, there is no additional middleware, proxy, or indirection between your code and the target resource.

### Can I limit which actions connectTo grants?

Not directly — `connectTo` grants a fixed set of permissions per resource type. If you need narrower permissions (e.g., read-only bucket access), do not use `connectTo` for that target's IAM. Instead, set environment variables manually via `$ResourceParam()` and add specific actions in `iamRoleStatements`.

### What is the aws:ses macro and when should I use it?

The `aws:ses` string in a `connectTo` array grants full AWS Simple Email Service permissions (`ses:*`) to the consuming resource. Use it when your function or service sends emails via the AWS SDK. Unlike resource names, it does not inject environment variables — it only adds IAM permissions.

### How do I see which environment variables are injected for my resources?

Run [`stacktape info:stack`](/cli/info-stack) after deploying to see all referenceable parameters for each resource. The injected env var names follow the `STP_[RESOURCE_NAME]_[PARAM]` pattern. You can also inspect the runtime environment of a running workload with [`stacktape debug:logs`](/cli/debug-logs) or by logging `process.env` in your handler.
