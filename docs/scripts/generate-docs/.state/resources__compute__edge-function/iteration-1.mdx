# Edge Function

An edge function is a Lambda function that runs at CloudFront edge locations worldwide, executing code as close to your users as possible. Use it to modify HTTP headers, rewrite URLs, implement authentication checks, or run A/B tests — all before requests reach your origin server or after responses leave it.

## When to use

**Use an edge function when:**

- You need to manipulate CDN requests or responses — add security headers, redirect users, rewrite URLs, or implement geo-based routing
- You want to add lightweight authentication or authorization checks at the edge, before traffic reaches your origin
- You need A/B testing or feature flagging at the CDN layer
- You want to normalize or modify cache keys for better cache hit rates

**Use something else when:**

| Situation | Better option |
|-----------|---------------|
| General-purpose API or backend logic | [Lambda function](/resources/compute/lambda-function) — full runtime support, env vars, VPC access |
| Always-on HTTP server or WebSocket API | [Web service](/resources/compute/web-service) |
| Background processing | [Worker service](/resources/compute/worker-service) |
| Complex logic needing env vars or database access | [Lambda function](/resources/compute/lambda-function) — edge functions cannot use environment variables |

## Basic example

Define an edge function as a resource, then reference it from a CDN's `edgeFunctions` configuration on another resource (such as a [hosting bucket](/resources/frontend/hosting-bucket) or [web service](/resources/compute/web-service)).

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, EdgeLambdaFunction, StacktapeLambdaBuildpackPackaging, HostingBucket } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const addHeaders = new EdgeLambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/add-headers.ts'
    }),
    timeout: 5
  });

  const site = new HostingBucket({
    cdn: {
      enabled: true,
      edgeFunctions: {
        onResponse: 'addHeaders'
      }
    }
  });

  return {
    resources: { addHeaders, site }
  };
});
// [!code focus-end]`
    }
  ]}
/>

The edge function handler receives a CloudFront event and returns a modified request or response:

```typescript
// src/add-headers.ts
export const handler = async (event: any) => {
  const response = event.Records[0].cf.response;
  response.headers['strict-transport-security'] = [
    { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubdomains; preload' }
  ];
  response.headers['x-content-type-options'] = [
    { key: 'X-Content-Type-Options', value: 'nosniff' }
  ];
  return response;
};
```

## Event types

Edge functions attach to one of four CloudFront lifecycle events. The event type determines when your code runs and what resource limits apply.

| Event | CDN property | When it runs | Max memory | Max timeout |
|-------|-------------|-------------|------------|-------------|
| Viewer request | `onRequest` | Before cache lookup — every request | 128 MB | 5 s |
| Viewer response | `onResponse` | Before returning to client | 128 MB | 5 s |
| Origin request | `onOriginRequest` | After cache miss, before forwarding to origin | 10,240 MB | 30 s |
| Origin response | `onOriginResponse` | After origin responds, before caching | 10,240 MB | 30 s |

**Viewer events** run on every request and have tight limits — keep your code fast and small. Use these for authentication, redirects, and header manipulation.

**Origin events** only run on cache misses, so they execute less frequently. They have higher limits and are suited for heavier logic like content generation or complex URL rewrites.

<Warning>Stacktape uses `onOriginRequest` internally for CDN setups on buckets, web services, and API Gateways. Overriding it may break default behavior. Only set `onOriginRequest` if you understand the implications.</Warning>

You can also attach edge functions to specific CDN routes using `routeRewrites`, so different URL patterns run different edge logic:

## Limitations

Edge functions run in a distributed environment across CloudFront edge locations. This architecture imposes constraints that differ from standard [Lambda functions](/resources/compute/lambda-function):

- **No environment variables.** Edge functions cannot read environment variables at runtime. Hard-code configuration values or fetch them from an external source (like S3 or DynamoDB).
- **No VPC access.** Edge functions cannot connect to VPC resources like relational databases or Redis clusters.
- **Node.js and Python only.** Supported runtimes are Node.js (18.x–22.x) and Python (3.8–3.13). The runtime is auto-detected from your file extension.
- **x86_64 architecture only.** ARM (Graviton) is not available for edge functions.
- **Memory and timeout limits depend on event type.** Viewer events are capped at 128 MB / 5 seconds. Origin events allow up to 10,240 MB / 30 seconds.

Defaults: `memory` is `128` MB, `timeout` is `3` seconds.

## Connecting to resources

The `connectTo` property works differently for edge functions than for other compute resources. Because edge functions cannot use environment variables or access VPC resources, `connectTo` **only grants IAM permissions** — it does not inject connection strings or open network access.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'stacktape.ts',
      lang: 'typescript',
      code: `import { defineConfig, EdgeLambdaFunction, StacktapeLambdaBuildpackPackaging, Bucket } from 'stacktape';

export default defineConfig(() => {
  const myBucket = new Bucket({});

  const edgeFn = new EdgeLambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/edge-handler.ts'
    }),
    // [!code focus-start]
    connectTo: ['myBucket']
    // [!code focus-end]
  });

  return {
    resources: { myBucket, edgeFn }
  };
});`
    }
  ]}
/>

This grants the edge function IAM permissions to access `myBucket`, but you must construct the bucket name or ARN manually in your handler code (since no `STP_*` environment variables are injected).

**Resources that work with `connectTo`:** Buckets, DynamoDB tables, SQS queues, SNS topics, Lambda functions, event buses, state machines, user auth pools, OpenSearch domains, Kinesis streams, and `aws:ses`.

**Resources that do NOT work with `connectTo`:** Relational databases, Redis clusters (require VPC access), and MongoDB Atlas clusters.

For permissions not covered by `connectTo`, use `iamRoleStatements` to add raw IAM policy statements.

## Logging

Edge function logs are sent to CloudWatch Logs, but **in the region where the function executed** — not your stack's region. If a user in Tokyo triggers your edge function, the logs appear in the `ap-northeast-1` CloudWatch console.

Configure log retention with the `logging` property.
