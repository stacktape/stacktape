# Edge Function

A Stacktape edge function is Lambda code that runs on CloudFront edge events, close to the viewer or origin request flow. Use edge functions when CDN-layer code needs to modify headers, rewrite URLs, implement A/B testing, or add lightweight auth checks before a request reaches the origin.

## When to use

An edge function is the right choice when request or response logic belongs at the CDN layer instead of inside your origin application. Stacktape defines the function as an `EdgeLambdaFunction` resource, and CloudFront invokes that function only after a CDN-capable resource references it from its `edgeFunctions` configuration.

- **Request rewriting** — normalize URLs or route legacy paths before a request reaches your application
- **Header manipulation** — add, remove, or transform request and response headers at the CloudFront edge
- **Lightweight access checks** — reject obviously invalid requests before they reach an origin, while keeping full authentication flows in your application
- **Edge experiments** — implement simple A/B testing decisions where the decision needs to happen during the CloudFront event flow

## When NOT to use

- **General backend compute** — use a [Lambda function](/resources/compute/lambda-function) when the code does not need to run on CloudFront viewer or origin events.
- **Containerized APIs** — use a [web-service](/resources/compute/web-service) when you need an always-on HTTP application with container packaging, load balancing, and application-level routing.
- **Environment-variable based integrations** — edge functions cannot use environment variables. Avoid edge code that depends on Stacktape-injected names, secrets, connection strings, or runtime configuration.
- **VPC resource access** — edge functions cannot connect to VPC resources. Keep database-backed or Redis-backed logic behind the CDN in a [Lambda function](/resources/compute/lambda-function), [web-service](/resources/compute/web-service), or [private-service](/resources/compute/private-service).
- **CDN behavior without code** — prefer CDN configuration when routing or cache behavior can be expressed without Lambda code. Add an edge function only when code must inspect or mutate the CloudFront event.

## Basic example

This example defines a Node.js edge function packaged from a local TypeScript file. The function becomes active only when a CDN-capable resource references it from `edgeFunctions`; the `EdgeLambdaFunction` resource itself does not expose a standalone URL or event trigger.

// [!code focus-start]
export default defineConfig(() => {
  const rewriteRequest = new EdgeLambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/edge/rewrite-request.ts'
    }),
    runtime: 'nodejs22.x',
    memory: 128,
    timeout: 3
  });

  return {
    resources: { rewriteRequest }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`runtime` is optional because Stacktape can auto-detect the runtime from the file extension. Set `runtime` explicitly when you want to pin a supported edge runtime: `nodejs18.x`, `nodejs20.x`, `nodejs22.x`, `nodejs24.x`, or Python `3.8` through `3.13`. `memory` defaults to `128` MB, and `timeout` defaults to `3` seconds.

```ts
export const handler = async (event: any) => {
  const request = event.Records[0].cf.request;

  if (request.uri === '/docs') {
    request.uri = '/docs/';
  }

  request.headers['x-edge-rewrite'] = [{ key: 'X-Edge-Rewrite', value: 'true' }];

  return request;
};
```

The handler example uses the CloudFront event shape and returns the modified request. Keep edge handlers small: request and response manipulation is a better fit than database calls, heavy computation, or application workflows.

## Attachment

A Stacktape edge function is invoked only after a CDN-capable resource references it from that resource's `edgeFunctions` configuration. The attachment shape is defined on the CDN-capable resource, not on `EdgeLambdaFunctionProps`, so the edge-function page defines the reusable Lambda code rather than the CDN routing rule.

Treat `EdgeLambdaFunction` as the deployable unit for edge code: packaging, runtime, IAM permissions, memory, timeout, and logging live on the function. The CDN-capable resource decides which CloudFront event invokes that code. If the CDN-side source for a resource is not visible on this page, avoid inferring attachment properties from the edge function alone.

## Packaging

Edge functions use Lambda packaging, not container packaging. Stacktape supports two Lambda packaging modes for this resource: `StacktapeLambdaBuildpackPackaging` for source-based packaging and `CustomArtifactLambdaPackaging` for a pre-built deployment package.

For most teams, `StacktapeLambdaBuildpackPackaging` is the better default because Stacktape packages the entry file and dependencies from source. Choose a Node.js or Python runtime for edge functions, even though the general Lambda buildpack supports more languages, because the `EdgeLambdaFunctionProps` runtime list is restricted to Node.js and Python.

Use `CustomArtifactLambdaPackaging` when your team already has a build pipeline that produces the Lambda deployment package. The `packagePath` points to the pre-built artifact, and `handler` uses the `{{filepath}}:{{functionName}}` form documented by the packaging type.

## Runtime limits

Edge function memory and timeout limits depend on the CloudFront event type that invokes the function. Stacktape exposes `memory` and `timeout` on the `EdgeLambdaFunction` resource, but the maximum values are determined by whether the function is attached to viewer events or origin events.

| Event placement | Memory limit | Timeout limit | Good fit |
|---|---:|---:|---|
| Viewer event | 128 MB | 5 seconds | Header changes, redirects, URL rewrites, request blocking |
| Origin event | 10,240 MB | 30 seconds | Heavier request/response logic that still belongs at the CDN layer |

The default `memory` is `128`, and the default `timeout` is `3`. Use the defaults for viewer-request and viewer-response functions unless you have measured a need to change them. Increase timeout or memory only for origin events where the CDN attachment path allows the higher limits.

<Info>Edge function runtimes are limited to Node.js and Python in Stacktape: Node.js 18, 20, 22, or 24, and Python 3.8 through 3.13. The `runtime` property is optional and can be auto-detected from the file extension.</Info>

## Permissions

Edge functions can receive IAM permissions through `connectTo` or explicit `iamRoleStatements`, but edge functions cannot use Stacktape's normal runtime environment-variable injection or VPC connectivity. This makes edge functions a poor fit for code that needs database connection strings, Redis endpoints, or private network access.

Use `connectTo` when IAM permissions are enough, such as S3 bucket access, DynamoDB access, or SES permissions. Use `iamRoleStatements` for finer-grained permissions that are not covered by `connectTo`.

`connectTo: ['assets']` grants IAM permissions for the referenced bucket. Unlike `connectTo` on regular functions or containers, edge-function `connectTo` does not inject variables such as bucket names or connection strings, so avoid code that depends on Stacktape-injected environment variables.

## Logging

Edge function logs are sent to CloudWatch Logs in the AWS region where the function executes, not necessarily the stack's AWS region. This matters when debugging global traffic, because requests from different viewer locations can produce logs in different regions.

Use edge-function logging for operational visibility, but avoid relying on one regional log group as the complete story for globally distributed requests. If you are debugging a specific user path, check the CloudWatch region where CloudFront executed the Lambda function for that request.

## FAQ

### What is a Stacktape edge function?

A Stacktape edge function is an `EdgeLambdaFunction` resource that runs Lambda code on CloudFront events. The function is meant for request and response manipulation at the CDN edge, such as URL rewrites, header changes, lightweight auth checks, and A/B testing. Edge functions are referenced from CDN `edgeFunctions` configuration on a CDN-capable resource.

### How do I attach an edge function to CloudFront?

Stacktape defines the edge function as a separate `EdgeLambdaFunction` resource. CloudFront invokes the function only after a CDN-capable resource references it from that resource's `edgeFunctions` configuration. The attachment property belongs to the CDN-capable resource, not to `EdgeLambdaFunctionProps`.

### Which runtimes can an edge function use?

A Stacktape edge function can use Node.js and Python runtimes only. The supported values are `nodejs18.x`, `nodejs20.x`, `nodejs22.x`, `nodejs24.x`, `python3.8`, `python3.9`, `python3.10`, `python3.11`, `python3.12`, and `python3.13`. The `runtime` property is optional because Stacktape can auto-detect it from the file extension.

### Can an edge function use environment variables?

No. Stacktape's edge-function source explicitly says edge Lambda functions cannot use environment variables. `connectTo` grants IAM permissions only, so do not design edge code around injected connection strings, API keys, or VPC endpoints. For environment-variable-heavy code, use a [Lambda function](/resources/compute/lambda-function) or [web-service](/resources/compute/web-service).

### Can an edge function connect to a database or Redis?

No. Stacktape edge functions cannot connect to VPC resources, and `connectTo` does not open network access for databases or Redis. Keep database-backed business logic in a regular Lambda function, web service, private service, or worker service. Use [connecting resources](/configuration/connecting-resources) for resources that support normal connection injection.

### What is the timeout limit for Lambda at the edge?

Stacktape exposes `timeout` on edge functions with a default of `3` seconds. Viewer events can run for at most 5 seconds, while origin events can run for at most 30 seconds. If the logic regularly needs more time, move it behind the CDN in a [Lambda function](/resources/compute/lambda-function) or [web-service](/resources/compute/web-service).

### How much memory can an edge function use?

Stacktape exposes `memory` on edge functions with a default of `128` MB. Viewer events are limited to 128 MB, while origin events can use up to 10,240 MB. Keep viewer-event functions small, and only raise memory for origin-event functions where the CDN event type supports the larger allocation.

### Where do edge function logs appear?

Stacktape edge-function logs go to CloudWatch Logs in the region where the function executed, not necessarily the region where the stack was deployed. This is normal for Lambda at the edge because CloudFront runs code near viewers. When debugging, check the CloudWatch region associated with the request execution path.

### When should I use an edge function vs a Lambda function?

Use an edge function when code must run during a CloudFront viewer or origin request/response flow. Use a regular [Lambda function](/resources/compute/lambda-function) for backend work that does not need CloudFront edge events, especially code that needs environment variables or normal Stacktape resource connections. Edge functions are CDN event handlers, not a general backend compute layer.

### When should I use an edge function vs CDN configuration?

Use CDN configuration when routing, caching, or origin behavior can be expressed without code. Add an edge function only when code needs to inspect or mutate the CloudFront request or response event. Code at the edge adds deployment and debugging complexity, so prefer configuration for static behavior and reserve edge functions for logic.
