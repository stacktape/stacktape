# Edge Function

A Stacktape edge function is a Lambda function for CloudFront edge events, where code can inspect or change requests and responses close to the viewer. Use edge functions for header changes, URL rewrites, A/B routing, or lightweight auth checks that must run before traffic reaches the origin.

## When to use

An edge function is the right choice when request or response logic belongs at the CDN layer instead of inside your origin application. Stacktape defines the function as an `EdgeLambdaFunction` resource, and edge functions run on CloudFront events after they are referenced from CDN `edgeFunctions` configuration.

- **Request rewriting** — normalize URLs, route legacy paths, or send traffic to different origins before the request reaches your app
- **Header manipulation** — add, remove, or transform request and response headers at the CloudFront edge
- **Lightweight access checks** — reject obviously invalid requests before they reach an origin, while keeping full authentication flows in your application
- **Edge experiments** — implement simple A/B routing decisions where the decision needs to happen before origin selection

## When NOT to use

- **General backend compute** — use a [Lambda function](/resources/compute/lambda-function) when the code does not need to run on CloudFront viewer or origin events.
- **Containerized APIs** — use a [web-service](/resources/compute/web-service) when you need an always-on HTTP application with container packaging, custom domains, load balancing, and application-level routing.
- **Environment-variable based integrations** — edge functions cannot use environment variables. Use a regular Lambda function or container workload for code that needs injected configuration, secrets, or connection strings.
- **VPC resource access** — edge functions cannot connect to VPC resources. Keep database-backed or Redis-backed logic behind the CDN in a [Lambda function](/resources/compute/lambda-function), [web-service](/resources/compute/web-service), or [private-service](/resources/compute/private-service).
- **CDN routing without code** — prefer plain [CDN](/resources/networking/cdn) configuration when cache behavior, origin routing, or static redirects are enough. Add an edge function only when code must inspect or mutate the CloudFront event.

## Basic example

This example defines a Node.js edge function packaged from a local TypeScript file. The function becomes useful when CDN `edgeFunctions` configuration references it, because edge functions run on CloudFront events rather than from a standalone URL.

<CodeBlock

  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import { defineConfig, EdgeLambdaFunction, StacktapeLambdaBuildpackPackaging } from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const rewriteRequest = new EdgeLambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/edge/rewrite-request.ts',
      handlerFunction: 'handler'
    }),
    runtime: 'nodejs22.x',
    memory: 128,
    timeout: 3
  });

  return {
    resources: { rewriteRequest }
  };
});
// [!code focus-end]`
    }
  ]}
/>

`runtime: 'nodejs22.x'` selects Node.js 22; Stacktape edge functions support Node.js 18/20/22/24 and Python 3.8 through 3.13. `memory` defaults to `128` MB, and `timeout` defaults to `3` seconds.

```ts
export const handler = async (event: any) => {
  const request = event.Records[0].cf.request;

  if (request.uri === '/docs') {
    request.uri = '/docs/';
  }

  request.headers['x-edge-rewrite'] = [{ key: 'X-Edge-Rewrite', value: 'true' }];

  return request;
};
```

The handler example uses the CloudFront event shape and returns the modified request. Keep edge handlers small: request and response manipulation is a better fit than database calls, heavy computation, or application workflows.

## Attachment

A Stacktape edge function runs only when CloudFront invokes it through CDN `edgeFunctions` configuration. Define the `EdgeLambdaFunction` resource separately, then reference that function from the CDN configuration on the resource that owns the CDN behavior.

The exact CDN-side property shape belongs to the CDN-capable resource configuration, not the edge function itself. Treat the edge function as reusable edge logic: the resource defines code, runtime, IAM permissions, memory, timeout, and logging; CDN `edgeFunctions` configuration decides which CloudFront event invokes that code.

## Packaging

Edge functions use Lambda packaging, not container packaging. Stacktape supports two Lambda packaging modes for this resource: `StacktapeLambdaBuildpackPackaging` for source-based packaging and `CustomArtifactLambdaPackaging` for a pre-built deployment package.

For most teams, `StacktapeLambdaBuildpackPackaging` is the better default because Stacktape packages the entry file and dependencies from source. For edge functions, choose a Node.js or Python runtime even though the general Lambda buildpack supports more languages, because the edge function runtime list is restricted to Node.js and Python in the resource type.

Use `CustomArtifactLambdaPackaging` when your team already has a build pipeline that produces the Lambda deployment package. The `packagePath` points to the pre-built artifact, and `handler` uses the `{{filepath}}:{{functionName}}` form documented by the packaging type.

## Runtime limits

Edge function memory and timeout limits depend on the CloudFront event type that invokes the function. Stacktape exposes `memory` and `timeout` on the function resource, but the maximum values are determined by whether the function is used for viewer events or origin events.

| Event placement | Memory limit | Timeout limit | Good fit |
|---|---:|---:|---|
| Viewer event | 128 MB | 5 seconds | Header changes, redirects, URL rewrites, request blocking |
| Origin event | 10,240 MB | 30 seconds | Heavier request/response logic that still belongs at the CDN layer |

The default `memory` is `128`, and the default `timeout` is `3`. Use the defaults for viewer-request and viewer-response functions unless you have measured a need to change them. Increase timeout or memory only for origin events where the CDN attachment path allows the higher limits.

<Info>Edge function runtimes are limited to Node.js and Python in Stacktape: Node.js 18, 20, 22, or 24, and Python 3.8 through 3.13.</Info>

## Permissions

Edge functions can receive IAM permissions through `connectTo` or explicit `iamRoleStatements`, but edge functions cannot use Stacktape's normal runtime environment-variable injection or VPC connectivity. This makes edge functions a poor fit for code that needs database connection strings, Redis endpoints, or private network access.

Use `connectTo` when IAM permissions are enough, such as S3 bucket access, DynamoDB, or SES. Use `iamRoleStatements` for finer-grained permissions that are not covered by `connectTo`.

`connectTo: ['assets']` adds IAM permissions for the referenced bucket; the exact permission set is defined by Stacktape's `connectTo` behavior. Unlike `connectTo` on regular functions or containers, edge-function `connectTo` does not inject variables such as bucket names or connection strings, so pass required static names through code or another supported configuration path instead.

## Logging

Edge function logs are sent to CloudWatch Logs in the AWS region where the function executes, not necessarily the stack's AWS region. This matters when debugging global traffic, because requests from different viewer locations can produce logs in different regions.

Use edge-function logging for operational visibility, but avoid relying on one regional log group as the complete story for globally distributed requests. If you are debugging a specific user path, check the region where CloudFront executed the Lambda function for that request.

## FAQ

### What is a Stacktape edge function?

A Stacktape edge function is an `EdgeLambdaFunction` resource that runs Lambda code on CloudFront events. The function is meant for request and response manipulation at the CDN edge, such as URL rewrites, header changes, lightweight auth checks, and A/B routing. Edge functions run on CloudFront events and are referenced from CDN `edgeFunctions` configuration.

### How do I attach an edge function to CloudFront?

Stacktape defines the edge function as a separate resource, and the source states that edge functions are referenced from CDN `edgeFunctions` configuration. Configure the function with `new EdgeLambdaFunction(...)`, then reference it from the CDN-side configuration. See [CDN](/resources/networking/cdn) for CDN behavior and routing concepts.

### Which runtimes can an edge function use?

A Stacktape edge function can use Node.js and Python runtimes only. The supported runtime values are Node.js 18, 20, 22, and 24, plus Python 3.8 through 3.13. If you need another language, use a regular [Lambda function](/resources/compute/lambda-function) unless the code must run on CloudFront edge events.

### Can an edge function use environment variables?

No. Stacktape's edge-function source explicitly says edge Lambda functions cannot use environment variables. `connectTo` grants IAM permissions only, so do not design edge code around injected connection strings, API keys, or VPC endpoints. For environment-variable-heavy code, use a [Lambda function](/resources/compute/lambda-function) or [web-service](/resources/compute/web-service).

### Can an edge function connect to a database or Redis?

No. Stacktape edge functions cannot connect to VPC resources, and `connectTo` does not open network access for databases or Redis. Keep database-backed business logic in a regular Lambda function, web service, private service, or worker service. Use [connecting resources](/configuration/connecting-resources) for resources that support normal connection injection.

### What is the timeout limit for Lambda at the edge?

Stacktape exposes `timeout` on edge functions with a default of `3` seconds. Viewer events can run for at most 5 seconds, while origin events can run for at most 30 seconds. If the logic regularly needs more time, move it behind the CDN in a [Lambda function](/resources/compute/lambda-function) or [web-service](/resources/compute/web-service).

### How much memory can an edge function use?

Stacktape exposes `memory` on edge functions with a default of `128` MB. Viewer events are limited to 128 MB, while origin events can use up to 10,240 MB. Keep viewer-event functions small, and only raise memory for origin-event functions where the CDN event type supports the larger allocation.

### Where do edge function logs appear?

Stacktape edge-function logs go to CloudWatch Logs in the region where the function executed, not necessarily the region where the stack was deployed. This is normal for Lambda at the edge because CloudFront runs code near viewers. When debugging, check the CloudWatch region associated with the request execution path.

### When should I use an edge function vs a Lambda function?

Use an edge function when code must run during a CloudFront viewer or origin request/response flow. Use a regular [Lambda function](/resources/compute/lambda-function) for event-driven backend work, API integrations, scheduled jobs, or code that needs environment variables and normal Stacktape resource connections. Edge functions are a CDN tool first, not a general backend compute layer.

### When should I use an edge function vs CDN configuration?

Use plain [CDN](/resources/networking/cdn) configuration when cache behavior, routing, or origin settings are enough. Add an edge function only when code needs to inspect or mutate requests or responses. Code at the edge adds deployment and debugging complexity, so prefer configuration for static routing decisions and reserve edge functions for logic.
