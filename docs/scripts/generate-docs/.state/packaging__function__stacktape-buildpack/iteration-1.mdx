# Stacktape Buildpack for Lambda

The Stacktape Lambda buildpack (`StacktapeLambdaBuildpackPackaging`) packages your source code into an optimized AWS Lambda deployment artifact with zero configuration. Point it to an entry file — Stacktape detects the language, bundles your code and dependencies, generates source maps (JS/TS), and uploads the result. This packaging mode applies to [Lambda function](/resources/compute/lambda-function) resources and supports JavaScript, TypeScript, Python, Java, Go, Ruby, PHP, and .NET.

## When to use

Use the Stacktape Lambda buildpack when you want the fastest path from source code to a deployed [Lambda function](/resources/compute/lambda-function). It handles bundling, dependency resolution, and artifact creation automatically. This is the right choice for most Lambda-based projects — API handlers, event-driven processors, scheduled tasks, webhook endpoints, and background functions.

Choose the buildpack if:

- You want zero-config packaging from a source file
- You're using a supported language with standard dependency management
- You don't need a custom build pipeline or post-processing step beyond what the buildpack provides

| Criteria | Stacktape buildpack | [Custom artifact](/packaging/function/custom-artifact) |
|---|---|---|
| Build handled by | Stacktape (automatic) | You (external pipeline) |
| Input | Source file path | Pre-built zip or directory |
| Languages | JS, TS, Python, Java, Go, Ruby, PHP, .NET | Any (you build it) |
| Best for | Most projects | Custom toolchains, pre-compiled artifacts |

## When NOT to use

Use [custom artifact packaging](/packaging/function/custom-artifact) instead when:

- You have a pre-built zip from a separate CI pipeline or build tool
- Your build process requires steps the buildpack doesn't support (cross-compilation with custom toolchains, proprietary build tools)
- You need to include artifacts generated outside your project tree

For container-based workloads ([web services](/resources/compute/web-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload), [batch jobs](/resources/compute/batch-job)), see [container packaging](/packaging/containers/stacktape-buildpack) instead. Lambda packaging applies only to `LambdaFunction` resources.

The smallest valid configuration points the buildpack at your entry file. Stacktape detects the language from the file extension and handles everything else.

## Supported languages

The buildpack detects the language from your entry file extension and applies language-appropriate bundling. All configuration beyond `entryfilePath` is optional — defaults work for most projects.

| Language | Entry file example | Default runtime version | Key config options |
|---|---|---|---|
| JavaScript / TypeScript | `./src/handler.ts` | Node.js 18 | `nodeVersion`, `outputModuleFormat`, `tsConfigPath` |
| Python | `./src/handler.py` | Python 3.12 | `pythonVersion`, `packageManagerFile`, `minify` |
| Java | `./src/Handler.java` | Java 11 | `javaVersion`, `useMaven`, `packageManagerFile` |
| Go | `./src/main.go` | — | — |
| Ruby | `./src/handler.rb` | Ruby 3.3 | `rubyVersion` |
| PHP | `./src/handler.php` | PHP 8.3 | `phpVersion` |
| .NET | `./src/Handler.cs` | .NET 8 | `dotnetVersion`, `projectFile` |

For detailed language-specific tuning — runtime versions, module format, dependency exclusions, and build tool configuration — see [Language-specific configuration](/packaging/function/language-specific-config).

## Entry file

The `entryfilePath` property (required) tells the buildpack where your Lambda handler code lives. The path is relative to your Stacktape config file.

For JavaScript and TypeScript, the buildpack bundles your code starting from this file into a single output file using Bun's built-in bundler. Dependencies with native binaries (like `sharp`, `bcrypt`, or `@prisma/client`) are detected and installed separately in the deployment package rather than being statically bundled.

For Python, use the standard file path format: `./src/handler.py`. For Java, point to your main handler class. For Go, point to `main.go` or the file containing your handler.

## Handler function

The `handlerFunction` property overrides which exported function AWS Lambda invokes. This is optional — the buildpack uses the conventional default for your language. Set it when your entry file exports the handler under a different name than the default.

## Including and excluding files

By default, the buildpack includes only what your entry file transitively imports. Use `includeFiles` and `excludeFiles` to adjust when your function reads files at runtime that aren't statically imported — templates, configuration files, ML models, or data files.

### Including additional files

The `includeFiles` property accepts glob patterns relative to your Stacktape config file. Matched files are added to the deployment package alongside your bundled code. Use this for any file your handler reads at runtime via the filesystem (e.g., `fs.readFile`) rather than through an `import`.

### Excluding files

The `excludeFiles` property removes files from the deployment package that would otherwise be included. This is useful when `includeFiles` globs match more than you need, or when your project tree includes large files (test fixtures, documentation, local data) that the bundler picks up transitively.

### Excluding dependencies

The `excludeDependencies` property removes specific packages from the final deployment package. Use this when a dependency is only needed at build time, or when it's already provided by the Lambda runtime environment. This reduces package size and cold start time.

## Language-specific configuration

The `languageSpecificConfig` property tunes how the buildpack handles your specific language and runtime. Each language has its own set of options. The subsections below cover the most commonly adjusted settings; see [Language-specific configuration](/packaging/function/language-specific-config) for a complete reference.

### Node.js and TypeScript

For JavaScript and TypeScript projects, the buildpack uses Bun's built-in bundler to produce a single output file. Source maps are generated by default so CloudWatch stack traces map back to your original source.

Key options:

- **`nodeVersion`** — major Node.js version to target. Supported: 16, 17, 18, 19, 20, 21, 22, 23, 24. Default: `18`.
- **`outputModuleFormat`** — `'cjs'` (CommonJS, default) or `'esm'` (ES Modules, enables top-level `await`). Some npm packages don't support ESM, and ESM may produce less readable stack traces.
- **`tsConfigPath`** — path to your `tsconfig.json`, used to resolve path aliases during bundling.
- **`emitTsDecoratorMetadata`** — enable for frameworks that rely on TypeScript decorator metadata reflection (NestJS, TypeORM).
- **`dependenciesToExcludeFromBundle`** — packages treated as external (not bundled into the single output file). They're installed separately in the deployment package. Use `'*'` to exclude all dependencies from the bundle.
- **`dependenciesToExcludeFromDeploymentPackage`** — removes non-bundled dependencies from the final package. Only applies to dependencies already excluded from the bundle.
- **`disableSourceMaps`** — skips source map generation, reducing package size but making production errors harder to debug.
- **`outputSourceMapsTo`** — saves source maps to a local directory instead of uploading them. Useful for external error tracking tools like Sentry or Datadog.

The `nodeVersion` controls the build target for bundling and determines which Lambda runtime is used. `outputModuleFormat: 'esm'` enables ES Module output — choose this when you need top-level `await` or your dependencies are ESM-only. The `dependenciesToExcludeFromBundle` array is critical for packages with native binaries: marking `@prisma/client` as external ensures the buildpack installs the correct platform-specific binary for the Lambda environment instead of bundling your local machine's binary.

<Info>

Packages with native binaries (like `sharp`, `bcrypt`, or `@prisma/client`) should be excluded from the static bundle using `dependenciesToExcludeFromBundle`. The buildpack installs them separately so the correct platform-specific binary is included in the deployment package.

</Info>

When your stack has multiple Node.js Lambda functions, the buildpack automatically optimizes shared dependencies into a Lambda Layer so each function's individual package is smaller. This happens transparently and reduces both upload size and cold start time.

### Python

The buildpack packages your Python source and installs dependencies using `uv`, a fast Python package manager. It supports `requirements.txt`, `Pipfile`, and `pyproject.toml` as dependency sources.

Key options:

- **`pythonVersion`** — Python runtime version. Supported: 2.7, 3.6, 3.7, 3.8, 3.9, 3.11, 3.12, 3.13, 3.14. Default: `3.12`.
- **`packageManagerFile`** — path to your dependency file (`requirements.txt`, `Pipfile`, or `pyproject.toml`).
- **`minify`** — minifies Python source to reduce package size. Default: `true`. Disable if minification breaks libraries that rely on runtime source introspection.

Setting `pythonVersion: 3.12` targets the Python 3.12 Lambda runtime. The `packageManagerFile` tells the buildpack where to find your dependencies. <Warning>

Python minification (`minify: true`, the default) can break libraries that rely on runtime introspection of source code — Pydantic, SQLAlchemy, and similar metaprogramming-heavy frameworks. Set `minify: false` if you use these libraries.

</Warning>

### Java

The buildpack builds Java projects using Gradle by default. Set `useMaven: true` to switch to Maven.

Key options:

- **`javaVersion`** — Java runtime version. Supported: 8, 11, 17, 19. Default: `11`.
- **`useMaven`** — use Maven instead of Gradle for building.
- **`packageManagerFile`** — path to `pom.xml` (Maven) or `build.gradle` (Gradle).

### Other languages

The remaining supported languages require minimal or no language-specific configuration:

- **Go** — no configuration needed. Point `entryfilePath` to your `main.go` file. The buildpack compiles and packages the binary.
- **Ruby** — `rubyVersion` (supported: 3.2, 3.3; default: `3.3`).
- **PHP** — `phpVersion` (supported: 8.2, 8.3; default: `8.3`).
- **.NET** — `dotnetVersion` (supported: 6, 8; default: `8`) and `projectFile` (path to your `.csproj` file).

See [Language-specific configuration](/packaging/function/language-specific-config) for detailed examples of each language.

## Build caching

The Stacktape Lambda buildpack caches deployment packages based on a content-hash digest of your source files and dependencies. When you redeploy without changing any source code, the buildpack skips bundling entirely and reuses the cached artifact. This makes iterative deployments significantly faster — only functions with actual code changes are repackaged.

Caching works automatically. You don't need to configure it. The digest calculation covers your source files, dependency lock files, and buildpack configuration, so any change to code, dependencies, or packaging settings triggers a fresh build.

## Processor architecture

Lambda functions support two processor architectures: `x86_64` (default) and `arm64` (AWS Graviton). The architecture is configured on the [Lambda function resource](/resources/compute/lambda-function) itself, not on the packaging. When you choose `arm64`, the buildpack automatically builds for the ARM architecture — no packaging-level changes are needed.

```typescript
// Architecture is set on the LambdaFunction, not on packaging
const api = new LambdaFunction({
  packaging: new StacktapeLambdaBuildpackPackaging({
    entryfilePath: './src/handler.ts'
  }),
  architecture: 'arm64' // Graviton — ~20% cheaper per GB-second
});
```

ARM-based Lambda functions are typically ~20% cheaper per GB-second compared to x86. Most code works on both architectures without changes. If your function uses native binary dependencies, verify they support the ARM platform before switching.

## FAQ

### What languages does the Stacktape Lambda buildpack support?

The buildpack supports JavaScript, TypeScript, Python, Java, Go, Ruby, PHP, and .NET. It detects the language from your entry file extension and applies the appropriate bundling strategy. JS/TS code is bundled into a single file with source maps. Python dependencies are installed via `uv`. Java projects are built with Gradle or Maven. See [Language-specific configuration](/packaging/function/language-specific-config) for all tuning options.

### Should I use the Stacktape buildpack or a custom artifact?

Use `StacktapeLambdaBuildpackPackaging` for most projects — it handles bundling, dependency resolution, and optimization automatically with no configuration beyond `entryfilePath`. Use [custom artifact packaging](/packaging/function/custom-artifact) when you already have a pre-built zip from an external build pipeline, need a custom toolchain the buildpack doesn't support, or want full control over what goes into the deployment package.

### Can I use ES Modules (ESM) with the Lambda buildpack?

Yes. Set `outputModuleFormat: 'esm'` in `languageSpecificConfig` to enable ES Module output, which also enables top-level `await`. The default is `'cjs'` (CommonJS). Be aware that some npm packages don't support ESM, and ESM can produce less readable stack traces. Test your function after switching to catch compatibility issues.

### What is the maximum Lambda deployment package size?

AWS Lambda accepts deployment packages up to 50 MB (zipped) when uploaded directly, or 250 MB (unzipped). The buildpack's single-file bundling (for JS/TS), `excludeFiles`, and `excludeDependencies` options help keep packages small. For functions with very large dependencies that push past these limits, consider using [container-based packaging](/packaging/containers/stacktape-buildpack) instead — container images support up to 10 GB.

### How do I handle native binary dependencies like sharp or Prisma?

Add native binary packages to `dependenciesToExcludeFromBundle` in your `languageSpecificConfig`. The buildpack installs them separately with the correct platform-specific binaries for the Lambda runtime (Amazon Linux), instead of bundling binaries from your local development machine. This is the most common packaging issue — if a dependency works locally but fails in Lambda, excluding it from the bundle is usually the fix.

### Does the buildpack support TypeScript path aliases?

Yes. Set `tsConfigPath` in `languageSpecificConfig` to point to your `tsconfig.json`. The buildpack reads path alias mappings from `compilerOptions.paths` and resolves them during bundling. No additional plugin configuration is needed.

### How does Lambda cold start relate to package size?

Larger deployment packages increase cold start duration because AWS Lambda must download and extract the package before your function can execute. The buildpack helps minimize this by bundling code into a single file (for JS/TS), excluding unnecessary files, and automatically sharing common dependencies via Lambda Layers when your stack has multiple functions. Keeping packages small — under 5-10 MB for latency-sensitive functions — noticeably reduces cold start time.

### Can I deploy the same code as both a Lambda function and a container?

Yes, but you need different packaging configurations. [Lambda function](/resources/compute/lambda-function) resources use `StacktapeLambdaBuildpackPackaging` (this page), which produces a zip artifact. Container workloads like [web services](/resources/compute/web-service) use [container packaging](/packaging/containers/stacktape-buildpack) (`StacktapeImageBuildpackPackaging`), which produces an OCI image. Your application code can be shared, but the packaging configuration and entry point conventions differ.

### How do I debug a Lambda function packaged with the buildpack?

The buildpack generates source maps by default for JavaScript and TypeScript, so CloudWatch stack traces map back to your original source files. Use [`stacktape debug:logs`](/cli/debug-logs) to tail logs from the CLI, or view logs in the [Stacktape Console](/stacktape-console/console-overview). For rapid iteration without full redeployment, use [dev mode](/local-development/dev-mode-overview) (`stacktape dev`).

### What Node.js versions can I use with the Lambda buildpack?

The `nodeVersion` option in `languageSpecificConfig` accepts major versions 16 through 24, with a default of `18`. This controls both the build target for bundling and the Lambda runtime version. AWS periodically deprecates older Node.js runtimes — if you target an older version, check the AWS Lambda documentation for currently supported runtimes to avoid deployment failures.
