# Use the Dev Mode

Stacktape dev mode runs your workloads locally while connecting them to real (or locally emulated) AWS resources. Edit code, save, and see results in seconds — no redeploy needed. One command sets up everything: containers, databases, frontends, and Lambda functions, all wired together.

<Info>
Already familiar with dev mode? Skip ahead to [Deploy your first stage](/getting-started/deploy-your-first-stage).
</Info>

## What you need

You should have a `stacktape.ts` from [step 1](/getting-started/configure-your-stack). You also need:

- **Docker** running on your machine (for containers and local database emulators)
- **Stacktape CLI** installed (`npm install -g stacktape`)
- An AWS account connected — dev mode initializes AWS credentials and, in normal mode, deploys a dev stack when one does not already exist

## Start dev mode

Run this from the directory containing your `stacktape.ts`:

```bash
stacktape dev --stage dev --region eu-west-1
```

On first run, Stacktape deploys a lightweight dev stack to AWS (IAM roles, secrets, and routing needed for local development). Subsequent runs detect the existing dev stack and start without redeploying.

An interactive picker lets you choose which resources to run. To skip the picker and run everything:

```bash
stacktape dev --stage dev --region eu-west-1 --resources all
```

To run only specific resources:

```bash
stacktape dev --stage dev --region eu-west-1 --resources myApi,myDatabase
```

## What runs where

Dev mode splits your stack between local execution and AWS, depending on resource type:

| Resource type | Where it runs | How changes work |
|---|---|---|
| Container workloads (web-service, private-service, worker-service, multi-container-workload) | Docker on your machine | Rebuilt on file change (with `--watch`) or manually (`rs` command) |
| Lambda functions | Packaged locally and code updated on AWS | Repackaged and updated on file change or manual trigger; logs stream to your terminal |
| SSR frontends (Next.js, Astro, Nuxt, SvelteKit, SolidStart, TanStack Start, Remix) | Local dev server | Automatic hot reload via the framework's built-in HMR |
| Hosting buckets (with `dev` config) | Local dev server | Automatic hot reload |
| Databases (PostgreSQL, MySQL, Redis, DynamoDB, OpenSearch) | Docker-backed local emulators | Persistent data in `.stacktape/dev-data/{stage}/` |

Lambda functions have their code built and pushed to AWS, where they run. Stacktape creates tunnels so those Lambda functions can reach your locally emulated databases. Logs from Lambda invocations stream directly to your terminal.

## The edit-save-see loop

This is the core dev mode experience. Here is an example config that shows a typical setup:

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  RelationalDatabase,
  RdsEnginePostgresql16
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const myDatabase = new RelationalDatabase({
    engine: new RdsEnginePostgresql16({
      instanceSize: 'db.t4g.micro'
    })
  });

  const api = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handler.ts'
    }),
    connectTo: [myDatabase],
    environment: [{ name: 'STAGE', value: '$Stage()' }]
  });

  return {
    resources: { myDatabase, api }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The `instanceSize` property controls the RDS compute tier for production deployments. `db.t4g.micro` is the smallest general-purpose instance (2 vCPU, 1 GB RAM) and costs roughly ~$12/month. For workloads that need more headroom, use `db.t4g.small` (2 GB RAM) or `db.t4g.medium` (4 GB RAM). In dev mode, the instance size doesn't matter — the database runs as a local Docker container regardless.

With this config, running `stacktape dev --stage dev --region eu-west-1 --resources all`:

1. Starts a local PostgreSQL emulator (Docker container on your machine)
2. Builds and updates the Lambda function code on AWS, with tunnels so it can reach your local database
3. Injects environment variables so your code can connect — for a database resource named `myDatabase`, the connection string is available as `STP_MY_DATABASE_CONNECTION_STRING`

Now edit `./src/handler.ts`, save, and either:

- **With `--watch` flag**: Stacktape detects the change and automatically repackages and updates the Lambda code
- **Without `--watch`**: Type `rs` and press Enter in the terminal to trigger a rebuild

For SSR frontends (Next.js, Astro, etc.), changes are picked up instantly through the framework's built-in hot module replacement — no rebuild step needed.

## Enable file watching

Add `--watch` to automatically rebuild workloads when source files change:

```bash
stacktape dev --stage dev --region eu-west-1 --resources all --watch
```

Container workloads stop, rebuild, and restart. Lambda functions repackage and update on AWS. Frontends handle their own watching natively regardless of this flag.

## Local database emulation

Dev mode starts Docker-backed local emulators for supported database resources:

| Config resource type | Local Docker image |
|---|---|
| PostgreSQL / Aurora PostgreSQL | `postgres` |
| MySQL / Aurora MySQL | `mysql` / `mariadb` |
| Redis | `redis` |
| DynamoDB | `amazon/dynamodb-local` |
| OpenSearch | `opensearchproject/opensearch` |

Database data persists between sessions in `.stacktape/dev-data/{stage}/`. This means you can stop and restart dev mode without losing your test data.

To start with a clean database:

```bash
stacktape dev --stage dev --region eu-west-1 --freshDb
```

To skip local emulation and connect to an already-deployed database on AWS:

```bash
stacktape dev --stage dev --region eu-west-1 --remoteResources myDatabase
```

## Environment variables

Local workloads receive `STP_*` environment variables pointing to local endpoints. For a resource named `myDatabase`, the connection string is available as `STP_MY_DATABASE_CONNECTION_STRING`. The exact host and port values are assigned by dev mode at startup and visible in the dev-mode TUI output.

Your application code works identically in dev mode and production — it reads the same environment variable names regardless of whether it's connecting to a local emulator or a deployed AWS database. Other database parameters (host, port, database name) are available via the [`$ResourceParam()` directive](/configuration/directives) when needed.

## Terminal commands

While dev mode is running, type commands in your terminal:

| Command | Action |
|---|---|
| `rs` | Rebuild all workloads |
| `rs myApi` | Rebuild a specific workload |
| `c` or `clear` | Clear the log output |
| `q` or `quit` | Stop dev mode |

## Useful flags

| Flag | Description |
|---|---|
| `--resources` | Which resources to run (`all` or comma-separated names) |
| `--skipResources` | Exclude specific resources |
| `--watch` | Auto-rebuild on file changes |
| `--freshDb` | Delete local database data before starting |
| `--remoteResources` | Connect to deployed AWS resources instead of local emulation |
| `--noTunnel` | Disable Lambda-to-local-database tunneling |
| `--devMode` | Which mode to use: `normal` (default) or `legacy` |

<Tip>
Run `stacktape help dev` to see all available flags and their short aliases.
</Tip>

## Agent mode for AI coding assistants

Dev mode includes an HTTP API for programmatic control — useful when working with AI coding assistants like Claude Code or Cursor. Pass `--agent` to start as a detached daemon with the HTTP server:

```bash
stacktape dev --stage dev --region eu-west-1 --resources all --agent
```

The agent server starts on port 7331 by default (or the next available port) and exposes endpoints for status, logs, environment variables, rebuild triggers, stop, local-resource health checks, database query helpers, and AWS SDK access. See [Agent mode in dev](/using-with-ai/agent-mode-in-dev) for the full endpoint reference.

## Debugging

While dev mode handles the fast iteration loop, Stacktape also provides debugging commands for inspecting your running resources:

- [`stacktape debug:logs`](/cli/debug-logs) — Stream or search logs from your Lambda functions and container workloads
- [`stacktape debug:metrics`](/cli/debug-metrics) — View CloudWatch metrics for any resource

For deeper coverage of debugging workflows, see [Dev mode overview](/local-development/dev-mode-overview).

## Legacy mode

If you already have a fully deployed stack and want to run workloads against it (without local database emulation):

```bash
stacktape dev --stage prod --region eu-west-1 --devMode legacy
```

Legacy mode skips the dev-stack deployment and connects directly to your existing deployed resources. It requires the stack to already exist — use it when you need to test against production-like infrastructure. Local database emulation is not available in legacy mode; all database connections go to the deployed AWS resources.

## FAQ

### Does dev mode cost anything on AWS?

The dev stack itself is minimal (IAM roles, secrets, routing) and costs virtually nothing. Lambda function invocations during development are billed at standard AWS Lambda rates — typically pennies for development use. Local containers and database emulators run on your machine at no AWS cost.

### Can I run dev mode without Docker?

Docker is required for container workloads and local database emulation. If your config has only Lambda functions with no local databases, dev mode works without Docker. For database-heavy projects, Docker Desktop or a compatible runtime (Podman, Colima) must be running.

### How do Lambda functions reach my local databases?

Stacktape creates tunnels so that Lambda functions running on AWS can reach database emulators on your machine. Your function code uses the same `STP_*` connection strings in both dev and production. Disable this with `--noTunnel` if you don't need it.

### Can I use dev mode with multiple developers on the same project?

Yes. Each developer should use a unique, short stage name (e.g., `--stage dev-alice`, `--stage dev-bob`). Each stage gets its own isolated dev stack and local data directory.

### What happens if I change my Stacktape config?

Changes to `stacktape.ts` (adding resources, changing environment variables, modifying connections) require restarting dev mode. Dev mode reads the config once at startup. If you change your config in a way that affects the dev stack (new resources, new permissions), delete the existing dev stack with `stacktape delete --stage dev --region eu-west-1` and restart dev mode to create a fresh one.

### How is dev mode different from `stacktape deploy`?

Dev mode optimizes for iteration speed. It runs what it can locally (containers, databases, frontends) and pushes only Lambda function code to AWS. A full [`stacktape deploy`](/cli/deploy) creates the complete production-grade infrastructure (load balancers, auto-scaling, CDN, alarms). Use dev mode while building; deploy when you're ready to ship.

### Can I use dev mode with a monorepo?

Yes. Use `--resources` to specify which workloads to run, or `--skipResources` to exclude heavy resources you're not actively working on. Each workload tracks its own source files independently for rebuild detection.

### Does the local database data survive restarts?

Yes. Database data persists in `.stacktape/dev-data/{stage}/` between dev mode sessions. Use `--freshDb` to wipe the data and start clean when needed — useful after schema changes that aren't backward-compatible.

### What's the difference between normal and legacy dev mode?

Normal mode (default) deploys a lightweight dev stack and emulates databases locally with Docker — ideal for daily development. Legacy mode requires an existing fully-deployed stack and connects to the real AWS databases — use it when you need to test against production data or infrastructure you've already deployed.

### Can I connect to a remote database while emulating others locally?

Yes. Use `--remoteResources myDatabase` to connect a specific resource to its deployed AWS version while other databases remain locally emulated. This is useful when you need real production data for one database but want fast local iteration for others.
