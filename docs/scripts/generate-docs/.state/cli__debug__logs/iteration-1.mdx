# debug:logs

`stacktape debug:logs` fetches and displays logs from any deployed compute resource — Lambda functions, web services, private services, worker services, multi-container workloads, and batch jobs. It supports both simple log tailing with filter patterns and advanced analytics via CloudWatch Logs Insights queries.

## Usage

The command requires a deployed stack, a stage, a region, and the name of the resource whose logs you want to inspect.

Fetch the last hour of logs from a Lambda function:

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi
```

## How it works

The command resolves the CloudWatch log group for the given resource by inspecting the deployed CloudFormation stack. It then fetches log streams ordered by last event time and retrieves matching log events. Output is color-coded by log level — errors in red, warnings in yellow, debug in gray — with a summary footer showing total counts and error/warning breakdowns.

Two distinct modes are available:

**Standard mode** retrieves recent log events from the resource's log group. You can narrow results with `--startTime`, `--filter`, and `--limit`. This is the default when no `--query` flag is provided.

**Insights mode** runs a full [CloudWatch Logs Insights](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/AnalyzingLogData.html) query when you pass `--query`. This gives you SQL-like filtering, aggregation, and field extraction across all log streams simultaneously — useful for debugging patterns across many invocations.

## Important flags

### --startTime

Controls how far back to fetch logs. Defaults to 1 hour. Accepts two formats:

Relative time — a number followed by a unit (`h` for hours, `m` for minutes, `d` for days, `s` for seconds):

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --startTime "2h"
```

Absolute time — any string the JavaScript `Date` constructor accepts:

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --startTime "2025-03-15T10:00:00Z"
```

### --filter

Applies a CloudWatch Logs [filter pattern](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/FilterAndPatternSyntax.html) in standard mode. Only log events matching the pattern are returned. Use plain text for simple substring matching or CloudWatch's structured syntax for JSON logs.

Filter for error messages:

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --filter "ERROR"
```

Filter JSON structured logs by field value:

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --filter "{ $.level = \"error\" }"
```

### --query

Switches from standard mode to Insights mode. The value is a [CloudWatch Logs Insights query string](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/CWL_QuerySyntax.html). When `--query` is set, `--filter` is ignored — the query itself handles all filtering and aggregation.

Find the 20 slowest Lambda invocations in the last 6 hours:

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --startTime "6h" --query "fields @timestamp, @duration, @message | sort @duration desc | limit 20"
```

Count errors per hour:

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --startTime "1d" --query "filter @message like /ERROR/ | stats count(*) as errorCount by bin(1h)"
```

### --limit

Caps the number of log events returned in standard mode. Defaults to 100. Set a higher value when you need to scan more history.

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --limit 500
```

### --raw

Outputs log events as raw JSON instead of color-coded formatted output. Useful for piping into `jq` or other tools.

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --raw
```

### --container

For [multi-container workloads](/resources/compute/multi-container-workload), specifies which container's logs to fetch. If omitted, defaults to the main container.

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myWorkload --container sidecar
```

## Examples

### Tail recent Lambda errors

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --filter "ERROR" --startTime "30m"
```

### Investigate a specific time window

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --startTime "2025-03-15T08:00:00Z"
```

### Run an Insights query to find cold starts

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myApi --startTime "1d" --query "filter @type = \"REPORT\" | fields @duration, @initDuration | filter ispresent(@initDuration) | sort @initDuration desc | limit 50"
```

### Get raw JSON output for scripting

```bash
stacktape debug:logs --stage staging --region us-east-1 --resourceName orderProcessor --raw --limit 200
```

### View logs from a specific container in a multi-container workload

```bash
stacktape debug:logs --stage prod --region eu-west-1 --resourceName myWorkload --container nginx
```

## FAQ

### What resource types can I fetch logs from?

`stacktape debug:logs` works with [Lambda functions](/resources/compute/lambda-function), [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload), and [batch jobs](/resources/compute/batch-job). The command resolves the correct CloudWatch log group from the deployed CloudFormation stack automatically.

### What is the difference between --filter and --query?

`--filter` applies a CloudWatch Logs filter pattern in standard mode — it works like a substring or structured-field match against individual log events. `--query` switches to CloudWatch Logs Insights mode, which lets you write SQL-like queries with aggregation, sorting, field extraction, and statistical functions. Use `--filter` for simple searches; use `--query` when you need analytics across many log events.

### How far back can I fetch logs?

CloudWatch retains logs based on the retention policy configured for each log group. By default, Stacktape log groups retain logs indefinitely. The `--startTime` flag controls how far back to look within the retained logs — it defaults to 1 hour. You can set it to days or weeks with relative syntax like `--startTime "7d"`.

### Can I pipe log output to other tools?

Yes. Use `--raw` to get JSON output, then pipe to `jq`, `grep`, or any other tool. The JSON output includes `timestamp`, `message`, and `logStream` fields for each event.

### How do I view logs from a specific container in a multi-container workload?

Pass the `--container` flag with the container name as defined in your Stacktape configuration. If omitted, logs from the main container are returned.

### How much does CloudWatch Logs cost?

AWS CloudWatch Logs charges for ingestion (~$0.50/GB) and storage (~$0.03/GB/month). Logs Insights queries have an additional charge per GB of data scanned (~$0.005/GB). The `debug:logs` command itself is free — the cost comes from the underlying AWS CloudWatch service. See [AWS CloudWatch pricing](https://aws.amazon.com/cloudwatch/pricing/) for current rates.

### Can I view logs in the Stacktape Console instead of the CLI?

Yes. The [Stacktape Console](/stacktape-console/console-overview) provides a web-based log viewer with search, filtering, and live tail. The CLI command is useful for quick checks, scripting, and CI/CD pipelines where a browser is not available.

### What does the formatted output look like?

In standard (non-raw) mode, each log line shows a yellow timestamp followed by the message. Messages are color-coded by detected log level: red for errors, yellow for warnings, gray for debug messages. A summary footer shows total event count with error and warning breakdowns.
