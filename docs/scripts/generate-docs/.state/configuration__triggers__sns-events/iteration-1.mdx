# SNS Events

An SNS event trigger invokes a [Lambda function](/resources/compute/lambda-function) each time a message is published to an [SNS topic](/resources/messaging/sns-topic). SNS (Simple Notification Service) is a pub/sub messaging service — publishers send messages to a topic, and every subscriber receives its own copy. This makes SNS triggers the right fit for fan-out patterns where one event needs to reach multiple independent consumers.

## When to use

SNS event triggers suit scenarios where a single published event needs to fan out to multiple independent processors. A common pattern: an order-placed message publishes to an SNS topic, and separate Lambda functions handle billing, inventory, and email — each receiving its own copy without competing for messages.

Use SNS triggers when:

- **Multiple consumers need the same message.** Each subscriber gets an independent copy. Unlike an [SQS queue](/resources/messaging/sqs-queue), consumers don't compete — they all receive every message.
- **You want push-based delivery.** SNS pushes messages to your function on publish. There's no polling interval and no batching configuration — each message triggers one invocation.
- **You need to mix subscriber types.** A single SNS topic can deliver to Lambda functions, SQS queues, HTTP endpoints, and email simultaneously. The SNS trigger covers the Lambda subscriber path.

## When NOT to use

- **You need message retention or exactly-once processing.** SNS does not retain messages. If your function fails and no dead-letter queue is configured, the message is lost after retries. Use an [SQS trigger](/configuration/triggers/sqs-events) for reliable, at-least-once processing with built-in retention.
- **You need to filter on message body content.** SNS filter policies match on message **attributes** (metadata), not the message body itself. For content-based routing on event payloads, use [EventBridge](/configuration/triggers/event-bus-events).
- **You have a single consumer.** With one subscriber, an SQS queue is simpler and gives you batching, retention, and visibility timeout. SNS adds value when you need fan-out.
- **You need ordered delivery.** Standard SNS topics don't guarantee message order. FIFO SNS topics guarantee order, but FIFO topics cannot deliver to Lambda functions — only to FIFO SQS queues. For ordered processing, use [Kinesis](/configuration/triggers/kinesis-events) or subscribe a FIFO SQS queue to the topic.

## Basic example

A Lambda function subscribes to an SNS topic defined in the same stack. Every message published to the topic triggers one invocation of the function.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  SnsTopic,
  SnsIntegration
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const notificationTopic = new SnsTopic({});

  const logNotification = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/log-notification.ts'
    }),
    memory: 256,
    triggers: [
      new SnsIntegration({
        snsTopicName: 'notificationTopic'
      })
    ]
  });

  return {
    resources: { notificationTopic, logNotification }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The `snsTopicName` references an SNS topic by its resource key name — here, `notificationTopic`. Stacktape automatically creates the SNS subscription and grants the topic permission to invoke the function. You can attach multiple functions to the same topic to build a fan-out pattern.

The handler receives the standard AWS SNS event structure with a `Records` array. Each record contains the message body, subject, message ID, and timestamp:

```typescript
type SNSRecord = {
  Sns: { Subject: string; Message: string; MessageId: string; Timestamp: string };
};

const handler = async (event: { Records: SNSRecord[] }) => {
  for (const record of event.Records) {
    const data = JSON.parse(record.Sns.Message);
    // process the message
  }
};

export default handler;
```

## Filtering messages

SNS filter policies restrict which messages trigger your function based on message attributes. Filtering happens at the SNS level before invocation — your function is never called for non-matching messages, which reduces invocation cost and handler complexity.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  SnsTopic,
  SnsIntegration
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const orderEvents = new SnsTopic({});

  const highValueHandler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/high-value-orders.ts'
    }),
    triggers: [
      new SnsIntegration({
        snsTopicName: 'orderEvents',
        filterPolicy: {
          orderType: ['premium', 'enterprise'],
          amount: [{ numeric: ['>=', 1000] }]
        }
      })
    ]
  });

  return {
    resources: { orderEvents, highValueHandler }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The `filterPolicy` property accepts SNS subscription filter policy syntax. Filters match against **message attributes** — key-value metadata that the publisher attaches to each message, separate from the message body. Supported matching includes exact string values, string prefix matching, numeric comparisons, and `exists` checks. See the [AWS SNS filter policy documentation](https://docs.aws.amazon.com/sns/latest/dg/sns-subscription-filter-policies.html) for the full syntax.

<Tip>If you need to route based on the message body content rather than attributes, use an [EventBridge event bus](/configuration/triggers/event-bus-events) instead. EventBridge supports content-based pattern matching on event payloads.</Tip>

## Using external topics

To subscribe to an SNS topic not managed by your stack — for example, a topic in another AWS account or a shared infrastructure topic — use `snsTopicArn` instead of `snsTopicName`.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  SnsIntegration
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const handler = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/handle-external.ts'
    }),
    triggers: [
      new SnsIntegration({
        snsTopicArn: 'arn:aws:sns:us-east-1:123456789012:shared-events'
      })
    ]
  });

  return {
    resources: { handler }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

You must specify exactly one of `snsTopicName` or `snsTopicArn` — providing both (or neither) is a validation error. When using an external topic, ensure the topic's access policy allows your stack's AWS account to create a subscription.

## Handling delivery failures

SNS attempts to deliver each message to your Lambda function. In rare cases — for example, if the function can't scale fast enough during a traffic spike — delivery may fail. Configure `onDeliveryFailure` to route failed messages to an SQS queue so they aren't lost.

<CodeBlock
  intellisense
  tabs={[
    {
      label: 'TypeScript',
      lang: 'typescript',
      code: `import {
  defineConfig,
  LambdaFunction,
  StacktapeLambdaBuildpackPackaging,
  SnsTopic,
  SnsIntegration,
  SqsQueue
} from 'stacktape';

// [!code focus-start]
export default defineConfig(() => {
  const alerts = new SnsTopic({});
  const failedDeliveries = new SqsQueue({});

  const alertProcessor = new LambdaFunction({
    packaging: new StacktapeLambdaBuildpackPackaging({
      entryfilePath: './src/process-alert.ts'
    }),
    triggers: [
      new SnsIntegration({
        snsTopicName: 'alerts',
        onDeliveryFailure: {
          sqsQueueName: 'failedDeliveries'
        }
      })
    ]
  });

  return {
    resources: { alerts, failedDeliveries, alertProcessor }
  };
});
// [!code focus-end]
`
    }
  ]}
/>

The `onDeliveryFailure` property accepts either `sqsQueueName` (referencing an [SQS queue](/resources/messaging/sqs-queue) in your stack by its resource key) or `sqsQueueArn` (for an external queue). Stacktape configures the SNS subscription's redrive policy so that failed messages land in the queue with delivery failure metadata, allowing you to inspect and reprocess them.

<Warning>Without `onDeliveryFailure` configured, messages that fail all delivery attempts are dropped permanently. For critical workflows, always configure a dead-letter queue.</Warning>

## Throughput and invocation behavior

SNS triggers invoke your Lambda function once per published message. Unlike [SQS](/configuration/triggers/sqs-events) or [Kinesis](/configuration/triggers/kinesis-events) triggers, there is no `batchSize` or `maxBatchWindowSeconds` configuration — each message produces a separate invocation.

Because each message triggers a separate invocation, your function's concurrency scales with the publish rate on the topic. Publishing 100 messages per second can produce up to 100 concurrent Lambda invocations. Monitor your function's concurrency and consider configuring reserved concurrency on the [Lambda function](/resources/compute/lambda-function) if you need predictable scaling limits.

AWS SNS retries Lambda delivery using a backoff policy if the initial invocation fails. If all retries are exhausted, the message routes to the `onDeliveryFailure` destination (if configured) or is dropped.

## FIFO topic restriction

FIFO SNS topics (topics with `fifoEnabled: true`) guarantee strict message ordering and exactly-once delivery. However, FIFO topics **cannot deliver to Lambda functions** — they can only deliver to FIFO SQS queues. Stacktape validates this at deploy time and raises an error if you attempt to subscribe a Lambda function to a FIFO topic.

If you need ordered message processing, subscribe a FIFO SQS queue to the FIFO SNS topic, then use an [SQS trigger](/configuration/triggers/sqs-events) on your Lambda function to consume from that queue.

## FAQ

### When should I use SNS vs SQS as a trigger?

Use [SNS](/resources/messaging/sns-topic) when you need fan-out — one message delivered to multiple independent consumers, each getting its own copy. Use [SQS](/configuration/triggers/sqs-events) when you have a single consumer that needs reliable processing with message retention and batching. If you need both fan-out AND per-consumer reliability, publish to SNS and subscribe SQS queues to the topic — each queue gets its own message copy with independent retry and retention.

### When should I use SNS vs EventBridge?

[EventBridge](/configuration/triggers/event-bus-events) supports content-based filtering on event payloads, complex pattern matching, and routing to many AWS service targets. SNS is simpler and better suited for straightforward pub/sub where you filter on message attributes. Use EventBridge for event-driven architectures with many routing rules; use SNS for direct fan-out to a known set of subscribers.

### Can I subscribe multiple Lambda functions to the same topic?

Yes. Add an `SnsIntegration` trigger to each function, all referencing the same `snsTopicName`. Each function receives an independent copy of every published message. This is the core fan-out pattern. Each subscriber can apply its own `filterPolicy` to receive only relevant messages.

### Can container workloads subscribe to SNS topics?

SNS event triggers are available for [Lambda functions](/resources/compute/lambda-function) and [batch jobs](/resources/compute/batch-job). Container-based workloads ([web services](/resources/compute/web-service), [worker services](/resources/compute/worker-service), [multi-container workloads](/resources/compute/multi-container-workload)) use HTTP-based triggers instead. To process SNS messages in a container, subscribe an SQS queue to the SNS topic and poll the queue from your container application.

### What is the maximum SNS message size?

AWS SNS supports messages up to 256 KB. For larger payloads, store the data in [S3](/resources/storage/s3-bucket) and publish only a reference (the S3 object key) in the SNS message. Your Lambda function then fetches the full payload from S3 at invocation time.

### How do I publish messages to an SNS topic from my function?

Use [`connectTo`](/configuration/connecting-resources) to grant your function publish permissions and inject the topic's ARN. For a topic resource named `notificationTopic`, Stacktape injects `STP_NOTIFICATION_TOPIC_ARN` and `STP_NOTIFICATION_TOPIC_NAME` as environment variables. Use the AWS SDK's `PublishCommand` with the injected ARN to send messages.

### Can I use filter policies with external SNS topics?

Yes. The `filterPolicy` property works identically whether you reference a topic via `snsTopicName` or `snsTopicArn`. Filter policies are configured at the subscription level, not the topic level, so your stack controls its own filtering regardless of who owns the topic.

### Does SNS guarantee message ordering?

Standard SNS topics do not guarantee message ordering — messages may arrive at subscribers in a different order than they were published. FIFO SNS topics guarantee strict ordering within a message group, but FIFO topics cannot deliver directly to Lambda functions. For ordered Lambda processing, subscribe a FIFO SQS queue to a FIFO SNS topic and trigger your function from the [SQS queue](/configuration/triggers/sqs-events).
