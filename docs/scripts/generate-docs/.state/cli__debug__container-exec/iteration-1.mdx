# debug:container-exec

Execute a command inside a running container workload and capture the output. Use this to inspect file systems, check environment variables, run diagnostic scripts, or verify application state without opening an interactive session.

## Usage

```bash
stacktape debug:container-exec --stage prod --region eu-west-1 --resourceName myService --command "ls -la /app"
```

The command connects to a running ECS task via ECS Exec (AWS Systems Manager under the hood), runs your command, and returns structured JSON output containing the command result and exit code.

## How it works

Stacktape identifies the ECS service backing your container workload, finds a running task, and executes the specified command inside the container. The output is returned as JSON with the following structure:

```json
{
  "resourceName": "myService",
  "containerName": "myService",
  "taskArn": "arn:aws:ecs:eu-west-1:123456789:task/cluster/abc123",
  "command": "ls -la /app",
  "output": "total 24\ndrwxr-xr-x 4 root root ...",
  "exitCode": 0
}
```

This command works with any deployed container workload type: [web services](/resources/compute/web-service), [private services](/resources/compute/private-service), [worker services](/resources/compute/worker-service), and [multi-container workloads](/resources/compute/multi-container-workload).

<Info>

This command does not modify your stack or containers. It is read-only and safe to run against production workloads.

</Info>

## Important flags

**`--command`** (required) — The shell command to execute inside the container. Wrap in quotes if it contains spaces or special characters.

**`--resourceName`** (required) — The name of the container workload resource as defined in your Stacktape configuration.

**`--taskArn`** (optional) — The specific ECS task to target. When your service has multiple running instances (scaled out), Stacktape connects to the first available task by default. Use this flag to target a specific instance. You can pass the full ARN or just the task ID suffix.

**`--container`** (optional) — The container name within the task definition. Required when the workload has multiple containers (e.g., a [multi-container workload](/resources/compute/multi-container-workload) with side containers). If omitted and only one container exists, that container is used automatically.

## Examples

Check environment variables in a running service:

```bash
stacktape debug:container-exec --stage prod --region eu-west-1 --resourceName apiService --command "env"
```

Read a configuration file:

```bash
stacktape debug:container-exec --stage prod --region eu-west-1 --resourceName apiService --command "cat /app/config.json"
```

Target a specific task when multiple instances are running:

```bash
stacktape debug:container-exec --stage prod --region eu-west-1 --resourceName apiService --taskArn abc123def --command "ps aux"
```

Execute a command in a specific container of a multi-container workload:

```bash
stacktape debug:container-exec --stage prod --region eu-west-1 --resourceName myWorkload --container sidecar --command "curl localhost:8080/health"
```

Check disk usage inside the container:

```bash
stacktape debug:container-exec --stage prod --region eu-west-1 --resourceName apiService --command "df -h"
```

## When to use debug:container-exec vs container:session

Use `debug:container-exec` when you need to run a single command and capture its output programmatically — for example, in CI scripts, from an AI agent, or when you want structured JSON results.

Use [`container:session`](/cli/container-session) when you need an interactive shell session for hands-on debugging — exploring the file system, running multiple commands, or troubleshooting interactively.

## Common errors

If the resource has no running tasks, the command fails with a "No running tasks found" error. This happens when the service has scaled to zero or all tasks are in a non-RUNNING state. Wait for tasks to start or check your service's deployment status.

If you target a multi-container workload without specifying `--container` and more than one container exists in the task definition, the command fails and lists the available container names.
